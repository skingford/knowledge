# 组织架构及业务系统设计方案

## 组织结构

组织结构分为 公司、平台、合作商、商户、网点，其中合作商下可以在有子级合作商(合作商理论上可以有无限个子合作商)，或者直属商户，该如何设计这个组织可以

```txt
Company
  - Platform
   - Agent
      - Agent
        - Agent
          - Shop
            - Site
      - Shop
        - Site
```

1. 业务背景与挑战
组织特性：存在无限层级的合作商（Agent），且商户（Shop）可由平台或合作商直属。

数据量级：底层网点（Site）众多（单账号可能覆盖 1 万+），业务数据（订单、设备）规模达到千万级甚至亿级。

核心诉求：

高性能：查询 1 万个网点下的设备/订单时，严禁使用超长 IN 语句。

纯净性：业务表尽量只存 site_id，避免过多的冗余导致逻辑复杂。

动态性：支持组织架构平滑迁移，且不丢失历史统计口径。

2. 核心数据模型设计
采用 “三位一体” 的建模方式：parent_id（血缘）+ org_path（路径）+ org_closure（索引）。

2.1 组织架构表 (org_tree)
定义所有组织节点的唯一事实来源。

2.2 闭包索引表 (org_closure)
专门用于解决“祖先与后代”的快速 Join，是提升查询性能的关键。

2.3 业务表 (device / orders)
保持业务表简洁，仅通过 site_id 关联。

3. 核心查询策略
3.1 跨层级聚合查询（高性能场景）
场景：某代理商（ID: 5）查询旗下所有 12,000 个网点的设备列表。

4. 组织架构变动维护逻辑
组织变动（如 Agent A 从区域 1 迁移到区域 2）是低频但复杂的操作，采用**“同步修改 + 异步刷索引”**的策略。

原子更新：修改 org_tree.parent_id，保证物理关系立刻正确。

发送消息：通过 MQ 发送 OrgNodeMovedEvent。

异步处理 (Worker)：

更新 org_path：利用字符串替换函数更新该节点及下属所有子节点的 org_path。

重建 org_closure：

删除涉及该节点的旧 ancestor_id 映射。

根据新的 parent_id 重新插入祖先关系。

提示：此过程建议分批处理，防止数据库长事务锁表。

5. 方案总结与架构建议
5.1 方案优势
扩展性：支持无限层级代理商，业务逻辑不随层级增加而变复杂。

查询极速：闭包表方案规避了 MySQL 对 IN 或 LIKE 在大数据量下的性能抖动。

职责分离：业务表不关心组织结构，组织变动不影响业务表结构，仅影响索引表。

5.2 实施建议
索引优化：确保 org_closure 表在 (ancestor_id, descendant_id) 上有复合唯一索引。

数据隔离：在应用层（Service 层）封装数据权限拦截器，自动根据当前用户的 org_id 关联 org_closure，实现开发者无感知的权限隔离。

读写分离：如果业务量极大，可以将 org_closure 同步至 Redis（使用 Set 结构）或 Elasticsearch 中，进一步提升查询吞吐量。


## 闭包表

1. 闭包表的更新算法：其实是“三板斧”
当一个节点（例如 Agent A）带着它下面的所有子节点移动到新的父节点下时，闭包表的维护分为三个标准化步骤：

第一步：断开旧关联 (Severing)
删除所有“旧祖先”与“该节点及其后代”之间的关系。

第二步：建立新关联 (Linking)
将“新父节点的所有祖先”与“该节点及其后代”建立关联。
这是一个笛卡尔积操作，但因为组织深度通常不超过 10-20 层，计算量其实很小。

第三步：深度修复 (Depth Fix)
更新 depth 字段。如果你不需要 depth 字段做特定查询，这一步甚至可以省略。

2. 为什么你觉得它“麻烦”？（常见痛点与架构对策）
痛点 A：数据膨胀
疑虑：如果我有 1 万个 Site，每个 Site 深度为 5，闭包表就会有 5-6 万行数据。

架构对策：闭包表是典型的窄表（只有 3 个 Long 型字段）。在现代数据库（MySQL/PostgreSQL）中，千万行的窄表查询依然是毫秒级的。这比在百万行宽表（带几十个字段的业务表）里做 LIKE 快得多。

痛点 B：移动节点时的锁冲突
疑虑：移动一个大分支会产生大量 DELETE/INSERT，导致锁表。

架构对策：

异步化：组织变动在业务中是“低频”的。用户点击“保存”后，先更新 org_tree.parent_id（同步），然后通过 MQ 异步触发闭包表的更新（秒级延迟）。

软删除/版本号：如果实时性要求极高，可以给闭包表加版本号，更新完新路径后切换查询版本。

痛点 C：代码实现复杂
疑虑：逻辑嵌套太多，开发容易写错。

架构对策：利用 存储过程 或 成熟的 ORM 插件。作为一个架构师，你应该把这套逻辑封装在 OrgService 的底层，对业务开发同学透明。他们只需要调用 moveNode(nodeId, newParentId)。

4. 架构师的“偷懒”技巧：混合模式
既然你之前提到了 org_path，我给出的最终建议是：双重保障，各司其职。

org_path 作为辅助：在代码里做简单的权限判定，或者在前端展示 Breadcrumb（面包屑）时直接读取，无需查表。

org_closure 作为索引：只在需要**关联海量业务表（订单、设备）**时使用。

更新时的“防错”机制：
如果担心闭包表更新出错，可以写一个定期校验任务（Consistency Checker）。

每晚凌晨 3 点，扫描 org_tree 的 parent_id 关系，与 org_closure 对比。

如果发现不一致，以 parent_id 为准，重新全量生成该分支的闭包记录。

总结： parent_id 负责逻辑的正确性，org_path 负责业务的直观性，org_closure 负责查询的极限性能。三者合一，才是能支撑起万级网点、亿级订单的专业架构

架构师的最终交付清单
1. 核心表：org_tree
职责：维护“父子”的血缘关系。

特性：它是系统的“物理根基”，所有数据修复都以它的 parent_id 为准。

2. 索引表：org_closure
职责：打破层级，将“树状结构”铺平为“网状结构”。

特性：专门为 JOIN 订单和设备表准备。你甚至可以把它放在 Redis 中，用来做超高速的权限鉴权。

3. 查询策略
千万级 Join：走 org_closure。

树形展示：走 parent_id 递归。

权限过滤：在 SQL 拦截器里自动追加 WHERE org_path LIKE ...。
