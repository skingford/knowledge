以下我为您梳理的专项系统设计方案及 5 个高频面试难题：
## 一、 专项系统设计：千万级 DAU 运动数据中台

针对运动 APP（如跑步、骑行），核心挑战在于**瞬时高并发轨迹上传**与**实时排行榜**。

### 1. 接入层方案 (Ingestion Layer)

- **长连接 vs 短连接：** 运动过程中建议使用 **WebSocket 或 MQTT** 保持轻量化心跳，但在结束运动上传大批量轨迹点（GPS List）时，采用 **分片上传 + gRPC**。
    
- **流量削峰：** 在 LB（负载均衡）后接入 **Kafka**。千万级 DAU 意味着早晚高峰期会有数百万同时在线，直接写库会崩溃。Kafka 起到缓冲作用，后端 Consumer 异步消费写入分布式数据库。
    

### 2. 存储层设计 (Storage Layer)

- **热数据 (Redis)：** 存储用户当前运动状态、实时步数、实时心跳。
    
- **时序数据 (InfluxDB / ClickHouse)：** 运动轨迹是典型的时序数据。使用 ClickHouse 进行多维度的运动数据分析（如：分析过去一年的配速变化）。
    
- **持久化 (MongoDB / Cassandra)：** 运动详情（长文本、GPS 坐标数组）适合文档型数据库，水平扩展能力强。
    

### 3. 全球化部署 (Global Footprint)

- **Geo-Routing：** 用户请求就近接入。
    
- **数据归口：** 欧洲用户数据严格存储在法兰克福节点（合规要求），汇总后的匿名化统计数据再同步至总部。

## 二、 5 个高频架构师面试题（带 Golang 深度解析）

### Q1：运动 APP 的实时排行榜（如：今日步数全国排名）如何实现？

- **考点：** Redis ZSet 的局限性与分布式分片。
    
- **架构师回答：** * 千万级数据量下，单个 ZSet 会产生大 Key 风险且性能下降。
    
    - **方案：** 采用**多级桶架构**。按地域或用户 ID Hash 分成 1000 个桶（ZSet），每个桶记录自己的 Top 100。
        
    - **Golang 实现：** 利用 `Goroutine` 并发查询 1000 个桶，通过 `Context` 控制超时，最后使用**归并排序**提取全局 Top 100。
        

### Q2：如何保证用户运动轨迹点上传的“不丢失”与“不重复”？

- **考点：** 幂等性设计与消息可靠性。
    
- **架构师回答：** * **不丢失：** 客户端本地存储 + 序列号（Sequence ID）。上传失败则重试。服务端 Kafka 开启 `acks=all`。
    
    - **不重复：** 每个运动 Segment 分配全局唯一 ID。Golang 后端在写入数据库前，使用 **Redis SETNX** 或数据库唯一索引做幂等校验。
        

### Q3：Golang 服务在线上突然 CPU 飙升到 300%，作为架构师你怎么排查？

- **考点：** Pprof 实战与 Runtime 底层理解。
    
- **架构师回答：**
    
    - 立即通过 `go tool pprof` 获取 **CPU Profile**。
        
    - **常见原因：** 1. 某个 Goroutine 进入死循环； 2. 频繁的序列化/反序列化（如 JSON 解析大对象）； 3. **GC 压力过大：** 产生大量短生命周期对象（通过 `heap` 分析），考虑使用 `sync.Pool` 复用对象。
        
    - **决策：** 如果是热点函数，考虑用 `Atomic` 替代 `Mutex`，或优化算法复杂度。
        

### Q4：如果 AWS 某个区域（如新加坡）宕机了，你的架构如何实现分钟级灾备切换？

- **考点：** 异地多活与 DNS 切换。
    
- **架构师回答：**
    
    - **控制面：** 使用 Route53 的健康检查，一旦节点失效，自动将 CNAME 指向备用 Region（如东京）。
        
    - **数据面：** 采用数据库跨区域复制（Cross-Region Replication）。Go 服务需具备“只读模式”切换开关，在数据同步完成前，限制用户只能查看不能写入。
        

### Q5：如何设计一个支持“万人同跑”的实时位置共享系统？

- **考点：** 地理空间索引 (Geo-sharding) 与长连接。
    
- **架构师回答：**
    
    - 不能全量广播，必须**按需感知**。
        
    - 使用 **Google S2 几何算法** 或 **GeoHash** 将地图切成小格。
        
    - 用户只订阅自己所在网格及其周围 8 个网格的消息。
        
    - **Golang 选型：** 使用 `Nats` 或 `Redis Pub/Sub` 作为消息总线，结合 `Goroutine` 维护每个用户的 WebSocket 订阅流。
