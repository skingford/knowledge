## 为什么要专门做失败复盘

Agent 项目特别容易出现一种错觉：

> “每次失败的表现都不一样，所以只能个案处理。”

实际上，大多数失败是可以归类的，而且经常重复出现。只要分类做得好，你就能逐步把问题从“随机坏”变成“可治理”。

## 先把失败分成 6 类

| 类型 | 常见表象 | 更像该补哪层 |
| --- | --- | --- |
| **Prompt 失败** | 风格漂、格式错、指令没遵守 | Prompt / Structured Output |
| **Context 失败** | 模型没看到关键事实、看到太多噪音 | Context / RAG / Memory |
| **Tool 失败** | 工具选错、参数错、调用超时 | Tool schema / 执行器 / 重试 |
| **Workflow 失败** | 多步链路重复、早停、状态丢失 | Workflow / Harness |
| **权限失败** | 越权调用、高风险动作失控 | 权限、审批、Guardrails |
| **Eval 失败** | 改版后退化却没提前发现 | Eval / 回归 / 观测 |

## 案例 1：Prompt 看起来没问题，但输出还是飘

### 表象

- 有时候 JSON 合法，有时候不合法
- 同样的问题，多跑几次格式不一致
- 明明写了“不要输出解释”，模型还是加一堆废话

### 常见根因

- 约束不够明确
- few-shot 样例和目标格式不一致
- schema 约束没真正接进系统

### 更稳的改法

- 把自然语言要求改成更硬的结构约束
- 把样例、字段说明和禁止项写清楚
- 能用 schema 校验的，别只靠口头要求

## 案例 2：RAG 有结果，但答案还是错

### 表象

- 检索日志里明明有命中
- 最终答案却没有引用关键片段
- 或者引用了片段，但解释方向还是错

### 常见根因

- 召回到了“相关但不关键”的块
- 上下文拼装顺序不好
- 检索结果和模型任务指令混在一起，主次不清

### 更稳的改法

- 单独评测召回、重排和最终生成
- 给检索块加来源、标题和优先级
- 减少噪音块数量，别把“有命中”误当“可作答”

## 案例 3：工具选对了，但参数传错了

### 表象

- 工具调用方向是对的，但字段缺失、类型错误、枚举值不合法
- 同一个工具不同版本 schema 兼容性差

### 常见根因

- schema 设计太松或描述太模糊
- 工具名称和职责边界不清
- 没有在执行前做参数校验

### 更稳的改法

- 把字段语义写得更硬
- 在工具执行前做强校验和默认值补齐
- 收集失败参数样本，加入回归集

## 案例 4：多步任务中途跑偏

### 表象

- 重复执行同一步
- 本该交给人工时还在继续
- 任务状态前后不一致

### 常见根因

- 没有明确状态机或 step budget
- 检查点不清
- handoff 或路由条件模糊

### 更稳的改法

- 明确每一步输入、输出、终止条件
- 给任务加 step 限制和状态检查点
- 用 workflow 收紧大框架，而不是全靠模型自由决定

## 案例 5：权限边界没收住

### 表象

- 模型触发了不该触发的高风险写操作
- 一个只读任务却拿到了写权限工具
- 用户稍微诱导就越过了系统本意

### 常见根因

- 工具默认开放
- 没有审批节点
- 认为“模型应该懂边界”

### 更稳的改法

- 权限按工具、角色、场景最小化
- 高风险动作前置审批或确认
- 把权限边界放进系统层，不只写在 Prompt 里

## 案例 6：上线后越改越乱

### 表象

- 改一个地方，另外两个指标掉了
- 团队不断 patch prompt，但没有统一版本和回归
- 谁也说不清“这次变好”还是“只是换了个失败方式”

### 常见根因

- 没有回归体系
- 没有 trace 和版本号
- 失败案例没有沉淀进评测样本

### 更稳的改法

- 每次改动都跑小集回归
- 给 prompt / context / tool / workflow 做版本号
- 线上失败样本持续回灌到评测集

## 复盘模板

每次失败案例至少回答这 5 个问题：

1. 现象是什么？
2. 失败发生在哪一层？
3. 当时模型看到了什么、做了什么、系统走到了哪一步？
4. 真正应该补的是 Prompt、Context、Tool、Workflow、权限还是 Eval？
5. 这个案例如何变成未来的防回归样本？
