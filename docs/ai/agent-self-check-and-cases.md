# Agent 能力自检题与案例

## 适合人群

- 想判断自己学到现在到底停在“知道概念”还是“能做系统”的工程师
- 准备面试、复盘、带新人或做阶段性自测的人
- 想把 `/ai/` 这一专题真正转成“会判断、会取舍、会解释”的读者

## 学习目标

- 用问题驱动方式检查自己是否掌握关键边界
- 把零散概念收束成真实工程判断题
- 通过案例题训练“先归因，再给解法”的能力

## 怎么使用这份自检题

这份题不适合背答案，更适合：

- 自己先答
- 再对照文档查缺补漏
- 或在团队里拿来做 onboarding / sharing

如果你能把“为什么这么判断”讲清楚，往往比把术语解释清楚更重要。

## 基础判断题

1. Tool Calling 和 Handoff 的本质区别是什么？
2. 为什么很多“记忆问题”其实是状态设计问题？
3. 为什么 RAG 命中了文档，答案仍然可能错？
4. 什么情况下应该优先改 Prompt，而不是去做 Fine-tuning？
5. 为什么 Agent 不能只靠普通接口日志排障？

## 系统设计题

1. 做一个“内部文档问答 + 实时工单状态查询”的 Agent，你会怎么分模型、知识库和工具的职责？
2. 做一个会写日报、发消息、查数据库的办公 Agent，哪些步骤应该放进 workflow，哪些可以交给模型自由判断？
3. 如果用户要求“以后都用中文回复我”，这条信息应该放进会话上下文、长期记忆，还是别的地方？
4. 如果团队已经有 PostgreSQL，你会在什么情况下仍然建议别直接用 pgvector，而考虑专门向量数据库？

## 排障归因题

1. 模型总能选对工具，但参数经常缺字段。你会优先检查哪一层？
2. 多步任务经常重复执行相同步骤，但单步回答本身没问题。你会优先看哪一层？
3. 同一批测试样本里最终回答分数波动很大，但 trace 里工具调用基本正常。你会先怀疑 Prompt、Context、Tool 还是 Workflow？
4. 线上案例显示知识库确实召回了正确片段，但用户仍说“它老是答偏”。你下一步最该看什么？

## 产品化题

1. 什么信号说明一个 Agent demo 已经值得进入灰度？
2. 如果失败一次的代价很高，你会加哪些系统层措施，而不是只改 prompt？
3. 为什么“任务完成率提高了”还不能单独说明产品已经稳定？
4. 如果用户反馈 Agent 经常“说得像对的，但细节不靠谱”，你会如何把这个反馈变成评测样本？

## 一个简单评分标准

你可以用下面这个非常粗粒度的标准自查：

- **入门**：能解释术语，能跑最小 demo
- **进阶**：能定位问题大概属于 Prompt / Context / Tool / Workflow 哪层
- **工程化**：能提出更稳的改法，并知道如何回归和上线
- **可带人**：能把边界、取舍和失败模式讲成可复用的方法

如果你想更进一步，可以把每道题都扩成：

- 现象
- 归因
- 方案
- 风险
- 回归验证

这基本就已经接近真实工作里的设计和复盘方式了。
