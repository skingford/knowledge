# Agent 缓存、延迟与吞吐优化

## 适合人群

- 已经把 Agent 跑起来，但觉得又贵又慢、峰值扛不住的人
- 需要在体验、质量和成本之间做系统优化的工程师
- 想知道除了“换便宜模型”之外，还有哪些更有效优化手段的人

## 学习目标

- 理解 Agent 的性能问题通常不只在模型本身
- 学会从缓存、上下文、并发、流式、降级几层做优化
- 建立“先找高频高价值瓶颈，再优化”的工程思路

## 先别只盯模型单价

很多系统在成本和延迟升高时，第一反应是：

> “换个更便宜模型吧。”

这有时有效，但往往不是最先该做的。

Agent 的总成本和总延迟，通常由下面几项一起决定：

- 模型调用次数
- 每次调用的上下文长度
- 检索和工具调用次数
- 是否重复做相同工作
- 是否把低价值步骤也交给强模型

## Agent 性能瓶颈通常在哪

常见瓶颈一般集中在：

- 多轮链路太长
- 上下文太大
- 检索结果太多
- 工具调用过频
- 没有缓存或中间结果复用

一个经验是：

> 很多“模型太慢”的问题，本质上是系统让模型做了太多不必要的工作。

## 缓存能解决什么问题

缓存最适合解决高频、重复、模式稳定的问题。

常见缓存类型包括：

- **响应缓存**：同类请求直接复用结果
- **语义缓存**：近似问题命中近似答案
- **工具结果缓存**：外部查询结果短时间复用
- **中间结果缓存**：摘要、路由判断、结构化抽取结果复用

缓存的关键不是“能不能加”，而是：

- 命中率是否够高
- 过期策略是否合理
- 是否会缓存高风险或高漂移结果

## 上下文压缩为什么常常更有效

很多系统的真正大头不是模型单价，而是 token 数量。

所以非常高收益的优化往往是：

- 长对话做摘要
- RAG 结果减少噪音块
- 工具返回值先结构化再压缩
- 不把重复背景信息反复塞进上下文

这类优化经常同时改善：

- 成本
- 延迟
- 稳定性

## 并发和吞吐怎么设计

如果系统进入高频使用阶段，吞吐问题就会变得重要。

更稳的设计通常包括：

- 对外部工具限并发
- 对高成本模型限流
- 对长任务异步化
- 给不同任务类型分配不同资源池

别让一个重任务把所有轻任务都堵住。

## 流式输出和用户体感

有时候你没法把总耗时大幅压下去，但可以改善用户体感。

这时常见办法是：

- 流式输出
- 分阶段反馈
- 先回显“正在检索 / 正在分析”
- 长任务先返回任务状态，再异步补全结果

体感优化不是“假装变快”，而是把等待过程设计得更可接受。

## 高频路径优化顺序

如果你想要一个更稳的默认顺序，可以按这个来：

1. 先量化最贵、最慢、最频繁的路径
2. 先砍不必要调用次数
3. 再压上下文长度
4. 再做缓存和中间结果复用
5. 最后再考虑更复杂的架构优化

这个顺序往往比一开始就上很多花哨技巧更有效。

## 常见失败点

- 只盯模型单价，不看调用次数和 token 长度
- 对命中率很低的路径也硬加缓存
- 缓存了高漂移结果，反而制造错误答案
- 并发没分级，让重任务拖垮全局
- 一味追求低延迟，牺牲了关键高风险路径的质量
