# Context Engineering 实践指南

## 适合人群

- 已经会写 Prompt，但系统效果仍然不稳定、不知道问题出在哪的人

- 想搞清“模型答错到底是 Prompt 问题，还是上下文问题”的工程师

- 能区分 Prompt 问题与上下文问题
- 掌握上下文选择、裁剪、排序、摘要和动态组装的基本方法

## Context Engineering 是什么

如果说 Prompt Engineering 关注的是“怎么写指令”，那么 Context Engineering 关注的是：**给模型看什么信息**。

它处理的是推理时送进模型的整包输入，而不只是用户那一句话。

## 它为什么在 Agent 阶段变重要

在单轮问答里，很多问题看起来像 Prompt 问题。但一旦系统引入：

- RAG 检索
- 多轮对话历史
- 工具返回值
- 长期记忆
- 长上下文窗口

瓶颈就会变成“信息有没有被正确拿到、筛掉、排序好、拼进去”。这正是 Context Engineering 的职责。

## 演进过程

- **最早期**：上下文主要就是用户输入和少量对话历史，关注点还是 Prompt 本身
- **RAG 兴起后**：上下文扩展为检索片段、来源信息、FAQ、知识库片段，重点转向“召回什么”
- **Tool Calling 普及后**：工具返回值、数据库查询结果、浏览器抓取内容、代码执行结果都开始进入上下文，重点转向“怎么拼”
- **长上下文模型出现后**：问题从“放不下”变成“放什么、按什么顺序放、哪些必须摘要”，上下文编排反而更重要
- **当前阶段**：上下文被视为动态产物，由检索器、记忆系统、摘要器、路由器、工具链共同组装，而不是手工拼几段文本

## 主要作用

- 给模型提供当前任务所需的事实依据
- 让模型理解用户历史、会话状态和外部环境
- 降低幻觉，减少模型靠参数记忆胡乱补全
- 在有限 token 预算内，把最重要的信息摆到最容易被模型利用的位置

## 核心定义

Context Engineering 是设计和管理推理时提供给 LLM 的全部信息的学科，包括：

- System Prompt
- 对话历史
- 检索到的文档上下文
- 工具返回的数据
- 记忆摘要和状态信息
- 它们的排列顺序、标注方式和 token 预算分配

## 关键原则

| 原则 | 说明 |
|------|------|
| **上下文相关性** | 只传入与当前任务相关的信息，减少噪音 |
| **上下文排序** | 重要信息放在开头和结尾，帮助模型优先关注 |
| **Token 预算管理** | 给系统指令、历史、检索、工具结果分配明确预算 |
| **动态组装** | 根据任务意图决定拼入哪些上下文，而不是一股脑全塞进去 |
| **可追溯性** | 尽量保留来源、时间、文档类型等元信息，方便判断依据 |

## 与 Prompt Engineering 的区别

| 维度 | Prompt Engineering | Context Engineering |
|------|-------------------|-------------------|
| 关注点 | 指令的措辞和格式 | 信息的选择和编排 |
| 核心问题 | “怎么说”让模型理解 | “给什么”让模型有足够信息 |
| 典型操作 | Few-shot、CoT、角色设定 | 检索策略、上下文裁剪、记忆摘要 |
| 优化目标 | 行为稳定性和输出一致性 | 信息相关性和利用效率 |

## 实践技巧

- **上下文裁剪**：RAG 检索结果不要贪多，3-5 个高质量片段通常优于 10 个低质量片段
- **摘要注入**：长对话用摘要替代原始消息，保留关键状态同时节省 token
- **分层上下文**：把“必须遵守的规则”和“参考信息”分开标注
- **上下文模板化**：统一拼接模板，避免每次手工组装导致结构漂移
- **结果去重**：避免重复段落反复占用上下文窗口
- **保留来源**：给检索和工具结果标注来源，便于 grounded answer 和排障

> 深入阅读：Anthropic 的 [Context Engineering 文档](https://docs.anthropic.com/en/docs/build-with-claude/context-engineering)

## 常见失败点

- 把所有历史记录和检索结果一次性全塞给模型
- 检索内容看起来相关，但没有经过重排、过滤和来源标注
- 工具返回了很多原始数据，却没有做摘要和结构化整理
- 重要约束埋在中间，真正关键的信息被噪音淹没
- 没有为不同任务类型设计不同的上下文模板

## 和 Harness 的关系

Context Engineering 解决的是“信息怎么进模型”，Harness Engineering 解决的是“系统怎么把整个任务跑稳”。

两者通常一起工作：

- Harness 决定什么时候检索、什么时候摘要、什么时候调工具
- Context 决定这些结果最终以什么形式进入模型
