# Agent 数据集构建、标注与合成数据

## 适合人群

- 已经开始做 Eval，但评测样本主要靠临时手搓的人
- 线上已经有失败案例，却没有沉淀成长期资产的人
- 想搞清楚合成数据能做什么、不能做什么的工程师

## 学习目标

- 理解为什么 Agent 系统最终会卡在“数据层”而不只是 Prompt 层
- 学会搭一个最小可行的数据闭环：采样、标注、合成、版本化、回流
- 知道什么时候该用人工标注，什么时候可以用模型扩样

## 为什么数据层会成为瓶颈

很多团队前期主要靠 Prompt 和模型切换推动效果提升，但到了第二阶段，真正限制迭代速度的往往变成数据。

典型信号包括：

- 每次回归只会拿 10 条熟悉样本看一眼
- 线上已经反复踩过的坑，下次发布还会再出现
- 团队成员对“什么算通过”没有统一标准
- 合成了一堆样本，但和真实用户问题脱节

Agent 系统会持续演进 Prompt、模型、工具、知识库和工作流。如果没有稳定的数据集，这些改动就很难做对比。

## 样本应该从哪里来

不要把数据来源只理解成“人工写几条问答”。更稳的做法是混合取样。

| 来源 | 价值 | 注意点 |
| --- | --- | --- |
| **线上失败样本** | 最接近真实风险，优先级最高 | 要脱敏、去重、补充上下文 |
| **人工设计样本** | 能覆盖明确边界和对抗场景 | 容易理想化，不能替代真实流量 |
| **历史工单 / FAQ / SOP** | 贴近业务，适合做主路径 | 常需要清洗格式和补齐标签 |
| **合成扩样** | 补覆盖率、做扰动和风格扩展 | 不能把模型生成结果直接当真值 |

一个实用原则是：

1. 主路径样本尽量来自真实历史任务
2. 高风险边界样本由人工刻意设计
3. 合成数据主要用来补覆盖、补扰动、补长尾

## 先把样本桶设计清楚

比起先追求样本总量，更重要的是先把“样本桶”拆对。

| 样本桶 | 目的 |
| --- | --- |
| **主路径样本** | 验证系统在最常见任务上是否稳定 |
| **边界样本** | 验证长输入、缺字段、模糊意图、低质量上下文 |
| **对抗样本** | 验证注入、越权诱导、脏工具返回、错误知识片段 |
| **恢复样本** | 验证超时、空结果、重试、降级、转人工 |
| **回归样本** | 记录团队踩过的历史坑，避免再次出现 |

如果系统有多个业务子场景，还应该在每个样本上加业务标签，比如 `客服`、`报表生成`、`工单分流`、`代码执行`。

## 标注规范怎么写

很多团队不是缺标注，而是缺“统一标注口径”。

最小可行标注字段通常包括：

- `输入内容`：用户原始请求或轨迹片段
- `任务标签`：属于什么场景、什么能力
- `预期行为`：系统应该做什么，而不一定是唯一文字答案
- `必须 / 禁止工具`：哪些工具应该调用，哪些不应调用
- `参考依据`：知识来源、规则来源、标准答案片段
- `通过规则`：如何判断通过，规则判还是 LLM 判
- `风险等级`：低、中、高，决定是否需要人工复核

对于开放式任务，不要强行只留一条“标准答案”，更实用的是写成“可接受行为范围”。

例如：

- 是否先检索后回答
- 是否引用了正确来源
- 是否遗漏关键风险项
- 是否在高风险动作前转入审批

## 合成数据什么时候值得做

合成数据最适合解决“覆盖率不足”，不适合解决“真值不清楚”。

适合的场景：

- 从少量真实样本扩出多种表达方式
- 生成拼写错误、口语化、残缺字段等扰动版本
- 为分类、抽取、结构化输出补齐样本分布
- 为工具参数生成补边界用例

不适合直接依赖的场景：

- 高风险业务结论
- 复杂开放式长文质量评估
- 事实性要求很高但真值难确认的任务

更稳的实践是四步：

1. 用真实样本做种子
2. 用模型扩写或改写
3. 用规则和小集人工抽检过滤
4. 只把通过审核的样本纳入正式数据集

## 数据集如何版本化

数据集要像代码和 Prompt 一样管理版本，而不是散落在表格和聊天记录里。

建议至少记录下面几类信息：

- 数据集版本号
- 样本来源和生成方式
- 标注规范版本
- 脱敏规则版本
- 关联的 Prompt / 模型 / 工具 schema / workflow 版本

如果一个样本来自线上事故，最好还能追到：

- 首次出现时间
- 事故编号或复盘链接
- 修复后在哪个版本被加入回归集

这会让数据集不只是“样本仓库”，而是“系统学习历史”。

## 线上失败如何回流到离线评测

推荐闭环：

1. 在线上记录失败任务、人工接管案例和高风险误判
2. 每周筛出高价值样本，做脱敏和去重
3. 用统一标注规范补全预期行为和通过规则
4. 放入回归集，在下一次发布前自动验证

这一步做扎实后，数据集会逐渐从“人工拼出来的样本集合”，变成“系统真实问题的沉淀池”。

## 常见失败点

- 只收集最终回答，不保留轨迹、工具结果和上下文
- 用模型自动标注后不做抽检，导致错误标签污染数据集
- 合成数据比例太高，导致数据分布脱离真实用户
- 每次重写数据集，却没有版本和来源信息
- 线上失败案例只在群里讨论，没有进入回归集
