# Agent Eval 设计与回归实践

## 适合人群

- 已经能跑出 Demo，但每次改 Prompt、模型或工具都不知道到底变好还是变坏的人
- 正在把 Agent 接进生产链路，需要把“效果判断”从感觉变成数据的人
- 需要区分模型问题、上下文问题、工具问题和工作流问题的工程师

## 学习目标

- 建立 Agent Eval 的分层视角，而不是只看“最后答案对不对”
- 学会设计最小可行评测集、关键指标和回归流程
- 理解 LLM-as-Judge、轨迹评测、线上抽样复盘各自的边界

## 为什么 Agent 必须有 Eval

Prompt、模型、检索、工具、路由、重试策略只要改一个点，整条链路的行为都可能变化。

如果没有 Eval，团队最后只能靠下面几种不稳的方法做判断：

- “这个 Demo 看起来更聪明了”
- “线上投诉暂时没变多”
- “这个回答我个人更喜欢”

这些都不够。Agent 是系统，不是一次性展示品。只要系统会演进，就必须有回归能力。

## 先分清 4 层评测对象

很多团队一上来就评“最终答案”，结果定位问题非常慢。更稳的做法是把评测对象拆层。

| 层级 | 评什么 | 典型问题 |
| --- | --- | --- |
| **单步输出层** | 单轮回答、结构化输出、字段完整性 | JSON 不合法、格式漂移 |
| **工具 / 检索层** | 是否选对工具、参数是否正确、检索是否命中 | 不该调工具时乱调、该检索时没检索 |
| **轨迹层** | 多步链路是否按预期推进 | 步骤重复、早停、死循环、handoff 错位 |
| **业务结果层** | 是否真正完成业务目标 | 报告质量、任务完成率、人工接管率 |

一个典型经验是：

- `单步层` 更适合自动化精确判断
- `工具层` 更适合规则校验和日志比对
- `轨迹层` 更适合结合 Trace 做结构化检查
- `业务层` 往往还需要人工抽样或业务指标联动

## 最小可行评测集怎么建

刚开始不需要 1000 条样本。更重要的是样本结构要对。

建议先准备 20-50 条样本，并按下面几类分桶：

| 样本桶 | 目的 |
| --- | --- |
| **Happy Path** | 验证典型流程是否稳定通过 |
| **边界样本** | 验证长输入、缺字段、模糊意图、低质量检索等情况 |
| **对抗样本** | 验证注入、越权、误导输入、脏工具返回值 |
| **恢复样本** | 验证工具失败、超时、空结果、重试和降级 |

写样本时，最好同时保留：

- 用户输入
- 预期行为
- 是否必须调工具
- 可接受输出范围
- 人工备注

这样后面才能做更细的归因。

## 核心指标怎么定

别只放一个“准确率”。Agent 需要一组互补指标。

| 指标 | 含义 | 常见适用场景 |
| --- | --- | --- |
| **任务完成率** | 是否最终完成目标 | 任务型 Agent、表单填报、办公自动化 |
| **结构合法率** | 输出是否满足 schema | Structured Output、API 入参生成 |
| **工具选择正确率** | 是否选了该选的工具 | 搜索、数据库、浏览器、多工具系统 |
| **工具成功率** | 工具调用是否成功结束 | 外部 API、脚本执行、数据查询 |
| **轨迹通过率** | 多步流程是否按预期推进 | Workflow、状态机、handoff |
| **引用 / grounding 率** | 回答是否基于证据 | RAG、问答、知识库助手 |
| **延迟与成本** | 每任务耗时、token、调用次数 | 生产环境容量与预算 |
| **人工接管率** | 需要人工介入的比例 | 高风险或复杂任务系统 |

实操上，一般推荐：

1. 先定义 1 个北极星指标，例如 `任务完成率`
2. 再配 3-5 个护栏指标，例如 `工具成功率`、`延迟`、`成本`、`人工接管率`
3. 每次回归同时看北极星和护栏，而不是只盯最终回答质量

## LLM-as-Judge 该怎么用

LLM-as-Judge 很有用，但不是万能裁判。

适合的场景：

- 开放式文本质量判断
- 多版本输出相对比较
- 是否覆盖关键要点、是否遗漏风险项

不适合单独依赖的场景：

- 数值精确正确性
- 工具调用参数是否合法
- 安全边界、权限边界等强规则判断

更稳的实践是“三段式”：

1. **能规则校验的，优先规则校验**
2. **规则不好判的，用 LLM-as-Judge**
3. **高价值样本，保留人工抽样复核**

## 工具和检索怎么拆开评

Agent 失败常被笼统归因成“模型不行”，但很多时候真正的问题在工具和检索。

### 工具评测要看什么

- 是否调用了正确工具
- 参数是否齐全、格式是否对
- 是否在错误场景下滥用工具
- 失败后是否重试或降级

### 检索评测要看什么

- 是否召回了应该召回的片段
- 是否召回了太多无关噪音
- 重排后真正送入上下文的片段是否可用
- 最终答案是否引用了正确证据

一个常见做法是把 RAG 拆成三段分别评：

1. `召回`
2. `重排 / 裁剪`
3. `基于上下文作答`

这样才能区分“检不到”和“检到了但没用好”。

## 回归测试链路

把 Eval 接进工程后，最小闭环通常是下面这样：

1. 固化一批带标签的评测样本
2. 给 Prompt、模型、工具 schema、检索配置做版本号
3. 每次改动后自动跑评测集
4. 输出各指标的前后对比
5. 保留失败样本和 Trace，进入复盘池

如果系统复杂一些，可以把回归再分成两层：

- **PR 前小集回归**：几十条高价值样本，追求快
- **主干大集回归**：几百条样本，追求稳

## 线上观测和线下评测怎么连起来

离线样本再全，也不可能覆盖所有真实输入。线上观测的价值在于不断把新失败样本喂回评测集。

推荐闭环：

1. 在线上记录失败轨迹、异常输入、人工接管案例
2. 每周挑选高价值失败样本
3. 补成新的离线评测集
4. 下次改版时自动验证这些历史坑有没有再出现

这一步做起来之后，Eval 才不只是“验收工具”，而会变成“系统学习能力”。

## 常见失败点

- 用 5 条顺风样本就宣布效果提升
- 只评最终答案，不评工具选择和轨迹
- 用一个平均分掩盖高风险失败样本
- 线上出了问题，却没有沉淀回离线评测集
- 把 LLM-as-Judge 当成绝对标准，而不是辅助判断器
