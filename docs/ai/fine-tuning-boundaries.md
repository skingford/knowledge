# Fine-tuning 边界与适用场景

## 适合人群

- 已经开始做 Agent / LLM 应用，但经常听到“调个模型就好了”的团队
- 不确定当前问题该先改 Prompt、RAG、工具还是 Fine-tuning 的工程师
- 想理解微调的收益、代价和边界的人

## 学习目标

- 分清 Fine-tuning 能解决什么、不能解决什么
- 建立从 Prompt、Context、RAG、Workflow 到 Fine-tuning 的优先顺序
- 学会识别值得做微调的典型信号和不值得做的误区

## 先别把 Fine-tuning 当默认答案

很多团队在模型输出不稳定时，第一反应是：

> “是不是该微调一下？”

但在大多数 Agent 项目里，真正的问题更常见于：

- Prompt 约束不清
- 上下文给错了
- RAG 召回不准
- 工具描述或 schema 设计得不好
- 工作流没有把步骤和边界框住

如果这些基础层没做好，Fine-tuning 往往只是在放大系统噪音。

## Fine-tuning 更擅长解决什么

Fine-tuning 更适合处理下面这类问题：

- **固定风格 / 语气 / 输出模式** 需要长期稳定
- **领域表达习惯** 很强，例如法务、医疗、金融、客服
- **结构化抽取或分类** 有大量稳定标注样本
- **小模型能力补齐**，希望在成本更低的模型上逼近特定任务表现

一句话概括：

> Fine-tuning 更擅长把“已经定义清楚的输出习惯”学稳，而不是替你解决系统架构问题。

## Fine-tuning 不擅长解决什么

下面这些问题，通常不该优先指望 Fine-tuning：

- 缺实时知识
- 缺组织内部知识
- 工具调用不稳定，因为 schema 或执行器有问题
- 多步任务容易跑偏，因为 workflow 没收住
- 高风险写操作缺审批和权限控制

这些问题更可能要靠：

- **RAG**
- **Tool Calling**
- **Context Engineering**
- **Harness / Workflow**

## 遇到问题先按什么顺序排

比较稳的一条排查顺序通常是：

1. **先看 Prompt**
2. **再看 Context / RAG**
3. **再看 Tool / Workflow**
4. **最后才看 Fine-tuning**

这是因为：

- Prompt 决定规则
- Context 决定事实输入
- Tool / Workflow 决定系统行为
- Fine-tuning 只是在这些都相对明确之后，进一步把模型行为收紧

## 什么时候值得认真考虑微调

下面这几类信号比较强：

- 同类任务量大，而且输入输出模式稳定
- 已经有高质量标注样本，且样本规模足够
- Prompt 已经比较清楚，但还是存在风格漂移或结构不稳
- 想把任务从昂贵大模型迁到便宜模型

比如：

- 固定行业的工单分类
- 结构化字段抽取
- 稳定格式的回复生成
- 统一品牌语气的客服输出

## 训练数据该长什么样

决定微调值不值得做的，不只是“有没有数据”，而是数据质量是否真的支撑任务。

训练数据最好满足：

- 样本定义清楚
- 输入输出模式稳定
- 标签一致性高
- 覆盖主要分布和边界样本

如果数据本身噪音很大、标注口径不统一，那微调出来的往往不是能力，而是混乱。

## 成本和风险主要在哪

很多人只想到训练成本，却忽略了后面的维护成本。

常见成本包括：

- 数据准备和清洗
- 训练与评测成本
- 模型版本管理
- 回归测试与监控
- 数据分布变化后的再训练

风险则主要包括：

- 过拟合某类表达
- 风格更稳了，但事实性变差
- 迁移后影响工具调用、结构输出或多步决策
- 团队把“该做系统设计”的问题错误甩给了模型

## 常见失败点

- RAG 没做好就急着微调
- 工具调用不稳，却试图靠微调修工具问题
- 数据量不够、标签不一致就开始训练
- 只看离线样本，不看线上真实任务分布
- 把 Fine-tuning 当成“万能提效按钮”
