# Guardrails、审批与沙箱边界

## 适合人群

- 正在把 Agent 接进真实业务系统，需要控制越权、误操作和高风险动作的人
- 已经做了 Tool Calling，但还没把审批、最小权限和沙箱隔离系统化的人
- 想把“安全边界”从 Prompt 约束升级成系统设计的人

## 学习目标

- 理解 Guardrails 不是一条提示词，而是一套系统边界
- 分清拒答、审批、只读隔离、沙箱执行分别解决什么问题
- 学会把高风险动作前置到系统层控制，而不是把期望寄托给模型自觉

## 为什么 Guardrails 不能只写在 Prompt 里

很多系统一开始会这么做：

> “在 system prompt 里写：不要执行危险操作。”

这当然有帮助，但远远不够。因为 Prompt 只是在“提醒模型”，而不是在“限制系统”。

真正的 Guardrails，应该至少覆盖：

- **模型层**：它应该怎么判断和表达
- **工具层**：它到底能调用什么
- **执行层**：调用前后有哪些检查
- **权限层**：谁能做、做多大、何时审批

## 先分清 4 类控制手段

| 手段 | 主要解决什么问题 |
| --- | --- |
| **拒答 / 阻断** | 某类请求根本不该继续 |
| **审批 / 确认** | 某类操作能做，但必须经过人或规则确认 |
| **权限隔离** | 不同工具、角色、场景拥有不同能力边界 |
| **沙箱执行** | 允许执行，但必须在隔离环境里跑 |

这 4 类手段不是替代关系，而是可以叠加。

## 什么时候该拒答

适合直接拒答的典型场景：

- 越权请求
- 明显违反政策或安全边界的请求
- 缺必要信息、继续执行风险很高的请求

这里关键不是“模型礼貌地说不行”，而是**系统根本不把这类请求放到危险路径里**。

## 什么时候必须审批

下面这些动作通常更适合加审批或显式确认：

- 写生产数据
- 删除、覆盖、发布、支付、下发消息
- 发起外部高影响操作
- 会造成不可逆结果的动作

一个很好用的判断标准是：

> 只要一次误触的代价明显高于一次多点确认的摩擦，就值得审批。

## 什么时候需要沙箱

如果 Agent 需要：

- 执行代码
- 操作文件系统
- 调用 shell
- 访问不完全可信的外部内容

那么沙箱通常不是可选项，而是基础设施。

沙箱的价值不在于“让模型更会干活”，而在于把潜在破坏范围尽量收窄。

## 权限边界怎么收

最小权限原则在 Agent 系统里尤其重要。

比较稳的做法通常是：

- 只读工具和可写工具分开
- 高风险工具单独分组
- 不同场景绑定不同工具集合
- 给工具调用增加额度、预算、频次限制

别让一个看起来“方便”的万能工具，变成整个系统最大的风险源。

## 常见失败点

- 把 Guardrails 理解成 prompt 话术，而不是系统边界
- 所有工具默认开放，再指望模型自己克制
- 高风险动作没有审批、确认或审计
- 沙箱缺失，直接把模型接到真实执行环境
- 把只读查询和可写操作混在同一条执行路径里
