# 本地知识库构建实践

## 适合人群

- 准备把 Markdown、PDF、笔记、内部文档接进 Agent 的工程师
- 已经做了 RAG demo，但知识库质量不稳定、召回效果忽高忽低的人
- 想把“文档堆进去就能问答”升级成更像工程系统的人

## 学习目标

- 建立本地知识库构建的最小工程流程
- 理解清洗、切块、元信息、索引更新和评测之间的关系
- 知道为什么很多 RAG 效果问题其实不是模型问题，而是知识库构建问题

## 先别急着把文件全喂进去

很多本地知识库项目的起点是：

> “把所有 PDF 和 Markdown 扔进去，然后试试看。”

这样通常能很快出 demo，但效果不稳定的原因也会很快暴露：

- 文档里夹杂大量噪音
- 切块过碎或过粗
- 没有元信息，后续无法过滤和溯源
- 新旧版本混在一起
- 检索有结果，但不是用户真正需要的那一段

## 知识库构建的最小流程

一个比较稳的本地知识库流程通常包括：

1. 文档收集
2. 清洗与标准化
3. 切块
4. 生成 embedding
5. 建索引
6. 检索与重排
7. 评测与迭代

关键点不在于“步骤多”，而在于每一步都要能单独检查。

## 数据清洗要做什么

常见清洗动作包括：

- 去掉页眉页脚、导航、版权、重复模板段
- 统一编码、换行、标题层级
- 尽量保留结构信息，例如标题、章节、表格来源
- 对 PDF / 扫描件要特别小心 OCR 质量

一个经验是：

- **脏数据进来，后面不管 embedding 多高级，结果也很难稳**

## 切块策略怎么选

切块没有万能大小，重点看文档结构和提问方式。

常见思路：

- **按自然结构切**：标题、小节、段落
- **按固定窗口切**：固定 token 或字符长度
- **混合切块**：先按结构，再限制上限

一般来说：

- 过大：容易把无关信息带进去
- 过小：上下文不完整，语义断裂

最稳的做法通常不是盲选一个长度，而是拿一批真实问题做召回测试。

## 元信息为什么很关键

元信息决定你后面能不能做更细的过滤、重排和解释。

建议至少保留：

- 文档名
- 标题 / 小节路径
- 来源路径
- 更新时间
- 文档类型

如果做组织知识库，元信息往往比纯文本本身还重要，因为它决定了：

- 能不能按部门、产品、版本过滤
- 能不能向用户解释“答案来自哪里”
- 能不能清掉旧版本内容

## 索引更新怎么设计

知识库不是一次性建完就不管。

至少要考虑：

- 新文档如何进入索引
- 已更新文档如何替换旧块
- 被删除文档如何从索引中清理
- 不同版本是否需要并存

如果没有更新策略，知识库很容易出现：

- 旧文档和新文档同时命中
- 重复片段越来越多
- 用户拿到过期答案

## 评测怎么接进来

本地知识库也需要评测，不然你很难知道问题出在：

- 文档清洗
- 切块
- 召回
- 重排
- 生成

建议至少准备一小批真实问答样本，然后检查：

- 有没有召回到正确文档
- 召回的块是不是关键块
- 最终回答有没有基于正确块生成

## 常见失败点

- 不做清洗，直接把原始文档全量导入
- 没有元信息，只存一堆文本块
- 切块策略拍脑袋，不基于问题样本验证
- 文档更新后没有做增量替换和删除
- 只看最终回答，不看召回是否真的命中正确块
