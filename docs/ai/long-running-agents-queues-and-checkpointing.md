- 已经不再只是做单轮问答，而是开始跑多步骤任务的人
- 遇到任务超时、重复执行、半路失败后无法恢复的工程师
- 需要把 Agent 接进工单、审批、代码执行、数据处理等长链路场景的人

## 为什么长任务不能继续走同步链路

很多 Agent 原型最初都挂在一个 HTTP 请求里完成：进来一个请求，模型规划、调工具、拿结果、返回答案。

但只要任务开始包含下面这些特征，同步链路就会迅速失稳：

- 需要调多个慢工具
- 包含外部审批或人工等待
- 需要长时间处理文档、代码或批量数据
- 需要多次重试、分段推进或跨小时恢复

这时候真正的问题已经不是“模型够不够聪明”，而是“任务执行模型是不是还合理”。

## 哪些任务应该异步化

可以先做一个很简单的判断：

| 任务形态 | 更适合的执行方式 |
| --- | --- |
| **几秒内完成、无副作用、失败可直接重试** | 同步请求 |
| **几十秒到几分钟、依赖多个外部系统** | 异步 worker |
| **需要审批、人工确认、等待事件回调** | 异步状态机 |
| **跨小时、跨天、要断点续跑** | 队列 + 检查点 + 可恢复任务 |

典型长任务包括：

- 批量处理文档并构建知识库
- 多步骤调研、写作、审校链路
- 浏览器自动化、代码执行、数据报表生成
- 需要等待审批、支付结果、外部 webhook 的任务

## 最小可行架构长什么样

一个足够实用的长任务架构通常包括下面几层：

1. **入口层**：接收任务创建请求，立即返回任务 ID
2. **调度层**：把任务放进队列，控制优先级、限流和重试
3. **执行层**：worker 拉取任务，按步骤推进 Agent
4. **状态层**：保存任务状态、检查点、已产出结果和失败原因
5. **通知层**：把进度变化推给用户、人工队列或下游系统

如果任务中还有高风险动作，通常还要再挂一个审批或人工接管通道。

## 幂等和检查点为什么重要

长任务失败并不可怕，可怕的是失败后不知道“做到哪一步了”。

因此至少要有两个设计：

- **幂等键**：避免同一任务被重复执行出副作用
- **检查点**：让任务可以从中间继续，而不是每次从头再跑

一个实用做法是按 `任务 -> 步骤 -> 子步骤` 三层记录状态，例如：

- 当前处于哪一步
- 哪些工具已经调用成功
- 哪些结果已经落库
- 下一次恢复应该从哪里继续

如果某一步涉及写操作，还要记录“写操作是否已经提交”，不然重试时很容易重复发消息、重复建单、重复扣费。

## 重试超时和补偿怎么设计

不是所有失败都适合直接重试。

| 失败类型 | 典型处理方式 |
| --- | --- |
| **瞬时失败** | 短时间重试，例如网络超时、临时 5xx |
| **确定性失败** | 不要盲目重试，先修参数或修上下文 |
| **高风险副作用失败** | 需要人工确认是否重试 |
| **部分成功失败** | 需要补偿动作或从检查点恢复 |

补偿设计常见于这些场景：

- 已经创建了草稿，但后续步骤失败，需要撤销或标记异常
- 已经写入外部系统，需要回滚或人工确认残留状态
- 已经通知用户“成功”，但下游实际没有完成，需要补发说明

因此，长任务设计不只是 `retry`，还要考虑 `timeout`、`cancel`、`resume`、`compensate`。

## 用户反馈和状态同步怎么做

长任务如果没有状态同步，用户体验会非常差。

最小可行状态建议至少有：

- `queued`
- `running`
- `waiting_human`
- `failed`
- `completed`
- `cancelled`

如果任务会持续较长时间，还应提供：

- 当前阶段说明
- 最近一次失败原因
- 是否可重试或继续
- 部分产物或中间结果预览

这不仅是用户体验问题，也会直接影响人工接管效率。

## 常见失败点

- 把所有逻辑都塞在同步 API 里，导致超时后任务真实状态不明
- 没有幂等键，worker 重试后产生重复副作用
- 只保存最终结果，不保存中间检查点
- 工具失败后直接无限重试，没有区分瞬时失败和确定性失败
- 用户只能看到“处理中”，看不到任务卡在哪一步
