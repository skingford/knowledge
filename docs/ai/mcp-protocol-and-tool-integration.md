## MCP 先解决什么问题

MCP 最核心解决的不是“让模型更聪明”，而是：

> **让外部工具和上下文资源的接入方式更标准、更可复用。**

在没有 MCP 的情况下，很多 Agent 系统会遇到这些问题：

- 不同模型提供商的工具 schema 不一样
- 换一个宿主应用，就得重新接一遍工具
- 文档检索、浏览器、文件系统、设计工具各有各的接法
- 团队里同一个工具能力反复接、反复封装

MCP 的价值就是把“接入层”统一起来。

## MCP 和 Tool Calling 的区别

这两个概念很容易被混成一件事。

| 概念 | 解决什么问题 |
| --- | --- |
| **Tool Calling** | 模型如何表达“我想调用哪个工具、传什么参数” |
| **MCP** | 工具和资源如何以统一协议暴露给 Agent 或宿主应用 |

一个更直白的理解是：

- Tool Calling 更像“模型侧的调用意图”
- MCP 更像“系统侧的标准接入口”

所以 MCP 不是替代 Tool Calling，而是让 Tool / Resource 的接入更标准化。

## Client、Server、Resource、Tool 各自是什么

MCP 里最关键的几个角色可以这样理解：

| 角色 | 可把它理解成什么 |
| --- | --- |
| **MCP Server** | 对外暴露工具和资源的能力提供方 |
| **MCP Client** | 连接并使用这些能力的宿主应用或 Agent |
| **Tool** | 可执行动作，例如搜索、读文件、调 API |
| **Resource** | 可读取上下文，例如文档、设计稿、数据库视图 |

在很多工程场景里，真正有价值的不只是 Tool，还有 Resource。因为很多系统需要的不是“执行动作”，而是“稳定地拿到外部上下文”。

## 什么时候适合引入 MCP

下面这些信号比较强：

- 同一套工具能力希望被多个宿主复用
- 团队已经不只在一个 AI 应用里使用这些外部能力
- 文档、浏览器、设计工具、文件系统、数据库等接入越来越多
- 希望接入层和模型提供商解耦

这时引入 MCP 往往可以显著降低重复接入成本。

## 什么时候不必急着上 MCP

如果你现在只是：

- 一个很小的 demo
- 只有 1-2 个本地工具
- 只会在一个应用里使用
- 团队还在验证场景本身值不值得做

那先直接用原生 Tool Calling 或本地函数封装，通常更轻。

别把 MCP 当成“起步必备”，它更像是**接入层复杂度上来之后的标准化工具**。

## 接入时最该注意什么

### 1. 别只暴露工具，要想清资源

很多团队只会问“这个能力能不能做成 tool”，却忽略了：

- 文档
- 文件
- 设计稿
- 结构化上下文

这些往往更适合被建模成 resource。

### 2. 权限边界不能因为标准化而变松

MCP 统一的是接入方式，不代表权限可以统一放开。

仍然要做：

- 最小权限
- 白名单
- 审批节点
- 只读 / 可写隔离

### 3. 版本和稳定性要可追溯

如果一个 MCP Server 升级后工具签名变了、返回结构变了，却没有清晰版本和兼容策略，宿主侧会很难排障。

## 常见失败点

- 把 MCP 当成“更高级的 Tool Calling”
- 在还没形成复用需求前，过早引入协议层
- 只考虑 tool，不考虑 resource
- 标准化接入之后，反而忽略了权限、版本和审计
- 以为用了 MCP 就自动解决了系统设计问题
