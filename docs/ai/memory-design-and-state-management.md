# Agent Memory 设计与状态管理

## 适合人群

- 做过多轮对话或任务型 Agent，但经常觉得“记忆”这个词越用越乱的人
- 正在设计会话摘要、用户偏好、任务状态和长期信息存储的人
- 想分清什么该进上下文、什么该持久化、什么应该忘掉的工程师

## 学习目标

- 把 Memory 从笼统概念拆成几类不同职责的状态
- 理解写入、读取、更新、遗忘和纠错的工程边界
- 知道 Memory、Context、RAG、Harness 分别解决什么问题

## 为什么很多“记忆问题”本质上是状态设计问题

用户常说“Agent 没记住我说过的话”，但真正的问题可能完全不同：

- 没有把关键信息写入任何地方
- 写入了，但下次没读出来
- 读出来了，但没放进当前上下文
- 放进去了，但被噪音淹没
- 信息早就过期，却还在被当真

所以 Memory 不是一个“更聪明的 Prompt”。它本质上是**状态设计 + 生命周期管理**。

## 先分清 5 类 Memory

不同记忆的作用完全不同，混在一起设计最容易失控。

| 类型 | 作用 | 存多久 |
| --- | --- | --- |
| **工作记忆** | 当前一步推理或当前阶段要临时用的信息 | 只在当前运行内保留 |
| **会话记忆** | 本次对话的重要摘要、目标、约束 | 一次会话或一次任务周期 |
| **用户记忆** | 用户偏好、常用格式、身份背景 | 跨会话长期保留 |
| **任务状态** | 当前任务的步骤、检查点、产物、失败原因 | 一次任务生命周期 |
| **知识记忆** | 案例、文档、FAQ、工具说明等可检索知识 | 长期保留，按版本更新 |

一个非常实用的判断是：

- `当前正在做什么`，通常属于 **任务状态**
- `这个用户长期偏好什么`，通常属于 **用户记忆**
- `当前轮次推理必须看到什么`，通常属于 **工作记忆 / 会话记忆**
- `组织级事实或外部文档`，通常不叫 Memory，更接近 **RAG 知识库**

## 什么时候该写入

不是所有信息都值得写入长期记忆。

更稳的写入规则通常是：

- **用户明确表达的稳定偏好**：例如“以后默认给我中文回答”
- **工具返回且可验证的事实**：例如客户等级、任务状态、工单号
- **会话压缩后的关键摘要**：例如当前目标、已完成步骤、待确认事项

不建议直接写入的内容：

- 模型推断出来但没有证据支持的“猜测”
- 短期情绪或一次性上下文
- 原始长对话全文

一个常见的稳妥原则是：**越长期的记忆，越需要更严格的写入门槛**。

## 什么时候该读取

Memory 不是读得越多越好。读太多会把上下文污染得很严重。

建议按优先级读取：

1. 当前任务的任务状态
2. 本次会话的摘要
3. 与当前意图强相关的用户偏好
4. 必要时再补充长期知识或历史案例

如果一个字段对当前决策没有帮助，就不要因为“它存在”而塞进上下文。

## 遗忘、过期和纠错怎么做

记忆系统最大的风险之一不是“记不住”，而是“记错了还一直在用”。

推荐至少设计 3 个机制：

### 1. TTL / 过期时间

- 短期任务状态和工作记忆应有明确过期时间
- 会话摘要可按会话结束或任务结束清理

### 2. 版本和更新时间

- 每条长期记忆最好保留 `updated_at`
- 高价值事实最好保留来源和版本

### 3. 覆盖与纠错

- 用户显式更正时，应允许覆盖旧偏好或旧事实
- 对关键字段，优先以最新可信工具结果为准，而不是历史摘要

## 数据结构怎么设计更稳

刚开始不要追复杂语义图谱。多数业务里，把数据结构做清晰就已经很有用。

推荐至少区分这几类存储：

| 存储对象 | 推荐结构 |
| --- | --- |
| **用户偏好** | `key-value` 或结构化 profile |
| **会话摘要** | 时间戳 + 摘要文本 + 当前目标 |
| **任务状态** | 状态机字段、步骤列表、检查点、产物引用 |
| **长期事实** | 结构化记录 + 来源 + 更新时间 |

如果需要写入模型生成的摘要，最好同时保留：

- 摘要文本
- 生成时间
- 来源范围
- 版本号

这样后面做复盘和纠错才有抓手。

## 和 Context、RAG、Harness 的关系

这四者很容易混，但职责不同：

| 能力 | 回答的问题 |
| --- | --- |
| **Memory** | 有哪些状态应该被长期或阶段性保存 |
| **Context** | 当前这一步推理应该看到哪些信息 |
| **RAG** | 外部知识如何被召回和引用 |
| **Harness** | 这些状态何时读写、如何限流、如何回放和恢复 |

一个常见工程形态是：

1. Harness 触发一次任务
2. 先读任务状态和会话摘要
3. 再按需要检索用户偏好和外部知识
4. Context 负责把这些信息组装进模型输入
5. 任务结束后，再决定哪些结果值得写回 Memory

## 常见失败点

- 把“所有历史消息”直接当长期记忆存
- 把用户一次性指令写成永久偏好
- 没有 TTL 和更新时间，导致过期信息长期污染决策
- 任务状态和用户偏好混在一起，导致读写逻辑越来越乱
- Memory 写入太松，模型猜出来的内容也被当真
