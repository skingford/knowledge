# 模型路由与成本控制

## 适合人群

- 已经把 Agent 系统跑起来，但开始觉得成本、延迟和稳定性压力越来越大的人
- 想设计强模型、快模型、小模型协同策略的工程师
- 需要把“模型选型”从拍脑袋变成系统化规则的人

## 学习目标

- 理解模型路由不是单纯省钱，而是平衡质量、延迟和预算
- 建立按任务类型、风险级别和步骤价值分配模型的思路
- 学会把成本控制接进评测和回归，而不是上线后才补救

## 为什么模型路由不是可选优化

很多团队前期会先用一个强模型把一切跑通，这没问题。但继续往前走时，通常会同时遇到三件事：

- 成本开始明显上升
- 延迟开始影响体验
- 某些低价值步骤其实不需要最强模型

这时模型路由就不再只是优化项，而是系统设计的一部分。

## 先分清 4 类任务

不是所有步骤都值得用同一种模型。

| 任务类型 | 更关心什么 |
| --- | --- |
| **理解 / 路由** | 低延迟、稳定分类 |
| **抽取 / 结构化** | schema 稳定、成本可控 |
| **复杂推理 / 规划** | 推理质量、上下文处理能力 |
| **审核 / 最终输出** | 准确性、风格、风险控制 |

很多系统里最常见的浪费是：

> 把所有步骤都交给同一个最强模型。

## 一个常见的路由策略

一个比较实用的默认分层通常是：

- **轻量模型**：做路由、分类、抽取、简单改写
- **中等模型**：做一般问答、RAG 汇总、结构化生成
- **强模型**：做复杂规划、高风险决策、最终审核

这类分层的价值在于：

- 让贵的能力只出现在真正高价值步骤上
- 不让每个小任务都背最贵模型的成本

## 哪些地方最值得省

通常最值得优先优化的是：

- 路由与分类
- 重复性的结构化抽取
- 简单总结和格式转换
- 可缓存的中间结果

这些步骤往往量大、频繁、模式稳定，最适合下沉到更便宜模型或缓存层。

## 哪些地方不该盲目省

下面这些位置，如果省过头，通常会引发更大系统代价：

- 高风险动作前的最终判断
- 多步复杂任务的规划和校验
- 错一次就代价很高的用户交付内容

这些环节更适合优先保质量，而不是一味降成本。

## 预算控制怎么做

比较稳的预算控制通常包括：

- 每任务 token 预算
- 每类工具调用上限
- 高风险路径单独预算
- 超预算后的降级或人工接管

预算控制最好在系统层完成，而不是事后对账才发现问题。

## 怎么把成本接进回归

只看质量指标不够。模型路由调整之后，至少要同时比较：

- 任务完成率
- 结构合法率
- 延迟
- 单任务成本
- 人工接管率

否则很容易出现：

- 成本降了，但质量明显退化
- 质量没掉，但延迟恶化
- 低风险步骤省了钱，高风险步骤却出事故

## 常见失败点

- 所有步骤都用一个最强模型
- 所有步骤都为了省钱强行下沉到便宜模型
- 路由策略没有结合任务风险和业务价值
- 缺乏成本回归，只在线上靠账单发现问题
- 忽略缓存、摘要和中间结果复用，只盯模型单价
