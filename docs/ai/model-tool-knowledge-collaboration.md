# 模型、工具与知识库协同设计

## 适合人群

- 已经把 LLM、Tool Calling、RAG 都接起来了，但系统边界越来越糊的人
- 经常纠结“这个问题该让模型直接答、该查知识库，还是该调工具”的工程师
- 想把 Agent 从“能跑”升级到“职责清晰、可扩展、好排障”的团队

## 学习目标

- 分清模型、工具、知识库、记忆、工作流各自该解决什么问题
- 学会根据任务特征选择“直接回答 / 检索 / 调工具 / 多步编排”
- 建立更接近生产系统的协同设计视角，而不是单点堆能力

## 先别把所有问题都交给模型

很多 Agent 系统做着做着会变成一句话：

> “把所有相关信息塞进上下文，然后让模型自己判断。”

这在 Demo 阶段很方便，但上线后往往会遇到几个问题：

- 成本高，什么都要长上下文
- 不稳定，同一问题不同轮次决策可能漂
- 难排障，不知道错在知识、工具还是推理
- 难扩展，一旦工具和知识源变多，Prompt 会迅速失控

所以更稳的做法不是“让模型接管一切”，而是让模型和外部能力各做自己擅长的部分。

## 5 个核心角色分别做什么

| 组件 | 最擅长解决什么 |
| --- | --- |
| **模型（LLM）** | 理解意图、归纳、生成、轻规划、选择下一步 |
| **工具（Tools）** | 访问实时世界、执行动作、读写结构化系统 |
| **知识库（RAG）** | 提供外部事实、文档依据、历史案例、组织知识 |
| **记忆（Memory）** | 保存用户偏好、会话摘要、任务状态、长期稳定信息 |
| **工作流（Workflow / Harness）** | 控制顺序、重试、分支、审批、预算与收敛 |

一个非常实用的判断标准是：

- **需要“知道”什么**，优先想知识库
- **需要“做”什么**，优先想工具
- **需要“记住”什么**，优先想记忆
- **需要“按步骤稳定推进”什么**，优先想工作流
- **需要“理解、选择、归纳、组织语言”什么**，交给模型

## 什么时候该直接让模型回答

下面这类问题，通常可以直接依赖模型：

- 通用解释、归纳、改写、总结
- 对上下文要求不高的开放式表达
- 不依赖实时数据和外部事实的轻任务

但要注意：

- 这不代表模型“知道所有事实”
- 一旦进入组织知识、最新状态、业务规则，就不该只靠参数记忆

## 什么时候该查知识库

知识库最适合处理下面这类问题：

- 组织内部文档、FAQ、规范、历史案例
- 需要引用依据、来源、上下文的回答
- 模型参数里不一定可靠、而团队又希望 grounded 的内容

典型信号：

- 问题不是让模型“推”，而是让模型“基于已有资料答”
- 用户关心“依据是什么”
- 内容更新频繁，但没必要做在线工具操作

## 什么时候该调用工具

工具更适合处理“实时”和“动作型”任务。

典型场景：

- 查询数据库、搜索网页、读文件、调用 API
- 创建工单、发消息、写日历、执行脚本
- 需要结构化结果，而不是一段自然语言猜测

一个很关键的边界是：

- **知识库解决“资料是什么”**
- **工具解决“现在系统里是什么 / 我需要执行什么”**

比如：

- “公司报销流程是什么”更像知识库问题
- “我这张报销单当前审批到哪一步了”更像工具问题

## 什么时候必须进工作流

当任务具备下面这些特征时，最好别只靠“模型决定下一步”：

- 多步骤且顺序重要
- 某些步骤失败后需要重试或降级
- 包含高风险写操作或人工审批
- 需要阶段产物、检查点和恢复能力

这时更稳的方式是：

- 让工作流决定“步骤框架”
- 让模型在每一步里做局部判断

## 一个常用的协同判断顺序

很多任务其实可以用一个简单问题链来判断：

1. 这是纯语言组织任务，还是需要外部事实？
2. 如果需要外部事实，是静态知识还是实时状态？
3. 如果是实时状态，是只读查询还是会产生副作用的动作？
4. 如果任务有多步、审批或失败恢复，是不是应该交给工作流兜底？

把这个顺序跑一遍，设计就会清晰很多。

## 典型组合模式

### 1. 模型 + 知识库

适合：文档问答、知识助手、内部搜索解释器。

### 2. 模型 + 工具

适合：查天气、查数据库、自动填表、调用业务 API。

### 3. 模型 + 知识库 + 工具

适合：先查规范，再查实时状态，最后综合回答。

### 4. 工作流 + 模型 + 工具 + 知识库

适合：复杂业务流程、审批流、长任务、跨系统任务。

这里最常见的稳定形态不是“全自动自治 Agent”，而是：

> 工作流定框架，模型做理解与选择，知识库供事实，工具做动作，记忆保存状态。

## 常见失败点

- 把组织文档硬塞进 Prompt，而不是做检索
- 把实时业务状态放进知识库，结果内容过期
- 明明是强约束流程，却完全交给模型自由发挥
- 工具和知识库边界不清，出现“既查文档又查实时数据却不知道谁更可信”
- 让模型直接决定高风险写操作，没有工作流和审批兜底
