## 为什么 Agent 不能只靠普通日志

传统服务里，一次请求通常是确定性的：输入差不多，代码路径也差不多。

Agent 不一样。一次任务可能包含：

- 多轮模型调用
- 多个工具调用
- 动态上下文拼装
- 路由、重试、handoff
- 人工接管或审批

所以只看“接口请求日志”远远不够。你需要看到的是**整条任务轨迹**。

## 一次运行最少要记录什么

如果你想做到基本可排障，一次 Agent 运行至少应记录下面这些信息：

| 类别 | 最少字段 |
| --- | --- |
| **任务标识** | `trace_id`、`task_id`、用户 ID、会话 ID |
| **模型调用** | 模型名、参数、输入摘要、输出摘要、token、耗时 |
| **上下文信息** | system prompt 版本、上下文来源、检索片段 ID、摘要版本 |
| **工具调用** | 工具名、参数、结果摘要、耗时、错误码 |
| **控制流** | 当前 step、状态迁移、重试、handoff、终止原因 |
| **安全 / 预算** | 权限命中、审批节点、预算消耗、限流情况 |

这里的关键不是“全量存所有原文”，而是至少要有**可追溯的结构化事件**。

## Prompt、Context、Tool 为什么必须版本化

没有版本号，你很难解释“为什么上周能过，这周突然不行了”。

推荐至少给这些对象加版本：

- **Prompt 模板版本**
- **上下文模板版本**
- **工具 schema 版本**
- **检索配置版本**
- **工作流节点版本**

这样你看到失败样本时，才知道它到底跑在哪一版系统上。

## Trace 应该长什么样

一个可用的 Agent Trace，通常是按 step 组织的事件流，而不是一大段文本日志。

推荐的最小结构：

1. `task_started`
2. `context_built`
3. `model_called`
4. `tool_called`
5. `state_updated`
6. `handoff` 或 `retry`
7. `task_finished` 或 `task_failed`

如果你能把这套事件稳定打出来，后面再接可视化和检索平台会容易很多。

## 回放怎么做才有用

很多团队说自己“支持回放”，实际上只能重跑一遍请求，这不够。

真正有用的回放，至少要回答两个问题：

- 当时模型看到了什么
- 当时系统为什么走到了这一步

所以回放通常需要保留：

- 输入快照
- 上下文快照或上下文引用
- Prompt / Tool / Workflow 版本
- 工具结果摘要或关键外部依赖状态

要注意的是，Agent 回放通常不是 100% 确定性的。更现实的目标是：

- **结构性回放**：尽量复现步骤和状态迁移
- **证据性回放**：看清当时输入、工具结果和决策依据

## 线上指标和 SLO 怎么定

除了传统的成功率、延迟和错误率，Agent 还建议关注：

| 指标 | 说明 |
| --- | --- |
| **任务完成率** | 是否真正达到目标 |
| **人工接管率** | 多少任务需要人补救 |
| **工具错误率** | 外部依赖是否成为主要故障源 |
| **平均步数** | 是否出现链路膨胀或循环 |
| **每任务成本** | token、工具成本、调用次数 |
| **失败分类分布** | Prompt / Context / Tool / Workflow 各占多少 |

一个很实用的习惯是给失败打标签，而不是统一归为 `agent_failed`。

## 故障复盘闭环

真正让系统持续变稳的，不是“有日志”，而是“能复盘并回灌到工程流程”。

建议闭环如下：

1. 线上发现失败样本或异常峰值
2. 用 Trace 和回放定位根因
3. 归类为 Prompt / Context / Tool / Workflow / 权限问题
4. 修复后，把这个案例沉淀进离线评测集
5. 后续每次改版自动回归

这样可观测性就不只是“看报表”，而是直接推动系统进化。

## 常见失败点

- 只记录最终回答，不记录中间步骤
- 没有 prompt、context、tool 的版本信息
- 工具失败只有一行错误字符串，没有输入参数和上下文
- 线上失败样本没有进入离线回归
- 指标只看延迟，不看任务完成率、人工接管率和成本
