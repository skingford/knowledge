# Prompt 版本化、实验与回滚

## 适合人群

- 已经开始频繁改 Prompt、Context 模板或工具说明，但团队靠口口相传管理版本的人
- 想把 Prompt 从“临场发挥”变成“可追踪、可回归、可回滚”的工程资产的人
- 需要设计 A/B、影子流量和回滚策略的工程师

## 学习目标

- 建立 Prompt 版本化和实验管理的基本工程方法
- 理解 prompt、context、tool schema、workflow 这些“行为契约”为什么都要一起版本化
- 学会在上线和回归中管理 prompt 变更，而不是靠记忆和截图

## 为什么 Prompt 必须版本化

一旦系统开始演进，Prompt 就不再只是“一段文本”，而更像系统行为的一部分。

如果不做版本化，团队很快会遇到这些问题：

- 不知道这次线上结果到底用了哪版 Prompt
- 改完一条约束后，另一个场景退化了，但没人说得清是从哪次开始的
- 一个同事说“我昨天改过”，另一个同事说“我本地还有旧版”

所以 Prompt 工程里最关键的一步，往往不是“写得更聪明”，而是“变更可追溯”。

## 哪些对象应该一起版本化

很多团队只给 prompt 起版本号，但真正影响行为的对象不止这一项。

建议至少一起管理：

- **system prompt 模板**
- **few-shot 样例**
- **上下文拼装模板**
- **tool schema / 工具描述**
- **workflow 路由规则**
- **检索配置**

因为很多时候，线上“Prompt 出问题”其实根因在：

- context 模板换了
- schema 改了
- 检索块排序变了
- workflow 节点逻辑变了

## 最小可行版本策略

最简单但够用的做法通常是：

- 把 prompt 模板放进代码仓库
- 明确文件名和用途
- 给每次变更写简短说明
- 在 trace 里记录实际运行版本

如果系统更复杂，可以把版本抽成结构化字段，例如：

- `prompt_version`
- `context_version`
- `tool_schema_version`
- `workflow_version`

这样观察线上问题时就能更快定位。

## 实验怎么做才不乱

Prompt 实验最容易犯的错是：

> “先改一版试试看，如果感觉好就留着。”

这种做法在 demo 阶段还行，但一旦系统接近上线，就很容易制造长期混乱。

更稳的实验闭环通常是：

1. 明确这次实验要改善什么
2. 选定对照版本
3. 规定成功指标和护栏指标
4. 保留失败样本
5. 不通过就能快速撤回

## 什么时候适合 A/B

`A/B` 更适合下面这类情况：

- 用户量已经有一定规模
- 可以容忍两个版本同时存在一段时间
- 想比较总体任务完成率、结构合法率、人工接管率等结果指标

它的优势是更接近真实使用场景，但前提是：

- 实验目标明确
- 指标事先定义好
- 回滚路径清楚

## 什么时候适合影子流量

影子流量更适合：

- 不想直接影响真实用户输出
- 想在生产环境中验证新版本行为
- 想对比新旧版本 trace、token、成本和路径差异

影子流量的价值通常在于：

- 先看真实输入分布
- 不立刻暴露给用户
- 可以提早暴露退化点

## 回滚怎么做更稳

Prompt 回滚如果靠“把旧文案找回来”，基本上不算真正可回滚。

更稳的回滚通常要求：

- 旧版本可直接切换
- 回滚条件提前定义
- 版本切换对 trace 和观测可见
- 失败样本能保留下来继续分析

一个很实用的原则是：

> 回滚不是承认失败，而是给实验留出生路。

## 常见失败点

- Prompt 有版本号，但 context 和 tool schema 没有
- 线上实验没有事先定义指标
- 只看“回答好像更自然了”，不看完成率和人工接管率
- 改 prompt 时顺手改了 schema、workflow，却没有一起记录
- 回滚只能靠人工复制粘贴旧文本
