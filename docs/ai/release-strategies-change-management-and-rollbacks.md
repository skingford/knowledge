# Agent 灰度发布、变更管理与回滚

## 适合人群

- 已经开始频繁改 Prompt、模型、工具和知识库配置的人
- 准备把 Agent 从内测推向试点、灰度或正式流量的团队
- 想把“可以发布”变成“可控地发布、可控地回滚”的工程师或负责人

## 学习目标

- 理解 Agent 变更为什么比普通页面发布更容易出连锁反应
- 学会把 Prompt、模型、工具、知识库和 workflow 视为一个变更包管理
- 建立灰度发布、观测、回滚和复盘的最小闭环

## 为什么 Agent 变更比普通发布更脆弱

普通前端改一个样式，影响范围通常比较清楚。Agent 不是这样。

一次看似很小的改动，例如：

- 改了一句 system prompt
- 切了一个新模型
- 调整了工具参数 schema
- 更新了知识库切块策略
- 修改了路由和预算策略

都可能对最终行为造成连锁影响。

因此 Agent 发布更像“行为系统发布”，而不只是“代码发布”。

## 什么才算一个完整变更包

更稳的做法是把下面这些一起看成发布对象，而不是只看代码 diff。

| 变更项 | 典型内容 |
| --- | --- |
| **Prompt 变更** | system prompt、few-shot、输出规则 |
| **模型变更** | 模型版本、temperature、reasoning 配置 |
| **工具变更** | schema、权限、超时、重试、幂等逻辑 |
| **知识库变更** | 切块、索引、重排、过滤规则 |
| **工作流变更** | 路由、状态机、handoff、人工接管条件 |
| **治理变更** | 审批策略、预算上限、降级策略 |

一旦把这些视为一个整体，发布前后就更容易定位问题，也更容易整包回滚。

## 常见灰度策略怎么选

| 策略 | 适用场景 | 特点 |
| --- | --- | --- |
| **离线回归** | 所有变更的基础动作 | 成本低，但看不到真实流量分布 |
| **影子流量** | 想验证新版本轨迹但不影响用户结果 | 风险低，适合先看行为偏差 |
| **小流量灰度** | 准备进入真实结果输出 | 能看到真实反馈，但要盯指标 |
| **A/B 对比** | 需要比较两套版本优劣 | 适合做收益判断，不只是安全验证 |
| **分场景切流** | 高低风险场景差异大 | 更容易控制风险扩散面 |

很多团队真正稳的顺序不是“改完就上”，而是：

1. 先离线回归
2. 再影子流量
3. 再低风险场景灰度
4. 最后才全量

## 发布前闸门应该检查什么

最小可行发布闸门通常包括：

- 关键回归集是否通过
- 高风险样本是否人工抽检
- 关键工具是否做了联调验证
- 回滚版本是否随时可切回
- 人工接管通道是否可用
- 观测面板和告警是否准备好

如果这几个条件都没有，最好不要把“灰度”理解成“拿线上用户帮你测”。

## 发布中应该盯哪些信号

不要只盯一个“整体成功率”，要同时看行为和成本。

推荐重点观察：

- 任务完成率
- 工具成功率
- 人工接管率
- 延迟和 token 成本
- 高风险错误数
- 用户纠错或投诉信号

如果系统是分场景发布，还要按场景拆开看，而不是看一个总平均值。

## 回滚条件和回滚方式怎么定

回滚条件最好在发布前就写清楚，而不是出事后再讨论。

常见触发条件包括：

- 任务完成率明显下跌
- 高风险错误出现连续增长
- 人工接管率突然升高
- 单任务成本或延迟超出预算
- 工具副作用错误开始出现

回滚方式至少应有两层：

1. **快速回切**：把版本指向上一套稳定 Prompt / 模型 / workflow
2. **整包回滚**：把知识库、工具 schema、治理配置一起退回上一版本

如果回滚只能退代码，退不了 Prompt、索引和策略，那很多事故其实仍然回不到安全状态。

## 发布后复盘怎么做

每次重要变更后，最好都保留一份轻量复盘：

- 本次改了什么
- 指标如何变化
- 新增了哪些失败模式
- 哪些失败样本要进入回归集
- 下一次发布前需要补什么闸门

这样发布体系才会越来越稳，而不是每次都从头踩坑。

## 常见失败点

- 只记录代码版本，不记录 Prompt、知识库和工具 schema 版本
- 一次改太多层，没有把变更包拆清楚
- 灰度期间只看平均分，不看高风险场景
- 没有明确回滚阈值，出事时全靠临场判断
- 回滚只能退服务代码，退不了配置、索引和策略
