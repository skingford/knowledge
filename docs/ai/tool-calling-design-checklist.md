# Tool Calling 设计清单

## 适合人群

- 正在做 Agent 或 LLM 应用开发的工程师
- 已经会调用模型，但工具调用效果不稳定
- 需要把 Demo 调整到可维护、可调试的状态

## 学习目标

- 理解 Tool Calling 的核心设计点
- 降低误调用、漏调用、参数错误等问题
- 建立一套可复用的工具设计清单

## 设计原则

- 工具要小而明确，不要一个工具承担太多职责
- 工具名要表达动作，不要过于抽象
- 参数要尽量结构化，减少模型自由发挥空间
- 返回值要面向下一步决策，而不是只返回大段原始文本
- 能用规则解决的事情，不要完全依赖模型判断

## 工具定义清单

### 工具名称

- 使用动词开头，例如 `search_web`、`get_weather`、`query_orders`
- 避免 `handle_task`、`do_action` 这类泛化名称

### 工具描述

- 明确“什么时候该调用”
- 明确“什么时候不该调用”
- 明确输入范围和边界条件

### 参数设计

- 参数名要直接反映业务含义
- 必填参数尽量少，但要覆盖关键约束
- 枚举值优于自由文本
- 时间、金额、状态等字段优先结构化

### 返回结果

- 优先返回结构化字段
- 保留必要状态，例如 `success`、`error_code`、`not_found`
- 避免让模型从超长文本里二次提取关键信息

## 调用流程清单

### 调用前

- 是否真的需要工具，而不是模型直接回答
- 是否已具备调用所需参数
- 是否需要先澄清用户意图

### 调用中

- 记录工具名、参数、耗时、结果状态
- 对失败结果做重试或降级
- 对高风险操作做确认或权限校验

### 调用后

- 校验返回值是否完整
- 判断是否需要继续调用下一个工具
- 将工具结果转成对用户可读的结论

## 常见失败点

- 工具描述太泛，模型不知道何时调用
- 参数过多，模型经常填错
- 返回结果非结构化，后续步骤难处理
- 一个工具包含多个分支逻辑，模型容易误用
- 没有日志，出了问题无法复盘

## 调试方法

- 收集真实失败样本，而不是只看成功案例
- 对每个工具单独做测试集
- 记录“为什么调用”和“为什么没调用”
- 区分是 Prompt 问题、Schema 问题，还是工具本身问题
- 对高频失败场景增加 few-shot 或规则分流

## 推荐实践项目

- 搜索问答 Agent：重点练搜索工具与总结工具协作
- 办公助手 Agent：重点练日历、天气、提醒类工具设计
- 数据助手 Agent：重点练查询类工具的参数约束与结构化输出
