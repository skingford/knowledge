## 先说结论

- **Redlock 是“多数派租约算法”**：客户端要在多个彼此独立的 Redis 主节点上拿到多数派锁，且剩余有效期仍大于 0，才算成功
- **它提升的是锁服务可用性，不是把 Redis 变成共识系统**
- **它适合效率型互斥，不适合账务级、资金级最终正确性**
- **真正难的问题不是“有没有拿到锁”，而是“旧持有者过期后还能不能继续写”**
- **如果业务需要把 stale owner 收敛住，关键不是 Redlock 本身，而是 `fencing token + 下游存储校验`，或者直接使用数据库条件更新、状态机、etcd / ZooKeeper**

一句话概括：

> Redlock 可以理解成“比单 Redis 锁更高可用的租约”，但不能把它理解成“严格互斥且永不出错的分布式锁”。

## Redlock 要解决什么问题

单实例 Redis 锁的标准做法通常是：

```text
SET lock:resource unique_value NX PX 30000
```

释放时再用 Lua 脚本校验 `value` 后删除：

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
  return redis.call("DEL", KEYS[1])
end
return 0
```

这个方案已经能解决很多工程问题，比如：

- 同一时刻只让一个 worker 处理任务
- 防止重复提交
- 在热点入口先挡掉并发冲突

但它有一个明显边界：**锁状态只保存在一个 Redis 实例里**。一旦这个实例宕机、不可用或故障切换异常，锁服务本身就不可靠了。

Redlock 提出的目标不是“让锁更强一致”，而是：

- 避免把锁能力完全压在一个 Redis 实例上
- 在部分实例故障时，客户端仍然有机会完成加锁和解锁
- 让“锁变量”存在于多个独立节点上，而不是只存在一个节点上

## 单实例 Redis 锁为什么不够

### 问题 1：单点故障

如果只有一个 Redis：

- Redis 挂了，客户端没法加锁
- Redis 重启丢数据，锁状态直接消失
- Redis 网络抖动，锁语义会被客户端超时放大

### 问题 2：主从切换不能天然保证锁正确

很多人会想：那我用 Redis 主从 + Sentinel 不就行了？

这里的关键问题是：**Redis 主从复制是异步的**。可能出现下面的经典场景：

```text
1. Client A 在主节点 M 上加锁成功
2. 锁还没复制到从节点 R
3. M 宕机，R 被提升为新的主节点
4. Client B 在新主节点上再次加锁成功
```

于是 A 和 B 都认为自己拿到了锁。

所以，Redlock 的设计前提不是“一个主从集群”，而是：

- 使用 `N` 个**相互独立**的 Redis 主节点
- 这些节点之间**不做复制**
- 客户端自己对这 `N` 个节点执行加锁和解锁

## Redlock 核心思路

Redlock 一般使用 `N = 5` 个独立 Redis 主节点，要求拿到多数派：

```text
quorum = floor(N / 2) + 1
```

例如：

- `N = 5`，多数派是 `3`
- `N = 3`，多数派是 `2`

客户端会对所有节点使用**同一个锁 key**、**同一个随机 value**、**同一个 TTL** 尝试加锁。

只有同时满足两个条件，才算真正持有锁：

1. 在至少 `quorum` 个节点上加锁成功
2. 从开始尝试到完成多数派加锁所消耗的总时间，小于锁的 TTL，并且扣除漂移后仍有剩余有效期

直观图如下：

```text
Client
  │
  ├─ SET lock x NX PX 10000 ──> Redis-1  success
  ├─ SET lock x NX PX 10000 ──> Redis-2  success
  ├─ SET lock x NX PX 10000 ──> Redis-3  timeout
  ├─ SET lock x NX PX 10000 ──> Redis-4  success
  └─ SET lock x NX PX 10000 ──> Redis-5  fail

成功数 = 3，达到多数派
但如果总耗时太长，导致有效期接近 0，仍然算失败
```

### 为什么多数派有意义

如果两个客户端都声称自己拿到了多数派，那么它们的成功节点集合一定有交集。

例如在 5 个节点里：

- A 想拿到至少 3 个成功
- B 也想拿到至少 3 个成功

两个大小都为 3 的集合，在 5 个节点里不可能完全不相交。也就是说，它们至少会在一个节点上冲突，而同一个节点在同一时刻只能让一个 `value` 成功写入。

这解释了为什么**在理想条件下**，不会同时出现两个有效多数派锁。

但注意，这里只是“理想条件下”。一旦出现租约过期、长暂停、网络延迟、时钟漂移，问题又会复杂很多，这也是 Redlock 后来争议的焦点。

## 获取锁完整流程

### 步骤 1：准备参数

客户端在加锁前需要准备：

- `resource`：锁资源名，例如 `lock:order:123`
- `value`：本次加锁的唯一随机值，例如 UUID
- `ttl`：锁租约时长，例如 10 秒
- `quorum`：多数派阈值
- `perNodeTimeout`：单个 Redis 实例的请求超时，必须明显小于 `ttl`

这里 `perNodeTimeout` 很关键。因为某个节点慢或卡住时，客户端不能一直等，否则锁还没拿齐，多数派有效期就已经快耗尽了。

### 步骤 2：记录开始时间

```text
startTime = now()
```

Redlock 的有效期是从**开始尝试加锁**那一刻算起，不是从最后一个节点成功那一刻才开始算。

### 步骤 3：依次或并发尝试加锁

对每个独立 Redis 节点执行：

```text
SET resource value NX PX ttl
```

要求：

- 使用相同的 `resource`
- 使用相同的 `value`
- 使用相同的 `ttl`
- 每个节点都要受 `perNodeTimeout` 约束

原始描述常写成“依次尝试”，但很多工程实现会做并行化或多路复用，以降低总耗时。核心不变：**最终判断还是看成功节点数和剩余有效期**。

### 步骤 4：统计成功数和耗时

```text
elapsed = now() - startTime
successCount = numberOfSuccessNodes
```

只有当：

```text
successCount >= quorum
```

并且：

```text
elapsed < ttl
```

时，才能进入下一步校验。

### 步骤 5：计算实际有效期

工程上通常不会把 `ttl - elapsed` 直接当成剩余有效期，而会再扣掉一个漂移余量：

```text
validity = ttl - elapsed - drift
```

其中 `drift` 用来覆盖：

- Redis 过期处理不是绝对实时
- 客户端和 Redis 之间存在时钟偏差
- 网络传播和本地调度有抖动

常见工程写法会把 `drift` 设成一个固定小量加上 `ttl` 的一个比例。重点不在精确公式，而在这个意识：

- **多数派成功，不等于你还有足够长的租约**
- **如果剩余有效期已经太短，这把锁在业务上基本不可用**

### 步骤 6：成功或失败

如果：

- `successCount >= quorum`
- `validity > 0`

则加锁成功。

否则加锁失败，并且客户端要对**所有成功加锁的节点**执行解锁，清理掉这次半成功留下的锁痕迹。

### 伪代码

```text
value = random()
start = now()
successNodes = []

for redis in redisNodes:
  ok = SET resource value NX PX ttl with perNodeTimeout
  if ok:
    successNodes.append(redis)

elapsed = now() - start
drift = ttl * driftFactor + fixedMargin
validity = ttl - elapsed - drift

if len(successNodes) >= quorum and validity > 0:
  return acquired(value, validity)

for redis in successNodes:
  unlock(resource, value)

return failed
```

## 释放锁与失败回滚

Redlock 的释放逻辑和单实例 Redis 锁本质一样：**只删除 value 属于自己的锁**。

Lua 仍然是标准方式：

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
  return redis.call("DEL", KEYS[1])
end
return 0
```

区别在于：Redlock 不是对一个实例执行，而是对所有节点都尝试执行。

### 正常释放

持锁客户端在业务完成后：

1. 向全部 Redis 节点发起解锁
2. 每个节点都只删除 `value` 匹配的锁
3. 某些节点解锁失败也没关系，因为 TTL 最终会兜底清理

### 失败回滚

如果本次加锁没有达到多数派，或者达到多数派但有效期已经不足：

1. 仍然需要对已经成功加锁的节点做 best-effort 解锁
2. 不能什么都不做直接返回失败
3. 否则会在部分节点上残留短暂锁，增加后续冲突概率

## 为什么必须关心有效期

Redlock 不是“永久锁”，它本质上是**租约**。

### 关键语义

当客户端认为自己持有锁时，它真正拥有的是：

- 在一段有限时间内
- 在多数派节点上
- 由同一个 `value` 标识的一组租约

这意味着一件很重要的事：

> 只要租约时间过去了，即使客户端自己还没停下，它也不该再认为自己有写入资格。

### 典型危险场景

```text
Client A 获取 Redlock，ttl = 10s
Client A 执行业务时发生 GC STW / 线程挂起 / 容器调度卡顿，暂停了 15s
期间锁自动过期
Client B 获取到新的多数派锁并开始写入
Client A 恢复后继续执行，误以为自己仍然持锁
```

这时就出现了经典的 stale owner 问题：

- A 不是“没拿到锁”
- A 是“拿到过锁，但租约已经失效”
- 如果下游存储不做额外校验，A 仍可能写出错误结果

这也是为什么很多资料会强调：

- Redlock 解决的是“锁服务高可用”和“尽量少冲突”
- 它不能单独解决“过期持有者恢复后仍继续写”的最终正确性问题

## Redlock 真的安全吗

这个问题不能直接回答“安全”或“不安全”，而要先问：

- 你要的是**效率型互斥**，还是**正确性型互斥**

### 对效率型互斥，它通常够用

例如：

- 定时任务去重
- 重复消费削峰
- 短临界区串行化
- 缓存重建时的互斥保护

这些场景的共同点是：

- 偶发双执行成本可控
- 即使锁偶尔失效，业务还有其他补偿手段
- 锁的主要价值是减冲突、降重复，而不是充当最终裁判

### 对正确性型互斥，它不够闭环

例如：

- 资金扣减
- 账务记账
- 库存最终扣减
- 需要绝对拒绝过期写入的状态变更

Redlock 的争议核心不在于“多数派会不会算错”，而在于它依赖这些假设：

- 时钟漂移是有界的
- 网络延迟是有界的
- 进程暂停不会长到超过租约很多
- 客户端在租约失效后不会继续影响下游

在真实生产环境里，下面这些情况都可能破坏最后一条假设：

- GC Stop-The-World
- 容器被调度挂起
- 长尾网络抖动
- 机器时钟异常
- 应用线程池阻塞

因此，很多工程实践会把 Redlock 的结论说得更保守：

> 它可以作为高可用分布式租约，但不能单靠它承担最终正确性。

### 真正能收住 stale owner 的是 fencing token

更稳妥的思路通常是：

1. 锁服务分配一个单调递增的 token
2. 每次写请求都带上这个 token
3. 下游存储拒绝旧 token 的写入

这样即使旧持有者恢复执行，它带着的 token 也已经过期，下游可以直接拒绝。

如果下游根本不校验 token，那么：

- 你即使做了 Redlock
- 即使做了续期
- 即使做了安全解锁

也仍然无法从根上解决“旧持有者恢复后继续写”的问题。

## 适用场景与不适用场景

### 适用场景

- 任务调度去重
- 消息消费侧的并发收敛
- 幂等前置保护
- 缓存重建互斥
- 对可用性要求高于单 Redis、但对绝对正确性要求没到共识级的场景

### 不适用场景

- 资金和账务类核心链路
- 必须靠锁本身保证绝对互斥的写操作
- 下游无法做 token 校验、条件更新或状态机约束的场景
- 跨地域高延迟部署，导致加锁耗时和漂移难以控制的场景

有一个实用判断标准：

如果业务在锁失效后发生双执行，会直接造成不可接受的错误结果，那么你就不该把 Redlock 当最终兜底。

## 工程落地建议

### 1. 先把单实例锁写对，再谈 Redlock

很多线上问题不是出在“没用 Redlock”，而是单实例锁本身就没写对，例如：

- 只用 `SETNX` 不带 TTL
- 直接 `DEL` 解锁
- `value` 不是唯一值
- TTL 明显短于业务 P99 执行时长

如果这些基础都没做好，上 Redlock 只是把复杂度放大。

### 2. 节点必须独立，且通常使用奇数个

经验上常见配置是 3 个或 5 个独立节点：

- 1 个节点不是 Redlock
- 2 个节点没有清晰多数派容错意义
- 4 个节点相比 5 个节点通常没有优势

“独立”强调的是：

- 不是同一个主从复制组
- 不共享同一个故障域更好
- 不建议跨地域拉太远，否则 RTT 会直接侵蚀有效期

### 3. TTL 要按业务最慢路径来定

至少覆盖：

- 业务处理 P99
- 下游 RPC 抖动
- 数据库慢 SQL
- 重试和序列化开销

TTL 太短会频繁出现租约过期后的脑裂，TTL 太长则会增加失败重试的等待成本。

### 4. 每个节点超时要远小于 TTL

例如：

- `ttl = 10s`
- 单节点超时可能只给几十毫秒到几百毫秒

目标是让坏节点尽快失败，不要拖累整轮加锁。

### 5. 失败后必须做 best-effort 清理

Redlock 加锁失败不是简单返回 `false` 就完了。

如果已经在部分节点写入成功，就应该尽快回滚：

- 减少残留锁窗口
- 降低下一次加锁冲突
- 避免测试环境里出现“偶现锁还在”的假象

### 6. 续期不是银弹

一些客户端库会给锁做 watchdog 自动续期。它能解决的是：

- 业务时间超出初始 TTL 时，不至于过早过期

它解决不了的是：

- 旧持有者在长暂停恢复后继续写
- 下游不知道谁才是最新持有者

所以续期只能改善体验，不能替代正确性设计。

### 7. 最终正确性放回业务状态机或存储约束

比较稳妥的组合通常是：

- Redlock 负责降低并发冲突概率
- 数据库唯一索引 / 条件更新 / 乐观锁负责最终不写错
- 必要时用 fencing token 让下游识别过期持有者

这和仓库里其他文档的口径是一致的：

> Redis 锁负责“尽量少冲突”，数据库约束、状态机和条件更新负责“最终不做错”。

## Redlock vs 其他方案

| 方案 | 优点 | 局限 | 适合场景 |
| --- | --- | --- | --- |
| 单 Redis 锁 | 简单、低延迟、实现成本最低 | 单点可用性弱，主从切换有复制空窗 | 防重、轻量互斥、效率优先 |
| Redlock | 比单 Redis 更高可用，能容忍部分节点故障 | 仍是租约语义，无 fencing token，复杂度更高 | 需要高可用互斥，但能接受租约边界 |
| etcd / ZooKeeper 锁 | 语义更强，天然更接近一致性协调器 | 延迟和运维成本更高 | 需要明确锁语义、租约管理、版本控制 |
| 数据库唯一索引 / 条件更新 | 最贴近业务状态，最终正确性强 | 吞吐和热点能力通常不如内存锁 | 账务、订单、库存、状态流转 |

如果只说一句选型建议：

- **轻量防重**：先用单 Redis 锁
- **高可用互斥**：再考虑 Redlock
- **强正确性**：优先数据库约束、状态机、fencing token、etcd / ZooKeeper

## 高频问题

### Q1：Redlock 最少需要几个节点？

理论上至少 3 个独立节点才有多数派意义。工程上更常见的是 5 个节点，因为它允许容忍 2 个节点故障，但成本和延迟也会更高。

### Q2：Redlock 一定比单 Redis 锁好吗？

不一定。如果你的场景只是：

- 重复请求削峰
- 短任务互斥
- 偶发双执行可接受

那么单 Redis 锁往往更简单、延迟更低、维护成本更小。

### Q3：Redlock 和 Redisson watchdog 是一回事吗？

不是。Redlock 是**多独立 Redis 主节点上的多数派算法**；watchdog 是**对锁租约自动续期**的机制。两者可以组合，但语义并不相同。

### Q4：Redlock 能完全避免双持锁吗？

在理想条件下，多数派交集能显著降低双持锁概率；但在租约过期、长暂停、网络异常、时钟漂移等现实条件下，仍然可能出现旧持有者继续执行的问题。所以它不能被理解为绝对不会双写的强一致锁。

### Q5：如果我已经做了安全解锁，为什么还不够？

Lua 安全解锁只能保证“我不会删掉别人的锁”。它解决不了“我的锁已经过期，但我自己还在继续执行”这个 stale owner 问题。

### Q6：什么时候应该直接放弃 Redlock？

当你发现业务正确性必须依赖锁本身兜底，而且下游又无法做 token 校验、条件更新、唯一约束或状态机控制时，就应该放弃把 Redlock 当核心方案，转向更强的一致性组件或把正确性收回数据库。
