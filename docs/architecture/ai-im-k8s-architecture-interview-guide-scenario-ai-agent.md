## 场景题一：AI Agent 编排服务的高并发调度与架构抽象

假设题目是：

> 服务接收复杂用户输入，内部做意图识别，并发调度多个外部工具，最后由 LLM 汇总并流式返回。

这个题不要平均发力。最容易拉开差距的是三件事：

- 长耗时、多 IO 场景下的调度与资源保护
- 工具编排的抽象模型是否可扩展
- 流式输出、失败补偿和线上稳定性是否成体系

答题时我通常分两段来讲：

- 先讲底层并发控制与系统防雪崩，这是防守篇
- 再讲图编排、状态传递和 AI 与核心系统之间的边界，这是进攻篇

### 1. 先定目标和边界

我会先明确四个约束：

- 用户请求是同步流式返回，还是异步任务 + 回调/轮询
- 工具调用是否允许并发，是否要求严格顺序
- 外部依赖是否不稳定，是否存在长尾延迟
- 是否需要会话记忆、审计、可重放和调试追踪

如果这些约束不先说清，后面的设计很容易漂。

### 2. 微服务边界

我通常会拆成下面几层：

#### 入口层

- API Gateway：鉴权、配额、审计、租户限流、SSE 连接管理

#### 编排层

- Agent Orchestrator：负责意图识别、计划生成、工作流推进、状态机管理

#### 路由层

- Model Router：负责不同 LLM 的路由、重试、fallback、成本和 SLA 治理

#### 工具层

- Tool Adapter：统一封装向量检索、数据库查询、搜索、内部结算系统等工具

#### 状态层

- Session/Memory Service：保存会话状态、摘要、长期记忆、工具执行历史

#### 异步层

- Async Queue + Worker：处理超时任务、重试、补偿、离线总结、异步评测

### 3. 核心调用链路

一个比较稳的主链路是：

1. 请求进入网关，完成鉴权、限流、请求 ID 注入
2. 编排服务创建根 `context`，挂上总超时、租户预算和 trace
3. 意图识别与计划器生成执行图
4. 调度器按 DAG 或状态图并发触发可执行节点
5. 工具结果统一回写到运行态上下文
6. LLM 根据中间结果继续规划或生成最终回答
7. 输出通过 SSE 分段返回，包括中间状态、工具进度和最终答案

### 4. Go 层面的调度与生命周期管理

这是这个题的核心。

#### 1. 全链路 `context` 级联管控

原则是绝不裸起 goroutine。每个外部工具调用、模型请求、流式推送协程，都必须绑定派生自顶层请求的 `context.Context`。

设计重点：

- 顶层请求进入时创建根 `context`
- 每个节点执行时派生子 `context`
- SSE 连接断开、全局超时触发、用户主动取消时统一 `cancel`
- 所有阻塞点都必须 `select` 监听 `ctx.Done()`

这样做的价值不是“代码更规范”，而是：

- 客户端一断开，底层挂起请求能立刻感知并退出
- 达到全局 P99 超时后，整条执行图能安全收敛
- 从根上避免 goroutine 泄漏、悬挂重试和内存暴涨

#### 2. 不做无界 goroutine 扩张

我不会让每个工具调用都无限制起 goroutine，而是：

- 按请求内最大并发数限制
- 按租户限制
- 按工具类型限制
- 按外部依赖实例做舱壁隔离

这样单个热点租户或单个慢工具不会把整机 goroutine、连接和内存拖满。

#### 3. 舱壁隔离与独立 worker pool

如果某个外部大模型接口突然长时间卡死，最危险的不是单个请求超时，而是整个服务被它拖进雪崩。

所以我会按下游依赖拆独立并发池，例如：

- LLM 调用池
- 向量检索池
- 内部数据库查询池
- 核心交易或清结算调用池

实现上可以是：

- 信号量 + goroutine
- 带缓冲 channel 控制并发
- 或独立 worker pool

关键不是用什么库，而是保证慢依赖只能耗尽自己的池子，不能侵占全局调度资源。这就是典型的 Bulkhead Pattern。

#### 4. 所有子任务都挂根 context

设计原则：

- 请求超时，所有子工具立刻取消
- 客户端断开 SSE，编排服务感知后停止无意义计算
- 某一步已经判定不可继续，就级联取消后续分支

如果没有这套取消树，最容易出现的就是“用户早就走了，机器还在拼命跑工具调用”。

#### 5. 结果汇聚要可中断、可降级

外部 LLM 或工具突然卡死时，我会这样处理：

- 每个工具单独超时
- 给重要工具和非重要工具分级
- 慢工具触发熔断后，降级成默认回答、缺省结果或异步补全
- 对需要 quorum 的结果设定最小成功数，不要求所有分支都结束才继续

#### 6. 流式通道与 channel 安全关闭

流式响应场景里，LLM 节点和 HTTP 层之间通常会通过 channel 传递 token 块或中间事件。

这里要注意两个点：

- 发送方和关闭方职责必须唯一，避免多处关闭 channel 导致 panic
- 接收方必须同时监听数据 channel 和 `ctx.Done()`

安全模式一般是：

- 节点侧只负责发送
- 聚合器或生产者侧负责在完成时关闭 channel
- HTTP 层 `select` 监听 token、错误和取消信号

这样客户端断开时，不会继续向已关闭 channel 写数据，也不会把响应协程吊死。

#### 7. 资源隔离

至少做三层隔离：

- CPU/内存层：不同服务、不同 Pod 资源隔离
- 连接层：不同外部依赖独立连接池
- 调度层：不同工具类型独立 worker pool 或信号量

### 5. 编排抽象：DAG + 状态机，而不是写死流程

面对未来的循环判断、多 Agent 协作、人工确认节点，我更倾向于：

- 用状态机描述运行阶段
- 用 DAG 或节点图描述依赖关系
- 用统一节点接口封装执行、重试、超时、回滚和观测

节点抽象至少包含：

- 输入上下文
- 执行函数
- 超时与重试策略
- 成功条件
- 失败处理策略
- 输出写回规则

这样后续新增工具、插入审批节点、增加循环判断时，不需要把主流程改成一团 `if else`。

### 6. 全局状态：统一 `State` 对象做不可变传递

复杂 Agent 流最容易失控的点，不是节点多，而是多个节点并发修改共享上下文。

所以我更倾向于把节点协议设计成：

- 节点接收当前 `State`
- 节点返回新的 `State`
- 状态写入集中在调度器或状态合并器中完成

`State` 可以包含：

- 会话历史
- 中间推理结果
- 工具调用记录
- 当前执行阶段
- 幂等键、trace 信息和审计字段

这样做有三个好处：

- 避免多 goroutine 直接读写全局变量造成 data race
- 方便回放、调试和断点恢复
- 为后续持久化状态机和异步续跑打基础

### 7. 确定性系统与非确定性系统的边界处理

这是这道题的深水区。

LLM 输出天然是概率性的，可能产生幻觉、重复调用工具，甚至构造出非法参数。但内部结算、扣费、订单这些核心系统要求的是强确定性。

所以在 LLM 节点和核心业务节点之间，必须有一层防腐层：

- 工具 schema 严格约束输入参数
- 所有高风险工具都做参数校验和权限校验
- 业务接口实现幂等，不相信模型只会调用一次
- 对非法、越权、重复调用直接拒绝，不把风险传给底层核心系统

如果涉及资金操作，我会明确：

- Agent 不直接决定交易结果
- Agent 只能发起受控指令
- 底层交易系统仍由状态机、幂等、对账和补偿负责兜底

### 8. 数据与一致性

这类系统通常要分清三类数据：

- 请求运行态：短期存在，适合放内存 + Redis
- 会话记忆：适合放数据库或专用记忆存储
- 审计和回放：适合事件流或持久日志

如果工具链里接入清结算系统、下单系统这类核心业务能力，我会明确：

- Agent 本身不直接承担交易一致性
- 交易链路仍走原有状态机、幂等和对账体系
- Agent 只是调用者，不应该破坏底层交易系统边界

### 9. 流式返回设计

SSE 事件建议拆成几类：

- `planning`：计划已生成
- `tool_start`：某工具开始调用
- `tool_result`：某工具完成
- `token`：模型流式输出 token
- `final`：最终答案
- `error`：可恢复或不可恢复错误

这比只吐文本更容易做前端交互、问题定位和链路回放。

### 10. 观测与运维

这个题如果想答完整，最后一定要收在观测和故障处理：

- Trace 关联用户请求、编排节点、工具调用、模型调用
- Metrics 看成功率、P95、工具超时率、取消率、队列长度、SSE 断流率
- Log 保留结构化字段，例如租户、会话、节点类型、工具名、错误码
- 对慢模型、慢工具、积压队列设专门告警

图例参考：观测不是三套系统各摆一份大盘。最好用同一个 `trace_id` 把 Metrics、Logs、Trace 串成证据链，先发现异常，再定位阶段，最后回到根因和修复动作。

### 11. 一段适合口述的总结版答案

如果面试现场时间不多，我会压缩成下面这段：

> 我会把它设计成“入口网关 + 编排服务 + 模型路由 + 工具适配层 + 会话记忆 + 异步任务”的分层架构。编排层内部不用写死流程，而是用 DAG 配合状态机推进节点执行。Go 层面重点做三件事：第一，用根 `context` 管所有子任务，保证超时、断连、失败时能级联取消；第二，用分层限流、信号量和独立 worker pool 做背压与资源隔离，避免慢工具拖垮整机；第三，把流式输出、重试、熔断、降级和观测做成统一框架能力。这样它既能支持当前多工具并发调用，也能在后续扩展到循环判断、多 Agent 协作和异步补偿。
