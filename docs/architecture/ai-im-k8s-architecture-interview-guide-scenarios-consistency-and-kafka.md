## 场景题二：百万并发下的存储高可用与资金强一致性

问题背景：

> 在高并发场景下，如直播间百万在线、头部主播热点，如何处理 Redis 热点 Key 导致的集群倾斜？同时，在涉及核心资金的交互，如打赏扣费、清结算中，如何保证分布式系统下 MySQL 的数据一致性？

这道题实际上考两件完全不同的事：

- Redis 在极端热点流量下如何不被单点打穿
- MySQL 在资金链路里如何在高并发下既保住正确性，又不把吞吐打崩

答题时最好分“扛热点”和“守资金”两部分说。

### 1. 打透热点流量：多级缓存而不是只会说 Redis Cluster

单靠 Redis Cluster 解决不了极端热点。因为头部主播、全站公告、热门房间状态，最终仍可能打到同一个热点 key 或同一个热点槽位。

更稳的方案是 L1/L2 多级缓存：

- L1：Go 进程本地缓存，例如 BigCache、FreeCache
- L2：Redis 作为共享缓存和分布式协调层
- L3：MySQL 或其他持久化存储作为最终数据源

这样做的价值是：

- 极热读请求直接在应用层命中
- 降低网络 IO 和 Redis 单节点 CPU 压力
- 避免热点流量全部穿透到底层集群

### 2. 热点探测与动态降级

只做缓存还不够，还要主动发现热点。

比较实用的策略是：

- 在网关、代理层或业务层做滑动窗口统计
- 发现热点 key 后，主动把热点数据广播到各业务节点的本地缓存
- 对非核心查询链路做柔性降级，例如延迟刷新、返回近实时快照、关闭高频装饰性接口

这里的思路不是“所有请求都必须实时命中最新值”，而是先保证系统不被热点拖死，再按业务等级恢复精度。

### 3. 房间广播和热点房间数据要拆开设计

直播场景里很容易把两类问题混在一起：

- 全站广播、进场动效、房间消息分发
- 房间基础状态、榜单、在线人数、礼物聚合等热点数据读取

前者更适合消息总线和网关 fan-out，后者更适合多级缓存、预聚合和异步刷新。

如果把广播也压到 Redis 热点 key 上，本质上是在用缓存做消息系统，扩展性会很差。

### 4. 资金绝对安全：避免应用层查后改

涉及打赏扣费、账户余额、清结算时，我会先明确一个原则：

- 绝不在应用层做非原子的 `read-modify-write`

更稳的扣减方式是：

- 乐观锁，基于 `version` 字段控制并发更新
- 或直接在 SQL 里写条件更新，例如 `balance >= amount`

典型模式：

```sql
UPDATE account
SET balance = balance - 100
WHERE id = 1
  AND balance >= 100;
```

这类写法把并发竞争收敛到数据库原子更新里，比应用层先查余额再扣减安全得多。

### 5. 分布式事务不要默认上 2PC

订单服务、账户服务、清结算服务分开后，最容易犯的错是同步串行调用一路锁到底。

这会带来两个问题：

- 长时间持有数据库行锁
- 下游稍慢就把上游并发能力拖崩

所以我一般会这样选：

- 单库单服务：优先本地事务
- 强业务一致场景：TCC 或显式状态机
- 允许最终一致场景：本地消息表 + MQ + 补偿

### 6. 本地消息表保证最终一致

资金链路里，如果跨服务同步事务代价太高，我更偏向可靠消息最终一致性方案。

图例参考：Outbox 的关键不是“把消息也存下来”，而是“业务写库和消息落 outbox 在同一事务里”，这样才能先兜住不丢，再接受后续可能重复。

<GoMicroserviceDiagram kind="outbox-pattern" />

核心流程是：

1. 在本地 MySQL 事务里同时完成账户变更和 outbox 消息写入
2. 后台任务或 binlog 订阅组件把待发送消息投递到 Kafka
3. 下游消费成功后更新消息状态
4. 失败时依赖重试、补偿和对账机制收敛

这样做的优点是：

- 不长时间占住跨服务事务
- 吞吐量明显好于同步强一致链路
- 资金结果仍然可以通过幂等、状态机、对账做最终闭环

### 7. 一段适合口述的总结版答案

> 这类题我会拆成两层来看。对 Redis 热点问题，核心不是单纯扩 Redis，而是用本地缓存加 Redis 的多级缓存体系，再配合热点探测、预热和动态降级，把极热流量挡在应用层。对资金链路，核心不是追求链路上每一步同步完成，而是先用数据库原子更新、幂等和状态机保证扣费正确，再用本地消息表和 Kafka 实现跨服务最终一致。这样既扛得住百万并发，又不会为了吞吐牺牲资金正确性。 

## 面试核心场景题三：海量数据削峰与严格防乱序（Kafka 深度实战）

问题背景：

> 系统使用 Kafka 处理海量异步任务。如果下游 AI 算力节点消费极慢导致千万级消息积压，如何排查恢复？在特定业务下，如何保证消息的严格顺序且不丢失？

这道题的关键不只是“会不会调 Kafka 参数”，而是你能不能讲出事故处理顺序和架构取舍。

### 1. 先止血，再排查，再提速

发现千万级积压后，不要直接开始调参数，先做止血动作：

- 判断是不是下游 AI 节点、向量检索或外部依赖大面积故障
- 对非核心生产流量限流或降级
- 必要时暂停部分低优先级 topic 或消费链路

目标是先阻止 lag 继续失控，而不是一边灌水一边救火。

### 2. 排查要按链路顺序来

我一般按这条链路排：

- Producer 是否突然放量
- Broker CPU、磁盘、网络、页缓存是否异常
- 分区是否倾斜，lag 是否集中在少数 partition
- Consumer group 是否频繁 rebalance
- 单条消息处理耗时是否异常
- 下游 AI 节点是否成为慢依赖

这样排的原因是，积压只是表象，根因通常在某个阶段吞吐突然跌破生产速率。

### 3. 拉、处分离，榨干单机消费能力

如果下游 AI 处理很慢，消费者不要把“从 Kafka 拉消息”和“业务处理”绑死在同一个阻塞循环里。

更实用的做法是拉、处分离：

- 主 goroutine 专门快速 fetch 消息
- 拉到消息后按业务 key 分发到内部队列
- 多个 worker goroutine 异步执行具体处理

如果追求极致吞吐，可以进一步用 ring buffer 或无锁队列承接内部分发。

这样做的价值是：

- Kafka 拉取侧保持高吞吐
- 业务处理侧可以独立扩展并发
- 单机 CPU 和网络资源利用率更高

### 4. 严格顺序必须同时在 Kafka 层和进程内层成立

很多人只说“同一个 key 发到同一个 partition”，这只解决了 Kafka 层的顺序。

如果消费端拿到消息后又随意分发给不同 worker，同样会乱序。

所以严格顺序至少要满足：

- 同一个业务 key 固定到同一个 Kafka partition
- 消费端再基于同样的 key 做一致性 hash
- 同一个 key 永远落到同一个内部 worker 串行处理

这时即便你在单机内部做了并发模型，也不会打乱单 key 的事件顺序。

### 5. 防丢失是全链路问题

严格不丢失不能只看消费者。

比较标准的防线是：

- Producer：`acks=all`、重试开启、幂等生产
- Broker：副本数至少 3，`min.insync.replicas=2`
- Consumer：处理成功再提交 offset，业务侧实现幂等

如果业务特别关键，还要加：

- 死信队列
- 重试主题
- 消费失败审计
- 人工补偿和回放机制

图例参考：Kafka 里“丢失”和“重复”经常被混讲。把这两个窗口拆开之后，为什么要业务幂等、为什么要成功后再提交 offset，就都容易讲清。

### 6. 一段适合口述的总结版答案

> Kafka 积压我会按“先止血、再排查、再提速”处理。先限制非核心生产流量，避免 lag 继续膨胀；再按 producer、broker、partition、consumer、下游依赖这条链路定位瓶颈；最后在消费端做拉、处分离，把 fetch 和业务处理解耦。如果业务要求严格顺序，我会让同一个 key 固定到同一个 partition，并在消费进程内部再次基于同 key 做一致性 hash，保证永远由同一个 worker 串行处理。防丢失则靠 producer、broker、consumer 三端一起兜住，而不是只靠 Kafka 本身。
