## 软实力与项目复盘

这部分很多人准备不足，但在高级岗位里经常决定最终评级。

### 1. 项目复盘

建议至少准备 1 到 2 个项目，每个都按 STAR 结构讲：

- 背景：业务规模、性能瓶颈、故障场景、团队约束
- 任务：你负责什么，关键目标是什么
- 行动：你做了哪些架构、治理、排障或优化动作
- 结果：指标提升多少，故障恢复多久，沉淀了什么机制

最好优先准备下面几类故事：

- 一次容量瓶颈优化
- 一次线上事故止血与复盘
- 一次核心链路重构
- 一次跨团队推动的平台化治理

图例参考：项目复盘最怕讲成流水账。尤其是线上事故，最好按“发现 -> 止血 -> 定位 -> 修复 -> 验证 -> 复盘”的顺序讲，面试官更容易判断你是不是有真实一线经验。

### 2. 质量管理

可以围绕这几个点展开：

- 单元测试不是覆盖率数字游戏，重点是覆盖核心状态流转和边界条件
- Code Review 的价值是防止架构腐化和隐患进入主干，而不是只看代码风格
- 设计模式不是背概念，而是为了控制复杂度，比如状态机、策略模式、责任链、模板方法

## 高频追问题库

下面这些题不是让你逐字背，而是让你练“结论 -> 原理 -> 方案 -> 风险”的表达。

### Q1：并发调用多个外部大模型 API 时，Go 的 GMP 如何工作？如何做背压和限流？

回答骨架：

- 大量 goroutine 等待外部 API 时，主要是 IO 阻塞，不会等价于创建大量 OS 线程
- 运行时会通过网络 poller 和调度器管理阻塞 goroutine，但 goroutine 数量失控仍会推高内存、定时器、上下文和连接资源成本
- 我会把请求处理拆成入口限流、租户限流、模型限流、并发 worker 池和下游超时控制五层
- 所有子调用挂在同一个根 `context` 上，超时、用户取消、上游断链时统一级联取消
- 如果某个模型突然变慢，先快速熔断和降级，再切备用模型或异步化，而不是把请求堆死在内存里

面试官重点看：

- 你是否理解 syscall/网络阻塞对调度器的影响
- 你是否会把限流、背压和取消机制落到代码结构上

图例参考：这题最容易漏掉的是“子调用不是各跑各的”，而是挂在同一棵取消树上统一收束。先把根 `context` 到子任务的传播关系看清，再往上叠信号量、worker pool 和 fallback。

<GoChannelDiagram kind="context-tree" />

::: details 完整参考答案

> 并发调用多个大模型 API 时，goroutine 大多在等网络 IO，运行时通过 netpoller 把它们挂起，不会占用 P，所以活跃线程数仍然只有 GOMAXPROCS 个，不会像 Java 线程模型那样快速耗尽线程池。
>
> 但 goroutine 数量失控仍然有问题：每个挂起的 goroutine 占几 KB 栈内存，加上 HTTP 连接缓冲区、请求上下文、超时定时器，十万级 goroutine 可能消耗几 GB 内存。所以我不会无限制开 goroutine，而是做分层限流。
>
> 第一层是入口限流，在 API Gateway 按 QPS 或并发数控制总入口。第二层是租户限流，防止单个大租户挤占所有资源。第三层是模型维度限流，不同模型有不同的 QPS 配额和 token 预算。第四层是 worker pool，用信号量或 buffered channel 控制同时并发的外部调用数。第五层是每个调用的超时控制，挂在请求级 context 上。
>
> 所有子调用共享同一个根 context，客户端断连或总超时触发时，所有子调用通过 ctx.Done() 立刻感知并退出。如果某个模型突然变慢，熔断器触发后自动切到 fallback 模型或返回降级结果，而不是让请求堆积在内存里拖垮整个服务。

:::

### Q2：你做过最深的一次 Go 性能优化是什么？

回答骨架：

- 先说场景和指标异常，例如 CPU 飙升、P99 上升、内存泄漏、GC 次数异常
- 再说排查手段，例如 `pprof`、`trace`、运行时指标、火焰图、对象分配热点
- 再说优化动作，例如减少逃逸、复用对象、分离热点锁、缩短大对象生命周期、调 `GOGC` 或 `GOMEMLIMIT`
- 最后讲结果和副作用，例如吞吐提升、GC 暂停下降，但代码复杂度是否增加

图例参考：性能优化如果说不清“对象为什么上堆、上堆之后为什么 GC 会更重”，面试回答会很虚。先把栈和堆的成本差异看清，再解释逃逸、对象复用和生命周期收缩。

<GoRuntimeDiagram kind="stack-vs-heap" />

::: details 完整参考答案

> 我做过一次 AI 推理网关的性能优化。当时现象是高峰期 P99 从 200ms 飙到 2 秒，同时 GC 暂停从 0.5ms 涨到 5ms，堆内存持续增长。
>
> 排查过程：先用 `pprof` 看 CPU 火焰图，发现大量时间花在 JSON 序列化和 HTTP 响应体拷贝上。再看 heap profile，发现每个请求都在堆上分配一个大的 response buffer（模型返回的 token 流拼接），而且用了 `[]byte` 到 `string` 的频繁转换导致逃逸。另外日志库的 `fmt.Sprintf` 也贡献了大量小对象分配。
>
> 优化动作：第一，用 `sync.Pool` 复用 response buffer，避免每次请求都在堆上新分配；第二，把关键路径的 JSON 序列化换成代码生成的版本（如 easyjson），减少反射开销；第三，用 `go build -gcflags=-m` 检查逃逸点，修复了几处不必要的指针传递；第四，把 GOGC 从默认 100 调到 200，配合 GOMEMLIMIT 设为物理内存的 80%，降低 GC 频率但不允许 OOM。
>
> 结果：P99 从 2 秒降回 300ms，GC 暂停回到 1ms 以内，堆分配量下降约 60%。副作用是 sync.Pool 的使用增加了代码复杂度，需要注意归还时清空 buffer 状态。

:::

### Q3：直播平台做全站广播和大主播房间热点数据，Redis 热点 key 怎么彻底治理？

回答骨架：

- 先区分“广播链路”和“热点房间状态”是两类问题
- 广播更适合消息总线 + 网关分发，不要让所有节点反复从 Redis 拉同一个 key
- 热点房间数据做多级缓存，本地缓存承接绝大多数读流量，Redis 只做共享状态和失效协调
- 对超热点 key 做拆分、预聚合、分区路由或旁路静态化，不要把全部读压在单个 Redis 槽位上
- 必要时把”高频查询状态”和”低频准确数据”分层存储

图例参考：这题其实在考两条链路。广播链路要看“消息怎么扇出到网关”，而不是盯着 Redis 轮询；先把这个推送路径看清，回答就不容易混成缓存题。

<GoDataCacheDiagram kind="redis-pubsub" />

图例参考：热点 key 真正的治理重点是拆热点、分路由、分散单槽位压力，而不是简单把 Redis 实例做大。先把分片路由看清，再讲本地缓存、预聚合和动态降级。

<GoDataCacheDiagram kind="sharding-route" />

::: details 完整参考答案

> 我会先把”广播链路”和”热点房间状态”拆开来处理，因为它们的访问模式完全不同。
>
> 广播链路（进场动效、全站公告、系统消息）本质是一对多推送。我不会让所有节点都去 Redis 轮询同一个 key，而是通过 Kafka 或 Redis Pub/Sub 把广播消息推到每个网关节点，网关节点维护本地的连接列表直接下推。这样广播的扇出发生在消息层而不是缓存层。
>
> 热点房间数据（在线人数、榜单、礼物统计）我用多级缓存：L1 是 Go 进程本地缓存（BigCache / 自建 LRU），TTL 设 1-3 秒，承接 90%+ 的读请求；L2 是 Redis，作为共享缓存和失效协调。本地缓存过期后从 Redis 刷新，Redis 过期后从 DB 重建。对超热 key，还会做 key 分片：把 `room:12345:stats` 拆成 `room:12345:stats:0` 到 `room:12345:stats:3`，读取时随机选一个分片，写入时全部更新。这样单个 Redis 槽位的读压力降为 1/4。
>
> 动态降级方面：监控 Redis 节点 CPU 和慢查询，如果某个热点 key 被探测到（滑动窗口内 QPS 超阈值），自动把该 key 的数据广播到所有业务节点的本地缓存，直到热度下降。非核心数据（装饰性排名、动效配置）在极端情况下降级为返回上一个缓存快照。

:::

### Q4：打赏扣费这类交易链路，怎么保证 MySQL 强一致？节点故障时怎么补偿？

回答骨架：

- 单服务单库内先用本地事务解决，不要一上来就分布式事务
- 跨服务时按业务等级选 TCC、Outbox+MQ、Saga，而不是统一方案
- 资金扣减要有幂等键、账户流水、状态机和对账
- 网络分区或节点宕机时，系统目标不是”永远不出错”，而是”最终可收敛、可补偿、可追账”

图例参考：资金题里最容易漏的是“先防重，再谈一致性”。把幂等键和唯一约束放到入口层看清，后面再讲本地事务、流水和补偿才不会乱。

<GoMicroserviceDiagram kind="idempotency-key" />

图例参考：跨服务失败后的关键不是“某一步失败了”，而是“后续怎么反向收敛”。先把补偿链路看清，再区分什么时候适合 Saga，什么时候更适合 Outbox 或 TCC。

<GoMicroserviceDiagram kind="saga-compensation" />

::: details 完整参考答案

> 打赏扣费我会分三层设计。
>
> 第一层是单服务内的原子扣减。不在应用层做 read-modify-write，直接用 SQL 条件更新：`UPDATE account SET balance = balance - 100 WHERE id = 1 AND balance >= 100`。影响行数为 0 就是余额不足，不需要先查再扣。每次扣费操作带幂等键（订单号 + 操作类型），写入流水表做去重。
>
> 第二层是跨服务一致性。打赏涉及扣费服务、礼物服务、直播间服务。我不用 2PC，因为长时间持锁在高并发下吞吐崩塌。核心链路用本地消息表（Outbox）：在扣费事务内同时写入 outbox 消息，后台异步投递到 Kafka，下游服务消费并确认。如果业务要求更强一致（比如扣费成功但礼物必须立刻展示），用 TCC：try 阶段冻结余额 + 预创建礼物记录，confirm 阶段执行扣减 + 确认礼物，cancel 阶段回滚冻结。
>
> 第三层是兜底机制。每个账户都有独立的流水表，资金变动全部有据可查。T+1 对账任务对比各服务的流水，发现不一致自动生成补偿工单。节点宕机时，未完成的事务通过状态机恢复：pending 状态的操作由定时任务补偿推进或回滚。系统目标不是”永远不出错”，而是”出错时能发现、能收敛、能追账”。

:::

### Q5：Kafka 出现千万级积压，如何排查与处理？如何同时保证顺序和不丢失？

回答骨架：

- 先看 lag 是否集中在少数分区，再看 Broker 资源、消费者耗时和下游依赖
- 对慢消息分类，区分是单条消息过大、外部依赖变慢，还是消费者逻辑阻塞
- 临时止血可以扩消费者、提分区、隔离慢主题、降级非关键消费
- 真正要求顺序时按业务 key 固定分区，消费端串行处理，并接受吞吐受限
- “不丢失”依赖生产幂等、副本保障、消费幂等、延迟提交和失败重试一起成立

图例参考：顺序这个词一定要落到“同 key 进同 partition”上，否则很容易只会空谈“Kafka 支持顺序”。先把路由规则看清，再讲消费端为什么还要同 key 串行 worker。

::: details 完整参考答案

> 发现千万级积压，我按”止血 → 定位 → 提速”三步走。
>
> **止血**：先判断是不是下游整体故障（AI 算力节点挂了）。如果是，立刻限制生产端非核心流量的投递，暂停低优先级 topic 的消费，避免 lag 继续失控。目标是阻止情况恶化。
>
> **定位**：按链路排查。先看 Producer 是否突然放量（上游活动、重试风暴）。再看 Broker 的磁盘 IO、网络、页缓存命中率。再看 lag 是分散还是集中在少数 partition（如果集中说明分区倾斜或热点 key）。再看 Consumer 是否频繁 rebalance（心跳超时、消费时间过长触发 session timeout）。最后看单条消息处理耗时，通常瓶颈在 AI 推理或下游 RPC。
>
> **提速**：消费端做拉、处分离。主 goroutine 专门快速 fetch，拉到消息后按业务 key hash 分发到内部 channel，多个 worker goroutine 异步处理。这样 Kafka fetch 不被业务处理阻塞。如果可以接受临时降级，对非关键消息走快速通道（跳过耗时的 AI 推理，标记为待补处理）。如果分区数本身是瓶颈，扩分区 + 扩消费者。
>
> **保顺序**：同一个业务 key 通过 partition key 固定到同一 partition，消费端内部也用同样的 key 做一致性 hash，保证同 key 永远落到同一个 worker 串行处理。这样即使内部并发处理，单 key 的事件顺序也不会乱。代价是：热点 key 的吞吐上限受限于单 worker。
>
> **防丢失**：三端一起做。Producer 开 `acks=all` + 幂等生产。Broker 副本数 ≥ 3，`min.insync.replicas=2`。Consumer 处理成功后才手动提交 offset，失败的消息进死信队列，定时重试或人工处理。

:::

### Q6：从零设计一个支持多模型、上下文记忆和工具调用的 AI 服务，微服务边界怎么划？

回答骨架：

- API Gateway 负责鉴权、限流、租户隔离和请求入口
- Orchestrator 负责意图识别、工作流推进和工具调度
- Model Router 负责模型路由、配额、fallback 和成本治理
- Tool Service 负责向量检索、数据库查询、内部业务能力调用
- Session/Memory Service 负责上下文、会话摘要和长期记忆
- Async Worker 负责长任务、批处理和补偿

图例参考：这题本质是在考“边界怎么划、治理怎么挂上去”。先把服务层次总览看清，再往下讲 Gateway、Orchestrator、Router、Tool、Memory 各自的职责。

<GoMicroserviceDiagram kind="overview" />

图例参考：AI 服务边界里最容易讲虚的是 Model Router。把失败降级和 fallback 路径看成一张图，为什么要独立出 Router 这一层就更容易说清。

<GoMicroserviceDiagram kind="degradation-fallback" />

::: details 完整参考答案

> 我会拆成六个服务。
>
> **API Gateway**：统一入口，做鉴权、租户识别、限流、审计日志。SSE 长连接也在这一层管理，Gateway 持有客户端连接，编排层通过内部 channel 或 gRPC stream 推 token 过来。
>
> **Agent Orchestrator（编排服务）**：核心服务。接收用户输入后，先做意图识别，然后生成执行计划（DAG 或状态机）。按计划调度工具调用和模型推理，管理整个请求的生命周期。关键能力：并发调度、超时控制、失败重试、降级和级联取消。
>
> **Model Router（模型路由）**：封装不同 LLM 提供商的调用差异（OpenAI / Claude / 私有模型），做模型选择、配额管理、成本统计、失败 fallback。比如主模型超时，自动降级到更快但精度略低的模型。
>
> **Tool Adapter（工具适配层）**：统一封装所有外部工具——向量检索、数据库查询、Web 搜索、内部业务 API。每个工具实现统一接口（输入 schema、输出 schema、超时、重试策略）。这一层是确定性系统和非确定性 LLM 之间的防腐层，负责参数校验、权限控制和幂等保障。
>
> **Session / Memory Service**：管理会话上下文、对话历史摘要、长期记忆。短期记忆放 Redis，长期记忆持久化到数据库。支持上下文窗口管理（超过 token 限制时自动摘要压缩）。
>
> **Async Worker**：处理超时任务、批量推理、离线评测、失败补偿。通过 Kafka 解耦。前台返回任务 ID，后台推进状态机，完成后通过 WebSocket/SSE/回调通知。

### Q7：K8s 里 AI 服务怎么做资源限制和弹性伸缩？动态 IP 下白名单怎么解决？

回答骨架：

- 在线推理服务要按 CPU、内存、并发数、队列长度联合看，不要只盯 CPU
- `requests` 保障调度，`limits` 防止单 Pod 抢占，但 limit 过紧会导致频繁 OOM 或节流
- HPA 指标优先选真实业务指标，例如活动请求数、排队长度、P95
- 第三方白名单问题通常要解决固定出口 IP，而不是试图固定 Pod IP

> AI 服务的资源管理和普通 Web 服务不一样，核心差异在负载特征。
>
> **资源配置**：AI 推理服务的内存占用高且稳定（模型常驻内存），CPU 使用极端波动（空闲时接近 0，推理时吃满）。所以我会把内存 requests 设成 = limits = 模型大小 + 20% buffer，防止 OOM；CPU 用 burstable 策略，requests 设为平均用量，limits 适当放宽或不设。GPU 节点用 taint + toleration 做专用节点池隔离，避免和普通服务互相干扰。
>
> **弹性伸缩**：HPA 不能只看 CPU。AI 服务更适合看业务指标：并发请求数、推理队列长度、P95 延迟。通过 Prometheus + KEDA 或自定义 metrics adapter 把业务指标暴露给 HPA。设置合理的 scale-up 速率（突发流量快速扩）和 scale-down 稳定窗口（避免抖动导致频繁缩容）。极端情况下用 VPA 自动调整单 Pod 的 requests/limits。
>
> **固定出口 IP**：Pod IP 是动态的，不可能给每个第三方都更新白名单。方案是在 VPC 内配置 NAT 网关，所有出集群流量统一走 NAT 网关的固定弹性 IP。如果只是部分服务需要固定出口 IP，用 Istio Egress Gateway 或 Calico 的 egress IP 策略做细粒度控制。这和"固定 Pod IP"或"固定 Service IP"是完全不同的事。
