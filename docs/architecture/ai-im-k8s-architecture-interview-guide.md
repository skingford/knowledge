## 这类岗位到底在筛什么

这类 JD 通常不是在招“会写 CRUD 的后端”，而是在筛下面四种能力能不能同时成立：

- 你是否理解基础设施的运行边界，而不只是会调用 API
- 你是否能把高并发、强一致、低延迟、长耗时 IO 这些冲突目标做出取舍
- 你是否能在 AI、直播、IM 这类复杂业务里把工程问题讲清楚
- 你是否具备线上稳定性负责人的意识，包括容量、降级、容灾、观测和复盘

一句话概括：面试官想确认你是“能独立扛一条核心链路”的人。

## 核心技术栈深度

这是入场券。这里如果讲不深，后面的 AI、直播、K8s 都很难拿到高评价。

### Go：调度、内存、并发控制

#### 1. 调度模型

必须能讲清：

- `G` 是 goroutine，`M` 是线程，`P` 是调度所需的运行上下文
- `P` 持有本地运行队列，调度器通过本地队列、全局队列和 work stealing 分配 `G`
- Goroutine 遇到阻塞 syscall 或网络 IO 时，运行时会尽量把 `P` 从阻塞线程上摘下来，交给其他 `M` 继续跑
- Go 1.14 之后抢占更完善，长时间不主动让出的 goroutine 也更容易被调度器打断

面试里不要只说 GMP 定义，要能继续往下说：

- 大量外部 API 调用为什么不会像线程模型那样很快把线程打满
- `syscall`、网络 poller、`GOMAXPROCS`、CPU 密集型任务分别怎么影响吞吐
- 如果 goroutine 数量飙升但大多在等待 IO，瓶颈往往不在 goroutine 本身，而在下游连接池、限流器、内存占用和超时治理

#### 2. 内存管理

必须能讲清：

- Go GC 的核心是并发三色标记清除
- 写屏障的作用是保证并发标记期间不漏标
- 逃逸分析决定对象留在栈上还是堆上，堆对象增多会直接增加 GC 压力
- GC 调优不是“调小暂停时间”这么简单，而是延迟、吞吐、内存占用三者平衡

建议能顺口说出这些实战点：

- 用 `go build -gcflags=-m` 看逃逸分析
- 用 `pprof` 看堆分配热点和对象生命周期
- 用 `GOGC` 控制 GC 触发频率，用 `GOMEMLIMIT` 控制进程内存上界
- 长耗时请求、超大响应体、无界缓存、goroutine 泄漏，经常会一起放大 GC 问题

#### 3. 并发控制

必须能讲清：

- `channel` 本质上是带锁的队列 + 发送/接收等待队列，不是“天然高性能”
- `context` 是一棵取消树，重点不是传参数，而是传播截止时间、取消信号和请求边界
- `sync.Mutex` 存在正常模式和饥饿模式，锁竞争严重时调度行为会发生变化

高分回答通常会带上这些判断：

- IO 聚合场景优先考虑有限并发，不要盲目“一请求一 goroutine 无脑开”
- `channel` 适合表达协作和背压，不适合拿来替代所有共享状态
- `atomic`、分片锁、本地缓存、无锁读多写少结构，往往比一把大锁更稳
- 优雅退出时必须有统一的 `context` 根节点，否则 worker、重试器、流式推送协程很容易泄漏

### MySQL 与 Redis：从原理到架构取舍

#### 1. MySQL

必须能讲清：

- B+ Tree 索引为什么适合范围查询和排序
- 聚簇索引、回表、覆盖索引、最左前缀分别意味着什么
- 事务隔离级别下 MVCC 如何工作，读已提交和可重复读差异在哪里
- 深分页为什么慢，以及游标翻页、延迟关联、覆盖索引这些常见优化思路

在交易或计费场景里，还要能继续展开：

- 单库事务能解决什么，跨服务后为什么要引入 TCC、Saga、Outbox
- `select ... for update`、间隙锁、死锁重试这些坑在高并发扣费里怎么处理
- 资金类场景优先用状态机 + 幂等 + 对账兜底，而不是盲目追求“链路上每一步都同步强一致”

#### 2. Redis

必须能讲清：

- 缓存穿透、击穿、雪崩的定义差异和治理组合拳
- RDB 和 AOF 的恢复速度、数据安全和性能取舍
- Redis Cluster 的槽位机制、主从切换和高可用边界
- 分布式锁的核心不是“能锁住”，而是“业务是否真的需要锁”和“锁失效时如何收敛”

高频追问里最容易拉开差距的是：

- 热点 key 怎么从架构层面处理，而不是只说“加 Redis”
- 多级缓存怎么做，本地缓存命中、失效广播和一致性怎么平衡
- Redlock 的争议点是什么，为什么很多核心业务更愿意用数据库约束、状态机或单线程串行化代替“看起来高级”的分布式锁

### Kafka：可靠性、积压与顺序

必须能讲清：

- Topic、Partition、Consumer Group、ISR 这些基本概念
- 为什么顺序性本质上只能在单分区或单业务 key 维度上保证
- 消息不丢失不是一句“Kafka 很可靠”，而是 Producer、Broker、Consumer 三端一起设计

高频回答骨架建议固定成这条链路：

#### 1. Producer 侧

- 开启 `acks=all`
- 合理设置重试
- 开启幂等生产
- 对关键链路控制超时、批量和压缩策略

#### 2. Broker 侧

- 副本数充足
- `min.insync.replicas` 配置合理
- 落盘和刷盘策略匹配业务等级
- 监控 ISR 缩减、磁盘打满、页缓存抖动和网络拥塞

#### 3. Consumer 侧

- 业务处理成功后再提交 offset
- 幂等消费，避免“至少一次”带来的重复副作用
- 消费能力不足时优先扩分区、扩消费者、拆慢消息、异步化下游

面试时如果只会背可靠性参数，往往不够。一定要能说出：

- 消息积压本质上是“生产速率超过消费速率”的系统容量问题
- 排查顺序一般是：Broker 资源、分区分布、消费者 lag、单条消息耗时、下游依赖、重平衡抖动
- 真正要求强顺序时，往往要接受吞吐下降，并用业务 key 分区、单线程消费或状态机串行化来换一致性

## 专项场景突破

这部分是 JD 里最容易做差异化的地方。你的回答必须从“会用组件”上升到“理解业务链路”。

### AI 工程化能力

#### 1. 支持流式输出的接口设计

必须能讲清：

- 为什么 AI 场景适合用 Streaming/SSE，而不是所有结果都等到最后一次性返回
- 服务端如何把模型 token、工具调用中间态、最终结论拆成不同事件
- 客户端断开、超时取消、幂等重试和落库策略怎么处理

常见接口分层：

- 同步请求接口：适合短请求、简单问答
- SSE 流式接口：适合对话、Agent 中间过程透传
- 异步任务接口：适合超长耗时、批处理、复杂工具链编排

#### 2. 多模型调度与异步任务

高分回答要覆盖：

- 按模型、租户、接口类型做限流，不要只做全局 QPS 限流
- 失败时优先做快速失败、降级和 fallback，不要无限重试
- 长耗时任务进入异步队列，前台返回任务 ID，后台推进状态机
- 推理节点池要和普通 API 节点隔离，避免彼此争抢 CPU、内存和连接资源

#### 3. 数据处理

常见考点：

- 数据清洗、切分、去重、脱敏、标签化
- 向量存储的基本概念，例如 embedding、索引构建、召回、重排
- RAG 和 Agent 不是一回事：RAG 解决检索增强，Agent 解决规划与工具编排

### IM/直播高并发架构

#### 1. 长连接管理

必须能讲清：

- 长连接网关和业务服务尽量拆开
- 连接状态、会话状态、用户在线状态不要混成一层
- 心跳、超时、断线重连、连接迁移和多端登录策略要提前定义

百万连接场景常见做法：

- 网关无状态化，用户连接通过一致性 hash 或会话路由分散到多个节点
- 用户在线状态放到独立存储层或状态服务，不直接绑在单机内存里
- 广播、房间消息、系统通知通过消息总线分发，不走点对点同步调用

#### 2. 消息即时性

必须能讲清：

- 写扩散适合群规模较小、读取极高频、要求极致读性能的场景
- 读扩散适合大群、超大房间、在线状态强波动场景
- IM 和直播房间消息并不一定用同一套投递模型

一个比较稳的表达是：

- 私聊和小群偏写扩散
- 大群、直播公屏、系统广播偏读扩散或在线用户实时 fan-out
- 明确“离线存储”和“在线推送”是两条链路，不要混成一个组件

#### 3. 稳定性治理

必须能讲清：

- 限流放在入口、房间、用户、下游多个层次
- 熔断的重点不是报错，而是阻止慢依赖把线程池、连接池和 goroutine 拖死
- 降级要提前定义，例如关闭进场动效、降低消息刷新频率、只保留核心互动链路

## 架构与云原生

这部分考的是你能不能既看得懂系统边界，又能把服务落到可运维的生产环境里。

### 分布式架构

#### 1. 微服务治理

需要能讲清这些能力的边界：

- 服务发现解决的是实例动态变化，不解决熔断和重试策略
- 配置中心解决的是运行时配置收敛，不解决发布流程混乱
- 链路追踪解决的是跨服务定位问题，不等于完整可观测性

回答时建议按这条线说：

- 服务边界如何拆
- 同步链路和异步链路如何组合
- 配置、注册发现、鉴权、灰度、观测如何接入
- 哪些公共能力沉到平台层，哪些保留在业务服务内

#### 2. 高可用设计

常见考点：

- 单机故障、机房故障、跨地域故障分别怎么处理
- 数据库主从切换的 RPO/RTO 如何权衡
- 缓存预热、降级开关、只读模式、限流阈值如何提前准备

高分回答一般会强调：

- 高可用不是“做主从”就结束，而是故障发生时系统如何有序退化
- 真正重要的是演练、切换脚本、告警门槛和回滚路径

### Kubernetes 与交付

#### 1. Pod 生命周期和资源管理

必须能讲清：

- Pod 从调度、拉镜像、启动、探针通过到终止的关键阶段
- `requests` 决定调度保底，`limits` 决定可用上界
- AI 服务不能简单照搬普通 Web 服务的资源配置，因为推理负载波动和内存占用波动都更明显

比较稳的回答方式：

- 对在线推理服务，CPU/内存 `requests` 不要压得太低，否则节点会被过度装箱
- 对明显 CPU 密集或 GPU 密集型工作负载，优先做专用节点池和资源隔离
- HPA 不能只看 CPU，很多 AI 服务更应该看并发数、队列长度、P95 延迟或自定义业务指标

#### 2. Service 与流量入口

要能讲清：

- 集群内稳定访问依赖 `Service`
- 对外流量一般经过 `LoadBalancer`、Ingress、Gateway
- “固定出口 IP”“第三方白名单”“Pod 动态 IP”是三件不同的事

这是非常高频的追问点，尤其是远程岗位和云原生岗位。

#### 3. CI/CD

建议固定成一条流水线：

1. 代码提交后触发测试、构建、镜像扫描
2. 产出版本化 Docker 镜像
3. 通过 Helm、Kustomize 或 GitOps 系统发布到 K8s
4. 用探针、灰度和回滚机制保证上线风险可控

## 软实力与项目复盘

这部分很多人准备不足，但在高级岗位里经常决定最终评级。

### 1. 项目复盘

建议至少准备 1 到 2 个项目，每个都按 STAR 结构讲：

- 背景：业务规模、性能瓶颈、故障场景、团队约束
- 任务：你负责什么，关键目标是什么
- 行动：你做了哪些架构、治理、排障或优化动作
- 结果：指标提升多少，故障恢复多久，沉淀了什么机制

最好优先准备下面几类故事：

- 一次容量瓶颈优化
- 一次线上事故止血与复盘
- 一次核心链路重构
- 一次跨团队推动的平台化治理

### 2. 质量管理

可以围绕这几个点展开：

- 单元测试不是覆盖率数字游戏，重点是覆盖核心状态流转和边界条件
- Code Review 的价值是防止架构腐化和隐患进入主干，而不是只看代码风格
- 设计模式不是背概念，而是为了控制复杂度，比如状态机、策略模式、责任链、模板方法

### 3. 远程协作

远程岗位常见关注点：

- 你是否会写清楚设计文档、复盘文档和发布说明
- 你是否能把同步沟通前置成异步文档，减少低效会议
- 你是否能明确风险、依赖和时间边界，而不是等问题爆出来再说

## 高频追问题库

下面这些题不是让你逐字背，而是让你练“结论 -> 原理 -> 方案 -> 风险”的表达。

### Q1：并发调用多个外部大模型 API 时，Go 的 GMP 如何工作？如何做背压和限流？

回答骨架：

- 大量 goroutine 等待外部 API 时，主要是 IO 阻塞，不会等价于创建大量 OS 线程
- 运行时会通过网络 poller 和调度器管理阻塞 goroutine，但 goroutine 数量失控仍会推高内存、定时器、上下文和连接资源成本
- 我会把请求处理拆成入口限流、租户限流、模型限流、并发 worker 池和下游超时控制五层
- 所有子调用挂在同一个根 `context` 上，超时、用户取消、上游断链时统一级联取消
- 如果某个模型突然变慢，先快速熔断和降级，再切备用模型或异步化，而不是把请求堆死在内存里

面试官重点看：

- 你是否理解 syscall/网络阻塞对调度器的影响
- 你是否会把限流、背压和取消机制落到代码结构上

### Q2：你做过最深的一次 Go 性能优化是什么？

回答骨架：

- 先说场景和指标异常，例如 CPU 飙升、P99 上升、内存泄漏、GC 次数异常
- 再说排查手段，例如 `pprof`、`trace`、运行时指标、火焰图、对象分配热点
- 再说优化动作，例如减少逃逸、复用对象、分离热点锁、缩短大对象生命周期、调 `GOGC` 或 `GOMEMLIMIT`
- 最后讲结果和副作用，例如吞吐提升、GC 暂停下降，但代码复杂度是否增加

### Q3：直播平台做全站广播和大主播房间热点数据，Redis 热点 key 怎么彻底治理？

回答骨架：

- 先区分“广播链路”和“热点房间状态”是两类问题
- 广播更适合消息总线 + 网关分发，不要让所有节点反复从 Redis 拉同一个 key
- 热点房间数据做多级缓存，本地缓存承接绝大多数读流量，Redis 只做共享状态和失效协调
- 对超热点 key 做拆分、预聚合、分区路由或旁路静态化，不要把全部读压在单个 Redis 槽位上
- 必要时把“高频查询状态”和“低频准确数据”分层存储

### Q4：打赏扣费这类交易链路，怎么保证 MySQL 强一致？节点故障时怎么补偿？

回答骨架：

- 单服务单库内先用本地事务解决，不要一上来就分布式事务
- 跨服务时按业务等级选 TCC、Outbox+MQ、Saga，而不是统一方案
- 资金扣减要有幂等键、账户流水、状态机和对账
- 网络分区或节点宕机时，系统目标不是“永远不出错”，而是“最终可收敛、可补偿、可追账”

### Q5：Kafka 出现千万级积压，如何排查与处理？如何同时保证顺序和不丢失？

回答骨架：

- 先看 lag 是否集中在少数分区，再看 Broker 资源、消费者耗时和下游依赖
- 对慢消息分类，区分是单条消息过大、外部依赖变慢，还是消费者逻辑阻塞
- 临时止血可以扩消费者、提分区、隔离慢主题、降级非关键消费
- 真正要求顺序时按业务 key 固定分区，消费端串行处理，并接受吞吐受限
- “不丢失”依赖生产幂等、副本保障、消费幂等、延迟提交和失败重试一起成立

### Q6：从零设计一个支持多模型、上下文记忆和工具调用的 AI 服务，微服务边界怎么划？

回答骨架：

- API Gateway 负责鉴权、限流、租户隔离和请求入口
- Orchestrator 负责意图识别、工作流推进和工具调度
- Model Router 负责模型路由、配额、fallback 和成本治理
- Tool Service 负责向量检索、数据库查询、内部业务能力调用
- Session/Memory Service 负责上下文、会话摘要和长期记忆
- Async Worker 负责长任务、批处理和补偿

### Q7：K8s 里 AI 服务怎么做资源限制和弹性伸缩？动态 IP 下白名单怎么解决？

回答骨架：

- 在线推理服务要按 CPU、内存、并发数、队列长度联合看，不要只盯 CPU
- `requests` 保障调度，`limits` 防止单 Pod 抢占，但 limit 过紧会导致频繁 OOM 或节流
- HPA 指标优先选真实业务指标，例如活动请求数、排队长度、P95
- 第三方白名单问题通常要解决固定出口 IP，而不是试图固定 Pod IP

## 场景题一：AI Agent 编排服务的高并发调度与架构抽象

假设题目是：

> 服务接收复杂用户输入，内部做意图识别，并发调度多个外部工具，最后由 LLM 汇总并流式返回。

这个题不要平均发力。最容易拉开差距的是三件事：

- 长耗时、多 IO 场景下的调度与资源保护
- 工具编排的抽象模型是否可扩展
- 流式输出、失败补偿和线上稳定性是否成体系

答题时我通常分两段来讲：

- 先讲底层并发控制与系统防雪崩，这是防守篇
- 再讲图编排、状态传递和 AI 与核心系统之间的边界，这是进攻篇

### 1. 先定目标和边界

我会先明确四个约束：

- 用户请求是同步流式返回，还是异步任务 + 回调/轮询
- 工具调用是否允许并发，是否要求严格顺序
- 外部依赖是否不稳定，是否存在长尾延迟
- 是否需要会话记忆、审计、可重放和调试追踪

如果这些约束不先说清，后面的设计很容易漂。

### 2. 微服务边界

我通常会拆成下面几层：

#### 入口层

- API Gateway：鉴权、配额、审计、租户限流、SSE 连接管理

#### 编排层

- Agent Orchestrator：负责意图识别、计划生成、工作流推进、状态机管理

#### 路由层

- Model Router：负责不同 LLM 的路由、重试、fallback、成本和 SLA 治理

#### 工具层

- Tool Adapter：统一封装向量检索、数据库查询、搜索、内部结算系统等工具

#### 状态层

- Session/Memory Service：保存会话状态、摘要、长期记忆、工具执行历史

#### 异步层

- Async Queue + Worker：处理超时任务、重试、补偿、离线总结、异步评测

### 3. 核心调用链路

一个比较稳的主链路是：

1. 请求进入网关，完成鉴权、限流、请求 ID 注入
2. 编排服务创建根 `context`，挂上总超时、租户预算和 trace
3. 意图识别与计划器生成执行图
4. 调度器按 DAG 或状态图并发触发可执行节点
5. 工具结果统一回写到运行态上下文
6. LLM 根据中间结果继续规划或生成最终回答
7. 输出通过 SSE 分段返回，包括中间状态、工具进度和最终答案

### 4. Go 层面的调度与生命周期管理

这是这个题的核心。

#### 1. 全链路 `context` 级联管控

原则是绝不裸起 goroutine。每个外部工具调用、模型请求、流式推送协程，都必须绑定派生自顶层请求的 `context.Context`。

设计重点：

- 顶层请求进入时创建根 `context`
- 每个节点执行时派生子 `context`
- SSE 连接断开、全局超时触发、用户主动取消时统一 `cancel`
- 所有阻塞点都必须 `select` 监听 `ctx.Done()`

这样做的价值不是“代码更规范”，而是：

- 客户端一断开，底层挂起请求能立刻感知并退出
- 达到全局 P99 超时后，整条执行图能安全收敛
- 从根上避免 goroutine 泄漏、悬挂重试和内存暴涨

#### 2. 不做无界 goroutine 扩张

我不会让每个工具调用都无限制起 goroutine，而是：

- 按请求内最大并发数限制
- 按租户限制
- 按工具类型限制
- 按外部依赖实例做舱壁隔离

这样单个热点租户或单个慢工具不会把整机 goroutine、连接和内存拖满。

#### 3. 舱壁隔离与独立 worker pool

如果某个外部大模型接口突然长时间卡死，最危险的不是单个请求超时，而是整个服务被它拖进雪崩。

所以我会按下游依赖拆独立并发池，例如：

- LLM 调用池
- 向量检索池
- 内部数据库查询池
- 核心交易或清结算调用池

实现上可以是：

- 信号量 + goroutine
- 带缓冲 channel 控制并发
- 或独立 worker pool

关键不是用什么库，而是保证慢依赖只能耗尽自己的池子，不能侵占全局调度资源。这就是典型的 Bulkhead Pattern。

#### 4. 所有子任务都挂根 context

设计原则：

- 请求超时，所有子工具立刻取消
- 客户端断开 SSE，编排服务感知后停止无意义计算
- 某一步已经判定不可继续，就级联取消后续分支

如果没有这套取消树，最容易出现的就是“用户早就走了，机器还在拼命跑工具调用”。

#### 5. 结果汇聚要可中断、可降级

外部 LLM 或工具突然卡死时，我会这样处理：

- 每个工具单独超时
- 给重要工具和非重要工具分级
- 慢工具触发熔断后，降级成默认回答、缺省结果或异步补全
- 对需要 quorum 的结果设定最小成功数，不要求所有分支都结束才继续

#### 6. 流式通道与 channel 安全关闭

流式响应场景里，LLM 节点和 HTTP 层之间通常会通过 channel 传递 token 块或中间事件。

这里要注意两个点：

- 发送方和关闭方职责必须唯一，避免多处关闭 channel 导致 panic
- 接收方必须同时监听数据 channel 和 `ctx.Done()`

安全模式一般是：

- 节点侧只负责发送
- 聚合器或生产者侧负责在完成时关闭 channel
- HTTP 层 `select` 监听 token、错误和取消信号

这样客户端断开时，不会继续向已关闭 channel 写数据，也不会把响应协程吊死。

#### 7. 资源隔离

至少做三层隔离：

- CPU/内存层：不同服务、不同 Pod 资源隔离
- 连接层：不同外部依赖独立连接池
- 调度层：不同工具类型独立 worker pool 或信号量

### 5. 编排抽象：DAG + 状态机，而不是写死流程

面对未来的循环判断、多 Agent 协作、人工确认节点，我更倾向于：

- 用状态机描述运行阶段
- 用 DAG 或节点图描述依赖关系
- 用统一节点接口封装执行、重试、超时、回滚和观测

节点抽象至少包含：

- 输入上下文
- 执行函数
- 超时与重试策略
- 成功条件
- 失败处理策略
- 输出写回规则

这样后续新增工具、插入审批节点、增加循环判断时，不需要把主流程改成一团 `if else`。

### 6. 全局状态：统一 `State` 对象做不可变传递

复杂 Agent 流最容易失控的点，不是节点多，而是多个节点并发修改共享上下文。

所以我更倾向于把节点协议设计成：

- 节点接收当前 `State`
- 节点返回新的 `State`
- 状态写入集中在调度器或状态合并器中完成

`State` 可以包含：

- 会话历史
- 中间推理结果
- 工具调用记录
- 当前执行阶段
- 幂等键、trace 信息和审计字段

这样做有三个好处：

- 避免多 goroutine 直接读写全局变量造成 data race
- 方便回放、调试和断点恢复
- 为后续持久化状态机和异步续跑打基础

### 7. 确定性系统与非确定性系统的边界处理

这是这道题的深水区。

LLM 输出天然是概率性的，可能产生幻觉、重复调用工具，甚至构造出非法参数。但内部结算、扣费、订单这些核心系统要求的是强确定性。

所以在 LLM 节点和核心业务节点之间，必须有一层防腐层：

- 工具 schema 严格约束输入参数
- 所有高风险工具都做参数校验和权限校验
- 业务接口实现幂等，不相信模型只会调用一次
- 对非法、越权、重复调用直接拒绝，不把风险传给底层核心系统

如果涉及资金操作，我会明确：

- Agent 不直接决定交易结果
- Agent 只能发起受控指令
- 底层交易系统仍由状态机、幂等、对账和补偿负责兜底

### 8. 数据与一致性

这类系统通常要分清三类数据：

- 请求运行态：短期存在，适合放内存 + Redis
- 会话记忆：适合放数据库或专用记忆存储
- 审计和回放：适合事件流或持久日志

如果工具链里接入清结算系统、下单系统这类核心业务能力，我会明确：

- Agent 本身不直接承担交易一致性
- 交易链路仍走原有状态机、幂等和对账体系
- Agent 只是调用者，不应该破坏底层交易系统边界

### 9. 流式返回设计

SSE 事件建议拆成几类：

- `planning`：计划已生成
- `tool_start`：某工具开始调用
- `tool_result`：某工具完成
- `token`：模型流式输出 token
- `final`：最终答案
- `error`：可恢复或不可恢复错误

这比只吐文本更容易做前端交互、问题定位和链路回放。

### 10. 观测与运维

这个题如果想答完整，最后一定要收在观测和故障处理：

- Trace 关联用户请求、编排节点、工具调用、模型调用
- Metrics 看成功率、P95、工具超时率、取消率、队列长度、SSE 断流率
- Log 保留结构化字段，例如租户、会话、节点类型、工具名、错误码
- 对慢模型、慢工具、积压队列设专门告警

### 11. 一段适合口述的总结版答案

如果面试现场时间不多，我会压缩成下面这段：

> 我会把它设计成“入口网关 + 编排服务 + 模型路由 + 工具适配层 + 会话记忆 + 异步任务”的分层架构。编排层内部不用写死流程，而是用 DAG 配合状态机推进节点执行。Go 层面重点做三件事：第一，用根 `context` 管所有子任务，保证超时、断连、失败时能级联取消；第二，用分层限流、信号量和独立 worker pool 做背压与资源隔离，避免慢工具拖垮整机；第三，把流式输出、重试、熔断、降级和观测做成统一框架能力。这样它既能支持当前多工具并发调用，也能在后续扩展到循环判断、多 Agent 协作和异步补偿。 

## 场景题二：百万并发下的存储高可用与资金强一致性

这道题实际上考两件完全不同的事：

- Redis 在极端热点流量下如何不被单点打穿
- MySQL 在资金链路里如何在高并发下既保住正确性，又不把吞吐打崩

答题时最好分“扛热点”和“守资金”两部分说。

### 1. 打透热点流量：多级缓存而不是只会说 Redis Cluster

单靠 Redis Cluster 解决不了极端热点。因为头部主播、全站公告、热门房间状态，最终仍可能打到同一个热点 key 或同一个热点槽位。

更稳的方案是 L1/L2 多级缓存：

- L1：Go 进程本地缓存，例如 BigCache、FreeCache
- L2：Redis 作为共享缓存和分布式协调层
- L3：MySQL 或其他持久化存储作为最终数据源

这样做的价值是：

- 极热读请求直接在应用层命中
- 降低网络 IO 和 Redis 单节点 CPU 压力
- 避免热点流量全部穿透到底层集群

### 2. 热点探测与动态降级

只做缓存还不够，还要主动发现热点。

比较实用的策略是：

- 在网关、代理层或业务层做滑动窗口统计
- 发现热点 key 后，主动把热点数据广播到各业务节点的本地缓存
- 对非核心查询链路做柔性降级，例如延迟刷新、返回近实时快照、关闭高频装饰性接口

这里的思路不是“所有请求都必须实时命中最新值”，而是先保证系统不被热点拖死，再按业务等级恢复精度。

### 3. 房间广播和热点房间数据要拆开设计

直播场景里很容易把两类问题混在一起：

- 全站广播、进场动效、房间消息分发
- 房间基础状态、榜单、在线人数、礼物聚合等热点数据读取

前者更适合消息总线和网关 fan-out，后者更适合多级缓存、预聚合和异步刷新。

如果把广播也压到 Redis 热点 key 上，本质上是在用缓存做消息系统，扩展性会很差。

### 4. 资金绝对安全：避免应用层查后改

涉及打赏扣费、账户余额、清结算时，我会先明确一个原则：

- 绝不在应用层做非原子的 `read-modify-write`

更稳的扣减方式是：

- 乐观锁，基于 `version` 字段控制并发更新
- 或直接在 SQL 里写条件更新，例如 `balance >= amount`

典型模式：

```sql
UPDATE account
SET balance = balance - 100
WHERE id = 1
  AND balance >= 100;
```

这类写法把并发竞争收敛到数据库原子更新里，比应用层先查余额再扣减安全得多。

### 5. 分布式事务不要默认上 2PC

订单服务、账户服务、清结算服务分开后，最容易犯的错是同步串行调用一路锁到底。

这会带来两个问题：

- 长时间持有数据库行锁
- 下游稍慢就把上游并发能力拖崩

所以我一般会这样选：

- 单库单服务：优先本地事务
- 强业务一致场景：TCC 或显式状态机
- 允许最终一致场景：本地消息表 + MQ + 补偿

### 6. 本地消息表保证最终一致

资金链路里，如果跨服务同步事务代价太高，我更偏向可靠消息最终一致性方案。

核心流程是：

1. 在本地 MySQL 事务里同时完成账户变更和 outbox 消息写入
2. 后台任务或 binlog 订阅组件把待发送消息投递到 Kafka
3. 下游消费成功后更新消息状态
4. 失败时依赖重试、补偿和对账机制收敛

这样做的优点是：

- 不长时间占住跨服务事务
- 吞吐量明显好于同步强一致链路
- 资金结果仍然可以通过幂等、状态机、对账做最终闭环

### 7. 一段适合口述的总结版答案

> 这类题我会拆成两层来看。对 Redis 热点问题，核心不是单纯扩 Redis，而是用本地缓存加 Redis 的多级缓存体系，再配合热点探测、预热和动态降级，把极热流量挡在应用层。对资金链路，核心不是追求链路上每一步同步完成，而是先用数据库原子更新、幂等和状态机保证扣费正确，再用本地消息表和 Kafka 实现跨服务最终一致。这样既扛得住百万并发，又不会为了吞吐牺牲资金正确性。 

## 场景题三：海量异步任务的削峰、积压与严格顺序

这道题的关键不只是“会不会调 Kafka 参数”，而是你能不能讲出事故处理顺序和架构取舍。

### 1. 先止血，再排查，再提速

发现千万级积压后，不要直接开始调参数，先做止血动作：

- 判断是不是下游 AI 节点、向量检索或外部依赖大面积故障
- 对非核心生产流量限流或降级
- 必要时暂停部分低优先级 topic 或消费链路

目标是先阻止 lag 继续失控，而不是一边灌水一边救火。

### 2. 排查要按链路顺序来

我一般按这条链路排：

- Producer 是否突然放量
- Broker CPU、磁盘、网络、页缓存是否异常
- 分区是否倾斜，lag 是否集中在少数 partition
- Consumer group 是否频繁 rebalance
- 单条消息处理耗时是否异常
- 下游 AI 节点是否成为慢依赖

这样排的原因是，积压只是表象，根因通常在某个阶段吞吐突然跌破生产速率。

### 3. 拉、处分离，榨干单机消费能力

如果下游 AI 处理很慢，消费者不要把“从 Kafka 拉消息”和“业务处理”绑死在同一个阻塞循环里。

更实用的做法是拉、处分离：

- 主 goroutine 专门快速 fetch 消息
- 拉到消息后按业务 key 分发到内部队列
- 多个 worker goroutine 异步执行具体处理

如果追求极致吞吐，可以进一步用 ring buffer 或无锁队列承接内部分发。

这样做的价值是：

- Kafka 拉取侧保持高吞吐
- 业务处理侧可以独立扩展并发
- 单机 CPU 和网络资源利用率更高

### 4. 严格顺序必须同时在 Kafka 层和进程内层成立

很多人只说“同一个 key 发到同一个 partition”，这只解决了 Kafka 层的顺序。

如果消费端拿到消息后又随意分发给不同 worker，同样会乱序。

所以严格顺序至少要满足：

- 同一个业务 key 固定到同一个 Kafka partition
- 消费端再基于同样的 key 做一致性 hash
- 同一个 key 永远落到同一个内部 worker 串行处理

这时即便你在单机内部做了并发模型，也不会打乱单 key 的事件顺序。

### 5. 防丢失是全链路问题

严格不丢失不能只看消费者。

比较标准的防线是：

- Producer：`acks=all`、重试开启、幂等生产
- Broker：副本数至少 3，`min.insync.replicas=2`
- Consumer：处理成功再提交 offset，业务侧实现幂等

如果业务特别关键，还要加：

- 死信队列
- 重试主题
- 消费失败审计
- 人工补偿和回放机制

### 6. 一段适合口述的总结版答案

> Kafka 积压我会按“先止血、再排查、再提速”处理。先限制非核心生产流量，避免 lag 继续膨胀；再按 producer、broker、partition、consumer、下游依赖这条链路定位瓶颈；最后在消费端做拉、处分离，把 fetch 和业务处理解耦。如果业务要求严格顺序，我会让同一个 key 固定到同一个 partition，并在消费进程内部再次基于同 key 做一致性 hash，保证永远由同一个 worker 串行处理。防丢失则靠 producer、broker、consumer 三端一起兜住，而不是只靠 Kafka 本身。 

## 推荐复习顺序

如果你要在较短时间内准备这类岗位，建议按这个顺序：

1. 先补 Go 调度、GC、并发控制，把 runtime 层面讲顺
2. 再补 MySQL、Redis、Kafka，把一致性、缓存、消息治理讲顺
3. 再把 AI 工程化、IM/直播和 K8s 这几个场景题串起来
4. 最后准备 1 到 2 个项目故事，把方案、事故、指标和复盘讲顺
