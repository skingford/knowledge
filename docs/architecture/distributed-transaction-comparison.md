## 为什么需要分布式事务

当一个业务操作跨多个服务、数据库或消息系统时，单机事务已经无法覆盖整个链路。这时需要考虑：

- 资金、库存、订单状态是否必须一致
- 某个步骤失败后如何补偿
- 是否允许短暂不一致

## 主流方案对比

### 2PC

特点：

- 强一致思路更直接
- 参与方统一提交或回滚

问题：

- 协调者压力大
- 阻塞明显
- 可用性和性能较差

适用：

- 参与方少、链路短、对一致性要求极高的场景

### TCC

特点：

- 业务层显式拆成 Try、Confirm、Cancel
- 可控性强

问题：

- 业务侵入重
- 开发成本高
- 幂等、空回滚、悬挂处理复杂

适用：

- 核心交易、资金、库存这类强业务一致性场景

### 本地消息表 / 最终一致性

特点：

- 主业务事务和消息发送解耦
- 吞吐更高
- 更适合互联网业务

问题：

- 存在短暂不一致
- 需要补偿、重试、对账机制

适用：

- 订单、营销、通知、积分等允许延迟一致的场景

## 怎么选

### 如果业务要求强一致

- 先评估是否真的必须强一致
- 如果必须，再看是否值得接受 TCC 的复杂度

### 如果业务允许最终一致

- 优先考虑消息驱动方案
- 配合幂等、重试、补偿和对账

### 如果只是单库多表

- 优先用本地事务，不要过早引入分布式事务

## 常见风险

- 幂等没做好，重复执行导致脏数据
- 补偿逻辑缺失，失败后无法收敛
- 过度追求强一致，把系统复杂度拉得过高
- 没有对账机制，问题长期潜伏

## 推荐实践项目

- 订单创建与库存扣减
- 支付成功与订单状态更新
- 积分发放与消息通知
