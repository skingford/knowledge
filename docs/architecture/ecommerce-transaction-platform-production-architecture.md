## 一、先把可用性目标算清楚

交易平台如果目标是 `99.99%` 可用性，就不能只喊口号，必须把它拆成具体工程约束。

### 1. 99.99% 到底意味着什么

`99.99%` 可用性大致意味着：

- 年故障时间不超过 `52.56 分钟`
- 月故障时间不超过 `4.38 分钟`

这会直接推导出几个硬约束：

- 单次发布或变更不能长时间影响用户
- 任意单点故障必须快速自动恢复
- 主从切换、实例摘除和服务重启必须尽量对业务透明

如果系统做不到：

- `故障发现 < 20 秒`
- `切换动作 < 10 秒`
- `用户恢复 < 30 秒`

那四个 9 就很容易在几次事故里被打穿。

### 2. 三个不能妥协的原则

生产级交易平台里，最重要的三条原则通常是：

- `没有单点`
  每个关键组件至少双副本，且跨机架或跨可用区部署
- `故障自愈`
  人工介入之前系统已经开始自动切换，告警是通知，不是触发器
- `严格隔离`
  核心链路和非核心链路必须拆开，推荐、评论、足迹不能拖垮下单和支付

一句话概括：

> 交易平台不是“尽量别挂”，而是“挂了也要自动恢复，且优先保核心交易”。

## 二、两地三中心与单元化多活

如果面对千万 DAU，所有流量都打一个主数据中心，系统再强也会在容量、切换和故障域上暴露明显风险。

### 1. 两地三中心整体拓扑

比较常见的生产级拓扑可以概括为：

```text
                    ┌─────────────────┐
                    │   全局流量调度   │
                    │   GTM / GSLB    │
                    └────────┬────────┘
                             │
          ┌──────────────────┼──────────────────┐
          │                  │                  │
   ┌──────▼──────┐    ┌──────▼──────┐    ┌──────▼──────┐
   │  城市 A 主 DC │    │  城市 A 备 DC │    │   城市 B DC  │
   │    60% 流量   │    │    20% 流量   │    │    20% 流量   │
   │    完整服务   │    │    完整服务   │    │    完整服务   │
   └─────────────┘    └─────────────┘    └─────────────┘
```

它想解决的不是“把机器铺开”，而是三个问题：

- 单个机房故障时，流量可以被其他中心接管
- 单个城市故障时，仍有异地承接能力
- 平时就有真实流量，灾备中心不会长期冷着

### 2. 单元化路由为什么是交易多活的关键

真正的大规模电商交易系统，不能让所有用户都依赖同一个中心写入。

更现实的做法是：

- 按用户维度切单元
- 同一用户稳定路由到同一单元
- 强一致数据在单元内自洽

例如：

- `uid % 100 < 60` -> `unit-a`
- `60 <= uid % 100 < 80` -> `unit-b`
- `uid % 100 >= 80` -> `unit-c`

一个简单的路由 SDK 大致如下：

```java
public class UnitRouter {
    public String route(long uid) {
        int slot = (int) (uid % 100);
        if (slot < 60) return "unit-a";
        if (slot < 80) return "unit-b";
        return "unit-c";
    }
}
```

这套方案的关键约束是：

- 同一用户请求必须稳定落到同一单元
- 订单、支付、库存等强一致写入尽量留在单元内完成
- 商品、配置、活动模板这类偏只读数据再跨单元同步

### 3. 故障切换 SLA 怎么拆解

如果目标是 `RTO < 30 秒`，那“自动切换”就必须拆成明确时间预算。

一个常见分解可以是：

- 故障检测：`5 ~ 15 秒`
- 熔断摘流：`< 1 秒`
- 路由切换：`5 ~ 10 秒`
- 实例恢复或新实例接管：`10 ~ 30 秒`
- 数据库自动选主：`5 ~ 10 秒`

这类目标通常要求：

- 更高频的健康检查
- 更短的 DNS TTL
- 新实例预热
- 单元迁移按 `5% -> 20% -> 100%` 灰度推进

需要明确的一点是：

- `RTO` 不是等人看见告警后再操作
- 而是系统自己完成绝大部分恢复动作

## 三、接入层与流量治理

交易平台真正的大流量，不应该直接落到业务服务上，必须靠接入层分层削峰。

### 1. 多级限流体系

比较成熟的做法通常是四层：

#### 第一层：公网入口与清洗

- BGP Anycast 或云厂商流量调度
- DDoS 清洗
- 单 IP、单 ASN、异常 UA 的自动拦截

这一层解决的是：

- 恶意流量
- 攻击流量
- 明显不该进入业务网关的垃圾请求

#### 第二层：Nginx / L4 限流

典型目标是限制：

- 单 IP 最大并发连接
- 单 IP 请求速率

例如：

```nginx
limit_conn_zone $binary_remote_addr zone=conn:10m;
limit_conn conn 100;
limit_req_zone $binary_remote_addr zone=req:10m rate=200r/s;
```

#### 第三层：API Gateway 业务限流

这一层开始按业务语义做：

- 全局容量限流
- 接口级限流
- 用户级限流
- 商户级或租户级配额

例如：

- `/api/order/create` 限流
- 单用户每分钟请求数限制
- 不同商家等级配置不同配额

#### 第四层：服务内部熔断与降级

服务层一般还要继续做：

- 慢调用降级
- 异常比例熔断
- 下游依赖隔离

也就是说：

- 接入层负责“别让流量打穿入口”
- 服务层负责“局部故障别扩散成全链路雪崩”

### 2. API Gateway 的核心处理链路

网关本质上通常是一条责任链。

```java
public class GatewayPipeline {
    private final List<Filter> filters = List.of(
        new AuthFilter(),
        new RateLimitFilter(),
        new RouteFilter(),
        new GrayFilter(),
        new CircuitBreakerFilter(),
        new LogFilter(),
        new TimeoutFilter()
    );
}
```

典型职责包括：

- `Auth`：JWT 验签、黑名单校验
- `RateLimit`：令牌桶限流
- `Route`：服务发现和单元路由
- `Gray`：灰度规则匹配
- `CircuitBreaker`：熔断
- `Log`：结构化日志和 Trace 注入
- `Timeout`：入口总超时保护

灰度规则也最好稳定，不要让同一用户在不同版本间来回跳：

- 按 `uid % 100`
- 按设备 ID 哈希
- 按明确 header 标记测试流量

## 四、核心交易链路设计

交易平台真正的复杂度，不在于单个服务，而在于订单、库存、营销、支付如何一起收敛。

### 1. 订单服务：用 Saga 编排整条下单链路

一个更适合生产的下单主链路通常不是“一个接口干完一切”，而是由 Saga 协调器驱动：

```text
创建订单草稿
-> 预扣库存
-> 冻结优惠券
-> 发起支付
-> 全部成功则确认
-> 任一步失败则逆序补偿
```

Saga 协调器必须把实例状态持久化，不然协调器自己重启后就会丢上下文。

```java
@Entity
public class SagaInstance {
    private String sagaId;
    private String sagaType;
    private SagaStatus status;
    private String currentStep;
    private String contextJson;
    private LocalDateTime updatedAt;
}
```

这里最重要的不是框架名，而是：

- Saga 状态要持久化
- 每步上下文要能拿回来补偿
- 协调器重启后能继续执行或继续补偿

### 2. 库存服务：三层超卖防护

库存服务真正稳的方案，通常不是只靠一层，而是三层一起上。

#### 第一层：Redis 预扣减

用 Lua 保证：

- 判断库存
- 扣减库存

在 Redis 内原子执行。

```lua
local stock = redis.call('GET', KEYS[1])
if tonumber(stock) < tonumber(ARGV[1]) then
    return -1
end
redis.call('DECRBY', KEYS[1], ARGV[1])
return 1
```

这一层的价值是：

- 快速拒绝
- 把热点流量挡在数据库前面

#### 第二层：数据库冻结库存

最终真实约束仍然要落回数据库：

```sql
UPDATE inventory
SET frozen_qty = frozen_qty + ?,
    available_qty = available_qty - ?
WHERE product_id = ?
  AND available_qty >= ?;
```

如果 `affected_rows = 0`，就说明：

- 库存不足
- 或并发下已经被别人抢走

#### 第三层：异步对账与校准

Redis 预扣减只是前置容量保护，不应被当作唯一真相。

所以还要保留：

- Redis 与 DB 周期对账
- 偏差超阈值告警
- 大促前强制回源校准

### 3. 热点商品要做分片库存

秒杀或爆款商品里，一个单独的库存 key 迟早会变成热点。

比较常见的办法是把库存拆片：

- 总库存 `1000`
- 拆成 `100` 个 Redis 分片
- 每片 `10`

请求按用户哈希稳定落到某个分片：

```java
String shardKey = "stock:" + productId + ":" + (uid % SHARD_COUNT);
```

这么做的收益是：

- 降低单 key 热点
- 分散 Redis 和下游处理压力

代价则是：

- 分片售罄后的合并和回补更复杂
- 需要后台任务处理“假售罄”问题

### 4. 支付服务：资金安全优先于吞吐

支付最怕的不是慢，而是：

- 重复扣款
- 渠道成功、本地失败
- 本地成功、渠道未知

所以幂等屏障通常要做成多层：

1. 客户端或前端生成 `paymentToken`
2. 网关层 `SET NX` 防重复提交
3. 数据库唯一约束兜底
4. 实时对账 + T+1 批量对账

比较稳妥的原则是：

- 同步返回只做阶段性结果
- 最终结果靠异步通知、主动查询和对账收口

### 5. 订单号为什么必须应用层生成

交易平台里，`orderId`、`paymentNo`、`refundNo` 这类业务主键，通常不应该依赖数据库自增。

原因很直接：

- 下单前就需要全局唯一业务 ID，供库存、支付、营销、Saga、MQ 一起使用
- 分库分表和单元化之后，数据库自增天然只保证分片内唯一
- 两地三中心、多单元场景下，业务主键需要跨库、跨单元、跨消息链路稳定传播
- 数据库主备切换透明，不等于“全局业务唯一性”天然成立

更现实的做法通常是：

- `数据库内部代理主键` 可以保留自增
- `业务主键` 用应用层生成，比如 Snowflake、号段、Sequence 服务或带单元位的自定义 ID

## 五、数据层高可用

交易平台的数据层目标通常不是“绝对单一强一致”，而是分层做取舍。

### 1. MySQL 架构怎么拆

交易核心库更偏向：

- 单主写
- 同城强同步或半同步
- 跨城异步灾备

读写分层也必须按一致性等级区分：

- `强一致读`：订单状态确认、支付确认，走主库
- `最终一致读`：商品详情、历史订单列表，可走从库

### 2. 分库分表为什么常按 uid 路由

订单库常见做法是按用户维度路由：

- `uid % 256` 路由逻辑库
- 库内再分多张表

这样做的好处是：

- 单用户订单尽量落同一分片
- 用户视角查询更容易命中单分片

但跨商家、跨维度查询会变得困难，所以通常会补一份搜索或分析索引：

- `Binlog -> Canal / CDC -> MQ -> ES`

### 3. 四级缓存体系

生产级交易平台的缓存体系一般不会只有一个 Redis。

比较常见的是：

- `L0`：浏览器缓存 / CDN
- `L1`：进程内缓存，比如 Caffeine
- `L2`：Redis Cluster
- `L3`：数据库读库兜底

这里最重要的是明白：

- 越靠前越快，但越容易不一致
- 越靠后越慢，但越接近真实状态

### 4. 缓存一致性不要只靠“更新后立刻删缓存”

高并发下，最危险的误区之一就是以为：

- 更新 DB
- 删缓存

就一定安全。

更常见的现实做法是：

- 更新 DB
- 发 MQ 或 Binlog 订阅删除缓存
- 必要时做延迟双删

延迟双删想解决的是：

- 并发读在脏窗口里把旧值重新写回缓存

## 六、可用性数学与错误预算

高可用不是感性目标，最好用数学约束团队行为。

### 1. 串联系统为什么天然拉低可用性

如果每个组件可用性都是 `99.9%`，10 个组件串起来并不是 `99.9%`，而是：

```text
0.999 ^ 10 ≈ 99.0%
```

这意味着：

- 不能只堆单组件可用性
- 必须依赖冗余、副本和快速切换

### 2. 错误预算为什么值得单独管理

四个 9 对应的年错误预算大约是：

- `52.56 分钟 / 年`

这份预算一般会被分给：

- 计划内变更
- 非计划故障
- 第三方依赖问题

比较成熟的团队会把预算消耗和发布节奏挂钩：

- 预算消耗过快，暂停新功能，优先做稳定性
- 预算长期过低，说明可能过度保守，可以加快发布

所以错误预算真正约束的不是事故复盘，而是：

- 发布频率
- 变更规模
- 风险承受能力

## 七、大促保障与发布体系

交易平台真正的生产级能力，往往在大促和发布时被验证出来。

### 1. 大促保障不是临时加机器，而是提前演练

比较成熟的节奏通常是：

- `T-7`：全链路压测、影子流量、容量评估
- `T-1`：缓存预热、阈值调整、降级开关确认
- `T-0`：核心指标看板、值班、限流兜底

大促期间最重要的一条原则通常是：

- 超过系统能力边界时，优先“有损拒绝”，而不是让全站一起崩

### 2. 降级必须按优先级设计

比较常见的优先级大致是：

- `P0`：下单、支付、库存扣减
- `P1`：商品推荐
- `P2`：评论、足迹、浏览历史

也就是说，大促真正的设计目标不是：

- 所有功能都全开且不降级

而是：

- 核心交易链路必须活着

### 3. 发布体系本身就是高可用能力的一部分

如果你想做四个 9，发布体系不能只是“CI 跑完就发”。

更稳妥的做法通常是：

- 金丝雀或蓝绿发布
- 每一步看错误率、延迟和业务指标
- 指标恶化自动回滚

Feature Flag 也非常关键，因为它能把：

- 代码发布
- 功能生效

拆成两步，减少紧急发版频率。

### 4. 数据库变更也必须当成生产级事件处理

交易平台最危险的变更之一，不是代码，而是：

- 加索引
- 改表结构
- 批量回填

更稳妥的原则是：

- 能 Online DDL 就不要锁表
- 大流量写入前，先把索引准备好
- 把数据库变更也纳入灰度和回滚预案

## 八、为什么云数据库统一入口不能替代分布式 ID

这个问题在交易平台里非常高频，结论可以先直接说：

> 云数据库统一入口可以让单主自增更好用，但不能替代真正的分布式业务 ID 设计。

### 1. 它能解决什么

如果你的数据库是：

- 单主写
- 主备切换对业务透明
- 应用始终连统一入口

那么数据库 `auto_increment` 确实可以继续工作，并且：

- 主从切换后应用基本无感
- 单库内的主键唯一性通常不会出问题

### 2. 它解决不了什么

它解决不了的部分恰恰是交易平台最在意的：

- 下单前就要拿到 `orderId`
- `orderId` 要同时传给库存、支付、营销、MQ、Saga
- 分库分表、多单元、多活时需要跨库全局唯一
- 业务侧需要按 ID 做幂等、对账和日志追踪

也就是说：

- 数据库自增只适合做 `库内代理主键`
- 不适合做 `跨系统传播的业务主键`

### 3. 生产建议

更稳妥的做法通常是：

- `DB 自增`：保留给库内技术主键
- `业务主键`：统一应用层生成

例如：

- 订单号
- 支付流水号
- 退款号
- SagaId

这类 ID 可以带上：

- 时间位
- 单元位
- 机房位
- 序列位

从而同时满足：

- 全局唯一
- 趋势递增
- 分库分表路由
- 跨系统追踪

## 结论

电商交易平台的生产级架构，本质上不是堆组件，而是把下面几件事同时做好：

- 用多活和单元化缩小故障域
- 用分层限流和降级保护核心交易链路
- 用 Saga、库存冻结、支付幂等和对账把交易结果收口
- 用缓存、数据库高可用和错误预算控制整体系统风险
- 用大促演练和发布治理把“理论可用”变成“真实可用”

最值得记住的一句话是：

> 四个 9 不是靠某个组件单独堆出来的，而是靠冗余消除单点、靠自动化缩短恢复时间、靠错误预算约束变更节奏共同实现的。
