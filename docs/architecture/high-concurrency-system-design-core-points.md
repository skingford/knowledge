# 高并发系统设计核心要点

高并发系统设计可以抽象为一个"漏斗型"分层架构：从流量进入到数据落盘，每一层都在做两件事——**拦截无效流量**和**加速有效处理**。本文按这条链路层层递进，从扛住流量、保证正确、不让系统挂，到基础设施支撑，最后用秒杀系统串联全部知识点。

## 阅读指引

**适合人群**：需要设计高并发接口或活动系统的后端工程师 / 准备系统设计能力自检的人 / 需要把"会说概念"变成"会做方案"的开发者

**学习目标**：建立高并发系统设计的检查框架，能从流量、性能、稳定性、数据一致性几个维度审题，输出更完整的方案而不只是讲缓存和 MQ

**文章结构**：全文分五部分递进——第一部分解决"流量怎么扛住"，第二部分解决"数据怎么不出错"，第三部分解决"系统挂了怎么办"，第四部分补齐微服务与分布式环境的基石，第五部分用秒杀系统做一次端到端串联。附录收录设计原则、指标参考和自检框架。

## 设计自检清单

### 先确认业务约束

- 峰值流量多大
- 是否存在突发流量
- 是否允许排队、降级、限流
- 是否要求强一致
- 用户体验要求是什么

### 再确认系统容量

- QPS、TPS、并发连接数
- 数据量级和增长速度
- 缓存命中率预期
- 数据库、消息队列、下游服务瓶颈

### 最后确认保护策略

- 限流 / 熔断 / 降级 / 超时 / 重试 / 隔离

### 典型风险

- 热点 Key 打爆缓存
- 数据库被穿透或写爆
- 重复请求导致脏数据
- 消息堆积导致链路延迟放大
- 下游故障级联传染

## 第一部分：接入层与流量承载

> 一个请求从用户端出发，经过网关、缓存、消息队列，最终到达数据库。这一部分解决的核心问题是：**如何在每一层拦截尽可能多的无效流量，让到达数据库的请求降到最低。**

### 1.1 架构层面

<HcScaleOutDiagram />

#### 分层架构——生产级请求链路

一个用户请求在生产环境中的完整链路：

<details>
<summary>展开详情</summary>

```
用户 → DNS → CDN（静态资源） → 负载均衡（L4/L7）→ API 网关 → 业务服务 → 数据层
```

每一层独立扩展，职责清晰：

| 层级 | 核心职责 | 典型组件 | 扩展方式 |
|------|---------|----------|----------|
| DNS 层 | 域名解析、就近接入、机房级容灾 | 云 DNS / GSLB | 多线路解析、故障自动切换 |
| CDN 层 | 静态资源加速、DDoS 防护 | CloudFlare / 阿里云 CDN | 全球边缘节点 |
| 负载均衡层 | 流量分发、健康检查、SSL 卸载 | Nginx / ALB / SLB | LVS 四层 + Nginx 七层 |
| 网关层 | 路由、限流、鉴权、协议转换 | APISIX / Kong / Spring Cloud Gateway | 无状态水平扩展 |
| 业务服务层 | 业务逻辑处理 | Spring Boot / Go 微服务 | 无状态水平扩展 |
| 数据层 | 持久化存储、缓存 | MySQL / Redis / Kafka | 主从 / 分片 / 集群 |

</details>

#### 水平扩展（Scale Out）

系统设计需无状态化，通过增加机器而非升级单机来提升吞吐量。

<details>
<summary>展开详情</summary>

**无状态化核心要求**——以下状态必须外置，服务实例才能任意增删：

| 需要外置的状态 | 外置到哪 | 常见问题 |
|---------------|---------|---------|
| 用户 Session | Redis / JWT | 本地 Session 导致请求必须粘连到固定节点 |
| 文件 / 图片 | OSS / S3 | 写本地磁盘导致其他节点读不到 |
| 定时任务 | XXL-JOB / K8s CronJob | 本地 cron 导致多实例重复执行 |
| 配置 | Nacos / Apollo | 本地配置文件修改后需逐台重启 |
| 本地缓存 | 加版本号或广播失效 | 各节点缓存不一致 |

</details>

#### 负载均衡

**四层（L4）vs 七层（L7）**

<details>
<summary>展开详情</summary>

| 维度 | L4 负载均衡 | L7 负载均衡 |
|------|-----------|-----------|
| 工作层 | 传输层（TCP/UDP） | 应用层（HTTP/gRPC） |
| 转发依据 | IP + 端口 | URL / Header / Cookie |
| 性能 | 极高（内核态转发） | 较高（需解析应用协议） |
| 功能 | 简单转发 | 路由、改写、限流、鉴权 |
| 典型组件 | LVS / 云 SLB (四层) | Nginx / HAProxy / ALB |
| 适用场景 | 入口流量分发 | 细粒度路由、灰度发布 |

**生产常见架构**：DNS → LVS（四层分发）→ 多台 Nginx（七层路由）→ 后端服务集群

**负载均衡算法选择**

| 算法 | 原理 | 适用场景 |
|------|------|----------|
| 轮询 (Round Robin) | 依次分配 | 实例配置一致时的默认选择 |
| 加权轮询 | 按权重比例分配 | 实例配置不一致（8 核和 4 核混部署） |
| 最少连接 | 分配给当前连接数最少的实例 | 长连接场景（WebSocket、gRPC 流） |
| 一致性哈希 | 相同 key 路由到同一实例 | 需要本地缓存命中率的场景（详见 4.3 节） |
| IP Hash | 相同 IP 路由到同一实例 | 需要会话保持但无法用共享 Session 的遗留系统 |

**健康检查**：负载均衡器定时探活（HTTP 200 / TCP 连通），故障节点自动从集群摘除，恢复后自动加回。Nginx 可用 `max_fails` + `fail_timeout` 配置；K8s 用 Readiness Probe。

</details>

#### 容量估算速算法

从 DAU 到 QPS 的快速估算：

<details>
<summary>展开详情</summary>

```
日请求量 ≈ DAU × 每用户日均请求数
日均 QPS ≈ 日请求量 / 86400
峰值 QPS ≈ 日均 QPS × 峰值系数（通常 3~5 倍）
所需实例数 ≈ 峰值 QPS / 单实例极限 QPS × 冗余系数（1.5~2）
```

**示例**：100 万 DAU，人均 50 次请求/天

```
日请求量 = 100万 × 50 = 5000万
日均 QPS = 5000万 / 86400 ≈ 580
峰值 QPS = 580 × 4 ≈ 2300
单实例极限 QPS = 500（假设）
所需实例 = 2300 / 500 × 1.5 ≈ 7 台
```

::: warning 注意
这是粗算，实际需通过压测（3.2 节）验证。不同接口 QPS 差异大，热点接口需单独估算。
:::

</details>

#### 分布式 Session 方案

高并发场景下的用户会话管理方案选择。

<details>
<summary>展开详情</summary>

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| Cookie + Redis Session | Spring Session 将 Session 存入 Redis，服务无状态 | 成熟稳定，需维护 Redis |
| JWT Token | 无状态令牌，服务端不存储 Session | 性能好，但 Token 吊销困难（需黑名单） |
| Token + Redis | JWT 做身份标识 + Redis 存储权限/状态 | 兼顾无状态和可控性，推荐方案 |

::: tip 建议
高并发且需要主动踢人/吊销的场景用 Token + Redis；纯无状态 API 用 JWT。
:::

</details>

#### 架构选型决策

根据 QPS 量级和团队规模选择合适的架构方案。

<details>
<summary>展开详情</summary>

| 场景 | 推荐架构 | 关键考量 |
|------|---------|---------|
| QPS < 1000，团队 < 10 人 | 单体 + 读写分离 | 不要过早微服务化，运维成本远大于收益 |
| QPS 1000~1万，需独立部署 | 微服务 + 网关 + 注册中心 | 按业务域拆分（详见 4.1 节），每个服务独立数据库 |
| QPS 1万~10万，大促场景 | 微服务 + 多级缓存 + MQ 削峰 | 缓存（1.2 节）和异步（1.4 节）是关键杠杆 |
| QPS > 10万，秒杀/抢购 | 全链路优化 + 弹性伸缩 | 需要本文所有部分的综合运用（详见 5.1 节） |

</details>

#### 流量峰值分级架构参考

按 DAU 和峰值 QPS 从万级到十亿级，给出每个量级的典型架构组合和关键瓶颈。

<details>
<summary>展开详情</summary>

| 级别 | DAU 量级 | 峰值 QPS | 典型架构 | 数据库 | 缓存 | 消息队列 | 关键瓶颈与应对 |
|------|---------|---------|---------|--------|------|---------|--------------|
| L1 万级 | 1万~10万 | 百~千 | 单体应用 + Nginx | MySQL 单主 + 读写分离 | 单节点 Redis | 可选（不强制） | 数据库连接数 → 连接池调优即可 |
| L2 十万级 | 10万~100万 | 千~万 | 微服务 + API 网关 | MySQL 主从 + 分库 | Redis 哨兵 | Kafka / RocketMQ | 单表瓶颈 → 分库分表；缓存命中率是关键 |
| L3 百万级 | 100万~1000万 | 万~十万 | 微服务 + 多级缓存 + MQ 削峰 | 分库分表 / TiDB | Redis Cluster（3主3从起） | Kafka 集群（多 Partition） | 热点 Key → 本地缓存 + 分桶；需要全链路压测 |
| L4 千万级 | 1000万~1亿 | 十万~百万 | 全链路优化 + 同城双活 | 分库分表 + 读写分离 | Redis Cluster（大规模） | Kafka 集群 + 独立消费组 | 资源耗尽 → 连接池/线程池/FD 精细治理；需异地容灾 |
| L5 亿级 | 1亿~10亿 | 百万~千万 | 异地多活 + 单元化部署 | 按用户分区路由到不同机房 | 每单元独立 Redis 集群 | 每单元独立 MQ 集群 | 跨机房一致性 → 数据分区 + 异步同步；全链路灰度 |
| L6 十亿级 | > 10亿 | 千万+ | 全球化多 Region + 边缘计算 | NewSQL / 自研存储引擎 | 多层缓存（CDN → 边缘 → 本地 → 集群） | 自研消息中间件 | 全球路由 → GSLB + 就近写入；基础设施全部自研或深度定制 |

**各级别关键转折点**

| 从哪到哪 | 核心变化 | 典型触发事件 |
|---------|---------|------------|
| L1 → L2 | 单体拆微服务，引入网关和注册中心 | 团队 > 10 人，部署耦合严重 |
| L2 → L3 | 引入多级缓存和 MQ 削峰，数据库分片 | 大促/活动导致数据库被打满 |
| L3 → L4 | 全链路压测常态化，同城双活 | 单机房故障导致全站不可用 |
| L4 → L5 | 单元化架构，数据按用户分区路由 | 单机房容量天花板，需要异地扩展 |
| L5 → L6 | 全球化部署，基础设施自研 | 业务出海，需要多 Region 就近服务 |

::: warning 注意
不要跳级。每一级都有对应的组织能力、运维成熟度和成本要求。90% 的系统处于 L1~L3，过早引入 L4+ 的复杂度会拖慢业务迭代。
:::

</details>

### 1.2 缓存策略

<HcCacheDiagram />

多级缓存：本地缓存（Caffeine）→ 分布式缓存（Redis）→ 数据库，命中率层层递增
缓存穿透：布隆过滤器拦截不存在的 key
缓存击穿：热点 key 加互斥锁或永不过期
缓存雪崩：过期时间加随机抖动，避免同时失效

<details>
<summary>展开详情</summary>

**缓存与数据库双写一致性**
- Cache Aside Pattern（推荐）：先更新 DB，再删缓存。读时 miss 则从 DB 加载并回填缓存
- 为什么不"先删缓存再更新 DB"：并发读请求会将旧值重新写入缓存，导致长期不一致
- 延迟双删：更新 DB 后删缓存 → 延迟 500ms 再删一次，覆盖并发读回填的脏数据
- Binlog 异步同步（最可靠）：Canal 监听 MySQL Binlog → 异步更新/删除 Redis，解耦业务代码
- 适用准则：对一致性要求不高的用 Cache Aside；强一致场景用 Binlog 同步或加分布式锁

</details>

### 1.3 数据库优化

<HcDatabaseDiagram />

读写分离：主库写，从库读，适合读多写少场景
分库分表：按业务或 ID 哈希拆分，单表控制在千万行以内
连接池调优：控制最大连接数，避免连接风暴
索引优化：覆盖索引、避免回表、慢查询监控

<details>
<summary>展开详情</summary>

**数据库中间件与 NewSQL**

| 方案 | 类型 | 特点 | 适用场景 |
|------|------|------|----------|
| ShardingSphere-JDBC | 客户端分片 | 无代理开销，Java 生态集成 | Java 应用分库分表 |
| ShardingSphere-Proxy | 代理层分片 | 对应用透明，支持多语言 | 异构语言、DBA 友好 |
| TiDB | NewSQL | 兼容 MySQL 协议，原生水平扩展 | 免分库分表，大数据量 OLTP+OLAP |
| CockroachDB | NewSQL | 强一致分布式，兼容 PostgreSQL | 全球化部署、强一致需求 |

::: tip 建议
数据量 < 5 亿行优先用分库分表中间件；数据量更大或想避免分片复杂性则考虑 TiDB 等 NewSQL。
:::

</details>

### 1.4 异步与削峰

<HcAsyncDiagram />

消息队列（Kafka / RocketMQ）：将同步调用改为异步，生产者快速返回，消费者按能力消费，天然削峰填谷。
线程池隔离：不同业务使用独立线程池，防止慢任务拖垮整个系统（Hystrix/Resilience4j）。

### 1.5 限流与熔断

<HcRateLimitDiagram />

四种核心流量防护策略：令牌桶、漏桶、滑动窗口和熔断器，分别适用于不同场景。

<details>
<summary>展开详情</summary>

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 令牌桶 | 允许突发流量 | API 网关限流 |
| 漏桶 | 平滑输出速率 | 调用第三方接口 |
| 滑动窗口 | 精确计数 | 用户维度限频 |
| 熔断器 | 下游故障自动断开 | 微服务防雪崩 |

</details>

### 1.6 热点数据处理

<HcHotspotDiagram />

一个爆款商品、一个明星带货直播，都可能把单个 Redis 节点或数据库行打崩。

<details>
<summary>展开详情</summary>

**热点识别**：实时统计访问频次（Redis 热 key 探测、京东 HotKey 框架）

**读热点方案**
- 本地缓存兜底：JVM 级别 Caffeine / Guava Cache，绕过 Redis 直接返回
- 多副本分散：同一 key 加随机后缀分散到多个 slot（如 `product:123:r0` ~ `product:123:r7`），读时随机选一个

**写热点方案**
- 合并写入：先在 Redis 中累加，定期批量刷入数据库（如点赞计数、浏览量）
- 分桶计数：将一个 key 拆成 N 个子桶（如 `stock:123:bucket_0` ~ `stock:123:bucket_7`），写请求随机落桶，读时汇总

**典型案例**：秒杀库存用分桶扣减，每个桶分配部分库存，大幅降低单点压力。

</details>

### 1.7 连接与资源耗尽防护

<HcResourceDiagram />

高并发下最常见的故障不是逻辑错误，而是资源耗尽 — 连接池满、FD 用尽、OOM。

<details>
<summary>展开详情</summary>

**数据库连接耗尽**
- HikariCP 推荐公式：`connections = (core_count * 2) + effective_spindle_count`
- 慢查询长时间占用连接 → 设置 statement timeout
- 事务未提交 / 未关闭连接是常见代码 bug

**线程池耗尽**
- 核心线程数、最大线程数、队列容量需合理配置
- CallerRunsPolicy 看似安全，实则阻塞调用线程，可能拖垮上游
- 不同业务必须使用独立线程池（与 1.4 节呼应）

**文件描述符（FD）限制**
- Linux `ulimit -n` 默认 1024，生产环境需调大至 65535+
- TCP TIME_WAIT 堆积：开启 `tcp_tw_reuse`，缩短 `tcp_fin_timeout`

**内存问题**
- 大对象分配触发 Full GC → STW 导致超时雪崩
- 堆外内存泄漏（Netty ByteBuf 未释放）
- 监控 JVM 各区域使用率，设置合理的 GC 参数

::: danger 警告
对连接数、线程数、FD、内存使用率必须有监控和阈值告警，在耗尽之前预警。资源耗尽是高并发下最常见的致命故障。
:::

</details>

## 第二部分：业务正确性

> 流量扛住了，但如果数据乱了——超卖、重复扣款、订单丢失——比系统崩了更糟。这一部分解决的核心问题是：**高并发下如何保证每一笔业务操作的正确性。**

### 2.1 并发安全——超卖问题与库存扣减

<HcOversellDiagram />

问题本质：经典的"先检查后执行"（Check-Then-Act）竞态条件。多个请求同时读到库存 > 0，各自扣减，导致库存变为负数。

<details>
<summary>展开详情</summary>

**方案一：数据库乐观锁**
`UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0`，利用行锁 + WHERE 条件原子判断。
优点：实现简单；缺点：高并发下大量更新失败，数据库压力大。

**方案二：Redis 原子操作**
使用 Lua 脚本保证"判断 + 扣减"原子性：
```lua
if tonumber(redis.call('get', KEYS[1])) > 0 then // [!code highlight]
    return redis.call('decr', KEYS[1]) // [!code highlight]
else
    return -1
end
```
优点：性能极高（10万+ QPS）；缺点：需解决 Redis 与数据库的最终一致性。

**方案三：预扣库存 + 异步落库**
Redis 预扣 → 生成订单 → MQ 异步扣减数据库库存 → 超时未支付则回补。
适用于秒杀等大流量场景，兼顾性能和一致性。

**方案对比**

| 方案 | 性能 | 一致性 | 复杂度 | 适用场景 |
|------|------|--------|--------|----------|
| 数据库乐观锁 | 低 | 强 | 低 | 并发量不高的常规业务 |
| Redis Lua | 高 | 最终一致 | 中 | 秒杀、抢购 |
| 预扣 + 异步 | 高 | 最终一致 | 高 | 大促、高并发电商 |

</details>

### 2.2 分布式锁

<HcDistLockDiagram />

分布式环境下 JVM 级别的 synchronized / ReentrantLock 失效，需要跨进程的互斥机制。

<details>
<summary>展开详情</summary>

**Redis 分布式锁（Redisson）**
基本原理：`SET key value NX EX timeout`
关键要点：
- value 必须唯一（防误删他人锁）
- 释放时用 Lua 原子判断 value 后再 DEL
- Redisson 看门狗（Watchdog）自动续期，防止业务未完成锁已过期

**ZooKeeper 分布式锁**
临时有序节点 + Watch 机制，CP 模型，适合强一致场景。可靠性高于 Redis，但性能较低。

**常见陷阱**

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 锁过期业务未完成 | GC STW 或慢操作 | Watchdog 自动续期 |
| 主从切换锁丢失 | Redis 异步复制 | RedLock 或换 ZooKeeper |
| 线程堆积 | 未设等待超时 | tryLock 设置合理超时 |
| 误删他人锁 | value 不唯一 | UUID + Lua 原子释放 |

</details>

### 2.3 幂等性设计

<HcIdempotentDiagram />

网络重试、MQ 重复消费、用户重复点击在高并发下是常态。非幂等接口会导致重复扣款、重复下单。

<details>
<summary>展开详情</summary>

**核心原则**：同一操作执行一次和执行多次效果相同 — f(f(x)) = f(x)

**需要幂等的场景**：支付回调、MQ 消费者、创建订单接口、第三方 Webhook

**实现方案**

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| 唯一请求 ID | 前端生成 Token → Redis SETNX 判重 | 表单提交、创建类接口 |
| 数据库唯一约束 | 利用业务唯一键（如订单号）防重 | 订单创建、支付记录 |
| 状态机控制 | 状态单向流转，重复请求检查状态直接返回 | 订单状态变更 |
| 乐观锁版本号 | `WHERE version = ?` + `SET version = version + 1` | 更新类操作 |

**MQ 消费幂等**：消费前查 Redis/DB 是否已处理该 messageId，处理完毕后记录。

</details>

### 2.4 分布式事务与数据一致性

<HcDistTxDiagram />

分库分表、微服务拆分后，本地事务失效，跨服务/跨库的数据一致性是最难的分布式问题之一。

<details>
<summary>展开详情</summary>

**CAP 与 BASE**：分布式系统通常在一致性和可用性间权衡，大多数业务选择最终一致性（BASE）。

**方案对比**

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| 2PC / XA | 强一致 | 低 | 中 | 传统数据库跨库 |
| TCC | 强一致 | 中 | 高 | 资金交易（Try-Confirm-Cancel） |
| SAGA | 最终一致 | 高 | 高 | 长事务流程、编排型业务 |
| 本地消息表 | 最终一致 | 高 | 中 | 跨服务异步通知 |
| 事务消息（RocketMQ） | 最终一致 | 高 | 低 | 异步场景首选 |

**本地消息表流程**：业务操作 + 写消息表放在同一本地事务 → 定时扫描未发送消息 → 发 MQ → 消费端幂等消费。

**TCC 注意事项**：需处理空回滚（Cancel 在 Try 之前到达）和悬挂（Try 在 Cancel 之后到达）问题。

::: tip 建议
90% 的场景用最终一致性即可，不要过度追求强一致。
:::

</details>

## 第三部分：高可用与安全保障

> 单点故障是必然会发生的事。这一部分解决的核心问题是：**挂了用户能不能无感，以及如何在故障发生前就做好准备。**

### 3.1 高可用架构

<HcHaDiagram />

系统不崩溃是底线。高可用回答的核心问题是：**挂了怎么办，挂了用户能不能无感。**

<details>
<summary>展开详情</summary>

**可用性等级**

| 等级 | 年停机时间 | 说明 |
|------|-----------|------|
| 99.9%（3 个 9） | 8.76 小时 | 一般业务 |
| 99.99%（4 个 9） | 52.6 分钟 | 核心交易系统 |
| 99.999%（5 个 9） | 5.26 分钟 | 金融级别 |

**服务层高可用**
- 无状态设计：任意节点可被替换，Session 外置到 Redis（与 1.1 节呼应）
- 健康检查 + 自动摘除：负载均衡器定时探活，故障节点自动从集群摘除
- 优雅关机（Graceful Shutdown）：收到 SIGTERM 后停止接收新请求，处理完存量请求再退出

**数据层高可用**
- MySQL：主从复制 + 半同步，MHA / Orchestrator 自动故障切换
- Redis：Sentinel 哨兵模式（自动主从切换）或 Cluster 模式（数据分片 + 副本）
- 消息队列：Kafka 多副本（ISR 机制），RocketMQ 主从同步

**多机房与异地多活**
- 同城双活：两个机房同时提供服务，数据通过同步复制保持一致
- 异地多活：核心思路是"数据分区 + 就近写入 + 异步同步"，如按用户 ID 分区路由到不同机房
- 关键难点：跨机房数据一致性、流量调度（DNS / GSLB）、故障时的流量切换

**降级与兜底**
- 非核心功能降级：大促时关闭推荐、评论等非核心模块，保障下单支付链路
- 兜底数据：缓存失效时返回静态兜底数据，而非直接报错
- 预案平台：提前配置好各级降级开关，一键执行

</details>

### 3.2 全链路压测与容量规划

<HcPressureDiagram />

不做压测就上大促，等于蒙眼开车。容量规划回答的问题是：**系统能扛多少，瓶颈在哪。**

<details>
<summary>展开详情</summary>

**压测类型**

| 类型 | 目的 | 工具 |
|------|------|------|
| 基准压测 | 确定单接口性能基线 | JMeter / wrk / ab |
| 全链路压测 | 模拟真实流量验证端到端能力 | 阿里 PTS / 自研 |
| 容量探测 | 逐步加压找到系统极限点 | 阶梯加压法 |
| 稳定性压测 | 长时间运行发现内存泄漏等问题 | 持续 2-4 小时恒压 |

**全链路压测关键技术**
- 流量染色：压测请求打标（Header / TraceId 标记），全链路透传
- 影子库 / 影子表：压测数据写入影子存储，不污染线上数据
- Mock 外部依赖：支付、短信等第三方接口用 Mock 替代
- 流量构造：基于线上日志回放或按业务模型生成仿真流量

**容量规划公式**
```
所需机器数 = 峰值 QPS / 单机极限 QPS × 冗余系数（通常 1.5~2）
```
峰值 QPS 通常取日常峰值的 3~5 倍（应对突发流量和大促）。

**压测必看指标**：QPS / TPS、P99 响应时间、错误率、CPU / 内存 / 网络 IO、GC 频率与耗时。

::: warning 注意
上线前必须压测，扩容后必须回归压测，每次大促前全链路压测至少一轮。
:::

</details>

### 3.3 可观测性

<HcObservabilityDiagram />

系统出问题不可怕，可怕的是出了问题找不到原因。可观测性是高并发系统的"眼睛"。

<details>
<summary>展开详情</summary>

**三大支柱**

| 支柱 | 作用 | 常用工具 |
|------|------|----------|
| Metrics（指标） | 量化系统状态，告警触发 | Prometheus + Grafana |
| Tracing（链路追踪） | 追踪请求在微服务间的调用链 | SkyWalking / Zipkin / Jaeger |
| Logging（日志） | 记录详细的运行时信息 | ELK（Elasticsearch + Logstash + Kibana） |

**Metrics 关键指标**
- RED 方法（面向服务）：Rate（请求速率）、Errors（错误率）、Duration（响应耗时）
- USE 方法（面向资源）：Utilization（使用率）、Saturation（饱和度）、Errors（错误数）

**链路追踪核心价值**
- 一个请求跨 10 个微服务，哪个环节慢了一目了然
- TraceId 全链路透传，日志、Metrics、Trace 三者通过 TraceId 关联

**告警设计**
- 分级告警：P0（立即响应） / P1（15 分钟内） / P2（工作时间处理）
- 避免告警疲劳：合理设置阈值和沉默期，告警必须可操作（收到告警后知道该做什么）

</details>

### 3.4 灰度发布与流量调度

<HcCanaryDiagram />

高并发系统最怕的不是 bug，而是一次性把 bug 推给所有用户。灰度发布让变更逐步生效，控制爆炸半径。

<details>
<summary>展开详情</summary>

**发布策略**

| 策略 | 原理 | 适用场景 |
|------|------|----------|
| 金丝雀发布 | 先切 1%~5% 流量到新版本，观察无异常后逐步放量 | 常规迭代发布 |
| 蓝绿部署 | 新旧版本同时部署，流量一键切换 | 需要快速回滚的场景 |
| AB 测试 | 按用户特征分流，对比不同版本的业务指标 | 功能实验、算法优化 |
| 滚动更新 | 逐台替换实例，始终保持部分可用 | K8s 默认策略 |

**流量调度手段**
- 网关层：Nginx / Kong / Spring Cloud Gateway 按权重、Header、用户 ID 分流
- 服务网格：Istio VirtualService 实现精细化流量规则
- DNS 层：GSLB 做机房级别流量切换

**灰度发布核心流程**
1. 部署新版本（不接流量）
2. 切入小比例流量（1%~5%）
3. 观察 Metrics / 日志 / 告警（至少 10~30 分钟）
4. 无异常则逐步放量（10% → 50% → 100%）
5. 发现问题则立即回滚（流量切回旧版本）

::: tip 建议
任何变更都应该可灰度、可监控、可回滚。
:::

</details>

### 3.5 安全防护——防刷与风控

<HcAntiCheatDiagram />

高并发系统面对的不仅是正常用户，还有恶意爬虫、黄牛脚本、CC 攻击。没有安全防护，限流和缓存形同虚设。

<details>
<summary>展开详情</summary>

**接口防刷（多层防线）**

| 层级 | 手段 | 说明 |
|------|------|------|
| 客户端 | 按钮防重、前端加密签名 | 防君子不防小人，但能拦大部分误操作 |
| 网关层 | IP 限频、用户维度限频 | Nginx `limit_req` / Redis 滑动窗口计数 |
| 业务层 | 设备指纹、行为分析 | 识别同设备多账号、异常操作频率 |

**验证码体系**（安全性递增）
1. 图形验证码 — 基础防护，易被 OCR 破解
2. 滑块验证 — 增加破解成本
3. 行为验证（无感验证） — 基于鼠标轨迹、操作行为判断人机，体验最好
4. 风险自适应 — 正常用户无感通过，可疑用户弹出更强验证

**防重放攻击**
请求携带 `timestamp + nonce + sign`，服务端校验：
- 时间戳超过 5 分钟的请求直接拒绝
- nonce 用 Redis 判重（过期时间与时间窗口一致）
- sign = HMAC(参数排序 + timestamp + nonce, secretKey)

**风控引擎**
- 规则引擎（Drools / Aviator）：配置化风控规则，如"同 IP 1 分钟下单超过 10 次"
- 实时特征计算（Flink）：滑动窗口聚合用户行为特征
- 风险评分模型：对每笔请求打分，高风险拦截或增加验证

**CC 攻击防护**：WAF 层拦截 + 人机识别 + IP 信誉库 + 动态封禁

</details>

## 第四部分：分布式基础设施

> 前三部分讲的是"做什么"，这一部分讲的是"在什么环境下做"。微服务治理、ID 生成、一致性哈希、网络优化、配置中心、弹性伸缩——这些是支撑前面所有设计的底座。

### 4.1 微服务治理——服务注册发现与 API 网关

<HcMicroserviceDiagram />

前面多处提及微服务（熔断、分布式事务、链路追踪），但微服务的基础设施——服务如何找到彼此、流量如何统一管理——是一切的前提。

<details>
<summary>展开详情</summary>

**服务注册与发现**

| 组件 | 一致性模型 | 健康检查 | 适用场景 |
|------|-----------|----------|----------|
| Nacos | AP/CP 可切换 | 心跳 + 主动探测 | 国内微服务首选，兼顾配置中心 |
| Eureka | AP | 心跳续约 | Spring Cloud 生态 |
| Consul | CP (Raft) | 多种检查方式 | 跨语言、Service Mesh |
| ZooKeeper | CP (ZAB) | 临时节点 | 强一致场景（但不推荐做注册中心） |

**核心流程**：服务启动 → 注册到注册中心 → 消费者订阅服务列表 → 注册中心推送变更 → 消费者本地缓存 + 负载均衡调用

**API 网关**

网关是高并发系统的统一入口，承担多种职责：

| 职责 | 说明 | 与现有章节的关系 |
|------|------|-----------------|
| 路由转发 | 根据路径/Header 将请求路由到对应微服务 | — |
| 限流 | 全局限流 + 租户限流 + 接口限流 | 1.5 节 |
| 鉴权 | JWT 校验、OAuth2、API Key | — |
| 熔断降级 | 后端故障时快速失败或返回兜底 | 1.5 节 + 3.1 节 |
| 协议转换 | HTTP → gRPC、WebSocket 代理 | — |
| 请求聚合 | 将多个微服务调用合并为一个接口返回 | 减少前端请求数 |

**网关选型**

| 网关 | 语言 | 性能 | 生态 |
|------|------|------|------|
| Kong | Lua (OpenResty) | 高 | 插件丰富，企业级 |
| APISIX | Lua (OpenResty) | 极高 | 国内活跃，动态路由 |
| Spring Cloud Gateway | Java | 中 | Spring 生态无缝集成 |
| Envoy | C++ | 极高 | Service Mesh 标配（Istio 数据面） |

**服务拆分原则**
- 按业务域拆分（DDD 限界上下文），而非按技术层拆分
- 避免拆得过细：服务间调用链过长会增加延迟和故障概率
- 数据库不共享：每个服务独立数据库，跨服务数据通过接口或消息同步
- 拆分判据：团队规模（两个披萨原则）、独立部署需求、性能隔离需求

</details>

### 4.2 分布式 ID 生成

<HcDistIdDiagram />

分库分表后数据库自增 ID 失效，需要全局唯一、趋势递增、高性能的 ID 生成方案。这是分库分表（1.3 节）的**前置依赖**。

<details>
<summary>展开详情</summary>

**方案对比**

| 方案 | 有序性 | 性能 | 可用性 | 复杂度 | 适用场景 |
|------|--------|------|--------|--------|----------|
| UUID | 无序 | 极高 | 极高 | 低 | 日志 TraceId（不适合做 DB 主键，索引效率差） |
| 数据库自增 | 有序 | 低 | 低（单点） | 低 | 单库小规模系统 |
| 号段模式（Leaf-segment） | 趋势递增 | 高 | 高 | 中 | 通用业务 ID |
| 雪花算法（Snowflake） | 趋势递增 | 极高 | 高 | 中 | 高并发场景首选 |

**雪花算法结构（64 位）**
```
0 - 41位时间戳 - 10位机器ID - 12位序列号
```
- 41 位时间戳：支持约 69 年
- 10 位机器 ID：支持 1024 个节点
- 12 位序列号：每毫秒每节点可生成 4096 个 ID
- 单机 QPS：理论上限 409.6 万/秒

**关键问题：时钟回拨**
- 机器时间回退会导致 ID 重复
- 解决方案：等待时钟追上（小幅回拨）、报错拒绝（大幅回拨）、预留扩展位容忍
- 美团 Leaf 采用 ZooKeeper 检测时钟回拨，百度 UidGenerator 使用环形缓冲区预生成

**号段模式原理**
数据库维护当前最大 ID → 应用启动时批量获取一段（如 1000 个）缓存到内存 → 用完再取下一段。双 Buffer 预加载避免取号段时的短暂阻塞。

</details>

### 4.3 一致性哈希

<HcConsistHashDiagram />

分库分表、缓存集群都依赖哈希分片路由。普通取模在节点变更时代价极大，一致性哈希是标准解法。

<details>
<summary>展开详情</summary>

**普通取模的问题**
`hash(key) % N`，当 N 变为 N+1 时，约 `N/(N+1)` 的 key 需要迁移。例如 10 个节点变 11 个，约 91% 的数据需要重新分布。

**一致性哈希原理**
1. 将哈希空间组成一个环（0 ~ 2^32-1）
2. 节点通过哈希映射到环上的某个位置
3. key 哈希后顺时针找到最近的节点
4. 节点增删时，只影响环上相邻区间的数据，迁移量最小化

**数据倾斜与虚拟节点**
物理节点少时，哈希分布不均导致某些节点负载过重。解决方案：每个物理节点映射 100~200 个虚拟节点到环上，使数据分布更均匀。

**实际应用**

| 场景 | 实现方式 |
|------|----------|
| Redis Cluster | 16384 个固定槽位，手动/自动分配给节点 |
| Nginx 负载均衡 | `consistent_hash` 指令，按请求参数路由到固定后端 |
| 分库分表路由 | 按分片键哈希到固定槽位，槽位映射到物理库表 |
| Memcached 集群 | 客户端一致性哈希，节点增删只迁移 1/N 数据 |

**跳跃一致性哈希（Jump Consistent Hash）**
Google 提出的轻量算法，O(ln N) 时间复杂度，分布均匀度优于环形一致性哈希，但只支持节点追加不支持任意删除，适合节点编号连续的场景。

</details>

### 4.4 网络层优化

<HcNetworkDiagram />

在高并发场景下，网络往往是被忽视的瓶颈。一次 TCP 握手 1~2ms，乘以百万请求就是巨大的开销。

<details>
<summary>展开详情</summary>

**连接策略**

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 短连接 | 每次请求新建连接，用完关闭 | 低频调用 |
| 长连接 (Keep-Alive) | 连接复用，减少握手开销 | HTTP 服务间调用 |
| 连接池 | 预建连接 + 复用 + 健康检查 | 数据库、Redis、RPC |
| 多路复用 | 单连接并发多个请求 | HTTP/2、gRPC |

**协议选择**

| 协议 | 序列化 | 性能 | 适用场景 |
|------|--------|------|----------|
| HTTP/1.1 + JSON | 文本 | 低 | 对外 API、兼容性优先 |
| HTTP/2 | 二进制帧 | 中 | 浏览器通信、多路复用 |
| gRPC (Protobuf + HTTP/2) | 二进制 | 高 | 微服务内部通信首选 |
| Dubbo (Hessian2) | 二进制 | 高 | Java 微服务生态 |

**序列化性能差异**
同一对象序列化：JSON ~500ns、Protobuf ~100ns、Kryo ~80ns。高并发场景下序列化占 CPU 时间可达 30%+，选择高效序列化格式能显著提升吞吐。

**网络参数调优**
- `TCP_NODELAY`：关闭 Nagle 算法，减少小包延迟（RPC 场景必开）
- `SO_REUSEPORT`：允许多进程绑定同一端口，内核级负载均衡
- TCP 缓冲区：`net.core.rmem_max` / `net.core.wmem_max` 根据带宽调整
- 连接队列：`net.core.somaxconn` 调大至 65535，避免连接被拒绝

**CDN 与边缘计算**
- 静态资源（JS/CSS/图片）走 CDN，减少源站压力
- 边缘节点预处理：参数校验、鉴权等逻辑下沉到 CDN 边缘（如 Cloudflare Workers）

</details>

限流阈值、降级开关、线程池参数、灰度规则——这些都需要不重启即时生效。配置中心是高并发系统的"遥控器"。

<summary>展开详情</summary>

**核心能力**

### 4.6 延迟队列与定时任务

<HcDelayQueueDiagram />

1.4 节讲了消息队列削峰，但"延迟执行"是另一个高频场景——订单超时取消、延迟重试、定时推送。

<details>
<summary>展开详情</summary>

**典型场景**
- 订单 30 分钟未支付自动取消
- 失败重试指数退避（1s → 2s → 4s → 8s）
- 会员到期提醒、优惠券过期通知

**方案对比**

| 方案 | 精度 | 可靠性 | 吞吐量 | 适用场景 |
|------|------|--------|--------|----------|
| Redis ZSet | 秒级 | 中（需自行保障） | 高 | 轻量延迟任务 |
| RocketMQ 延迟消息 | 固定级别 | 高 | 高 | 与 MQ 生态集成 |
| RabbitMQ 死信队列 (DLX) | 消息级 | 高 | 中 | RabbitMQ 技术栈 |
| 时间轮 (HashedWheelTimer) | 毫秒级 | 低（内存级） | 极高 | 进程内定时器（Netty/Kafka） |

**Redis ZSet 实现**
```
# 添加延迟任务：score = 执行时间戳
ZADD delay_queue <execute_timestamp> <task_data>

# 轮询获取到期任务
ZRANGEBYSCORE delay_queue 0 <current_timestamp> LIMIT 0 100
```
注意：需分布式锁防止多消费者重复消费，或用 Lua 脚本原子取出并删除。

**分布式定时任务调度**

| 框架 | 特点 |
|------|------|
| XXL-JOB | 轻量、Web 控制台、分片广播 |
| Elastic-Job | 基于 ZooKeeper，弹性扩缩 |
| PowerJob | 支持 MapReduce、DAG 工作流 |

**可靠性保障**：任务必须幂等（与 2.3 节呼应），支持失败重试和执行日志追溯。

</details>

### 4.7 数据倾斜问题

<HcSkewDiagram />

分片后数据分布不均，导致某些节点过载而其他节点空闲，是分库分表和缓存集群的经典隐患。

<details>
<summary>展开详情</summary>

**常见倾斜类型**

| 类型 | 表现 | 原因 |
|------|------|------|
| 分库分表倾斜 | 某些库/表数据量远大于其他 | 分片键选择不当（如按日期分片，热门日期数据暴增） |
| Redis 大 Key | 单个 key 占用内存过大，影响该节点性能 | Hash/Set/List 元素过多（如百万粉丝列表） |
| MQ 分区倾斜 | 某些 partition 消息堆积 | 相同 key 的消息集中在同一分区 |
| 计算倾斜 | MapReduce / Spark 某些 task 特别慢 | 数据分布不均导致个别 task 处理量远超其他 |

**检测手段**
- 各分片数据量 / QPS 监控对比（Grafana 面板）
- Redis `MEMORY USAGE key`、`DEBUG OBJECT key` 检测大 key
- `redis-cli --bigkeys` 扫描集群大 key
- Kafka `kafka-consumer-groups.sh` 查看各 partition lag

**解决方案**
- 分片键优化：选择散列度高的字段（如用户 ID），避免用时间、地区等倾斜字段
- 二次哈希：对倾斜严重的 key 追加随机后缀再哈希
- 大 key 拆分：将一个大 Hash 拆为多个小 Hash（如 `user:fans:0` ~ `user:fans:9`）
- MQ rebalance：手动调整 partition 分配，或增加 partition 数量
- 热点与倾斜的区别：热点是访问频率高（1.6 节），倾斜是数据分布不均，两者可能同时存在

</details>

### 4.8 容器化与弹性伸缩

<HcElasticDiagram />

水平扩展（1.1 节）讲了原则，容器化和弹性伸缩是落地手段。在云原生时代，"扛住流量洪峰"依赖自动化的扩缩容。

<details>
<summary>展开详情</summary>

**容器化基础**
- Docker：应用 + 依赖打包为镜像，环境一致性、秒级启动
- Kubernetes：容器编排，管理成百上千个容器的部署、扩缩、自愈

**弹性伸缩策略**

| 策略 | 触发条件 | 响应速度 | 适用场景 |
|------|----------|----------|----------|
| HPA（水平 Pod 自动伸缩） | CPU / 内存 / 自定义指标超阈值 | 分钟级 | 常规流量波动 |
| 定时 HPA (CronHPA) | 预设时间点 | 提前扩容 | 可预期的流量高峰（如早高峰、大促） |
| VPA（垂直伸缩） | 资源使用率分析 | 需重启 Pod | 单 Pod 资源不足 |
| KEDA（事件驱动伸缩） | MQ 堆积量、HTTP 请求率 | 分钟级 | 消费者按队列深度伸缩 |

**冷启动问题**
新 Pod 启动到可服务需要时间（JVM 预热、连接池初始化、缓存加载），流量打到未就绪实例会导致超时。
- Readiness Probe：Pod 就绪前不接流量
- 预热机制：启动后逐步增加流量权重（如从 10% 逐步到 100%）
- 提前扩容：大促前按预估量提前扩好，而非等流量来了再扩

**Serverless / FaaS**
- 函数计算（如 AWS Lambda、阿里云 FC）：按调用次数计费，天然弹性
- 适合突发、低频、事件驱动的场景（如图片处理、Webhook 回调）
- 不适合长连接、有状态服务

**成本优化**
- 混合部署：基线流量用预留实例，峰值增量用 Spot / 竞价实例（成本降 60%~80%）
- 资源配额：设置 request / limit 防止单个服务吃光集群资源
- 自动缩容：低谷期缩到最小副本数，避免资源浪费

</details>
