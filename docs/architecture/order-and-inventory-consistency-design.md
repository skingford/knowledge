## 为什么订单和库存总是绑在一起讲

在交易系统里，最经典的一条主链路往往就是：

- 用户下单
- 系统创建订单
- 系统扣减库存
- 后续支付、取消、退款再继续影响库存和订单状态

这里最难的地方在于：

- 订单和库存都很重要
- 两者都可能被高并发同时修改
- 它们往往又不一定在同一个库、同一个服务里

如果处理不当，最常见的后果就是：

- 超卖
- 少卖
- 订单创建成功但库存没扣
- 库存扣了但订单没创建
- 订单取消了但库存没回滚

所以“库存扣减与订单创建一致性”本质上并不是一个单点技术问题，而是一个完整的交易链路设计问题。

## 先明确你到底要解决什么问题

很多讨论一上来就问：

- 库存到底怎么扣？

但真正应该先问的是：

- 你是在防超卖，还是在做强一致？
- 你是普通电商场景，还是秒杀场景？
- 你是单库单服务，还是订单和库存已经拆成独立系统？
- 你要求“绝不能多卖一件”，还是允许短暂不一致后再补偿？

如果这些前提不先明确，方案讨论很容易失焦。

更准确地说，这类设计通常同时要处理四件事：

- 高并发下不能超卖
- 订单与库存状态最终要一致
- 失败后要能恢复
- 正常路径不能太重，不能把系统自己压垮

## 最简单的情况：单库事务

如果订单表和库存表还在同一个数据库里，那么最优先的做法通常不是上 MQ、上 Saga、上 TCC，而是先用好本地事务。

最常见的单库方案是：

1. 在事务里扣减库存
2. 在同一个事务里写订单
3. 成功后提交

例如：

```sql
UPDATE inventory
SET available = available - 1,
    sold = sold + 1
WHERE sku_id = ?
  AND available >= 1;
```

如果扣减成功，再继续写订单。

### 为什么这个方案最稳

因为在单库里：

- 数据一致性最容易保证
- 失败可以直接回滚
- 不需要额外引入分布式事务复杂度

所以一条很重要的工程原则是：

> 能用单库事务解决的问题，不要过早升级成分布式一致性问题。

## 一旦拆成订单服务和库存服务会发生什么

一旦订单和库存被拆成两个服务、两个库，问题就变了。

此时你已经没法用一个本地事务同时覆盖：

- 订单创建
- 库存扣减

最典型的两种错误写法是：

### 1. 先创建订单，再远程扣库存

问题：

- 订单已经创建成功
- 但库存服务调用失败

结果：

- 产生无效订单
- 后续还要补偿关闭或回滚

### 2. 先远程扣库存，再创建订单

问题：

- 库存已经扣掉
- 但订单服务落库失败

结果：

- 库存被“吃掉”
- 需要额外补偿释放

这就是订单和库存在分布式场景下的核心矛盾：

> 无论先做哪边，另一边失败时都需要补偿收敛。

## 常见方案总览

| 方案 | 核心思路 | 优点 | 缺点 | 更适合什么场景 |
| --- | --- | --- | --- | --- |
| 单库事务 | 同一事务内扣库存 + 写订单 | 最简单、最强一致 | 只适用于同库 | 单体或未拆库场景 |
| 本地事务 + Outbox | 主库先提交，再异步推动下游一致 | 工程上成熟、适合最终一致性 | 需要补偿和幂等 | 互联网常见交易系统 |
| TCC / 预扣减冻结 | Try 预留库存，Confirm 扣减，Cancel 释放 | 一致性更强，表达清晰 | 侵入业务，复杂度高 | 核心库存、强一致交易 |
| Saga / 事件驱动补偿 | 每步本地成功，失败时逆向补偿 | 易扩展，适合微服务 | 存在短暂不一致 | 允许最终一致的链路 |
| Redis 预扣减 + 异步落库 | 先在缓存里快速扣减，再异步写库 | 吞吐高，适合秒杀 | 复杂度高，补偿要求高 | 大促、秒杀、高峰抢购 |

## 如何防止超卖

库存一致性里最基础、也最容易被忽略的一件事是：

- 不管你整体架构多复杂
- 最终都必须有一层真正的扣减屏障

最常见的数据库做法是条件更新：

```sql
UPDATE inventory
SET available = available - ?,
    sold = sold + ?
WHERE sku_id = ?
  AND available >= ?;
```

它的价值很直接：

- 库存不足时不会扣成负数
- 并发请求下只有满足条件的线程能成功
- `affected_rows = 0` 就可以明确判定扣减失败

### 为什么不能先查库存再扣减

错误写法通常是：

```text
SELECT available FROM inventory WHERE sku_id = ?
if available >= 1:
    UPDATE inventory SET available = available - 1 WHERE sku_id = ?
```

在高并发下，多个请求可能同时查到库存足够，然后同时去扣，结果自然就超卖了。

所以防超卖的第一原则是：

> 判断和扣减不能分开，必须合并到一条原子更新里。

## 预扣减冻结库存为什么常见

很多交易系统并不会在“下单成功”那一刻就把库存彻底卖掉，而是更倾向于：

- 先冻结库存
- 支付成功后再确认扣减
- 超时取消或支付失败时再释放库存

这就是典型的“预扣减 / 冻结库存”思路。

例如库存表可以拆成：

- `available`
- `frozen`
- `sold`

下单时：

```sql
UPDATE inventory
SET available = available - 1,
    frozen = frozen + 1
WHERE sku_id = ?
  AND available >= 1;
```

支付成功时：

```sql
UPDATE inventory
SET frozen = frozen - 1,
    sold = sold + 1
WHERE sku_id = ?
  AND frozen >= 1;
```

订单取消时：

```sql
UPDATE inventory
SET frozen = frozen - 1,
    available = available + 1
WHERE sku_id = ?
  AND frozen >= 1;
```

### 这种模型的优点

- 更贴近真实交易流程
- 更适合“下单占库存、支付再确认”的业务语义
- 可以更自然地处理超时取消和支付失败

### 它的代价

- 状态更多
- 补偿链路更长
- 对订单状态机和库存状态机协同要求更高

## Outbox 在这里起什么作用

一旦订单和库存不在同一个库里，Outbox 就非常自然地进入方案里。

典型链路可以是：

1. 订单服务本地事务里创建订单，并写一条 `OrderCreated` 到 Outbox
2. 投递器把 `OrderCreated` 发到 MQ
3. 库存服务消费后执行库存冻结或扣减
4. 成功后再发 `StockReserved`
5. 订单服务消费后更新订单状态

或者反过来：

1. 库存服务先冻结库存并写 Outbox
2. 发送 `StockReserved`
3. 订单服务创建订单
4. 如果失败，再发补偿事件释放库存

这两种设计没有绝对统一答案，但都说明一件事：

- Outbox 不是解决业务规则本身
- 而是解决“本地事务成功了，事件不能丢”这个问题

## 补偿链路怎么设计

库存和订单一旦拆开，补偿就不再是“可选优化”，而是主方案的一部分。

常见补偿场景包括：

- 订单创建成功，但库存冻结失败 -> 关闭订单
- 库存冻结成功，但订单创建失败 -> 释放库存
- 支付成功，但确认扣减失败 -> 重试或人工介入
- 超时取消成功，但库存释放失败 -> 补偿任务继续释放

比较稳妥的工程做法通常是：

- 正常路径异步化
- 补偿路径事件化
- 最终再保留兜底扫描

例如定时扫描：

- 长时间处于 `PENDING_STOCK`
- 长时间处于 `FROZEN`
- 长时间处于 `REFUNDING`

这类异常状态，再继续补偿收敛。

## 高并发下还要考虑哪些问题

库存问题之所以麻烦，不只是因为一致性，还因为它经常天然带热点。

### 1. 热点 SKU

大促时，某个 SKU 会成为超级热点，导致：

- 同一行库存频繁更新
- 行锁竞争加重
- 数据库写入放大

常见优化方向：

- 预热库存到 Redis
- 分段库存 / 分桶库存
- 异步削峰
- 抢购资格先过滤

### 2. 重复请求

用户连点、重试、网络抖动都可能导致同一请求重复提交。

因此必须有：

- 幂等单号
- 防重复下单
- 订单维度锁或唯一键

### 3. 消息重复

只要用了 MQ，就要默认：

- 库存冻结事件可能重复
- 释放库存事件可能重复

所以库存服务也必须幂等。

### 4. 惊群和回滚风暴

大促结束或大批订单超时取消时，可能会瞬间产生大量库存释放请求。

这时候要考虑：

- 限流
- 批量释放
- 线程池隔离
- 事件抖动和错峰

## 面试回答模板

如果面试官问“订单创建和库存扣减怎么保证一致性”，可以用下面这版口径来回答：

> 这个问题要先看订单和库存是不是还在同一个库里。如果还在同一个库，最优先的方案通常是本地事务，在一个事务里用条件更新扣减库存，再写订单，这样最直接也最稳。  
> 如果已经拆成订单服务和库存服务，那就变成典型的分布式一致性问题。此时无论先创建订单还是先扣库存，另一边失败时都需要补偿。  
> 工程上比较常见的是两类方案：一种是最终一致性方案，比如本地事务 + Outbox + MQ，把事件可靠发出去，再通过幂等和补偿收敛；另一种是更强一些的 Try-Confirm-Cancel 或冻结库存模型，先冻结库存，支付成功后再确认扣减，超时取消或失败时再释放。  
> 不管上层方案怎么选，真正防止超卖的底层手段都应该是数据库条件更新，比如 `UPDATE inventory ... WHERE available >= ?`，不能先查再扣。  
> 最后还要补幂等、超时取消、库存释放和异常状态扫描，否则主路径一旦失败，库存和订单还是会长期不一致。

如果继续追问，可以展开：

1. 为什么不能先查库存再扣减
2. 为什么订单和库存拆服务后一定要有补偿
3. 为什么冻结库存比直接 sold 更适合支付前场景
4. 为什么 Outbox 只解决“消息别丢”，不解决业务规则本身

## 落地检查清单

### 1. 数据模型

- 是否明确区分 `available / frozen / sold`
- 是否对热点库存表建立了合理索引
- 是否有订单状态和库存状态协同定义

### 2. 防超卖

- 是否使用条件更新原子扣减
- 是否避免了“先查再扣”
- 是否有库存不足时的明确失败语义

### 3. 一致性

- 单库是否优先使用本地事务
- 分布式是否明确采用 Outbox / Saga / TCC 中哪一种
- 是否定义了失败补偿路径

### 4. 幂等与重复

- 是否防重复下单
- 是否对库存冻结、释放、确认都做了幂等
- 是否对消息重复消费有处理方案

### 5. 高并发治理

- 是否识别热点 SKU
- 是否有削峰、限流、隔离、错峰策略
- 是否考虑大批取消时的释放风暴

### 6. 恢复能力

- 是否有异常状态扫描和补偿任务
- 是否能定位“订单成功库存失败”或“库存成功订单失败”
- 是否有链路监控、重试和人工介入能力

## 结论

库存扣减与订单创建一致性设计，真正难的从来不是“扣减 SQL 怎么写”，而是：

- 正常路径怎么高效
- 并发下怎么不超卖
- 失败后怎么补回来
- 服务拆分后怎么最终收敛

所以最值得记住的一句话是：

> 库存方案选型不只是“扣不扣得掉”，而是“能不能不超卖、失败后能不能恢复、系统高峰时还能不能扛住”。
