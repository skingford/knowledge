## 风控与反欺诈的区别

风控和反欺诈经常被混为一谈，但它们的定位和方法论有本质差异：

| 维度 | 风控（Risk Control） | 反欺诈（Fraud Detection） |
| --- | --- | --- |
| 核心目标 | 合规 + 风险管控 | 识别并阻止欺诈行为 |
| 驱动模式 | 规则驱动、政策驱动 | 对抗驱动、数据驱动 |
| 时间视角 | 交易发起时决策 | 事前 + 事中 + 事后全链路 |
| 典型手段 | 限额限频、KYC、黑名单 | 行为分析、模型评分、关联图谱 |
| 对手 | 不合规行为（可能非恶意） | 有组织的欺诈者（主动对抗） |
| 迭代速度 | 季度级调整 | 周级甚至日级迭代 |
| 详细设计 | [支付交易风控设计](/architecture/payment/risk-control) | 本文 |

::: tip 一句话总结
风控回答"这笔交易是否符合规定"，反欺诈回答"这笔交易背后的人是否在骗钱"。
:::

反欺诈的核心难点在于**对抗性**——欺诈者会持续调整策略来绕过检测。这意味着：

- 规则不能一成不变，需要持续迭代
- 仅靠规则不够，需要模型来捕获未知模式
- 需要事后分析能力，发现新的欺诈手法并快速响应

## 支付场景常见欺诈类型

### 盗卡交易（Card-Not-Present Fraud）

**特征**：使用被盗的银行卡信息（卡号、CVV、有效期）进行线上支付。

- 来源：数据泄露、钓鱼网站、暗网购买
- 典型行为：短时间多笔交易、收货地址与持卡人不符、首次使用该卡

**识别要点**：
- 卡 BIN 与 IP 地理位置不匹配
- 同一设备短时间内尝试多张卡
- 交易金额与用户历史消费模式偏差大

### 试卡攻击（Card Testing）

**特征**：用大量卡号发起小额交易（通常 $0.01~$1），测试哪些卡号是有效的。

- 有效的卡号随后会用于更大金额的盗刷
- 通常通过自动化脚本批量发起

**识别要点**：
- 同一 IP/设备短时间大量小额交易
- 大量交易失败（无效卡号）
- 交易金额异常集中在极小值

### 账户接管（Account Takeover / ATO）

**特征**：攻击者通过撞库、社工、钓鱼等方式获取用户账号权限，冒充用户操作。

- 登录后可能修改密码、绑定新设备、转移余额
- 常见于有余额或绑卡的支付账户

**识别要点**：
- 登录设备/IP/地理位置突变
- 登录后立即执行敏感操作（改密码、加新收款人、大额转账）
- 异常时段登录

### 友好欺诈（Friendly Fraud / Chargeback Fraud）

**特征**：持卡人自己完成了交易，但事后向发卡行申请拒付（Chargeback），声称"未授权"或"未收到商品"。

- 最难检测的欺诈类型，因为交易本身是"正常"的
- 在数字商品（游戏充值、虚拟货币）场景尤其常见

**识别要点**：
- 同一用户多次发起拒付
- 数字商品交易后短时间内申请退款
- 高价值交易完成后立即更换设备/IP

### 薅羊毛（Promo Abuse / Bonus Abuse）

**特征**：利用多账号、虚假身份或脚本批量获取优惠、红包、返现。

- 与秒杀防刷有重叠，但支付场景还涉及返现、首单优惠等
- 养号 → 批量注册 → 领取优惠 → 变现

**识别要点**：
- 同一设备/IP 关联大量账号
- 新账号集中使用优惠后不再活跃
- 收货地址、联系方式重复或高度相似

### 洗钱与资金转移（Money Mule）

**特征**：利用支付系统作为资金清洗的通道。

- "钱骡"账户接收非法资金后快速分散转出
- 常见模式：大额充值 → 分散转账 → 提现

**识别要点**：
- 资金快进快出，停留时间极短
- 多个账户之间形成环状或链状转账
- 金额故意拆分到大额报告阈值以下（化整为零 / structuring）

### 商户欺诈（Merchant Fraud）

**特征**：商户自身参与欺诈，如虚假交易刷单、利用退款漏洞套现。

- 自买自卖产生流水
- 与客户串通制造虚假退款

**识别要点**：
- 商户交易对手高度集中
- 退款率异常偏高
- 交易时间分布不自然（如凌晨集中交易）

## 一条推荐的反欺诈链路

<InlineSvg src="/images/architecture/payment/fraud-detection.svg" alt="支付反欺诈全链路设计" />

反欺诈不是只在"交易发起时"做判断，而是覆盖全生命周期：

### 事前：准入与基线

| 环节 | 做什么 | 目标 |
| --- | --- | --- |
| 用户注册 | 设备指纹采集、手机号校验、IP 风险评估 | 阻止批量注册 |
| KYC 认证 | 身份验证、人脸比对 | 确认用户身份真实性 |
| 行为基线 | 建立用户正常行为画像 | 后续对比的基准 |
| 商户准入 | 营业执照、行业审核、历史记录 | 过滤高风险商户 |

### 事中：实时决策

```
交易请求 → 特征提取 → 规则匹配 + 模型评分 → 综合决策
                                                │
                         ┌──────────────────────┤
                         ▼                      ▼                      ▼
                      低风险                  中风险                  高风险
                      直接通过              增强验证                  拦截
                                      (3DS / 短信 / 人脸)         (人工审核)
```

**关键设计点**：

- 特征提取需要在 **50ms 以内** 完成，否则影响交易体验
- 规则和模型并行执行，综合两者结果做最终决策
- 中风险的"增强验证"是核心体验设计——不是拒绝，而是增加一道确认

### 事后：分析与闭环

| 环节 | 做什么 | 目标 |
| --- | --- | --- |
| 交易监控 | 实时监控欺诈指标、异常波动 | 快速发现新型攻击 |
| 拒付处理 | 接收 Chargeback、举证应诉 | 减少资损和罚款 |
| 案件调查 | 关联分析、溯源定位 | 发现欺诈网络 |
| 策略迭代 | 新增规则、模型重训、黑名单更新 | 持续对抗升级 |
| 情报共享 | 与同行、卡组织共享欺诈情报 | 行业联防 |

## 特征工程与评分模型

### 实时特征

实时特征在交易发生时即时计算，用于实时决策：

| 特征类别 | 具体特征 | 说明 |
| --- | --- | --- |
| 速度特征 | 最近 1h/24h 交易笔数 | 短时间大量交易可能是盗刷 |
| 金额特征 | 本次金额与历史均值的偏差 | 突然大额消费 |
| 地理特征 | 两次交易之间的地理跳跃速度 | 不可能的旅行（1 小时内跨城市） |
| 设备特征 | 当前设备是否为新设备 | 新设备 + 大额 → 高风险 |
| 时间特征 | 是否在异常时段（凌晨 2-5 点） | 非正常时段交易 |
| 收款特征 | 收款方是否为首次交易对象 | 首次向未知账户大额转账 |

### 离线特征

离线特征通过批处理或流处理提前计算，存入特征库：

| 特征类别 | 具体特征 | 更新频率 |
| --- | --- | --- |
| 用户画像 | 历史消费水平、常用设备、常用地点 | 日级 |
| 行为序列 | 最近 30 天的交易模式 | 日级 |
| 关联图谱 | 设备-账户-IP-收款方关联关系 | 小时级 |
| 社交网络 | 与已确认欺诈账户的关联度 | 小时级 |
| 商户画像 | 退款率、拒付率、交易时间分布 | 日级 |

### 模型架构

```
实时特征 ──┐
           ├──→ 特征向量 → 在线模型 → 风险分 (0~100)
离线特征 ──┘                  │
                              ▼
                        风险分 + 规则 → 最终决策
```

**关键决策点**：

| 问题 | 推荐方案 |
| --- | --- |
| 模型选择 | 初期用 LightGBM / XGBoost，效果好且可解释 |
| 训练数据 | 已确认的欺诈案例 + 正常交易（注意类别不平衡） |
| 更新频率 | 每周或每两周重训一次 |
| 冷启动 | 新用户缺少历史数据时回退到规则兜底 |
| 模型上线 | A/B 切流验证，新模型先 shadow 再切量 |

### 规则与模型协同

- **规则做硬性兜底**：确定性高的场景（如黑名单命中）直接用规则拦截，不走模型
- **模型做精细化补充**：规则覆盖不到的模式由模型识别
- **模型反哺规则**：模型发现的新模式可以转化为规则，提高响应速度

## 设备指纹与行为分析

### 设备指纹

设备指纹是反欺诈的基础能力之一。核心目标是：**即使用户换了账号，也能识别出同一个物理设备**。

| 采集端 | 采集信息 | 技术手段 |
| --- | --- | --- |
| 浏览器 | UA、屏幕分辨率、时区、语言、Canvas 指纹、WebGL 指纹 | JS SDK |
| APP（Android） | IMEI、Android ID、设备型号、SIM 信息 | Native SDK |
| APP（iOS） | IDFA（受限）、设备型号、系统版本 | Native SDK |
| 综合 | IP 地址、Wi-Fi SSID、基站信息 | 网络层 |

::: warning 隐私合规
设备指纹采集需要遵守 GDPR、个人信息保护法等隐私法规。需要用户明确授权，且采集的数据需要加密存储、定期清理。
:::

**设备指纹的关键指标**：

- **同一设备关联账户数**：一个设备关联 5 个以上账户 → 疑似批量注册
- **设备更换频率**：用户短时间内频繁更换设备 → 可能是 ATO 或设备模拟器
- **设备环境异常**：Root/越狱、模拟器、代理工具 → 风险加权

### 行为生物识别

超越传统设备指纹，通过用户的操作行为建立身份认证：

| 行为特征 | 采集方式 | 识别能力 |
| --- | --- | --- |
| 击键节奏 | JS 监听 keydown/keyup 时间间隔 | 识别是否为真人操作 |
| 鼠标/触摸轨迹 | 记录移动路径、速度、加速度 | 区分人和脚本 |
| 滑动模式 | APP 端记录滑动力度、角度 | 个体化身份特征 |
| 页面停留 | 各页面的停留时间分布 | 识别自动化工具 |

### 关联图谱分析

通过设备、账户、IP、收款方之间的关联关系构建图谱：

```
设备 A ──── 账户 1 ──── IP X
  │            │
  │            │
账户 2 ──── 设备 B ──── IP Y
  │
  │
账户 3 ──── 收款方 Z
```

如果多个账户通过同一设备、IP 或收款方关联起来，且其中有一个已被确认为欺诈，那么关联的其他账户也需要重点排查。

## 拒付与争议处理

### Chargeback 流程

Chargeback（拒付）是持卡人向发卡行发起的交易争议，是支付反欺诈中最直接的资损来源。

```
持卡人发起争议
    │
    ▼
发卡行审核
    │
    ├── 接受争议 → 通知卡组织
    │                │
    │                ▼
    │         卡组织扣款收单行
    │                │
    │                ▼
    │         收单行通知商户
    │                │
    │                ▼
    │         商户决定：接受 or 应诉
    │                │
    │      ┌─────────┴─────────┐
    │      ▼                   ▼
    │   接受拒付            提交举证
    │   承担损失            (representment)
    │                         │
    │                         ▼
    │                   卡组织仲裁
    │                         │
    │              ┌──────────┴──────────┐
    │              ▼                     ▼
    │           商户胜诉               商户败诉
    │           资金退回               损失确认
    │
    └── 拒绝争议 → 持卡人可升级投诉
```

### 举证与应诉策略

应诉成功率取决于证据质量。需要留存的证据：

| 证据类型 | 内容 | 保留方式 |
| --- | --- | --- |
| 交易记录 | 订单号、金额、时间、支付方式 | 数据库 |
| 交付证明 | 物流签收、下载记录、使用记录 | 对象存储 |
| 身份验证 | 3DS 验证结果、短信确认记录 | 日志系统 |
| 设备信息 | 设备指纹、IP 地址、地理位置 | 特征库 |
| 用户交互 | 交易前后的页面操作记录 | 行为日志 |

::: tip 关键：3DS 验证是最好的举证
如果交易经过 3D Secure 验证（如 Visa Secure、Mastercard Identity Check），责任转移到发卡行（liability shift）。商户在大多数情况下不用承担欺诈损失。
:::

### 拒付率监控与卡组织处罚

卡组织（Visa、Mastercard）对商户的拒付率有严格要求：

| 卡组织 | 拒付率阈值 | 超标后果 |
| --- | --- | --- |
| Visa | > 0.9%（连续两月） | 进入 VDMP 计划，罚款 $25,000~$75,000/月 |
| Mastercard | > 1.0% | 进入 EFM 计划，罚款递增 |
| 持续超标 | > 1.5%（连续多月） | 可能被终止收单资格 |

::: warning 严重后果
被终止收单资格意味着无法接受该卡组织的卡支付，对商户来说可能是致命的。拒付率是支付业务的生命线指标。
:::

### 预防性措施

| 措施 | 效果 | 适用场景 |
| --- | --- | --- |
| 3DS 验证 | 转移责任到发卡行 | 高风险交易、跨境交易 |
| AVS 地址验证 | 比对账单地址 | 北美市场 |
| CVV 校验 | 验证物理卡持有 | 所有 CNP 交易 |
| 订单确认邮件 | 提供交易凭证 | 所有交易 |
| 延迟发货/发放 | 给拦截留出时间 | 高价值数字商品 |

## 跨渠道与跨境欺诈

### 多渠道关联

现代支付的欺诈不限于单一渠道。攻击者可能：

- 在线上获取用户信息 → 线下门店盗刷
- APP 端控制账户 → 网页端转账
- 社交渠道钓鱼 → 支付渠道变现

反欺诈系统需要打通多渠道数据，建立统一的用户/设备/交易视图。

### 跨境支付的特殊风险

| 风险类型 | 表现 | 应对 |
| --- | --- | --- |
| 身份验证难度大 | 海外用户无法本地验证身份 | 依赖卡组织 3DS、当地 KYC 服务 |
| 时区差异 | 异常时段判断失效 | 按用户注册地时区判断 |
| 汇率波动 | 利用汇率差套利 | 实时汇率 + 价差异常检测 |
| 法规差异 | 不同国家的反欺诈和隐私要求不同 | 按目标市场分别配置规则 |
| 语言障碍 | 争议处理沟通成本高 | 多语言客服 + 自动化举证 |

### 应对策略

- 建立按国家/地区分层的风控策略
- 高风险地区（如拒付率高的国家）增加验证要求
- 与当地支付服务商合作获取本地化的欺诈情报
- 跨境交易强制 3DS 验证

## 反欺诈运营与监控

### 核心指标大盘

| 指标 | 计算方式 | 健康范围 | 说明 |
| --- | --- | --- | --- |
| 欺诈率 | 确认欺诈交易 / 总交易 | < 0.1% | 低于万分之十 |
| 拒付率 | Chargeback 笔数 / 总交易笔数 | < 0.5% | 卡组织红线是 0.9% |
| 检出率 (Recall) | 检出欺诈 / 实际欺诈 | > 80% | 漏掉的欺诈越少越好 |
| 误报率 (FPR) | 误拦正常交易 / 正常交易 | < 1% | 影响用户体验 |
| 精确率 (Precision) | 实际欺诈 / 标记为欺诈 | > 50% | 审核效率的保障 |
| 应诉成功率 | 商户胜诉 / 提交应诉 | > 40% | 举证质量的体现 |

### 监控与告警

| 监控项 | 告警条件 | 响应动作 |
| --- | --- | --- |
| 欺诈率突增 | 环比上升 50% 以上 | 紧急排查是否有新型攻击 |
| 拒付率接近阈值 | > 0.7% | 启动拒付预防方案 |
| 模型评分分布异常 | 高风险占比突增 | 检查模型输入和特征是否正常 |
| 特定商户拒付率高 | 单商户 > 2% | 联系商户排查、考虑暂停合作 |
| 试卡攻击信号 | 短时间大量小额失败交易 | 临时限制 IP/设备、启用 CAPTCHA |

### 案件管理与调查流程

```
可疑交易标记
    │
    ▼
自动归类（按欺诈类型）
    │
    ▼
案件分配（按优先级和专长）
    │
    ▼
调查分析
├── 关联分析（同设备/IP/收款方的其他交易）
├── 时间线还原（用户操作全链路）
└── 外部核实（联系用户、联系商户）
    │
    ▼
处置决定
├── 确认欺诈 → 冻结账户 + 加黑名单 + 上报
├── 确认正常 → 释放交易 + 优化规则
└── 无法确认 → 持续监控 + 标记观察
    │
    ▼
策略反馈
├── 新增规则
├── 模型重训样本
└── 情报沉淀
```

### 持续对抗与策略迭代

反欺诈是一场持续的对抗。欺诈者会不断调整策略：

| 阶段 | 欺诈者行为 | 防守方响应 |
| --- | --- | --- |
| 初期 | 简单盗刷、批量注册 | 基础规则即可覆盖 |
| 中期 | 使用代理、模拟器、养号 | 需要设备指纹、行为分析 |
| 成熟期 | 深度伪造、社工攻击、真人代刷 | 需要模型 + 图谱 + 情报 |
| 高级 | 利用系统漏洞、内外勾结 | 需要内控 + 审计 + 安全团队 |

::: warning 核心认知
反欺诈没有"一劳永逸"的方案。投入是持续性的，团队需要具备快速响应新型欺诈手法的能力。
:::

## 常见反模式

### 1. 只做事中拦截，忽略事前和事后

只在交易发起时检查，不做注册阶段的设备/行为分析，不做事后的拒付分析和策略迭代，是最常见的"半吊子"反欺诈。

### 2. 把反欺诈等同于风控规则

仅靠规则应对欺诈，忽略行为分析和模型评分。规则能覆盖已知模式，但无法发现新型欺诈。

### 3. 过度依赖第三方黑名单

第三方黑名单有滞后性，且覆盖面有限。不能替代自建的设备指纹和行为分析能力。

### 4. 忽略友好欺诈

很多团队只关注"外部攻击者"，不重视持卡人自身发起的虚假拒付。友好欺诈在电商和数字商品场景中占比可达 30%~40%。

### 5. 反欺诈团队与业务团队割裂

反欺诈策略的调整需要与业务方紧密配合。拦截率上升时，业务方应该知道原因并参与决策。

### 6. 不记录决策过程

每笔交易的风险评估过程（命中了哪些规则、模型评分多少、最终决策是什么）都需要完整记录，用于事后分析和合规审计。

## 面试回答模板

> **面试官问**：你们的支付系统是怎么做反欺诈的？

参考回答：

我们的反欺诈体系按事前、事中、事后三个阶段来设计：

**事前**：在用户注册和 KYC 阶段就开始采集设备指纹，建立行为基线。同时对商户做准入审核。

**事中**：每笔交易实时提取特征——包括速度特征（短时间交易频率）、金额偏差、地理跳跃、设备变更等。这些特征同时送入规则引擎和评分模型。规则引擎做硬性兜底（如黑名单直接拒绝），模型做精细化评分。综合两者的结果，决定通过、增强验证（如 3DS）、或拦截转人工。

**事后**：我们有完整的拒付处理流程——收到 Chargeback 后自动收集举证材料（交易记录、3DS 验证结果、设备信息），在截止日前提交应诉。同时通过关联分析（设备-账户-IP 图谱）发现欺诈网络，案件调查结果反馈到规则和模型中形成闭环。

核心指标我们关注四个：欺诈率 < 0.1%、拒付率 < 0.5%（远低于卡组织 0.9% 的红线）、检出率 > 80%、误报率 < 1%。

## 结论

支付反欺诈的核心是**对抗思维**：

1. **全链路覆盖**——事前建基线、事中做决策、事后做闭环，缺一不可
2. **规则 + 模型协同**——规则覆盖已知模式、快速响应；模型发现未知模式、精细判断
3. **持续迭代**——欺诈手法不断进化，防守方必须保持同等甚至更快的迭代速度
4. **数据驱动**——设备指纹、行为分析、关联图谱是反欺诈的数据基础

建议优先做三件事：集成设备指纹 SDK、接入 3DS 验证、搭建基础的规则引擎。这三项能覆盖最高频的欺诈场景，后续再逐步引入模型和图谱能力。
