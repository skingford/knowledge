## 为什么支付系统必须有独立风控层

很多团队在支付系统早期，风控逻辑散落在各个业务接口里：

- 转账接口判一下"余额够不够"
- 提现接口卡一下"每日限额"
- 充值接口校验一下"银行卡是否绑定"

这种做法在交易量小的时候还能凑合，一旦业务规模上来，问题就会集中爆发：

- **规则散落**：相同的限额逻辑在多个服务中重复实现，版本不一致
- **无法联动**：A 渠道消费和 B 渠道消费没有全局视角，各管各的额度
- **合规缺失**：监管要求的反洗钱上报、可疑交易识别没有统一出口
- **响应滞后**：新的欺诈手段出现后，需要逐个服务改代码、上线

::: warning 关键认知
支付风控不是"限流"的升级版，而是一套独立的决策系统。它需要全局视角、统一规则、实时响应和可审计的决策链路。
:::

支付风控的核心矛盾是：

| 目标 | 矛盾 |
| --- | --- |
| 安全 | 规则太严 → 误杀正常用户、损失收入 |
| 体验 | 规则太松 → 漏过欺诈交易、产生资损 |
| 合规 | KYC/AML 要求 → 增加用户操作成本 |
| 效率 | 实时决策 → 需要低延迟的规则和模型服务 |

好的风控系统，是在这四个维度之间找到动态平衡，而不是一味地"拦截更多"。

## 支付风控与秒杀风控的区别

秒杀场景下的风控（可参考 [秒杀系统风控与资格校验设计](/architecture/seckill-risk-control-and-eligibility-design)）和支付场景下的风控，目标和手段有本质差异：

| 维度 | 秒杀风控 | 支付风控 |
| --- | --- | --- |
| 核心目标 | 过滤无效流量、保护库存 | 保护资金安全、满足合规 |
| 风险后果 | 活动体验下降、库存被脚本抢走 | 资金损失、合规处罚、牌照风险 |
| 时间窗口 | 短时高峰（几分钟到几小时） | 7×24 持续运行 |
| 决策维度 | 用户资格、设备、IP、行为频率 | 账户、金额、频率、收款方、地理、设备、历史画像 |
| 容错性 | 误拦一个用户影响有限 | 误拦可能导致用户投诉、商户流失 |
| 合规要求 | 几乎没有 | KYC/AML 是硬性要求 |
| 规则复杂度 | 相对简单、偏静态 | 多维组合、需要动态调整 |

::: tip 一句话总结
秒杀风控重"快速拦截"，支付风控重"精准判断"。
:::

## 支付风控的核心维度

支付风控的判断不是"单一条件命中就拦截"，而是多维度组合决策：

### 账户维度

- **KYC 等级**：未实名、基础实名、高级实名，不同等级对应不同限额
- **账户年龄**：新注册账户天然高风险
- **历史行为**：有无退款、拒付、投诉记录
- **信用评分**：基于历史交易积累的内部信用分

### 交易维度

- **单笔金额**：是否超过该用户/该渠道的合理范围
- **交易频率**：短时间内大量小额交易可能是洗钱或盗刷
- **累计金额**：日/周/月累计是否超过阈值
- **收款方**：首次收款对象、高风险商户类别

### 设备与环境维度

- **设备指纹**：同一设备关联多少账户
- **IP/地理位置**：登录地和交易地是否一致，是否有地理跳跃
- **网络环境**：是否使用代理/VPN/Tor
- **操作时间**：凌晨大额交易需要额外关注

### 业务维度

- **交易类型**：消费、转账、提现、充值的风险等级不同
- **商户类别**：赌博、虚拟商品、高风险行业
- **渠道来源**：APP 内支付 vs 第三方跳转 vs API 调用
- **跨境标志**：涉及跨境的交易额外增加合规检查

## 一条推荐的风控决策链路

<InlineSvg src="/architecture/payment/risk-control.svg" alt="支付风控六层决策链路" />

```
请求进入
  │
  ▼
┌──────────────────┐
│ 第一层：准入校验   │  KYC 等级 × 交易类型，不满足直接拒绝
└──────┬───────────┘
       ▼
┌──────────────────┐
│ 第二层：限额限频   │  单笔/日累计/月累计 + 频率窗口
└──────┬───────────┘
       ▼
┌──────────────────┐
│ 第三层：规则引擎   │  多维度条件组合 → 通过 / 拒绝 / 转人工
└──────┬───────────┘
       ▼
┌──────────────────┐
│ 第四层：模型评分   │  实时特征 + 离线模型 → 风险分
└──────┬───────────┘
       ▼
┌──────────────────┐
│ 第五层：人工审核   │  高风险交易拦截、人工复核
└──────┬───────────┘
       ▼
┌──────────────────┐
│ 第六层：黑白名单   │  已知高风险 → 直接拦截，已知安全 → 跳过部分检查
└──────┬───────────┘
       ▼
    决策结果
  通过 / 拒绝 / 降级 / 人工
```

### 每一层的设计要点

**第一层：准入校验**

这是最前置、最硬性的一关。核心逻辑是：

- KYC 等级不满足该交易类型要求 → 直接拒绝，引导用户完成认证
- 账户状态异常（冻结、注销中）→ 直接拒绝
- 不需要复杂规则，只需要查账户基础信息

::: tip 设计原则
准入校验的结果只有两种：通过 或 拒绝。不存在"降级"或"转人工"。
:::

**第二层：限额限频**

限额限频是支付风控的基础防线。详细设计见 [限额限频设计](#限额限频设计) 章节。

**第三层：规则引擎**

规则引擎处理的是多维度条件组合判断。详细设计见 [规则引擎设计](#规则引擎设计) 章节。

::: warning 关键决策
规则引擎的输出不只是"通过/拒绝"，还要支持"降级处理"（如增加短信验证、降低限额）和"转人工"（高风险但不确定的交易）。
:::

**第四层：模型评分**

- 适用于规则难以覆盖的复杂场景
- 实时特征 + 离线训练模型 → 输出风险分 0~100
- 与规则引擎协同：规则做硬性兜底，模型做精细化补充
- 初期可以用规则代替，等数据积累够了再引入模型

**第五层：人工审核**

- 设定人工审核阈值（如模型评分 60~80 区间的交易）
- 审核队列需要有 SLA（如 30 分钟内处理）
- 超时未审核的交易需要有默认策略（通过或拒绝）
- 审核结果要反馈到规则和模型，形成闭环

**第六层：黑白名单**

- **黑名单**：已确认的欺诈账户、设备、IP → 直接拦截
- **白名单**：内部账户、VIP 用户、合作商户 → 跳过部分检查
- 名单需要有有效期和审批流程，避免无限膨胀
- 黑名单可以放在第一层（准入校验后）以提前拦截，减少后续计算

## 规则引擎设计

### 规则与策略的区别

| 概念 | 定义 | 示例 |
| --- | --- | --- |
| 规则（Rule） | 单条判断逻辑 | "单笔金额 > 50000" |
| 策略（Strategy） | 多条规则的组合 + 动作 | "金额 > 50000 且 新用户 且 跨境 → 拒绝" |
| 规则集（RuleSet） | 一组策略的集合 | "转账风控规则集" 包含 10 条策略 |

### 规则配置化

规则应该可以在不发版的情况下动态修改。典型的规则配置结构：

```json
{
  "rule_id": "R001",
  "name": "大额转账拦截",
  "conditions": [
    { "field": "amount", "op": "gt", "value": 50000 },
    { "field": "account_age_days", "op": "lt", "value": 30 },
    { "field": "tx_type", "op": "eq", "value": "transfer" }
  ],
  "logic": "AND",
  "action": "reject",
  "priority": 100,
  "enabled": true,
  "effective_time": "2025-01-01T00:00:00Z",
  "expire_time": "2025-12-31T23:59:59Z"
}
```

### 规则版本管理与灰度

- 每次规则变更生成新版本，支持回滚
- 新规则上线前先灰度（如只对 1% 的交易生效，观察误杀率）
- 灰度期间新规则只记录、不拦截（shadow mode），用于评估效果
- 正式启用后保留历史版本，出问题可以秒级回滚

### 规则冲突处理

多条规则可能对同一笔交易给出不同结论：

- **优先级机制**：高优先级规则覆盖低优先级
- **最严原则**：多条规则命中时，取最严格的结论
- **白名单优先**：白名单命中时跳过风控规则（需要审计）
- **冲突记录**：记录冲突情况，供运营分析和规则调优

## KYC / AML 合规要点

### KYC 分级

KYC（Know Your Customer）是金融监管的基础要求。分级策略平衡安全与体验：

| 等级 | 验证内容 | 开放能力 |
| --- | --- | --- |
| 简易 KYC | 手机号 + 短信验证 | 小额消费（日限 1000 元） |
| 标准 KYC | 实名 + 身份证号 + 人脸识别 | 标准消费/转账（日限 50000 元） |
| 增强 KYC | 银行卡绑定 + 视频面签 + 地址证明 | 大额/跨境/提现（更高限额） |

::: warning 合规底线
KYC 等级决定了交易限额的上限。即使规则引擎判断"安全"，也不能突破 KYC 等级的限额约束。KYC 是硬性合规要求，不是风险偏好可以调整的参数。
:::

### 反洗钱（AML）监控

AML（Anti-Money Laundering）关注的不是单笔交易安全，而是交易模式：

- **大额交易报告**：单笔超过监管阈值（如 5 万元）需要上报
- **可疑交易识别**：
  - 化整为零：多笔小额交易规避大额报告
  - 快进快出：资金短暂停留后快速转出
  - 频繁跨境：短时间大量跨境转账
  - 对手异常：频繁与高风险地区/行业交易
- **上报流程**：识别 → 内部审查 → 合规官审批 → 监管上报
- **留存要求**：交易记录至少保留 5 年

### 跨境支付的合规差异

- 不同国家和地区的 KYC 和 AML 要求不同
- FATF（反洗钱金融行动特别工作组）成员国的要求相对统一
- 跨境交易需要额外检查：制裁名单、外汇管制、税务合规
- 需要对接当地监管的上报系统

## 限额限频设计

### 限额层次

```
单笔限额
  │
  ▼ 叠加
日累计限额
  │
  ▼ 叠加
月累计限额
  │
  ▼ 叠加
年累计限额（部分场景）
```

每一层都需要按多个维度设置：

| 维度 | 示例 |
| --- | --- |
| 用户 + 交易类型 | 用户 A 转账日限 5 万 |
| 用户 + 渠道 | 用户 A 通过快捷支付日限 2 万 |
| 商户 + 交易类型 | 商户 B 收款日限 100 万 |
| 用户 + 商户 | 用户 A 向商户 B 月限 10 万 |

### 窗口策略

| 策略 | 优点 | 缺点 |
| --- | --- | --- |
| 固定窗口（自然日） | 实现简单 | 窗口交界处有突增风险 |
| 滑动窗口 | 平滑限制 | 实现复杂、存储成本高 |
| 令牌桶 | 允许突发 | 不适合限额场景 |

::: tip 推荐
限额场景推荐固定窗口（自然日/自然月），用 Redis `INCRBY + EXPIRE` 实现。限频场景（如每分钟最多 N 笔）推荐滑动窗口。
:::

### Redis 实现要点

```
// 日累计限额检查
key = "limit:{user_id}:{tx_type}:daily:{date}"
current = INCRBY key amount
if current > daily_limit:
    DECRBY key amount   // 回滚
    return REJECT

EXPIRE key 86400+3600   // 过期时间多留 1 小时余量
```

注意事项：

- `INCRBY` 和 `EXPIRE` 不是原子操作，需要用 Lua 脚本或 `SET NX EX` 组合
- 限额回滚：交易失败或退款时需要释放已占用的额度
- 多维度叠加：用户日限和商户日限需要同时检查，全部通过才放行

## 误杀恢复与运营治理

### 误杀的成本

风控误杀（False Positive）的代价远比想象中大：

- 正常用户被拦截 → 投诉、流失
- 商户交易被拒绝 → 商户流失、赔付
- 大额正常交易被卡 → 用户体验差、业务受阻
- 客服压力 → 人工审核成本上升

### 误杀检测

- **事后分析**：定期统计拦截交易中"用户申诉成功"的比例
- **A/B 对比**：新规则灰度期间对比拦截率和申诉率
- **阈值监控**：拦截率突增时自动告警

### 申诉通道设计

```
用户被拦截
  │
  ▼
展示拦截原因（脱敏）+ 申诉入口
  │
  ▼
用户提交申诉材料
  │
  ▼
人工审核（SLA ≤ 2 小时）
  │
  ▼
通过 → 加白名单 + 重试交易
拒绝 → 告知用户 + 记录
```

### 风控效果度量

| 指标 | 计算方式 | 健康范围 |
| --- | --- | --- |
| 拦截率 | 被拦截交易 / 总交易 | 0.1% ~ 1% |
| 误杀率 | 申诉成功 / 被拦截 | < 5% |
| 漏杀率 | 事后确认欺诈 / 总交易 | < 0.01% |
| 资损率 | 欺诈资损金额 / 总交易金额 | < 0.001% |

::: warning 注意
拦截率不是越高越好。如果拦截率超过 2%，大概率是规则太严或者阈值设置不合理，应该立即排查。
:::

## 数据模型建议

### 风控事件表（risk_event）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| event_id | bigint | 主键 |
| tx_id | varchar(64) | 关联交易 ID |
| user_id | bigint | 用户 ID |
| event_type | varchar(32) | 事件类型（交易、登录、绑卡等） |
| risk_score | int | 风险评分 0~100 |
| decision | varchar(16) | 决策结果（pass/reject/review/downgrade） |
| hit_rules | jsonb | 命中的规则列表 |
| features | jsonb | 本次决策使用的特征快照 |
| review_status | varchar(16) | 审核状态 |
| created_at | timestamp | 创建时间 |

### 规则配置表（risk_rule）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| rule_id | varchar(32) | 规则 ID |
| rule_name | varchar(128) | 规则名称 |
| conditions | jsonb | 条件表达式 |
| action | varchar(16) | 命中动作 |
| priority | int | 优先级 |
| version | int | 版本号 |
| enabled | boolean | 是否启用 |
| effective_time | timestamp | 生效时间 |
| expire_time | timestamp | 失效时间 |

### 审核记录表（risk_review）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| review_id | bigint | 主键 |
| event_id | bigint | 关联风控事件 |
| reviewer | varchar(64) | 审核人 |
| decision | varchar(16) | 审核结论 |
| reason | text | 审核理由 |
| created_at | timestamp | 审核时间 |

## 常见反模式

### 1. 风控逻辑散落在业务代码中

风控逻辑应该收敛到独立的风控服务中调用，而不是每个接口自己写一遍。否则规则更新需要改 N 个服务。

### 2. 规则只增不删

规则会随着时间膨胀。定期清理过期、低命中率的规则是运营的日常工作。建议每季度做一次规则审计。

### 3. 用"拦截所有"替代精细化风控

一些团队遇到欺诈后的应激反应是：加很多严格的规则把可疑交易全拦住。短期止血可以，长期会导致正常用户大量流失。

### 4. 忽略异步回调场景的风控

支付回调也可能被伪造或重放。风控不只在"用户发起交易"时触发，异步回调阶段也需要验签和重复性检查。

### 5. 限额只防用户不防商户

商户侧也需要限额——防止单商户异常收款（如被盗用 API 密钥）。

## 面试回答模板

> **面试官问**：你们的支付系统是怎么做风控的？

参考回答：

我们的支付风控分成六层来做：

1. **准入校验**：先检查 KYC 等级和账户状态，不满足要求的直接拒绝
2. **限额限频**：按用户 × 交易类型 × 渠道做多维度限额，用 Redis 滑动窗口实现频控
3. **规则引擎**：支持配置化的多条件组合规则，规则变更不需要发版，支持灰度和回滚
4. **模型评分**：对规则难以覆盖的复杂场景，引入实时特征 + 离线模型做风险评分
5. **人工审核**：高风险但不确定的交易进入审核队列，设 SLA 和超时策略
6. **黑白名单**：已确认的高风险实体直接拦截，VIP 跳过部分检查

关键点是：规则引擎的输出不只是通过/拒绝，还支持降级（加验证）和转人工。我们用 shadow mode 做新规则上线前的效果评估，用拦截率 + 误杀率 + 漏杀率三个指标来度量风控效果。

## 落地检查清单

- [ ] 风控决策逻辑收敛到独立服务，业务侧只做调用
- [ ] KYC 等级与交易限额有明确的对应关系
- [ ] 限额覆盖单笔、日累计、月累计，按用户 × 交易类型 × 渠道分维度
- [ ] 规则引擎支持配置化，变更不需要发版
- [ ] 新规则上线有灰度流程和 shadow mode
- [ ] 规则有版本管理，支持秒级回滚
- [ ] 人工审核队列有 SLA 和超时默认策略
- [ ] 黑白名单有有效期和审批流程
- [ ] 有误杀率、拦截率、漏杀率的监控大盘
- [ ] 异步回调链路也经过风控校验
- [ ] AML 可疑交易有上报通道
- [ ] 风控决策全量记录，支持审计追溯

## 结论

支付风控的核心不是"拦得多"，而是"判得准"。一个好的风控系统需要：

1. **分层决策**——从硬性准入到柔性评分，逐层过滤
2. **配置化管理**——规则可以快速调整，不需要等发版
3. **可度量效果**——用拦截率、误杀率、漏杀率持续优化
4. **运营闭环**——误杀有恢复通道，规则有定期审计

建议先把前三层（准入 + 限额 + 规则引擎）做扎实，这已经能覆盖 80% 以上的风控场景。等数据积累够了再引入模型评分层。
