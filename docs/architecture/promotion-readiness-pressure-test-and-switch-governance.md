# 大促活动预热、压测与开关治理手册

## 适合人群

- 需要参与双十一、618、秒杀节、抢券活动等大促保障的后端工程师

- 理解为什么很多活动事故不是因为代码写错，而是因为预热不充分、压测不真实、开关准备不完整

- 能整理出一套更贴近真实大促保障场景的活动准备和现场执行方案

## 为什么大促保障不能只靠“代码写好了”

很多团队在活动前最容易高估的一件事是：

- 只要功能开发完、联调通过、测试环境跑通，就说明活动能扛住

这在平时流量下也许问题不大，但在大促场景里，真正决定结果的往往不是功能是否存在，而是：

- 容量估算是否靠谱
- 热点数据是否提前预热
- 压测流量是否接近真实分布
- 降级和开关是否能在现场快速生效
- 出现抖动时是否有清晰的止血动作

所以大促保障的核心认知是：

> 大促不是一次普通上线，而是一场有明确时间窗口、明确洪峰预期、明确风险边界的系统作战。

很多事故并不是因为服务完全没设计，而是因为：

- 预热少做了一层
- 压测漏了一条关键链路
- 开关准备得不够细
- 现场不知道先关什么、保什么、放弃什么

## 先把大促问题拆成三个阶段

大促保障最适合按阶段思考，而不是只盯着活动开始那一刻。

### 1. 活动前

这个阶段重点不是“快上线”，而是“把未知变少”。

最关键的问题通常是：

- 流量峰值是多少
- 哪些资源是热点
- 哪些数据必须预热
- 哪些链路必须压测
- 哪些开关必须提前配置好

### 2. 活动中

这个阶段重点不是“所有功能都正常”，而是“核心目标是否稳定达成”。

最关键的问题通常是：

- 当前通过率和成功率是否健康
- 哪一层先开始抖动
- 是否要收缩放量
- 是否要触发降级或切回旧链路

### 3. 活动后

这个阶段重点不是“活动结束就结束了”，而是“把经验沉淀成下一次更稳的容量模型和预案”。

最关键的问题通常是：

- 哪些阈值设置过紧或过松
- 哪些热点没有提前识别出来
- 哪些开关缺失或颗粒度不够
- 哪些补偿和恢复动作太慢

## 活动保障的总体目标

一个更成熟的大促保障方案，通常同时追求这些目标：

- 在活动开始前明确系统可承诺的处理边界
- 在活动开始时尽量减少动态计算和冷启动成本
- 在活动中快速识别风险并把损失收敛在局部
- 在必要时优先保核心链路，而不是强保全部功能
- 在活动结束后把容量和预案更新为可复用资产

这里很值得强调一句：

- 大促保障不是追求“任何时候都不降级”
- 而是追求“在降级时仍然知道怎样保住最核心目标”

## 第一步：先做容量估算和承诺边界

很多大促事故的起点，是业务和技术对“系统能接多少量”没有统一认识。

### 至少要回答的问题

- 峰值 QPS 预计是多少
- 峰值订单创建速率是多少
- 热点 SKU 会集中到什么程度
- Redis、MQ、数据库、下游服务的安全水位是多少
- 哪些指标是硬上限，超过就必须降级

### 为什么这一步是第一步

因为如果边界都不清楚，后面的压测和预案就会失去目标。

例如你说系统能抗：

- 10 万 QPS

但这 10 万是：

- 网关入口 QPS
- 通过资格过滤后的有效 QPS
- 还是最终落到订单库的写入 QPS

这些不是一回事。

### 一个更稳的做法

容量模型最好分层建立：

- 入口层容量
- 资格与 Redis 层容量
- 队列层容量
- 订单层容量
- 支付和回调层容量

这样你才能知道真正的瓶颈在哪一层，而不是只拿一个总 QPS 数字安慰自己。

## 第二步：活动前资源预热和数据准备

大促开始时，系统最怕的不是“请求来了”，而是“请求来了以后很多资源还是冷的”。

### 常见预热对象

- CDN 和静态页面资源
- 活动配置、白名单、限购规则
- Redis 热 Key、库存 Key、资格数据
- JVM、连接池、线程池、数据库连接
- 消息队列 Topic、消费组和关键 worker

### 为什么预热有用

因为冷启动成本在大促里会被成倍放大。

例如：

- 活动页首次回源
- Redis 热点 Key 未建立
- 连接池刚启动还没拉满
- 消费者分区没分配稳定

这些在日常也许只是轻微抖动，但在洪峰下就可能变成明显故障。

### 一个实践建议

预热不要只理解成：

- 先访问一遍页面

更完整的预热应该包括：

- 数据预热
- 资源预热
- 线程和连接预热
- 缓存和热点路由预热
- 降级数据和兜底文案预热

## 第三步：全链路压测而不是只压单接口

很多团队说自己做过压测，但实际只做了：

- 某个 HTTP 接口压测

这对大促帮助有限。

### 为什么单接口压测不够

因为大促链路里真正会出问题的，通常是多个系统共同作用后的结果：

- 网关限流
- Redis 热 Key
- MQ 堆积
- 订单库写入
- 支付回调并发
- 补偿任务回补

如果只压单接口，你很难发现：

- 某个队列先堆积
- 某个数据库索引在真实写入模式下失效
- 某个依赖在高峰下出现 RT 飙升

### 更推荐的压测层次

- 单接口压测：看单点极限
- 单机压测：看单机资源瓶颈
- 全链路压测：看真实流量穿透后的系统行为
- 故障注入压测：看依赖抖动和降级是否生效
- 回归压测：看优化后阈值是否真的提高

### 大促压测最容易漏的点

- 压测流量和真实流量分布不一致
- 只压下单，不压查询、回调、取消和回补
- 压测不经过缓存和 MQ 的真实路径
- 压测结果没有回写到开关阈值和容量预案

没有回写阈值的压测，很容易沦为一次“知道系统会抖，但不知道该怎么处理”的演习。

## 第四步：灰度放量和可逆切流

大促活动不应该默认“一秒切满”。

更稳的方式通常是：

- 先小流量验证
- 再逐步放量
- 每一步都保留回退能力

### 为什么灰度放量重要

因为即使压测做过了，线上真实环境仍然会出现：

- 用户行为分布变化
- 某些热点比预估更集中
- 某些下游配置和压测时不一致

灰度放量的价值就在于：

- 让问题暴露在更小的损失范围里

### 可逆切流的关键原则

- 读流量先灰度
- 写流量最后切
- 每一步都要有回切开关
- 回切动作必须可演练，不是口头预案

### 大促里特别值钱的一点

不是“敢不敢切”，而是：

- 切了以后出了问题，能不能在几分钟内明确回退路径

## 第五步：开关矩阵、降级策略与应急预案

真正的大促现场，最值钱的不是某个高深架构词，而是：

- 现场到底先关什么
- 先保什么
- 哪些功能允许牺牲
- 哪些动作必须最后才动

所以大促保障最好提前准备开关矩阵。

### 常见开关类型

- 活动入口开关
- 用户资格开关
- 限流阈值开关
- 队列消费并发开关
- 降级文案开关
- 推荐、埋点、通知等非核心链路开关
- 新旧链路切换开关

### 为什么要叫“矩阵”

因为开关不是单个存在的，而是要配合场景使用。

例如：

- Redis RT 升高时先收紧资格放量
- 队列堆积严重时降低令牌发放速度
- 订单库写入异常时暂停部分活动入口
- 新链路错误率升高时切回旧链路

### 应急预案至少要明确什么

- 触发条件
- 执行动作
- 负责人
- 回滚条件
- 对用户的提示文案

如果这些都没有提前定义，大促现场就很容易陷入：

- 大家都知道有问题
- 但没人敢先动开关
- 或者开关乱动导致次生事故

## 第六步：活动现场值守与指标判读

大促现场不是只盯一个大盘看 QPS。

更重要的是看：

- 哪个指标先变坏
- 是入口问题、缓存问题、队列问题还是数据库问题
- 当前最应该做的是扩容、降级、限流还是回切

### 建议重点盯的指标

- 活动页和网关 QPS、错误率、RT
- 资格命中率、限流拒绝率、风控拦截率
- Redis RT、热点 Key 命中率、脚本失败率
- MQ 堆积长度、消费延迟、失败重试次数
- 订单创建成功率、取消率、平均耗时
- 数据库 CPU、锁等待、慢 SQL、连接数
- 开关触发次数、降级持续时间、恢复耗时

### 指标判读要避免什么

不要只盯单个值，要看变化趋势和因果顺序。

例如：

- 先看到 Redis RT 升高
- 接着资格通过率下降
- 再看到队列堆积加剧

这说明问题起点更可能在前置资格和热点竞争层，而不是订单库本身。

### 一个现场原则

大促时不要一上来就做重操作。

更常见、更稳的止血顺序通常是：

- 先收紧放量
- 再降级非核心链路
- 必要时回切
- 最后才做更重的结构性操作

## 第七步：活动后复盘与容量回写

很多团队活动结束后的最大浪费是：

- 现场经验没有沉淀
- 下次还要再踩一遍

### 活动后最值得回写的东西

- 真正峰值和原估算的差距
- 哪些热点资源预判错了
- 哪个链路最先成为瓶颈
- 哪个开关最有效
- 哪些开关缺失或粒度不够
- 哪些告警触发过晚或噪音太大

### 为什么回写重要

因为容量模型和保障预案不是写一次就完事的文档，而是需要每次活动后不断修正。

真正成熟的团队会把每次大促沉淀成：

- 更新后的容量模型
- 更合理的阈值
- 更清晰的开关矩阵
- 更贴近真实的压测脚本

这样下一次活动，准备工作才不是从零开始。

## 常见误区

### 1. 误区一：压测通过，就说明大促一定稳

不一定。

压测模型如果不真实，结论也会失真。

### 2. 误区二：开关准备得越多越好

开关很多但没有命名规范、触发条件和演练流程，现场一样会乱。

### 3. 误区三：大促保障只属于运维或 SRE

不是。

大促保障通常需要研发、测试、运维、DBA、业务和客服一起协同。

### 4. 误区四：活动开始前再做预热也来得及

很多热点资源、连接、缓存和路由，如果不提前准备，活动开始后再补往往已经太晚。

### 5. 误区五：活动结束就不用管了

如果没有复盘和回写，下次活动的准备质量通常不会真正提升。

## 面试回答模板

如果面试官问“怎么做一次大促活动保障”，可以用下面这版口径回答：

> 我会先把大促保障拆成活动前、活动中和活动后三段。活动前重点做容量估算、热点识别、资源预热、全链路压测和开关预案；活动中重点看通过率、成功率、Redis/MQ/数据库这些关键指标的变化趋势，根据情况收紧放量、降级非核心能力或者回切旧链路；活动后则把真实峰值、瓶颈位置、开关效果和恢复耗时回写到容量模型和预案里。  
> 具体做法上，我不会只看一个总 QPS，而是把入口层、Redis 层、队列层、订单层分别建容量边界；压测也不会只压单接口，而会做全链路压测和故障注入。  
> 同时我会提前准备开关矩阵，明确每个开关的触发条件、执行人和回滚条件，因为大促现场最重要的不是“知道有开关”，而是知道在什么指标下先动哪个开关、先保哪个核心目标。  
> 所以大促保障本质上不是一次发布动作，而是一套围绕容量、预热、演练、开关和复盘的治理机制。

如果继续追问，可以顺着讲：

1. 容量模型怎么分层估算
2. 预热应该覆盖哪些资源
3. 为什么大促一定要做全链路压测和故障演练
4. 开关矩阵应该怎么设计
5. 活动现场先看哪些指标、先做哪些止血动作

## 落地检查清单

### 1. 容量与边界

- 是否明确入口、Redis、MQ、数据库等分层容量边界
- 是否定义硬上限和必须降级的水位线
- 是否对业务承诺和技术容量达成一致

### 2. 预热准备

- 是否预热了活动页面、配置、白名单、热点 Key 和库存数据
- 是否预热了连接池、线程池、消费者和关键 worker
- 是否准备了降级数据、兜底文案和备用路由

### 3. 压测与演练

- 是否做了单接口、全链路和故障注入压测
- 是否覆盖了下单、查询、取消、回补等完整路径
- 是否把压测结果回写到阈值和开关配置中

### 4. 灰度与切流

- 是否支持活动前小流量验证和逐步放量
- 是否明确了新旧链路的回切路径
- 是否演练过回切和恢复过程

### 5. 开关与应急

- 是否整理了活动入口、限流、消费并发、降级文案等开关矩阵
- 是否定义了开关触发条件、责任人和回滚条件
- 是否明确了先保什么、先关什么、最后才动什么

### 6. 现场值守与复盘

- 是否有核心指标大盘和分层告警
- 是否有活动期间的值守分工和沟通机制
- 是否能在活动后沉淀复盘结论、容量模型和压测脚本

## 结论

大促活动预热、压测与开关治理真正要解决的，不只是“系统能不能扛住一波流量”，而是：

- 怎么在活动前把未知风险尽量变少
- 怎么在活动中快速判断问题起点并收缩损失
- 怎么在必要时用开关和回切保住核心目标
- 怎么把一次活动经验沉淀成下一次更稳的能力资产

所以最值得记住的一句话是：

> 大促保障的本质，不是临场救火，而是用容量边界、预热、压测、开关和复盘，把高峰风险前移治理。
