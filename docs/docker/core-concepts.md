# Docker 核心概念与对象模型

> 回答 Docker 相关问题时，最好按“解决什么问题 -> 对象模型 -> 底层机制 -> 生命周期 -> 常见误区”这条线讲。先把概念讲稳，再谈命令和平台细节。

## 高频问题总览

| 题目 | 核心矛盾 | 首先要答什么 |
| --- | --- | --- |
| Docker 到底解决什么问题 | 程序能跑，但环境不一致 | 交付单元标准化 + 环境一致性 |
| 镜像和容器是什么关系 | 经常把两者混成一回事 | 镜像是模板，容器是运行实例 |
| Docker 和虚拟机差别是什么 | 都在“隔离运行”，但实现不同 | 容器共享内核，虚拟机带 Guest OS |
| 容器为什么能隔离 | 不是单纯“Docker 自己做了沙箱” | `namespace` + `cgroup` + 文件系统分层 |
| 镜像层和容器可写层是什么关系 | 数据会不会跟着容器一起消失 | 镜像只读，容器额外挂可写层 |
| `save/load` 和 `export/import` 有什么区别 | 都像“导出导入”，但语义不同 | 前者迁镜像，后者导文件系统快照 |

## 1. Docker 到底解决什么问题

Docker 的核心价值，不是“把程序跑起来”，而是把**应用和运行环境封装成一个稳定、可复制、可分发的交付单元**。

它主要解决下面这些高频问题：

- 开发、测试、生产环境不一致
- 应用依赖复杂，迁机器容易坏
- 交付过程靠手工命令，不可复用
- 一个服务上线，需要同时带配置、依赖、运行时和启动方式

可以直接记住这句口径：

> Docker 解决的是交付标准化和环境一致性问题，让应用可以用镜像作为统一制品在不同环境里重复运行。

## 2. 先把对象模型分清楚

| 对象 | 它是什么 | 你至少要会什么 |
| --- | --- | --- |
| `Image` | 应用运行模板，由多个只读层组成 | 会 `build / pull / tag / push / history / inspect` |
| `Container` | 镜像的运行实例，本质上是宿主机上的进程 | 会 `run / ps / logs / exec / stop / rm / inspect` |
| `Dockerfile` | 描述如何构建镜像的脚本 | 会写 `FROM / COPY / RUN / ENTRYPOINT / CMD / USER` |
| `Registry` | 存储镜像的仓库 | 会 `login / push / pull`，理解 tag 和 digest |
| `Volume` | 由 Docker 管理的持久化目录 | 会和 bind mount 区分使用 |
| `Network` | 容器之间与容器对外通信的连接方式 | 会区分默认 bridge、用户自定义 bridge、host、none |
| `Compose` | 多容器应用定义与管理方式 | 会写 `services / networks / volumes / env / healthcheck` |

### 镜像和容器的关系

最容易混淆的，就是把镜像和容器说成同一个东西。

- 镜像是只读模板
- 容器是镜像启动后的运行实例
- 一个镜像可以启动多个容器
- 删掉容器，不等于删掉镜像

标准口径：

> 镜像像类，容器像实例。镜像定义了运行模板，容器是这个模板在某个时间点的实际运行态。

## 3. Docker 和虚拟机不是一回事

| 对比项 | Docker 容器 | 虚拟机 |
| --- | --- | --- |
| 隔离基础 | 共享宿主机内核 | 自带 Guest OS |
| 启动速度 | 通常更快 | 通常更慢 |
| 资源开销 | 一般更小 | 一般更大 |
| 适合场景 | 应用交付、CI/CD、本地开发、微服务 | 强隔离、多操作系统并存、传统基础设施 |

必须讲清楚的结论：

- 容器不是“轻量虚拟机”
- 容器共享宿主机内核，所以不能像虚拟机一样随便跑另一套内核
- Docker 提供的是工程上足够强的隔离和更快的交付，而不是绝对隔离

## 4. Docker、containerd、runc、K8s 的关系

这几个词经常一起出现，但职责不一样：

- Docker：开发者最常接触的 CLI、镜像构建、容器管理体验
- `containerd`：更底层的容器生命周期管理组件
- `runc`：按 OCI 规范真正把容器进程创建出来
- Kubernetes：站在多机编排层，解决调度、发布、伸缩和治理

一个不容易答错的说法是：

> Docker 偏单机容器体验和镜像交付；Kubernetes 偏多机编排和运行治理。K8s 管的不是某一个 `docker run`，而是一整套期望状态和控制循环。

## 5. 容器为什么能隔离

Docker 之所以能跑出“像一台独立机器”的效果，靠的是 Linux 内核能力，不是它自己模拟了一套操作系统。

| 机制 | 作用 | 你要知道什么 |
| --- | --- | --- |
| `namespace` | 隔离视图 | 容器能拥有自己的进程、网络、挂载点、主机名视图 |
| `cgroup` | 限制和统计资源 | 容器 CPU、内存、IO 能被约束和观测 |
| 分层文件系统 | 复用镜像层 | 多个镜像共享基础层 |
| copy-on-write | 延迟复制 | 没改的数据直接复用，改动才写入新层 |

### `namespace` 重点理解什么

至少要知道容器通常会隔离这些视图：

- PID：容器里看到的是自己的进程树
- Network：容器里有自己的网络命名空间
- Mount：容器里有自己的挂载视图
- UTS：容器里有自己的 hostname

### `cgroup` 重点理解什么

- 容器不是无限吃资源
- `-m`、`--cpus` 之类的限制，本质上落到 `cgroup`
- 如果没有资源边界，单个容器就可能拖垮宿主机

## 6. 镜像层、容器可写层与 copy-on-write

Docker 镜像不是一个整体大文件，而是多层叠加。

至少要理解这条主线：

- 镜像由多个只读层组成
- `Dockerfile` 中很多指令会产生新层
- 容器启动后会额外挂一层临时可写层
- 容器里的变更默认写在可写层，不会回写镜像本身

必须掌握的结论：

- 基础镜像能被多个业务镜像复用
- 多个容器能共用同一个镜像层
- 数据如果只写在容器可写层里，删容器就可能一起没了

### 为什么层设计会影响构建速度和磁盘占用

- 常变的内容放后面，缓存更容易命中
- 不必要的大文件会被打进镜像层，后续每次分发都要跟着走
- 一条写坏的 `RUN` 指令，可能让整层缓存失效

## 7. 镜像和容器的生命周期怎么理解

### 镜像主线

```bash
docker build -t app:1.0 .
docker tag app:1.0 registry.example.com/team/app:1.0
docker push registry.example.com/team/app:1.0
docker pull registry.example.com/team/app:1.0
```

### 容器主线

```bash
docker run --name app -d app:1.0
docker ps
docker logs -f app
docker exec -it app sh
docker stop app
docker rm app
```

### 容器启动到底发生了什么

可以按这条线回答：

1. 确认本地有没有镜像，没有就拉取
2. 基于镜像准备容器可写层
3. 配置网络、挂载卷、环境变量、资源限制
4. 启动主进程
5. 主进程退出后，容器通常也随之退出

## 8. `save/load` 和 `export/import` 不要混着答

| 对比项 | `save / load` | `export / import` |
| --- | --- | --- |
| 操作对象 | 镜像 | 容器文件系统快照 |
| 是否保留镜像层 | 保留 | 不保留 |
| 是否保留镜像元数据 | 保留 | 大多不保留 |
| 适合场景 | 镜像离线分发、内网导入 | 临时导出当前容器文件内容 |

标准口径：

> `docker save/load` 用来迁镜像；`docker export/import` 更像把容器当时的文件系统打平带走。前者保留镜像语义，后者更像临时快照。

## 9. 常见误区

- 以为 Docker 容器就是轻量虚拟机
- 把镜像和容器当成一个对象
- 以为删容器就会顺带清理掉数据卷
- 以为容器里的数据天生持久化
- 以为 Docker 和 Kubernetes 只是同一层东西的不同实现

## 10. 面试前至少要背下来的清单

- Docker 解决的是交付标准化和环境一致性问题
- 镜像是模板，容器是运行实例
- 容器共享宿主机内核，不是轻量虚拟机
- 容器隔离依赖 `namespace`，资源限制依赖 `cgroup`
- 镜像是只读分层，容器会额外挂可写层
- 数据如果只写在容器层，删容器后可能丢失
- `save/load` 迁镜像，`export/import` 导文件系统快照
- Docker 偏单机交付与容器体验，K8s 偏多机编排与治理
