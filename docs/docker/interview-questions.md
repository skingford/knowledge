# Docker 高频问题与自检清单

这页不追求把 Docker 所有知识点列完，而是把最常被问、最能拉开理解深度的问题按主线排好。建议回答时固定三个动作：

- 先说它**解决什么问题**
- 再说它**怎么实现**
- 最后补一句**适用场景、风险或常见误区**

## 1. 入门必答：先把定位和对象说对

**1. Docker 到底解决什么问题？**

- 它解决的是交付标准化和环境一致性
- 把应用和运行环境打包成统一镜像，方便在不同环境复用
- 重点不是“帮你跑程序”，而是“让程序可复制地跑”

**2. 镜像和容器是什么关系？**

- 镜像是只读模板
- 容器是镜像的运行实例
- 一个镜像可以起多个容器，删容器不等于删镜像

**3. Docker 和虚拟机的根本区别是什么？**

- 容器共享宿主机内核，虚拟机带独立 Guest OS
- 容器通常更轻、更快，但隔离不是虚拟机级别
- 所以 Docker 不是“轻量虚拟机”

**4. Docker、containerd、runc、K8s 分别做什么？**

- Docker 偏镜像构建和单机容器体验
- `containerd` 偏底层容器生命周期管理
- `runc` 负责真正把容器进程拉起来
- Kubernetes 解决多机编排和治理

## 2. 底层机制：不要只背命令

**5. 容器为什么能隔离？**

- 视图隔离靠 `namespace`
- 资源限制靠 `cgroup`
- 文件系统复用靠分层和 copy-on-write

**6. 镜像层和容器可写层是什么关系？**

- 镜像由多个只读层组成
- 容器启动时会额外挂一层可写层
- 数据如果只写在容器层，删容器后可能丢失

**7. 为什么说容器本质上还是宿主机进程？**

- 容器没有独立内核
- 最终仍然是宿主机上的一组进程，只是视图和资源边界被隔离了

## 3. 镜像构建：最容易被追问深的一块

**8. 一份靠谱 Dockerfile 至少要满足什么要求？**

- 基础镜像明确
- 分层合理，缓存能命中
- 构建工具链不进最终镜像
- 运行用户非 root
- 密钥不进镜像

**9. `ENTRYPOINT` 和 `CMD` 有什么区别？**

- `ENTRYPOINT` 更像固定主命令
- `CMD` 更像默认参数或默认命令
- `docker run image xxx` 更常覆盖 `CMD`

**10. `ARG` 和 `ENV` 有什么区别？**

- `ARG` 偏构建时参数
- `ENV` 偏运行时默认环境变量
- 两者都不该被拿来硬编码密钥

**11. 为什么构建缓存总失效？**

- `COPY . .` 太早
- 依赖下载和源码复制没有拆层
- `.dockerignore` 没配好，构建上下文太大

**12. 多阶段构建到底解决什么问题？**

- 编译阶段和运行阶段分离
- 最终镜像更小、更干净
- 攻击面更小

**13. 为什么生产不建议长期用 `latest`？**

- 不稳定
- 不可回滚
- 环境之间容易产生“同名不同内容”的漂移

## 4. 仓库与交付：别把版本治理答成“推个 latest”

**14. tag 和 digest 有什么区别，什么时候更该用 digest？**

- tag 可读，适合人工识别和日常发布
- digest 指向固定内容，适合强确定性部署
- 越接近正式发布，越应该能落到 digest 或不可变 tag

**15. 私有仓库和 Harbor 的价值是什么？**

- 不只是“镜像放公司内网”
- 还包括项目隔离、权限控制、Robot Account、扫描、复制、保留策略和垃圾回收
- Harbor 是团队级治理能力，不只是一个下载站

**16. 为什么更稳的流程是 build once，promote by digest？**

- 同一版代码在不同环境分别重建，产物可能不完全一样
- 测试通过的是 A，生产重新 build 出 B，发布链路就断了
- 更稳的做法是同一份镜像做晋级，而不是每个环境重新造一遍

**17. `docker save/load` 和 `export/import` 有什么区别？**

- `save/load` 迁的是镜像，保留镜像层和元数据
- `export/import` 更像把容器当时的文件系统打平带走
- 离线分发镜像时通常答 `save/load`

## 5. 生命周期与进程模型：很多人会用，但答不细

**18. 为什么容器里的主进程一退出，容器通常也跟着退出？**

- 因为容器生命周期默认绑在 PID 1 对应的主进程上
- 容器不是轻量虚拟机，不会自己长期守着一个系统级 init
- 主进程执行完，容器通常也就结束

**19. `docker stop` 和 `docker kill` 的区别是什么？**

- `docker stop` 更偏优雅停止，通常先发终止信号，再等宽限期
- `docker kill` 更强硬，常用于快速终止
- 正常发布或维护一般优先 `stop`，前提是应用自己能处理信号

**20. 为什么启动脚本最后常常要 `exec`？**

- 不 `exec` 时，shell 可能留在 PID 1
- 真正业务进程拿不到预期信号
- 停止、重启和退出行为会变得不稳定

**21. 重启策略和健康检查分别解决什么问题？**

- 重启策略解决“挂了以后要不要拉起”
- 健康检查解决“它现在看起来健不健康”
- 两者不是一回事，健康不等于 ready，重启也不等于根因已经修好

## 6. 运行态：网络、挂载、Compose 最常出问题

**22. `EXPOSE` 和 `-p` 有什么区别？**

- `EXPOSE` 只是镜像元数据声明
- `-p` 才是把宿主机端口映射出去

**23. 默认 `bridge` 和用户自定义 `bridge` 有什么差别？**

- 都是桥接网络
- 用户自定义 `bridge` 更适合多容器服务发现，通常更常用

**24. `bind mount`、`volume`、`tmpfs` 各适合什么场景？**

- `bind mount`：本地开发、挂源码、挂宿主机已有文件
- `volume`：数据库、缓存等正式持久化数据
- `tmpfs`：内存中的临时敏感数据

**25. 为什么容器里的数据会丢？**

- 因为可能写在容器可写层而不是持久化卷里
- 容器删除后，可写层数据通常一起消失

**26. Compose 适合什么，不适合什么？**

- 适合单机多容器交付
- 不适合多节点编排、自动扩缩容和复杂发布治理

**27. `depends_on` 为什么不等于服务真的 ready？**

- 它主要表达依赖顺序
- 真正的“已可用”通常还要配合 `healthcheck`

## 7. 安全与生产：从能跑到能上线

**28. 为什么生产镜像建议非 root 运行？**

- 减少权限暴露
- 降低逃逸或误操作影响面
- 宿主机挂载目录的权限污染更少

**29. 为什么不能把密钥写进镜像？**

- 镜像一旦推到仓库，泄露范围会扩大
- 历史镜像层里可能长期保留敏感信息
- 环境差异配置也不该和镜像耦合

**30. 为什么挂 `docker.sock` 很危险？**

- 拿到 Docker socket 的容器可能间接控制宿主机
- 可以创建高权限容器、挂宿主机目录、访问其他容器

**31. 资源限制为什么重要？**

- 避免单个容器拖垮宿主机
- 让容量评估和治理更可控
- “能跑”不代表“跑得稳”

## 8. 排障：最能区分是否真干过活

**32. 容器一启动就退出，你先看什么？**

- `docker ps -a`
- `docker logs`
- `docker inspect`

先分清是主进程直接结束、启动命令错误、配置缺失，还是依赖不可达。

**33. 端口映射了却访问不到，你先看什么？**

- 应用是否启动成功
- 应用监听的是不是 `0.0.0.0`
- 宿主机端口是否真的映射
- 防火墙或安全组是否拦截

**34. 容器之间互访失败，你先看什么？**

- 是否在同一网络
- 服务名是否写对
- 目标服务是否真的 ready
- 应用监听端口是否正确

**35. 磁盘爆满时你先看什么？**

- `docker system df`
- 镜像、停止容器、孤立卷、构建缓存分别占了多少
- 再决定要不要 `prune`

**36. 构建突然变慢时你先看什么？**

- 哪一层开始不命中缓存
- `.dockerignore` 是否配置
- 依赖层和源码层是否拆开

## 9. 面试前至少要背下来的结论

- Docker 解决的是交付标准化和环境一致性
- 镜像是模板，容器是运行实例
- 容器共享宿主机内核，不是轻量虚拟机
- `namespace` 负责隔离，`cgroup` 负责资源边界
- 多阶段构建的核心是编译环境和运行环境分离
- tag 负责版本语义，digest 负责内容确定性
- 容器是否存活，本质上取决于 PID 1 对应的主进程
- `docker stop` 偏优雅退出，`docker kill` 偏强制终止
- `EXPOSE` 不等于对外开放端口
- 数据写在容器层不等于持久化
- Compose 适合单机交付，不适合代替 K8s 做集群编排
- 非 root、密钥不进镜像、tag 可回滚，是最基本的生产底线
- 排障不要猜，先看状态、日志、inspect、网络和挂载
