# Docker 安全与生产实践

> 容器能跑起来，不等于可上线。生产上真正要守住的是权限边界、镜像治理、配置注入、资源控制和可运维性。

## 1. 生产里最少要守住的底线

| 风险点 | 正确做法 |
| --- | --- |
| 容器默认以 root 运行 | 改成非 root 用户，最小权限 |
| 镜像过大、工具过多 | 多阶段构建，减少攻击面 |
| 密钥写进镜像 | 运行时注入，接 Secret 管理系统 |
| 使用漂移的基础镜像 | pin 版本或 digest，纳入扫描 |
| 滥用 `privileged` | 除非非常明确，否则不要开 |
| 随手挂 `/var/run/docker.sock` | 明确知道风险再用，否则等于把宿主机控制权交出去 |

## 2. 非 root 不是形式主义

很多教程为了图省事，容器里直接跑 root。开发环境可能还不明显，但生产里这是典型高风险默认值。

至少要理解：

- root 容器一旦被突破，横向风险更大
- root 写宿主机挂载目录时，容易制造权限脏数据
- 很多服务根本不需要 root 权限就能运行

最基本的做法：

```dockerfile
USER 10001:10001
```

或者使用镜像里预先创建的低权限用户：

```dockerfile
USER nonroot:nonroot
```

## 3. 权限边界不能只看用户

除了用户身份，还要看容器到底拿了什么能力。

需要特别警惕的配置：

- `privileged: true`
- `--cap-add=ALL`
- 挂载宿主机敏感目录
- 挂载 Docker socket

### 为什么 `docker.sock` 很危险

如果一个容器拿到了 Docker socket，它就可能：

- 再启动新的高权限容器
- 挂宿主机目录
- 访问其他容器
- 间接获得宿主机控制能力

标准口径：

> 挂 Docker socket 本质上不是“给容器更多工具”，而是把宿主机级控制权暴露给它。

## 4. 密钥和配置不要写进镜像

镜像应该尽量保持环境无关。

不该进镜像的内容：

- 数据库密码
- 云厂商密钥
- TLS 私钥
- 第三方 API token
- 按环境变化的配置值

更稳的做法：

- 环境变量或 `env-file`
- Docker Secret 或外部 Secret 管理
- 宿主机只读挂载配置文件
- 由编排平台注入配置

必须记住：

- 镜像一旦被推到仓库，泄露范围会远大于单机
- 把密钥写进镜像，即使后续删掉 Dockerfile 里的那行，历史镜像层里也可能还在

## 5. 基础镜像和供应链要纳入治理

生产镜像最好做到：

- 基础镜像来源可追溯
- tag 明确，不滥用 `latest`
- 定期补安全更新
- 纳入漏洞扫描

常见治理动作：

- 记录基础镜像来源
- 固定版本或 digest
- 做镜像扫描
- 尽量产出 SBOM
- 高要求场景增加签名校验
- 生产拉取账号最小权限化
- 关键发布 tag 设成不可变

如果团队已经在用 Harbor，这一层通常还会继续落到：

- Project 隔离
- Robot Account
- 保留策略和垃圾回收

## 6. 资源限制不是可选项

如果容器没有边界，单个实例就可能把宿主机资源吃穿。

最常见的限制项：

- 内存：`-m`
- CPU：`--cpus`
- 文件打开数、进程数等平台级限制

### 为什么资源限制重要

- 避免单个容器拖垮宿主机
- 让性能和容量评估更可控
- 在平台侧更容易做调度和治理

必须理解：

- “能跑”不代表“跑得稳”
- 不限资源时，出问题通常不是局部问题，而是宿主机级问题

## 7. 日志、退出和重启策略要统一

### 日志原则

- 应用日志优先输出到 stdout/stderr
- 不要在容器内自己造复杂日志轮转体系，除非场景非常明确
- 平台、宿主机或收集器负责统一采集

### 重启策略

常见策略：

- `no`
- `on-failure`
- `unless-stopped`
- `always`

单机守护型服务常见会用：

```bash
--restart unless-stopped
```

### 信号和优雅退出

主进程需要正确处理 `SIGTERM`，否则：

- 发布或重启时可能直接断请求
- 数据落盘、连接释放、缓冲刷新不完整

## 8. 健康检查要克制

健康检查有用，但不要做成重型业务探测。

适合检查：

- 进程是否还活着
- 基本 HTTP 健康端点是否可达

不适合默认每次都做：

- 全链路依赖探测
- 高成本 SQL
- 大量外部调用

## 9. 一个最小的生产清单

- 运行用户不是 root
- 基础镜像版本明确
- 密钥不在镜像里
- 镜像体积和攻击面可控
- 已配置资源限制
- 日志输出到 stdout/stderr
- 已考虑重启策略和优雅退出
- 镜像扫描纳入交付链
- 私有仓库和 tag 策略可回滚

## 10. 最常见的生产反模式

- 开发镜像直接拿去生产跑
- root 用户运行服务
- `latest` 到处流动
- 把 `.env` 和密钥打进镜像
- 容器内存无限制
- 依赖容器内 shell 手工修问题，而不是做可重复变更
