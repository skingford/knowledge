# Etcd 场景与模式：注册发现、配置中心、选主与分布式锁

## 服务注册与发现

这是 etcd 最经典的落地场景。

### 注册侧在做什么

一个服务实例启动后，典型流程是：

1. 创建 Lease
2. 把实例地址、端口、权重、zone 等信息写到某个前缀下
3. 让这个 Key 绑定 Lease
4. 后台持续 KeepAlive

例如：

- `/services/user/instance-1`
- `/services/user/instance-2`

这里实例 Key 的 value 通常会放：

- IP / 端口
- 协议类型
- 元数据标签
- zone / region
- 权重或版本信息

### 发现侧在做什么

消费者不是每次请求都实时查 etcd，而通常是：

1. 启动时全量 `Get` 目标服务前缀
2. 构建本地地址缓存
3. 启动 Watch 监听实例上下线
4. 本地做负载均衡选择

这能把 etcd 从高频请求链路里摘出来，让它只负责“状态同步”，不直接承担每个业务请求的查询压力。

### 为什么这套模式稳定

因为它天然把职责拆开了：

- etcd 负责强一致保存实例状态
- Lease 负责实例生命周期
- Watch 负责把变更同步给消费者
- 消费者本地缓存负责低延迟读

## 配置中心与配置热更新

配置中心和服务注册看起来很像，但关注点不同。

### 相同点

- 都依赖 KV 存储
- 都常用 Watch 做变更通知
- 都需要本地缓存最新配置

### 不同点

- 服务注册更强调“谁还活着”
- 配置中心更强调“哪个版本生效了”

### 一个更稳的配置模型

配置客户端通常会：

1. 启动时按前缀或指定 Key 拉取当前配置
2. 解析后写入本地内存配置对象
3. 用 Watch 追后续 revision
4. 收到变更后原子替换本地配置快照

这里最重要的不是“监听到变化就改字段”，而是要尽量做成**整份快照替换**，避免本地状态半更新。

### 配置更新时常见的两个问题

1. 先改了一半配置就被业务线程读到
2. 多个发布方并发覆盖，后写把前写静默冲掉

第一类问题靠本地原子替换解决，第二类问题通常靠 **CAS 更新** 解决。

## 基于 Txn 的 CAS 更新

etcd 的事务能力非常适合做“条件成立时才更新”。

### 最常见的 Compare 条件

- Key 不存在时才创建
- `mod_revision` 还是预期值时才更新
- `version == 0` 时说明这是首次写入

### 一个典型用途：防并发覆盖

假设配置发布平台先读到：

- `/config/payment`
- `mod_revision = 205`

编辑完成后，希望只在配置还没被别人改过的前提下写回。此时可以做：

- compare：`mod_revision == 205`
- then：执行新的 `put`
- else：返回“配置已被他人更新，请重新拉取”

这就是 etcd 版的乐观锁。

### etcd 事务更适合什么

- 条件写入
- 幂等创建
- 抢占式更新
- 锁和选主

### 不适合期待什么

- 跨大量业务对象的复杂事务
- 长事务
- 像关系型数据库那样的多表事务语义

## Leader 选举模式

很多系统里只允许一个节点执行关键任务，例如：

- 定时调度器主节点
- 主控任务分发器
- 某类资源的唯一协调者

这时 etcd 常被用来做 Leader 选举。

### 选主的常见思路

1. 为候选节点申请 Lease
2. 通过 `Txn` 尝试创建选举 Key
3. 创建成功的节点成为 Leader
4. 持续 KeepAlive 维持领导权
5. Lease 失效后，其他节点重新竞争

关键点不在于“有一个 Key”，而在于：

- 成为 Leader 必须是**原子竞争**
- 领导权必须能**自动失效**

所以选主本质上是：`Compare + Put + Lease`

### 选主时要特别注意什么

- Leader 身份最好和 Session / Lease 绑定
- 业务线程要能感知自己是否已经失去 Lease
- 失去领导权后要尽快停止关键任务，避免双主

## 分布式锁模式

分布式锁和选主非常像，本质上都是“唯一拥有者”问题。

### 锁的核心目标

- 同一时刻只有一个持有者
- 持有者故障后，锁能自动释放
- 其他竞争者能继续尝试获取

### etcd 里的稳妥做法

1. 创建 Lease
2. 用 `Txn(compare create_revision == 0)` 或“Key 不存在”条件去抢锁
3. 抢到后把锁 Key 绑定 Lease
4. 持有期间 KeepAlive
5. 业务完成后主动删除或让 Lease 结束

### 锁最容易出事的地方

- 只做 `put`，不做 CAS，多个客户端会互相覆盖
- 锁不绑 Lease，客户端挂了以后锁永远不释放
- 锁拿到后本地停顿太久，Lease 已失效却还以为自己持有锁

所以真正要保护的不是“拿锁那一刻”，而是“整个持锁生命周期”。

## Watch 客户端的本地缓存模型

etcd 的高频读场景，常见做法都不是“每次都查 etcd”，而是本地维护一个快照。

### 稳定模型

1. 全量 `Get`
2. 记录返回头里的 revision
3. 从 `revision + 1` 启动 Watch
4. 事件到来时按顺序修改本地缓存
5. 如果 Watch 中断或被 compaction 打断，则全量重建

### 这个模型适合哪些场景

- 服务实例列表
- 配置项字典
- 灰度路由元数据
- 任务分配与 worker 状态

### 为什么不要把 etcd 放进业务热路径

如果每次业务请求都同步查询 etcd：

- etcd 会承担不必要的读压
- 网络抖动会直接放大到业务请求时延
- 客户端会失去本地缓存带来的降本和降延迟优势

更稳的方式通常是“etcd 做控制面，本地缓存做数据面”。

## 常见误区与设计边界

### 1. 用 etcd 存大量配置详情或大 JSON

问题：

- 更新成本高
- Watch 负担重
- 版本和历史清理都会变得更沉重

更好的做法：

- etcd 放关键索引、开关、版本号、路由元数据
- 大对象放数据库或对象存储

### 2. 把 Watch 当消息队列

问题：

- Watch 是状态变更流，不是业务消息队列
- 不适合依赖它承载海量、长期堆积、必须逐条消费确认的业务消息

### 3. 只做客户端 SDK 调用，不做失效恢复

问题：

- 断线重连
- revision 补偿
- compacted 回跳
- Lease 丢失感知

这些如果不做，系统在“正常时看起来能跑”，但一到网络抖动和重启恢复就会露问题。

### 4. 以为拿到锁就万无一失

问题：

- GC 停顿
- 网络分区
- Lease 已过期但本地没及时感知

更稳的做法：

- 关键逻辑里持续校验 Session / Lease 状态
- 让锁和业务执行边界绑定，而不是只在入口检查一次

## 小结

把 etcd 的常见场景压缩一下，其实就是三套模式：

- 服务注册 / 配置推送：`KV + Lease + Watch`
- 选主 / 分布式锁：`Txn + Compare + Lease`
- 并发保护 / 幂等更新：`Txn + mod_revision`
