## 语言与底层

### 1. Slice 扩容机制是怎样的？

回答模板：

- 先说结论：Slice 本身不是数组，而是对底层数组的一个描述，`append` 超过容量时会触发扩容。
- 再说原理：扩容时 Runtime 会申请一块更大的新数组，把旧数据拷贝过去，然后返回新的 Slice 描述符。
- 再说能力自检重点：如果扩容前后仍共用底层数组，原 Slice 和新 Slice 的数据可能互相影响；如果已经扩容到新数组，就不会互相影响。
- 再补一句工程视角：高频 `append` 需要关注扩容带来的拷贝和分配成本，必要时提前 `make` 容量。

### 2. Map 为什么并发不安全？

回答模板：

- 先说结论：原生 Map 不支持并发读写。
- 再说原理：Map 写入时可能触发扩容和 bucket 搬迁，这时候并发读写会破坏内部状态。
- 再说表现：严重时会直接触发 `concurrent map read and map write`。
- 再补工程建议：简单共享状态用 Mutex，读多写少并且场景合适时可以考虑 `sync.Map`。

### 3. Interface 的底层结构是什么？

回答模板：

- 先说结论：Interface 底层保存的是类型信息和数据地址。
- 再说细节：空接口和非空接口的底层表示不同，非空接口还需要方法集相关信息。
- 再说能力自检重点：这也是接口调用有动态派发成本的原因之一。
- 再补场景：在高频路径里，大量 interface 转换和反射通常会带来额外开销。

### 4. Nil Interface 为什么容易出错？

回答模板：

- 先说结论：Interface 是否等于 nil，要看动态类型和值是否都为空。
- 再举典型例子：`var err error = (*MyErr)(nil)`，这个时候 err 不等于 nil，因为类型信息还在。
- 再说能力自检重点：很多 error 判断 bug 就出在这里。
- 再补建议：调试时要区分“接口是 nil”还是“接口里装的是 nil 指针”。

### 5. Defer 的执行顺序和参数求值规则是什么？

回答模板：

- 先说结论：`defer` 是后进先出，参数在注册那一刻就完成求值。
- 再说区别：普通 `defer` 用的是当时的参数值，闭包 `defer` 更容易拿到后续变化后的变量。
- 再说能力自检重点：分析多个 `defer` 修改返回值时，一定要分清注册顺序和执行顺序。
- 再补工程建议：复杂返回逻辑里不要过度依赖 `defer` 改返回值，否则可读性会变差。

### 6. Panic 和 Recover 应该怎么回答？

回答模板：

- 先说结论：`panic` 用于不可恢复错误，`recover` 只能在 `defer` 中捕获正在展开的 `panic`。
- 再说原理：发生 `panic` 后，函数会沿调用栈向上展开，并依次执行 `defer`。
- 再说能力自检重点：`recover` 放错位置不会生效，因为它必须处在当前展开链上的 `defer` 里。
- 再补工程建议：正常业务错误优先用 `error` 返回，不要把 `panic` 当普通控制流。

### 7. 逃逸分析是什么？

回答模板：

- 先说结论：逃逸分析是编译器判断变量应该分配在栈上还是堆上的过程。
- 再说原理：如果对象在函数外仍可能被引用，编译器通常会让它逃逸到堆上。
- 再说能力自检重点：逃逸到堆不一定是 bug，但会增加分配和 GC 压力。
- 再补工程建议：可以用 `-gcflags=-m` 辅助分析高频路径上的逃逸情况。

### 8. String 和 `[]byte` 转换时要注意什么？

回答模板：

- 先说结论：两者转换通常会带来拷贝和分配成本。
- 再说原理：String 是只读语义，`[]byte` 是可变字节切片，因此运行时需要保证语义安全。
- 再说能力自检重点：高频路径里大量字符串和字节切片转换，往往是性能热点来源之一。
- 再补工程建议：要关注是否真的需要转换，必要时减少中间态对象。

## 并发与调度

### 9. 什么是 GMP 调度模型？

回答模板：

- 先说结论：G 是 Goroutine，M 是线程，P 是运行 Go 代码必须持有的调度资源。
- 再说原理：M 真正映射系统线程，P 负责把可运行的 G 调度到 M 上执行。
- 再说为什么有 P：如果只有 G 和 M，调度和本地队列管理会更难做，P 的存在让调度、缓存和并发控制更高效。
- 再补能力自检点：工作窃取和本地队列机制，都是建立在 P 的基础上的。

### 10. Goroutine 为什么轻量？

回答模板：

- 先说结论：因为它初始栈小、可动态扩缩，而且由 Runtime 调度，不像线程那样直接绑定系统资源。
- 再说原理：线程创建和切换成本更高，Goroutine 则把这部分复杂度收敛到了 Runtime。
- 再说边界：轻量不代表没成本，大量 Goroutine 仍会占用内存、调度资源和下游连接资源。
- 再补工程建议：Goroutine 数量要和业务吞吐、IO、下游承载能力一起看。

### 11. Channel 和 Mutex 怎么选？

回答模板：

- 先说结论：共享状态保护优先考虑 Mutex，任务和数据传递优先考虑 Channel。
- 再说原因：Mutex 直接保护临界区，成本更直接；Channel 更适合做协程间通信和同步。
- 再说误区：不要为了“看起来优雅”把所有并发问题都改写成 Channel。
- 再补工程建议：先根据问题类型选工具，再谈风格。

### 12. Channel 的底层原理怎么讲？

回答模板：

- 先说结论：Channel 底层不只是一个队列，它同时承担通信和同步职责。
- 再说原理：内部会维护缓冲区、发送等待队列、接收等待队列和锁。
- 再说能力自检重点：无缓冲 Channel 的发送和接收必须配对完成，所以天然有同步语义。
- 再补工程建议：遇到 Channel 阻塞问题，要先想清楚是否有接收方、发送方和退出路径。

### 13. Select 的作用是什么？

回答模板：

- 先说结论：`select` 用来同时处理多个 Channel 操作。
- 再说原理：它会在多个可执行通信分支里选择一个执行，非常适合超时、取消和多路收发。
- 再说能力自检重点：它是 Go 并发控制里把“通信”和“状态判断”连起来的关键语法。
- 再补工程建议：和 `time.After`、`ctx.Done()` 配合时，要注意资源释放和超时管理。

### 14. Context 为什么重要？

回答模板：

- 先说结论：Context 是 Go 服务里超时、取消和请求链路控制的基础机制。
- 再说原理：父 Context 取消后，子 Context 都会收到信号，因此适合管理请求全链路。
- 再说工程价值：它能防止请求超时后后台任务还在继续跑，减少资源泄漏。
- 再补边界：不要滥用 `WithValue` 传业务对象，它更适合少量请求级上下文信息。

### 15. Atomic 和 Mutex 的区别是什么？

回答模板：

- 先说结论：Atomic 适合简单原子变量操作，Mutex 适合保护复杂共享状态。
- 再说原理：Atomic 依赖 CPU 原子指令，适合计数器、状态位等简单场景；Mutex 保护的是临界区。
- 再说能力自检重点：Atomic 不是“更高级的锁”，它只是适用范围更窄、开销更小。
- 再补工程建议：一旦涉及多个字段一致性或复杂状态机，优先用锁而不是拼 Atomic。

### 16. `sync.Map` 适合什么场景？

回答模板：

- 先说结论：`sync.Map` 适合读多写少、键动态变化的并发场景。
- 再说原理：它内部通过读写分层来减少锁竞争。
- 再说能力自检重点：它不是普通 Map 的通用替代品，很多场景直接用 Mutex + Map 反而更清晰。
- 再补工程建议：是否使用 `sync.Map`，要以访问模式和性能数据为依据。

### 17. 什么是 Goroutine 泄漏，怎么排查？

回答模板：

- 先说结论：Goroutine 泄漏就是本该结束的协程长期不退出。
- 再说常见原因：Channel 无人消费、Context 没取消、下游调用卡死、重试逻辑无限循环。
- 再说排查方法：先看 Goroutine Profile，再看堆积在什么调用点，最后回到代码找退出条件。
- 再补工程建议：凡是起 Goroutine，都要明确退出条件和取消路径。

### 18. WaitGroup 使用时的关键注意点是什么？

回答模板：

- 先说结论：WaitGroup 本质是等待计数归零。
- 再说原理：主协程调用 `Wait`，工作协程结束时调用 `Done`，等计数器归零后继续执行。
- 再说能力自检重点：`Add` 最好在启动 Goroutine 前完成，否则容易出现计数不一致问题。
- 再补工程建议：WaitGroup 负责“等结束”，不负责错误传递和取消控制，这些要结合 Context 或其他机制。

## 性能与排障

### 19. Go 的 GC 是如何工作的？

回答模板：

- 先说结论：Go 采用并发垃圾回收，目标是尽量降低停顿时间。
- 再说原理：核心是并发标记和清扫，配合写屏障维持对象引用关系的正确性。
- 再说能力自检重点：GC 追求的是低停顿和整体吞吐平衡，不是单纯“回收越彻底越好”。
- 再补工程视角：频繁分配、大对象和长生命周期对象都会加大 GC 压力。

### 20. GC 为什么会影响延迟？

回答模板：

- 先说结论：因为对象越多、分配越频繁，GC 的扫描和回收成本就越高。
- 再说原理：即使是并发 GC，也仍然需要一定的协作和停顿成本。
- 再说能力自检重点：GC 影响延迟的核心不是“有没有 GC”，而是分配模式和对象生命周期。
- 再补工程建议：性能优化时要优先关注分配频率和对象存活时间。

### 21. 如何用 Pprof 排查性能问题？

回答模板：

- 先说结论：先确定问题类型，再选对应 Profile。
- 再说方法：CPU 高看 CPU Profile，内存高看 Heap，Goroutine 堆积看 Goroutine，锁竞争看 Mutex，阻塞看 Block。
- 再说排查顺序：先看热点函数，再看调用链，最后回到业务请求和代码验证。
- 再补建议：不要只看采样图就直接优化，必须结合压测或线上现象验证收益。

### 22. 如何排查内存泄漏？

回答模板：

- 先说结论：内存泄漏本质上是对象被长期持有，GC 无法回收。
- 再说排查方法：看 Heap Profile、对象引用链、缓存生命周期和长连接持有关系。
- 再说常见来源：全局缓存无上限、切片或字符串长期持有大对象、协程不退出。
- 再补工程建议：先确认是真的泄漏，还是业务峰值带来的正常内存上涨。

### 23. 如何排查慢请求？

回答模板：

- 先说结论：慢请求通常不是单点问题，而是日志、下游、数据库、锁竞争、GC 等因素共同作用的结果。
- 再说排查顺序：先看请求耗时分布，再看 Trace、慢 SQL、Pprof、外部依赖耗时。
- 再说能力自检重点：要体现“逐层缩小范围”的思路，而不是直接猜代码有问题。
- 再补工程建议：没有可观测性，慢请求排查会非常低效。

### 24. `sync.Pool` 适合什么场景？

回答模板：

- 先说结论：适合短生命周期、频繁分配、可复用对象的临时缓存。
- 再说原理：它的目标是减少分配和 GC 压力，不是做长期对象池。
- 再说边界：对象可能随 GC 被清掉，所以不能依赖它保存状态。
- 再补工程建议：适合高频请求里的 Buffer、临时结构体等复用。

### 25. 如何减少内存分配？

回答模板：

- 先说结论：减少临时对象、减少拷贝、控制字符串和切片转换、复用对象，通常是最直接的方法。
- 再说原理：分配少了，GC 压力自然会下降。
- 再说能力自检重点：减少分配的目标不是“代码看起来高级”，而是降低 CPU 和 GC 成本。
- 再补工程建议：优化前先测量，避免为了省一点分配把代码复杂度拉高。

## 系统设计结合 Go

### 26. `net/http` 使用时要注意什么？

回答模板：

- 先说结论：`net/http` 很强大，但连接复用、超时、Transport 配置这些细节非常关键。
- 再说原理：服务端和客户端都内置了连接管理，尤其是 `http.Client` 和 `Transport` 的复用策略。
- 再说能力自检重点：错误使用 `http.Client`，例如频繁新建客户端、请求体不及时关闭，都会导致连接问题。
- 再补工程建议：客户端要关注超时、连接池、空闲连接复用，服务端要关注中间件、限流和可观测性。

### 27. Go 服务如何做高并发优化？

回答模板：

- 先说结论：高并发优化不是只改代码，而是要同时看 CPU、内存、锁、IO、下游依赖。
- 再说手段：减少分配、控制 Goroutine 数量、优化锁竞争、加缓存、做超时和限流、必要时异步化。
- 再说顺序：先测量，再定位，再做局部优化。
- 再补工程建议：没有压测和指标支撑的优化，很容易变成“自我感动”。

### 28. Go 在微服务架构中的优势是什么？

回答模板：

- 先说结论：Go 在服务端的优势主要是部署简单、启动快、并发模型清晰、资源占用相对可控。
- 再说工程价值：适合做 API 服务、网关、中间件、基础设施组件。
- 再说边界：并不是所有业务都天然适合 Go，团队经验和生态配套也要考虑。
- 再补能力自检建议：回答时不要只讲“快”，要讲为什么适合服务端。

### 29. 用 Go 设计高并发接口时重点看什么？

回答模板：

- 先说结论：重点不是语言本身，而是围绕语言特性把限流、超时、幂等、缓存、数据库压力控制好。
- 再说实现层：合理并发、控制连接池、减少热点锁、必要时异步化。
- 再说系统层：要考虑监控、告警、熔断、降级和故障兜底。
- 再补能力自检建议：回答时要同时覆盖代码层、运行时层和系统层。

### 30. Go 在服务拆分和可观测性里扮演什么角色？

回答模板：

- 先说结论：Go 很适合做微服务、网关和基础设施组件，因为它在并发、部署和资源控制上比较平衡。
- 再说工程价值：服务拆分后，Go 适合承载链路追踪、日志埋点、指标采集这类工程能力。
- 再说能力自检重点：不要只说“Go 快”，要说明它如何帮助稳定性、治理和交付效率。
- 再补建议：设计服务时，语言只是手段，可观测性和治理能力才决定长期可维护性。
