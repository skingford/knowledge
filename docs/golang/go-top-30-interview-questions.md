## 基础与底层

### 1. Slice 的底层结构是什么？

:::tabs
== 简答
底层是一个包含指针、长度、容量的结构体，分别指向底层数组、当前长度和容量。
== 源码线索
- 关键词：`runtime.slice`、切片描述符
- 重点原理：切片不是数组本身，而是对底层数组的一个描述
- 延伸：Slice / Map / Channel 传参为什么看起来像引用传递 → Go 全是值传递，只是描述符里装的是指针
:::

### 2. Slice 扩容机制是怎样的？

:::tabs
== 简答
追加元素超过容量时会扩容。小容量通常增长更快，大容量增长更平滑，具体比例和 Go 版本实现有关。
== 标准回答模板
- 先说结论：Slice 本身不是数组，而是对底层数组的一个描述，`append` 超过容量时会触发扩容。
- 再说原理：扩容时 Runtime 会申请一块更大的新数组，把旧数据拷贝过去，然后返回新的 Slice 描述符。
- 再说能力自检重点：如果扩容前后仍共用底层数组，原 Slice 和新 Slice 的数据可能互相影响；如果已经扩容到新数组，就不会互相影响。
- 再补一句工程视角：高频 `append` 需要关注扩容带来的拷贝和分配成本，必要时提前 `make` 容量。
== 源码线索
- 关键词：`growslice`
- 重点原理：容量不够时申请新数组并迁移旧数据
- 延伸：为什么 `append` 后原切片和新切片有时互相影响、有时不影响
:::

### 3. Map 为什么并发不安全？

:::tabs
== 简答
因为读写过程中可能触发扩容和迁移，并发访问会破坏内部状态，所以原生 Map 不保证并发安全。
== 标准回答模板
- 先说结论：原生 Map 不支持并发读写。
- 再说原理：Map 写入时可能触发扩容和 bucket 搬迁，这时候并发读写会破坏内部状态。
- 再说表现：严重时会直接触发 `concurrent map read and map write`。
- 再补工程建议：简单共享状态用 Mutex，读多写少并且场景合适时可以考虑 `sync.Map`。
== 源码线索
- 关键词：`hmap`、`bucket`、`overflow bucket`
- 重点原理：Map 通过哈希定位 bucket，同一 bucket 内再找 key；写入时可能触发扩容和 bucket 搬迁，并发读写会破坏内部状态
- 延伸：为什么 key 类型会影响性能；为什么会直接 panic `concurrent map read and map write`
:::

::: danger 警告
原生 Map 并发读写会直接触发 `concurrent map read and map write` panic，这不是数据竞争的"可能后果"，而是 Runtime 的强制检查。生产环境中务必使用 Mutex 或 `sync.Map` 保护。
:::

### 4. Channel 的底层原理是什么？

:::tabs
== 简答
Channel 内部维护队列、发送等待队列、接收等待队列和锁，用于在 Goroutine 之间做安全通信与同步。
== 标准回答模板
- 先说结论：Channel 底层不只是一个队列，它同时承担通信和同步职责。
- 再说原理：内部会维护缓冲区、发送等待队列、接收等待队列和锁。
- 再说能力自检重点：无缓冲 Channel 的发送和接收必须配对完成，所以天然有同步语义。
- 再补工程建议：遇到 Channel 阻塞问题，要先想清楚是否有接收方、发送方和退出路径。
== 源码线索
- 关键词：`hchan`、`sendq`、`recvq`
- 重点原理：内部有缓冲区、发送队列、接收队列、锁；发送或接收无法立即完成时，Goroutine 会被挂到等待队列
- 延伸：无缓冲 Channel 为什么既是通信也是同步；死锁出现的根本原因
:::

### 5. Interface 的底层结构是什么？

:::tabs
== 简答
Interface 本质上包含类型信息和数据指针。空接口和非空接口在底层表示上有所不同。
== 标准回答模板
- 先说结论：Interface 底层保存的是类型信息和数据地址。
- 再说细节：空接口和非空接口的底层表示不同，非空接口还需要方法集相关信息。
- 再说能力自检重点：这也是接口调用有动态派发成本的原因之一。
- 再补场景：在高频路径里，大量 interface 转换和反射通常会带来额外开销。
== 源码线索
- 关键词：`iface`（非空接口）、`eface`（空接口）
- 重点原理：接口值包含类型信息和数据地址
- 延伸：为什么接口调用有动态派发成本
:::

### 6. Nil Interface 和 Nil 指针有什么区别？

:::tabs
== 简答
Nil Interface 是类型和值都为空；如果 Interface 里装的是一个类型已知但值为 nil 的指针，它本身并不等于 nil。
== 标准回答模板
- 先说结论：Interface 是否等于 nil，要看动态类型和值是否都为空。
- 再举典型例子：`var err error = (*MyErr)(nil)`，这个时候 err 不等于 nil，因为类型信息还在。
- 再说能力自检重点：很多 error 判断 bug 就出在这里。
- 再补建议：调试时要区分"接口是 nil"还是"接口里装的是 nil 指针"。
== 源码线索
- 关键词：动态类型、动态值
- 重点原理：接口是否为 nil，要看"类型"和"值"是否都为空
- 延伸：为什么 `var err error = (*MyErr)(nil)` 不等于 nil
:::

::: warning 注意
`var err error = (*MyErr)(nil)` 中 err 不等于 nil，这是 Go 中最常见的接口陷阱之一。返回 error 时，避免将具体类型的 nil 指针赋给 interface 变量。
:::

### 7. Defer 的执行顺序是什么？

:::tabs
== 简答
后定义的 `defer` 先执行，整体是后进先出。
== 标准回答模板
- 先说结论：`defer` 是后进先出，参数在注册那一刻就完成求值。
- 再说区别：普通 `defer` 用的是当时的参数值，闭包 `defer` 更容易拿到后续变化后的变量。
- 再说能力自检重点：分析多个 `defer` 修改返回值时，一定要分清注册顺序和执行顺序。
- 再补工程建议：复杂返回逻辑里不要过度依赖 `defer` 改返回值，否则可读性会变差。
== 源码线索
- 关键词：后进先出、定义时求值
- 重点原理：`defer` 本质上像压栈，返回前再逐个弹出执行；参数在注册那一刻就确定了
- 延伸：多个 `defer` 修改返回值时要怎么分析；为什么闭包 `defer` 和普通 `defer` 表现不同
:::

### 8. Panic 和 Recover 的正确使用方式是什么？

:::tabs
== 简答
`panic` 用于不可恢复错误，`recover` 只能在 `defer` 中捕获。正常业务错误不应该大量依赖 `panic`。
== 标准回答模板
- 先说结论：`panic` 用于不可恢复错误，`recover` 只能在 `defer` 中捕获正在展开的 `panic`。
- 再说原理：发生 `panic` 后，函数会沿调用栈向上展开，并依次执行 `defer`。
- 再说能力自检重点：`recover` 放错位置不会生效，因为它必须处在当前展开链上的 `defer` 里。
- 再补工程建议：正常业务错误优先用 `error` 返回，不要把 `panic` 当普通控制流。
== 源码线索
- 关键词：异常展开、`defer` 链
- 重点原理：`panic` 会沿调用栈向上展开，`recover` 只能在 `defer` 中截获
- 延伸：为什么 `recover` 放错位置不生效
:::

::: warning 注意
不要把 `panic` 当作普通控制流使用。正常业务错误应优先通过 `error` 返回值处理，`panic` 仅用于不可恢复的程序错误。
:::

### 9. 什么是逃逸分析？

:::tabs
== 简答
编译器判断对象应该分配在栈上还是堆上的过程。逃逸到堆会增加 GC 压力。
== 标准回答模板
- 先说结论：逃逸分析是编译器判断变量应该分配在栈上还是堆上的过程。
- 再说原理：如果对象在函数外仍可能被引用，编译器通常会让它逃逸到堆上。
- 再说能力自检重点：逃逸到堆不一定是 bug，但会增加分配和 GC 压力。
- 再补工程建议：可以用 `-gcflags=-m` 辅助分析高频路径上的逃逸情况。
== 源码线索
- 关键词：编译器分析、`-gcflags=-m`；栈（生命周期短、分配快）vs 堆（增加 GC 压力）
- 重点原理：变量如果在函数外仍可能被引用，就会分配到堆上
- 延伸：哪些写法容易触发逃逸；为什么逃逸不一定是 bug，但常常是优化点
:::

::: tip 建议
使用 `go build -gcflags=-m` 可以直观查看哪些变量逃逸到了堆上，是性能优化的实用起点。
:::

### 10. String 和 `[]byte` 转换要注意什么？

:::tabs
== 简答
转换通常会涉及拷贝和分配，要关注性能影响，尤其是在高频路径中。
== 标准回答模板
- 先说结论：两者转换通常会带来拷贝和分配成本。
- 再说原理：String 是只读语义，`[]byte` 是可变字节切片，因此运行时需要保证语义安全。
- 再说能力自检重点：高频路径里大量字符串和字节切片转换，往往是性能热点来源之一。
- 再补工程建议：要关注是否真的需要转换，必要时减少中间态对象。
:::

## 并发与调度

### 11. Goroutine 为什么轻量？

:::tabs
== 简答
初始栈小、按需扩缩，调度由 Runtime 管理，不直接一一对应系统线程，所以创建和切换成本更低。
== 标准回答模板
- 先说结论：因为它初始栈小、可动态扩缩，而且由 Runtime 调度，不像线程那样直接绑定系统资源。
- 再说原理：线程创建和切换成本更高，Goroutine 则把这部分复杂度收敛到了 Runtime。
- 再说边界：轻量不代表没成本，大量 Goroutine 仍会占用内存、调度资源和下游连接资源。
- 再补工程建议：Goroutine 数量要和业务吞吐、IO、下游承载能力一起看。
== 源码线索
- 关键词：小栈、动态扩缩栈
- 重点原理：Goroutine 初始栈很小，由 Runtime 调度，不像线程那样创建成本高
- 延伸：大量 Goroutine 是否真的"没有成本"
:::

### 12. 什么是 GMP 模型？

:::tabs
== 简答
G 是 Goroutine，M 是线程，P 是调度上下文。P 负责把 G 调度到 M 上执行。
== 标准回答模板
- 先说结论：G 是 Goroutine，M 是线程，P 是运行 Go 代码必须持有的调度资源。
- 再说原理：M 真正映射系统线程，P 负责把可运行的 G 调度到 M 上执行。
- 再说为什么有 P：如果只有 G 和 M，调度和本地队列管理会更难做，P 的存在让调度、缓存和并发控制更高效。
- 再补能力自检点：工作窃取和本地队列机制，都是建立在 P 的基础上的。
== 源码线索
- 关键词：`G`、`M`、`P`；`work stealing`（工作窃取）
- 重点原理：P 是执行 Go 代码必须持有的调度资源，M 真正映射系统线程；某个 P 本地队列空了，会去别的 P 偷 Goroutine 来执行
- 延伸：为什么不是只有 G 和 M 还要多一个 P；工作窃取如何提升 CPU 利用率
:::

### 13. Go 的抢占式调度是什么？

Runtime 会在合适时机中断长时间运行的 Goroutine，避免其长期占用 CPU，提升调度公平性。

::: tip 建议
理解抢占式调度时，重点对比老版本协作式调度与新版本基于信号的异步抢占的差异，这是体现 Go 调度演进理解深度的好角度。
:::

**源码线索：** 关键词：Safe Point、异步抢占

### 14. Channel 和 Mutex 怎么选？

:::tabs
== 简答
共享状态保护优先考虑 Mutex；任务和数据传递优先考虑 Channel，不要为了"优雅"滥用 Channel。
== 标准回答模板
- 先说结论：共享状态保护优先考虑 Mutex，任务和数据传递优先考虑 Channel。
- 再说原因：Mutex 直接保护临界区，成本更直接；Channel 更适合做协程间通信和同步。
- 再说误区：不要为了"看起来优雅"把所有并发问题都改写成 Channel。
- 再补工程建议：先根据问题类型选工具，再谈风格。
:::

### 15. Select 有什么作用？

:::tabs
== 简答
`select` 用于同时监听多个 Channel 操作，适合超时控制、多路收发和取消信号处理。
== 标准回答模板
- 先说结论：`select` 用来同时处理多个 Channel 操作。
- 再说原理：它会在多个可执行通信分支里选择一个执行，非常适合超时、取消和多路收发。
- 再说能力自检重点：它是 Go 并发控制里把"通信"和"状态判断"连起来的关键语法。
- 再补工程建议：和 `time.After`、`ctx.Done()` 配合时，要注意资源释放和超时管理。
:::

### 16. Context 有什么用？

:::tabs
== 简答
用于传递取消信号、超时和少量请求级元数据，是请求链路控制、资源回收和服务超时治理的重要机制。
== 标准回答模板
- 先说结论：Context 是 Go 服务里超时、取消和请求链路控制的基础机制。
- 再说原理：父 Context 取消后，子 Context 都会收到信号，因此适合管理请求全链路。
- 再说工程价值：它能防止请求超时后下游调用和请求内 Goroutine 继续空跑，减少资源泄漏。
- 再补边界：`WithValue` 只适合少量请求级元数据，不要拿它传业务对象或依赖。
- 再补 HTTP 语境：服务端 `r.Context()` 会在客户端断开、HTTP/2 请求取消，或 `ServeHTTP` 返回时失效。
- 再补异步任务：如果任务要在请求结束后继续执行，不要直接复用 `r.Context()`，而要改用 `context.Background()` 或 Go 1.21+ 的 `context.WithoutCancel()`，并重新设置超时。
== 源码线索
- 关键词：链式取消、超时传播、`withoutCancelCtx`
- 重点原理：父 Context 取消后，子 Context 都会收到信号
- 延伸：为什么 Context 是服务超时治理的核心；使用边界详见 [Context 使用边界](./context-usage-boundaries.md)，源码实现详见 [context 包源码精读](./guide/source-reading/context.md)
:::

### 17. 为什么 Context 不应该滥用 `WithValue`？

因为它会让依赖关系变隐式，降低可读性和可维护性，而且 `valueCtx` 查找本身也是沿父链逐层向上找。它只适合少量跨层协作信息，不适合业务对象、配置或下游依赖。

::: warning 注意
`WithValue` 只适合传递 trace ID、request ID 等少量跨层协作信息。切记不要用它传递业务对象、配置或下游依赖，否则会导致隐式耦合和调试困难。
:::

### 18. 什么是 Goroutine 泄漏？

:::tabs
== 简答
Goroutine 本该退出却长期不退出，通常由 Channel 阻塞、缺少取消逻辑、死循环重试等导致。
== 标准回答模板
- 先说结论：Goroutine 泄漏就是本该结束的协程长期不退出。
- 再说常见原因：Channel 无人消费、Context 没取消、下游调用卡死、重试逻辑无限循环。
- 再说排查方法：先看 Goroutine Profile，再看堆积在什么调用点，最后回到代码找退出条件。
- 再补工程建议：凡是起 Goroutine，都要明确退出条件和取消路径。
== 源码线索
- 关键词：阻塞等待、未退出协程
- 重点原理：Goroutine 不会自动回收，只有执行完才结束
- 延伸：最常见是 Channel 无人消费、Context 没取消、下游调用卡死
:::

::: danger 警告
凡是启动 Goroutine，都必须明确退出条件和取消路径。没有退出机制的 Goroutine 就是潜在的内存泄漏。
:::

### 19. Atomic 和 Mutex 的区别是什么？

:::tabs
== 简答
Atomic 适合简单原子变量操作，开销更小；Mutex 适合保护复杂共享状态。
== 标准回答模板
- 先说结论：Atomic 适合简单原子变量操作，Mutex 适合保护复杂共享状态。
- 再说原理：Atomic 依赖 CPU 原子指令，适合计数器、状态位等简单场景；Mutex 保护的是临界区。
- 再说能力自检重点：Atomic 不是"更高级的锁"，它只是适用范围更窄、开销更小。
- 再补工程建议：一旦涉及多个字段一致性或复杂状态机，优先用锁而不是拼 Atomic。
== 源码线索
- 关键词：CAS、原子指令（Atomic）；自旋、阻塞（Mutex）；读写分离（RWMutex）
- 重点原理：Atomic 依赖 CPU 原子操作实现无锁更新；Mutex 锁竞争不激烈时可能先自旋，再挂起等待；RWMutex 允许多个读者并发，但写者需要独占
- 延伸：为什么 Atomic 不能替代所有锁；为什么读多写少才适合 RWMutex，否则可能更慢
:::

### 20. `sync.Map` 适合什么场景？

:::tabs
== 简答
适合读多写少、键集合动态变化的并发场景，不适合作为普通 Map 的默认替代品。
== 标准回答模板
- 先说结论：`sync.Map` 适合读多写少、键动态变化的并发场景。
- 再说原理：它内部通过读写分层来减少锁竞争。
- 再说能力自检重点：它不是普通 Map 的通用替代品，很多场景直接用 Mutex + Map 反而更清晰。
- 再补工程建议：是否使用 `sync.Map`，要以访问模式和性能数据为依据。
== 源码线索
- 关键词：Read Map、Dirty Map
- 重点原理：通过读写分层减少锁竞争
- 延伸：为什么它适合特定场景，不是 Map 的通用替代品
:::

### 21. WaitGroup 使用时的关键注意点是什么？

- 先说结论：WaitGroup 本质是等待计数归零。
- 再说原理：主协程调用 `Wait`，工作协程结束时调用 `Done`，等计数器归零后继续执行。
- 再说能力自检重点：`Add` 最好在启动 Goroutine 前完成，否则容易出现计数不一致问题。
- 再补工程建议：WaitGroup 负责"等结束"，不负责错误传递和取消控制，这些要结合 Context 或其他机制。

::: warning 注意
`Add` 必须在启动 Goroutine 之前调用，否则 `Wait` 可能在所有 Goroutine 注册前就返回，导致计数不一致。
:::

**源码线索：** 关键词：计数器同步

## 工程与性能

### 22. Go 的 GC 是如何工作的？

:::tabs
== 简答
Go 使用并发垃圾回收，以降低停顿时间，但频繁分配和大对象仍会对延迟产生影响。
== 标准回答模板
- 先说结论：Go 采用并发垃圾回收，目标是尽量降低停顿时间。
- 再说原理：核心是并发标记和清扫，配合写屏障维持对象引用关系的正确性。
- 再说能力自检重点：GC 追求的是低停顿和整体吞吐平衡，不是单纯"回收越彻底越好"。
- 再补工程视角：频繁分配、大对象和长生命周期对象都会加大 GC 压力。
== 源码线索
- 关键词：三色标记、并发标记、清扫；`write barrier`（写屏障）
- 重点原理：Go 通过并发 GC 尽量降低 STW 时间；并发标记时需要写屏障记录对象引用变化，保证三色不变式
- 延伸：GC 的目标是低停顿，不只是"回收干净"；为什么并发 GC 离不开写屏障
:::

### 23. GC 为什么会影响延迟？

:::tabs
== 简答
对象越多、分配越频繁、存活时间越长，GC 的扫描和回收成本越高，可能带来延迟抖动。
== 标准回答模板
- 先说结论：因为对象越多、分配越频繁，GC 的扫描和回收成本就越高。
- 再说原理：即使是并发 GC，也仍然需要一定的协作和停顿成本。
- 再说能力自检重点：GC 影响延迟的核心不是"有没有 GC"，而是分配模式和对象生命周期。
- 再补工程建议：性能优化时要优先关注分配频率和对象存活时间。
:::

### 24. 如何减少内存分配？

:::tabs
== 简答
减少临时对象、避免不必要拷贝、复用对象、控制字符串和切片分配、合理使用对象池。
== 标准回答模板
- 先说结论：减少临时对象、减少拷贝、控制字符串和切片转换、复用对象，通常是最直接的方法。
- 再说原理：分配少了，GC 压力自然会下降。
- 再说能力自检重点：减少分配的目标不是"代码看起来高级"，而是降低 CPU 和 GC 成本。
- 再补工程建议：优化前先测量，避免为了省一点分配把代码复杂度拉高。
:::

### 25. `sync.Pool` 适合什么场景？

:::tabs
== 简答
适合短生命周期、可复用对象的临时缓存，不适合拿来做长期资源池。
== 标准回答模板
- 先说结论：适合短生命周期、频繁分配、可复用对象的临时缓存。
- 再说原理：它的目标是减少分配和 GC 压力，不是做长期对象池。
- 再说边界：对象可能随 GC 被清掉，所以不能依赖它保存状态。
- 再补工程建议：适合高频请求里的 Buffer、临时结构体等复用。
:::

### 26. 如何用 Pprof 排查性能问题？

:::tabs
== 简答
先确认问题类型，再看对应 Profile，例如 CPU、Heap、Goroutine、Mutex、Block，然后结合代码和请求场景验证。
== 标准回答模板
- 先说结论：先确定问题类型，再选对应 Profile。
- 再说方法：CPU 高看 CPU Profile，内存高看 Heap，Goroutine 堆积看 Goroutine，锁竞争看 Mutex，阻塞看 Block。
- 再说排查顺序：先看热点函数，再看调用链，最后回到业务请求和代码验证。
- 再补建议：不要只看采样图就直接优化，必须结合压测或线上现象验证收益。
== 源码线索
- 关键词：CPU Profile、Heap Profile、Goroutine Profile、Mutex Profile、Block Profile
- 重点原理：通过采样定位热点函数、分配热点、阻塞热点
- 延伸：CPU 高、内存高、Goroutine 高，分别该看哪类 Profile；详细排障流程见 [Pprof 排障指南](./pprof-troubleshooting-guide.md)
:::

### 27. 如何排查内存泄漏？

:::tabs
== 简答
关注对象是否被长期持有，通过 Heap Profile、对象引用链和缓存生命周期分析定位问题。
== 标准回答模板
- 先说结论：内存泄漏本质上是对象被长期持有，GC 无法回收。
- 再说排查方法：看 Heap Profile、对象引用链、缓存生命周期和长连接持有关系。
- 再说常见来源：全局缓存无上限、切片或字符串长期持有大对象、协程不退出。
- 再补工程建议：先确认是真的泄漏，还是业务峰值带来的正常内存上涨。
:::

### 28. 如何排查慢请求？

:::tabs
== 简答
结合日志、Trace、Pprof、数据库慢查询、下游调用耗时，逐层定位瓶颈。
== 标准回答模板
- 先说结论：慢请求通常不是单点问题，而是日志、下游、数据库、锁竞争、GC 等因素共同作用的结果。
- 再说排查顺序：先看请求耗时分布，再看 Trace、慢 SQL、Pprof、外部依赖耗时。
- 再说能力自检重点：要体现"逐层缩小范围"的思路，而不是直接猜代码有问题。
- 再补工程建议：没有可观测性，慢请求排查会非常低效。
:::

## 系统设计结合 Go

### 29. `net/http` 使用时要注意什么？

- 先说结论：`net/http` 很强大，但连接复用、超时、Transport 配置这些细节非常关键。
- 再说原理：服务端和客户端都内置了连接管理，尤其是 `http.Client` 和 `Transport` 的复用策略。
- 再说能力自检重点：错误使用 `http.Client`，例如频繁新建客户端、请求体不及时关闭，都会导致连接问题。
- 再补工程建议：客户端要关注超时、连接池、空闲连接复用，服务端要关注中间件、限流和可观测性。

::: warning 注意
千万不要在每次请求时都新建 `http.Client`，也不要忘记关闭 Response Body。这两个是最常见的连接泄漏根源。
:::

### 34. 如何设计一个高并发的 WebSocket 推送服务？

:::tabs
== 简答
核心是 Goroutine-per-connection 模型 + Hub 广播机制。每个连接一个读写 Goroutine，Hub 管理所有连接的注册/注销/广播。用 Channel 传递消息，避免锁竞争。
== 标准回答模板
- 先说架构：每个 WebSocket 连接对应一个 Client 结构体（含 send channel），一个全局 Hub 管理所有 Client
- 再说并发模型：Hub 使用单 Goroutine 的 select 循环处理注册/注销/广播，避免 map 的并发读写
- 再说扩展性：单机 10 万+连接时的优化——epoll 替代 goroutine-per-connection（如 gobwas/ws）、消息批量发送、分组广播
- 再说分布式：多实例间通过 Redis Pub/Sub 或 NATS 同步消息
- 补充：心跳检测、断线重连、消息确认机制
:::

### 35. 如何实现分布式限流器？

:::tabs
== 简答
单机用 golang.org/x/time/rate（令牌桶），分布式用 Redis + Lua 脚本实现滑动窗口或令牌桶。
== 标准回答模板
- 先说单机：`rate.NewLimiter(rate.Every(time.Second/100), 10)` 每秒 100 个请求，突发 10 个
- 再说分布式：Redis EVAL Lua 脚本保证原子性，滑动窗口算法（ZADD + ZRANGEBYSCORE + ZCARD）
- 再说设计要点：per-IP、per-user、per-API 多维度限流、降级策略（限流后返回 429 + Retry-After）
- 再说 Go 实现：中间件模式，从 context 提取用户标识，查询 Redis 获取配额
- 补充：本地缓存 + Redis 的二级限流架构减少 Redis 压力
:::

### 36. Go 服务如何实现优雅关闭？

:::tabs
== 简答
监听 SIGTERM/SIGINT 信号，调用 http.Server.Shutdown() 停止接收新请求，等待进行中请求完成，最后关闭数据库等资源。
== 标准回答模板
- 先说流程：signal.NotifyContext 注册信号 → context 取消 → Shutdown(ctx) 停止监听 → 等待已有请求完成 → 关闭 DB/Redis/MQ 等资源
- 再说超时：Shutdown 需要带超时 context（如 30s），超时后强制关闭
- 再说 Kubernetes 配合：K8s 先发 SIGTERM → 等待 terminationGracePeriodSeconds → 发 SIGKILL。PreStop hook 可做额外清理
- 再说注意事项：关闭顺序很重要——先停 HTTP 入口，再停消费者，最后关 DB
- 补充：长连接（WebSocket、gRPC streaming）的特殊处理
:::

### 37. 如何设计 Go 服务的缓存策略？

:::tabs
== 简答
本地缓存（sync.Map 或 bigcache）+ 分布式缓存（Redis），Cache-Aside 模式，配合 singleflight 防击穿。
== 标准回答模板
- 先说模式：Cache-Aside（旁路缓存）最常用——先查缓存，miss 则查 DB 并回填
- 再说防护三件套：穿透（布隆过滤器或空值缓存）、击穿（singleflight 合并请求）、雪崩（随机过期时间）
- 再说一致性：写操作先更新 DB 再删除缓存（延迟双删）、或通过 binlog 异步更新缓存
- 再说 Go 实现：`golang.org/x/sync/singleflight` 合并并发查询、`github.com/dgraph-io/ristretto` 或 `bigcache` 做本地缓存
- 补充：多级缓存架构（L1 本地 + L2 Redis）减少网络 RT
:::

### 38. 如何在 Go 中实现可观测性（Observability）？

:::tabs
== 简答
三大支柱：Metrics（Prometheus）、Logging（slog/zap）、Tracing（OpenTelemetry）。通过中间件统一注入。
== 标准回答模板
- 先说 Metrics：Prometheus client_golang 暴露 /metrics 端点，核心指标——QPS、延迟分布（Histogram）、错误率、Goroutine 数、GC 暂停时间
- 再说 Logging：结构化日志（slog 或 zap），包含 trace_id、request_id 便于关联
- 再说 Tracing：OpenTelemetry SDK 自动注入 HTTP/gRPC 中间件，上报到 Jaeger/Tempo
- 再说告警：基于 Prometheus Alertmanager，规则举例——P99 延迟 > 500ms、错误率 > 1%
- 补充：Go runtime metrics（runtime/metrics 包）、Continuous Profiling（pyroscope）
:::

### 39. 如何设计一个高性能的批量任务处理系统？

:::tabs
== 简答
Worker Pool + Channel 分发 + errgroup 错误收集。任务入队（Redis/Kafka）→ 消费者拉取 → Worker 池并发处理 → 结果回写。
== 标准回答模板
- 先说架构：Producer-Consumer 模型，任务持久化到消息队列（Kafka/Redis Stream），消费者按 partition 并发消费
- 再说 Go 实现：固定大小 Worker Pool（N 个 goroutine 监听 task channel），errgroup 控制并发度和错误传播
- 再说可靠性：at-least-once 语义（消费成功后 ack）、幂等处理、死信队列
- 再说性能：批量拉取（batch fetch）减少网络 RT、本地聚合后批量写入 DB
- 补充：优雅关闭——收到 SIGTERM 后停止拉取新任务，等待进行中任务完成
:::
