## 1. 栈与堆：本质是在争 GC 成本

Go 编译器在编译期通过逃逸分析（Escape Analysis）决定变量分配在栈上还是堆上。栈分配成本极低，函数返回时自动释放；堆分配则需要 GC 参与回收。

- **栈（Stack）**：分配和回收都非常快，几乎没有 GC 压力
- **堆（Heap）**：生命周期更灵活，但会增加 GC 开销

编译器关心的核心问题只有一个：这个变量在当前函数返回之后，还可能被别的地方继续引用吗？

如果答案是“会”，它就逃逸到堆上；如果答案是“不会”，就尽量留在栈上。

## 2. 常见逃逸场景

最常见的几类场景：

- 返回局部变量指针
- 变量被发送到 Channel
- 变量赋值给接口类型（`interface{}` / `any`）
- 闭包引用外部变量
- Slice / Map 大小在编译期不可知或过大
- `fmt.Println` 等接受接口参数的函数调用

### 返回局部变量指针

局部变量本来属于当前函数栈帧，但如果把它的地址返回出去，那么函数结束后它还必须继续存活，编译器只能把它放到堆上。

### 接口装箱

很多人第一次看逃逸分析结果时会困惑：为什么只是 `fmt.Println(x)`，一个简单值也会“逃逸”？

原因在于 `fmt.Println` 的签名是：

```go
func Println(a ...any) (n int, err error)
```

也就是说，参数要先装箱成接口值。接口值会显著增加逃逸概率，尤其是在复杂调用链和泛化 API 中。

### 闭包捕获外部变量

闭包如果引用了外层函数的局部变量，而闭包本身又可能在外层函数返回后继续存在，那么这些变量就不能继续留在原来的栈帧里，只能转移到堆上。

### Slice / Map 大小不可知或过大

例如：

```go
func makeSlice() {
	s1 := make([]int, 10)     // 小且编译期可知，可能留在栈上
	n := 10
	s2 := make([]int, n)      // 更容易逃逸
	s3 := make([]int, 100000) // 太大，通常直接放堆上

	_, _, _ = s1, s2, s3
}
```

这里要注意：

- “编译期不可知”不代表 100% 一定逃逸，但会显著增加逃逸概率
- “太大”时即使生命周期不长，也可能因为栈空间成本不划算而被分配到堆上

## 3. 代码示例与逃逸结果

```go
package main

import "fmt"

// 返回指针 -> 逃逸到堆
func newInt(n int) *int {
	v := n
	return &v
}

// 不返回指针 -> 更容易留在栈上
func addOne(n int) int {
	v := n + 1
	return v
}

// 接口参数增加逃逸概率
func printValue(v interface{}) {
	fmt.Println(v)
}

// 闭包捕获导致逃逸
func closure() func() int {
	x := 0
	return func() int {
		x++
		return x
	}
}

func main() {
	p := newInt(42)
	fmt.Println(*p)

	r := addOne(10)
	_ = r

	printValue(100)

	f := closure()
	fmt.Println(f())
}
```

用下面命令查看：

```bash
go build -gcflags="-m -m" main.go
```

输出里常见这些关键词：

- `moved to heap`
- `escapes to heap`
- `does not escape`

它们分别表示变量被放到堆上、值沿调用链分析后最终逃逸、或者编译器确认它可以安全留在栈上。

## 4. 工程里怎么排查逃逸

### 先看编译器输出

最常用的是：

```bash
go build -gcflags="-m" ./...
```

如果要看更详细的原因链：

```bash
go build -gcflags="-m -m" ./...
```

单文件、小实验场景下也可以直接：

```bash
go tool compile -m main.go
```

### 只在热点路径上较真

不是所有 `escapes to heap` 都值得优化。更合理的排查顺序是：

1. 先用 pprof 或 benchmark 找到分配热点
2. 再结合 `-gcflags="-m -m"` 看是不是意外逃逸
3. 最后再决定是否为了减少几个堆对象去改代码结构

### 和 benchmark 一起看

```bash
go test -bench=. -benchmem
```

重点关注：

- `B/op`
- `allocs/op`

如果你改掉了一处逃逸，通常会直接反映在这两个指标上。

## 5. 优化时的取舍

- 逃逸分析是编译期行为，不影响运行时语义，只影响分配位置和 GC 压力
- `-gcflags="-m"` 看简要结果，`-gcflags="-m -m"` 看详细原因链
- 减少逃逸的常用手段：避免不必要的指针返回、用具体类型代替接口、预分配 Slice 容量、避免大结构体在逃逸路径中被频繁引用
- `sync.Pool` 的目的之一就是复用堆对象，减少 GC 压力

真正该优化的，通常是高频路径、低延迟路径、GC 压力明显的地方。很多时候，代码清晰比消灭一两个堆对象更重要。
