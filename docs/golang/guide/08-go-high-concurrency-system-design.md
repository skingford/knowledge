## 先纠正一个误区 {#mindset}

很多人一提到“Go 高并发系统设计”，第一反应是：

- Go 协程很轻量
- 一个请求一个 goroutine
- Redis 顶一层
- MQ 削峰
- 数据库分库分表

这些点都可能是对的，但它们不是“设计方法”，只是工具清单。

高并发系统设计真正要解决的是四个问题：

1. 流量进来时系统会不会被打垮
2. 系统没垮时结果会不会做错
3. 系统出故障时能不能止血和恢复
4. 峰值过去后成本是不是还能接受

所以高并发设计不等于“堆组件”，而是要同时考虑：

- **吞吐**：单位时间能处理多少请求
- **延迟**：P50、P95、P99 是否可接受
- **正确性**：会不会重复处理、超卖、脏写、乱序
- **稳定性**：下游超时、缓存失效、流量突刺、消息堆积时是否还能活
- **成本**：扩容方式、机器数、缓存内存、带宽、存储成本是否合理

## 第一步先审题不要一上来就说 Redis + MQ {#problem-framing}

高并发系统设计最常见的错误，是还没搞清业务约束，就开始说解决方案。

先问业务：

| 维度 | 必问问题 | 为什么重要 |
| --- | --- | --- |
| 流量规模 | 平均 QPS、峰值 QPS、峰值持续多久 | 决定是否要限流、排队、异步化 |
| 流量形态 | 平峰增长还是活动秒级突刺 | 决定网关、缓存、队列和自动扩容策略 |
| 读写比例 | 读多写少、读写均衡、重写场景 | 决定缓存收益和数据库瓶颈位置 |
| 数据正确性 | 是否允许最终一致、是否资金相关 | 决定能不能异步、能不能先成功后补偿 |
| 用户体验 | 能否排队、是否允许稍后到账、是否允许降级 | 决定同步链路和异步链路边界 |
| 数据量级 | 当前数据量、增长速度、冷热分布 | 决定分库分表、冷热分层和归档 |
| 下游依赖 | 有多少 RPC、DB、Redis、MQ、三方接口 | 决定最大瓶颈和级联风险 |

### 先把系统分型

不同系统的高并发重点完全不同：

| 系统类型 | 重点矛盾 | 常见手段 |
| --- | --- | --- |
| 热点读取型 | 缓存穿透、热点 Key、回源风暴 | 本地缓存、Redis、singleflight、预热 |
| 重写入型 | DB 写放大、锁竞争、索引压力 | 批量化、异步化、分区、分片 |
| 强一致交易型 | 重复执行、并发冲突、超时重试 | 幂等、唯一索引、状态机、补偿 |
| 实时推送型 | 长连接、广播风暴、连接治理 | 连接分层、Hub、分区广播、背压 |
| 任务消费型 | 消息堆积、重试雪崩、消费不均 | 分区、重试队列、死信、并发控制 |

### 再明确同步链路和异步链路

核心问题不是“要不要 MQ”，而是：

- 哪些动作必须同步返回
- 哪些动作可以延后处理
- 哪些动作失败后允许补偿
- 哪些动作必须在一个事务边界里完成

一句话判断：

> 用户感知强、结果必须立刻确定的动作放同步链路；可延迟、可重试、可补偿的动作放异步链路。

## 第二步做容量估算 {#capacity-planning}

高并发设计如果没有容量估算，后面的很多判断都不稳。

### 几个最基础的估算公式

#### 1. 并发数近似

```text
并发数 ≈ QPS × 平均响应时间（秒）
```

例如：

- 峰值 QPS = 8000
- 平均 RT = 120ms = 0.12s

那么系统同时在处理的请求数量大约是：

```text
8000 × 0.12 = 960
```

这 960 不是 goroutine 上限，但它会直接影响：

- 应用层并发执行数
- 数据库连接数
- Redis 连接数
- 下游 RPC 并发数
- 内存占用和排队时延

#### 2. 带宽估算

```text
带宽 ≈ QPS × 单次请求响应大小
```

如果 10000 QPS，每次响应 20KB：

```text
10000 × 20KB = 200000KB/s ≈ 195MB/s
```

这时网卡、网关、序列化和压缩策略都会成为瓶颈。

#### 3. 数据库写入估算

```text
写 QPS × 每次请求触发的 SQL 条数 = 总 SQL TPS
```

如果每次下单会：

- 写订单表 1 次
- 写明细表 2 次
- 写幂等表 1 次
- 更新库存 1 次

那一次请求可能就是 5 次写操作。5000 QPS 的下单峰值，数据库写压并不是 5000 TPS，而是 25000 TPS。

### 容量估算至少要回答 6 个问题

1. 峰值流量是多少，持续多久
2. 峰值时每层组件的 QPS / TPS / 连接数是多少
3. 是否有热点流量集中打到少量 Key / 行 / 分区
4. 单机极限是多少，水平扩容阈值是多少
5. 限流阈值、降级阈值、告警阈值分别是多少
6. 需要预留多少余量

工程上一般不要按 100% 利用率设计。常见做法是：

- CPU 平稳控制在 40% 到 60%
- 连接池高峰不长期贴满
- 队列积压有可吸收余量
- Redis / DB 内存和磁盘留足增长空间

## 第三步画出核心链路 {#critical-path}

系统设计说不清楚，很多时候不是不会，而是链路没画出来。

先把“读路径”和“写路径”分开：

<GoHighConcurrencySystemDesignDiagram kind="core-path" />

这张图真正要回答的是：

- 峰值流量先打到哪一层
- 哪一层最先扛不住
- 哪一层可以做短路、降级、排队
- 哪些请求必须命中数据库
- 哪些写动作必须同步完成

### 先找瓶颈，不要平均用力

高并发场景的瓶颈通常不是“所有层都差不多”，而是集中在几个点：

- 网关层：限流和鉴权过重
- Go 服务层：共享锁、Goroutine 暴涨、GC 抖动
- Redis：热点 Key、Big Key、回源风暴
- DB：热点行更新、长事务、索引维护开销
- MQ：堆积、消费组不均衡、重试风暴
- RPC：下游 RT 抖动导致线程和连接被拖住

设计时要先判断瓶颈最可能出现在哪里，再决定优化顺序。

## 第四步按层拆系统 {#layered-architecture}

高并发系统通常至少要拆成 4 层来看：

| 层次 | 关注点 | 常见问题 | 常见手段 |
| --- | --- | --- | --- |
| 接入层 | 扛流量、挡垃圾请求 | 洪峰、爬虫、防刷 | CDN、LB、网关限流、验证码 |
| 服务层 | 快速处理、少阻塞 | Goroutine 暴涨、锁竞争、下游拖慢 | 无状态扩容、背压、超时、隔离 |
| 数据层 | 低延迟与正确性 | 热点 Key、热点行、写放大 | 缓存、分片、异步写、唯一索引 |
| 治理层 | 稳定性和恢复 | 重试雪崩、级联故障、不可观测 | 熔断、监控、告警、演练、压测 |

### 1. 接入层

目标不是“转发请求”，而是尽早把不该进来的流量挡在外面。

接入层常见动作：

- 静态资源 CDN 化
- WAF / 黑白名单 / 防爬
- 网关鉴权、验签、参数校验
- 接口级和用户级限流
- 熔断和快速失败

如果接入层做得太薄，所有垃圾流量都会穿透到应用层，最后不是应用层的问题，而是整个系统设计过于“后置防御”。

### 2. 服务层

Go 服务层重点解决三个问题：

1. 单机效率够不够高
2. 并发控制是否可控
3. 下游慢时本层会不会一起卡死

服务层设计原则：

- 业务服务尽量无状态，便于水平扩容
- 热点逻辑和普通逻辑隔离，避免相互拖累
- 对外部依赖一定要设超时、并发上限和熔断
- 对本地资源一定要设队列长度和拒绝策略

### 3. 数据层

数据层不是只看数据库。

它至少包括：

- 本地缓存
- Redis / Memcached
- MySQL / PostgreSQL
- 对象存储
- MQ / Stream

高并发场景的数据层原则：

- 读路径优先减少对主库的直接命中
- 写路径优先减少热点争抢和同步链路长度
- 正确性由数据库约束和状态机兜底
- 缓存、锁、消息队列都只负责提升性能和抗峰值，不负责最终正确性

### 4. 治理层

治理层常被低估，但真正决定系统上限的往往不是功能，而是治理。

治理层至少要有：

- 指标：QPS、RT、错误率、队列积压、连接池使用率
- 日志：错误日志、慢日志、关键状态变更日志
- Trace：跨服务耗时和错误传播
- 告警：阈值、趋势、同比、环比
- 演练：限流、熔断、降级、机房切换、消息堆积

## 第五步用 Go 写服务时最容易忽略的点 {#go-service-pitfalls}

很多架构图看起来没问题，但一落到 Go 代码就开始失控。根因通常是把“系统并发”和“代码并发”混为一谈。

### 1. Goroutine 很轻，但不是免费

Go 的 goroutine 成本比线程低很多，但以下问题仍然存在：

- 每个 goroutine 都有栈和调度开销
- 阻塞在 IO、锁、channel、syscall 上的 goroutine 会堆积
- 大量 goroutine 会增加 GC root 扫描压力
- 如果入口没有限流，goroutine 数量会和流量一起失控

错误示例不是“用了 goroutine”，而是“来多少请求就放多少 goroutine 并且没有上限”。

### 2. 一定要做有界并发

对 CPU 密集、IO 风险高、下游脆弱的动作，都应该限制并发。

一个常见写法：

```go
package main

import (
	"context"
	"net/http"

	"golang.org/x/sync/semaphore"
)

var reqSem = semaphore.NewWeighted(200)

func handler(w http.ResponseWriter, r *http.Request) {
	ctx := r.Context()

	if err := reqSem.Acquire(ctx, 1); err != nil {
		http.Error(w, "server busy", http.StatusServiceUnavailable)
		return
	}
	defer reqSem.Release(1)

	// 真正的业务逻辑
	w.WriteHeader(http.StatusOK)
	_, _ = w.Write([]byte("ok"))
}
```

这类限制的价值是：

- 防止单机被无限压榨
- 给排队、降级、快速失败留出口
- 让系统在高峰时“变慢但不崩”

### 3. Context 不是装饰品

在高并发服务里，`context.Context` 至少承担三件事：

- 取消传播
- 超时控制
- 链路级元数据透传

如果你不把 `ctx` 贯穿到：

- HTTP / gRPC 请求
- Redis / SQL / MQ 客户端
- 下游 RPC

那么上游超时了，下游动作可能还在继续做，结果就是：

- 无效工作持续消耗资源
- 连接池被慢请求占满
- 用户已经放弃，系统还在忙

### 4. 连接池比 Goroutine 更早打满

Go 服务高并发下一个特别常见的误区是：

> goroutine 很多没事，Go 能扛。

但真正先打满的，往往是：

- 数据库连接池
- Redis 连接池
- HTTP 客户端连接池
- 下游线程池或连接上限

例如数据库连接池，如果 `SetMaxOpenConns(100)`，你的应用层即使放出 2000 个 goroutine 去打数据库，最后也只是 1900 个在排队。

这会造成：

- RT 拉长
- context 超时
- 重试叠加
- 连接池被慢 SQL 长时间占住

所以服务层并发上限一定要和连接池能力一起设计。

### 5. 共享锁要尽量缩小粒度

Go 服务层常见瓶颈不是 CPU 算力，而是锁竞争。

高频场景：

- 全局 map + mutex
- 大对象缓存 + 统一锁
- 统计计数器集中更新
- 广播 Hub 全局锁

优化方向：

- 分片锁
- 读写锁只用于读多写少场景
- 原子计数器替代短整型互斥
- 热点状态拆成多桶
- 避免在锁内做 IO 或复杂逻辑

### 6. 不要把所有工作都放同步链路

同步链路适合做：

- 鉴权
- 参数校验
- 幂等检查
- 关键状态落库
- 必须立刻返回的结果

不适合同步做：

- 批量通知
- 大量日志聚合
- 次要报表更新
- 非关键画像刷新
- 可补偿任务派发

一句话：

> 同步链路做“必须立刻确定”的事，异步链路做“可以稍后完成”的事。

## 第六步缓存设计 {#cache-design}

高并发系统几乎绕不开缓存，但缓存设计的重点不是“加 Redis”，而是要回答：

- 缓存什么
- 为什么缓存
- 失效时怎么办
- 错误数据怎么办

### 1. 优先判断缓存是否真的有效

缓存特别适合：

- 读多写少
- 热点集中
- 对极短暂不一致可接受
- 重建成本高但可接受

缓存不太适合直接扛：

- 强一致资金余额
- 高频写、低复用数据
- 随机写随机读且热点不明显的数据

### 2. 常见缓存分层

| 层级 | 优点 | 风险 | 适合场景 |
| --- | --- | --- | --- |
| 本地缓存 | 延迟最低，不走网络 | 容量小，实例间不一致 | 热点配置、字典、小对象 |
| Redis | 命中快，易共享 | 热点 Key、网络延迟、回源风暴 | 共享缓存、计数、锁 |
| 数据库 | 最终正确性来源 | 延迟高，扛峰值差 | 兜底查询、事务写入 |

生产里很常见的组合是：

```text
本地缓存 -> Redis -> 数据库
```

### 3. 缓存读路径的标准动作

1. 先读本地缓存
2. 本地没有，再读 Redis
3. Redis 没有，再查数据库
4. 回填 Redis
5. 如果是热点配置，再异步刷新本地缓存

### 4. 高并发缓存四大问题

#### 缓存穿透

查询不存在的数据，导致请求持续打到 DB。

手段：

- 参数校验
- 空值缓存
- 布隆过滤器

#### 缓存击穿

热点 Key 过期瞬间，大量请求同时回源。

手段：

- 热点 Key 永不过期或逻辑过期
- `singleflight`
- 互斥重建

#### 缓存雪崩

大量 Key 同时失效或 Redis 整体异常。

手段：

- TTL 加随机值
- 多级缓存
- 限流与降级
- 高可用部署

#### 热点 Key

少量 Key 被极高频访问，单节点 Redis 被打热。

手段：

- 本地缓存兜热点
- 请求合并
- 热点拆桶
- 异步刷新

### 5. Go 服务里用 singleflight 挡回源风暴

```go
package main

import (
	"context"
	"encoding/json"
	"time"

	"golang.org/x/sync/singleflight"
)

var group singleflight.Group

func GetConfig(ctx context.Context, key string) ([]byte, error) {
	v, err, _ := group.Do(key, func() (any, error) {
		// 先查 DB 或下游
		cfg := map[string]any{
			"feature": true,
			"expire":  time.Now().Unix(),
		}
		return json.Marshal(cfg)
	})
	if err != nil {
		return nil, err
	}
	return v.([]byte), nil
}
```

它的作用不是“提升吞吐”，而是：

- 避免同一个热点 Key 在失效瞬间被同时重建
- 避免数据库被同一份数据查询打爆

## 第七步异步削峰与消息队列 {#async-and-mq}

高并发系统常见一句话是“上 MQ 削峰”。这句话本身没错，但前提是你知道削什么峰、削到哪去、失败怎么兜底。

### 什么时候适合异步化

适合异步的动作：

- 短信、邮件、站内信
- 报表、统计、埋点聚合
- 搜索索引刷新
- 非关键下游通知
- 后置对账、补偿、清理任务

不适合轻易异步的动作：

- 用户必须同步看到结果的关键落库
- 没有幂等机制的资金扣减
- 强一致库存确认

### 异步化设计要解决 5 个问题

1. 消息有没有丢
2. 消息会不会重复
3. 消息顺序是否重要
4. 消费积压怎么办
5. 消费失败怎么重试和兜底

### 典型设计原则

- 生产端用本地消息表或事务消息，避免“业务成功但消息没发出去”
- 消费端一定做幂等
- 重试要有上限，超过上限进死信
- 对需要顺序的消息，用业务 key 做分区
- 消费并发不要无限拉高，要看下游和数据库能力

### Go Worker 不等于无限开协程

消费者侧的常见模式：

```go
package main

import "context"

type Job struct {
	ID string
}

func StartWorkers(ctx context.Context, jobs <-chan Job, n int, handler func(context.Context, Job) error) {
	for i := 0; i < n; i++ {
		go func() {
			for {
				select {
				case <-ctx.Done():
					return
				case job, ok := <-jobs:
					if !ok {
						return
					}
					_ = handler(ctx, job)
				}
			}
		}()
	}
}
```

关键点不在 `go func()`，而在：

- worker 数量是否受控
- 每个 worker 是否设置超时
- handler 是否幂等
- 失败后是否分类重试

## 第八步数据库与一致性 {#db-and-consistency}

高并发系统设计里，最终正确性通常都落在数据库约束和状态机上，而不是落在缓存或消息队列上。

### 1. 先统一一个原则

> Redis、MQ、限流、分布式锁负责提升吞吐和降低冲突；最终正确性由数据库唯一索引、事务、状态机、乐观锁兜底。

### 2. 常见数据库瓶颈

- 热点行更新
- 索引过多导致写放大
- 长事务持锁
- 批量写不批量
- 分页深翻页
- 事务里混入慢 RPC

### 3. 高并发写路径常见方案

| 方案 | 适合什么场景 | 优点 | 风险 |
| --- | --- | --- | --- |
| 单库单表 | 早期阶段 | 简单、正确性强 | 上限低 |
| 读写分离 | 读多写少 | 降低读压力 | 复制延迟 |
| 分库分表 | 数据量或写入过大 | 水平扩展 | 复杂度高 |
| 异步写入 | 非强同步链路 | 削峰明显 | 一致性更复杂 |
| 批量写入 | 日志、流水、埋点 | TPS 高 | 实时性下降 |

### 4. 幂等是高并发系统的底座

高并发下最怕的不是“慢”，而是“重复做错”。

典型重复来源：

- 用户重复点击
- 网关重试
- 客户端超时重试
- MQ 重复投递
- 消费失败重试
- 回调重复通知

常见幂等手段：

- 请求唯一号
- 数据库唯一索引
- 状态机校验
- 乐观锁版本号
- Redis 短期去重

### 5. 一个库存扣减的安全边界

如果只是靠 Redis 锁防超卖，设计是不完整的。

更稳的模式通常是：

1. 入口限流
2. Redis 做热点削峰和快速拦截
3. 数据库用条件更新或乐观锁做最终扣减
4. 失败后统一补偿或回滚可见状态

例如数据库层可以这样兜底：

```sql
UPDATE inventory
SET stock = stock - 1
WHERE sku_id = ? AND stock > 0;
```

如果影响行数为 0，就说明库存已经不足，不会因为并发而扣成负数。

## 第九步稳定性治理 {#stability}

高并发系统在正常流量下跑得快不算厉害，洪峰、抖动、下游异常时还能活下来才算设计到位。

### 稳定性常用五件套

#### 1. 超时

任何下游调用都必须设超时。

没有超时，就等于默认“永远等”。

#### 2. 重试

重试只适合瞬时失败，且必须有边界：

- 次数上限
- 总时间预算
- 幂等前提

没有预算的重试，会把单次失败放大成全链路雪崩。

#### 3. 限流

限流要分层：

- 网关限流
- 接口限流
- 用户 / 租户 / 商户维度限流
- 下游调用限流

#### 4. 熔断

当下游持续高错误或高延迟时，快速失败比持续等待更稳。

#### 5. 隔离

不同流量、不同依赖、不同业务池化隔离，避免一个问题拖垮全部服务。

隔离对象常见有：

- 线程 / worker 池
- 连接池
- 队列
- 限流桶
- 服务实例

### 常见故障传播链

```text
下游变慢
-> Go 服务请求堆积
-> Goroutine 增多
-> 连接池打满
-> RT 上升
-> 网关超时重试
-> 请求更多
-> 整体雪崩
```

所以真正关键的是：

- 不让慢请求无限堆积
- 不让重试无限叠加
- 不让失败在全链路传播

## 第十步压测观测与容量回归 {#testing-and-observability}

如果没有压测和观测，所谓“高并发设计”很可能只是静态想象。

### 1. 压测不是只看 QPS

至少要看：

- P50 / P95 / P99 RT
- 错误率
- CPU 使用率
- 内存、GC 次数、GC 暂停
- Goroutine 数量
- 数据库连接池等待时间
- Redis RT 和热点情况
- MQ 积压长度

### 2. 观测指标最少要有这些

| 类别 | 核心指标 |
| --- | --- |
| 流量 | QPS、TPS、并发数、入口流量、拒绝数 |
| 延迟 | P50、P95、P99、超时数 |
| Go runtime | goroutines、heap、GC 次数、GC pause、线程数 |
| 连接池 | open、in-use、wait-count、wait-duration |
| 缓存 | 命中率、miss、热点 Key、Big Key |
| 队列 | lag、消费速率、失败率、重试次数 |
| 数据库 | 慢 SQL、锁等待、行更新冲突、复制延迟 |

### 3. 压测要分层

不要只做“整链路压测”，还要做：

- 单接口压测
- 单机压测
- 依赖隔离压测
- 数据库压测
- 缓存命中率变化压测
- 故障注入压测

### 4. 压测之后要回归容量模型

压测不是结束，而是修正估算：

- 单机极限 QPS 是多少
- 多少 QPS 开始 RT 抖动
- 多少 QPS 开始连接池等待
- 多少 QPS 开始 GC 明显变重

最终把这些数据回写到：

- 自动扩容阈值
- 限流阈值
- 告警阈值
- 峰值预案

## 一个完整案例活动报名系统 {#case-study}

下面用“活动报名系统”把前面的原则串起来。

### 业务目标

- 活动开始后 10 秒内峰值 15000 QPS
- 用户希望 1 秒内知道是否报名成功
- 报名名额有限，不能超卖
- 可接受“短信通知稍后发送”

### 一条更稳的链路

<GoHighConcurrencySystemDesignDiagram kind="activity-registration" />

### 设计重点

#### 1. 接入层先限流

- 用户级限流，防止单用户刷接口
- 活动级限流，防止整体峰值超过系统承载
- 高峰无法承接时，明确返回“排队中”或“稍后再试”

#### 2. API 层先做幂等

报名请求必须带请求号或用户 + 活动唯一键。

这样即使：

- 用户重复点按钮
- 客户端超时重试
- 网关重试

最终也只会成功一次。

#### 3. Redis 只做快速拦截，不做最终裁决

Redis 可以保存：

- 活动剩余名额缓存
- 活动配置
- 用户报名短期去重

但最终是否报名成功，仍然要靠数据库条件更新或唯一约束确认。

#### 4. 数据库最终兜底

可以这样做：

- 报名表唯一键：`(activity_id, user_id)`
- 名额表条件更新：`remaining > 0`

结果就是：

- Redis 挡住大量无效请求
- 数据库保证绝不会超卖和重复报名

#### 5. 非关键动作异步化

报名成功后：

- 发短信
- 更新画像
- 写分析日志
- 发站内通知

全部可以异步。

这能显著缩短主链路 RT。

### 如果流量再扩大怎么办

当活动再升级到更高峰值时，常见演进路径是：

1. 本地缓存活动配置
2. Redis 做热点拦截
3. 网关前移限流
4. 数据库按活动或时间分区
5. 报名结果异步汇总
6. 热点活动做独立资源池

这就是高并发系统的常见演进方式：

> 先保证正确，再保证能扛，最后再保证成本和治理复杂度可控。

## 面试回答模板 {#interview-template}

如果面试官问：

> “你怎么设计一个 Go 高并发系统？”

建议按这个结构回答：

1. **先审题**：先确认峰值流量、读写比例、一致性要求、是否允许异步和降级
2. **再做容量估算**：估算 QPS、并发数、数据库写压、缓存命中率和连接池上限
3. **再拆系统层次**：接入层挡流量，服务层做无状态扩容和并发控制，数据层做缓存和数据库兜底，治理层做监控告警和熔断
4. **再讲 Go 侧重点**：控制 goroutine 数量、用 context 做超时取消、连接池和 worker 池做有界并发、避免共享锁热点
5. **再讲一致性**：Redis 和 MQ 负责抗峰值，数据库唯一索引、条件更新、状态机负责最终正确性
6. **最后讲稳定性**：限流、熔断、降级、重试预算、压测和容量回归

如果时间更短，可以压缩成一句：

> 我会先按业务约束和容量模型审题，再按接入层、服务层、数据层和治理层拆系统；Go 侧重点是有界并发、连接池治理、超时取消和锁竞争控制；最终一致性靠数据库约束兜底，稳定性靠限流熔断降级和压测回归。

## 常见错误清单

- 只会说 Redis + MQ，不会算容量
- 一味追求异步，却没讲清同步边界
- 只会说分布式锁，不会说数据库最终兜底
- 只会开 goroutine，不做有界并发
- 只会做接口限流，不做下游限流和重试预算
- 只会看平均 RT，不看 P99 和队列积压
- 只会压业务接口，不压数据库、缓存和消息链路
- 只会扩容应用，不处理热点 Key、热点行和锁竞争

## 上线前检查清单 {#launch-checklist}

- 是否完成了峰值流量和并发数估算
- 是否识别出最可能先出问题的瓶颈层
- 是否定义了同步链路和异步链路边界
- 是否为 Go 服务设置了超时、限流、并发上限和连接池边界
- 是否为数据库写路径设计了幂等和最终一致性兜底
- 是否为缓存设计了穿透、击穿、雪崩和热点 Key 治理方案
- 是否为 MQ 设计了幂等、重试上限和死信处理
- 是否有 P99、错误率、连接池等待、消息积压等关键指标
- 是否完成了压测，并把结果回写到阈值和预案
- 是否准备了降级、回滚、扩容和故障演练方案
