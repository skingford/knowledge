## 1. go generate 基础

`go generate` 是 Go 工具链内置的代码生成入口。它扫描源文件中的 `//go:generate` 注释指令，按顺序执行指定的命令。关键要理解：**`go generate` 在开发时手动运行，而非编译时自动触发**——它只是一个便捷的命令执行器，让代码生成步骤可以被记录在源码中、被版本控制追踪。

```go
package main

// 演示 go generate 的基本用法
// 在实际项目中，以下指令写在需要生成代码的源文件顶部

// 示例指令（实际执行时会运行对应命令）：
//go:generate echo "你好，这是 go generate 执行的命令"
//go:generate go run gen_version.go -output version_gen.go

import "fmt"

// 约定：生成的文件以 _gen.go 或 _string.go 结尾
// 这样在 code review 时一目了然哪些是生成代码

// Version 由 go generate 生成并写入 version_gen.go
// 这里仅做演示
var version = "v0.0.0-dev"

func main() {
	fmt.Println("=== go generate 基础演示 ===")
	fmt.Println()

	// 展示 go generate 的工作流程
	fmt.Println("go generate 工作流程：")
	fmt.Println("1. 开发者在源文件中添加 //go:generate 指令")
	fmt.Println("2. 运行 go generate ./... 扫描并执行所有指令")
	fmt.Println("3. 生成的 .go 文件与源文件一起提交到版本控制")
	fmt.Println("4. go build 正常编译，不会再次执行 generate")
	fmt.Println()

	// 展示可用的环境变量
	// go generate 执行时会设置以下环境变量：
	// $GOPACKAGE - 当前包名
	// $GOFILE    - 当前源文件名
	// $GOLINE    - 指令所在行号
	// $GOARCH    - 目标架构
	// $GOOS      - 目标操作系统
	fmt.Println("go generate 可用的环境变量：")
	fmt.Println("  $GOPACKAGE - 当前包名")
	fmt.Println("  $GOFILE    - 当前源文件名")
	fmt.Println("  $GOLINE    - 指令所在行号")
	fmt.Println()

	fmt.Println("当前版本:", version)
}
```

### 讲解重点

- **实际项目中的使用场景**：几乎所有 Go 代码生成工具（stringer、mockgen、protoc-gen-go、ent、sqlc 等）都通过 `//go:generate` 指令集成到项目中。团队成员 clone 项目后只需执行 `go generate ./...` 就能重新生成所有代码。
- **与其他语言的对比**：Java 有 Annotation Processor（编译期自动触发），Rust 有 proc_macro（编译期展开），而 Go 选择了最简单直接的方式——手动运行命令。这体现了 Go "显式优于隐式" 的设计哲学。
- **常见陷阱和最佳实践**：
  - `//go:generate` 与注释之间**不能有空格**，`// go:generate` 是普通注释而非指令。
  - 生成的文件开头应加注释 `// Code generated by xxx; DO NOT EDIT.`，这是 Go 官方约定，`go generate` 工具链和 IDE 会识别。
  - 将生成文件提交到版本控制（而非 `.gitignore`），确保 `go build` 无需依赖生成工具。
  - Makefile 中建议加 `generate` target，CI 中可对比生成结果是否已提交：`go generate ./... && git diff --exit-code`。

## 2. stringer 工具

`stringer` 是 Go 官方提供的代码生成工具，为整型常量枚举自动生成 `String()` 方法。在没有 stringer 之前，每新增一个枚举值都需要手动更新 `String()` 方法，极易遗漏；stringer 彻底消除了这类人为错误。

```go
package main

import "fmt"

// ========================================
// 步骤 1：定义枚举类型（写在 color.go 中）
// ========================================

// Color 表示颜色枚举
type Color int

const (
	Red    Color = iota // 红色
	Green               // 绿色
	Blue                // 蓝色
	Yellow              // 黄色
)

// 在 color.go 文件顶部添加以下指令：
//go:generate stringer -type=Color

// ========================================
// 步骤 2：运行 go generate 后会生成 color_string.go
// 内容大致如下（简化版）：
// ========================================

// --- 以下是 stringer 生成代码的原理演示 ---
// func _() {
//     // 编译期检查：确保枚举值没有改变
//     var x [1]struct{}
//     _ = x[Red-0]
//     _ = x[Green-1]
//     _ = x[Blue-2]
//     _ = x[Yellow-3]
// }
//
// const _Color_name = "RedGreenBlueYellow"
// var _Color_index = [...]uint8{0, 3, 8, 12, 18}
//
// func (i Color) String() string {
//     if i < 0 || i >= Color(len(_Color_index)-1) {
//         return "Color(" + strconv.FormatInt(int64(i), 10) + ")"
//     }
//     return _Color_name[_Color_index[i]:_Color_index[i+1]]
// }

// 为了演示，我们手写一个等效的 String() 方法
func (c Color) String() string {
	names := [...]string{"Red", "Green", "Blue", "Yellow"}
	if c < Red || int(c) >= len(names) {
		return fmt.Sprintf("Color(%d)", c)
	}
	return names[c]
}

// OrderStatus 展示另一个常见枚举
type OrderStatus int

const (
	OrderPending    OrderStatus = iota // 待处理
	OrderConfirmed                     // 已确认
	OrderShipped                       // 已发货
	OrderDelivered                     // 已送达
	OrderCancelled                     // 已取消
)

//go:generate stringer -type=OrderStatus

// 手写等效方法用于演示
func (s OrderStatus) String() string {
	names := [...]string{"OrderPending", "OrderConfirmed", "OrderShipped", "OrderDelivered", "OrderCancelled"}
	if s < OrderPending || int(s) >= len(names) {
		return fmt.Sprintf("OrderStatus(%d)", s)
	}
	return names[s]
}

func main() {
	fmt.Println("=== stringer 工具演示 ===")
	fmt.Println()

	// 使用生成的 String() 方法
	colors := []Color{Red, Green, Blue, Yellow}
	for _, c := range colors {
		fmt.Printf("Color(%d) = %s\n", int(c), c)
	}
	fmt.Println()

	// 处理越界值
	fmt.Printf("越界值: %s\n", Color(99))
	fmt.Println()

	// 订单状态
	statuses := []OrderStatus{OrderPending, OrderConfirmed, OrderShipped, OrderDelivered, OrderCancelled}
	for _, s := range statuses {
		fmt.Printf("OrderStatus(%d) = %s\n", int(s), s)
	}
	fmt.Println()

	// stringer 的实际价值：在日志和调试中自动展示可读名称
	currentStatus := OrderShipped
	fmt.Printf("当前订单状态: %s\n", currentStatus)
	// 输出: 当前订单状态: OrderShipped
	// 而非: 当前订单状态: 2
}
```

### 讲解重点

- **实际项目中的使用场景**：任何使用 iota 定义枚举的地方都应该用 stringer——HTTP 状态码、数据库状态字段、配置选项等。生成的 `String()` 方法让日志输出、JSON 序列化、调试信息都变得可读。
- **与其他语言的对比**：Java 的 enum 天生有 `name()` 和 `toString()`，Rust 可以 `#[derive(Debug)]`，而 Go 需要显式生成。stringer 是 Go 工具链哲学的体现：用简单工具解决简单问题。
- **常见陷阱和最佳实践**：
  - 新增枚举值后必须重新运行 `go generate`，否则新值会显示为 `Color(5)` 这样的数字。
  - stringer 生成的代码使用字符串拼接 + 索引表，性能极高（O(1)），无需担心性能问题。
  - 安装命令：`go install golang.org/x/tools/cmd/stringer@latest`。
  - 可以用 `-linecomment` 标志让生成的 String() 返回行尾注释内容，适合需要中文名称的场景。

## 3. mockgen / mockery

接口 mock 是 Go 单元测试的核心技能。`mockgen`（Go 官方维护）和 `mockery`（社区维护）是两个主流工具，它们为接口自动生成 mock 实现，避免手写大量样板代码。

```go
package main

import (
	"context"
	"errors"
	"fmt"
	"time"
)

// ========================================
// 步骤 1：定义业务接口（repository.go）
// ========================================

// User 用户模型
type User struct {
	ID        int64
	Name      string
	Email     string
	CreatedAt time.Time
}

// UserRepository 用户仓储接口
// mockgen 会为这个接口生成 mock 实现
type UserRepository interface {
	GetByID(ctx context.Context, id int64) (*User, error)
	Create(ctx context.Context, user *User) error
	Update(ctx context.Context, user *User) error
	Delete(ctx context.Context, id int64) error
	ListByName(ctx context.Context, name string, offset, limit int) ([]*User, error)
}

// 在接口文件顶部添加以下指令：
//go:generate mockgen -source=repository.go -destination=mock_repository.go -package=main

// 或使用 reflect 模式（推荐，不依赖源文件路径）：
//go:generate mockgen -destination=mock_repository.go -package=main . UserRepository

// ========================================
// 步骤 2：业务逻辑层（依赖接口，不依赖具体实现）
// ========================================

// UserService 用户服务
type UserService struct {
	repo UserRepository
}

// NewUserService 创建用户服务
func NewUserService(repo UserRepository) *UserService {
	return &UserService{repo: repo}
}

// GetUser 获取用户（带业务逻辑）
func (s *UserService) GetUser(ctx context.Context, id int64) (*User, error) {
	if id <= 0 {
		return nil, errors.New("无效的用户 ID")
	}
	user, err := s.repo.GetByID(ctx, id)
	if err != nil {
		return nil, fmt.Errorf("获取用户失败: %w", err)
	}
	return user, nil
}

// CreateUser 创建用户（带校验）
func (s *UserService) CreateUser(ctx context.Context, name, email string) (*User, error) {
	if name == "" {
		return nil, errors.New("用户名不能为空")
	}
	if email == "" {
		return nil, errors.New("邮箱不能为空")
	}
	user := &User{
		Name:      name,
		Email:     email,
		CreatedAt: time.Now(),
	}
	if err := s.repo.Create(ctx, user); err != nil {
		return nil, fmt.Errorf("创建用户失败: %w", err)
	}
	return user, nil
}

// ========================================
// 步骤 3：使用 mock 编写测试（user_service_test.go）
// 以下展示 gomock 的使用方式（伪代码风格，完整代码需要生成的 mock）
// ========================================

// --- 测试代码示例（需配合 mockgen 生成的 mock_repository.go）---
//
// func TestUserService_GetUser(t *testing.T) {
//     ctrl := gomock.NewController(t)
//     defer ctrl.Finish()
//
//     // 创建 mock 对象
//     mockRepo := NewMockUserRepository(ctrl)
//
//     // 设置期望：当调用 GetByID(ctx, 1) 时返回指定用户
//     mockRepo.EXPECT().
//         GetByID(gomock.Any(), int64(1)).
//         Return(&User{ID: 1, Name: "张三", Email: "zhangsan@example.com"}, nil).
//         Times(1)
//
//     // 执行被测方法
//     svc := NewUserService(mockRepo)
//     user, err := svc.GetUser(context.Background(), 1)
//
//     // 断言结果
//     assert.NoError(t, err)
//     assert.Equal(t, "张三", user.Name)
// }
//
// func TestUserService_GetUser_NotFound(t *testing.T) {
//     ctrl := gomock.NewController(t)
//     defer ctrl.Finish()
//
//     mockRepo := NewMockUserRepository(ctrl)
//
//     // 设置期望：返回错误
//     mockRepo.EXPECT().
//         GetByID(gomock.Any(), int64(999)).
//         Return(nil, errors.New("user not found")).
//         Times(1)
//
//     svc := NewUserService(mockRepo)
//     user, err := svc.GetUser(context.Background(), 999)
//
//     assert.Error(t, err)
//     assert.Nil(t, user)
//     assert.Contains(t, err.Error(), "获取用户失败")
// }
//
// func TestUserService_CreateUser_Validation(t *testing.T) {
//     ctrl := gomock.NewController(t)
//     defer ctrl.Finish()
//
//     mockRepo := NewMockUserRepository(ctrl)
//     // 不设置任何 EXPECT —— 校验失败不应调用 repo
//
//     svc := NewUserService(mockRepo)
//     _, err := svc.CreateUser(context.Background(), "", "test@example.com")
//
//     assert.Error(t, err)
//     assert.Contains(t, err.Error(), "用户名不能为空")
// }

func main() {
	fmt.Println("=== mockgen / mockery 演示 ===")
	fmt.Println()

	// 展示 mockgen 的两种模式
	fmt.Println("mockgen 两种模式：")
	fmt.Println("1. source 模式: mockgen -source=repository.go -destination=mock_repository.go")
	fmt.Println("   - 解析源文件，生成文件中所有接口的 mock")
	fmt.Println()
	fmt.Println("2. reflect 模式: mockgen -destination=mock_repo.go -package=main . UserRepository")
	fmt.Println("   - 使用反射，可以指定具体接口，更灵活")
	fmt.Println()

	// 对比 mockgen 和 mockery
	fmt.Println("mockgen vs mockery 对比：")
	fmt.Println("┌─────────────┬──────────────────────────┬──────────────────────────┐")
	fmt.Println("│ 特性        │ mockgen (官方)            │ mockery (社区)            │")
	fmt.Println("├─────────────┼──────────────────────────┼──────────────────────────┤")
	fmt.Println("│ 维护方       │ Go 官方团队               │ vektra 社区              │")
	fmt.Println("│ 断言库       │ gomock (EXPECT 链式调用)   │ testify/mock             │")
	fmt.Println("│ 配置方式     │ 命令行参数                │ .mockery.yaml 配置文件    │")
	fmt.Println("│ 批量生成     │ 需逐个指定                │ 支持全项目批量扫描         │")
	fmt.Println("│ 类型安全     │ 需手动匹配参数类型         │ v2 支持类型安全期望       │")
	fmt.Println("│ 推荐场景     │ 小型项目/官方工具链偏好    │ 大型项目/testify 用户     │")
	fmt.Println("└─────────────┴──────────────────────────┴──────────────────────────┘")
	fmt.Println()

	// gomock 常用匹配器
	fmt.Println("gomock 常用匹配器：")
	fmt.Println("  gomock.Any()      - 匹配任意值")
	fmt.Println("  gomock.Eq(x)      - 精确匹配")
	fmt.Println("  gomock.Nil()      - 匹配 nil")
	fmt.Println("  gomock.Not(x)     - 不匹配 x")
	fmt.Println("  gomock.InOrder()  - 指定调用顺序")
}
```

### 讲解重点

- **实际项目中的使用场景**：所有依赖外部系统（数据库、缓存、第三方 API）的业务逻辑都应该通过接口抽象，然后用 mock 进行单元测试。这是 Go 项目的标准实践。
- **与其他语言的对比**：Java 有 Mockito（运行时字节码操纵），Python 有 `unittest.mock`（动态语言天然支持），Go 由于静态类型和缺少运行时代理机制，必须通过代码生成实现 mock。
- **常见陷阱和最佳实践**：
  - mock 只用于**单元测试**，集成测试应使用真实依赖（如 testcontainers）。
  - 接口要小（Interface Segregation），一个接口 3-5 个方法为宜，过大的接口 mock 起来痛苦。
  - `gomock.Controller` 的 `Finish()` 会验证所有 EXPECT 是否满足，Go 1.14+ 可使用 `ctrl := gomock.NewController(t)` 自动 cleanup。
  - 安装命令：`go install go.uber.org/mock/mockgen@latest`（注意：已从 `github.com/golang/mock` 迁移到 `go.uber.org/mock`）。

## 4. wire 依赖注入

Google Wire 是编译时依赖注入框架，通过代码生成（而非运行时反射）完成依赖注入。开发者定义 Provider 函数和 Injector 函数签名，Wire 自动分析依赖关系并生成完整的构造代码。

```go
package main

import (
	"context"
	"database/sql"
	"fmt"
	"log"
	"net/http"
)

// ========================================
// 步骤 1：定义各层组件和 Provider 函数
// ========================================

// --- 配置层 ---

// Config 应用配置
type Config struct {
	DatabaseDSN string
	ServerAddr  string
	LogLevel    string
}

// NewConfig 从环境变量或配置文件加载配置
func NewConfig() *Config {
	return &Config{
		DatabaseDSN: "postgres://localhost:5432/myapp",
		ServerAddr:  ":8080",
		LogLevel:    "info",
	}
}

// --- 数据层 ---

// DB 数据库连接（简化示例）
type DB struct {
	conn *sql.DB
	dsn  string
}

// NewDB 创建数据库连接 —— 这是一个 Provider 函数
// Wire 会自动识别：输入 *Config → 输出 *DB
func NewDB(cfg *Config) (*DB, error) {
	// 实际项目中这里会调用 sql.Open
	fmt.Printf("连接数据库: %s\n", cfg.DatabaseDSN)
	return &DB{dsn: cfg.DatabaseDSN}, nil
}

// --- Repository 层 ---

// UserRepo 用户仓储
type UserRepo struct {
	db *DB
}

// NewUserRepo Provider 函数
func NewUserRepo(db *DB) *UserRepo {
	return &UserRepo{db: db}
}

// FindByID 查找用户
func (r *UserRepo) FindByID(ctx context.Context, id int64) (string, error) {
	return fmt.Sprintf("User-%d", id), nil
}

// OrderRepo 订单仓储
type OrderRepo struct {
	db *DB
}

// NewOrderRepo Provider 函数
func NewOrderRepo(db *DB) *OrderRepo {
	return &OrderRepo{db: db}
}

// --- Service 层 ---

// UserService 用户服务
type UserServiceWire struct {
	repo *UserRepo
}

// NewUserServiceWire Provider 函数
func NewUserServiceWire(repo *UserRepo) *UserServiceWire {
	return &UserServiceWire{repo: repo}
}

// OrderService 订单服务
type OrderService struct {
	repo     *OrderRepo
	userRepo *UserRepo
}

// NewOrderService Provider 函数
func NewOrderService(orderRepo *OrderRepo, userRepo *UserRepo) *OrderService {
	return &OrderService{repo: orderRepo, userRepo: userRepo}
}

// --- Handler 层 ---

// Server HTTP 服务器
type ServerApp struct {
	cfg      *Config
	userSvc  *UserServiceWire
	orderSvc *OrderService
	mux      *http.ServeMux
}

// NewServerApp Provider 函数
func NewServerApp(cfg *Config, userSvc *UserServiceWire, orderSvc *OrderService) *ServerApp {
	s := &ServerApp{
		cfg:      cfg,
		userSvc:  userSvc,
		orderSvc: orderSvc,
		mux:      http.NewServeMux(),
	}
	s.routes()
	return s
}

func (s *ServerApp) routes() {
	s.mux.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
		w.Write([]byte("OK"))
	})
}

// ========================================
// 步骤 2：定义 Wire Injector（wire.go）
// ========================================

// 实际的 wire.go 文件内容如下：
//
// //go:build wireinject
//
// package main
//
// import "github.com/google/wire"
//
// // ProviderSet 将所有 Provider 组织到一个 Set 中
// var ProviderSet = wire.NewSet(
//     NewConfig,
//     NewDB,
//     NewUserRepo,
//     NewOrderRepo,
//     NewUserServiceWire,
//     NewOrderService,
//     NewServerApp,
// )
```

- **实际项目中的使用场景**：Wire 特别适合三层/四层架构（Config → DB → Repository → Service → Handler），在中大型项目中能显著减少 main 函数中冗长的初始化代码。
- **与其他语言的对比**：Java 的 Spring DI 是运行时容器、TypeScript 的 NestJS 使用装饰器+反射，而 Wire 在编译时完成所有依赖注入——生成的代码就是普通的函数调用链，没有任何"魔法"。
- **常见陷阱和最佳实践**：
  - `wire.go` 必须有 `//go:build wireinject` 标签，`wire_gen.go` 有 `//go:build !wireinject`，两者互斥编译。
  - Provider 函数的返回值类型决定了 Wire 的匹配规则：同类型只能有一个 Provider，否则需要用 `wire.Bind` 绑定接口到具体类型。
  - 安装命令：`go install github.com/google/wire/cmd/wire@latest`，运行 `wire ./...` 生成代码。
  - 如果依赖关系简单（< 10 个组件），手写初始化代码可能比引入 Wire 更清晰。

## 5. go/ast 与 go/parser

`go/ast` 和 `go/parser` 是 Go 标准库提供的 AST（抽象语法树）解析工具。它们让你能以编程方式分析和操作 Go 源代码——这是构建自定义代码生成器、linter、重构工具的基础。

```go
package main

import (
	"fmt"
	"go/ast"
	"go/parser"
	"go/token"
	"strings"
)

func main() {
	fmt.Println("=== go/ast 与 go/parser 演示 ===")
	fmt.Println()

	// 要解析的 Go 源代码
	src := `
package example

import (
	"context"
	"time"
)

// User 用户模型
type User struct {
	ID        int64
	Name      string
	Email     string
	CreatedAt time.Time
}

// UserService 用户服务接口
type UserService interface {
	GetByID(ctx context.Context, id int64) (*User, error)
	Create(ctx context.Context, user *User) error
	List(ctx context.Context, offset, limit int) ([]*User, error)
}

// userServiceImpl 是 UserService 的实现
type userServiceImpl struct {
	db interface{}
}

// GetByID 根据 ID 获取用户
func (s *userServiceImpl) GetByID(ctx context.Context, id int64) (*User, error) {
	return nil, nil
}

// NewUserService 创建 UserService 实例
func NewUserService(db interface{}) UserService {
	return &userServiceImpl{db: db}
}

// helper 内部辅助函数
func helper() string {
	return "help"
}
`

	// 创建文件集（用于跟踪文件和位置信息）
	fset := token.NewFileSet()

	// 解析源代码
	// parser.ParseComments 保留注释信息
	file, err := parser.ParseFile(fset, "example.go", src, parser.ParseComments)
	if err != nil {
		fmt.Printf("解析错误: %v\n", err)
		return
	}

	// 1. 提取包名
	fmt.Printf("包名: %s\n\n", file.Name.Name)

	// 2. 提取所有导入
	fmt.Println("导入包：")
	for _, imp := range file.Imports {
		fmt.Printf("  %s\n", imp.Path.Value)
	}
	fmt.Println()

	// 3. 遍历所有声明，提取类型和函数信息
	fmt.Println("类型声明：")
	for _, decl := range file.Decls {
		switch d := decl.(type) {
		case *ast.GenDecl:
			// 处理 type、var、const 声明
			for _, spec := range d.Specs {
				if ts, ok := spec.(*ast.TypeSpec); ok {
					switch t := ts.Type.(type) {
					case *ast.StructType:
						fmt.Printf("  struct %s（%d 个字段）\n", ts.Name.Name, t.Fields.NumFields())
						// 提取字段
						for _, field := range t.Fields.List {
							for _, name := range field.Names {
								fmt.Printf("    - %s: %s\n", name.Name, formatExpr(field.Type))
							}
						}
					case *ast.InterfaceType:
						fmt.Printf("  interface %s（%d 个方法）\n", ts.Name.Name, t.Methods.NumFields())
						for _, method := range t.Methods.List {
							for _, name := range method.Names {
								fmt.Printf("    - %s%s\n", name.Name, formatFuncType(method.Type))
							}
						}
					}
				}
			}
		case *ast.FuncDecl:
			// 处理函数声明
			// （后面统一处理）
		}
	}
	fmt.Println()

	// 4. 提取所有函数签名
	fmt.Println("函数签名：")
	for _, decl := range file.Decls {
		if fn, ok := decl.(*ast.FuncDecl); ok {
			// 判断是否为方法（有接收者）
			receiver := ""
			if fn.Recv != nil && len(fn.Recv.List) > 0 {
				receiver = fmt.Sprintf("(%s) ", formatExpr(fn.Recv.List[0].Type))
			}

			// 判断是否为导出函数
			exported := "未导出"
			if fn.Name.IsExported() {
				exported = "已导出"
			}

			// 提取参数
			params := formatFieldList(fn.Type.Params)
			// 提取返回值
			results := formatFieldList(fn.Type.Results)

			fmt.Printf("  func %s%s(%s)", receiver, fn.Name.Name, params)
			if results != "" {
				fmt.Printf(" (%s)", results)
			}
			fmt.Printf("  [%s]\n", exported)

			// 提取函数注释
			if fn.Doc != nil {
				fmt.Printf("    注释: %s\n", strings.TrimSpace(fn.Doc.Text()))
			}
		}
	}
	fmt.Println()

	// 5. 使用 ast.Inspect 深度遍历 —— 统计所有标识符
	identCount := 0
	ast.Inspect(file, func(n ast.Node) bool {
		if _, ok := n.(*ast.Ident); ok {
			identCount++
		}
		return true // 返回 true 继续遍历子节点
	})
	fmt.Printf("源码中共有 %d 个标识符\n", identCount)
}

// formatExpr 将 ast.Expr 格式化为可读字符串
func formatExpr(expr ast.Expr) string {
	switch t := expr.(type) {
	case *ast.Ident:
		return t.Name
	case *ast.SelectorExpr:
		return fmt.Sprintf("%s.%s", formatExpr(t.X), t.Sel.Name)
	case *ast.StarExpr:
		return "*" + formatExpr(t.X)
	case *ast.ArrayType:
		return "[]" + formatExpr(t.Elt)
	case *ast.InterfaceType:
		return "interface{}"
	case *ast.FuncType:
		return "func" + formatFuncType(expr)
	default:
		return fmt.Sprintf("%T", expr)
	}
}

// formatFuncType 格式化函数类型
func formatFuncType(expr ast.Expr) string {
	ft, ok := expr.(*ast.FuncType)
	if !ok {
		return ""
	}
	params := formatFieldList(ft.Params)
	results := formatFieldList(ft.Results)
	if results != "" {
		return fmt.Sprintf("(%s) (%s)", params, results)
	}
	return fmt.Sprintf("(%s)", params)
}

// formatFieldList 格式化参数/返回值列表
func formatFieldList(fl *ast.FieldList) string {
	if fl == nil {
		return ""
	}
	var parts []string
	for _, field := range fl.List {
		typeName := formatExpr(field.Type)
		if len(field.Names) > 0 {
			names := make([]string, len(field.Names))
			for i, n := range field.Names {
				names[i] = n.Name
			}
			parts = append(parts, fmt.Sprintf("%s %s", strings.Join(names, ", "), typeName))
		} else {
			parts = append(parts, typeName)
		}
	}
	return strings.Join(parts, ", ")
}
```

### 讲解重点

- **实际项目中的使用场景**：go/ast 是构建自定义代码生成器的基石。例如：从接口定义自动生成 HTTP handler、从 struct 标签生成验证代码、扫描代码库查找特定模式等。ent（ORM）、buf（protobuf）等工具底层都使用了类似的 AST 分析技术。
- **与其他语言的对比**：Python 有 `ast` 模块、JavaScript 有 Babel/ESTree，但 Go 的 AST 工具链是标准库的一部分且与编译器共享同一套解析器，这保证了 100% 的语法兼容性。
- **常见陷阱和最佳实践**：
  - `token.FileSet` 必须在整个解析过程中保持同一实例，否则位置信息会错乱。
  - `parser.ParseFile` 解析的是语法结构，不做类型检查；需要类型信息时应使用 `golang.org/x/tools/go/packages` 或 `go/types`。
  - `ast.Inspect` 适合简单遍历，`ast.Walk` 需要实现 `ast.Visitor` 接口，更灵活。
  - 修改 AST 后可用 `go/printer` 或 `go/format` 输出格式化的代码。

## 6. 基于模板的代码生成

`text/template` 是 Go 代码生成最常用的方式之一。通过定义模板和数据模型，可以批量生成结构化的 Go 源代码——从简单的 CRUD 到复杂的 ORM 代码都能胜任。

```go
package main

import (
	"bytes"
	"fmt"
	"go/format"
	"strings"
	"text/template"
)

// ModelDef 模型定义（模拟从 JSON Schema 或配置文件读取）
type ModelDef struct {
	PackageName string
	ModelName   string
	TableName   string
	Fields      []FieldDef
}

// FieldDef 字段定义
type FieldDef struct {
	Name     string // Go 字段名
	Type     string // Go 类型
	Column   string // 数据库列名
	JSONName string // JSON 字段名
}

// crudTemplate CRUD 代码生成模板
const crudTemplate = `// Code generated by crud-gen; DO NOT EDIT.

package {{.PackageName}}

import (
	"context"
	"database/sql"
	"fmt"
	"time"
)

// 确保 time 包被使用（避免未使用导入错误）
var _ = time.Now

// {{.ModelName}} 数据模型
type {{.ModelName}} struct {
{{- range .Fields}}
	{{.Name}} {{.Type}} ` + "`" + `json:"{{.JSONName}}" db:"{{.Column}}"` + "`" + `
{{- end}}
}

// {{.ModelName}}Repository {{.ModelName}} 的数据访问层
type {{.ModelName}}Repository struct {
	db *sql.DB
}

// New{{.ModelName}}Repository 创建 Repository
func New{{.ModelName}}Repository(db *sql.DB) *{{.ModelName}}Repository {
	return &{{.ModelName}}Repository{db: db}
}

// GetByID 根据 ID 查询
func (r *{{.ModelName}}Repository) GetByID(ctx context.Context, id int64) (*{{.ModelName}}, error) {
	query := "SELECT {{columns .Fields}} FROM {{.TableName}} WHERE id = $1"
	row := r.db.QueryRowContext(ctx, query, id)

	var m {{.ModelName}}
	err := row.Scan({{scanFields .Fields}})
	if err != nil {
		return nil, fmt.Errorf("查询 {{.ModelName}} 失败: %w", err)
	}
	return &m, nil
}

// Create 创建记录
func (r *{{.ModelName}}Repository) Create(ctx context.Context, m *{{.ModelName}}) error {
	query := "INSERT INTO {{.TableName}} ({{columnsNoID .Fields}}) VALUES ({{placeholdersNoID .Fields}}) RETURNING id"
	return r.db.QueryRowContext(ctx, query, {{insertFields .Fields}}).Scan(&m.ID)
}

// Update 更新记录
func (r *{{.ModelName}}Repository) Update(ctx context.Context, m *{{.ModelName}}) error {
	query := "UPDATE {{.TableName}} SET {{updateSet .Fields}} WHERE id = ${{len .Fields}}"
	_, err := r.db.ExecContext(ctx, query, {{updateFields .Fields}})
	return err
}

// Delete 删除记录
func (r *{{.ModelName}}Repository) Delete(ctx context.Context, id int64) error {
	query := "DELETE FROM {{.TableName}} WHERE id = $1"
	_, err := r.db.ExecContext(ctx, query, id)
	return err
}

// List 分页查询
func (r *{{.ModelName}}Repository) List(ctx context.Context, offset, limit int) ([]*{{.ModelName}}, error) {
	query := "SELECT {{columns .Fields}} FROM {{.TableName}} ORDER BY id LIMIT $1 OFFSET $2"
	rows, err := r.db.QueryContext(ctx, query, limit, offset)
	if err != nil {
		return nil, fmt.Errorf("查询 {{.ModelName}} 列表失败: %w", err)
	}
	defer rows.Close()

	var results []*{{.ModelName}}
	for rows.Next() {
		var m {{.ModelName}}
		if err := rows.Scan({{scanFields .Fields}}); err != nil {
			return nil, err
		}
		results = append(results, &m)
	}
	return results, rows.Err()
}
`

func main() {
	fmt.Println("=== 基于模板的代码生成演示 ===")
	fmt.Println()

	// 定义模型（实际项目中从 YAML/JSON/数据库 schema 读取）
	model := ModelDef{
		PackageName: "repository",
		ModelName:   "Article",
		TableName:   "articles",
		Fields: []FieldDef{
			{Name: "ID", Type: "int64", Column: "id", JSONName: "id"},
			{Name: "Title", Type: "string", Column: "title", JSONName: "title"},
			{Name: "Content", Type: "string", Column: "content", JSONName: "content"},
			{Name: "AuthorID", Type: "int64", Column: "author_id", JSONName: "author_id"},
			{Name: "Status", Type: "int", Column: "status", JSONName: "status"},
			{Name: "CreatedAt", Type: "time.Time", Column: "created_at", JSONName: "created_at"},
			{Name: "UpdatedAt", Type: "time.Time", Column: "updated_at", JSONName: "updated_at"},
		},
	}

	// 创建模板函数映射
	funcMap := template.FuncMap{
		// 生成所有列名（逗号分隔）
		"columns": func(fields []FieldDef) string {
			cols := make([]string, len(fields))
			for i, f := range fields {
				cols[i] = f.Column
			}
			return strings.Join(cols, ", ")
		},
		// 生成除 ID 外的列名
		"columnsNoID": func(fields []FieldDef) string {
			var cols []string
			for _, f := range fields {
				if f.Name != "ID" {
					cols = append(cols, f.Column)
				}
			}
			return strings.Join(cols, ", ")
		},
		// 生成 Scan 参数
		"scanFields": func(fields []FieldDef) string {
			refs := make([]string, len(fields))
			for i, f := range fields {
				refs[i] = "&m." + f.Name
			}
			return strings.Join(refs, ", ")
		},
		// 生成 INSERT 占位符（除 ID 外）
		"placeholdersNoID": func(fields []FieldDef) string {
			var phs []string
			idx := 1
			for _, f := range fields {
				if f.Name != "ID" {
					phs = append(phs, fmt.Sprintf("$%d", idx))
					idx++
				}
			}
			return strings.Join(phs, ", ")
		},
		// 生成 INSERT 字段引用（除 ID 外）
		"insertFields": func(fields []FieldDef) string {
			var refs []string
			for _, f := range fields {
				if f.Name != "ID" {
					refs = append(refs, "m."+f.Name)
				}
			}
			return strings.Join(refs, ", ")
		},
		// 生成 UPDATE SET 子句（除 ID 外）
		"updateSet": func(fields []FieldDef) string {
			var sets []string
			idx := 1
			for _, f := range fields {
				if f.Name != "ID" {
					sets = append(sets, fmt.Sprintf("%s = $%d", f.Column, idx))
					idx++
				}
			}
			return strings.Join(sets, ", ")
		},
		// 生成 UPDATE 字段引用（除 ID 外，末尾加 ID）
		"updateFields": func(fields []FieldDef) string {
			var refs []string
			for _, f := range fields {
				if f.Name != "ID" {
					refs = append(refs, "m."+f.Name)
				}
			}
			refs = append(refs, "m.ID")
			return strings.Join(refs, ", ")
		},
	}

	// 解析模板
	tmpl, err := template.New("crud").Funcs(funcMap).Parse(crudTemplate)
	if err != nil {
		fmt.Printf("模板解析错误: %v\n", err)
		return
	}

	// 执行模板
	var buf bytes.Buffer
	if err := tmpl.Execute(&buf, model); err != nil {
		fmt.Printf("模板执行错误: %v\n", err)
		return
	}

	// 使用 go/format 格式化生成的代码（等效于 gofmt）
	formatted, err := format.Source(buf.Bytes())
	if err != nil {
		fmt.Printf("格式化错误: %v\n", err)
		fmt.Println("原始输出：")
		fmt.Println(buf.String())
		return
	}

	// 输出生成的代码
	fmt.Println("生成的 CRUD 代码：")
	fmt.Println(strings.Repeat("=", 60))
	fmt.Println(string(formatted))
	fmt.Println(strings.Repeat("=", 60))

	// 完整的代码生成流程总结
	fmt.Println()
	fmt.Println("代码生成完整流程：")
	fmt.Println("1. 定义数据源（JSON Schema / YAML / 数据库 DDL）")
	fmt.Println("2. 编写 Go 模板（text/template 语法）")
	fmt.Println("3. 解析数据源 → 填充模型结构体")
	fmt.Println("4. 执行模板 → 输出到 bytes.Buffer")
	fmt.Println("5. go/format.Source() 格式化（等效 gofmt）")
	fmt.Println("6. 写入 _gen.go 文件")
}
```

### 讲解重点

- **实际项目中的使用场景**：模板驱动的代码生成广泛用于 ORM（ent、sqlboiler）、API 客户端生成（openapi-generator）、protobuf 插件（protoc-gen-go）等。项目中重复性的 CRUD、DTO 转换、路由注册代码都是模板生成的好候选。
- **与其他语言的对比**：Java 用 Annotation Processor + JavaPoet 生成代码，Rust 用 proc_macro + quote! 宏，TypeScript 用 ts-morph。Go 选择 `text/template` 是最简单直接的方式，无需学习额外的 AST 构造 API。
- **常见陷阱和最佳实践**：
  - 生成代码后**必须**调用 `go/format.Source()` 格式化，否则缩进和空行可能不规范。
  - 模板中要处理好包导入——未使用的 import 会导致编译失败。可用 `goimports` 替代 `gofmt` 自动处理导入。
  - 生成的文件开头务必加 `// Code generated by xxx; DO NOT EDIT.` 注释。
  - 对于复杂的代码生成，考虑使用 `jennifer`（Go 代码生成 DSL）替代 text/template，可以避免模板拼接字符串的各种转义问题。

## 7. go:embed 静态资源嵌入

`//go:embed` 是 Go 1.16 引入的指令，可以在编译时将文件和目录嵌入到 Go 二进制文件中。这让部署变得极其简单——一个二进制文件包含所有需要的资源，无需额外的文件分发。

```go
package main

import (
	"embed"
	"fmt"
	"io/fs"
	"strings"
)

// ========================================
// 1. embed.String —— 嵌入单个文件为字符串
// ========================================

// 嵌入 SQL 迁移文件（实际项目中文件路径相对于当前 Go 源文件）
// //go:embed migrations/001_create_users.sql
// var createUsersSQL string

// 演示用：直接定义 SQL 内容
var createUsersSQL = `CREATE TABLE IF NOT EXISTS users (
    id         BIGSERIAL PRIMARY KEY,
    name       VARCHAR(100) NOT NULL,
    email      VARCHAR(255) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
```

- **实际项目中的使用场景**：Web 应用嵌入前端构建产物（SPA）、数据库迁移工具嵌入 SQL 文件、CLI 工具嵌入默认配置模板、嵌入 TLS 证书。典型案例：`hugo`、`gitea` 等项目都大量使用 embed 打包静态资源。
- **与其他语言的对比**：Rust 有 `include_str!` 和 `include_bytes!`（仅单文件），Java 通过 ClassLoader 加载 resources 目录，Go 的 embed 在功能上（支持整个目录、实现标准 fs 接口）是最完善的。
- **常见陷阱和最佳实践**：
  - `//go:embed` 与变量声明之间**不能有空行**，否则指令不生效。
  - 嵌入大文件（>几十 MB）会显著增大二进制体积，需要权衡。
  - `embed.FS` 是只读的，如果需要运行时修改，需先拷贝到临时目录。
  - 开发模式下可以用 `os.DirFS("static")` 替代 embed.FS，实现热重载。

## 8. 代码生成 vs 反射 vs 泛型

Go 提供了三种"元编程"方式：代码生成（编译前）、反射（运行时）、泛型（编译时）。理解它们各自的优势和限制，是做出正确技术选型的关键。

```go
package main

import (
	"encoding/json"
	"fmt"
	"reflect"
	"strings"
)

// ========================================
// 方式 1：反射 —— 运行时动态处理
// ========================================

// StructToMap 使用反射将任意结构体转换为 map（运行时通用方案）
func StructToMap(v interface{}) map[string]interface{} {
	result := make(map[string]interface{})
	val := reflect.ValueOf(v)
	// 处理指针
	if val.Kind() == reflect.Ptr {
		val = val.Elem()
	}
	if val.Kind() != reflect.Struct {
		return result
	}
	typ := val.Type()
	for i := 0; i < val.NumField(); i++ {
		field := typ.Field(i)
		if !field.IsExported() {
			continue
		}
		// 使用 json tag 作为 key
		key := field.Tag.Get("json")
		if key == "" || key == "-" {
			key = field.Name
		}
		// 处理 omitempty 等选项
		if idx := strings.Index(key, ","); idx != -1 {
			key = key[:idx]
		}
		result[key] = val.Field(i).Interface()
	}
	return result
}

// ========================================
// 方式 2：泛型 —— 编译时类型安全
// ========================================

// Map 泛型版 map 函数（编译时类型安全）
func Map[T any, R any](slice []T, fn func(T) R) []R {
	result := make([]R, len(slice))
	for i, v := range slice {
		result[i] = fn(v)
	}
	return result
}

// Filter 泛型版 filter 函数
func Filter[T any](slice []T, fn func(T) bool) []T {
	var result []T
	for _, v := range slice {
		if fn(v) {
			result = append(result, v)
		}
	}
	return result
}

// Reduce 泛型版 reduce 函数
func Reduce[T any, R any](slice []T, initial R, fn func(R, T) R) R {
	result := initial
	for _, v := range slice {
		result = fn(result, v)
	}
	return result
}

// ========================================
// 方式 3：代码生成 —— 编译前生成特定类型的代码
// ========================================

// 假设我们通过代码生成工具为 Product 类型生成了以下方法：
// （实际项目中这些会在 product_gen.go 中自动生成）

// Product 商品模型
type Product struct {
	ID    int64   `json:"id"`
	Name  string  `json:"name"`
	Price float64 `json:"price"`
}

// ProductToMap 代码生成版：为 Product 定制的转换方法（零反射开销）
func ProductToMap(p *Product) map[string]interface{} {
	return map[string]interface{}{
		"id":    p.ID,
		"name":  p.Name,
		"price": p.Price,
	}
}

// ProductFromMap 代码生成版：从 map 构造 Product
func ProductFromMap(m map[string]interface{}) *Product {
	p := &Product{}
	if v, ok := m["id"].(float64); ok {
		p.ID = int64(v)
	}
	if v, ok := m["name"].(string); ok {
		p.Name = v
	}
	if v, ok := m["price"].(float64); ok {
		p.Price = v
	}
	return p
}

func main() {
	fmt.Println("=== 代码生成 vs 反射 vs 泛型 ===")
	fmt.Println()

	// --- 反射示例 ---
	fmt.Println("【反射】StructToMap（运行时通用方案）：")
	p := Product{ID: 1, Name: "Go 语言圣经", Price: 59.90}
	m := StructToMap(p)
	fmt.Printf("  输入: %+v\n", p)
	fmt.Printf("  输出: %v\n\n", m)

	// --- 泛型示例 ---
	fmt.Println("【泛型】集合操作（编译时类型安全）：")
	prices := []float64{29.9, 59.9, 99.0, 15.5, 199.0}
	fmt.Printf("  原始价格: %v\n", prices)

	// Filter: 筛选 > 50 的价格
	expensive := Filter(prices, func(p float64) bool { return p > 50 })
	fmt.Printf("  筛选 > 50: %v\n", expensive)

	// Map: 全部打 8 折
	discounted := Map(prices, func(p float64) float64 { return p * 0.8 })
	fmt.Printf("  打 8 折:   %v\n", discounted)

	// Reduce: 求总价
	total := Reduce(prices, 0.0, func(acc, p float64) float64 { return acc + p })
	fmt.Printf("  总价:      %.2f\n\n", total)

	// --- 代码生成示例 ---
	fmt.Println("【代码生成】ProductToMap（零反射开销）：")
	p2 := &Product{ID: 2, Name: "Go 并发编程", Price: 79.00}
	m2 := ProductToMap(p2)
	fmt.Printf("  输入: %+v\n", *p2)
	fmt.Printf("  输出: %v\n\n", m2)

	// --- JSON 序列化：典型的反射场景 ---
	fmt.Println("【反射的典型场景】JSON 序列化：")
	data, _ := json.Marshal(p)
	fmt.Printf("  json.Marshal: %s\n\n", data)

	// ========================================
	// 三种方式的对比
	// ========================================
	fmt.Println("三种元编程方式对比：")
	fmt.Println("┌────────────────┬───────────────┬───────────────┬───────────────┐")
	fmt.Println("│ 维度           │ 代码生成       │ 反射          │ 泛型          │")
	fmt.Println("├────────────────┼───────────────┼───────────────┼───────────────┤")
	fmt.Println("│ 编译时安全性    │ ✅ 高         │ ❌ 低         │ ✅ 高         │")
	fmt.Println("│ 运行时性能     │ ✅ 最优       │ ❌ 较差       │ ✅ 接近最优    │")
	fmt.Println("│ 代码量         │ ❌ 多（生成的）│ ✅ 少         │ ✅ 适中       │")
	fmt.Println("│ 可读性         │ ⚠️ 模板难维护  │ ❌ 反射代码晦涩│ ✅ 较好       │")
	fmt.Println("│ 灵活性         │ ⚠️ 需重新生成  │ ✅ 最灵活     │ ⚠️ 受类型约束  │")
	fmt.Println("│ 调试难度       │ ✅ 可直接调试  │ ❌ 运行时报错  │ ✅ 编译期报错  │")
	fmt.Println("│ 适合场景       │ ORM、序列化   │ JSON、配置映射 │ 集合操作、容器 │")
	fmt.Println("└────────────────┴───────────────┴───────────────┴───────────────┘")
	fmt.Println()

	// 决策树
	fmt.Println("选择决策树：")
	fmt.Println("  需要处理任意类型？")
	fmt.Println("  ├── 是 → 类型之间是否有共同约束（接口/类型集合）？")
	fmt.Println("  │        ├── 是 → 用泛型（如 sort、sync.Pool 等通用容器）")
	fmt.Println("  │        └── 否 → 需要检查结构体字段/标签？")
	fmt.Println("  │                 ├── 是 → 反射（如 json.Marshal、ORM 映射）")
	fmt.Println("  │                 └── 否 → 用 interface{} + 类型断言")
	fmt.Println("  └── 否 → 是否有大量重复的类型特定代码？")
	fmt.Println("           ├── 是 → 代码生成（如 protobuf、ent、stringer）")
	fmt.Println("           └── 否 → 直接手写")
}
```

### 讲解重点

- **实际项目中的使用场景**：JSON/XML 序列化用反射（encoding/json）；集合操作、缓存容器用泛型（Go 1.18+）；ORM 代码、protobuf 消息、API 客户端用代码生成。大型项目中三者通常并存。
- **与其他语言的对比**：Java/C# 有强大的反射+注解系统；Rust 有宏系统（替代反射和代码生成）；TypeScript 有类型体操（类型级元编程）。Go 没有宏系统，这是有意的设计选择——避免"看不懂的魔法"，让代码保持可读性。
- **常见陷阱和最佳实践**：
  - 反射的性能开销主要在 `reflect.ValueOf` 的堆分配和类型查找，热路径上应避免反射（或缓存反射结果）。
  - 泛型不是万能的：Go 的泛型不支持方法级别的类型参数、不支持特化，复杂的类型约束会降低可读性。
  - 代码生成的最大缺点是工具链依赖和维护模板的成本，小项目不值得引入。
  - 优先级建议：手写 > 泛型 > 代码生成 > 反射（从简到繁）。

```go
	}
	for _, s := range scenarios {
		fmt.Printf("  • %s: %s\n", s.name, s.desc)
	}

	BadCode()
	GoodCode()
}
```

### 讲解重点

- **实际项目中的使用场景**：大型团队通过自定义分析器强制执行编码规范——比如禁止直接使用 `fmt.Println`（应使用结构化日志）、确保所有 HTTP handler 都有超时控制、检查 SQL 查询是否使用参数化等。这比 code review 中人工检查更可靠。
- **与其他语言的对比**：Java 有 Error Prone（Google）和 SpotBugs，TypeScript 有 ESLint 自定义规则，Rust 有 Clippy lint。Go 的 `analysis` 框架与编译器共享类型信息，分析能力远超简单的正则匹配。
- **常见陷阱和最佳实践**：
  - 分析器应该快速且确定性——避免网络请求、文件 I/O 等副作用。
  - 使用 `pass.TypesInfo` 而非仅靠 AST 判断类型，AST 只有语法信息。
  - 提供 `SuggestedFix` 让 IDE 可以一键修复问题。
  - 先用 `analysistest.Run()` 编写测试，通过 `// want "..."` 注释标记期望的诊断信息。

## 10. 构建标签与条件编译

构建标签（Build Tags）让你能根据操作系统、CPU 架构或自定义条件选择性编译代码。这是 Go 实现跨平台支持的核心机制——标准库中大量使用了此技术。

```go
package main

import (
	"fmt"
	"runtime"
)

// ========================================
// 构建标签的语法（Go 1.17+ 推荐写法）
// ========================================

// 新语法（Go 1.17+，推荐）：
// //go:build linux
// //go:build linux && amd64
// //go:build linux || darwin
// //go:build !windows
// //go:build (linux || darwin) && amd64

// 旧语法（Go 1.17 之前，仍兼容）：
// // +build linux
// // +build linux,amd64
// // +build linux darwin
// // +build !windows

// ========================================
// 按平台编译不同实现的示例
// ========================================

// 实际项目中，以下代码分布在三个文件中：

// --- file_linux.go ---
// //go:build linux
//
// package main
//
// func getPlatformInfo() string {
//     return "Linux 平台：使用 epoll 事件驱动"
// }
//
// func getConfigDir() string {
//     return "/etc/myapp"
// }

// --- file_darwin.go ---
// //go:build darwin
//
// package main
//
// func getPlatformInfo() string {
//     return "macOS 平台：使用 kqueue 事件驱动"
// }
//
// func getConfigDir() string {
//     home, _ := os.UserHomeDir()
//     return filepath.Join(home, "Library", "Application Support", "myapp")
// }

// --- file_windows.go ---
// //go:build windows
//
// package main
//
// func getPlatformInfo() string {
//     return "Windows 平台：使用 IOCP 事件驱动"
// }
//
// func getConfigDir() string {
//     return filepath.Join(os.Getenv("APPDATA"), "myapp")
// }

// 为了演示，我们用运行时判断模拟条件编译的效果
func getPlatformInfo() string {
	switch runtime.GOOS {
	case "linux":
		return "Linux 平台：使用 epoll 事件驱动"
	case "darwin":
		return "macOS 平台：使用 kqueue 事件驱动"
	case "windows":
		return "Windows 平台：使用 IOCP 事件驱动"
	default:
		return fmt.Sprintf("%s 平台", runtime.GOOS)
	}
}

func getConfigDir() string {
	switch runtime.GOOS {
	case "linux":
		return "/etc/myapp"
	case "darwin":
		return "~/Library/Application Support/myapp"
	case "windows":
		return "%APPDATA%\\myapp"
	default:
		return "/etc/myapp"
	}
}

// ========================================
// 自定义构建标签（debug/release）
// ========================================

// --- debug.go ---
// //go:build debug
//
// package main
//
// const isDebug = true
//
// func debugLog(msg string) {
//     log.Printf("[DEBUG] %s", msg)
// }

// --- release.go ---
// //go:build !debug
//
// package main
//
// const isDebug = false
//
// func debugLog(msg string) {
//     // release 模式下什么都不做，编译器会内联优化掉
// }

// 演示用
const isDebug = false

func debugLog(msg string) {
	if isDebug {
		fmt.Printf("[DEBUG] %s\n", msg)
	}
}

// ========================================
// 文件名约定与构建标签的关系
// ========================================

// Go 编译器会自动识别以下文件名模式（无需显式 //go:build 标签）：
//
// *_GOOS.go          → 仅在指定 OS 编译
//   file_linux.go    → 仅 Linux
//   file_darwin.go   → 仅 macOS
//   file_windows.go  → 仅 Windows
//
// *_GOARCH.go        → 仅在指定架构编译
//   file_amd64.go    → 仅 amd64
//   file_arm64.go    → 仅 arm64
//
// *_GOOS_GOARCH.go   → 指定 OS + 架构
//   file_linux_amd64.go → 仅 Linux amd64
//
// *_test.go          → 仅在 go test 时编译

func main() {
	fmt.Println("=== 构建标签与条件编译演示 ===")
	fmt.Println()

	// 当前平台信息
	fmt.Printf("当前平台: %s/%s\n", runtime.GOOS, runtime.GOARCH)
	fmt.Printf("平台信息: %s\n", getPlatformInfo())
	fmt.Printf("配置目录: %s\n", getConfigDir())
	fmt.Println()

	// 构建标签语法对照
	fmt.Println("构建标签语法（Go 1.17+ 新语法 vs 旧语法）：")
	fmt.Println("┌──────────────────────────────────┬──────────────────────────────┐")
	fmt.Println("│ 新语法（推荐）                    │ 旧语法（兼容）                │")
	fmt.Println("├──────────────────────────────────┼──────────────────────────────┤")
	fmt.Println("│ //go:build linux                 │ // +build linux              │")
	fmt.Println("│ //go:build linux && amd64        │ // +build linux,amd64        │")
	fmt.Println("│ //go:build linux || darwin       │ // +build linux darwin        │")
	fmt.Println("│ //go:build !windows              │ // +build !windows           │")
	fmt.Println("│ //go:build (linux || darwin)     │ // +build linux darwin        │")
	fmt.Println("│           && amd64               │ // +build amd64              │")
	fmt.Println("│ //go:build ignore                │ // +build ignore             │")
	fmt.Println("└──────────────────────────────────┴──────────────────────────────┘")
	fmt.Println()

	// 文件名约定
	fmt.Println("文件名约定（自动识别，无需显式标签）：")
	filePatterns := []struct {
		pattern string
		desc    string
	}{
		{"*_linux.go", "仅在 GOOS=linux 时编译"},
		{"*_darwin.go", "仅在 GOOS=darwin 时编译"},
		{"*_windows.go", "仅在 GOOS=windows 时编译"},
		{"*_amd64.go", "仅在 GOARCH=amd64 时编译"},
		{"*_arm64.go", "仅在 GOARCH=arm64 时编译"},
		{"*_linux_amd64.go", "仅在 linux/amd64 时编译"},
		{"*_test.go", "仅在 go test 时编译"},
	}
	for _, p := range filePatterns {
		fmt.Printf("  %-25s → %s\n", p.pattern, p.desc)
	}
	fmt.Println()

	// 使用自定义标签
	fmt.Println("使用自定义构建标签：")
	fmt.Println("  编译命令: go build -tags debug .")
	fmt.Println("  测试命令: go test -tags integration ./...")
	fmt.Println("  多标签:   go build -tags 'debug,feature_x' .")
	fmt.Println()
	fmt.Printf("  当前 isDebug = %v\n", isDebug)
	debugLog("这条日志只在 debug 模式下可见")
	fmt.Println()

	// 常见使用场景
	fmt.Println("常见使用场景：")
	scenarios := []struct {
		tag  string
		desc string
	}{
		{"//go:build linux", "平台特定的系统调用（syscall、epoll、kqueue 等）"},
		{"//go:build cgo", "需要 CGO 的代码与纯 Go 回退实现"},
		{"//go:build !race", "在竞态检测器开启时跳过某些测试"},
		{"//go:build integration", "集成测试（默认不运行，CI 中显式启用）"},
		{"//go:build wireinject", "Wire 依赖注入的注入器定义文件"},
		{"//go:build ignore", "临时禁用某个文件的编译"},
		{"//go:build tools", "记录项目工具依赖（tools.go 模式）"},
	}
	for _, s := range scenarios {
		fmt.Printf("  %-30s → %s\n", s.tag, s.desc)
	}
	fmt.Println()

	// tools.go 模式
	fmt.Println("tools.go 模式（管理工具依赖）：")
	fmt.Println(`  // tools.go
  //go:build tools

  package main

  import (
      _ "golang.org/x/tools/cmd/stringer"
      _ "go.uber.org/mock/mockgen"
      _ "github.com/google/wire/cmd/wire"
  )
  // 这些 import 确保 go.mod 追踪这些工具依赖
  // 但 //go:build tools 确保它们不参与正常编译`)
}
```

### 讲解重点

- **实际项目中的使用场景**：Go 标准库中 `os`、`syscall`、`net` 等包大量使用文件名约定实现跨平台。项目中常见的用法包括：集成测试隔离（`-tags integration`）、CGO 回退（有 CGO 用 C 库，无则用纯 Go 实现）、Wire 注入器隔离。
- **与其他语言的对比**：C/C++ 用 `#ifdef` 预处理器（任意条件编译），Rust 用 `#[cfg(target_os = "linux")]`（属性宏），Java 完全没有条件编译（靠运行时判断）。Go 的构建标签比 C 的 `#ifdef` 更安全——每个文件要么整个编译要么整个排除，不会出现 C 中条件编译导致的半截代码问题。
- **常见陷阱和最佳实践**：
  - `//go:build` 标签必须在文件第一行（或紧跟 `// Copyright` 注释之后），且与 `package` 声明之间有一个空行。
  - 新旧语法共存时，`gofmt` 会自动同步两种格式；Go 1.17+ 建议只写新语法。
  - 文件名约定的优先级低于显式标签——如果 `file_linux.go` 中有 `//go:build darwin`，则只在 darwin 编译。
  - 使用 <code v-pre>go list -tags debug -f '{{.GoFiles}}' .</code> 可以查看特定标签下会编译哪些文件。
