## 排障流程

### 1. 先确认问题类型

- CPU 高
- 内存上涨
- 延迟抖动
- Goroutine 堆积
- 锁竞争严重

### 2. 再选对应 Profile

- CPU 问题：看 `cpu`
- 内存问题：看 `heap`
- Goroutine 泄漏：看 `goroutine`
- 锁竞争：看 `mutex`
- 阻塞问题：看 `block`

### 3. 结合业务现场分析

- 先找热点函数
- 再看调用链
- 再回到代码和请求场景验证

## 常看指标

### CPU

- 热点函数占比
- 调用链路是否集中在序列化、反射、IO、锁等待

### Heap

- 哪些对象分配最多
- 哪些路径持续持有对象
- 是否是缓存、切片、字符串导致内存膨胀

### Goroutine

- 是否存在大量卡在同一位置的 Goroutine
- 是否有请求超时后未退出的后台任务

### Mutex / Block

- 是否有单点热点锁
- 是否存在 channel 阻塞或 IO 等待堆积

## 常见场景

### CPU 高

- JSON 编解码过重
- 日志过多
- 反射频繁
- 热点循环里有多余分配

### 内存高

- 大对象切片长期持有
- 缓存无上限
- 字符串复制过多
- 临时对象分配过于频繁

### Goroutine 泄漏

- Channel 无接收方
- Context 取消未透传
- 定时任务退出条件缺失
- 重试逻辑无限循环

## 排查误区

- 只看一个截图，不结合业务请求场景
- 看见热点函数就直接改，没有先验证收益
- 把偶发采样结果当成长期结论
- 不做压测复现，只在生产临时猜测

## 推荐实践项目

- 给一个 HTTP 服务接入 `pprof`
- 人为制造 CPU 热点并抓取 profile
- 人为制造 Goroutine 泄漏并定位堆积点
- 对优化前后做 Benchmark 对比

## 推荐整理模板

每次排障建议记录 6 列：

- 现象
- Profile 类型
- 热点位置
- 根因判断
- 优化动作
- 验证结果
