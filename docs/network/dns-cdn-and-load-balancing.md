## DNS 在解决什么问题

DNS 本质上解决的是：**人用域名，机器用 IP，需要一套分布式系统把域名翻译成 IP**。

除了最基础的域名到 IP 映射之外，DNS 还承担着：

- 按地域、运营商返回不同 IP（智能解析 / GeoDNS）
- 通过 CNAME 把流量导向 CDN 或云负载均衡
- 做灾备切换——故障时改解析指向备用集群
- 做服务发现——内部 DNS 或 Consul DNS 接口

### 你必须分清的角色

| 角色 | 做什么 | 举例 |
| --- | --- | --- |
| 本地缓存 | 浏览器、操作系统各自缓存解析结果，避免重复查询 | Chrome DNS cache、`/etc/hosts` |
| 递归 DNS | 替你完成"从根到权威"的完整查询，返回最终结果 | `8.8.8.8`、运营商 DNS、公司内网 DNS |
| 权威 DNS | 真正存储域名记录的服务器，对自己负责的区域给出权威回答 | Cloudflare DNS、阿里云云解析 |
| 根 DNS / TLD DNS | 递归查询的起点，根服务器指向 `.com`、`.cn` 等顶级域 | 全球 13 组根服务器 |

### 常见记录类型

| 类型 | 含义 | 典型场景 |
| --- | --- | --- |
| `A` | 域名 → IPv4 地址 | 最基础的解析 |
| `AAAA` | 域名 → IPv6 地址 | 双栈环境 |
| `CNAME` | 域名 → 另一个域名（别名） | 接入 CDN 时最常用，如 `www.example.com → example.cdn.provider.net` |
| `MX` | 邮件交换记录 | 配置企业邮箱 |
| `TXT` | 文本记录 | SPF 反垃圾邮件验证、域名所有权验证、DKIM 签名 |
| `NS` | 指定域名由哪组权威 DNS 管理 | 域名托管迁移 |
| `SRV` | 指定服务的主机和端口 | 微服务注册与发现 |

::: tip CNAME 为什么不能和 A 记录共存在同一个域名上
DNS 标准规定 CNAME 是"别名"，意味着该域名的所有查询都应该先去追 CNAME 指向的目标。如果同时有 A 记录，解析器不知道该用哪个结果。所以裸域（`example.com`）通常不能直接 CNAME 到 CDN，只能用 A 记录或部分服务商提供的 ALIAS/ANAME 记录。
:::

## DNS 解析过程

一个域名从浏览器到最终拿到 IP，经过的查询链条是：

1. **浏览器缓存**：最先查，命中直接用
2. **操作系统缓存**：包括 `/etc/hosts` 和系统 DNS 缓存
3. **递归 DNS 缓存**：递归解析器（如 ISP DNS、`8.8.8.8`）自身的缓存
4. **递归查询**：缓存都没命中，递归 DNS 从根开始逐级查询
   - 根 DNS → `.com` TLD → `example.com` 的权威 DNS → 拿到 A 记录
5. **结果沿链路返回并缓存**：每一级都会按 TTL 缓存结果

### 递归查询 vs 迭代查询

- **递归查询**：客户端把问题甩给递归 DNS，递归 DNS 负责查到底并返回最终答案
- **迭代查询**：递归 DNS 向根 / TLD / 权威逐级询问时，每一级只回答"你该去问谁"，而不是直接给最终答案

工程上你只需记住：**用户到递归 DNS 是递归，递归 DNS 到各级权威是迭代**。

### 智能解析（GeoDNS）

很多生产环境的权威 DNS 会根据请求来源返回不同的 IP：

- 按地域（华东用户解析到上海机房，华南用户解析到广州机房）
- 按运营商（电信用户解析到电信线路 IP，联通用户解析到联通线路 IP）
- 按权重（灰度发布时 90% 流量解析到旧集群，10% 到新集群）

这就是为什么同一个域名在不同地方 `dig` 可能拿到不同 IP——并不是 DNS "出了问题"。

### DNS 安全

需要知道的几个风险点：

- **DNS 劫持**：中间人篡改解析结果，把用户导向钓鱼站。运营商劫持广告注入也属此类
- **DNS 污染**：权威或递归层面返回错误 IP，导致某些域名在特定网络环境下无法正确解析
- **DNSSEC**：通过数字签名保证解析结果的真实性和完整性，防止篡改，但部署率仍不高
- **DoH / DoT**：DNS over HTTPS / DNS over TLS，加密 DNS 查询过程，防止窃听和中间人篡改

## 为什么 TTL 很重要

TTL（Time To Live）决定 DNS 解析结果能被缓存多久。它不只是一个"配置项"，而是直接影响线上运维操作的核心参数。

### TTL 影响什么

| 场景 | TTL 太长（如 86400s / 24h） | TTL 太短（如 30s） |
| --- | --- | --- |
| 发布新 IP | 最多要等 24 小时才能全网生效 | 几十秒内生效 |
| 故障切流 | 大量用户仍然解析到故障 IP | 快速切到备用 IP |
| DNS 查询压力 | 缓存命中率高，查询量小 | 递归 DNS 查询量大幅增加 |
| 对上游依赖 | 低 | 高——上游 DNS 抖动直接影响你 |

### 常见 TTL 策略

- **常规业务**：300s ~ 600s（5 ~ 10 分钟），平衡生效速度和缓存效率
- **故障预案前**：提前把 TTL 降到 60s，确保切流操作能快速生效
- **稳定期**：可以适当拉长到 3600s（1 小时）甚至更长
- **CDN 接入后**：CNAME 记录的 TTL 通常可以设得较长，因为流量调度由 CDN 内部 DNS 完成

::: warning TTL 不等于"精确到秒生效"
各级缓存的 TTL 计时不一定同步开始，且部分递归 DNS 实现可能不严格遵循 TTL。所以"把 TTL 设成 60s 就一定 60s 内生效"是理想情况，实际可能有几十秒到几分钟的偏差。
:::

## CDN 在解决什么问题

CDN（Content Delivery Network）的核心价值是三件事：

1. **就近访问，降低延迟**——用户请求被导向距离最近的边缘节点
2. **边缘缓存，减轻源站压力**——静态资源在边缘命中，不回源
3. **安全和访问控制**——部分 CDN 在边缘集成了 WAF、DDoS 防护、IP 黑名单

### 静态加速 vs 动态加速

- **静态加速**：缓存 JS、CSS、图片、字体等不常变化的资源。这是 CDN 最经典的用法
- **动态加速（DSA / DCDN）**：对 API 接口等无法缓存的请求，CDN 通过优化传输路径（选择更快的骨干网、减少中间跳数）来降低延迟，而不是缓存内容

一个非常常见的误区：**以为用了 CDN 就一定更快**。如果缓存策略不合理、回源链路不稳、边缘节点质量差，CDN 不仅不快，反而因为多了一跳而更慢。

## CDN 的工作机制

### CDN 如何通过 DNS 接入

最常见的接入方式是 CNAME：

1. 你在权威 DNS 把 `static.example.com` CNAME 到 CDN 提供的域名，如 `example.cdn.provider.net`
2. 用户解析 `static.example.com` 时，DNS 会跟着 CNAME 链最终到达 CDN 的调度 DNS
3. CDN 的调度 DNS 根据用户的地理位置、运营商、节点负载等信息，返回最优边缘节点的 IP
4. 用户直接访问该边缘节点

### 缓存层次

典型的 CDN 不只有一层缓存：

| 层级 | 说明 |
| --- | --- |
| L1 边缘节点 | 离用户最近的 POP 点，数量最多、覆盖最广 |
| L2 中间层 / 区域中心 | 当 L1 未命中时，先向 L2 请求而不是直接回源 |
| Origin Shield（源站盾） | 源站前面的最后一级缓存，所有区域的未命中请求先汇聚到这里，大幅减少源站压力 |
| 源站 | 你的实际服务器 |

这种分层的意义在于：**避免大量边缘节点同时回源冲垮你的源站**。

### 缓存命中、回源与失效

- **缓存命中**：请求到达边缘节点，节点本地有对应资源且未过期，直接返回
- **回源**（Cache Miss）：边缘没有缓存或缓存已过期，需要向上层或源站请求
- **缓存失效 / 刷新**：主动让某些缓存立即过期。通常通过 CDN 控制台提交 URL/目录刷新
- **预热**：提前把热点资源推到边缘节点，避免上线瞬间大量回源

### CDN 与缓存控制

CDN 节点遵循的缓存策略通常由这些 HTTP 头控制：

- `Cache-Control`：`max-age=3600` 表示资源可被缓存 1 小时
- `s-maxage`：专门给共享缓存（CDN、代理）用的 max-age，优先级高于 `max-age`
- `Expires`：绝对过期时间，优先级低于 `Cache-Control`
- `Vary`：告诉 CDN 按某些 Header（如 `Accept-Encoding`）区分缓存版本

::: warning 发版后用户还是看到旧资源？
多半是缓存未正确失效。解决思路：构建时给文件名加 content hash（如 `app.3a7c9f.js`），新发布自然是新 URL，不依赖缓存刷新。
:::

## 四层和七层负载均衡怎么区分

### 四层负载均衡

四层 LB 工作在传输层，基于：

- IP
- 端口
- TCP/UDP 连接

它的特点：

- **性能高**：只做连接转发，不解析应用层内容
- **延迟低**：接近硬件级别的包转发
- **不了解 HTTP 语义**：无法按 URL、Header、Cookie 做路由
- 典型实现：LVS、云厂商的 NLB/CLB（TCP 模式）、F5 硬件

### 七层负载均衡

七层 LB 工作在应用层，可以基于：

- Host（虚拟主机）
- Path（URL 路径）
- Header
- Cookie
- Method
- Query 参数

它的特点：

- **更灵活**：可以做细粒度的流量治理
- **适合做业务路由**：灰度发布、A/B 测试、蓝绿部署
- **可以做 TLS 终止**：在 LB 层解密 HTTPS，后端只走 HTTP
- **有一定性能开销**：需要解析完整的 HTTP 请求
- 典型实现：Nginx、HAProxy、Envoy、Traefik、云厂商的 ALB

### 四层与七层常见搭配

生产环境经常是**四层在前、七层在后**：

```
用户 → 四层 LB（NLB / LVS）→ 七层 LB（Nginx / Envoy）→ 应用实例
```

四层负责高性能的连接分发，七层负责根据业务规则精细路由。

## 负载均衡调度算法

| 算法 | 原理 | 适合场景 | 注意点 |
| --- | --- | --- | --- |
| 轮询（Round Robin） | 请求依次分配给各节点 | 节点性能一致的场景 | 不考虑节点负载差异 |
| 加权轮询（Weighted RR） | 按权重分配，高权重节点分到更多请求 | 节点性能不均 | 权重需要手动维护 |
| 最少连接（Least Connections） | 把请求发给当前活跃连接数最少的节点 | 请求处理时间差异大 | 需要实时维护连接计数 |
| 加权最少连接 | 结合权重和活跃连接数 | 性能不均且负载波动大 | 配置和调优更复杂 |
| IP Hash | 按客户端 IP 哈希，同一 IP 固定到同一节点 | 需要简单会话粘性 | 节点增减会导致大量重映射 |
| 一致性哈希（Consistent Hash） | 哈希环映射，节点增减只影响相邻区间 | 缓存场景、需要最小化重映射 | 虚拟节点数影响均衡性 |

## 健康检查与会话保持

### 健康检查

负载均衡需要持续检测后端节点是否正常，常见方式：

| 方式 | 检测内容 | 典型配置 |
| --- | --- | --- |
| TCP 探测 | 能否建立 TCP 连接 | 每 5s 检测一次，连续 3 次失败摘除 |
| HTTP 探测 | 访问健康检查接口（如 `/health`），检查状态码 | 期望 200，超时 2s |
| 主动上报 | 应用主动向注册中心报告健康状态 | Consul、Eureka 心跳模式 |

关键参数：检查间隔、超时时间、不健康阈值（连续几次失败摘除）、恢复阈值（连续几次成功恢复）。

::: warning 健康检查通过 ≠ 业务正常
健康接口返回 200 只代表进程活着，不代表它真的能处理业务请求。比如数据库连接池耗尽时进程仍然活着，但所有业务请求都会超时。更深度的健康检查应当探测关键依赖。
:::

### 会话保持（Session Affinity / Sticky Session）

某些场景需要保证同一用户的请求始终落到同一节点：

- **基于 Cookie**：LB 在响应里插入一个标识后端节点的 Cookie，后续请求按此 Cookie 路由
- **基于 IP**：按客户端 IP Hash 固定节点（即 IP Hash 算法）
- **基于 Header**：按自定义 Header（如 `X-User-ID`）路由

需要注意：会话保持会导致负载不均——如果一个"粘"在某节点的用户产生了大量请求，这个节点的负载会显著高于其他节点。

## DNS、CDN、LB 怎么配合

一条典型的访问链路：

```
用户 → DNS 解析（可能经过多级缓存 → 递归 → 权威）
     → 拿到 IP（可能是 CDN 边缘节点、也可能是 LB 入口）
     → CDN 判断本地是否命中缓存
        → 命中：直接返回
        → 未命中：回源到 LB 入口 或 Origin Shield
     → LB（四层 → 七层）根据策略转发到具体应用实例
     → 应用处理请求，返回响应
     → 响应沿原路返回，CDN 按策略缓存一份
```

这条链路上的慢点定位：

| 慢在哪 | 现象 | 排查方向 |
| --- | --- | --- |
| DNS | 首次访问明显慢，后续正常 | `dig` 看解析耗时、TTL、是否有多次 CNAME 跳转 |
| CDN 未命中 | 特定资源慢，其他正常 | 看 CDN 响应头 `X-Cache: MISS`，检查缓存策略 |
| CDN → 源站链路 | 回源请求延迟高 | 对比边缘到源站的 RTT，检查是否需要 Origin Shield |
| LB | 部分请求偶发超时 | 检查健康检查配置，是否有异常节点未被摘除 |
| 应用 | 所有链路都慢 | 问题在后端，不在入口层 |

::: tip 怎么判断慢在入口层还是应用层
最简单的办法：直接绕过 CDN 和 LB，用 `curl` 直连应用实例。如果直连快、经过入口层才慢，问题在入口；如果直连也慢，问题在应用。
:::

## 常见问题与误区

### 1. 域名能解析不代表服务一定可用

DNS 正常只代表你拿到了一个 IP，并不代表：

- 这个 IP 对应的服务在监听正确端口
- 后端实例是健康的
- TLS 证书配置正确
- 防火墙/安全组允许访问

### 2. CDN 命中率高不代表整体链路一定快

如果：

- 动态接口仍然频繁回源，而你的核心性能瓶颈就在 API 响应
- 某些边缘节点的 TLS 证书异常，导致部分用户看到证书错误
- 源站到 CDN 的回源链路不稳或距离太远

用户依然会觉得慢或出错。

### 3. 负载均衡均匀不代表业务一定稳定

即使流量"看起来平均"，系统也可能不稳：

- 某些节点因 GC、磁盘 IO 等原因实际处理能力下降
- 某类请求特别重（如导出报表），集中在一个节点会拖慢所有请求
- 会话粘性把大客户的高频请求全部"粘"在同一个节点

### 4. CDN 刷新了但用户还是看到旧内容

可能原因：

- CDN 多级缓存没有全部刷到
- 浏览器本地缓存未过期（`Cache-Control` 或 `Expires` 设得太长）
- Service Worker 拦截了请求并返回了旧缓存
- DNS 还在解析到旧的 CDN 节点

### 5. 把 DNS 和 CDN 搞混

常见的混淆：

- DNS 负责的是**域名到 IP 的翻译**
- CDN 负责的是**内容的缓存和加速分发**
- 它们之间的关系是：CDN 通常通过 DNS（CNAME）来接入，DNS 是 CDN 流量调度的入口
