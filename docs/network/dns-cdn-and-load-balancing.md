## DNS 在解决什么问题

DNS 本质上解决的是：**人用域名，机器用 IP，需要一套分布式系统把域名翻译成 IP**。

除了最基础的域名到 IP 映射之外，DNS 还承担着：

- 按地域、运营商返回不同 IP（智能解析 / GeoDNS）
- 通过 CNAME 把流量导向 CDN 或云负载均衡
- 做灾备切换——故障时改解析指向备用集群
- 做服务发现——内部 DNS 或 Consul DNS 接口

### 你必须分清的角色

| 角色 | 做什么 | 举例 |
| --- | --- | --- |
| 本地缓存 | 浏览器、操作系统各自缓存解析结果，避免重复查询 | Chrome DNS cache、`/etc/hosts` |
| 递归 DNS | 替你完成"从根到权威"的完整查询，返回最终结果 | `8.8.8.8`、运营商 DNS、公司内网 DNS |
| 权威 DNS | 真正存储域名记录的服务器，对自己负责的区域给出权威回答 | Cloudflare DNS、阿里云云解析 |
| 根 DNS / TLD DNS | 递归查询的起点，根服务器指向 `.com`、`.cn` 等顶级域 | 全球 13 组根服务器 |

### 常见记录类型

| 类型 | 含义 | 典型场景 |
| --- | --- | --- |
| `A` | 域名 → IPv4 地址 | 最基础的解析 |
| `AAAA` | 域名 → IPv6 地址 | 双栈环境 |
| `CNAME` | 域名 → 另一个域名（别名） | 接入 CDN 时最常用，如 `www.example.com → example.cdn.provider.net` |
| `MX` | 邮件交换记录 | 配置企业邮箱 |
| `TXT` | 文本记录 | SPF 反垃圾邮件验证、域名所有权验证、DKIM 签名 |
| `NS` | 指定域名由哪组权威 DNS 管理 | 域名托管迁移 |
| `SRV` | 指定服务的主机和端口 | 微服务注册与发现 |

::: tip CNAME 为什么不能和 A 记录共存在同一个域名上
DNS 标准规定 CNAME 是"别名"，意味着该域名的所有查询都应该先去追 CNAME 指向的目标。如果同时有 A 记录，解析器不知道该用哪个结果。所以裸域（`example.com`）通常不能直接 CNAME 到 CDN，只能用 A 记录或部分服务商提供的 ALIAS/ANAME 记录。
:::

## DNS 解析过程

一个域名从浏览器到最终拿到 IP，经过的查询链条是：

1. **浏览器缓存**：最先查，命中直接用
2. **操作系统缓存**：包括 `/etc/hosts` 和系统 DNS 缓存
3. **递归 DNS 缓存**：递归解析器（如 ISP DNS、`8.8.8.8`）自身的缓存
4. **递归查询**：缓存都没命中，递归 DNS 从根开始逐级查询
   - 根 DNS → `.com` TLD → `example.com` 的权威 DNS → 拿到 A 记录
5. **结果沿链路返回并缓存**：每一级都会按 TTL 缓存结果

### 递归查询 vs 迭代查询

- **递归查询**：客户端把问题甩给递归 DNS，递归 DNS 负责查到底并返回最终答案
- **迭代查询**：递归 DNS 向根 / TLD / 权威逐级询问时，每一级只回答"你该去问谁"，而不是直接给最终答案

工程上你只需记住：**用户到递归 DNS 是递归，递归 DNS 到各级权威是迭代**。

### 智能解析（GeoDNS）

很多生产环境的权威 DNS 会根据请求来源返回不同的 IP：

- 按地域（华东用户解析到上海机房，华南用户解析到广州机房）
- 按运营商（电信用户解析到电信线路 IP，联通用户解析到联通线路 IP）
- 按权重（灰度发布时 90% 流量解析到旧集群，10% 到新集群）

这就是为什么同一个域名在不同地方 `dig` 可能拿到不同 IP——并不是 DNS "出了问题"。

### DNS 安全

需要知道的几个风险点：

- **DNS 劫持**：中间人篡改解析结果，把用户导向钓鱼站。运营商劫持广告注入也属此类
- **DNS 污染**：权威或递归层面返回错误 IP，导致某些域名在特定网络环境下无法正确解析
- **DNSSEC**：通过数字签名保证解析结果的真实性和完整性，防止篡改，但部署率仍不高
- **DoH / DoT**：DNS over HTTPS / DNS over TLS，加密 DNS 查询过程，防止窃听和中间人篡改

## 为什么 TTL 很重要

TTL（Time To Live）决定 DNS 解析结果能被缓存多久。它不只是一个"配置项"，而是直接影响线上运维操作的核心参数。

### TTL 影响什么

| 场景 | TTL 太长（如 86400s / 24h） | TTL 太短（如 30s） |
| --- | --- | --- |
| 发布新 IP | 最多要等 24 小时才能全网生效 | 几十秒内生效 |
| 故障切流 | 大量用户仍然解析到故障 IP | 快速切到备用 IP |
| DNS 查询压力 | 缓存命中率高，查询量小 | 递归 DNS 查询量大幅增加 |
| 对上游依赖 | 低 | 高——上游 DNS 抖动直接影响你 |

### 常见 TTL 策略

- **常规业务**：300s ~ 600s（5 ~ 10 分钟），平衡生效速度和缓存效率
- **故障预案前**：提前把 TTL 降到 60s，确保切流操作能快速生效
- **稳定期**：可以适当拉长到 3600s（1 小时）甚至更长
- **CDN 接入后**：CNAME 记录的 TTL 通常可以设得较长，因为流量调度由 CDN 内部 DNS 完成

::: warning TTL 不等于"精确到秒生效"
各级缓存的 TTL 计时不一定同步开始，且部分递归 DNS 实现可能不严格遵循 TTL。所以"把 TTL 设成 60s 就一定 60s 内生效"是理想情况，实际可能有几十秒到几分钟的偏差。
:::

## CDN 在解决什么问题

CDN（Content Delivery Network）的核心价值是三件事：

1. **就近访问，降低延迟**——用户请求被导向距离最近的边缘节点
2. **边缘缓存，减轻源站压力**——静态资源在边缘命中，不回源
3. **安全和访问控制**——部分 CDN 在边缘集成了 WAF、DDoS 防护、IP 黑名单

### 静态加速 vs 动态加速

- **静态加速**：缓存 JS、CSS、图片、字体等不常变化的资源。这是 CDN 最经典的用法
- **动态加速（DSA / DCDN）**：对 API 接口等无法缓存的请求，CDN 通过优化传输路径（选择更快的骨干网、减少中间跳数）来降低延迟，而不是缓存内容

一个非常常见的误区：**以为用了 CDN 就一定更快**。如果缓存策略不合理、回源链路不稳、边缘节点质量差，CDN 不仅不快，反而因为多了一跳而更慢。

## CDN 的工作机制

### CDN 如何通过 DNS 接入

最常见的接入方式是 CNAME：

1. 你在权威 DNS 把 `static.example.com` CNAME 到 CDN 提供的域名，如 `example.cdn.provider.net`
2. 用户解析 `static.example.com` 时，DNS 会跟着 CNAME 链最终到达 CDN 的调度 DNS
3. CDN 的调度 DNS 根据用户的地理位置、运营商、节点负载等信息，返回最优边缘节点的 IP
4. 用户直接访问该边缘节点

### 智能 DNS 路由（GSLB）

CDN 的流量调度本质上是一次 **DNS 重定向**。当用户访问 `cdn.example.com` 时，通常会经历这条链路：

1. 用户的 Local DNS / 递归 DNS 向 CDN 的 GSLB 发起查询
2. GSLB 根据 LDNS 的出口 IP 判断用户大致位置
3. 如果接入了 `EDNS-Client-Subnet`（ECS），还能拿到更接近真实用户的客户端网段信息
4. 调度系统综合多种策略，返回当前最优接入点

典型的调度因素包括：

- **地理位置**：尽量让用户接入物理距离最近的 PoP
- **运营商 / ISP**：移动、电信、联通等不同网络通常需要不同接入策略
- **节点健康状态**：故障 PoP 需要立刻摘除，避免继续返回异常 IP
- **节点实时负载**：热门节点过载时，需要把新流量分流到相邻区域

所以从工程上看，**DNS 不只是把域名翻译成 IP，它还是 CDN 全局负载均衡（GSLB）的入口层**。

### 缓存层次

典型的 CDN 不只有一层缓存：

| 层级 | 说明 |
| --- | --- |
| L1 边缘节点 | 离用户最近的 POP 点，数量最多、覆盖最广 |
| L2 中间层 / 区域中心 | 当 L1 未命中时，先向 L2 请求而不是直接回源 |
| Origin Shield（源站盾） | 源站前面的最后一级缓存，所有区域的未命中请求先汇聚到这里，大幅减少源站压力 |
| 源站 | 你的实际服务器 |

这种分层的意义在于：**避免大量边缘节点同时回源冲垮你的源站**。

### 三层物理拓扑

很多 CDN 的物理架构可以抽象成一个金字塔式的三层拓扑：

<InlineSvg src="/images/network/cdn-three-layer-topology.svg" alt="CDN 三层物理拓扑图" />

| 层级 | 主要职责 | 工程特点 |
| --- | --- | --- |
| 中心站（Center Node） | 对接源站，做全局控制与最终汇总 | 通常承载 GSLB 控制中心、日志汇聚、统计分析、回源兜底 |
| 区域中心（Regional Node） | 作为边缘和源站之间的缓冲层 | 边缘未命中后优先访问区域中心，减少源站被直接打穿 |
| 边缘节点（Edge Node / PoP） | 直接面向用户提供访问 | 部署最广，负责缓存命中、就近接入、首跳加速 |

边缘节点内部一般又会包含几类核心组件：

- **Cache 模块**：常见实现基于 Nginx、Varnish、Squid 等深度定制，负责存储静态资源
- **L4 / L7 接入层**：负责接收用户请求、做连接分发和协议处理；极致性能场景会引入 DPDK 等技术优化收包和转发
- **本地存储层**：通常采用内存 + SSD 的分层缓存，兼顾热点数据访问速度和容量

把计算和存储不断往边缘推，目标只有一个：**让大多数请求在离用户最近的一跳内结束，而不是一路打到中心或源站**。

### CDN 缓存机制

CDN 缓存不是简单地“把文件存一份”，它至少要回答 4 个问题：

1. **拿什么当缓存键**
2. **把内容存到哪里**
3. **什么时候算过期**
4. **缓存满了先淘汰谁**

<InlineSvg src="/images/network/browser-cdn-origin-cache-compare.svg" alt="浏览器缓存、CDN 缓存与源站缓存对比图" />

#### 1. 缓存键（Cache Key）

CDN 必须先决定“两个请求是不是同一份内容”。这个判断依据就是缓存键。

最基础的缓存键通常包含：

- 协议：`HTTP` / `HTTPS`
- Host：如 `static.example.com`
- Path：如 `/assets/app.js`
- Query String：如 `?v=123`

在更复杂的场景里，缓存键还可能受这些因素影响：

- `Vary: Accept-Encoding`：区分 `gzip`、`br` 等压缩版本
- 特定 Query 参数白名单：只让 `version`、`lang` 参与缓存键，忽略 `utm_source` 这类营销参数
- 特定 Header 或 Cookie：例如按语言、设备类型拆分缓存

这里最容易踩的坑是：**把不该进缓存键的动态参数也放进去了**。这样会把同一资源切成大量碎片，导致命中率急剧下降。

#### 2. 存储介质

CDN 的缓存通常不是单一存储，而是分层的：

| 存储层 | 特点 | 适合存什么 |
| --- | --- | --- |
| 内存缓存 | 访问最快，但容量小、成本高 | 超热点小对象，如首页片段、小图标、热门 JS |
| SSD 缓存 | 容量更大，性能稳定 | 大多数静态资源，如图片、脚本、样式、音视频分片 |
| 上级缓存 / 区域中心 | 本地未命中的补充层 | 跨边缘共享的热点资源 |

所以“缓存命中”内部还有细分：

- **Memory Hit**：内存命中，速度最快
- **Disk Hit**：磁盘命中，仍然远快于回源
- **Parent Hit**：本地未命中，但命中上级缓存

#### 3. 过期、协商缓存与回源校验

CDN 不只是判断“有没有这份文件”，还要判断“这份文件现在还能不能继续用”。

常见判断方式包括：

- **强缓存**：根据 `Cache-Control: max-age`、`s-maxage`、`Expires` 判断是否仍可直接使用
- **协商缓存**：缓存过期后，带着 `If-None-Match` / `If-Modified-Since` 去源站验证
- **源站返回 `304 Not Modified`**：说明文件没变，CDN 续租本地缓存，不必重新下载整个对象
- **源站返回新内容**：说明对象已变更，CDN 用新响应覆盖旧缓存

这意味着“过期”不一定等于“必须重新拉完整文件”。很多时候只是做一次轻量校验。

#### 4. 淘汰策略

边缘节点容量是有限的，缓存放满后必须淘汰一部分对象。常见思路包括：

- **LRU**（Least Recently Used）：淘汰最近最少使用的对象
- **LFU**（Least Frequently Used）：淘汰访问频次最低的对象
- **分层淘汰**：优先保留高热、小体积、高回源成本的对象

工程上通常不会只看“最近访问没访问过”，还会综合：

- 对象热度
- 文件大小
- 回源成本
- 是否为大文件或冷门长尾资源

所以 CDN 的目标不是“什么都缓存”，而是**用有限空间换取最高命中收益**。

#### 5. 哪些内容不该缓存

不是所有响应都适合进入 CDN 缓存，典型的不缓存对象包括：

- 带用户身份的私有内容
- 强实时接口，如余额、库存、订单状态
- 明确声明 `Cache-Control: no-store` 或 `private` 的响应
- 会因 Cookie、Authorization 强烈变化的动态接口

否则最危险的问题不是“命中率低”，而是**把 A 用户的数据缓存给了 B 用户**。

#### 6. 缓存穿透、击穿、雪崩

CDN 也会遇到和业务缓存类似的问题：

| 问题 | 现象 | 常见治理方式 |
| --- | --- | --- |
| 缓存穿透 | 请求的对象根本不存在，大量请求持续打到源站 | 404 短缓存、恶意流量拦截、边缘限速 |
| 缓存击穿 | 单个热点对象刚过期，瞬时大量请求同时回源 | 合并回源、预热、热点对象提前续期 |
| 缓存雪崩 | 大量对象在同一时间批量过期，造成回源洪峰 | 打散 TTL、分批刷新、加上级缓存和源站保护 |

所以 CDN 缓存机制的核心不是“存没存”，而是**围绕命中率、回源成本和一致性做持续权衡**。

### 缓存命中、回源与失效

- **缓存命中**：请求到达边缘节点，节点本地有对应资源且未过期，直接返回
- **回源**（Cache Miss）：边缘没有缓存或缓存已过期，需要向上层或源站请求
- **缓存失效 / 刷新**：主动让某些缓存立即过期。通常通过 CDN 控制台提交 URL/目录刷新
- **预热**：提前把热点资源推到边缘节点，避免上线瞬间大量回源

<InlineSvg src="/images/network/cdn-hit-miss-flow.svg" alt="CDN 命中、未命中与回源流程图" />

如果把一次内容请求拆开看，它通常是这样流转的：

1. 用户请求先到边缘节点
2. 如果本地命中缓存，边缘直接返回 `HTTP 200`
3. 如果未命中，边缘先向区域中心或 Origin Shield 请求
4. 上层仍未命中时，才真正回源到源站
5. 资源返回后写入边缘的内存或 SSD，并根据缓存策略决定生命周期

这里有一个很关键的机制：**合并回源（Request Collapsing）**。

在高并发下，如果大量用户同时请求同一个刚过期的热点资源，CDN 不会让每个请求都独立回源，而是通常只放行第一个请求去取数据，其余相同请求先挂起，等首个请求拿到结果后复用同一份响应返回。这样可以显著降低源站在瞬时热点下被打爆的风险。

### CDN 与缓存控制

CDN 节点遵循的缓存策略通常由这些 HTTP 头控制：

- `Cache-Control`：`max-age=3600` 表示资源可被缓存 1 小时
- `s-maxage`：专门给共享缓存（CDN、代理）用的 max-age，优先级高于 `max-age`
- `Expires`：绝对过期时间，优先级低于 `Cache-Control`
- `Vary`：告诉 CDN 按某些 Header（如 `Accept-Encoding`）区分缓存版本

<InlineSvg src="/images/network/cache-control-header-map.svg" alt="CDN 缓存控制头关系图" />

::: warning 发版后用户还是看到旧资源？
多半是缓存未正确失效。解决思路：构建时给文件名加 content hash（如 `app.3a7c9f.js`），新发布自然是新 URL，不依赖缓存刷新。
:::

### 动态加速、边缘计算与一致性

很多人把 CDN 理解成“静态文件缓存”，这只覆盖了它的一部分能力。现代 CDN 在工程上还会处理三类更难的问题。

<InlineSvg src="/images/network/cdn-capability-compare.svg" alt="静态加速、动态加速与边缘计算对比图" />

#### 1. 动态加速（Dynamic Acceleration）

对 API、登录、下单这类**不可缓存**的请求，CDN 主要做的是链路优化而不是内容缓存：

- **路由优化**：实时探测跨网链路质量，选择丢包率更低、时延更小的路径，也就是常说的 Overlay Network
- **协议优化**：边缘到源站之间复用长连接，或使用 QUIC、HTTP/2 等方式减少 TCP 和 TLS 握手成本
- **内容压缩**：对文本响应启用 Brotli、Zstd 等压缩算法，减少跨网传输体积

这类能力本质上是在优化“边缘到源站”这段链路，而不是提高缓存命中率。

#### 2. 边缘计算（Edge Computing）

现代 CDN 往往允许在边缘节点直接执行轻量逻辑，比如 Cloudflare Workers、CloudFront Functions、基于 Wasm 的边缘运行时。

常见用途包括：

- **A/B Testing**：在边缘根据用户特征直接决定走哪个版本
- **鉴权**：在边缘校验 JWT、签名或时间戳，请求不合法就地拦截
- **请求改写**：补充 Header、改写 Path、做地区化路由

这样做的核心价值是：**把原本要进入核心应用层才能处理的逻辑，尽量前移到离用户更近的位置**。

#### 3. 一致性与刷新

CDN 带来的最大工程挑战之一不是“怎么缓存”，而是“怎么让缓存一致地失效”。

常见方案有两类：

- **主动失效（Invalidation / Purge）**：源站内容更新后，通过 CDN API 主动刷新指定 URL、目录或缓存标签
- **版本化发布**：给静态资源文件名加 Hash，如 `app.v123.js` 或 `app.3a7c9f.js`，通过新 URL 天然绕开旧缓存

实践里通常优先推荐版本化发布，因为它比“全网等刷新完成”更可控，也更不容易受到多级缓存和浏览器缓存的干扰。

## 四层和七层负载均衡怎么区分

最实用的区分方式不是死记“第几层”，而是看它**能看到什么信息、能做什么决策、代价是什么**。

| 对比项 | 四层负载均衡（L4） | 七层负载均衡（L7） |
| --- | --- | --- |
| 工作层次 | 传输层 | 应用层 |
| 主要依据 | IP、端口、TCP/UDP 连接 | Host、Path、Header、Cookie、Method、Query 参数 |
| 是否理解 HTTP 语义 | 不理解 | 理解 |
| 转发粒度 | 连接级转发 | 请求级转发 |
| 性能特点 | 性能高、延迟低，接近纯连接转发 | 更灵活，但需要解析应用层协议，开销更高 |
| 典型能力 | 高性能流量分发、TCP/UDP 转发 | 灰度发布、A/B 测试、蓝绿部署、鉴权、请求改写、TLS 终止 |
| 适合场景 | 大流量入口、非 HTTP 协议、对性能敏感的基础转发 | Web/API 网关、按业务规则路由、需要精细化流量治理 |
| 局限 | 无法按 URL、Header、Cookie 做路由 | 性能通常不如 L4，配置和治理复杂度更高 |
| 典型实现 | LVS、云厂商 NLB / CLB（TCP 模式）、F5 硬件 | Nginx、HAProxy、Envoy、Traefik、云厂商 ALB |

一句话记忆：

- **L4 负责把连接高效地送进去**
- **L7 负责把请求按业务规则分清楚送到谁**

### 四层与七层常见搭配

生产环境经常是**四层在前、七层在后**：

```
用户 → 四层 LB（NLB / LVS）→ 七层 LB（Nginx / Envoy）→ 应用实例
```

四层负责高性能的连接分发，七层负责根据业务规则精细路由。

## 负载均衡调度算法

| 算法 | 原理 | 适合场景 | 注意点 |
| --- | --- | --- | --- |
| 轮询（Round Robin） | 请求依次分配给各节点 | 节点性能一致的场景 | 不考虑节点负载差异 |
| 加权轮询（Weighted RR） | 按权重分配，高权重节点分到更多请求 | 节点性能不均 | 权重需要手动维护 |
| 最少连接（Least Connections） | 把请求发给当前活跃连接数最少的节点 | 请求处理时间差异大 | 需要实时维护连接计数 |
| 加权最少连接 | 结合权重和活跃连接数 | 性能不均且负载波动大 | 配置和调优更复杂 |
| IP Hash | 按客户端 IP 哈希，同一 IP 固定到同一节点 | 需要简单会话粘性 | 节点增减会导致大量重映射 |
| 一致性哈希（Consistent Hash） | 哈希环映射，节点增减只影响相邻区间 | 缓存场景、需要最小化重映射 | 虚拟节点数影响均衡性 |

## 健康检查与会话保持

### 健康检查

负载均衡需要持续检测后端节点是否正常，常见方式：

| 方式 | 检测内容 | 典型配置 |
| --- | --- | --- |
| TCP 探测 | 能否建立 TCP 连接 | 每 5s 检测一次，连续 3 次失败摘除 |
| HTTP 探测 | 访问健康检查接口（如 `/health`），检查状态码 | 期望 200，超时 2s |
| 主动上报 | 应用主动向注册中心报告健康状态 | Consul、Eureka 心跳模式 |

关键参数：检查间隔、超时时间、不健康阈值（连续几次失败摘除）、恢复阈值（连续几次成功恢复）。

::: warning 健康检查通过 ≠ 业务正常
健康接口返回 200 只代表进程活着，不代表它真的能处理业务请求。比如数据库连接池耗尽时进程仍然活着，但所有业务请求都会超时。更深度的健康检查应当探测关键依赖。
:::

### 会话保持（Session Affinity / Sticky Session）

某些场景需要保证同一用户的请求始终落到同一节点：

- **基于 Cookie**：LB 在响应里插入一个标识后端节点的 Cookie，后续请求按此 Cookie 路由
- **基于 IP**：按客户端 IP Hash 固定节点（即 IP Hash 算法）
- **基于 Header**：按自定义 Header（如 `X-User-ID`）路由

需要注意：会话保持会导致负载不均——如果一个"粘"在某节点的用户产生了大量请求，这个节点的负载会显著高于其他节点。

## DNS、CDN、LB 怎么配合

一条典型的访问链路：

```
用户 → DNS 解析（可能经过多级缓存 → 递归 → 权威）
     → 拿到 IP（可能是 CDN 边缘节点、也可能是 LB 入口）
     → CDN 判断本地是否命中缓存
        → 命中：直接返回
        → 未命中：回源到 LB 入口 或 Origin Shield
     → LB（四层 → 七层）根据策略转发到具体应用实例
     → 应用处理请求，返回响应
     → 响应沿原路返回，CDN 按策略缓存一份
```

这条链路上的慢点定位：

| 慢在哪 | 现象 | 排查方向 |
| --- | --- | --- |
| DNS | 首次访问明显慢，后续正常 | `dig` 看解析耗时、TTL、是否有多次 CNAME 跳转 |
| CDN 未命中 | 特定资源慢，其他正常 | 看 CDN 响应头 `X-Cache: MISS`，检查缓存策略 |
| CDN → 源站链路 | 回源请求延迟高 | 对比边缘到源站的 RTT，检查是否需要 Origin Shield |
| LB | 部分请求偶发超时 | 检查健康检查配置，是否有异常节点未被摘除 |
| 应用 | 所有链路都慢 | 问题在后端，不在入口层 |

<InlineSvg src="/images/network/cdn-troubleshooting-signals.svg" alt="CDN 线上排障观察点速查图" />

::: tip 怎么判断慢在入口层还是应用层
最简单的办法：直接绕过 CDN 和 LB，用 `curl` 直连应用实例。如果直连快、经过入口层才慢，问题在入口；如果直连也慢，问题在应用。
:::

## 常见问题与误区

### 1. 域名能解析不代表服务一定可用

DNS 正常只代表你拿到了一个 IP，并不代表：

- 这个 IP 对应的服务在监听正确端口
- 后端实例是健康的
- TLS 证书配置正确
- 防火墙/安全组允许访问

### 2. CDN 命中率高不代表整体链路一定快

如果：

- 动态接口仍然频繁回源，而你的核心性能瓶颈就在 API 响应
- 某些边缘节点的 TLS 证书异常，导致部分用户看到证书错误
- 源站到 CDN 的回源链路不稳或距离太远

用户依然会觉得慢或出错。

### 3. 负载均衡均匀不代表业务一定稳定

即使流量"看起来平均"，系统也可能不稳：

- 某些节点因 GC、磁盘 IO 等原因实际处理能力下降
- 某类请求特别重（如导出报表），集中在一个节点会拖慢所有请求
- 会话粘性把大客户的高频请求全部"粘"在同一个节点

### 4. CDN 刷新了但用户还是看到旧内容

可能原因：

- CDN 多级缓存没有全部刷到
- 浏览器本地缓存未过期（`Cache-Control` 或 `Expires` 设得太长）
- Service Worker 拦截了请求并返回了旧缓存
- DNS 还在解析到旧的 CDN 节点

### 5. 把 DNS 和 CDN 搞混

常见的混淆：

- DNS 负责的是**域名到 IP 的翻译**
- CDN 负责的是**内容的缓存和加速分发**
- 它们之间的关系是：CDN 通常通过 DNS（CNAME）来接入，DNS 是 CDN 流量调度的入口
