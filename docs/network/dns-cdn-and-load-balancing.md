## DNS 在解决什么问题

DNS 本质上解决的是：

- 域名如何映射到 IP
- 解析结果如何缓存
- 解析记录如何按策略控制

你必须分清的角色有：

- 本地缓存 / 浏览器缓存
- 系统解析器
- 递归 DNS
- 权威 DNS

你也需要认识常见记录：

- `A`：域名到 IPv4
- `AAAA`：域名到 IPv6
- `CNAME`：别名
- `MX`：邮件交换
- `TXT`：文本记录，常用于验证和策略配置

## 为什么 TTL 很重要

TTL 决定的是：

- 解析结果能被缓存多久

所以 TTL 会直接影响：

- 新发布 IP 多久生效
- 故障切流多久生效
- 访问是否持续打到旧节点

TTL 太长的问题：

- 切流慢
- 故障恢复慢

TTL 太短的问题：

- 递归查询压力大
- 稳定性更依赖上游 DNS 服务质量

## CDN 在解决什么问题

CDN 的核心价值通常有三类：

1. 就近访问，降低延迟
2. 边缘缓存，减轻源站压力
3. 某些情况下在边缘做安全和访问控制

CDN 最关键的几个概念：

- 缓存命中
- 回源
- 缓存失效
- 热点资源预热
- 静态加速与动态加速的边界

一个非常常见的误区是：

- 以为用了 CDN 就一定更快

真实情况是：

- 如果缓存策略不合理、回源链路不稳、边缘节点质量差，CDN 也可能成为问题点

## 四层和七层负载均衡怎么区分

### 四层负载均衡

四层 LB 基于：

- IP
- 端口
- 连接

它的特点：

- 性能高
- 转发更接近网络层 / 传输层
- 不了解 HTTP 的业务语义

### 七层负载均衡

七层 LB 可以基于：

- Host
- Path
- Header
- Cookie
- Method

它的特点：

- 更灵活
- 可以做更细粒度的流量治理
- 更适合 API 网关、灰度、A/B 测试和业务路由

## 一次请求怎么经过 DNSCDNLB 到应用

一条典型链路可能是：

1. 浏览器先做 DNS 解析
2. 域名解析到 CDN 或负载均衡入口
3. CDN 先判断本地是否命中缓存
4. 未命中则回源到源站或七层网关
5. 入口负载均衡再把请求转发到具体应用实例
6. 应用返回结果，经由同一路径回到客户端

这里的慢点可能出现在：

- DNS 解析慢
- CDN 节点缓存未命中导致大量回源
- LB 健康检查失真，把流量打到异常节点
- 七层代理规则写错，路由到不该到的服务

## 常见问题与误区

### 1. 域名能解析不代表服务一定可用

DNS 正常只代表：

- 你拿到了一个 IP

并不代表：

- 这个 IP 在监听正确端口
- 后端实例是健康的
- TLS 配置是正确的

### 2. CDN 命中率高不代表整体链路一定快

如果：

- 动态接口仍然频繁回源
- 某些节点证书异常
- 边缘到源站链路不稳

用户依然会觉得慢。

### 3. 负载均衡均匀不代表业务一定稳定

如果后端：

- 某些节点资源紧张
- 某类请求特别重
- 会话粘性配置不合理

即使流量“看起来平均”，系统也可能不稳。
