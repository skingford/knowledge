# 网络必备知识

## 适合谁看

- 想系统补齐网络基础，不再把 TCP、HTTP、DNS、CDN 当成零散名词来背的工程师
- 做后端、基础设施、云原生或系统设计时，发现很多性能和故障问题都绕不开网络的人
- 面试里经常被问三次握手、HTTPS、DNS、负载均衡，但回答容易只停在单点概念的人
- 已经知道 TCP、HTTP、HTTPS 是什么，但还没把它们串成一条访问链路的人

## 学习目标

- 建立一张从链路层到应用层的统一网络知识地图
- 讲清 IP、TCP、TLS、HTTP 分别负责什么，以及它们怎么串成一次完整请求
- 理解一次请求从输入域名到拿到响应，中间到底发生了什么
- 看到慢请求、重传、握手失败时，知道大致可能卡在哪里
- 知道网络排障时如何按层定位，而不是一上来就盲猜问题点

## 先建立一张网络全景图

<InlineSvg src="/network/network-layers.svg" alt="TCP/IP 四层网络模型" />

最值得先记住的不是单个协议，而是网络问题可以按层拆：

| 层次 | 你最该先掌握什么 | 典型问题 |
| --- | --- | --- |
| 物理 / 链路层 | 网卡、MAC、交换、ARP、MTU | 同网段通信异常、丢包、ARP 冲突 |
| 网络层 | IP、子网、路由、NAT、ICMP | 路由不通、跨网段访问失败、Traceroute 中断 |
| 传输层 | TCP、UDP、端口、连接状态 | 建连超时、RST、丢包重传、连接堆积 |
| 应用层 | HTTP、HTTPS、DNS、WebSocket、gRPC | 502/504、证书错误、域名解析异常、跨层代理问题 |

工程里最常见的误区是：

- 把所有问题都当成应用问题
- 或者一看到超时就只盯着代码

更稳的思路通常是：

1. 先确认问题发生在哪一层
2. 再看这一层最关键的状态、指标和日志
3. 最后再判断是否要往上层或下层继续追

::: tip OSI 七层和 TCP/IP 四层怎么讲
不必死背教科书映射，重点是知道每层大概负责什么。TCP/IP 四层在工程实践里更常用：链路层、网络层、传输层、应用层。回答时最好结合真实协议：IP、TCP、UDP、HTTP、DNS、TLS。
:::

## 一次请求到底经过哪些环节

<InlineSvg src="/network/request-lifecycle.svg" alt="一次 HTTPS 请求完整链路" />

假设浏览器访问 `https://example.com/api/orders`，大致会经过：

1. DNS 解析域名：先查本地缓存，没命中再去递归解析拿到目标 IP
2. 选定目标 IP，按路由规则决定走哪个网关
3. 发起 TCP 建连（三次握手）
4. 完成 TLS 握手：协商加密套件、校验证书、生成会话密钥
5. 发送 HTTP 请求
6. 请求经过 CDN / WAF / 负载均衡 / 网关，再到应用服务
7. 应用服务处理请求
8. 返回响应
9. 客户端决定是否复用连接、缓存内容或继续请求资源

所以很多"接口慢"其实可能卡在完全不同的位置：

- DNS 慢
- TCP 建连慢 / 重传多
- TLS 握手慢 / 失败
- HTTP 头过大被拒绝
- 网关转发慢 / 错误
- 上游服务处理慢
- 下游数据库或缓存慢

## IP 和路由在做什么

IP 层解决的核心问题是：

- 数据包往哪发
- 下一跳是谁
- 如何跨网段到达目标机器

必须掌握的几个关键词：

- 源 IP / 目标 IP
- 子网掩码
- 默认网关
- 路由表
- NAT
- TTL

一个很实用的理解是：

- TCP 更像"连接和传输规则"
- IP 更像"把包送到目的地的寻路系统"

所以如果根本路由不到，TCP 和 HTTP 再正确也没用。

## TCP 和 UDP 各自解决什么问题

可以先用一张表记住它们各自想解决的问题：

| 对比维度 | TCP | UDP |
| --- | --- | --- |
| 核心要解决的问题 | 在不可靠网络上提供面向连接、可靠、有序的字节流传输 | 提供尽量简单、低开销的报文发送能力，把可靠性和顺序控制留给上层 |
| 连接方式 | 面向连接，通信前先建连 | 无连接，直接发包 |
| 可靠性 | 保证到达，依赖 ACK、重传等机制 | 不保证到达 |
| 顺序性 | 保证按序交付 | 不保证顺序 |
| 控制能力 | 有流量控制、拥塞控制 | 没有内建流量控制和拥塞控制 |
| 数据形态 | 字节流，没有天然消息边界 | 报文传输，天然保留消息边界 |
| 开销与时延 | 头部和状态更多，实现更重 | 头部更轻，协议开销更低，通常更适合低时延场景 |
| 更适合的场景 | HTTP / HTTPS、数据库连接、RPC 调用、大多数要求正确性和顺序性的服务端通信 | 音视频实时传输、DNS 查询、某些游戏场景、可容忍少量丢包但更关注实时性的场景 |

一个常见理解方式是 TCP 重"正确"，UDP 重"快"。但不要简单理解成"UDP 一定更快"：如果上层还要自己做重传、排序、补偿，整体复杂度可能更高。

## TCP 连接管理

<InlineSvg src="/network/tcp-handshake.svg" alt="TCP 三次握手与四次挥手" />

### 三次握手

三次握手最关键的不是记住 `SYN -> SYN+ACK -> ACK`，而是知道它确认了什么：

1. 客户端确认自己能发
2. 服务端确认自己能收，也能发
3. 客户端再确认服务端的发送能力和自己的接收能力都没问题

> 三次握手的本质，是双方互相确认"发送能力和接收能力都正常"，并同步初始化序列号。如果少一次，就无法可靠确认双向通信状态；旧连接延迟包或伪造连接也更难处理。

#### 半连接队列和全连接队列

线上建连问题经常会追到：

- `SYN backlog`（半连接队列）：存放尚未完成三次握手的连接
- `accept queue`（全连接队列）：存放已经建立但应用尚未 `accept` 的连接

当这两处堆积异常时，常见表现就是建连超时。所以"三次握手"不能只背协议包，还要知道它和服务端监听队列、连接积压、SYN Flood 防护之间的关系。

### 四次挥手

因为 TCP 是全双工：

- 我不发了，不代表我也不收了
- 对方收到 FIN 后，可能还有数据没发完

所以关闭连接通常是两边分别关闭各自的发送方向，拆成四次交互。

### TIME_WAIT

TIME_WAIT 的价值主要是两件事：

- 防止旧连接的延迟包污染新连接
- 确保最后的 ACK 如果丢了，对方还有机会重发 FIN

线上常见问题不是"为什么有 TIME_WAIT"，而是：

- 为什么 TIME_WAIT 太多
- 为什么短连接风暴导致端口耗尽
- 为什么连接没有复用好

## TCP 可靠传输

<InlineSvg src="/network/tcp-reliable-transmission.svg" alt="TCP 可靠传输机制" />

TCP 的可靠，不是靠"永远不丢"，而是靠一组机制把不可靠网络尽量收敛成可靠连接：

- **序列号**：保证字节流有顺序
- **ACK 确认**：对方收到什么、没收到什么要能反馈
- **超时重传**：超时没确认就重发
- **滑动窗口**：控制未确认数据量，避免把对方压垮
- **流量控制**：接收方通过窗口告诉发送方"我还能吃多少"
- **拥塞控制**：根据网络拥堵情况调整发送速度

### 滑动窗口

最简单的记法是：

- 发送方不能无限发
- 接收方也不能无限吃
- 网络中未确认的数据量需要被控制在一个窗口里

因此窗口大小其实同时反映：接收方处理能力和当前网络连接状态。

### 拥塞控制

最少应知道四个阶段：

- **慢启动**：从小窗口开始指数增长
- **拥塞避免**：到阈值后改为线性增长
- **快重传**：收到三个重复 ACK 立即重传
- **快恢复**：不回到慢启动，直接进入拥塞避免

不要求每次都把算法细节背全，但要知道它们在解决：网络拥堵时不能继续无节制地打满。

## HTTP 版本演进

<InlineSvg src="/network/http-evolution.svg" alt="HTTP 版本演进对比" />

### HTTP/1.1

- Keep-Alive 连接复用
- 文本协议，易调试
- 同一连接上的并发能力有限，容易有队头阻塞

### HTTP/2

引入二进制分帧、Header 压缩、多路复用和 Stream 优先级。主要解决 HTTP/1.1 下多个请求并发效率差的问题。

::: warning 为什么 HTTP/2 多路复用了还会卡
HTTP/2 的多路复用发生在应用层，底层仍然是同一条 TCP 连接。一旦 TCP 层丢包，整个连接上的数据推进都会受到影响——这就是 TCP 层队头阻塞。
:::

### HTTP/3

基于 QUIC（运行在 UDP 之上），核心价值是：

- 把传输控制放到 UDP 之上
- 降低连接迁移成本
- 避免 TCP 层队头阻塞对整条连接的拖累

所以它在移动网络、弱网切换场景里通常更有优势。

| 版本 | 核心改进 | 你必须记住什么 |
| --- | --- | --- |
| HTTP/1.1 | Keep-Alive、管线化（实践中少用） | 容易有队头阻塞，一个连接串行语义更明显 |
| HTTP/2 | 多路复用、头部压缩、二进制分帧 | 同一个 TCP 连接里可并发多个 stream |
| HTTP/3 | 基于 QUIC（UDP） | 解决 TCP 层队头阻塞，移动网络场景更友好 |

## HTTPS 与 TLS

<InlineSvg src="/network/tls-handshake.svg" alt="TLS 握手流程" />

HTTPS 本质上就是 HTTP over TLS。

HTTP 负责请求方法、Header、Body、状态码、缓存与连接复用语义。TLS 负责身份认证、密钥协商、传输加密、报文完整性保护。

所以讲 HTTPS 时不要只讲"加了一层 SSL"——更应该讲出完整链路：先 DNS 解析，再 TCP 建连，再 TLS 握手，最后才是 HTTP 请求和响应。

### TLS 握手主线

至少要能讲到这几步：

1. **ClientHello**：客户端带上支持的 TLS 版本、加密套件等
2. **ServerHello**：服务端选择协议参数并返回证书
3. 客户端校验证书链和域名
4. 双方协商会话密钥
5. 后续 HTTP 报文通过会话密钥加密传输

### HTTPS 排障

很多 HTTPS 问题不只是证书过期，还可能是：

- 证书链不完整
- SNI 配置错误
- 仅支持旧 TLS 版本
- 证书和域名不匹配
- 中间代理把 TLS 终止位置改了

::: tip HTTPS 到底慢在哪
可能慢在 DNS、TCP、TLS、CDN、网关、应用或下游依赖。不能只凭"HTTPS 比 HTTP 多了一层加密"就断言问题在 TLS 本身。
:::

## DNS、CDN 与负载均衡

<InlineSvg src="/network/dns-cdn-lb.svg" alt="DNS、CDN 与负载均衡架构" />

### DNS

DNS 解决的是域名到 IP 的映射、多级缓存、递归查询和权威应答。

你至少要知道：

- 本地缓存、系统缓存、递归 DNS、权威 DNS 是怎么配合的
- `A`、`AAAA`、`CNAME` 分别是什么
- TTL 为什么会影响切流和故障恢复——TTL 太长，旧 IP 会保留更久，切流生效慢；TTL 太短，会增加解析压力并更依赖上游 DNS 稳定性

::: tip 域名能解析成功，为什么服务仍然不可用
解析成功只代表拿到了一个 IP，不代表这个 IP 正在监听目标端口，不代表后端实例健康，也不代表 TLS 和代理链路配置正确。
:::

### CDN

CDN 解决的是：

- 静态资源就近访问
- 边缘缓存减轻源站压力
- 某些场景下的边缘安全防护和回源治理

你至少要知道：什么内容适合缓存、什么不适合；缓存命中、回源、缓存失效分别意味着什么；为什么"用户访问慢"有时是 CDN 节点问题而不是源站问题。

### 负载均衡

负载均衡解决的是把流量分散到多个后端实例、做健康检查和故障摘除、按连接或业务规则进行流量分配。

常见维度：

- **四层 LB**：基于 IP + 端口转发，性能高，但不理解 HTTP 业务语义
- **七层 LB**：基于 HTTP Host、Path、Header、Cookie 等规则转发，更灵活，适合 API 网关、灰度和业务路由

最常见的线上链路往往是：DNS -> CDN -> WAF / LB -> Nginx / Gateway -> 应用服务

想深入 DNS、CDN 和负载均衡，继续读 [DNS、CDN 与负载均衡](./dns-cdn-and-load-balancing.md)。

## 排障时先按哪一层下手

<InlineSvg src="/network/troubleshooting-flowchart.svg" alt="网络排障决策流程" />

### 1. 先问"是连不上，还是连上了但慢"

- 连不上：先看 DNS、路由、防火墙、端口监听、健康检查
- 连上但慢：再看 RTT、握手、TLS、应用耗时、下游依赖

### 2. 再问"是单点问题，还是全局问题"

- 只有某个机房慢
- 只有某个运营商慢
- 只有某个域名慢
- 只有 HTTPS 慢，HTTP 不慢

这些差异很重要，因为它们能快速缩小范围。

### 3. 最后才去做更细的抓包和证据分析

常用命令通常包括：

| 命令 | 用途 |
| --- | --- |
| `ping` | 检查连通性（注意某些环境禁 ICMP，ping 不通不一定等于服务不可用） |
| `traceroute` / `mtr` | 追踪路由路径 |
| `dig` / `nslookup` | 排查 DNS 解析 |
| `curl -v` | 快速看状态码、响应头、重定向和 TLS 协商信息，成本低信息量大 |
| `ss -tanp` | 看端口监听、连接状态（ESTAB、TIME-WAIT、CLOSE-WAIT）、连接堆积 |
| `tcpdump` | 当其他手段看不清楚时，确认包到底有没有到、到了哪个节点被改写或被丢弃 |

如果想把这些命令放回真实问题里看，继续读：[网络排障与常用命令](./troubleshooting-and-commands.md)

## 高频自检题

在你说自己"网络基础已经够用了"之前，至少先问自己这些问题：

::: details 自检清单
- 能不能把一次 HTTPS 请求从 DNS 到响应完整讲顺
- 能不能解释三次握手、四次挥手、TIME_WAIT 的工程意义
- 能不能说清 HTTP/2 为什么仍可能受到 TCP 丢包影响
- 能不能解释 TLS 握手至少在做哪些事
- 能不能说出 DNS 的缓存层次和 TTL 的影响
- 能不能区分 CDN、四层 LB、七层 LB 分别在解决什么问题
- 能不能在故障时快速决定先用 `dig`、`curl`、`ss` 还是 `tcpdump`
- 能不能把"协议原理"和"线上排障动作"讲成一条线
:::

建议回答统一用这三步：先说它解决什么问题，再说关键机制或实现路径，最后补一句工程场景、风险点或常见误区。
