# 网络必备知识

## 适合谁看

- 想系统补齐网络基础，不再把 TCP、HTTP、DNS、CDN 当成零散名词来背的工程师
- 做后端、基础设施、云原生或系统设计时，发现很多性能和故障问题都绕不开网络的人
- 面试里经常被问三次握手、HTTPS、DNS、负载均衡，但回答容易只停在单点概念的人

## 学习目标

- 建立一张从链路层到应用层的统一网络知识地图
- 理解一次请求从输入域名到拿到响应，中间到底发生了什么
- 知道网络排障时如何按层定位，而不是一上来就盲猜问题点

## 先建立一张网络全景图

最值得先记住的不是单个协议，而是网络问题可以按层拆：

| 层次 | 你最该先掌握什么 | 典型问题 |
| --- | --- | --- |
| 物理 / 链路层 | 网卡、MAC、交换、ARP、MTU | 同网段通信异常、丢包、ARP 冲突 |
| 网络层 | IP、子网、路由、NAT、ICMP | 路由不通、跨网段访问失败、Traceroute 中断 |
| 传输层 | TCP、UDP、端口、连接状态 | 建连超时、RST、丢包重传、连接堆积 |
| 应用层 | HTTP、HTTPS、DNS、WebSocket、gRPC | 502/504、证书错误、域名解析异常、跨层代理问题 |

工程里最常见的误区是：

- 把所有问题都当成应用问题
- 或者一看到超时就只盯着代码

更稳的思路通常是：

1. 先确认问题发生在哪一层
2. 再看这一层最关键的状态、指标和日志
3. 最后再判断是否要往上层或下层继续追

## 一次请求到底经过哪些环节

假设浏览器访问 `https://example.com/api/orders`，大致会经过：

1. 先查本地 DNS 缓存，没命中再去递归解析
2. 拿到目标 IP，按路由规则决定走哪个网关
3. 发起 TCP 建连
4. 如果是 HTTPS，再进行 TLS 握手、校验证书、协商加密套件
5. 建连成功后发送 HTTP 请求
6. 请求可能先到 CDN、WAF、SLB、反向代理，再到应用服务
7. 服务处理后返回响应，浏览器再决定是否缓存、复用连接或继续发下一个请求

所以很多“接口慢”其实可能卡在完全不同的位置：

- DNS 慢
- TCP 建连慢
- TLS 握手慢
- 网关转发慢
- 上游服务处理慢
- 下游数据库或缓存慢

## TCP 和 UDP 各自解决什么问题

### TCP

TCP 解决的是：

- 面向连接
- 有序传输
- 可靠到达
- 流量控制和拥塞控制

因此它更适合：

- HTTP / HTTPS
- 数据库连接
- RPC 调用
- 大多数要求正确性和顺序性的服务端通信

### UDP

UDP 的特点是：

- 无连接
- 不保证到达
- 不保证顺序
- 头部更轻，延迟更低

它更适合：

- 音视频实时传输
- DNS 查询
- 某些游戏场景
- 对实时性要求更高、可容忍少量丢包的场景

一个常见理解方式是：

- TCP 重“正确”
- UDP 重“快”

但不要简单理解成“UDP 一定更快”。如果上层还要自己做重传、排序、补偿，整体复杂度可能更高。

## 为什么 TCP 能可靠传输

TCP 的可靠，不是靠“永远不丢”，而是靠一组机制把不可靠网络尽量收敛成可靠连接：

- 序列号：保证字节流有顺序
- ACK 确认：对方收到什么、没收到什么要能反馈
- 超时重传：超时没确认就重发
- 滑动窗口：控制未确认数据量，避免把对方压垮
- 流量控制：接收方通过窗口告诉发送方“我还能吃多少”
- 拥塞控制：根据网络拥堵情况调整发送速度

### 三次握手必须会讲清什么

至少要能说清：

- 为什么不是两次
- 为什么不是四次
- SYN、ACK 各自表示什么
- 服务端为什么会有半连接队列、全连接队列

最短回答口径可以是：

> 三次握手的本质，是双方互相确认“发送能力和接收能力都正常”，并同步初始化序列号。如果少一次，就无法可靠确认双向通信状态。

### 四次挥手为什么通常是四次

因为 TCP 是全双工：

- 我不发了，不代表我也不收了
- 对方收到 FIN 后，可能还有数据没发完

所以关闭连接通常是两边分别关闭各自的发送方向。

### TIME_WAIT 为什么重要

TIME_WAIT 的价值主要是两件事：

- 防止旧连接的延迟包污染新连接
- 确保最后的 ACK 如果丢了，对方还有机会重发 FIN

线上常见问题不是“为什么有 TIME_WAIT”，而是：

- 为什么 TIME_WAIT 太多
- 为什么短连接风暴导致端口耗尽
- 为什么连接没有复用好

## HTTPHTTPS 和 TLS 怎么串起来看

HTTP 负责的是：

- 请求方法
- Header
- Body
- 状态码
- 缓存与连接复用语义

TLS 负责的是：

- 身份认证
- 密钥协商
- 传输加密
- 报文完整性保护

HTTPS 本质上就是：

- HTTP over TLS

所以讲 HTTPS 时不要只讲“加了一层 SSL”。更应该讲：

1. 先 DNS 解析
2. 再 TCP 建连
3. 再 TLS 握手
4. 最后才是 HTTP 请求和响应

### HTTP/1.1、HTTP/2、HTTP/3 该怎么区分

| 版本 | 核心改进 | 你必须记住什么 |
| --- | --- | --- |
| HTTP/1.1 | Keep-Alive、管线化（实践中少用） | 容易有队头阻塞，一个连接串行语义更明显 |
| HTTP/2 | 多路复用、头部压缩、二进制分帧 | 同一个 TCP 连接里可并发多个 stream |
| HTTP/3 | 基于 QUIC（UDP） | 解决 TCP 层队头阻塞，移动网络场景更友好 |

很多面试题会追问：

- HTTP/2 明明多路复用了，为什么还会卡

因为：

- 它只是把应用层并发搬进了同一条 TCP 连接里
- 如果 TCP 本身丢包，还是会影响整个连接上的数据推进

## DNSCDN 和负载均衡分别在做什么

### DNS

DNS 解决的是：

- 域名到 IP 的映射
- 多级缓存
- 递归查询和权威应答

你至少要知道：

- 本地缓存、系统缓存、递归 DNS、权威 DNS 是怎么配合的
- `A`、`AAAA`、`CNAME` 分别是什么
- TTL 为什么会影响切流和故障恢复

### CDN

CDN 解决的是：

- 静态资源就近访问
- 边缘缓存减轻源站压力
- 某些场景下的边缘安全防护和回源治理

你至少要知道：

- 什么内容适合缓存，什么内容不适合
- 缓存命中、回源、缓存失效分别意味着什么
- 为什么“用户访问慢”有时是 CDN 节点问题而不是源站问题

### 负载均衡

负载均衡解决的是：

- 把流量分散到多个后端实例
- 做健康检查和故障摘除
- 按连接、请求或业务规则进行流量分配

常见维度：

- 四层 LB：基于 IP + 端口转发，性能高
- 七层 LB：基于 HTTP Host、Path、Header 等规则转发，更灵活

最常见的线上链路往往是：

- DNS -> CDN -> WAF / LB -> Nginx / Gateway -> 应用服务

## 排障时先按哪一层下手

一个很实用的顺序是：

### 1. 先问“是连不上，还是连上了但慢”

- 连不上：先看 DNS、路由、防火墙、端口监听、健康检查
- 连上但慢：再看 RTT、握手、TLS、应用耗时、下游依赖

### 2. 再问“是单点问题，还是全局问题”

- 只有某个机房慢
- 只有某个运营商慢
- 只有某个域名慢
- 只有 HTTPS 慢，HTTP 不慢

这些差异很重要，因为它们能快速缩小范围。

### 3. 最后才去做更细的抓包和证据分析

常用命令通常包括：

- `ping`
- `traceroute` / `mtr`
- `dig` / `nslookup`
- `curl -v`
- `ss -tanp`
- `tcpdump`

如果想把这些命令放回真实问题里看，继续读：

1. 先把“分层模型 + 一次请求链路”讲顺
2. 再把 TCP、HTTP、TLS 这些高频协议讲清
3. 然后补 DNS、CDN、负载均衡这些线上访问链路上的基础设施
4. 最后把排障命令和抓包思路转成固定动作

## 高频自检题

- 为什么三次握手不能减少成两次
- 为什么 HTTP/2 多路复用后仍然会受到 TCP 丢包影响
- 为什么 HTTPS 排障不能只看证书是否过期
- 为什么 DNS TTL 会影响故障切流和发布生效时间
- 为什么很多“接口超时”问题最后根因并不在应用代码里
