# 网络高频问题与自检清单

这页不追求把所有网络知识都展开，而是把最常被问、最容易答散、最适合拿来做阶段复习的问题按主题排好。

建议回答顺序统一成这三步：

- 先说它解决什么问题
- 再说关键机制或实现路径
- 最后补一句工程场景、风险点或常见误区

如果你的网络回答总是停留在“背定义”，这个结构很重要。

## 1. 分层与整体链路

**1. 一次 HTTPS 请求从输入域名到返回响应，中间经过了哪些环节？**

- DNS 解析：先查本地缓存，再走递归解析拿到目标 IP
- TCP 建连：客户端和服务端完成三次握手
- TLS 握手：协商加密套件、校验证书、生成会话密钥
- HTTP 请求转发：可能经过 CDN、WAF、负载均衡、网关、反向代理，再到应用服务
- 响应返回：客户端再决定是否缓存、是否复用连接

**2. 为什么网络问题最好按层排查，而不是一上来就看应用代码？**

- 因为 DNS、路由、建连、TLS、代理转发、应用处理都可能导致“接口超时”
- 如果不先分层，容易把非应用问题误判成业务代码问题
- 排障成本最低的方式，是先判断问题大概落在哪一层，再取证

**3. OSI 七层和 TCP/IP 四层模型，到底面试时该怎么讲？**

- 不必死背教科书映射，重点是知道“每层大概负责什么”
- TCP/IP 在工程实践里更常用：链路层、网络层、传输层、应用层
- 回答时最好结合真实协议：IP、TCP、UDP、HTTP、DNS、TLS

## 2. TCP 与连接

**4. 为什么 TCP 要三次握手，不能两次？**

- 三次握手的本质，是双方确认发送能力、接收能力和初始序列号都已同步
- 两次无法可靠确认客户端是否收到服务端确认，也无法完整建立双向通信状态
- 少一次确认，旧连接延迟包或伪造连接更难处理

**5. 为什么 TCP 挥手通常是四次？**

- TCP 是全双工，双方发送方向可以分别关闭
- 一方发 FIN 只表示“我不再发送”，不代表它也不再接收
- 另一方可能还有数据没发完，所以通常拆成两边各关闭一次

**6. TIME_WAIT 为什么存在？**

- 防止旧连接的延迟包影响新连接
- 确保最后 ACK 如果丢了，对方还能重发 FIN
- 线上重点不是“为什么有”，而是为什么 TIME_WAIT 太多、端口是否被耗尽、连接复用是否合理

**7. TCP 怎么保证可靠传输？**

- 序列号：保证顺序
- ACK：确认哪些数据已到达
- 超时重传：没收到确认就重发
- 滑动窗口：限制未确认数据量
- 流量控制：避免接收端被打爆
- 拥塞控制：网络拥堵时主动减速

**8. 半连接队列和全连接队列分别在说什么？**

- 半连接队列存放尚未完成三次握手的连接
- 全连接队列存放已经建立但应用尚未 `accept` 的连接
- 当这两处堆积异常时，常见表现就是建连超时、SYN backlog 满或服务端 accept 不及时

## 3. HTTP、HTTPS 与 TLS

**9. HTTP/1.1、HTTP/2、HTTP/3 的核心区别是什么？**

- HTTP/1.1：文本协议、连接复用能力有限
- HTTP/2：二进制分帧、多路复用、头部压缩
- HTTP/3：基于 QUIC，运行在 UDP 之上，重点改善 TCP 层队头阻塞和连接迁移

**10. 为什么 HTTP/2 多路复用了，还是会卡？**

- 因为 HTTP/2 的多路复用发生在应用层
- 底层仍然是一条 TCP 连接
- 一旦 TCP 丢包，整条连接上的数据推进都会受影响

**11. HTTPS 到底比 HTTP 多了什么？**

- HTTPS 本质是 HTTP over TLS
- 多出来的是身份认证、密钥协商、加密传输和完整性校验
- 所以讲 HTTPS 不能只说“加密”，还要讲证书、TLS 握手和信任链

**12. TLS 握手至少要讲清哪几步？**

- ClientHello：客户端声明支持的版本、套件等
- ServerHello：服务端选择参数并返回证书
- 客户端校验证书链和域名
- 双方协商出会话密钥
- 后续 HTTP 报文基于会话密钥加密传输

**13. HTTPS 排障为什么不能只看证书有没有过期？**

- 还可能是证书链不完整
- 也可能是域名不匹配、SNI 配置错误、TLS 版本不兼容
- 某些问题甚至出在中间代理或 TLS 终止位置错误

## 4. DNS、CDN 与负载均衡

**14. DNS 查询最少要能说出哪些角色？**

- 本地缓存 / 浏览器缓存
- 系统解析器
- 递归 DNS
- 权威 DNS

只要能把这四个角色的职责和顺序说清，就已经比只会背“域名转 IP”强很多。

**15. TTL 为什么会影响发布和故障恢复？**

- 因为 TTL 决定解析结果缓存多久
- TTL 太长，旧 IP 会保留更久，切流和恢复生效慢
- TTL 太短，会增加解析压力并更依赖上游 DNS 稳定性

**16. CDN 的核心价值是什么？**

- 就近访问，降低延迟
- 边缘缓存，减轻源站压力
- 某些情况下承担边缘安全防护和流量治理能力

**17. 四层和七层负载均衡怎么区分？**

- 四层：基于 IP、端口和连接，性能更高，但不理解 HTTP 业务语义
- 七层：基于 Host、Path、Header、Cookie 等规则转发，更灵活，适合 API 网关、灰度和业务路由

**18. 域名能解析成功，为什么服务仍然可能不可用？**

- 解析成功只代表你拿到了一个 IP
- 不代表这个 IP 正在监听目标端口
- 不代表后端实例健康
- 也不代表 TLS 和代理链路配置正确

## 5. 排障与工程实践

**19. 为什么 `ping` 不通不一定等于服务不可用？**

- 某些环境会禁 ICMP
- ping 检查的是 ICMP，不是 HTTP 或 TCP 业务语义
- 真实服务可能仍然能在 TCP/443 或 TCP/80 上正常工作

**20. 为什么 `curl -v` 常常是排 HTTP/HTTPS 问题的第一步？**

- 它能快速看到状态码、响应头、重定向和部分 TLS 协商信息
- 成本低，信息量大，适合先判断问题大致在哪个阶段
- 比浏览器错误页更适合做第一轮定位

**21. 什么时候应该用 `ss`？**

- 看端口是否监听
- 看连接状态是否异常，比如 `ESTAB`、`TIME-WAIT`、`CLOSE-WAIT`
- 看是不是连接堆积或进程没有及时处理 accept

**22. 什么时候应该抓包？**

- 当前面的 DNS、监听、HTTP 层都看不清楚时
- 怀疑 TCP 重传、RST、FIN、握手不完整时
- 需要确认包到底有没有到、到了哪个节点被改写或被丢弃时

## 6. 一份自检清单

在你说自己“网络基础已经够用了”之前，至少先问自己这些问题：

- 能不能把一次 HTTPS 请求从 DNS 到响应完整讲顺
- 能不能解释三次握手、四次挥手、TIME_WAIT 的工程意义
- 能不能说清 HTTP/2 为什么仍可能受到 TCP 丢包影响
- 能不能解释 TLS 握手至少在做哪些事
- 能不能说出 DNS 的缓存层次和 TTL 的影响
- 能不能区分 CDN、四层 LB、七层 LB 分别在解决什么问题
- 能不能在故障时快速决定先用 `dig`、`curl`、`ss` 还是 `tcpdump`
- 能不能把“协议原理”和“线上排障动作”讲成一条线
