## IP 层解决什么问题

IP 层解决的核心问题是：

- 数据包往哪发
- 下一跳是谁
- 如何跨网段到达目标机器

换句话说，IP 层更像“寻路系统”，它不保证可靠，也不关心应用语义，它关心的是：

- 目标地址是谁
- 我应该把包先交给谁
- 这个包还能活多久

## 必须掌握的关键词

| 关键词 | 最值得记住什么 |
| --- | --- |
| 源 IP / 目标 IP | 说明包从哪来、要去哪 |
| 子网掩码 / CIDR | 决定哪些地址属于同一网段 |
| 默认网关 | 不知道更精确路径时，先把包交给它 |
| 路由表 | 决定下一跳和出接口 |
| ARP | 在局域网里把 IP 解析成 MAC |
| ICMP | 承载可达性和差错信息，如 `ping`、TTL 超时 |
| TTL | 控制一个包最多能经过多少跳，避免死循环 |
| MTU | 一条链路能承载的最大 IP 包大小 |
| NAT | 在地址转换和访问边界上改写源目地址或端口 |

### IP 地址和子网掩码最少要会什么

不一定要现场手算很复杂的网段，但至少要能判断：

- `192.168.1.10/24` 和 `192.168.1.20/24` 在同一网段
- `192.168.1.10/24` 和 `192.168.2.20/24` 不在同一网段
- 不同网段通信通常要经过路由器或网关

## 同网段和跨网段通信怎么走

### 同网段通信

如果源主机判断目标 IP 和自己在同一网段，典型流程是：

1. 先查 ARP 缓存，看目标 IP 对应哪个 MAC
2. 没命中就发 ARP 请求
3. 拿到目标 MAC 后，直接把二层帧发给对方

这时候不需要经过默认网关。

### 跨网段通信

如果目标不在本地网段，典型流程就变成：

1. 查路由表
2. 发现目标应该走默认网关或某条更具体路由
3. 先通过 ARP 找到“下一跳”的 MAC
4. 把包交给网关，由网关继续转发

所以很多“能不能访问到服务”的问题，第一层不是 TCP，而是：

- 目标地址是否可达
- 默认网关是否正确
- 中间路由是否知道怎么回包

## 路由表到底怎么决定下一跳

工程里不用把所有协议细节背全，但最好知道这两个判断原则：

### 最长前缀匹配

如果路由表里同时有：

- `10.0.0.0/8`
- `10.1.2.0/24`

访问 `10.1.2.5` 时，会优先匹配更具体的 `10.1.2.0/24`。

### 默认路由

如果没有更具体的匹配，通常会走默认路由，也就是常见的：

- `0.0.0.0/0`

默认路由错了，最常见表现就是：

- 本机访问外网失败
- 容器或虚拟机只能访问本地网段
- 请求发得出去，但回程不对

## ARP、ICMP、TTL、MTU 分别在做什么

### ARP

ARP 主要在局域网里解决：

- 我知道对方 IP
- 但我还不知道它的 MAC

如果 ARP 缓存异常、ARP 冲突或交换网络有问题，就可能出现：

- 同网段机器互访异常
- 偶发访问飘忽
- `ping` 某些地址时很怪异

### ICMP

ICMP 常被误解成“就是 ping”。更准确地说，它承担的是：

- 可达性探测
- 差错回报
- 路径观察

比如：

- `ping` 本质上在用 ICMP Echo
- `traceroute` / `tracepath` 常会利用 TTL 递减和超时报文

### TTL

TTL 每经过一跳就减 1，减到 0 时，路由器会丢弃这个包并返回超时信息。它的核心价值是：

- 防止路由环路里的包无限转发
- 帮助我们借助 `traceroute` 观察路径

### MTU 和 MSS

这是排障里特别容易被忽略的一组概念：

- **MTU**：一条链路上可承载的最大 IP 包大小
- **MSS**：TCP 负载里单次可发送的最大数据段大小

如果链路 MTU 不一致、分片受限或 Path MTU Discovery 受阻，常见现象会是：

- `ping` 小包通，大包不通
- TCP 建连成功，但传大一点的数据就卡
- HTTPS / 文件上传 / 某些 VPN 场景偶发超时

## 常见 IP 层问题怎么排

### 现象一：同网段访问异常

优先怀疑：

- IP 配置错误
- 子网掩码错误
- ARP 缓存异常
- 二层网络问题

第一轮可先看：

```bash
ip addr
ip neigh
arp -a
ping 192.168.1.10
```

### 现象二：跨网段访问不通

优先怀疑：

- 默认网关错误
- 路由表缺失
- 安全组 / ACL / 防火墙拦截
- 回程路由不通

第一轮可先看：

```bash
ip route
traceroute example.com
tracepath example.com
```

### 现象三：小请求正常，大请求超时

优先怀疑：

- MTU 黑洞
- VPN / 隧道额外封装后有效 MTU 变小
- ICMP 被错误拦截，导致 PMTUD 失效

第一轮可先试：

```bash
ping -M do -s 1472 example.com
tracepath example.com
```

## IP 和 TCP 的关系

一个很实用的理解是：

- TCP 更像“连接和传输规则”
- IP 更像“把包送到目的地的寻路系统”

所以如果根本路由不到，TCP 和 HTTP 再正确也没用。

### NAT 为什么常和 IP 放在一起讲

NAT 本质上就是在网络边界改写地址或端口，它和 IP 路由天然相关，但它不等于路由本身。

最常见的两类是：

- **SNAT**：改写源地址，常见于内网访问公网
- **DNAT**：改写目标地址，常见于端口映射和入口转发

它解决的是地址复用和访问边界问题，但也会带来：

- 回包路径更复杂
- 原始源地址丢失
- 排障路径更长

## 高频误区

- “能 `ping` 通就说明业务没问题”是误区。ICMP 可达不代表 TCP 端口和应用层可用。
- “访问不通就是服务没启动”是误区。很多问题其实卡在默认网关、路由、ARP 或回程路径。
- “NAT 就是路由”也是误区。NAT 是地址转换，路由是决定包往哪走。
