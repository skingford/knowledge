# 抓包、tcpdump 与 Wireshark

## 适合谁看

- 会用 `curl`、`dig`、`ss`，但到了“最后一层证据”还不知道怎么抓包的人
- 线上排障时经常怀疑重传、RST、握手失败、代理中途断开，却缺少直接证据的人
- 想把 `tcpdump` 和 Wireshark 从“看起来很专业”变成日常可用工具的人

## 学习目标

- 知道什么场景下应该抓包，什么场景下还没必要抓
- 学会用 `tcpdump` 抓到足够小、足够干净、能回答问题的样本
- 看懂最常见的 TCP、TLS、HTTP 关键信号，知道问题大概卡在哪一段

## 抓包前先确定要验证什么

抓包最怕的不是不会命令，而是：

- 包抓了一大堆
- 但没有明确要验证的现象

开始前先把问题收缩成一句话：

- 我要确认客户端有没有把 `SYN` 发出去
- 我要确认服务端有没有回 `SYN,ACK`
- 我要确认是不是代理回了 `RST`
- 我要确认 TLS 握手是不是在证书校验前就断了

如果你能先说清“我要验证什么”，抓包通常就会简单很多。

## 应该在哪个节点抓包

抓包位置决定你能看到什么证据。

| 抓包位置 | 更适合回答什么问题 |
| --- | --- |
| 客户端节点 | 请求有没有发出去、收到什么错误、连接是否被本地或中间层打断 |
| 服务端节点 | 请求有没有到、服务端有没有回包、是不是服务端主动断开 |
| 代理 / 网关节点 | 代理是否收到了请求、是否成功转发给上游、哪一段断了 |
| 中间网络设备 | 更适合专门网络团队定位链路或安全设备问题 |

最实用的经验是：

- **如果不知道从哪抓，先从客户端或代理节点开始**
- **如果怀疑服务没收到，再到服务端抓**

## tcpdump 常用命令

### 最常用的基础抓法

```bash
tcpdump -i any host example.com and port 443
```

适合快速确认：

- 是否有流量经过
- 大概是哪个 IP、哪个端口
- 是否能看到建连和关闭

### 只看某台主机和某个端口

```bash
tcpdump -i eth0 -nn host 10.0.0.8 and port 8443
```

说明：

- `-i eth0`：指定网卡
- `-nn`：不做域名和端口名解析，输出更快更直观

### 抓下来给 Wireshark 分析

```bash
tcpdump -i any -nn host 10.0.0.8 and port 443 -w /tmp/https.pcap
```

适合：

- 线上先快速保留证据
- 之后再本地用 Wireshark 细看

### 控制抓包量

```bash
tcpdump -i any -nn host 10.0.0.8 and port 443 -c 200
```

`-c 200` 很有用，能避免一不小心抓太多。

### 看 TCP 标志位

```bash
tcpdump -i any -nn 'tcp[tcpflags] & (tcp-syn|tcp-fin|tcp-rst) != 0'
```

适合快速筛出：

- 建连
- 关闭
- 异常复位

## wireshark 常用过滤思路

Wireshark 更像“细看和回放工具”，它的价值主要在：

- 更容易顺着一条流看完整过程
- 能高亮重传、RST、TLS Alert 等信号
- 可视化地看每个阶段花了多久

常用显示过滤器可以先记这些：

```text
ip.addr == 10.0.0.8
tcp.port == 443
dns
http
tls
tcp.flags.reset == 1
tcp.analysis.retransmission
```

很实用的一个动作是：

- 右键一条包
- `Follow` 某条 TCP Stream

这样能把同一条连接单独拎出来看。

## tcp 三次握手在抓包里怎么看

最基本的正常建连应该看到：

1. 客户端发 `SYN`
2. 服务端回 `SYN, ACK`
3. 客户端回 `ACK`

### 只看到 SYN 没有回应

更常见的怀疑方向：

- 目标不可达
- 中间链路丢弃
- 防火墙 / ACL / 安全组丢包
- 目标服务没有正确到达这个路径

### 看到了 SYN 和 SYN ACK 但没有最后 ACK

更常见的怀疑方向：

- 回包能到客户端，但客户端回不去
- 中间链路不对称
- 客户端本地策略或网络异常

## 连接异常信号怎么看

### RST

RST 通常意味着“强制断开”，它经常说明：

- 对端端口没监听
- 协议不匹配
- 应用或代理主动拒绝连接
- 已关闭连接上又来了数据

### Retransmission

重传多不一定等于“服务挂了”，但通常说明：

- 丢包
- 对端处理太慢，ACK 回不来
- 链路质量差
- 某个中间设备丢了包

### FIN

FIN 更像“正常关闭”，要结合方向看：

- 是客户端主动关
- 还是服务端主动关
- 是空闲超时收的
- 还是业务处理完自然关闭

## TLS 握手在抓包里怎么看

HTTPS 抓包最先别想着“看到完整明文内容”，第一步应该先看：

- 有没有 `ClientHello`
- 服务端有没有回 `ServerHello`
- 是不是很快出现 `Alert`
- 连接是在握手前、握手中还是握手后断的

### 最少应该能认出这些阶段

1. `ClientHello`
2. `ServerHello`
3. 证书相关消息
4. 后续加密应用数据

如果你看到：

- `ClientHello` 发出后就超时：更像链路、SNI、代理转发或对端 TLS 能力问题
- 很快收到 `Alert`：更像证书、版本、密码套件、mTLS 等协商失败

### 为什么 HTTPS 内容通常看不到

因为握手完成后应用数据已经加密了。没有额外密钥材料时，你通常能看到：

- 谁和谁通信
- 握手是否成功
- 连接什么时候断
- 包大小和时序

但看不到明文 HTTP 路径和 Body。

如果你需要在自有环境里解密调试，常见思路是：

- 使用浏览器或客户端导出 `SSLKEYLOGFILE`
- 在 Wireshark 里配置会话密钥日志

## HTTP 请求和响应在抓包里怎么看

### 明文 HTTP

如果是明文 HTTP，你通常能直接看到：

- 请求方法
- URL 路径
- Header
- 状态码

这对排查代理改写、Host 问题、Header 丢失非常直接。

### HTTPS

如果是 HTTPS，除非你具备解密条件，否则更多只能结合：

- TLS 阶段
- 包长
- 时序
- 应用日志

一起判断问题。

## 几类典型问题怎么抓

### 现象一 连接超时

优先看：

- 客户端有没有持续发 `SYN`
- 服务端或代理有没有回 `SYN,ACK`
- 有没有中间节点只进不出

### 现象二 connection reset by peer

优先看：

- 谁发了 `RST`
- `RST` 之前是否已经完成握手
- 是代理发的还是上游服务发的

### 现象三 TLS handshake failure

优先看：

- 是否至少发出了 `ClientHello`
- 服务端有没有直接回 `Alert`
- 是不是代理终止 TLS 后再向上游转发

### 现象四 小请求通 大请求卡

优先怀疑：

- MTU / 分片 / PMTUD 问题

这时抓包要特别看：

- 是否反复重传同一段数据
- 是否某个大小以上的数据开始异常

## 抓包时的安全和性能注意事项

- 先缩小过滤范围，避免抓到过多无关流量。
- 生产环境优先抓短时间、少量样本，避免长时间高负载抓全量。
- 抓到的 `pcap` 可能包含敏感信息，尤其是明文 HTTP、Cookie、内网地址和部分认证头。
- 在线上主机抓包前，最好先确认磁盘空间和保存路径，避免把节点写满。

## 高频误区

- “只要抓包就一定能看到真相”是误区。抓包也需要选对节点和时间窗口。
- “HTTPS 抓包就能直接看到 URL 和 Body”是误区。大多数时候你先看到的是握手、时序和加密后的应用数据。
- “看到重传就一定是网络问题”也是误区。对端处理过慢、队列积压、代理卡住也会表现成重传。
