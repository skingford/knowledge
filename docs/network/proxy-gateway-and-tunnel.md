## 先分清正向代理反向代理网关和隧道

很多人第一次学到这里会把它们都记成“中间转发层”，但最稳的拆法其实是先看“它代表谁”：

| 概念 | 更像在替谁工作 | 核心作用 |
| --- | --- | --- |
| 正向代理 | 替客户端 | 帮客户端去访问目标服务 |
| 反向代理 | 替服务端 | 帮服务端接入和分发外部请求 |
| 网关 | 替服务集群或业务域 | 在反向代理基础上叠加认证、路由、限流、观测、协议治理 |
| 隧道 | 替两端保留一条透明通路 | 把中间层变成“只转发字节流”的通道 |

一句话记忆：

- **正向代理**：客户端知道代理，服务端未必知道真实客户端
- **反向代理**：客户端以为自己在连服务，实际上先连到代理
- **网关**：是更“有业务意识”的反向代理
- **隧道**：中间层尽量少解释协议，只负责把通道打通

## 正向代理在解决什么问题

正向代理最常见的价值是：

- 统一出网
- 访问控制
- 缓存加速
- 隐藏客户端地址

典型场景：

- 企业内网统一出互联网
- 浏览器或 CLI 配置 HTTP 代理
- 安全审计和访问白名单

### 正向代理的关键特征

- 客户端主动把请求发给代理
- 代理再替客户端去请求目标站点
- 目标服务看到的往往是代理地址，而不是真实客户端地址

所以如果你在命令行里配置：

```bash
export http_proxy=http://proxy.example.com:8080
export https_proxy=http://proxy.example.com:8080
```

本质上就是在让客户端走正向代理。

## 反向代理在解决什么问题

反向代理更常见于服务端入口，它主要解决：

- 统一接入
- 路由转发
- TLS 终止
- 静态资源分发
- 连接复用
- 灰度发布和流量治理

典型链路通常是：

1. 客户端访问 `api.example.com`
2. 先打到 Nginx / Envoy / Ingress / SLB
3. 反向代理按规则转发到上游服务
4. 上游服务把结果返回给代理
5. 代理再回给客户端

### 反向代理最常见的工程价值

- 把多个上游服务藏在统一域名后面
- 统一做 TLS 证书和 HTTPS 入口
- 为上游服务复用连接，减轻短连接压力
- 在代理层做超时、重试、熔断和可观测性

## 网关和反向代理有什么区别

它们不是完全割裂的两类东西，更多是能力层次差异。

### 反向代理更偏“流量入口”

反向代理更强调：

- 请求转发
- 上游选择
- 静态资源与 TLS 处理
- 基础连接治理

### 网关更偏“协议和业务治理入口”

网关通常会在反向代理能力上再加：

- 认证鉴权
- 限流熔断
- 统一日志和 tracing
- 灰度、AB、路由编排
- 请求改写、Header 注入
- API 生命周期治理

所以你可以粗略理解成：

- 所有网关几乎都带反向代理属性
- 但不是所有反向代理都应该叫网关

## 隧道和升级连接怎么理解

### CONNECT 隧道

当客户端通过 HTTP 代理访问 HTTPS 站点时，常见流程不是“代理解密后帮你转 HTTP”，而是：

1. 客户端先向代理发 `CONNECT example.com:443`
2. 代理建立到目标站点的 TCP 连接
3. 后续双方在这条连接里直接走 TLS 握手和应用数据

此时代理更像在提供一条“字节流隧道”。

### WebSocket Upgrade

WebSocket 常见问题在于它不是普通短请求，而是先走一次 HTTP Upgrade，再进入长连接双向通信。

这意味着代理层除了转请求，还要正确处理：

- `Connection: Upgrade`
- `Upgrade: websocket`
- 长连接超时
- 读写缓冲和反压

如果代理配置不完整，常见现象就是：

- 握手失败
- 建好后很快断开
- 空闲一段时间后被代理回收

### gRPC 和 HTTP/2

gRPC 更常见地跑在 HTTP/2 之上，因此代理层还要支持：

- HTTP/2 协议协商
- 长连接和多路复用
- 更细粒度的超时和流控

所以“HTTP 能通但 gRPC 不通”经常不是服务逻辑问题，而是代理层协议能力或转发配置不匹配。

## 代理和 NAT LB CDN 怎么区分

这些概念都在入口链路里，最容易混：

| 概念 | 重点解决什么 | 更偏哪一层 |
| --- | --- | --- |
| NAT | 改地址或端口 | 网络层 / 边界 |
| 负载均衡 | 把流量分摊到多个后端 | L4 或 L7 |
| 反向代理 | 代表服务端接收并转发请求 | 多见于 L7 |
| 网关 | 在代理基础上做统一业务治理 | L7 |
| CDN | 边缘接入、缓存和加速 | DNS + L7 + 边缘基础设施 |

更实用的理解是：

- **NAT** 更像“地址转换”
- **LB** 更像“流量分发”
- **代理** 更像“协议入口和转发控制”
- **网关** 更像“统一治理入口”
- **CDN** 更像“离用户最近的边缘入口”

## 真实客户端 IP 为什么会丢

只要中间多了一层代理，源地址就可能发生变化。

最常见原因是：

- TCP 连接是客户端和代理建的
- 上游服务看到的对端就是代理本身

所以才需要一些额外信息把原始来源传递下去，例如：

- `X-Forwarded-For`
- `X-Real-IP`
- `Forwarded`

### 这里最容易出错的点

- 信任边界配错，把伪造的 `X-Forwarded-For` 当真
- 多层代理链下，只拿第一跳或最后一跳，导致真实来源判断错误
- 日志里记录的是代理地址，不是真实客户端

## 常见代理层错误怎么倒推

| 现象 / 错误码 | 更常见的原因 | 第一反应先查什么 |
| --- | --- | --- |
| `502 Bad Gateway` | 代理连不上上游，或上游返回了非法响应 | 上游监听、协议匹配、代理错误日志 |
| `503 Service Unavailable` | 上游被摘除、无健康实例、限流或过载保护触发 | 健康检查、实例状态、网关限流 |
| `504 Gateway Timeout` | 代理等待上游响应超时 | 上游耗时、下游依赖、代理超时设置 |
| WebSocket `101` 失败 | Upgrade 头未透传、长连接策略不对 | 代理配置、超时、Header |
| gRPC 报协议不支持 | HTTP/2 / ALPN / TLS 配置不匹配 | 代理协议能力、TLS 协商、上游配置 |

### 为什么 502 和 504 容易混

可以这样粗记：

- **502**：更像“代理拿到的是坏结果，或者根本没拿到合法上游连接”
- **504**：更像“代理等上游等太久了”

## 排障时先看哪些信号

### 看代理日志

代理层问题最有价值的通常不是业务日志，而是入口日志：

- 上游地址是哪个
- 上游连接是否成功
- 响应时间卡在连接、首包还是总耗时
- 是代理拒绝的，还是上游返回的

### 看上游连接状态

如果怀疑代理到上游连不上，可以先看：

```bash
ss -tanp | head
curl -v http://upstream-service:8080/health
```

重点确认：

- 代理所在节点是否能直连上游
- 上游是否真的在监听
- 协议是否对齐，比如明文 HTTP 和 HTTPS 有没有配反

### 看代理是否改了请求

重点检查：

- Host 是否透传或重写
- `X-Forwarded-*` 是否符合预期
- 超时、重试、连接复用是否被代理层改变

## 高频误区

- “反向代理就是负载均衡”不够准确。负载均衡强调流量分摊，反向代理强调协议入口和转发控制。
- “网关只是加了一层转发”也太浅。网关的重点在统一治理，而不只是多跳一次。
- “有 502 就一定是应用挂了”是误区。很多时候问题在代理到上游的连接、协议、TLS 或超时配置。
- “真实客户端 IP 一定能从连接上看到”也是误区。经过代理后，连接对端常常只剩代理地址。
