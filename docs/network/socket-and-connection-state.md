## 端口、Socket、文件描述符分别是什么

最容易混的是这三个词：

| 概念 | 可以怎么理解 | 关键点 |
| --- | --- | --- |
| 端口 | 进程对外暴露的逻辑入口号 | 只有 `IP + 协议 + 端口` 才足够定位到某类服务 |
| Socket | 一条网络通信端点的抽象 | 既包含本地地址，也包含远端地址和协议状态 |
| 文件描述符（FD） | 进程里引用内核对象的句柄 | 网络连接、文件、管道本质上都可能占用 FD |

工程里更实用的说法是：

- 一个服务可以监听 `0.0.0.0:8080`
- 每来一个客户端连接，内核会为它建立一条具体的连接 Socket
- 应用 `accept` 之后，拿到一个新的 FD 去读写数据

所以“端口被占用”和“连接太多”不是一回事：

- 端口被占用：通常是监听冲突
- 连接太多：通常是 FD、连接池、内存、队列或下游处理能力问题

## 一条 TCP 连接在服务端怎么流转

把服务端看成一条简单流水线会更容易记：

1. 应用先 `listen` 某个地址和端口
2. 客户端发起 `SYN`
3. 内核处理三次握手，把连接放进对应队列
4. 应用 `accept`，拿到一条已建立连接
5. 应用开始读写请求和响应
6. 双方关闭连接，进入一组结束状态

### 监听 Socket 和已连接 Socket

最值得先分清的就是：

- **监听 Socket**：负责“等新连接来”
- **已连接 Socket**：负责“和某个具体客户端收发数据”

因此 `ss -ltn` 看到的是监听入口，`ss -tan` 看到的是所有连接状态，二者不要混为一谈。

### 半连接队列和全连接队列

很多“偶发建连超时”都不在业务代码里，而在这两个位置：

- **半连接队列**：三次握手还没完成的连接
- **全连接队列**：握手已完成，但应用还没及时 `accept` 的连接

当服务进程处理不过来、`backlog` 太小、SYN Flood 防护触发或节点压力过高时，队列积压就会表现成：

- 建连变慢
- 建连超时
- 某些连接看起来随机失败

## 常见 TCP 状态怎么理解

下面这张表不要求死记，但最好能看懂它们各自在说明什么：

| 状态 | 常见含义 | 工程上更该关注什么 |
| --- | --- | --- |
| `LISTEN` | 服务正在监听端口 | 入口是否真的打开了 |
| `SYN-SENT` | 客户端已发起建连，等待对方回应 | 目标不可达、丢包、ACL、安全组、对端没响应 |
| `SYN-RECV` | 服务端收到 `SYN`，握手未完成 | 半连接积压、客户端 ACK 没回来、被攻击或网络不稳 |
| `ESTAB` | 连接已建立 | 数量异常多时要看是否泄漏、慢请求或连接池打满 |
| `FIN-WAIT-1/2` | 本端主动关闭，等待对方完成收尾 | 关闭流程拖长，可能是对方没及时结束 |
| `CLOSE-WAIT` | 对方已关闭，本端还没真正 `close` | 典型应用层连接泄漏信号 |
| `LAST-ACK` | 本端已发 FIN，等待最后 ACK | 通常是关闭尾声，持续很多就不正常 |
| `TIME-WAIT` | 主动关闭方等待旧包自然消失 | 短连接过多、端口消耗、连接复用不足 |

### TIME_WAIT 为什么不能简单当成“异常”

`TIME_WAIT` 的存在本身是正常的，它主要解决两件事：

- 防止旧连接延迟到达的报文污染新连接
- 确保最后一个 ACK 丢失时，对端还能重发 FIN

真正要关心的是：

- 为什么短连接这么多
- 为什么连接复用没有起作用
- 为什么本机临时端口被耗尽

### CLOSE_WAIT 为什么更像代码问题

如果 `CLOSE_WAIT` 持续堆积，通常说明：

- 对端已经关了
- 但你的应用迟迟没有调用 `close`

这经常意味着：

- 连接泄漏
- goroutine / 线程卡住
- 上层协议读写流程没收尾

这类问题通常要回到应用代码、连接池或中间件逻辑看。

## 常见错误和状态异常怎么倒推

把错误文案和状态放在一起看，会更容易定位：

| 现象 / 报错 | 更常见的原因 | 第一反应先查什么 |
| --- | --- | --- |
| `connection refused` | 目标端口没监听，或被 RST 拒绝 | `ss -ltn`、服务进程、LB 后端健康状态 |
| `i/o timeout` / 建连超时 | 网络不可达、ACL、安全组、对端太慢 | `ping`、`traceroute`、`mtr`、`ss` |
| `connection reset by peer` | 对端主动发 RST，中途强制断开 | 服务日志、代理超时、协议不匹配 |
| `address already in use` | 端口监听冲突，或本地端口复用受限 | `lsof -i`、`ss -ltnp` |
| `too many open files` | FD 用尽 | `ulimit -n`、进程 FD 数、连接和文件泄漏 |
| `TIME_WAIT` 很多 | 短连接风暴、连接复用差、上游压测 | keep-alive、连接池、客户端复用策略 |
| `CLOSE_WAIT` 很多 | 应用没及时关闭连接 | 代码路径、连接池释放、阻塞 goroutine |

### 临时端口耗尽是怎么来的

很多人只盯服务端端口，其实客户端也会出问题。

当本机频繁主动建立大量短连接时，会消耗本地临时端口。如果连接关闭后还停留在 `TIME_WAIT`，临时端口回收不够快，就可能出现：

- 新连接建立失败
- 压测时本机先撑不住
- 看起来像“下游挂了”，其实是调用方自己耗尽了资源

## 排障时先看哪些命令

### 看监听和连接状态

```bash
ss -ltn
ss -tan
ss -tanp | head
ss -s
```

重点观察：

- 服务端口是否真的在监听
- 哪类状态数量异常多
- 是客户端建连不成功，还是服务端关闭不及时

### 看谁占用了端口

```bash
lsof -i :8080
```

适合查：

- 端口冲突
- 进程是否真的起来了
- 哪个进程在占用监听端口

### 看进程句柄是否快耗尽

```bash
ulimit -n
ls /proc/<pid>/fd | wc -l
```

这类检查适合在 Linux 上排：

- 连接泄漏
- 文件句柄上限过低
- 服务高峰期突然拒绝新连接

## 高频误区

- “端口开着就代表服务没问题”是误区。监听正常只说明入口还在，不能说明应用处理一定正常。
- “`TIME_WAIT` 多就一定要调内核参数”是误区。很多时候真正的问题是短连接过多、客户端没复用连接。
- “所有连接问题都在网络层”也是误区。`CLOSE_WAIT`、FD 耗尽、accept 不及时往往是应用或运行时行为导致。
