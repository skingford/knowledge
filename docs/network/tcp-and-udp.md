## TCP 和 UDP 各自解决什么问题

可以先用一张表记住它们各自想解决的问题：

| 对比维度 | TCP | UDP |
| --- | --- | --- |
| 核心要解决的问题 | 在不可靠网络上提供面向连接、可靠、有序的字节流传输 | 提供尽量简单、低开销的报文发送能力，把可靠性和顺序控制留给上层 |
| 连接方式 | 面向连接，通信前先建连 | 无连接，直接发包 |
| 可靠性 | 保证到达，依赖 ACK、重传等机制 | 不保证到达 |
| 顺序性 | 保证按序交付 | 不保证顺序 |
| 控制能力 | 有流量控制、拥塞控制 | 没有内建流量控制和拥塞控制 |
| 数据形态 | 字节流，没有天然消息边界 | 报文传输，天然保留消息边界 |
| 开销与时延 | 头部和状态更多，实现更重 | 头部更轻，协议开销更低，通常更适合低时延场景 |
| 更适合的场景 | HTTP / HTTPS、数据库连接、RPC 调用、大多数要求正确性和顺序性的服务端通信 | 音视频实时传输、DNS 查询、某些游戏场景、可容忍少量丢包但更关注实时性的场景 |

一个常见理解方式是 TCP 重"正确"，UDP 重"快"。但不要简单理解成"UDP 一定更快"：如果上层还要自己做重传、排序、补偿，整体复杂度可能更高。

## TCP 连接管理

<InlineSvg src="/network/tcp-handshake.svg" alt="TCP 三次握手与四次挥手" />

### 三次握手

三次握手最关键的不是记住 `SYN -> SYN+ACK -> ACK`，而是知道它确认了什么：

1. 客户端确认自己能发
2. 服务端确认自己能收，也能发
3. 客户端再确认服务端的发送能力和自己的接收能力都没问题

> 三次握手的本质，是双方互相确认"发送能力和接收能力都正常"，并同步初始化序列号。如果少一次，就无法可靠确认双向通信状态；旧连接延迟包或伪造连接也更难处理。

#### 半连接队列和全连接队列

线上建连问题经常会追到：

- `SYN backlog`（半连接队列）：存放尚未完成三次握手的连接
- `accept queue`（全连接队列）：存放已经建立但应用尚未 `accept` 的连接

当这两处堆积异常时，常见表现就是建连超时。所以"三次握手"不能只背协议包，还要知道它和服务端监听队列、连接积压、SYN Flood 防护之间的关系。

### 四次挥手

因为 TCP 是全双工：

- 我不发了，不代表我也不收了
- 对方收到 FIN 后，可能还有数据没发完

所以关闭连接通常是两边分别关闭各自的发送方向，拆成四次交互。

### TIME_WAIT

TIME_WAIT 的价值主要是两件事：

- 防止旧连接的延迟包污染新连接
- 确保最后的 ACK 如果丢了，对方还有机会重发 FIN

线上常见问题不是"为什么有 TIME_WAIT"，而是：

- 为什么 TIME_WAIT 太多
- 为什么短连接风暴导致端口耗尽
- 为什么连接没有复用好

## TCP 可靠传输

<InlineSvg src="/network/tcp-reliable-transmission.svg" alt="TCP 可靠传输机制" />

TCP 的可靠，不是靠"永远不丢"，而是靠一组机制把不可靠网络尽量收敛成可靠连接：

- **序列号**：保证字节流有顺序
- **ACK 确认**：对方收到什么、没收到什么要能反馈
- **超时重传**：超时没确认就重发
- **滑动窗口**：控制未确认数据量，避免把对方压垮
- **流量控制**：接收方通过窗口告诉发送方"我还能吃多少"
- **拥塞控制**：根据网络拥堵情况调整发送速度

### 滑动窗口

最简单的记法是：

- 发送方不能无限发
- 接收方也不能无限吃
- 网络中未确认的数据量需要被控制在一个窗口里

因此窗口大小其实同时反映：接收方处理能力和当前网络连接状态。

### 拥塞控制

最少应知道四个阶段：

- **慢启动**：从小窗口开始指数增长
- **拥塞避免**：到阈值后改为线性增长
- **快重传**：收到三个重复 ACK 立即重传
- **快恢复**：不回到慢启动，直接进入拥塞避免

不要求每次都把算法细节背全，但要知道它们在解决：网络拥堵时不能继续无节制地打满。
