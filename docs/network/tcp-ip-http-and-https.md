## 先看 IP 和路由在做什么

IP 层解决的核心问题是：

- 数据包往哪发
- 下一跳是谁
- 如何跨网段到达目标机器

必须掌握的几个关键词：

- 源 IP / 目标 IP
- 子网掩码
- 默认网关
- 路由表
- NAT
- TTL

一个很实用的理解是：

- TCP 更像“连接和传输规则”
- IP 更像“把包送到目的地的寻路系统”

所以如果根本路由不到，TCP 和 HTTP 再正确也没用。

## TCP 为什么要三次握手

三次握手最关键的不是记住 `SYN -> SYN+ACK -> ACK`，而是知道它确认了什么：

1. 客户端确认自己能发
2. 服务端确认自己能收，也能发
3. 客户端再确认服务端的发送能力和自己的接收能力都没问题

### 半连接队列和全连接队列

线上建连问题经常会追到：

- `SYN backlog` 满了没有
- `accept queue` 堆积没有
- 应用进程有没有及时 `accept`

所以“三次握手”不能只背协议包，还要知道它和服务端监听队列、连接积压、SYN Flood 防护之间的关系。

## TCP 怎么保证可靠传输

可靠传输核心依赖：

- 序列号
- 确认应答
- 超时重传
- 滑动窗口
- 流量控制
- 拥塞控制

### 滑动窗口最值得怎么记

最简单的记法是：

- 发送方不能无限发
- 接收方也不能无限吃
- 网络中未确认的数据量需要被控制在一个窗口里

因此窗口大小其实同时反映：

- 接收方处理能力
- 当前网络和连接状态

### 拥塞控制最少要记住什么

最少应知道：

- 慢启动
- 拥塞避免
- 快重传
- 快恢复

不要求每次都把算法细节背全，但要知道它们在解决：

- 网络拥堵时不能继续无节制地打满

## HTTP 版本演进到底在解决什么问题

### HTTP/1.1

HTTP/1.1 时代，重点是：

- Keep-Alive 连接复用
- 文本协议，易调试
- 同一连接上的并发能力有限

### HTTP/2

HTTP/2 引入：

- 二进制分帧
- Header 压缩
- 多路复用
- Stream 优先级

它主要解决：

- HTTP/1.1 下多个请求并发效率差的问题

但要知道它没有消灭所有阻塞，因为底层仍然是 TCP。

### HTTP/3

HTTP/3 基于 QUIC，核心价值是：

- 把传输控制放到 UDP 之上
- 降低连接迁移成本
- 避免 TCP 层队头阻塞对整条连接的拖累

所以它在移动网络、弱网切换场景里通常更有优势。

## HTTPS 和 TLS 握手怎么理解

HTTPS = HTTP + TLS。

TLS 主要负责：

- 校验证书和身份
- 协商密钥
- 加密传输内容
- 保证完整性

### TLS 握手最值得记住的主线

你至少要能讲到：

1. ClientHello：客户端带上支持的 TLS 版本、加密套件等
2. ServerHello：服务端选择协议参数并返回证书
3. 客户端校验证书链和域名
4. 双方协商会话密钥
5. 后续 HTTP 报文通过会话密钥加密传输

### 排障时最容易忽略什么

很多 HTTPS 问题不只是证书过期，还可能是：

- 证书链不完整
- SNI 配置错误
- 仅支持旧 TLS 版本
- 证书和域名不匹配
- 中间代理把 TLS 终止位置改了

## 一次浏览器请求的完整链路

以访问一个 API 为例，常见链路可以拆成：

1. DNS 解析域名
2. 选定目标 IP
3. 发起 TCP 建连
4. 完成 TLS 握手
5. 发送 HTTP 请求
6. 经过 CDN / WAF / 负载均衡 / 网关
7. 应用服务处理请求
8. 返回响应
9. 客户端决定是否复用连接、缓存内容或继续请求资源

这里每一步都可能单独出问题：

- DNS 慢
- TCP 重传多
- TLS 握手失败
- HTTP 头过大被拒绝
- 网关转发错误
- 上游应用超时

## 高频问题怎么回答

### 1. 为什么三次握手不能只有两次

最短口径：

> 因为必须双向确认发送和接收能力，并且同步初始序列号。两次无法可靠确认客户端是否收到服务端的确认，也无法完整建立双向通信状态。

### 2. 为什么 HTTP/2 还会卡

> 因为 HTTP/2 虽然在应用层实现了多路复用，但底层仍然依赖同一条 TCP 连接。一旦 TCP 层丢包，整个连接上的数据推进都会受到影响。

### 3. HTTPS 到底慢在哪

> 可能慢在 DNS、TCP、TLS、CDN、网关、应用或下游依赖。不能只凭“HTTPS 比 HTTP 多了一层加密”就断言问题在 TLS 本身。
