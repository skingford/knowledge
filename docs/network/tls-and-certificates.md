## TLS 在解决什么问题

TLS 的核心价值可以概括成三件事：

- **机密性**：避免明文传输被窃听
- **完整性**：避免报文中途被篡改
- **身份认证**：让客户端确认“我连的确实是这个服务”

但它不直接解决：

- 业务登录态
- 权限控制
- CSRF / XSS 这类应用层安全问题
- 代码里的越权、注入、逻辑漏洞

所以 HTTPS 不是“安全问题总开关”，它解决的是传输链路可信和加密。

## 证书和信任链怎么理解

最实用的理解方式是：证书不是“加密本身”，而是“身份证明 + 公钥载体”。

### 证书里通常有什么

- 域名信息（现在主要看 SAN）
- 公钥
- 颁发者
- 有效期
- 用途限制
- 数字签名

### 证书链为什么存在

客户端一般不会直接信任某个业务站点证书，而是信任操作系统或浏览器内置的一组根 CA。校验过程通常是：

1. 服务端返回站点证书和中间证书
2. 客户端向上验证签名链
3. 最终链到受信任的根 CA
4. 同时校验证书是否过期、域名是否匹配、用途是否正确

因此“证书看起来没过期但还是报错”很常见，问题可能在：

- 证书链不完整
- 域名不匹配
- 根证书不受信任
- 中间证书缺失

### 域名匹配到底在看什么

现代 TLS 更应该记住的是：

- 主要看证书里的 **SAN**
- 不要再只用旧认知里的 **CN**

如果你访问的是 `api.example.com`，但证书只覆盖 `www.example.com`，就会触发域名不匹配错误。

## TLS 1.2 和 TLS 1.3 握手主线

### TLS 1.2 可以怎么讲

面试或排障里，讲到这条主线通常就够用了：

1. 客户端发 `ClientHello`，带上支持的版本、密码套件、随机数、SNI 等信息
2. 服务端回 `ServerHello`，选择协议参数并返回证书
3. 客户端校验证书链和域名
4. 双方完成密钥协商
5. 后续应用数据用会话密钥加密传输

### TLS 1.3 相比 1.2 主要变化是什么

不必背完整报文，但要知道这些方向：

- 握手轮次更少
- 弱算法被移除
- 密钥协商更现代化
- 0-RTT 成为可选能力，但有重放风险边界

| 版本 | 你最该记住什么 |
| --- | --- |
| TLS 1.2 | 还会看到较多历史兼容场景，排障时经常涉及协议和套件兼容 |
| TLS 1.3 | 握手更快、更简化，很多旧算法和老式配置被淘汰 |

## SNI、ALPN、mTLS 分别在做什么

### SNI

SNI（Server Name Indication）解决的是：

- 同一个 IP / 端口上托管多个 HTTPS 站点时
- 服务端需要在握手早期就知道客户端想访问哪个域名

如果不带正确的 SNI，常见现象是：

- 返回了别的站点证书
- 域名不匹配
- 某些网关场景下直接握手失败

### ALPN

ALPN（Application-Layer Protocol Negotiation）解决的是：

- TLS 握手阶段就协商上层应用协议

典型场景：

- `h2` 对应 HTTP/2
- `http/1.1` 对应 HTTP/1.1

所以“服务支持 HTTP/2 吗”很多时候不只看应用配置，也要看 TLS 协商里是否带了合适的 ALPN。

### mTLS

mTLS（双向 TLS）是在服务端校验客户端证书的基础上，把认证从“只有服务端证明自己是谁”提升到“客户端也要证明自己是谁”。

常见场景：

- 内网服务间身份认证
- 零信任网络接入
- B2B 接口调用
- Service Mesh 东西向流量认证

mTLS 更适合回答“你是谁”，但通常不单独承担全部业务授权逻辑。

## 常见 TLS 报错怎么排

| 现象 / 报错 | 更可能的原因 | 第一反应先查什么 |
| --- | --- | --- |
| `certificate has expired` | 证书过期 | 证书有效期、自动续期链路 |
| `hostname mismatch` / `not valid for` | 域名不匹配 | 访问域名、SAN、SNI |
| `unknown ca` | 根 CA 不受信任或中间证书问题 | 客户端信任库、证书链 |
| `handshake failure` | 协议版本、密码套件、SNI、双向认证配置不兼容 | `openssl s_client`、服务端 TLS 配置 |
| `remote error: tls: bad certificate` | mTLS 场景下客户端证书异常 | 客户端证书、私钥、签发链 |
| `tls: first record does not look like a TLS handshake` | 明明连的是明文端口，却按 TLS 去握手 | 端口、代理终止位置、协议是否对齐 |

### 特别容易忽略的两个点

- **代理终止位置**：TLS 可能在 CDN、SLB、Nginx、网关处被终止，不一定直达应用。
- **证书链完整性**：很多“浏览器能开、代码里报错”的问题，根源就是不同客户端对缺失中间证书的容忍度不同。

## 常用验证命令

### 看握手、证书链、SNI

```bash
openssl s_client -connect example.com:443 -servername example.com
```

重点看：

- 是否真的握手成功
- 服务端返回了哪张证书
- 证书链是否完整

### 看证书详细内容

```bash
openssl x509 -in server.crt -noout -text
```

适合确认：

- SAN
- 有效期
- 颁发者
- Key Usage / Extended Key Usage

### 看客户端侧真实错误

```bash
curl -vk https://example.com
```

适合快速观察：

- TLS 协商是否通过
- 是否命中重定向
- 是证书错、握手错还是应用返回错

## 高频误区

- “HTTPS 就是把 HTTP 加密一下”太浅。真正要讲清的是证书校验、密钥协商和协议协商。
- “证书没过期就说明没问题”是误区。很多问题其实是 SNI、域名匹配或证书链不完整。
- “mTLS 就等于权限控制”也是误区。mTLS 更像连接建立时的强身份认证，不等于细粒度授权。
