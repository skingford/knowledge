# Nginx 核心概念与请求处理模型

> 学 Nginx 最容易卡住的地方，不是某个配置项记不住，而是脑子里没有“请求到底怎么经过它”的统一模型。先把角色、对象和请求流向讲清楚，再学配置会稳很多。

## 1. Nginx 到底解决什么问题

Nginx 最常见的价值不是“运行一个网站”，而是充当整个 Web 请求链路前面的统一入口。

它最常承担这些角色：

- 反向代理：把外部请求转给后端应用
- 静态资源服务：直接返回 HTML、CSS、JS、图片、下载文件
- 负载均衡：把流量分发给多个上游实例
- TLS 终止：处理 HTTPS 握手、证书和加解密
- 基础流量治理：连接限制、请求限流、缓存头、访问控制

可以记住一句最稳的口径：

> Nginx 是一个高性能事件驱动的 Web 服务器和反向代理服务器，常被放在应用前面做统一入口、静态资源分发、负载均衡和 HTTPS 终止。

## 2. 它不是什么

要避免把 Nginx 讲得过重。

- 它不是业务应用服务器，不负责核心业务逻辑
- 它不是数据库，也不负责持久化业务状态
- 它可以做缓存和限流，但不等于完整 API 网关平台
- 它可以代理四层和七层流量，但不等于 Service Mesh

工程上最实用的理解是：

> Nginx 负责接流量、转流量、改一部分请求和响应、做基础治理；真正的业务处理仍在上游应用里完成。

## 3. 先把对象模型分清楚

| 对象 | 它是什么 | 你至少要会什么 |
| --- | --- | --- |
| `master` 进程 | 管理配置、拉起 worker、处理 reload | 知道它不处理业务请求 |
| `worker` 进程 | 真正处理连接和请求的进程 | 知道事件驱动和并发能力主要在这里 |
| `http` 块 | HTTP 配置总入口 | 放日志格式、压缩、upstream、server 等全局配置 |
| `server` 块 | 一个虚拟主机 | 按端口和域名接住请求 |
| `location` 块 | 路径级路由规则 | 根据 URI 匹配具体处理逻辑 |
| `upstream` 块 | 后端服务池 | 描述一组上游实例和负载均衡策略 |
| access log | 访问日志 | 看请求结果、耗时、状态码 |
| error log | 错误日志 | 看配置错误、连接失败、权限和运行异常 |

## 4. `master` / `worker` 模型必须讲清楚

### `master` 做什么

- 读取并校验配置
- 绑定端口
- 拉起和回收 worker
- 接收信号，例如 reload、quit、stop

### `worker` 做什么

- 真正接受客户端连接
- 读取请求
- 做 `server` / `location` 匹配
- 返回静态文件或代理到上游
- 写访问日志和错误日志

这两个结论必须稳：

- 大部分请求处理发生在 worker，不是 master
- 平滑 reload 的本质，是 master 拉起新 worker，旧 worker 处理完手头请求再退出

## 5. 为什么 Nginx 性能高

核心原因不是“它是 C 写的”这么简单，而是事件驱动模型减少了线程/进程切换成本。

可以按这条线解释：

- 一个 worker 不会为每个连接都开一个线程
- 它更像基于事件循环管理大量连接
- 空闲连接不会长期占着大量执行资源
- 静态文件、keepalive、大量短连接场景下收益明显

最不容易答错的说法：

> Nginx 的高并发能力主要来自事件驱动、非阻塞 I/O 和多 worker 进程模型，而不是“一请求一线程”。

## 6. 一次请求是怎么经过 Nginx 的

把下面这条主线背熟，很多配置就不容易写散：

1. 客户端先连到某个 `listen` 端口
2. Nginx 根据 `Host` 和端口选中一个 `server`
3. 再根据 URI 在这个 `server` 下匹配 `location`
4. 匹配结果决定是直接回静态文件、做重写、还是代理到 `upstream`
5. 响应返回客户端，同时写 access log
6. 过程中如果有连接失败、权限错误、配置异常，再写 error log

可以压成一句：

> Nginx 先按端口和域名选 `server`，再按 URI 选 `location`，最后决定本地处理还是转给上游。

## 7. `server` / `location` / `upstream` 的职责不要混

### `server`

重点是回答：

- 这个请求属于哪个站点
- 监听哪个端口
- 绑定哪个域名
- 默认日志、根目录、证书、错误页用什么

### `location`

重点是回答：

- 这个 URI 应该按什么规则处理
- 是返回本地文件，还是代理到后端
- 要不要改写路径、加头、加缓存控制

### `upstream`

重点是回答：

- 后端服务有哪些节点
- 用什么策略转发
- 是否启用 keepalive

一句话区分：

- `server` 选站点
- `location` 选处理规则
- `upstream` 定义后端池

## 8. 反向代理到底是什么意思

“反向代理”最容易被说成抽象名词，其实就做了三件事：

- 对外暴露一个统一地址
- 对内把请求转发到真实后端
- 在中间补充协议、头部、超时、缓存、限流等治理能力

和正向代理的区别：

- 正向代理站在客户端前面，帮客户端访问外部资源
- 反向代理站在服务端前面，帮后端服务统一接流量

Nginx 最典型的场景就是反向代理。

## 9. 静态资源和动态请求为什么常常分开处理

因为它们关注点不同：

- 静态资源更适合 Nginx 直接返回，并加长期缓存头
- 动态请求通常要交给应用处理

所以常见做法是：

- `/assets/`、`/static/` 直接由 Nginx 返回
- `/api/` 反向代理到应用服务

这样做的好处：

- 减少应用服务器压力
- 降低响应链路复杂度
- 让缓存策略更清晰

## 10. 访问日志和错误日志分别看什么

### access log

适合看：

- 某请求有没有进来
- 返回了什么状态码
- 响应耗时大概多少
- 请求路径和来源 IP 是什么

### error log

适合看：

- 配置语法错误
- 权限错误
- 上游连接失败
- 文件不存在或内部错误

排障时一定记住：

> access log 看结果和流量，error log 看异常和失败原因。

## 11. 常见边界和误区

- 以为 Nginx 会自动帮你保留真实客户端 IP，其实通常需要显式转发头或配置 real IP
- 以为只要改了配置就会自动生效，其实需要校验并 reload
- 以为 `server` 和 `location` 谁写前面谁优先，实际上看匹配规则，不看文件顺序
- 以为 502 一定是 Nginx 挂了，很多时候是上游服务不可达或返回异常
- 以为所有静态资源都应该长期缓存，HTML 往往不能和 hash 文件同策略

## 12. 学 Nginx 的推荐主线

1. 先建立对象模型：`server`、`location`、`upstream`
2. 再掌握匹配规则：域名、路径、正则、优先级
3. 再学反向代理：头、超时、WebSocket、负载均衡
4. 再补 HTTPS、缓存、压缩与安全基线
5. 最后用日志、状态码和命令把排障流程固定下来
