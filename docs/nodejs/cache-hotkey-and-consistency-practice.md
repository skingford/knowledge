# Node.js 缓存、热点 Key 与缓存一致性实践

Node.js 服务一旦进入真实高并发场景，数据库和第三方依赖很快就会碰到另一条主线：

- 热点读请求很多
- 同一个对象被反复查
- 某些聚合结果现场算太贵
- 接口一到峰值就先把数据库打穿
- Redis 是上了，但一致性、热点 Key 和回源风暴越来越难控

这时候缓存就会出现。

但缓存真正难的，通常不是 `get/set` 这两个 API，而是：

- 什么数据值得缓存
- 本地缓存、Redis 和两级缓存怎么分工
- TTL 怎么设
- 写入时先改库还是先改缓存
- 缓存穿透、击穿、雪崩、热点 Key 怎么处理
- Node.js 里请求合并、内存边界、事件循环和一致性怎么一起看

这页补的，就是 Node.js 服务里的缓存工程主线。

## 适合谁看

- 已经在做 Node.js API、BFF、NestJS、Express 或 Fastify 服务，开始遇到热点查询、数据库读压和 Redis 治理问题
- 会用 Redis，但对缓存一致性、回源风暴、热点 Key 和多级缓存边界还没有统一模型
- 正在做商品详情、活动页、配置中心、首页聚合、排行榜、用户画像等读多写少场景
- 在高并发方案里经常说“上缓存”，但还说不清什么时候该本地缓存、什么时候该 Redis、什么时候不该做两级缓存

## 学习目标

- 分清缓存在解决延迟、吞吐和下游保护中的具体角色
- 能根据共享范围、热点程度、一致性要求选择本地缓存、Redis 或两级缓存
- 理解缓存穿透、击穿、雪崩、热点 Key 和大 Key 的不同风险与治理动作
- 把 TTL、回填、删除、预热、观测和压测放回一套工程主线

## 先分清：缓存解决的是吞吐和延迟，不是数据建模

缓存通常在换三件事：

- 用空间换时间
- 用复杂度换吞吐
- 用可控不一致换更低延迟

所以更稳的思路不是“哪里慢就先加缓存”，而是先回答：

1. 这个数据是不是明显读多写少
2. 不一致能容忍多久
3. 丢缓存后能不能从数据库或下游重建
4. 这个热点是真的业务热点，还是代码重复查自己制造出来的

如果这些问题说不清，缓存大概率会先把系统复杂度抬上去。

## 先做架构判断：本地缓存、Redis 还是两级缓存

| 方案 | 更适合什么场景 | 主要代价 |
| --- | --- | --- |
| 进程内缓存 | 单实例热点、只读配置、短 TTL 派生结果 | 多实例不共享，重启即失效 |
| Redis | 多实例共享缓存、跨服务共享读模型、统一失效 | 多一跳网络、序列化成本、运维成本 |
| 两级缓存 | 极热点路径，希望把 Redis 压力再降一层 | 一致性、排障和失效复杂度更高 |

可以先记一个很务实的判断：

- 单实例或低并发工具：先别急着上 Redis
- 多实例共享热点读：优先考虑 Redis
- 极热点且读取远高于写入：再考虑本地缓存 + Redis 两级缓存

不要把两级缓存当默认答案。  
它只在热点足够热时才值得。

## 默认优先掌握 cache-aside，不要一开始就追复杂模式

大多数 Node.js 业务服务，先把 cache-aside 做稳就够了。

### 读路径

1. 先查缓存
2. miss 后查数据库或下游
3. 回填缓存
4. 返回结果

### 写路径

更稳的默认策略通常是：

1. 先写数据库
2. 再删除缓存

为什么默认先选这个？

- 直接双写数据库和缓存，顺序竞态更复杂
- “更新缓存”比“删除缓存”更容易把旧值覆盖新值
- 写库后删缓存更容易和事务边界配合

所以第一版通常优先：

- 读用 cache-aside
- 写用“写库后删缓存”

## Key、TTL 和缓存对象边界要先收口

缓存最常出问题的地方，不是命令本身，而是 key 和数据边界。

### 1. Key 设计

至少做到：

- 带业务前缀，例如 `product:detail:{id}`
- 带环境、租户、语言、版本维度时显式编码
- 不把超长 JSON、原始 SQL 直接塞进 key
- key 构造逻辑集中到基础设施层，不要在各处手拼字符串

### 2. TTL 设计

TTL 不应该拍脑袋。

先回答：

- 数据多久允许过期
- 写后最晚多久要看到新值
- 失效时能承受多大回源流量

几个务实建议：

- 热 key TTL 不要完全一样，适当加抖动
- 负缓存 TTL 通常比正常缓存更短
- 没有明确失效策略时，不要随手设超长 TTL
- 如果缓存只是数据库读模型投影，TTL 应服从业务可接受的不一致窗口

### 3. 缓存对象边界

缓存里更适合放：

- 稳定的读模型
- 已经脱离请求上下文的纯数据对象
- 比数据库实体更贴近读取场景的 view model

不要直接把下面这些当缓存对象长期塞进去：

- 带大量无关字段的 ORM 实体
- 非序列化友好的临时对象
- 超大聚合对象
- 请求级上下文对象

## Node.js 里的本地缓存不要默认无边界

Node.js 项目里最常见的本地缓存起手式通常是：

- `Map`
- LRU
- 某个全局对象

这些工具本身没问题，但问题往往出在：

- 没有最大容量
- 没有 TTL
- 没有失效策略
- 多实例下误以为它是共享缓存

结果通常是：

- 内存持续上涨
- 实例之间数据不一致
- 发布、扩容、重启后命中率突然抖动
- 排障时不知道是 Redis 失效还是本地缓存旧值

更稳的本地缓存原则通常是：

- 只缓存明确短生命周期、可重建的数据
- 设置容量上限和 TTL
- 不把本地缓存当最终权威
- 只在极热点路径再考虑两级缓存

## Redis 更适合作为共享缓存边界，不是散落的命令集合

更稳的做法通常是：

- 启动时初始化 Redis client、连接池和序列化策略
- 封成缓存 gateway / repository 风格的基础设施层
- 业务层只调用语义化方法

不要把这些动作散落在 route、service、job handler 各处：

- `get`
- `setex`
- `del`
- `expire`

否则很快会出现：

- key 规则不统一
- TTL 到处不同
- 删除时机说不清
- 排障时不知道谁在写、谁在删、谁在预热

## 高并发最常见的 5 个坑

### 1. 缓存穿透

定义通常是：

- 请求的数据本来就不存在
- 每次都 miss
- 每次都回源

典型应对：

- 参数校验
- 布隆过滤器
- 负缓存
- 对异常 key 做更严格限流

### 2. 缓存击穿

定义通常是：

- 某个热点 key 过期瞬间，大量请求同时回源

典型应对：

- 请求合并 / singleflight 思路
- 热点 key 永不过期 + 异步刷新
- 热 key 提前预热
- 风险接口配合限流和降级

### 3. 缓存雪崩

定义通常是：

- 大量 key 同时过期，或 Redis 整体异常，导致回源风暴

典型应对：

- TTL 加随机抖动
- 多级缓存
- 降级兜底
- 对核心热点做独立保护

### 4. 热点 Key

定义通常是：

- 某个 key 的访问远高于其他 key

常见风险：

- Redis 单点热压
- 缓存失效后回源更猛
- 更新时形成读写争抢

典型应对：

- 本地缓存 + Redis 两级缓存
- 后台异步刷新
- key 分片
- 在入口加限流

### 5. 大 Key

定义通常是：

- 单个 key 挂的数据太大

常见风险：

- 序列化开销大
- 网络传输大
- Redis 慢命令风险增高
- Node.js 请求里解析大对象拖慢事件循环

典型应对：

- 按业务维度拆分对象
- 控制集合规模
- 避免一次性缓存巨型聚合结果
- 需要大对象时优先考虑分页、分片或预处理

## 缓存一致性先抓默认主线：写库后删缓存

缓存一致性的问题，很多时候不是“必须强一致”，而是：

- 不一致窗口是否可接受
- 出现脏值时有没有恢复路径
- 旧值能存在多久

大多数业务服务里，一个够稳的默认策略是：

- 写数据库成功后删除缓存

它仍然不是绝对完美，但比“写库同时更新缓存”更容易把风险收住。

尤其要注意：

- 删除失败要有重试或补偿
- 删除操作最好带日志和观测
- 热点对象更新频繁时，要评估是否需要异步刷新或版本控制

如果你的写路径已经涉及：

- 事务
- 回调
- 队列
- 多系统同步

那缓存一致性就不能单独看，而要和事务边界、Outbox、异步补偿一起看。

## 请求合并、预热和降级比“多加几个 Redis 实例”更常用

很多缓存问题，真正有效的动作不是先扩 Redis，而是先减少无效回源。

### 1. 请求合并

典型思路：

- 对同一个 miss key 的并发请求，只让一个真正回源
- 其他请求等待同一份结果

这对热点 key 击穿很有效。

### 2. 预热

适合：

- 活动页
- 商品详情
- 首页聚合
- 配置中心

不要等流量打上来才临时开始构建缓存。

### 3. 降级

当 Redis 或下游明显异常时，更稳的做法通常是：

- 返回默认值
- 返回旧值
- 只返回核心字段
- 暂时关闭低价值功能

而不是让所有请求一起穿透到底层数据库。

## 4 类典型场景怎么落地

### 1. 商品详情 / 活动页

更适合：

- Redis 缓存
- 热点 key 预热
- 请求合并
- TTL 抖动

如果是极热点：

- 再考虑本地缓存 + Redis

### 2. 配置中心 / 字典数据

更适合：

- 本地缓存
- 后台异步刷新
- Redis 作为共享版本源或兜底

这类数据通常读取极多、变更较少，非常适合做强工程化缓存。

### 3. BFF 聚合接口

更适合：

- 缓存聚合后的读模型
- 对慢下游结果做短 TTL 缓存
- 控制单次请求 fan-out

不要让每次请求都现场把多个依赖重新拼一遍。

### 4. 排行榜 / 统计结果

更适合：

- 预计算
- 定时刷新
- 分片存储
- 读写分离

不适合：

- 每个请求都现场扫描数据库再计算

## 指标与排障怎么做

至少建议观测：

- 缓存命中率
- Redis 延迟
- 热 key 命中分布
- 回源请求数
- 缓存删除失败次数
- 请求合并命中次数
- 大对象序列化 / 反序列化耗时
- 因缓存失效引发的数据库 QPS 抬升

很实用的标签设计包括：

- `cache_name`
- `key_type`
- `operation`
- `result`
- `fallback_reason`

排障时尤其要先分清：

- 是缓存 miss 变多
- 是 Redis 本身变慢
- 是某个热点 key 过热
- 还是 Node.js 在解析大对象时自己被拖慢

如果你想把日志、Tracing 和指标边界单独补齐，继续看：

- [Node.js 配置管理、日志、Tracing 与可观测性实践](./config-logging-tracing-and-observability-practice.md)

## 常见坏味道

- 所有慢接口一律先上缓存，不先做读路径分析
- 本地缓存没有 TTL、没有容量上限
- 进程内缓存被误当成共享缓存
- 热点 key 失效时没有请求合并，直接把数据库打穿
- 更新时到处直接 `set` 缓存，没有统一删除或失效策略
- 把超大对象直接塞进 Redis，再在 Node.js 请求路径里现解析
- 只有 Redis 命令，没有缓存语义层和观测

## 一组很实用的落地原则

1. 先确认数据是否值得缓存，再决定放哪层缓存。
2. 默认先掌握 cache-aside，写路径优先考虑写库后删缓存。
3. 热 key、击穿、雪崩、穿透是四类不同问题，不要混着处理。
4. 两级缓存只给极热点路径，不要默认到处开。
5. 本地缓存必须有 TTL 和容量上限。
6. 请求合并、预热和降级，往往比单纯扩 Redis 更先见效。
7. 缓存治理必须有命中率、回源量和删除失败观测。

## 高频自检题

- 本地缓存、Redis、两级缓存分别适合什么场景
- 为什么默认推荐 cache-aside，而不是一开始就双写缓存
- 缓存穿透、击穿、雪崩、热点 Key 分别怎么区分
- 为什么本地缓存如果没有容量边界，很容易变成 Node.js 内存问题
- 为什么缓存问题常常要和数据库、限流、异步刷新和可观测性一起看

## 一句话收尾

Node.js 缓存实践真正难的，不是“会不会调 Redis”，而是能不能把缓存层次、TTL、热点治理、回源保护和一致性窗口一起收成一条稳定主线。  
只要这条线立住，缓存才会成为高并发放大器，而不是新的事故源。
