# Node.js 数据库、事务与 ORM/Query Builder 实践

Node.js 服务一旦进入真实业务，很快就会碰到这些问题：

- 连接池应该放在哪
- 事务到底由谁发起
- Prisma / TypeORM / Sequelize / Query Builder 到底怎么选
- route、service、repository 谁负责数据库边界
- 为什么有些项目 ORM 用得很快，后面却越来越难改

这页要补的，就是 Node.js 服务里数据库和事务主线。

## 先立一条总原则

**先定事务和分层边界，再选 ORM 或 Query Builder。**

不要先问：

- “现在哪个 ORM 最火”
- “是不是所有查询都应该 ORM 化”
- “是不是写 SQL 就不优雅”

先问：

1. 哪一层负责事务
2. 哪一层负责数据访问
3. 哪些查询适合 ORM，哪些更适合 Query Builder 或 SQL
4. 查询对象、领域对象、响应对象是否分层

## 先分清：连接、事务、模型、查询工具不是一回事

很多工程问题都是因为这几件事混在一起。

### 1. 连接管理

它解决的是：

- 什么时候创建连接
- 怎样复用连接
- 连接池参数怎么控制

### 2. 事务边界

它解决的是：

- 哪些数据库操作必须原子完成
- 谁提交，谁回滚

### 3. 数据模型

它解决的是：

- 数据库存什么
- 应用内部怎么表达

### 4. ORM / Query Builder / SQL

它解决的是：

- 用什么方式访问数据库

如果这四层不分清，后面很容易越写越乱。

## 连接池先别“无感使用”

Node.js 服务里最怕的是：

- 开发阶段一切正常
- 压测或线上一上量，连接池先出问题

至少要先想清：

- 连接池生命周期是否和应用生命周期一致
- API 服务和 worker 是否共用同一套连接口径
- 超时、最大连接数、空闲连接策略是否清楚

一个很实用的原则通常是：

- 连接池是应用级基础设施
- 不要在每个请求里临时创建数据库客户端

## 事务通常应该落在 service 层

比较稳的思路通常是：

- repository 负责单一数据访问动作
- service 负责把多个动作编排成一个业务事务

如果每个 repository 方法里都偷偷提交事务，后面遇到：

- 订单创建
- 库存预扣
- 审计记录
- Outbox 写入

这类跨表一致性流程时，就很难收口。

## Repository 的价值不是“多包一层”

Repository 真正应该解决的是：

- 查询条件表达
- 数据库存取细节封装
- ORM / Query Builder / SQL 差异收口

它不该承担的通常是：

- 复杂业务决策
- HTTP 语义
- 权限判断

一个很实用的判断标准通常是：

- repository 解决“怎么取/怎么写”
- service 解决“为什么现在要这么取/这么写”

## ORM 最大的价值和最大的问题都在抽象

ORM 的价值通常是：

- CRUD 很快
- 模型映射统一
- 一部分事务和关系表达更方便

但风险也很常见：

- 查询语义被藏起来
- 性能边界不清
- N+1 不容易第一时间看出来
- 复杂查询越来越难表达

所以更稳的原则通常不是：

- 全用 ORM

而是：

- 让 ORM 负责适合它的边界
- 复杂查询、批量操作、性能热点允许落回 Query Builder 或 SQL

## 什么时候适合 ORM

ORM 更适合的场景通常包括：

- 常规 CRUD
- 模型关系比较清楚
- 团队希望统一数据访问口径
- 业务迭代速度优先于极致 SQL 控制

## 什么时候更适合 Query Builder 或 SQL

更适合直接写 Query Builder / SQL 的场景通常包括：

- 复杂聚合
- 报表类查询
- 大批量写入 / 更新
- 明显性能热点
- 需要非常清晰地控制锁、索引和查询计划

这不是“反 ORM”，而是把不同工具放回更合适的边界。

## 查询对象、领域对象、响应对象不要混在一起

Node.js 项目里很常见的一种坏味道是：

- ORM model 直接当响应对象
- 请求 DTO 直接当数据库写入对象

短期省事，长期非常危险。

更稳的边界通常是：

- 请求 DTO：只负责输入校验
- 领域对象：只表达业务语义
- 持久化模型：只负责数据库映射
- 响应 DTO / View Model：只负责对外输出

## 常见事务场景先有口径

至少这几类事务场景要先讲清：

### 1. 单请求内多表写入

例如：

- 创建订单
- 写订单明细
- 写审计记录

### 2. 本地事务 + 异步事件

例如：

- 创建订单后写 outbox
- 支付成功后写本地状态并发事件

### 3. 长流程一致性

例如：

- 支付回调
- 库存释放
- 异步补偿

这类问题通常不能靠单个数据库事务解决，需要把：

- 状态机
- 幂等
- 队列 / outbox
- 补偿

一起看。

## 连接、事务和请求上下文要一起看

数据库问题很多时候不只是 SQL 本身，而是：

- 请求上下文没贯穿
- 慢查询打不到具体请求
- 事务超时看不到业务来源

更稳的做法通常是：

- 数据库访问日志带 request id / trace id
- 慢查询和业务日志能对齐
- 事务边界在 service 中显式可见

如果你想把日志、Tracing、Metrics 和告警链路单独补齐，继续看：

## 常见坏味道

- 每个请求里临时 new 数据库客户端
- repository 方法内部偷偷提交事务
- ORM model 直接当请求 DTO 和响应对象
- 复杂报表查询硬塞 ORM，结果谁也看不懂
- SQL 明显是性能热点，但团队仍然不敢落回 Query Builder / SQL
- 慢查询、死锁、超时和具体请求完全对不上

## 推荐实践项目

### 1. 订单服务数据库边界收口

目标：

- 建立 route / service / repository 分层
- 明确事务落点
- 把查询对象、领域对象、响应对象拆开

### 2. 报表查询改造

目标：

- 把复杂查询从 ORM 抽到 Query Builder / SQL
- 对比改造前后的可维护性和性能

### 3. 支付回调与本地状态收口

目标：

- 把幂等、事务和异步事件边界讲清
- 让本地写库和后续异步链路能稳定衔接

## 高频自检题

- 为什么事务通常应该在 service 层收口
- ORM 适合解决什么，什么时候应该退回 Query Builder 或 SQL
- 为什么 repository 的价值不是“形式上多包一层”
- 为什么连接池应该被视为应用级基础设施
- 为什么数据库问题常常要和日志、Tracing、请求上下文一起排

## 一句话收尾

Node.js 数据库实践真正难的，不是“会不会用某个 ORM”，而是能不能把连接、事务、Repository、查询工具和业务边界一起收住。  
只要这条线立住，数据库层才不会从“开发方便”变成“线上不可控”。
