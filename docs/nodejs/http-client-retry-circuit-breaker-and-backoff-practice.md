# Node.js HTTP Client、重试、熔断与退避实践

Node.js 服务一旦进入真实业务，很快就会碰到另一条主线：

- 调第三方 HTTP API
- 调公司内部服务
- 做 BFF 聚合
- 处理超时、认证、重试和降级
- 把外部错误和本地业务错误分开

这件事看起来只是“发个请求”，但工程里真正难的通常不是 API 能不能调通，而是：

- Client 怎么建
- 超时怎么设
- 哪些错误能重试
- 哪些请求必须幂等
- 熔断和退避应该落在哪
- 日志、Tracing 和告警应该记什么

这页补的，就是 Node.js 外部 HTTP 调用的工程边界。

## 适合谁看

- 已经在做 Node.js API、BFF、NestJS、Express 或 Fastify 服务，开始频繁调用第三方接口或内部服务
- 在聚合接口、支付、短信、风控、用户中心、配置中心这类依赖外部 API 的链路上，遇到过超时、重试、雪崩和级联故障
- 会写 `fetch`、`axios` 或 `undici`，但对连接池、超时、重试预算、幂等和熔断边界还没有统一模型
- 想把“下游依赖调用”从散落的工具函数收成一套稳定基础设施

## 学习目标

- 理解 HTTP client 在 Node.js 服务里其实是“下游边界层”，不是临时工具函数
- 知道 client 为什么必须复用，以及 timeout、retry、backoff、breaker 为什么不能各自散写
- 分清哪些错误可以重试，哪些请求不能随便重试
- 把外部 API 调用和高并发保护、可观测性、异步化一起收成同一条主线

## 先分清：这是调用边界，不是语法细节

无论你是在：

- BFF 里聚合多个后端服务
- API 服务里调支付、短信、地图、风控
- worker 里批量访问外部系统

本质上都在做同一件事：

- 从本地进程边界跨到外部系统边界

这意味着你必须显式处理：

- 网络不可靠
- 上游状态码不是你能控制的
- 协议可能漂移
- 超时和重试可能带来副作用

所以这条线真正要收住的，不是 `fetch` 怎么写，而是：

- 连接复用
- timeout
- 错误分层
- 重试预算
- breaker / fallback
- tracing 和观测

## Client 一定要复用，不要每次请求都新建

Node.js 里一个很常见的反模式是：

```ts
export async function getUser(id: string) {
  return fetch(`https://example.com/users/${id}`)
}
```

这段代码在 demo 里能跑，但工程里很容易演变成：

- 连接池配置散落
- timeout 散落
- header 散落
- tracing 散落
- 重试散落

更稳的做法通常是：

- 启动时创建并复用 HTTP client
- 统一配置 keep-alive、超时、认证和基础 header
- 把 client 封在 adapter / gateway 层

如果你在 Node.js 服务里高频调下游，每次请求都新建 client，会更容易出现：

- 连接复用价值丢失
- 连接数量抖动
- 配置无法统一收口

## 把 HTTP 调用停在 adapter / client 边界层

很多 Node.js 项目一开始都会这样写：

- route 里直接 `fetch`
- service 里直接拼 URL
- 各处手工塞 token 和 headers

这会导致：

- URL 规则散落
- 认证逻辑散落
- 错误边界散落
- timeout 和重试策略散落

更稳的方式通常是：

- 定义一个外部 API client / gateway
- 由它负责 URL、认证、超时、状态码映射、响应解析
- 业务层只调用语义化方法

例如：

```ts
export class UserCenterClient {
  constructor(
    private readonly http: HttpClient,
    private readonly baseUrl: string,
  ) {}

  async getProfile(userId: string): Promise<UserProfileDto> {
    return this.http.getJson<UserProfileDto>(`${this.baseUrl}/users/${userId}`)
  }
}
```

这样做的价值是：

- 上游协议变更只改一处
- 认证和 headers 策略只改一处
- 测试时更容易替换依赖
- route / service 不会和外部协议直接耦死

## timeout 必须显式设计，而且要分层

timeout 最容易被误解成：

- “只要设了一个超时，系统就安全了”

其实更稳的思路通常至少分两层：

### 1. Client 级默认 timeout

它解决的是：

- 默认不要无限等待
- 所有下游请求有统一底线

### 2. 单接口覆盖 timeout

它解决的是：

- 某些接口天然更慢
- 某些接口必须更快失败

例如：

- 用户资料读取和风控评分，不一定该用同一个超时
- 短信发送和支付查询，也不应该默认同一个超时

还要注意：

- 请求总超时
- connect timeout
- 响应 body 读取时间

这些并不是完全一回事。

## 错误要分层，不要都抹平成一种失败

外部 API 错误通常至少可以分成四层：

1. 传输层错误
2. HTTP 状态码错误
3. 响应解析错误
4. 上游业务错误

把这四层都抹成一个：

- `request failed`

排障会非常痛苦。

更稳的方式通常是明确区分：

- 是没连上
- 还是超时
- 还是返回了 429 / 500
- 还是 200 但 body 格式变了
- 还是上游业务上明确拒绝

一个很实用的原则是：

- 业务层不要直接依赖原始 HTTP 错误
- adapter / client 层把外部错误翻译成稳定的本地错误类型

## 重试不是默认开关，而是预算策略

很多系统的第二次雪崩，往往不是第一次失败造成的，而是重试放大的。

重试真正要先回答的是：

- 什么错误值得重试
- 最大重试次数是多少
- 每次间隔多久
- 整体重试预算是多少

更稳的做法通常是：

- 只对明显瞬时错误重试
- 限定最多重试次数
- 限定总等待时间
- 让重试和 timeout 一起算总预算

一个反模式是：

- 任何失败都自动重试 3 次

这在高峰和下游抖动时，常常只是把流量放大成第二波洪峰。

## 退避、抖动和熔断为什么必须和重试一起看

如果重试没有退避，结果通常是：

- 所有请求同时失败
- 又在同一时刻再次打过去

所以更稳的做法通常是：

- 指数退避
- 加随机抖动

这样可以减少重试同步放大。

而熔断的价值通常是：

- 当下游已经明显异常时，直接快速失败
- 不再把更多请求继续压给它

可以把它理解成：

- timeout 负责收等待时间
- retry 负责给瞬时错误一次恢复机会
- breaker 负责在故障明显时停止继续放大

如果你只做其中一个，往往不够。

## 幂等边界没立住之前，不要随便重试写请求

这是很关键的一条。

对这些请求，默认要非常谨慎：

- 支付
- 发券
- 下单
- 回调确认
- 三方系统写入

因为你看到本地超时，不代表对方一定没成功。

所以写请求如果要重试，通常必须先想清：

- 是否有幂等键
- 是否有唯一约束
- 是否能查询最终状态
- 是否有补偿或人工兜底

如果这条线还没立住，就不要因为“怕失败”而先加自动重试。

## 高并发下游保护：连接池、并发限制和快速失败

Node.js 调下游时，最常见的误区是：

- 以为问题只在 timeout 和重试

实际上更容易先出问题的是：

- 下游 HTTP client 连接池
- 单请求 fan-out 过大
- 慢下游导致 in-flight 请求堆积
- BFF 聚合把一个用户请求放大成几十个下游请求

更稳的做法通常是：

- client 级并发限制
- 对高风险依赖单独设置熔断 / 降级
- 给不同下游分开资源池
- 对慢依赖优先快速失败

不要让所有下游共享同一套无限制资源。

## 4 类典型接口怎么设计

### 1. BFF 聚合接口

更适合：

- client 复用
- 下游并发上限
- 请求总超时
- 部分依赖失败时降级返回

这里最危险的通常不是入口流量，而是：

- 一个请求放大成多个下游请求

### 2. 第三方短信 / 邮件 / 风控

更适合：

- 较短 timeout
- 幂等或去重机制
- 明确的错误分类
- 必要时异步化

不要把这类慢依赖长期挂在用户交互主路径上。

### 3. 支付 / 回调确认

更适合：

- 最小本地事务
- 谨慎重试
- 查询最终状态
- 明确审计日志

这类链路重点通常不是“请求成功率看起来高”，而是：

- 不重复执行
- 不错误推进状态

如果你想把 webhook、原始 body 验签、重复投递和 2xx / 5xx 返回语义单独理顺，继续看：

- [Node.js Webhook、回调与签名校验实践](./webhook-and-callback-practice.md)

### 4. 后台批量同步

更适合：

- 队列或 worker
- 并发上限
- 重试预算
- 死信和补偿

不适合：

- 在一个 HTTP 请求里直接把外部系统打很久

## 指标、日志和压测怎么验证

至少建议观测：

- 下游请求总量
- 按依赖分组的错误率
- timeout 次数
- retry 次数
- breaker 打开次数
- 平均延迟、P95、P99
- 当前 in-flight 请求数
- 依赖级降级触发次数

很实用的标签包括：

- `dependency`
- `operation`
- `status_class`
- `error_type`
- `retry_result`

日志里至少建议能回答：

- 调的是哪个依赖
- 哪个操作失败
- 用的哪个 request id / trace id
- 超时、状态码还是解码失败
- 有没有发生重试和最终结果是什么

压测时至少验证：

- 下游变慢时，Node.js 是否会快速积压
- 重试是否把压力进一步放大
- breaker 打开后，整体 P99 是否更可控
- 高优先级接口是否仍能存活

如果你想把日志、trace 和指标闭环单独补齐，继续看：

## 常见坏味道

- route 里直接写 `fetch` / `axios`
- 每个地方各自 new client
- timeout、header、认证和 base URL 到处散写
- 所有失败统一自动重试
- 写请求没幂等也直接重试
- 没有退避和抖动，所有重试一起冲下游
- 只统计接口失败，不统计依赖失败和 timeout
- 明明适合异步化，却坚持把慢依赖挂在在线请求里

## 一组很实用的落地原则

1. 把 HTTP 调用当成边界层，而不是工具函数。
2. Client 必须复用，配置必须集中收口。
3. timeout、retry、backoff、breaker 要一起设计，不要零散配置。
4. 默认只重试明显瞬时错误，写请求先确认幂等边界。
5. BFF 和聚合接口必须控制 fan-out 和依赖级并发。
6. 依赖异常时优先快速失败或降级，不要硬扛。
7. 指标和日志必须能区分 timeout、状态码错误、解码错误和上游业务拒绝。

## 高频自检题

- 为什么 HTTP client 应该被视为基础设施边界，而不是随手可写的工具函数
- 为什么只设 timeout 还不够，还要和 retry、backoff、breaker 一起看
- 为什么写请求在没有幂等保障之前不能随便重试
- 为什么 BFF 聚合接口在高并发下特别容易放大下游压力
- 为什么依赖调用问题通常要和连接池、并发限制、trace 和降级一起看

## 一句话收尾

Node.js 外部 HTTP 调用真正难的，不是“请求能不能发出去”，而是能不能把 client 复用、超时、重试、退避、熔断、幂等和观测一起收成一条稳定边界。  
只要这条线立住，下游依赖才不会从“一个 API 调用”变成整个系统的故障放大器。
