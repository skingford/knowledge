# Node.js 幂等、状态机与 Outbox 实践

Node.js 服务一旦进入真实业务，很快就会碰到另一条很容易线上出事故的主线：

- 同一个请求为什么会被重复打进来
- 为什么支付回调会重复
- 为什么状态会被乱改、回退或覆盖
- 为什么“写库成功但消息没发出去”会很危险
- 为什么补偿任务总是最后才想起来补

这些问题表面上像不同专题，实际上都在回答同一件事：

- 一个业务事实如何被稳定地写入、传播、重试和收口

这页补的，就是 Node.js 服务里的幂等、状态机和 Outbox 主线。

## 适合谁看

- 已经在做 Node.js API、NestJS、后台任务或交易链路，开始碰到重复请求、状态乱跳和异步事件不一致问题
- 在支付回调、订单状态推进、发券、库存扣减、通知投递这类链路上，想把幂等和一致性讲清楚
- 会写事务和队列，但还没把状态机、Outbox、补偿和幂等收成一条稳定主线
- 想把“避免重复执行”和“最终一致性”从经验写法提升成工程边界

## 学习目标

- 分清幂等、状态机、Outbox、补偿分别在解决什么问题
- 知道哪些链路该靠数据库唯一约束，哪些该靠状态机条件更新，哪些该靠 Outbox 和异步补偿
- 理解为什么“本地事务正确”并不等于“整条链路最终一致”
- 把重复请求、回调、MQ、延迟任务和补偿放回一张统一执行图里看

## 先分清 4 件不同的事

这几个词经常一起出现，但职责并不一样。

### 1. 幂等

它回答的是：

- 同一请求重复到来时，结果要不要重复执行

它更适合解决：

- 用户重复点击
- 回调重复通知
- 消费重试
- MQ 重复投递

### 2. 状态机

它回答的是：

- 当前状态能不能推进到目标状态

它更适合解决：

- 状态被乱改
- 已终态又被回退
- 并发更新互相覆盖

### 3. Outbox

它回答的是：

- 本地业务写库成功后，异步事件怎么可靠发出去

它更适合解决：

- 写库成功但发消息失败
- 事件丢失后如何恢复
- 消息投递怎么重试

### 4. 补偿

它回答的是：

- 主路径没走完或走歪了，怎么把系统拉回一致

它更适合解决：

- 下游超时
- 回调丢失
- 异步任务失败
- 对账发现长时间不一致

一个很实用的记法是：

- 幂等：重复来了怎么办
- 状态机：这个状态能不能改
- Outbox：事件怎么可靠发
- 补偿：失败后怎么拉回

## 为什么 Node.js 服务更容易把这些边界写散

Node.js 服务端很容易从“先跑起来”开始：

- 一个 controller 里直接写业务
- service 里顺手写事务
- 成功后顺手发 MQ
- 失败了再随手重试

这在早期很快，但业务一复杂就容易变成：

- 状态判断散落
- 幂等键散落
- 重试逻辑散落
- 事件投递散落
- 出问题时根本不知道哪一步真正成功了

所以这条线真正要收住的不是某一个技巧，而是：

- 业务事实先落库
- 状态推进先有规则
- 异步传播先有恢复机制
- 失败收口先有补偿出口

## 幂等先解决“重复来了怎么办”

很多链路的第一类事故，不是请求失败，而是：

- 同一个请求成功执行了两次

常见来源包括：

- 用户连点提交
- 前端超时后重试
- 网关或客户端自动重试
- MQ 至少一次投递
- 第三方回调重复通知

更稳的幂等手段通常有几类：

### 1. 幂等键

适合：

- 明确的业务请求
- 写请求需要跨重试保留同一身份

### 2. 数据库唯一约束

适合：

- 最终落账唯一性
- 用户限领、唯一订单、唯一回调记录

### 3. 条件更新

适合：

- 只有在某个旧状态下才允许推进

### 4. 已处理结果缓存

适合：

- 回调或短时重复请求
- 希望快速返回上一次处理结果

一个很实用的原则是：

- 真正关键的幂等，最终仍要落到数据库或持久化状态上

Redis 可以帮你挡一层，但不要把最终正确性全压在短期缓存上。

## 状态机先解决“状态能不能这么改”

很多业务真正出错时，根因不是重复请求，而是：

- 状态被不合法地推进了

例如：

- 已取消订单又被改成已支付
- 已发放优惠券又被重复发放
- 已完成任务被重新开始

更稳的方式通常是：

- 明确状态集合
- 明确允许的迁移路径
- 用条件更新保证迁移只在合法旧状态上发生

一个很务实的原则是：

- 状态机不是画图好看，而是数据库更新时的真正约束

如果最终还是靠：

- 先查状态
- 再在内存里 `if`
- 再无条件更新

那高并发下还是很容易出并发覆盖问题。

## Outbox 先解决“本地成功了但事件没发出去”

最经典的问题通常是：

1. 本地数据库提交成功
2. 发送 MQ 或事件时失败

于是就出现：

- 业务表里已经成功
- 下游却完全不知道

这就是为什么“写库后顺手发消息”很危险。

Outbox 更稳的核心思路通常是：

1. 在同一个本地事务里同时写：
   - 业务表
   - outbox_event 表
2. 由独立投递器扫描 Outbox 记录
3. 成功投递后再标记为已发送
4. 失败则继续重试

它解决的不是：

- 消息只会被消费一次

它解决的是：

- 本地成功的事件最终不会悄悄丢失

所以消费端仍然必须幂等。

## 本地事务、Outbox 和补偿怎么分工

一个更稳的分工通常是：

### 1. 本地事务

负责：

- 同库多表原子写
- 把本地业务事实和 outbox 记录一起写进去

### 2. Outbox

负责：

- 把“待传播的业务事实”可靠送出

### 3. 消费端幂等

负责：

- 就算收到重复事件，也不重复做错事

### 4. 补偿 / 扫表 / 主动查询

负责：

- 长时间未收敛时再拉回一致

一个很务实的原则是：

- 事务负责本地正确
- Outbox 负责最终送达
- 补偿负责最后收口

## 4 类典型场景怎么设计

### 1. 支付回调

更稳的方式通常是：

- 验签
- 幂等键或唯一回调记录
- 状态机条件更新
- 必要时主动查询补偿

不要看到回调成功一次就直接：

- 无条件改状态
- 然后顺手发一堆事件

如果你想把原始 body 验签、时间戳、防重放和回调入口返回语义单独理顺，继续看：

- [Node.js Webhook、回调与签名校验实践](/nodejs/webhook-and-callback-practice)

### 2. 秒杀 / 抢券结果落账

更稳的方式通常是：

- Redis 预占位
- 数据库最终落账
- 用户维度唯一约束
- 失败时回补和补偿

### 3. 订单创建后发事件

更稳的方式通常是：

- 订单写库和 Outbox 同事务
- 由投递器发 `OrderCreated`
- 下游消费端自己幂等

### 4. 后台任务重试

更稳的方式通常是：

- 任务状态机
- 幂等 handler
- 明确最大重试次数
- 失败后进死信或补偿队列

## 可观测性、扫表和人工补偿怎么落地

这类链路最怕的是：

- 逻辑写了
- 出事时却根本看不见

至少建议观测：

- 幂等命中次数
- 状态迁移失败次数
- Outbox 待发送数量
- Outbox 重试次数
- 长时间 `NEW` / `FAILED` 的事件数量
- 补偿任务执行次数
- 主动查询命中次数

很实用的标签包括：

- `biz_type`
- `state_from`
- `state_to`
- `event_type`
- `retry_result`

还要明确：

- 定时扫哪些异常记录
- 超过多久触发补偿
- 是否需要人工处理后台

一个经验判断通常是：

- 真正关键的链路，如果没有扫表和人工补偿口，就不算闭环

## 常见坏味道

- 把幂等理解成“前端别重复点”
- 所有状态迁移都靠业务代码里的零散 `if`
- 数据库提交成功后直接同步发消息，没有 Outbox 或补偿
- 消费端假设消息不会重复
- 支付回调、活动发券、库存扣减都没有唯一约束或条件更新
- 出现不一致后只能人工查日志，没有补偿任务和异常列表

## 一组很实用的落地原则

1. 幂等、状态机、Outbox、补偿是四条不同职责线，不要混成一个概念。
2. 关键幂等最终要落到持久化约束，不要只靠短期缓存。
3. 状态机要体现在数据库条件更新里，不要只停留在代码里的注释和流程图。
4. 本地写库和异步事件之间，优先考虑 Outbox，而不是“顺手发消息”。
5. 消费端默认要按至少一次投递来设计，幂等不能省。
6. 长流程一致性必须预留扫表、主动查询和人工补偿出口。
7. 关键链路没有观测，就等于没有闭环。

## 高频自检题

- 幂等、状态机、Outbox、补偿分别在解决什么问题
- 为什么“写库后顺手发消息”很危险
- 为什么状态机最终要体现在条件更新和唯一约束上
- 为什么 Outbox 不能替代消费端幂等
- 为什么关键一致性链路一定要有扫表和补偿出口

## 一句话收尾

Node.js 一致性实践真正难的，不是“会不会写事务”，而是能不能把幂等、状态机、Outbox 和补偿一起收成一条可恢复、可观测、可收口的执行主线。  
只要这条线立住，服务才不会从“本地正确”走向“全链路失控”。
