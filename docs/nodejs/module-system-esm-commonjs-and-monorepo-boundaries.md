# Node.js 模块系统、ESM/CommonJS 与 monorepo 工程边界

很多 Node.js 项目真正开始变复杂，往往不是因为业务量先爆炸，而是工程边界先乱了：

- 这个包到底该用 `import` 还是 `require`
- 为什么本地能跑，构建后就报模块找不到
- `type: module` 一开，工具链和脚本全都开始互相打架
- monorepo 看起来很先进，但共享包一多就开始循环依赖
- workspace 能安装成功，不代表边界就是健康的

这页的目标，不是罗列一堆语法差异，而是把 Node.js 模块系统和 monorepo 工程边界讲成一条可落地主线。

## 先立一条总原则

**先定边界，再选模块格式和仓库结构。**

不要先问：

- “现在流行 ESM 还是 CommonJS”
- “是不是所有包都应该搬进 monorepo”
- “是不是上 workspace 就自动工程化了”

先问：

1. 这个包是应用入口，还是共享库
2. 它主要跑在 Node.js、浏览器、测试环境，还是构建工具链里
3. 它需要被谁消费
4. 它应该暴露什么 API，隐藏什么实现

只有这些边界先清楚，模块系统和 monorepo 才不会变成新的混乱来源。

## 先分清：文件、模块、包、workspace、monorepo 不是一回事

很多工程问题都来自这几个层级混在一起。

### 1. 文件

这是最小代码组织单元。

要关心的是：

- 一个文件职责是否单一
- 是否暴露了过多内部实现

### 2. 模块

模块关注的是：

- 一段代码如何被导入和复用
- 它的导出接口是什么

### 3. 包

包关注的是：

- 发布、安装、版本和依赖
- 对外暴露哪些入口

### 4. workspace

workspace 解决的是：

- 多个包如何在一个仓库里统一安装、联动开发和共享锁文件

### 5. monorepo

monorepo 更像一种仓库组织策略。

它不自动等于：

- 包边界清晰
- 依赖关系健康
- 构建链路合理

如果边界没立住，monorepo 只会把混乱放大。

## ESM 和 CommonJS 先别站队，先理解边界

### 1. CommonJS 在解决什么

CommonJS 更贴近 Node.js 早期生态：

- `require`
- `module.exports`
- 运行时加载

它的优点通常是：

- 老生态兼容多
- 工具链和历史包支持成熟

### 2. ESM 在解决什么

ESM 的核心价值通常是：

- 标准化模块格式
- 更清晰的 `import` / `export`
- 更利于静态分析和 tree-shaking

### 3. 工程上真正要关心的不是“谁更先进”

而是：

- 你的运行环境支持如何
- 你的测试工具、构建工具、脚本工具是否和它兼容
- 团队是否会长期同时维护两套导入心智模型

## 最常见的坑不是语法，而是混用

最常见的问题往往是：

- 应用代码用 ESM
- 构建脚本和测试配置还是 CommonJS
- 某些老包只能 `require`
- 某些共享包的导出写法又不一致

结果就是：

- 本地一个命令能跑
- CI 或构建阶段开始报错
- TypeScript 路径映射、运行时解析、打包器解析三套逻辑互相不一致

一个更稳的原则通常是：

- 同一个包内部尽量保持单一模块心智
- 工具脚本和业务代码的格式差异要显式管理
- 不要让“能跑”掩盖“边界不清”

## `package.json` 的边界比很多人想象得更重要

模块系统工程里最容易被低估的，是包元数据本身。

至少要搞清这些字段在控制什么：

- `type`
- `main`
- `module`
- `exports`
- `types`

真正高频的工程问题通常不是“怎么写导出”，而是：

- 对内和对外的入口是否一致
- 是否暴露了不该直接 import 的内部路径
- 构建产物和类型声明是否对齐

## `exports` 不是装饰项，而是边界控制器

很多包一开始为了省事，会默认允许：

- 随便深层 import 内部文件

例如调用方直接写：

- `xxx/dist/utils/internal.js`

短期很方便，长期会非常痛苦，因为：

- 你再也改不了内部结构
- 一改目录就变成破坏性升级
- 调用方会越来越依赖你的实现细节

更稳的做法通常是：

- 用 `exports` 明确公开入口
- 把内部实现留在包边界之后

一个很实用的判断标准是：

- 能不能被外部依赖，不应该由“别人恰好找得到路径”决定
- 应该由你显式声明的导出边界决定

## TypeScript 路径别名不等于运行时模块边界

很多团队在 TS 里配了：

- `paths`
- alias

然后以为模块边界已经清楚了。

其实这只是编译期或 IDE 层面的便利，不自动等于：

- Node.js 运行时也能解析
- 测试工具也能解析
- 构建产物也能解析

所以更稳的原则是：

- 别名用于提升可读性可以
- 但包与包之间的真实边界，最好仍然回到 workspace 包和显式导出上

## monorepo 真正适合解决什么

monorepo 更适合这些场景：

- 多个服务共享一部分类型、工具和基础库
- 需要统一 lint、test、build、release 流程
- 团队确实会长期维护多个相关包

它不太适合解决的，是这些问题：

- 因为一个项目目录有点乱，就想靠 monorepo 重启人生
- 把还没稳定的内部模块过早包化
- 把本该直接留在应用内部的代码也拆成共享包

一个务实原则通常是：

- 共享稳定能力，再抽包
- 不稳定、强耦合、只在单应用里使用的代码，优先留在应用内

## 共享包最容易踩的 4 类坑

### 1. 循环依赖

表现通常是：

- A 依赖 B
- B 又偷偷依赖 A

短期也许还能跑，后面维护成本会越来越高。

### 2. 把应用细节抽成“伪共享能力”

例如：

- 某个包名叫 shared
- 里面其实塞满了某个具体业务应用的耦合逻辑

这类包会越来越大，也越来越难复用。

### 3. 类型共享和运行时代码共享混在一起

更稳的做法通常是分清：

- 纯类型/协议包
- 真正可运行的工具包

### 4. 发布边界不清

要先回答清楚：

- 这些包只在仓库内消费
- 还是未来可能独立发布

这会直接影响：

- 版本策略
- 导出策略
- 构建策略

## 一个够用的 monorepo 思路

很多 Node.js monorepo 真正需要的不是“最先进”，而是“边界稳定”。

一个比较稳的基本思路通常是：

```text
repo/
├── apps/
│   ├── api/
│   └── admin-bff/
├── packages/
│   ├── config/
│   ├── logger/
│   ├── contracts/
│   └── utils/
├── package.json
├── pnpm-workspace.yaml
└── tsconfig.base.json
```

关键点在于：

- `apps/` 放可部署应用
- `packages/` 放真正稳定、可复用的共享能力
- contracts、config、logger 这类边界天然更容易稳定
- 不要把所有“以后可能复用”的代码都提前塞进 `packages/`

- 测试时依赖是否好替换
- monorepo 里共享 DTO、配置、logger 是否健康

## 常见坏味道

- 同一个包里同时混用多套模块心智，但没人知道规范是什么
- `paths` 能跳就到处跳，最后运行时和构建解析全不一致
- shared 包越来越大，实际是“公共垃圾场”
- 调用方深层 import 内部文件，包作者却还以为自己保留了重构空间
- monorepo 里所有代码都抽包，结果改一个功能要跨很多包联动

## 推荐实践项目

### 1. 单仓单服务改成 workspace

目标：

- 把配置、logger、contracts 收敛成独立共享包
- 保持应用层和共享包边界清楚
- 避免出现 shared 大杂烩

### 2. 给现有共享包补 `exports`

目标：

- 收口公开入口
- 禁止深层 import
- 明确类型声明和运行时入口

### 3. 拆一次伪共享包

目标：

- 识别哪些代码根本不该共享
- 把高耦合业务实现移回应用层
- 把真正稳定的协议、工具和配置独立出来

## 高频自检题

- 为什么 monorepo 不会自动带来清晰边界
- `exports` 为什么比很多人想象得更重要
- 为什么 TypeScript `paths` 不能替代真实运行时边界
- 什么样的代码应该抽成共享包，什么样的代码应该留在应用内部
- 为什么模块系统问题经常在 CI、构建或发布阶段才暴露

## 一句话收尾

Node.js 模块系统最难的，从来不是语法细节，而是边界治理。  
只要把包导出、workspace、共享包和应用边界放在一张图里看，ESM/CommonJS 和 monorepo 才会真正服务工程，而不是制造更多工程噪音。
