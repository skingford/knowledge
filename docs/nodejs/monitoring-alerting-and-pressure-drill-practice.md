# Node.js 监控、告警与压测演练实践

很多团队在 Node.js 服务稳定性上会卡在一个很尴尬的阶段：

- 日志打了不少
- Metrics 也接了
- Grafana 面板越来越多
- 活动前也说做过压测
- 真出事时还是靠人肉猜

最后常见的现场往往是：

- 用户先反馈接口慢了，团队才知道出问题
- 大盘看起来很多图，但说不清到底是 event loop、Redis、数据库还是第三方接口先抖
- 告警很多，却不知道哪条该先处理
- 压测只压了主接口，结果真实活动是查询风暴、重复点击和热点 key 一起放大
- 故障恢复后没有把阈值、脚本和预案回写，下一次还会再踩

这页要补的，不只是“监控怎么接”，而是把 Node.js 服务的监控、告警、压测和演练收成一条稳定性闭环。

## 适合谁看

- 已经在做 Node.js API、BFF、NestJS 或活动系统，开始承担线上值守、活动保障或容量评估工作
- 会接日志和指标，但还没把 runtime、依赖、队列和业务指标收成一套判断链
- 正在做高并发接口、秒杀 / 抢券、批量任务、Webhook 或第三方依赖编排，希望把“可观测”继续推进到“可行动”
- 面试或方案评审里经常能说日志、trace、metrics，但说不清压测模型、阈值和演练动作

## 学习目标

- 建立一套更适合 Node.js 服务的稳定性观察模型，而不是只盯 CPU 和 500
- 分清运行时指标、依赖指标、异步链路指标和业务指标分别在解决什么问题
- 能设计更可执行的告警分级、阈值和路由，减少告警噪音
- 能把压测、演练、复盘和阈值回写放回同一条工程主线

## 先讲结论：Node.js 稳定性保障不是图表越多越好，而是越早发现越好

Node.js 服务在线上高峰场景里，很少是“突然一下完全挂掉”。

更常见的劣化顺序通常是：

- event loop delay 先升高
- P99 先抬起来
- 数据库或 Redis 连接池先逼近上限
- 某个第三方 timeout 率先上升
- 队列积压开始增长
- 业务成功率开始下降
- 最后才出现明显 5xx 或大面积超时

所以 Node.js 稳定性保障最重要的原则不是“把监控工具都接上”，而是：

1. 先能在用户投诉前发现异常
2. 先能区分是 runtime、依赖、队列还是业务链路的问题
3. 告警要能驱动动作，而不是只制造噪音
4. 压测和演练要验证保护策略是否真的生效
5. 复盘后要把阈值、脚本和预案回写到系统里

如果你还没把 Node.js 高并发主线理顺，建议先看：

- [Node.js 高并发解决方案实践](./high-concurrency-solution-practice.md)
- [Node.js 配置管理、日志、Tracing 与可观测性实践](./config-logging-tracing-and-observability-practice.md)

## 先把 Node.js 监控问题拆成 4 层

一个更稳的 Node.js 服务，监控通常至少要分成 4 层来看：

1. runtime 和进程层
2. 依赖与资源层
3. 异步链路与一致性层
4. 业务结果层

如果这 4 层混在一起，常见结果就是：

- 看到了错误率，但不知道是不是某个下游先慢
- 看到了 Redis 延迟，但不知道业务成功率是不是已经被拖垮
- 看到了队列积压，但不知道是不是 Outbox sender 卡住还是消费端超时
- 看到了活动成功率下跌，但说不清该先限流、降级还是停某个后台任务

## 一套够用的 Node.js 监控、告警与演练闭环

```text
用户请求 / 任务进入系统
  -> 采集 runtime 指标: QPS / P99 / event loop delay / RSS / heap / GC
  -> 采集依赖指标: DB / Redis / HTTP client / queue latency and errors
  -> 采集业务指标: success rate / reject reason / result state / backlog
  -> 告警引擎按级别触发
  -> 值守按 runbook 做限流 / 降级 / 回切 / 暂停任务 / 扩容
  -> 压测和故障演练验证这些动作是否有效
  -> 活动后复盘并回写阈值、容量模型和预案
```

这条闭环真正想建立的是：

- 先发现
- 再定位
- 再止血
- 再验证
- 最后回写

而不是：

- 出事了
- 大家进群
- 临时看日志
- 靠经验猜

## 第一层：先盯住 Node.js runtime 和进程健康

Node.js 服务和很多其他服务不太一样的地方在于：

- CPU 没完全打满，接口也可能已经明显变慢
- 主线程被拖住时，整批请求会一起排队
- 内存和对象积压会让峰值过去后恢复得很慢

所以 runtime 层至少建议盯这些指标：

- QPS、错误率、P95、P99
- event loop delay
- RSS、heap used、heap total
- GC 次数、GC 耗时
- CPU、load
- 当前 in-flight 请求数
- 进程重启次数
- active handles / active requests（如果你的埋点体系支持）

### 为什么 event loop delay 很值钱

很多 Node.js 线上事故里，最先抖的不是 CPU，而是：

- event loop delay

因为下面这些问题都会先反映到这里：

- 大 JSON 序列化 / 反序列化
- 同步压缩、加密、模板拼装
- 请求内 Promise 链和 microtask 堆积
- 对象堆积导致 GC 抖动

一个很实用的经验是：

- 只盯平均延迟不够，要同时看 P99 和 event loop delay

如果你还没把 event loop、阻塞和 Worker 边界理顺，继续看：

- [Node.js 运行时、事件循环与性能排障](./runtime-event-loop-and-performance-troubleshooting.md)

## 第二层：依赖指标要能定位谁先拖慢了 Node.js

很多 Node.js 服务的真实瓶颈不在 Node.js 本身，而在依赖。

至少建议分开观测这些依赖：

### 1. 数据库层

重点通常包括：

- 连接池使用率
- 获取连接等待时间
- 慢查询数量
- 超时次数
- 事务冲突或死锁数量

一个常见误区是：

- 应用扩容了就以为更稳

但如果每个副本都把连接池开得很大，总连接数反而会更快失控。

如果你想把数据库连接、事务和查询边界单独补齐，继续看：

- [Node.js 数据库、事务与 ORM/Query Builder 实践](./database-transaction-and-orm-practice.md)

### 2. Redis 层

重点通常包括：

- Redis latency
- 命令超时率
- 热点 key 命中情况
- 大 key / 慢命令数量
- 连接数和连接复用情况
- 缓存命中率、回源量、回填失败次数

很多高并发读场景真正先坏的，不是应用实例，而是：

- 热 key 过热
- 回源风暴
- Redis 慢命令把整条链路拖慢

如果你想把缓存、热点 key、Pipeline 和 Lua 边界单独展开，继续看：

- [Node.js 缓存、热点 Key 与缓存一致性实践](./cache-hotkey-and-consistency-practice.md)
- [Node.js Redis 实战：客户端、连接池、Pipeline 与 Lua 脚本边界](./redis-client-pipeline-and-lua-practice.md)

### 3. HTTP Client / 第三方依赖层

重点通常包括：

- client latency
- timeout rate
- retry rate
- breaker open 次数
- status code class
- dependency concurrency

一个非常常见的问题是：

- 入口接口看起来没什么问题
- 其实是下游第三方 timeout 开始把请求慢慢堆住

如果你想把 HTTP client、重试、熔断和退避单独理顺，继续看：

## 第三层：异步链路和一致性指标不能缺席

Node.js 服务只看在线请求指标是不够的。

因为很多问题最后会堆在：

- 队列
- 后台 worker
- Outbox sender
- 定时扫表
- 补偿任务

至少建议观测：

- queue backlog
- oldest message age / queue lag
- consumer latency
- retry count
- dead-letter count
- job timeout count
- Outbox pending 数量
- Outbox send failed 数量
- 长时间停留在某个状态的记录数

很实用的标签包括：

- `queue_name`
- `job_type`
- `retry_result`
- `event_type`
- `state`

如果这层没有建起来，很容易出现：

- 入口已经返回“处理中”
- 但真正的消费链路早就卡住了
- 业务成功率下降却不会立刻体现在 5xx 上

如果你想把异步任务和一致性主线单独理顺，继续看：

## 第四层：业务指标决定你有没有真正发现故障

很多服务只盯技术指标，不盯业务结果。

这会带来一个问题：

- 图没红
- 用户其实已经在失败

Node.js 高并发业务里，业务指标通常比机器指标更早暴露问题。

### 秒杀 / 抢券场景

建议至少盯：

- 资格通过率
- 限流拒绝率
- Redis 预扣减成功率 / 回补量
- 排队中数量
- 最终成功率
- 结果查询成功率
- 用户重复点击率

### 支付回调 / Webhook 场景

建议至少盯：

- 验签失败率
- 幂等命中次数
- 回调处理成功率
- 状态推进失败率
- 查询补偿次数

### 导出 / 批量任务场景

建议至少盯：

- 任务提交成功率
- 平均等待时长
- 处理成功率
- 超时任务数量
- 人工补偿数量

一个很实用的原则通常是：

- 业务指标要能回答“用户现在到底有没有得到正确结果”

## 告警设计：先区分快速止血告警和趋势风险告警

告警做不好，监控基本就只能停留在“图很多”。

一个很实用的思路通常是把告警分成两类：

### 1. 快速止血告警

特点通常是：

- 需要马上处理
- 直接影响可用性
- 很可能要立刻执行限流、降级或开关动作

典型例子包括：

- 错误率短时间内突增
- P99 明显失控
- event loop delay 持续高于安全阈值
- 数据库连接池接近打满
- Redis timeout 率明显升高
- 队列 backlog 快速放大
- breaker 大面积打开

### 2. 趋势风险告警

特点通常是：

- 不一定马上止血
- 但说明系统正在靠近边界
- 值守需要提前观察和准备动作

典型例子包括：

- cache hit rate 持续下降
- Outbox backlog 持续升高
- 某个活动的重复点击率异常升高
- 查询接口流量开始放大到主链路数倍
- GC 耗时和 RSS 缓慢上涨

### 告警设计的几个实用原则

1. 告警要直接带路由、依赖、队列、活动或业务类型标签。
2. 告警文案最好直接带建议动作，例如限流、切降级、暂停某个 consumer。
3. 同类告警要做聚合，避免单次故障刷出几百条。
4. 不要把完整错误栈当标签，否则指标基数会失控。
5. 没有 runbook 的告警，通常都很难在高压现场真正发挥价值。

## 压测设计：别只压一个接口，要还原真实流量结构

很多团队说“做过压测”，实际只做了：

- 压一个主接口
- 看一下平均 RT
- 没报错就算通过

这对 Node.js 高并发系统通常远远不够。

更稳的压测模型通常至少要还原这些行为：

- 活动页、详情页和主接口是一起放大的
- 用户会重复点击
- 结果查询会在主链路成功前持续轮询
- 热点 SKU / 券模板会高度集中
- 某些接口会在高峰前先预热
- 峰值过去后会有延迟消费和恢复阶段

### 压测至少建议验证什么

- 峰值流量下 event loop delay 是否仍在可控范围
- in-flight 请求数是否持续堆高
- 数据库 / Redis / HTTP client 连接数是否被限制在安全边界内
- 限流、timeout、breaker 和快速失败是否真的触发
- 队列 backlog 是否能在活动结束后收敛
- 峰值过去后 RSS、heap 和 GC 是否能恢复

### 一个很值钱的 Node.js 视角

压测结果不要只看“峰值有没有顶住”，还要看：

- 尾延迟是不是已经明显失控
- event loop delay 有没有提前报警
- 峰值过去后进程是不是恢复很慢
- 查询风暴和补偿任务会不会把系统二次拖慢

## 演练设计：把常见故障变成预演过的场景

对 Node.js 服务来说，最值钱的演练不是“随机搞点 chaos”，而是优先演练那些最常见、最容易放大成事故的故障。

### 1. Redis 延迟尖刺或热点 key 过热

要验证：

- 本地缓存或降级是否接得住
- 回源量会不会把数据库一起拖慢
- 限流和快速失败是否生效

### 2. 数据库连接池饱和或慢查询放大

要验证：

- 应用是否能快速失败，而不是无限排队
- 高优先级接口是否有资源隔离
- 异步任务是否会被及时降速或暂停

### 3. 第三方依赖 timeout 或高错误率

要验证：

- client timeout 是否足够短
- 重试是否受预算约束
- breaker 是否能打开
- 是否有降级结果或异步补偿路径

如果你想把 webhook 回调入口的验签、重试语义和异步收口单独理顺，继续看：

- [Node.js Webhook、回调与签名校验实践](./webhook-and-callback-practice.md)

### 4. 队列 backlog、consumer timeout 或死信上升

要验证：

- backlog 告警是否及时
- consumer 并发是否可调
- 死信是否有人工处理口和回放策略
- 入口是否需要收缩放量

### 5. Outbox sender 卡住或补偿任务停摆

要验证：

- 长时间 pending 是否会被告警
- 主流程是否有查询或人工兜底
- 复盘时能否明确积压时长和影响范围

### 6. 应用自己阻塞 event loop

要验证：

- event loop delay 告警是否先于 5xx 出现
- 是否能快速定位到发布版本、路由或某类大对象处理
- 回滚、降级或限流动作是否有明确 runbook

一个很实用的原则通常是：

- 演练一定要带动作，不要只证明“确实会坏”

## 活动后和发布后：一定要做阈值回写与容量回归

很多团队稳定性建设卡住，不是因为没有监控，而是因为：

- 每次活动都重新紧张一遍
- 每次发布都重新猜一遍阈值

更稳的做法通常是活动后和发布后都回写这几类信息：

- 真实峰值 QPS、P99、event loop delay 峰值
- Redis、数据库、队列和第三方依赖的实际安全水位
- 最有效的限流、降级、回切动作
- 原有阈值哪里过松或过紧
- 哪些告警噪音太大，哪些关键指标缺失

## 常见坏味道

- 只盯 CPU、内存和 500，不看 event loop delay、P99 和 backlog
- 技术指标很多，但业务成功率、结果态和失败分类没有建起来
- 压测只压主接口，不压结果查询、重复点击和热点集中
- 告警很多，但没有分级、没有聚合、没有动作建议
- 队列、Outbox、补偿任务没有指标和告警
- 只在服务彻底挂掉后才有人知道，缺少前置信号
- 故障演练只做一次展示，不沉淀脚本、阈值和 runbook

## 推荐实践项目

### 1. 给现有 Node.js 服务补一套稳定性大盘

目标：

- 补齐 runtime、依赖、队列和业务四层指标
- 让关键接口具备 route、dependency、queue 和 result_state 标签
- 为 event loop delay、连接池和 backlog 设计最小可用告警

### 2. 给热点活动链路补一套真实压测模型

目标：

- 把主接口、结果查询和重复点击一起建模
- 验证限流、降级、timeout 和异步削峰是否真实生效
- 给出活动前安全水位和停止放量阈值

### 3. 给异步链路补一次故障演练

目标：

- 选一个 Redis、数据库、第三方依赖或 MQ 故障场景
- 验证告警、runbook 和止血动作是否有效
- 把演练结果回写到值守手册和阈值配置里

## 高频自检题

- 为什么 Node.js 稳定性建设不能只盯 CPU 和 500
- 为什么 event loop delay、P99 和 backlog 经常比 5xx 更早暴露问题
- 告警为什么要分成快速止血告警和趋势风险告警
- 为什么压测不能只压一个主接口，还要模拟重复点击和结果查询
- 为什么异步链路、Outbox 和补偿任务没有指标，就不算真正闭环

## 一句话收尾

Node.js 监控、告警与压测演练真正要解决的，不是“图表够不够多”，而是能不能在用户大规模感知之前发现问题、定位问题、执行动作并把经验回写成下一次更稳的默认配置。  
只要这条闭环立住，Node.js 服务的高并发治理才算真正进入工程化阶段。
