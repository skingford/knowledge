# Node.js 队列、定时任务与 Worker 实战

很多 Node.js 服务写到一定阶段，都会遇到一个明显拐点：

- 发邮件、短信、通知
- 批量导出
- 同步第三方系统
- 延迟取消订单
- 周期刷新配置
- 清理过期数据

这些工作和 HTTP 请求生命周期完全不是一回事。

如果继续把它们硬塞进接口里，常见结果就是：

- 请求超时和后台任务耦在一起
- 重试逻辑散落在各处
- 服务重启时任务直接丢
- 多副本部署后重复跑、并发跑、漏跑

这页补的，就是 Node.js 服务里的异步任务主线。

## 先分清 4 种执行模型

### 1. 请求内并发

典型场景：

- 一个请求里并发查多个下游
- 聚合多个服务结果后再返回

这种任务仍然属于当前请求生命周期。  
它不是“后台任务”。

### 2. 进程内轻量后台任务

典型场景：

- 周期刷新本地缓存
- 低价值、可重建的异步通知
- 本进程内串行消费轻量任务

适合前提通常是：

- 任务丢失成本可接受
- 不要求跨重启保留
- 不需要独立扩容

### 3. 持久化队列消费

典型场景：

- 失败后必须重试
- 服务重启后不能丢
- 需要积压、死信、延迟重试治理

这时通常要上：

- Redis / BullMQ
- MQ
- 数据库任务表

### 4. 独立 worker 服务

典型场景：

- 任务执行时间明显长于 HTTP 请求
- 需要和 API 服务分开扩缩容
- 消费吞吐和接口吞吐应该分离

一个很实用的判断标准是：

**如果任务需要跨实例接力、跨重启保留、失败重试和消费治理，它就已经不是简单的 `setTimeout` 或 `Promise` 问题了。**

## `setTimeout` 和 `cron` 不是任务系统

很多项目一开始都会这么写：

- 延迟任务就 `setTimeout`
- 周期任务就 `cron`

这在 demo 阶段没问题，但它们没有自动解决：

- 服务重启后的任务恢复
- 多副本部署后谁来跑
- 重试、死信、幂等
- 任务积压与背压
- 优雅关闭与收尾

所以更稳的原则通常是：

- 触发方式和执行治理要分开
- scheduler 负责“什么时候发任务”
- worker 负责“任务怎么执行”

## 什么时候还可以只用进程内任务

下面这些场景通常还适合进程内轻量方案：

- 刷新本地只读缓存
- 写低价值埋点
- 失败后可以重新触发的辅助任务
- 任务作用范围只在当前实例内

但前提通常是：

- 丢了也能补
- 重启不致命
- 不要求跨实例协同

## 什么时候应该升级成持久化队列

出现这些信号时，通常该升级：

1. 任务执行时间可能很长
2. 失败后必须重试
3. 服务重启不能丢
4. 多副本之间需要接力消费
5. 需要看积压、消费延迟、死信数量

这时更该先想的是：

- 任务数据模型
- 幂等键
- 重试策略
- 死信策略

而不是只想“consumer 怎么写”。

## 队列设计先于 handler 设计

后台任务真正先要想清楚的，通常不是处理函数本身，而是任务边界。

### 1. payload 尽量最小化

更稳的 payload 通常只传：

- 业务主键
- 幂等键
- attempt 信息
- trace / request 关联字段

而不是把整个巨大对象直接塞进队列。

### 2. 先定义幂等键

任何会重试的任务，都要提前回答：

- 同一个任务重复执行会不会出错
- 如何判断“已经成功处理过”

常见方式包括：

- 业务唯一键
- 去重表 / 幂等记录
- 条件更新 / 乐观锁

### 3. 重试要分可重试和不可重试

更稳的思路通常是：

- 网络抖动、临时超时：可重试
- 参数非法、业务状态不允许：不可重试

如果这两类不区分，队列只会不断制造噪音失败。

### 4. 死信要能落地处理

死信不是“扔到一个地方就算完”。  
至少要回答：

- 谁看
- 怎么告警
- 能不能补偿
- 需要人工介入还是自动修复

## 定时任务先分清语义

很多团队最容易混掉的，是这些东西：

- interval
- cron
- 延迟任务
- 队列消费
- 单实例调度

它们分别解决的问题并不一样。

### interval / fixed delay

更像：

- 每隔多久做一次

适合：

- 刷缓存
- 周期同步轻量状态

### cron / 日历型调度

更像：

- 每天几点
- 每周几几点

适合：

- 报表
- 对账
- 清理任务

### 延迟任务

更像：

- 某一条业务在未来某个时间点执行

例如：

- 30 分钟未支付自动取消
- 5 分钟后重试回调

延迟任务很多时候更适合：

- 延迟队列
- 任务表 + 扫描

而不是只靠进程内 `setTimeout`。

## 多副本部署下，谁来跑定时任务

多副本环境里，最容易出事故的问题就是：

- 到底谁触发任务

更实用的判断通常是：

- 每实例私有任务：都可以跑
- 全局任务：应该只有一个调度者负责发任务

比较稳的做法通常不是：

- 某个实例里写个内存布尔值

而是：

- 基于租约 / 锁 / 任务表状态的单实例调度

## 优雅关闭是后台任务的高频坑

Node.js 服务关闭时，最怕的是：

- HTTP 不再接新流量了，但后台任务还没收尾
- consumer 处理到一半被硬杀
- 任务 ack / commit 时机不清楚

至少要提前想清：

- 收到 `SIGTERM` 后是否停止拉取新任务
- 已在处理中的任务如何收尾
- 超过多长时间强制退出
- 是否需要把任务状态回滚或重试

## 队列和请求链路怎么分工

一个很稳的思路通常是：

- HTTP 请求只做快速确认
- 真正的重任务异步化
- 结果通过状态查询、回调或通知返回

不要让接口既承担：

- 用户交互
- 重计算
- 补偿重试
- 长时间等待第三方结果

否则接口和任务系统会一起失控。

## 和 NestJS 怎么衔接

如果你在 NestJS 里做这些事：

- `@Cron()`
- BullMQ consumer
- 导出 / 回调 / 通知任务

更稳的方式通常是：

- Module 里清楚隔离 API、scheduler、consumer
- Service 层承接真正业务逻辑
- 任务 handler 不直接耦合 HTTP 请求对象

## 常见坏味道

- 导出、压缩、对账直接放在 HTTP 请求里同步做
- 定时任务只靠单实例内存状态保证“不重复”
- 队列任务没有幂等键
- 所有失败都无限重试
- 死信没有告警和处理流程
- 服务关闭时 consumer 直接中断，没有收尾策略

## 推荐实践项目

### 1. 邮件 / 短信异步通知服务

目标：

- 把通知从 HTTP 请求里拆出来
- 设计重试、幂等和失败告警
- 区分可重试错误和不可重试错误

### 2. 订单超时取消链路

目标：

- 设计延迟取消任务
- 明确调度、执行、状态机和补偿逻辑
- 让多副本部署下只会按预期触发

### 3. 导出任务中心

目标：

- 把大导出从请求线程迁移到异步 worker
- 支持任务状态查询
- 支持优雅关闭与失败恢复

- 为什么 `setTimeout` 和 `cron` 不能自动解决可靠任务执行问题

## 一句话收尾

Node.js 异步任务真正难的，不是“把它异步化”，而是把触发、执行、重试、幂等、死信和关闭主线一起设计。  
只要这条线立住，后台任务才不会从“先能跑”变成“线上事故源”。
