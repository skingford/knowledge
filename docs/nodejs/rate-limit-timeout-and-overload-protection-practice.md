# Node.js 限流、超时与过载保护实践

很多 Node.js 服务的问题，不是“业务不会写”，而是压力一上来以后不会保护自己：

- 某个接口一慢，整个服务都跟着抖
- 数据库连接先满了，CPU 还没满
- 明明配了超时，还是把下游越打越慢
- 请求不是很快失败，而是先排队很久再失败
- 活动流量一来，日志里全是重试和超时，却不知道该先收哪里

这类问题的共同点是：

**它们不只是性能问题，而是服务过载时的边界设计问题。**

这页补的，就是 Node.js 服务在压力下怎么保护自己。

## 适合谁看

- 已经在做 Node.js API、BFF、NestJS、Express 或 Fastify 服务，开始遇到峰值流量、慢下游和队列积压问题
- 能说缓存、队列、扩容，但对 rate limit、concurrency limit、timeout、backpressure 之间的区别还不够清楚
- 正在设计登录、搜索、抢券、导出、Webhook、第三方 API 聚合这类容易被流量或慢依赖打穿的接口
- 想把“过载保护”从几个零散中间件升级成一套工程边界

## 学习目标

- 分清限流、并发限制、超时、排队、快速失败分别在解决什么问题
- 知道哪些保护该放在网关、哪些放在路由层、哪些放在下游 client 和队列消费者边界
- 理解 Node.js 为什么特别容易因为请求堆积、连接池占满和 event loop delay 上升而出现级联放大
- 能为热点接口、慢依赖接口和后台任务设计更清晰的保护组合

## 先分清 5 件不同的事

这几个词经常一起出现，但职责并不一样。

### 1. timeout

回答的是：

- 一个请求、本地处理段或下游调用，最多等多久

它解决的是：

- 慢请求长期占住连接和内存
- 失败边界不清楚

它不自动解决的是：

- 下游一定停止执行
- 写操作一定没有副作用
- 总体吞吐一定稳定

### 2. concurrency limit

回答的是：

- 同一时刻最多允许多少个 in-flight 请求或任务进入某段逻辑

它更像：

- 给数据库连接池、Redis、CPU 重任务、第三方接口前面加一道闸门

### 3. rate limit

回答的是：

- 单位时间内最多允许多少请求进入

它更适合：

- 平滑突发流量
- 保护登录、短信、搜索、抢券这类容易被瞬时冲击的接口
- 做用户级、租户级、接口级配额

### 4. backpressure / 有界排队

回答的是：

- 当下游暂时处理不过来时，请求是先短暂排队，还是把压力往上游传回去

它决定的是：

- 过载时系统更偏向“延迟增加”还是“快速拒绝”

### 5. load shedding / 快速失败

回答的是：

- 当服务已经处理不过来时，要不要直接拒绝一部分新请求

它通常适合：

- 你已经明确知道，再排队只会让整体更糟
- 你宁愿部分请求立刻失败，也不想把所有请求一起拖死

## 不要把这些策略当成“同一把锤子”

一个很务实的对应关系通常是：

- `timeout`：收等待时间
- `concurrency limit`：收在途量
- `rate limit`：收进入速率
- `backpressure`：控制排队与传播压力
- `load shedding`：过载时优先拒绝

如果这几件事混在一起，典型结果往往是：

- 明明该控并发，却只配了超时
- 明明应该快速失败，却在应用层偷偷排了很长的队
- 明明需要全局配额，却误把单实例限流当分布式限流
- 明明需要保护下游 client，却只在 Web 框架入口加中间件

## Node.js 为什么更需要过载保护视角

Node.js 服务在压力下，最容易先出现的常常不是“CPU 打满”，而是：

- event loop delay 上升
- 请求对象在内存里堆积
- 数据库 / Redis / HTTP client 连接池先满
- 慢下游引发 Promise 堆积和等待链放大

这意味着 Node.js 里的过载保护，重点通常不是“榨干最后一点 CPU”，而是：

- 不让慢请求无限占着资源
- 不让请求内 fan-out 把下游放大
- 不让低价值请求拖垮高价值接口
- 不让排队把延迟慢慢变成系统性雪崩

## 保护应该放在最靠近风险边界的地方

更稳的原则通常是：

### 1. 网关 / 接入层压力

优先放在：

- CDN / WAF / API Gateway
- 用户级、租户级、接口级流控
- 防刷、签名校验、黑名单

### 2. Web 路由压力

优先放在：

- Express / Fastify / NestJS 的路由或模块边界
- 高风险接口的 route-level 保护
- 请求总超时、局部并发上限、快速失败

### 3. 下游 client 压力

优先放在：

- 数据库访问层
- Redis client
- 第三方 HTTP client
- RPC / SDK adapter 层

这里更需要的是：

- 下游调用超时
- client 级并发限制
- 熔断、退避、快速失败

### 4. 后台 worker / 队列压力

优先放在：

- 队列消费者并发数
- 本地有界任务池
- 任务超时、重试预算和死信

所以不要把“负载保护”狭义理解成：

- “给 HTTP 服务挂几个 middleware 就结束了”

## 常见保护组合：入口、路由、下游 client、队列消费者

一个更稳的分层组合通常是：

### 1. 入口层

- 网关 rate limit
- 基础防刷
- 热点接口独立配额

### 2. 路由层

- 请求总超时
- route 级 concurrency limit
- 对导出、搜索、报表等接口做更严格保护

### 3. 下游 client 层

- 外部 HTTP 调用 timeout
- 第三方接口并发上限
- 数据库连接池前的容量控制

如果你想把第三方 HTTP client、错误分层、重试预算、退避和熔断单独展开，继续看：

- [Node.js HTTP Client、重试、熔断与退避实践](./http-client-retry-circuit-breaker-and-backoff-practice.md)

### 4. 队列和 worker 层

- 消费并发上限
- 单任务处理超时
- 有界积压和重试预算

真正要避免的是：

- 入口无限放行
- 应用无限 fan-out
- 下游无限等待
- 队列无限重试

## timeout 不是“系统已经安全了”

这是最常见误区之一。

timeout 真正保证的是：

- 本地不会无限等待

它不保证：

- 下游没有继续执行
- 写请求没有部分成功
- 重试不会造成重复副作用

所以对有副作用的接口，timeout 必须和这些一起看：

- 幂等键
- request id / trace id
- 结果查询
- 状态机
- 补偿或人工兜底

典型场景包括：

- 支付回调
- Webhook
- 发券
- 扣库存

## concurrency limit 和 rate limit 不要互相替代

很多服务只加其中一个，然后以为够了。

实际上它们解决的问题不同。

### concurrency limit 更像容量保护

更适合保护：

- 数据库连接池
- CPU 重任务
- 大对象响应
- 单实例可承受的在途量

### rate limit 更像流量整形

更适合处理：

- 短时间突发流量
- 登录 / 验证码 / 搜索
- 明确的每秒放行上限

一个很务实的组合通常是：

- 在入口或路由层做 rate limit
- 在风险依赖前做 concurrency limit

## 本地限流不是全局配额

这一点必须讲清。

如果你在单个 Node.js 实例上设置：

- 每秒 100 请求

那在 10 个副本上，更接近于：

- 每个实例各自 100

而不是：

- 整个集群统一 100

所以如果业务要求是：

- 用户级全局频控
- 租户级配额
- 活动库存的集群级闸门

那通常应该优先考虑：

- 网关统一控制
- Redis / 共享状态方案
- 专门的分布式令牌桶或滑动窗口

不要把进程内 rate limit 误当成分布式限流。

## 排队不是默认更安全，很多时候应该快速失败

很多团队一过载就本能地“先排队”，但排队并不总是好事。

当下面这些条件成立时，快速失败往往更稳：

- 下游已经明显变慢
- 队列长度不可控
- 请求本身时效性很高
- 用户重试行为会进一步放大流量

因为这时继续排队，通常只会得到：

- 更高的延迟
- 更多的超时
- 更多堆积对象
- 更慢的恢复时间

更适合短暂排队的，通常是：

- 下游只是瞬时抖动
- 队列是有界的
- 调用方能接受短时等待
- 任务本身比实时交互更适合异步化

一个很实用的原则是：

- 在线接口优先考虑清晰拒绝
- 后台任务优先考虑有界排队

## 4 类典型接口怎么配保护

### 1. 登录 / 验证码 / 搜索

更适合：

- 用户级或 IP 级 rate limit
- 较短 timeout
- 明确的快速失败

不适合：

- 把所有请求先放进长队列再说

### 2. 聚合型 BFF 接口

更适合：

- 请求总超时
- 下游 client timeout
- 请求内 fan-out 并发上限
- 熔断和降级返回

这里最危险的点往往不是入口流量，而是：

- 一个请求打太多下游

### 3. 导出 / 报表 / 大结果集

更适合：

- 路由级并发限制
- 更严格的请求体积和结果体积边界
- 直接异步化成任务中心

不适合：

- 让几十秒任务长期占着在线请求资源

### 4. Webhook / 支付回调 / 三方回调

更适合：

- 验签 + timeout
- 幂等键
- 最小本地事务
- 后续异步化

重点通常不是极致吞吐，而是：

- 不重复处理
- 不无限等待
- 不把回调慢依赖放大到主链路

## 指标与压测怎么验证保护是否生效

至少建议观测：

- 请求超时次数
- route 级 rate limit 命中次数
- concurrency limit 拒绝次数
- 快速失败次数
- 当前 in-flight 请求数
- 数据库连接池使用率
- 外部 HTTP client 超时率
- event loop delay

很实用的标签设计包括：

- `route`
- `dependency`
- `reject_reason`
- `status_class`

更稳的 `reject_reason` 例如：

- `timeout`
- `rate_limited`
- `overload`
- `dependency_slow`

不要把完整异常字符串直接当指标标签。

压测时至少验证：

- 峰值流量下，是快速拒绝还是排队拖死
- 数据库和 Redis 连接数是否被控制在上限内
- 下游慢时，Node.js 是否还能保护住高优先级接口
- 限流后 P99 是否更可控，而不是整体更差

## 常见误区

### 误区 1：只加 timeout，不控并发

结果通常是：

- 请求最终会失败
- 但连接池和下游还是先被打满

### 误区 2：把本地限流当全局配额

多副本一放大，真实放行量就远超预期。

### 误区 3：一过载就先加长队列

这常常只是把故障从“拒绝”变成“更慢地失败”。

### 误区 4：所有接口套同一层最严格保护

最后往往会把低风险和高风险接口都压得过紧，整体吞吐反而更差。

### 误区 5：只在 HTTP 入口做保护，不看下游 client

真正把系统打穿的，很多时候是：

- 第三方 HTTP 调用
- 数据库慢查询
- Redis 热点 key

## 一组很实用的落地原则

1. 先分清 timeout、并发限制、限流、排队和快速失败，不要混用术语。
2. 服务保护优先放在最靠近风险的地方，而不是全部堆在全局入口。
3. timeout 只负责收等待时间，不替代幂等、结果确认和补偿。
4. 本地 rate limit 不是分布式限流。
5. 在线请求默认谨慎排队，优先考虑清晰拒绝和降级。
6. 路由级保护、下游 client 级保护和队列消费保护要分层设计。
7. 发布前要用真实压测验证保护策略，而不是只看代码感觉。

## 高频自检题

- timeout、concurrency limit、rate limit、backpressure、load shedding 分别在保护什么
- 为什么只加 timeout 还不够保护数据库和下游
- 为什么本地 rate limit 不能直接等同于全局配额
- 什么场景更适合快速失败，而不是排队等待
- 为什么路由级保护、client 级保护和队列消费者保护不能只做其中一层

## 一句话收尾

Node.js 过载保护真正难的，不是“挂几个中间件”，而是把等待时间、在途量、进入速率、排队长度和拒绝策略分别放到正确边界上。  
只要这几条线分清，Node.js 服务在高峰和故障下就不会只剩“加实例”和“等超时”两种办法。
