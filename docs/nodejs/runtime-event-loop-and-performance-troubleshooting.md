# Node.js 运行时、事件循环与性能排障

Node.js 真正难讲清的，通常不是 `async/await` 语法，而是：

- 事件循环到底怎么转
- 为什么一个接口会“偶尔卡死”
- 为什么 CPU 没满但延迟很高
- 为什么 RSS 一直涨但看不到明显泄漏
- 为什么本地能跑，线上一压就抖

这页要补的，就是 Node.js 服务端里最影响稳定性的运行时主线。

## 先把 Node.js 运行时放回正确位置

Node.js 的核心不是“单线程所以简单”，而是：

- JavaScript 主线程负责执行用户代码
- libuv 负责事件循环和一部分底层异步能力
- 部分任务会进入线程池或交给内核异步机制
- 最终再把回调、promise continuation 或 I/O 结果送回事件循环

所以 Node.js 服务端最关键的判断不是“是不是异步”，而是：

- 哪些工作会阻塞主线程
- 哪些工作会挤爆线程池
- 哪些工作会在 microtask 阶段把事件循环拖慢
- 哪些工作其实更适合拆到队列或 worker

## 先分清：调用栈、事件循环、线程池、worker thread 不是一回事

很多线上问题都来自这几层概念混在一起。

### 1. 调用栈

这就是 JavaScript 当前正在执行的同步代码。

只要这里还没跑完：

- 定时器不会继续推进
- I/O 回调不会及时执行
- HTTP 请求也可能开始堆积

### 2. 事件循环

事件循环负责一轮轮推进：

- timer
- pending callbacks
- poll
- check
- close callbacks

你不需要死背每个 phase 的细枝末节，但至少要知道：

- “异步”不等于“马上执行”
- callback 到底什么时候拿到执行权，取决于当前调用栈和事件循环是否被拖住

### 3. 线程池

Node.js 不是“绝对单线程”。  
像这些工作常会压到 libuv 线程池：

- 文件系统一部分操作
- DNS 某些解析路径
- `crypto`
- `zlib`

所以线上有时不是主线程先满，而是线程池先堵。

### 4. Worker Threads

它解决的不是“所有异步任务”，而是：

- CPU 密集型计算
- 需要和主线程隔离的重处理任务
- 不适合长期占用主线程的同步计算

一个很实用的原则是：

- I/O 型异步：先看事件循环和下游依赖
- CPU 型计算：优先考虑 worker threads 或拆到独立 worker 服务

## `async/await` 不会自动让代码变快

很多人对 `async/await` 最大的误解是：

- 写成 `async` 之后，代码就不会阻塞了

其实不是。

如果你在一个 `async` 函数里做这些事：

- 大 JSON 序列化 / 反序列化
- 大对象深拷贝
- 正则灾难回溯
- 图片处理、压缩、加密
- 大数组排序和聚合

它照样会卡住主线程。

所以要先问的不是“这个函数是不是 `async`”，而是：

- 里面有没有长时间占用 CPU 的同步工作

## 事件循环最常见的 4 类问题

### 1. 同步 CPU 工作太重

典型表现：

- 接口偶发卡顿
- P99 拉高
- 机器 CPU 高
- 某些请求一来，其他请求一起受影响

常见根因：

- 大 JSON 处理
- 模板渲染
- 加解密
- 压缩解压
- 批量数据格式转换

### 2. microtask 挤压事件循环

Promise 回调、`queueMicrotask` 这类任务如果持续堆积，会让事件循环迟迟回不到 I/O 推进阶段。

典型现象：

- 看起来没有明显同步死循环
- 但事件循环延迟仍然持续升高

### 3. 线程池耗尽

典型表现：

- 文件、压缩、加密类操作突然整体变慢
- CPU 不一定顶满，但吞吐明显下降

这时要想到：

- 问题未必只在主线程，也可能是线程池排队太长

### 4. 下游 I/O 慢，导致请求堆积

很多“Node.js 性能问题”本质不是运行时本身，而是：

- 数据库慢查询
- Redis 抖动
- 第三方 API 超时
- 内网 RPC 堵塞

Node.js 会把这些等待放大成：

- 连接积压
- 超时堆积
- 内存上涨
- 尾延迟恶化

所以排障时不要只盯事件循环，要把外部依赖一起放进图里。

## 先建立一套够用的排障顺序

很多人排 Node.js 问题的方式是：

- 先打开监控
- 随机猜一个点
- 改完再观察

这不够稳。

更实用的顺序通常是：

1. **先分型**：CPU 高、内存涨、延迟抖动、事件循环延迟、线程池排队、下游超时，先分清主问题
2. **再定范围**：是某个接口、某类任务、某个租户，还是全局都慢
3. **先看趋势再看瞬时**：持续增长、周期抖动、发布后回归，这三类问题排法不同
4. **从 runtime 指标回到业务路径**：指标只能告诉你哪里异常，必须回到具体请求和代码路径
5. **优化后再验证**：重新压测、对比 event loop delay、CPU、RSS、P95/P99，而不是凭感觉

## 最值得盯的几类运行时指标

如果是线上服务，至少建议有这些指标：

- 请求量、错误率、P95/P99 延迟
- event loop delay
- active handles / requests
- 进程 RSS、heap used、heap total
- GC 次数与停顿趋势
- CPU 使用率
- 线程池相关等待迹象
- 下游数据库 / Redis / HTTP 客户端延迟与超时

关键点在于：

- 不要把 heap 当成内存全貌
- 不要把 CPU 当成唯一性能指标
- 不要脱离下游依赖只看 Node.js 自身

## 内存问题最容易被误判

Node.js 内存问题常见有 4 类：

### 1. 真正的对象持有过久

比如：

- 全局缓存没上限
- Map / Set 持续增长
- listener 或 closure 意外持有大对象

### 2. 请求堆积带来的瞬时内存上涨

如果下游变慢，请求进来但迟迟不返回，内存也会涨。

这不一定是“泄漏”，但会一样危险。

### 3. Buffer / 流处理不当

例如：

- 一次性把大文件读进内存
- HTTP body 直接整体缓存
- 多层复制 Buffer

### 4. 外部资源没及时释放

例如：

- 连接没收
- stream 没关闭
- 事件监听器没解绑

所以看到 RSS 涨，不要立刻下“内存泄漏”结论，先问：

- 是对象长期持有
- 还是请求积压
- 还是 Buffer / native memory / 外部资源占用

## CPU 高时优先查什么

CPU 高时通常先看这些方向：

- 大 JSON 编解码
- 正则或字符串处理
- 压缩、加密、哈希
- 大量同步循环
- 热点日志格式化
- 数据聚合或导出逻辑跑在请求线程里

一个特别常见的反模式是：

- 管理后台点一次导出
- Node.js 在 HTTP 请求里直接做几万行数据拼装和 CSV / Excel 生成

这种工作更适合拆到后台任务或 worker，不应该长期占住请求线程。

## 流和背压是 Node.js 服务端的关键边界

Node.js 里很多性能问题，不是“功能写错”，而是：

- 没有尊重 stream/backpressure 模型

典型错误包括：

- 一次性把大结果全读进内存
- `data` 事件里疯狂堆积处理
- 管道链路没有统一错误处理
- 只关注吞吐，不关注消费者是否跟得上

更稳的原则通常是：

- 能流式处理就不要整体读入
- 明确生产速度和消费速度
- 用 pipeline 思维而不是手搓一堆事件回调

## 什么时候该上 worker threads

出现下面这些信号时，可以优先考虑 worker threads：

- 单次 CPU 任务明显重
- 任务天然可序列化输入输出
- 逻辑不适合占用 HTTP 请求线程
- 希望和主线程隔离

但也要知道它的边界：

- 不是任务队列系统
- 不自动提供幂等、重试、死信、审计
- 不适合直接解决跨进程可靠执行问题

如果问题已经进入：

- 跨实例接力
- 跨重启保留
- 重试和死信治理

那往往该升级成队列或独立 worker，而不是只加 worker threads。

## 一套够用的定位工具链

实际排障时，至少建议准备：

- APM / tracing
- 结构化日志
- event loop delay 指标
- heap snapshot
- CPU profile
- 关键接口压测脚本

工具不是目的，目的是回答清楚：

- 哪个请求最慢
- 慢在哪里
- 是 CPU、内存、I/O 还是排队
- 是偶发还是持续
- 和发布、流量、下游依赖有没有对应关系

## 排障时最常见的误区

### 1. 看到 `async` 就以为不会阻塞

错。  
同步 CPU 工作放进 `async` 函数里，还是会阻塞。

### 2. 一看到内存涨就判定泄漏

错。  
也可能是请求堆积、Buffer 或 native memory 占用。

### 3. 只看 Node.js 进程，不看下游依赖

错。  
数据库、Redis、第三方 API 慢，最终都会体现成 Node.js 的延迟和内存问题。

### 4. 用平均值掩盖尾延迟

真正伤线上体验的，通常是 P99，而不是平均值。

### 5. 只修现象，不修执行模型

很多“性能优化”只是把某个热点挪了位置，没有真正解决：

- 请求线程做了不该做的事
- 流式处理缺失
- 队列和背压边界没建立

## 推荐实践项目

### 1. API 网关 / BFF 性能排障演练

目标：

- 压测聚合接口
- 识别 JSON 聚合、下游超时和事件循环延迟
- 输出一份从指标到代码的排障报告

### 2. 大文件导出改造

目标：

- 把请求内同步导出改成后台任务或流式导出
- 对比改造前后的 event loop delay、P99 和 RSS

### 3. Node.js 内存问题定位演练

目标：

- 人工制造缓存无上限或请求积压问题
- 区分 heap 增长和 RSS 增长
- 形成一套“先分型再定位”的固定流程

## 高频自检题

- `async/await` 为什么不能自动消除阻塞
- event loop delay 高但 CPU 不高时，可能先查哪些方向
- Node.js 里的线程池和 worker threads 分别在解决什么问题
- 为什么很多导出、压缩、聚合任务不适合直接放在 HTTP 请求里
- 看到内存上涨时，为什么不能立刻把问题定性成泄漏

## 一句话收尾

Node.js 性能问题很少只是“某个函数慢”，更常见的是执行模型和业务边界没划清。  
只要把事件循环、CPU、内存、下游 I/O、队列和背压放在一张图里看，排障才会真正稳定。
