# Node.js 秒杀 / 抢券系统实践

如果把 Node.js 高并发、缓存、Redis、限流、队列这些能力放回一个最典型的业务场景里，最常见的落点通常就是：

- 秒杀
- 抢券
- 限量活动
- 热点资格发放

这类系统真正难的，通常不是“能不能写一个下单接口”，而是：

- 流量洪峰怎么挡
- 无效请求怎么尽量前置过滤
- 库存或配额怎么扣得稳
- Redis 和数据库怎么分工
- 队列怎么削峰
- 失败后怎么补偿
- 用户该看到什么结果和排队态

这页补的，就是 Node.js 在秒杀 / 抢券系统里的工程主线。

## 适合谁看

- 已经在做 Node.js API、BFF、NestJS 或活动系统，开始碰到热点活动、瞬时流量和资源扣减问题
- 会说缓存、MQ、Redis 和限流，但还没把它们收成一个完整业务方案
- 想把秒杀、抢券、限量发放这类高并发写热点场景讲成一条工程主线
- 面试或方案评审里经常被问到“如果用 Node.js 做秒杀 / 抢券，你怎么设计”

## 学习目标

- 把秒杀 / 抢券系统拆成入口治理、资格过滤、资源扣减、异步排队、结果查询和补偿几条线
- 理解 Node.js 在这类场景里的真正风险点：连接池、热 key、回源风暴、队列积压、重复请求和内存堆积
- 能说清 Redis、数据库、队列和网关分别在承担什么责任
- 把用户体验、系统稳定性和最终一致性放回同一张图里看

## 先讲结论：秒杀 / 抢券系统经常不是先死在库存，而是先死在入口和背压

很多人聊秒杀，第一反应是：

- 库存怎么扣

这当然重要，但线上真实崩溃顺序通常更早：

- 活动页和详情页先被打爆
- 网关和活动服务入口先抖
- Redis 热 key 先过热
- 队列和订单消费先积压
- 数据库连接池先满

所以这类系统里很重要的一条认知是：

- 库存正确性解决的是“不能卖错”
- 限流、削峰、背压和降级解决的是“系统不能先死”

## 先把问题拆成 6 层

一个更稳的秒杀 / 抢券系统，通常要同时看这 6 层：

1. 活动预热
2. 资格过滤
3. 流量闸门
4. 资源扣减
5. 异步削峰
6. 结果收口

如果只盯其中一层，很容易出现：

- 资源扣减很对，但入口已经扛不住
- 限流做了，但重复请求还是把链路打满
- 排队也有了，但结果查询和补偿没跟上

## Node.js 在这类场景里最容易先抖在哪里

Node.js 做这类系统时，最容易先出问题的通常不是“语言本身不行”，而是下面这些点：

- 请求内同步做太多事，事件循环延迟升高
- 热点 key 过期后回源风暴把数据库打穿
- 同一个请求 fan-out 太多，把 Redis / DB / 下游一起放大
- 多实例扩容后，数据库和 Redis 总连接数失控
- 队列消费者只顾并发拉满，不看下游承载能力
- 大量排队对象和结果查询把内存慢慢堆高

## 一条够用的秒杀 / 抢券链路

```text
用户请求
  -> CDN / Gateway / 风控 / 登录校验
  -> 活动时间、资格、限购校验
  -> 资格令牌 / 前置限流 / 排队闸门
  -> Redis 预扣减 / 配额判断
  -> 成功后写入异步队列
  -> worker 创建订单 / 发券 / 落用户资产
  -> 成功结果写回查询态
  -> 用户轮询 / 回调 / 通知拿结果
```

这条链路里最重要的不是组件多，而是职责明确：

- 入口层负责挡流量
- Redis 负责快速判定和短状态控制
- 数据库负责最终正确性
- 队列负责削峰
- 结果查询负责用户体验和收口

## 第一层：活动预热和静态化

秒杀 / 抢券系统的第一层治理，往往不是库存，而是预热。

典型动作包括：

- 活动页静态化或半静态化
- 商品 / 券模板基础信息预热
- Redis 热 key 预热
- 把活动开始时间、限购规则、资格信息提前加载到缓存
- 提前压测、演练和灰度开关准备

一个很实用的原则是：

- 活动真正开始的那一秒，不要把资源花在“现场拼配置和查规则”上

如果活动开始时还在动态查大量配置、拼大 JSON、临时回源数据库，系统很容易在扣库存之前就先抖。

## 第二层：资格校验与无效流量前置过滤

秒杀 / 抢券系统不能让每个请求都直接冲向库存或配额扣减。

更稳的做法通常是先做前置过滤：

- 是否登录
- 活动是否开始
- 是否命中风控
- 是否已经参与过
- 是否达到单用户限购 / 限领阈值
- 是否满足券活动资格

这一层的价值不只是业务正确，更是：

- 尽量把无效流量挡在最外层

如果无效流量也能一路打到库存、订单、发券或队列，那高峰时大量资源都会被白白消耗掉。

## 第三层：限流、令牌和排队闸门

真正稳定的活动系统，通常不是“直接扣库存”，而是先过几道闸门。

### 1. 网关级限流

负责：

- IP 级限流
- 用户级限流
- 活动级限流
- 异常流量初筛

### 2. 资格令牌

负责：

- 防止请求伪造
- 防止重放
- 只让符合资格的请求进入真正扣减链路

### 3. 排队闸门

负责：

- 把系统承诺量控制在能处理的范围里
- 把瞬时高峰变成可消费流量

一个很务实的原则是：

- 不是所有通过资格校验的请求都应该立刻去扣数据库库存

在很多活动里，更稳的方式通常是：

- 先拿资格
- 再拿排队态
- 再逐步进入资源扣减

## 第四层：Redis 预扣减与数据库最终落账怎么分工

这类系统最容易吵起来的问题通常是：

- 到底全在 Redis 扣
- 还是全在数据库扣

更稳的理解通常是：

- Redis 负责高并发快速判定和前置拦截
- 数据库负责最终正确性和结果落账

例如：

- Redis 里维护活动库存或券配额的快速计数
- Redis 里维护用户限领、短期幂等、资格令牌
- 最终订单、用户券资产、状态流转落在数据库

一个常见错误是：

- 以为 Redis 扣成功就等于业务最终成功

其实更常见也更稳的方案是：

1. Redis 预扣减 / 预占位
2. 异步进入下单或发券队列
3. 数据库最终落账成功
4. 失败时做回补、补偿或重试

## 第五层：异步削峰、队列消费和补偿

这类链路最忌讳的一种做法是：

- 用户请求一进来，就同步做完整下单 / 发券 / 通知 / 审计

更稳的方式通常是：

- 请求快速确认进入排队态
- 真正创建订单 / 发券放到异步链路
- 后续通知、积分、审计继续异步化

异步层至少要想清：

- 消费并发上限是多少
- 重试预算是多少
- 死信怎么处理
- 失败时 Redis 预扣减怎么回补
- 是否需要人工补偿通道

一个高频坑是：

- 队列是上了，但消费没有上限，结果把数据库或第三方依赖一起打死

如果你想把队列、重试和 worker 主线单独理顺，继续看：

## 第六层：结果查询、排队态与用户体验

秒杀 / 抢券系统经常不只是技术问题，也是用户体验问题。

如果用户请求进来以后，你只能给两种结果：

- 成功
- 失败

那在异步化链路里，体验往往不够。

更稳的做法通常是明确几种态：

- 排队中
- 处理中
- 成功
- 失败
- 已经参与 / 已达上限

这样做的价值是：

- 接口不必长时间等待最终结果
- 用户知道系统还在处理，不会疯狂重复点击
- 可以减少一部分无意义重试和重复请求

一个很实用的原则是：

- 高并发活动里，明确排队态往往比假装同步成功更重要

## 秒杀和抢券最关键的差异

这两个系统很像，但也不能完全照搬。

### 秒杀更偏交易链路

重点通常是：

- 库存
- 订单
- 支付
- 超时取消

### 抢券更偏营销资产链路

重点通常是：

- 券模板配额
- 用户限领
- 用户券资产入账
- 核销、过期、作废状态

所以抢券系统里往往更重：

- 单用户去重
- 资产归属
- 券状态机

而秒杀系统里往往更重：

- 库存扣减
- 订单状态推进

## 指标、压测和演练怎么做

至少建议观测：

- 网关限流命中率
- 资格过滤命中率
- Redis 预扣减成功率 / 回补次数
- 队列积压长度
- 消费延迟
- 数据库写入成功率
- 排队态数量
- 结果查询成功率
- P95 / P99
- event loop delay

很实用的标签包括：

- `activity_id`
- `resource_type`
- `result_state`
- `reject_reason`
- `queue_name`

压测时至少验证：

- 活动开始瞬时峰值是否把入口打抖
- 热 key 过热时是否出现明显回源风暴
- 队列积压后系统是否还能保持清晰拒绝
- 回补和补偿逻辑是否能收敛

## 常见坏味道

- 一上来就把所有请求打进数据库扣库存
- 无资格请求也能一路打到核心链路
- Redis 扣减成功就被当成业务最终成功
- 队列消费没有并发上限和重试预算
- 结果查询态不清楚，用户只能疯狂刷新和重复点击
- 秒杀和抢券完全用同一套资源模型，不考虑资产生命周期差异
- 只盯库存正确性，不盯入口限流、积压和排队体验

## 一组很实用的落地原则

1. 活动系统先做入口治理，再做资源扣减。
2. 资格过滤和无效流量前置，越外层越值钱。
3. Redis 负责快判和预占位，数据库负责最终正确性。
4. 同步请求只做最小必要动作，其余尽量异步化。
5. 排队态和结果查询不是附属物，而是高并发体验的一部分。
6. 队列消费必须有并发上限、重试预算和回补逻辑。
7. 秒杀和抢券共享高并发框架，但资源模型和生命周期要分开设计。

## 高频自检题

- 为什么秒杀 / 抢券系统经常不是先死在库存，而是先死在入口和背压
- Redis 在这类系统里更适合承担什么，不适合承担什么
- 为什么同步请求只做最小必要动作，后面要异步化
- 为什么排队态和结果查询会直接影响系统稳定性
- 为什么抢券系统不能简单等同于秒杀系统

## 一句话收尾

Node.js 秒杀 / 抢券系统真正难的，不是“扣库存这一步”，而是能不能把入口流量、资格过滤、Redis 预占位、异步削峰、最终落账和结果体验一起收成一条稳定链路。  
只要这条线立住，Node.js 就不只是能写活动接口，而是能真正承接热点活动工程实战。
