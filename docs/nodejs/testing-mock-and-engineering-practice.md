# Node.js 测试、Mock 与工程化实践

很多 Node.js 项目一开始都推进得很快，后面变乱通常不是因为框架，而是工程习惯没立住。

最常见的症状是：

- 目录结构越写越散
- 包管理器混用，锁文件策略不清
- 改一个 service 不敢发版，因为测试不稳
- Mock 到处都是，但没人知道真假边界
- 线上问题只能靠零散日志猜

这页就是把 Node.js 服务端最值得尽早建立的工程化基本盘收起来。

如果你还没把包导出、workspace、共享包和 monorepo 的边界理顺，建议继续看：

- [Node.js 模块系统、ESM/CommonJS 与 monorepo 工程边界](./module-system-esm-commonjs-and-monorepo-boundaries.md)

## 先建立一个现实目标

不是每个 Node.js 项目都要一上来做成平台化工程，但只要项目会被：

- 多人协作
- 反复上线
- 持续重构
- 长期维护

就至少该做到：

- 环境可复现
- 依赖可追踪
- 目录结构清楚
- 核心逻辑可测试
- 日志、配置和 CI 有统一入口

## 项目结构先别散

一个够用的 Node.js / TypeScript 服务结构，通常至少可以这样：

```text
project/
├── package.json
├── pnpm-lock.yaml
├── tsconfig.json
├── src/
│   ├── main.ts
│   ├── config/
│   ├── modules/
│   ├── common/
│   └── infra/
├── tests/
│   ├── unit/
│   ├── integration/
│   └── e2e/
├── scripts/
└── .github/workflows/
```

关键点不在于“目录长什么样”，而在于至少分清：

- 入口层
- 业务层
- 基础设施层
- 测试代码
- 运维/脚本代码

如果 `src/` 下面所有东西都平铺，后面模块边界会越来越差。

## 包管理先统一口径

Node.js 项目最怕的是：

- 一部分人用 `npm`
- 一部分人用 `pnpm`
- CI 又偷偷用别的方式装依赖

更稳的做法通常是：

- 团队统一一个包管理器
- 锁文件进版本库
- 本地、CI、容器构建都走同一套安装入口

### 为什么这件事重要

因为 Node.js 生态里依赖树非常深：

- 版本浮动
- peer dependency 差异
- install script 行为

都可能让“我本地没问题”变成“线上或 CI 出问题”。

## 测试先分层，不要一把梭

很多 Node.js 项目测试痛苦，不是因为没工具，而是没分层。

更实用的划分通常是：

### 1. 单元测试

重点验证：

- 纯业务规则
- 参数边界
- 状态流转
- 错误分支

适合：

- service 里的纯逻辑
- util 的稳定规则
- policy / validator

### 2. 集成测试

重点验证：

- 模块之间接起来是否正常
- 数据库、缓存、消息组件的真实协作
- 配置和依赖注入是否正确

### 3. e2e 测试

重点验证：

- HTTP 入口到响应出口的完整链路
- 鉴权、校验、中间件、错误处理是否都接对了

一个很实用的原则是：

- 单元测试解决“规则是否正确”
- 集成测试解决“边界是否接通”
- e2e 测试解决“真实请求链路是否稳定”

## Mock 不是越多越好

Node.js 测试里最容易失控的地方，就是 Mock。

### 1. Mock 适合解决什么

- 下游 HTTP 调用
- 时间、随机数
- 文件、消息发送、短信、邮件
- 很慢或很不稳定的外部依赖

### 2. Mock 不适合解决什么

- 业务核心规则本身
- 数据库查询语义本身
- 复杂流程的全部真相

如果核心逻辑全靠 Mock 堆出来，测试只会越来越脆。

### 3. 更稳的顺序

- 先写容易纯化的业务逻辑
- 再在边界层 Mock 外部依赖
- 对数据库、缓存、队列这类关键边界保留一定比例的真实集成测试

## Fake、Stub、Mock、Test Double 要会区分

虽然团队里不一定天天精确区分术语，但工程上最好知道：

- Fake：一个简化但可运行的替代实现
- Stub：返回固定数据
- Mock：验证调用行为
- Test Double：这些替身的总称

很多时候，Fake 比 Mock 更稳，因为：

- 更贴近真实协作
- 不容易把测试写死在实现细节上

## 写出好测代码，比补测试更重要

更健康的 Node.js 项目通常会刻意做到：

- 业务逻辑和 I/O 分开
- 配置集中加载
- 时间和随机数可替换
- 外部依赖通过 adapter / client 收口
- 模块初始化和运行逻辑解耦

这会直接决定：

- 你的测试是轻松还是痛苦
- 你的 Mock 是少而准，还是到处都是

## 配置和日志别散在业务里

### 1. 配置要统一入口

不要让每个文件都直接读：

- `process.env.DB_URL`
- `process.env.REDIS_URL`
- `process.env.NODE_ENV`

更稳的方式是：

- 启动时统一加载
- 启动时统一校验
- 暴露成结构化 config 对象

### 2. 日志至少要结构化

日志至少应该能回答：

- 哪个请求出问题
- 哪个模块出问题
- 哪个用户或租户受影响
- 哪个下游依赖超时

所以最好至少做到：

- request id
- trace id
- 关键业务字段上下文
- 错误栈

### 3. 不要把敏感信息打进日志

尤其避免：

- token
- password
- 手机号、身份证号
- 支付敏感字段

## CI 的价值不是“多一个流程”，而是防止口径分裂

一个够用的 CI 至少应包括：

- 安装依赖
- lint / typecheck
- 单元测试
- 关键集成或 e2e 测试
- 构建产物验证

CI 真正解决的是：

- 团队机器环境不同
- 本地没跑全
- 发布前没有统一质量门槛

## 测试覆盖优先级怎么排

不要一开始追求覆盖率数字，先追求最有价值的覆盖：

1. 最核心的业务规则
2. 最容易回归的边界条件
3. 最关键的鉴权、校验、错误路径
4. 最容易因为重构出问题的模块
5. 线上故障复盘里真正出过问题的地方

## Node.js 项目里很常见的坏味道

- 只有 e2e，没有单元测试，导致定位成本极高
- 只有单元测试，没有真实边界测试，导致一接真实依赖就出问题
- Mock 过度，测试验证的是实现细节而不是行为
- 配置到处读 `process.env`
- 入口、业务、基础设施混在一起
- 本地跑 `npm test`，CI 跑的是另一套命令

## 和性能、异步任务的关系

测试和工程化不是独立话题，它会直接影响：

- 性能问题能不能复现
- 队列任务能不能做回归验证
- 发布时能不能放心改动

## 推荐实践项目

### 1. 给现有 Node.js 服务补最小工程化骨架

目标：

- 收敛配置入口
- 增加结构化日志
- 把测试拆成 unit / integration / e2e
- 让 CI 至少能跑 typecheck + test + build

### 2. 给一个 NestJS 服务补测试主线

目标：

- 给 service 层补单元测试
- 给关键接口补 e2e
- 给数据库和缓存边界补集成测试
- 明确哪些地方用 mock，哪些地方保留真实依赖

### 3. 复盘一次线上问题并沉淀测试

目标：

- 从真实故障回溯到缺失的测试类型
- 补一组能稳定防回归的用例
- 把“踩过的坑”变成 CI 能挡住的问题

## 高频自检题

- 为什么 Node.js 项目里包管理器和锁文件策略必须统一
- 单元测试、集成测试、e2e 测试分别在防什么问题
- 为什么 Mock 不能替代所有集成测试
- 为什么配置不应该散落在 `process.env` 的直接读取里
- 为什么很多团队测试很多，但仍然不敢重构

## 一句话收尾

Node.js 工程化真正要解决的，不是“工具装齐了没有”，而是让环境、依赖、测试、日志和发布口径保持一致。  
只要这条线立住，项目才会从“能写”走到“敢改、敢发、能维护”。
