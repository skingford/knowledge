# Python AI 工程、RAG 与服务化实践

Python 现在几乎是 AI 应用开发里最常见的语言之一，但很多项目的问题并不在“模型不会调”，而在工程边界没有立住。

最常见的问题通常是：

- 模型调用散落在各个模块里，没人知道哪段提示词对应哪条链路
- RAG 能跑，但检索、切块、重排、评测全混在一起
- Tool 调用很多，但失败、重试、超时和权限边界没设计
- Demo 能跑，真正上线后成本、延迟、观测和回归测试都接不住

这篇文档的目标，是从 Python 工程视角，把 AI 应用、RAG 和服务化实践收成一条更清楚的主线。

## 一、先建立一个判断：AI 项目首先是软件项目，然后才是模型项目

模型很重要，但上线后的大部分问题通常出在这些地方：

- 输入输出边界不清
- 配置和密钥管理混乱
- 结果没有评测
- 没有可观测性
- 流程不可回放

所以更稳的原则是：

- 把模型当成系统中的一个能力组件
- 用工程方法管理提示词、工具、检索和配置

## 二、一个更稳的最小分层

在 Python AI 项目里，比较常见也比较稳的分层包括：

- API / CLI 层：接收请求、参数和任务入口
- Application 层：编排一次问答、一次 Agent 任务或一次工作流
- RAG / Tool 层：负责检索、工具调用、上下文构造
- Adapter 层：封装模型 SDK、向量库、数据库、对象存储、第三方接口
- Domain 层：保存业务语义、任务状态、会话与权限信息

重点不是分层名字，而是这几个边界要清楚：

- 谁负责拼 prompt
- 谁负责调模型
- 谁负责检索和重排
- 谁负责记录状态和评测

## 三、模型调用一定要收口

### 1. 不要在业务代码里到处直接调用 SDK

否则后面很容易出现：

- 提示词版本不清
- 超时和重试策略不一致
- 日志和 token 统计不一致

### 2. 一个更稳的习惯

把模型调用统一收口到 adapter 或 gateway 层，至少统一这些能力：

- 模型名
- 超时
- 重试
- 流式输出与非流式输出
- token 和成本统计
- 错误映射

### 3. Prompt 也应该有边界

提示词不是随手拼几段字符串就结束了。

更稳的做法是：

- 把系统提示、用户上下文、工具结果分开管理
- 给关键 prompt 做版本管理
- 保持输入结构化，减少隐式拼接

## 四、RAG 不只是“向量库 + Prompt”

一个最小可维护的 RAG 链路通常包括：

1. 文档接入与清洗
2. 切块
3. embedding 与索引
4. 检索
5. 重排或过滤
6. 上下文拼接
7. 生成答案
8. 引用来源与评测

真正高频的问题通常在这些环节：

- 文档切块过粗或过碎
- 元数据不完整
- 检索结果很多，但真正相关的很少
- 没有来源展示，无法排查答案依据

## 五、Tool 调用和 workflow 要区分“确定性步骤”和“模型步骤”

### 1. 不是所有流程都该交给模型自由决定

更稳的思路是：

- 确定性步骤交给代码
- 需要语言理解、推理和生成的步骤交给模型

例如：

- 权限校验
- 参数标准化
- 幂等检查
- 数据落库

这些通常都应该由代码明确控制。

### 2. Tool 定义要像 API 一样严格

至少要清楚：

- 输入是什么
- 输出是什么
- 超时多长
- 失败如何返回
- 是否允许副作用

如果 Tool 边界模糊，Agent 跑起来后很难排障。

## 六、Pydantic 和结构化输出在 AI 工程里非常关键

很多 AI 应用最后要的不是“一段自由文本”，而是：

- 表单草稿
- 工作流参数
- 分类结果
- 提取后的结构化字段

所以很适合用结构化模型约束：

- 模型输入
- Tool 参数
- 模型输出
- 中间状态

这样做的价值是：

- 减少解析歧义
- 降低 prompt 修改后的连锁错误
- 让测试和回放更容易

## 七、评测是 AI 项目能不能持续迭代的关键

### 1. 没有评测，就很难知道系统到底有没有变好

只靠“我感觉这次回答更自然了”是不够的。

至少应该逐步建立：

- 一组真实问题样本
- 对应的期望答案或评分标准
- 检索质量评估
- 最终回答质量评估

### 2. 评测要拆层

不要把所有问题都丢给“最终主观感觉”。

更稳的拆法是：

- 检索层是否召回了正确片段
- 工具调用是否走到了正确工具
- 最终答案是否引用了足够依据

## 八、可观测性要覆盖成本、延迟和失败路径

AI 项目除了普通服务指标，还特别要关注：

- token 消耗
- 模型调用次数
- 平均响应时延
- 失败重试次数
- 检索命中率
- Tool 调用成功率

如果没有这些信息，后面很难回答：

- 为什么这次更贵
- 为什么这次更慢
- 为什么答案突然变差

## 九、异步、队列和后台任务会越来越重要

这些场景尤其适合异步化或后台化：

- 文档导入和切块
- embedding 批量生成
- 批量评测
- 长时间 Tool 工作流
- 报告生成

更稳的原则是：

- 快速交互请求保持轻
- 重任务异步化
- 明确任务状态、重试和回放

## 十、配置、密钥和多环境管理不能靠手工记忆

AI 项目常见配置非常多，例如：

- 模型名
- API Key
- 向量库地址
- embedding 模型
- 检索阈值
- 重排开关
- 评测集位置

如果这些配置散落在代码里，后面会很难维护。

更稳的做法是：

- 统一配置对象
- 启动阶段校验关键配置
- 不同环境清楚区分
- 密钥永远不要写死在仓库里

## 十一、上线时要优先考虑可回放和可降级

### 1. 可回放

要尽量保留：

- 输入参数
- 关键上下文
- 检索结果
- prompt 版本
- 输出结果

这样才能在问题出现时真正复盘。

### 2. 可降级

如果模型调用失败、检索失败或 Tool 超时，系统应该知道：

- 能不能回退到更简单的回答
- 能不能提示用户稍后重试
- 能不能关闭高成本环节

## 十二、测试不要只测“模型回了东西”

更有价值的是测试：

- 检索管道是否按预期过滤和拼接上下文
- Tool 参数组装是否正确
- 结构化输出是否能稳定解析
- 配置切换是否影响模型和索引选择
- Prompt 改动后关键样例是否退化

其中很多测试其实不需要真的调线上模型，也可以先做：

- 适配层测试
- 解析测试
- 流程级回放测试

## 十三、常见坏味道

- 模型 SDK 调用散落在多个模块
- 提示词没有版本和变更记录
- 只关注生成结果，不观察检索质量
- Tool 输入输出没有结构化约束
- 没有成本、延迟和失败指标
- Demo 直接上线，没有评测基线

## 十四、面试或 code review 最值得说清的点

- 为什么 AI 项目首先是软件工程问题
- 为什么模型调用、提示词和工具边界要统一收口
- 为什么 RAG 至少要拆成切块、检索、重排、生成几个层次来看
- 为什么结构化输入输出比自由文本更容易维护
- 为什么评测、回放和可观测性决定了 AI 应用能不能持续迭代
