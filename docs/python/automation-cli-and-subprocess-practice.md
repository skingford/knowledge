# Python CLI 自动化与 subprocess 实践

Python 很适合写内部工具、运维脚本、批处理任务和命令行程序。  
但很多 CLI 或自动化脚本一开始虽然能跑，后面会逐渐出现这些问题：

- 参数越来越乱
- 错误码不稳定
- 依赖外部命令时不安全
- 日志不足，失败很难排查
- 重跑时可能把数据或文件搞坏

这篇文档把 Python 在 CLI 和自动化方向最常见的实践主线收起来。

## 一、CLI 和自动化脚本到底适合做什么

特别适合：

- 批量文件处理
- 数据导入导出
- 内部运维工具
- 定时任务触发器
- 调 API 的小工具
- 流程编排脚本

不太适合：

- 需要长期复杂交互、复杂状态管理的大型前端式应用
- 对极致性能要求很高的超大规模 CPU 任务

## 二、先把脚本和工具区分开

很多项目最初只是一个脚本，但一旦会被反复使用，就应该逐步把它升级成工具。

一个更像“工具”的 CLI，至少应具备：

- 明确参数
- 稳定输出
- 清晰错误码
- 日志
- 配置边界

## 三、参数解析：先清晰，再复杂

### 1. 一个更稳的起点

如果工具还不复杂，优先把：

- 子命令
- 必填参数
- 可选参数
- 帮助信息

设计清楚。

### 2. 参数设计常见原则

- 参数名语义要明确
- 布尔开关要可预测
- 默认值不要掩盖危险行为
- 输出路径、覆盖行为、dry-run 要显式

### 3. 为什么 `--dry-run` 很有价值

对会改文件、发请求、删资源的工具来说，`dry-run` 往往能显著降低误操作成本。

## 四、项目结构：别让 CLI 入口变成垃圾堆

即使是小工具，也建议尽量分成：

- 入口层：参数解析、配置加载、错误码处理
- 业务层：真正处理逻辑
- 边界层：文件、HTTP、数据库、外部命令

这样后面：

- 更好测试
- 更好扩展子命令
- 更容易做重构

## 五、`subprocess`：自动化里最容易踩坑的边界

### 1. 什么时候会用到

- 调 Git
- 调系统命令
- 调 ffmpeg、tar、ssh、kubectl 等外部程序

### 2. 更稳的调用方式

更推荐：

- 传参数列表，而不是拼整段 shell 字符串
- 显式处理返回码
- 为外部命令设置超时

这样能降低：

- shell 注入风险
- 转义问题
- 行为不透明

### 3. 至少要关注这几件事

- 超时
- 返回码
- 标准输出和标准错误
- 当前工作目录
- 环境变量

## 六、文件处理与批处理

CLI 工具最常见的事情之一，就是处理文件。

更稳的原则：

- 用 `pathlib`
- 显式编码
- 大文件优先流式处理
- 批处理要考虑失败回滚或部分成功策略

### 一个很重要的问题

工具是“原地修改”，还是“输出新文件”，这个边界一定要提前说清楚。

## 七、日志、输出与退出码

### 1. 标准输出和日志要分角色

更稳的做法通常是：

- 标准输出给用户看结果
- 日志记录过程和上下文

### 2. 退出码要稳定

CLI 一旦被别的脚本或 CI 调用，退出码就是接口的一部分。

至少要区分：

- 成功
- 参数错误
- 运行失败
- 外部依赖失败

### 3. 不要让输出格式随意漂移

如果工具会被自动化系统消费，输出格式应该尽量稳定。

## 八、幂等、重试和可重跑性

自动化任务一个很大的现实问题是：

- 任务可能失败
- 任务可能重试
- 人可能手动再次执行

所以要尽量回答：

- 重跑会不会重复改数据
- 是否会重复发消息
- 是否会覆盖已有产物

### 更稳的设计方式

- 关键步骤可幂等
- 输出文件可覆盖策略明确
- 外部副作用可追踪

## 九、调外部 API 的 CLI 要注意什么

至少要考虑：

- 超时
- 重试
- 限流
- 认证信息来源
- 错误输出是否可理解

不要把 API Token、密码等敏感信息直接写死在脚本里。

## 十、调度和定时任务的边界

很多 Python 自动化工具最后会进入：

- cron
- CI
- 任务平台

这时最重要的问题通常不是“代码能不能跑”，而是：

- 失败如何告警
- 重试策略是什么
- 并发执行会不会冲突
- 同一个任务能不能重复触发

## 十一、测试怎么做才有意义

CLI 工具至少值得覆盖：

- 参数解析
- 核心业务逻辑
- 文件输出
- 外部依赖替身
- 错误路径和退出码

对 `subprocess` 这类边界，尤其适合用测试替身，而不是每次真调系统命令。

## 十二、常见坏味道

- 把所有逻辑都写在 `main()`
- 拼接 shell 字符串执行命令
- 没有超时
- 没有退出码约定
- 重跑后副作用不可控
- 没有 `dry-run`
- 出错只打印一行模糊信息

## 十三、面试或 code review 最值得说清的点

- 为什么自动化工具的稳定性重点在边界，而不只是业务逻辑
- 为什么 `subprocess` 更推荐参数列表而不是拼 shell 字符串
- 为什么 CLI 输出和退出码属于接口设计的一部分
- 为什么定时任务必须考虑幂等和重跑
- 为什么自动化脚本一旦反复使用，就该按项目而不是一次性脚本来治理
