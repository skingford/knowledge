# Python 并发、GIL 与 asyncio

Python 并发是最容易被讲成“玄学”的主题之一。  
真正要学清楚，关键不是记 API，而是先分清几个问题：

- 你要解决的是并发，还是并行
- 任务是 IO 密集，还是 CPU 密集
- 你要的是更高吞吐，还是更低等待时间
- 你的库生态是同步的，还是异步的

这篇文档重点解决这些判断问题。

## 一、先把几个词拆开

### 1. 并发和并行不是一回事

- 并发：多个任务在时间上交错推进
- 并行：多个任务在同一时刻真正一起执行

在 Python 里：

- 协程非常适合并发
- 多进程更适合真正的 CPU 并行

### 2. IO 密集和 CPU 密集必须分开判断

#### IO 密集

典型特征：

- 大量时间在等网络、磁盘、数据库、RPC
- CPU 不是瓶颈

这类场景常见方案：

- 线程
- `asyncio`

#### CPU 密集

典型特征：

- 大量时间在计算、压缩、图像处理、特征计算
- 主要瓶颈是 CPU

这类场景常见方案：

- 多进程
- 更底层语言或原生扩展

## 二、GIL 要理解到什么程度

### 1. GIL 是什么

常见 Python 实现里，GIL 会影响多个线程同时执行 Python 字节码。

这意味着：

- 多线程不一定能让 CPU 密集 Python 代码线性提速
- 但这不代表 Python 不能做并发

### 2. GIL 不等于“线程没用”

线程在下面这些场景仍然非常有价值：

- 网络请求并发
- 文件 IO 并发
- 调用会阻塞的第三方同步库
- 把等待时间重叠起来

### 3. 一个更实用的记忆方式

> GIL 主要影响“多线程跑 Python 计算代码”的并行能力，不会否定线程在 IO 场景下的价值。

## 三、线程、进程、协程怎么选

| 方案 | 适合场景 | 优点 | 风险或代价 |
| --- | --- | --- | --- |
| `threading` | IO 密集、同步库较多 | 接入简单，共享内存方便 | 共享状态复杂，CPU 密集收益有限 |
| `multiprocessing` | CPU 密集 | 可用多核、隔离性更强 | 进程开销大，通信和序列化成本更高 |
| `asyncio` | 大量网络 IO、高并发连接 | 轻量、吞吐高、调度可控 | 要求链路异步化，阻塞调用会卡事件循环 |

### 一个简化决策

- 同步生态，任务量不大，主要是 IO：先线程
- CPU 密集：先多进程
- 大量网络并发、连接数高：优先 `asyncio`

## 四、线程：什么时候够用，什么时候会乱

### 1. 线程最适合解决什么问题

- 并发抓取多个 HTTP 接口
- 并发读多个文件或磁盘任务
- 同时调多个阻塞型 SDK

### 2. 线程的核心风险

最大风险不是性能，而是共享状态。

常见问题：

- 多个线程同时修改同一个列表或字典
- 没有清楚的锁边界
- 死锁
- 任务退出条件混乱

### 3. 线程里至少要知道的同步原语

- `Lock`
- `RLock`
- `Event`
- `Condition`
- `Semaphore`
- `Queue`

其中最值得优先掌握的是：

- `Lock`：保护共享状态
- `Queue`：线程间安全传递任务
- `Event`：通知停止或状态变化

### 4. 一个常见建议

如果你的线程模型开始需要非常复杂的锁设计，通常说明：

- 数据共享边界不清楚
- 或者这个场景更适合消息传递而不是共享状态

## 五、多进程：CPU 密集任务的第一选择

### 1. 多进程为什么适合 CPU 密集

因为进程之间更独立，更容易利用多核并行能力。

典型场景：

- 批量图像处理
- 大规模文本解析
- 数据清洗和计算
- 特征提取

### 2. 多进程的真实代价

需要注意：

- 进程创建成本更高
- 数据跨进程传递需要序列化
- 大对象传来传去会有明显成本

### 3. 什么时候不该轻易上多进程

- 任务很小、粒度太碎
- 共享大对象成本过高
- 实际瓶颈并不在 CPU

## 六、`asyncio` 的核心模型

### 1. 协程不是线程

协程是用户态调度单元，特点是：

- 很轻量
- 主动让出控制权
- 适合高并发 IO

### 2. 事件循环在做什么

事件循环负责：

- 调度协程
- 等待 IO 完成
- 在协程可继续执行时恢复它

你可以先把它理解成一个任务协调中心。

### 3. 最基础的三个概念

#### 协程函数

```python
async def fetch():
    ...
```

#### `await`

表示当前协程等待另一个异步结果，同时把控制权交回事件循环。

#### 任务

任务是被调度执行的协程包装体。

## 七、`asyncio` 最基本的代码形态

```python
import asyncio

async def fetch(name: str, delay: float) -> str:
    await asyncio.sleep(delay)
    return f"done: {name}"

async def main():
    results = await asyncio.gather(
        fetch("a", 1),
        fetch("b", 2),
    )
    print(results)

asyncio.run(main())
```

这段代码至少体现了几件事：

- `async def` 定义协程
- `await` 表示等待异步结果
- `asyncio.gather` 可以并发等待多个协程
- `asyncio.run` 负责驱动事件循环

## 八、`asyncio` 必须掌握的边界

### 1. 协程里不能随便放阻塞调用

下面这种写法就是典型问题：

```python
async def bad():
    data = requests.get("https://example.com")
    return data.text
```

如果这里 `requests.get` 是阻塞的，它会卡住整个事件循环。

### 2. 正确思路

- 用异步版库
- 或者把阻塞调用放到线程池

例如：

- `asyncio.to_thread(...)`

### 3. 一句经验话

`async def` 不是性能咒语。  
只有当整条链路大体异步化，它才真正发挥价值。

## 九、取消、超时和资源释放

这部分非常关键，因为生产问题很多都出在“任务没收干净”。

### 1. 协程可能被取消

取消是正常控制流的一部分，不是异常情况中的意外。

必须理解：

- 超时可能触发取消
- 父任务取消可能传播到子任务
- 取消后的清理逻辑非常重要

### 2. 超时控制

任何外部依赖调用都应该考虑超时：

- HTTP 请求
- 数据库调用
- RPC
- 队列等待

没有超时的异步系统，很容易把问题积压成雪崩。

### 3. 清理一定要落到 `finally`

无论同步还是异步，资源管理都要清楚：

- 连接要释放
- 文件要关闭
- 锁要解开
- 临时状态要回滚

## 十、结构化并发的思路

现代并发更推荐的思路是：

- 任务有清晰父子关系
- 谁负责创建，谁负责收尾
- 出现错误时，相关任务一起收敛

如果你的异步代码到处都是“散落创建任务但没人管理”，后面一定很难排查。

### 什么时候说明你的并发结构不健康

- 任务创建后没人 `await`
- 超时和取消没有统一策略
- 子任务报错但主流程无感
- 进程退出时还有大量后台任务悬挂

## 十一、并发里的背压与限流

不管是线程池、进程池还是协程，都不能只关心“能不能并发”，还要关心：

- 一次并发多少
- 下游是否扛得住
- 失败重试会不会放大流量

因此至少要建立这些意识：

- 用有界队列
- 控制并发数
- 控制超时
- 控制重试次数

## 十二、几个高频误区

### 1. 以为异步一定比线程快

不一定。  
如果规模不大、依赖库都是同步的，线程可能更省心。

### 2. 以为多线程能自动让 CPU 密集任务提速

通常不应该这样期待。

### 3. 以为 `async def` 里写什么都异步

错。  
只有真正异步的等待点才会让出控制权。

### 4. 以为并发越高越好

高并发没有背压和超时控制，往往只是更快把系统压垮。

## 十三、一个实际可落地的选择框架

遇到并发需求时，可以按这四步判断：

1. 任务主要在等 IO，还是在做计算
2. 当前依赖库是同步生态，还是异步生态
3. 是否需要共享大量内存状态
4. 是否需要极高连接并发

大多数场景会收敛成：

- 普通 IO 并发：线程
- 大规模网络并发：`asyncio`
- CPU 密集：多进程

## 十四、面试里最值得说清的点

- GIL 影响的是什么，不影响的又是什么
- 为什么 IO 密集任务仍然可以从线程中收益
- 为什么 CPU 密集任务更适合多进程
- 为什么异步代码里混入阻塞调用会拖垮事件循环
- 为什么并发系统必须有超时、取消和背压设计
