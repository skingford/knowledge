# Python 数据模型与常见陷阱

如果说 Python 有哪一层最值得尽早学透，那通常不是“语法糖”，而是数据模型。  
很多线上 bug、面试追问和维护困难，最后都能追溯到这几个问题：

- 变量到底是什么
- 对象到底什么时候共享
- 为什么看起来一样的值，行为却不同
- 为什么一段代码“能跑”，但改一处就开始出问题

这篇文档就是专门收敛这部分内容。

## 为什么这一层最重要

Python 的很多特性表面上看很轻松，但背后都依赖一套统一模型：

- 赋值是名字绑定对象
- 参数传递是对象引用共享
- 比较分身份比较和值比较
- 容器行为和可变性有关
- 迭代、切片、`with`、`len`、`in` 等语法都依赖对象协议

如果这层没建立起来，后面学模块、类、装饰器、并发和工程化时会反复混乱。

## 一、名字绑定，不是“变量盒子”

### 1. 赋值到底做了什么

```python
a = [1, 2, 3]
b = a
b.append(4)
```

执行完后，`a` 和 `b` 都会变成 `[1, 2, 3, 4]`。

原因不是 Python “自动同步变量”，而是：

- 列表对象只有一份
- `a` 和 `b` 只是两个名字
- 它们同时绑定到同一个对象

所以更准确的理解是：

- 变量名只是引用标签
- 对象是实际存放数据的实体
- 赋值通常不会复制对象

### 2. 这个认知会影响什么

它直接影响：

- 函数参数的传递效果
- 列表和字典在多处修改时的联动
- 默认参数陷阱
- 浅拷贝和深拷贝
- `is` 与 `==`

## 二、身份、类型、值：看对象的三个维度

理解对象时，可以从三个角度看：

- 身份：这个对象是谁
- 类型：这个对象属于哪一类
- 值：这个对象当前的内容是什么

```python
x = [1, 2]
y = [1, 2]
z = x

print(x == y)  # True
print(x is y)  # False
print(x is z)  # True
```

这里：

- `x == y` 为真，因为值一样
- `x is y` 为假，因为不是同一个对象
- `x is z` 为真，因为 `z` 和 `x` 绑定的是同一个对象

## 三、可变与不可变对象

### 1. 常见分类

常见不可变对象：

- `int`
- `float`
- `bool`
- `str`
- `tuple`
- `bytes`
- `frozenset`

常见可变对象：

- `list`
- `dict`
- `set`
- 大多数自定义类实例

### 2. 不可变到底是什么意思

不可变的意思不是“变量不能重新赋值”，而是：

- 对象创建后，内容不能原地修改

```python
s = "abc"
s = s + "d"
```

上面不是把原字符串改长了，而是创建了一个新字符串，再让 `s` 绑定过去。

### 3. 为什么这件事重要

它会影响：

- 对象是否适合做字典键
- 拷贝时的行为
- 并发场景下共享状态的风险
- API 设计时是否应该暴露可变对象

## 四、函数参数传递：共享对象引用

Python 常被说成“值传递”或“引用传递”，但都不够准确。  
更实用的理解是：

- 函数调用时，形参名绑定到实参对象
- 如果对象可变，函数内部原地修改会影响外部
- 如果对象不可变，函数内部重新绑定不会影响外部名字

```python
def add_one(n):
    n += 1

def append_one(items):
    items.append(1)
```

对 `int` 这种不可变对象，`n += 1` 通常会绑定新对象。  
对 `list` 这种可变对象，`append` 是原地修改。

### 必须记住的判断方法

不要去背“到底是值传递还是引用传递”，更稳的做法是问自己：

- 这个对象是否可变
- 这里发生的是重新绑定，还是原地修改

## 五、`==`、`is`、哈希和字典键

### 1. `==` 和 `is`

- `==` 比较值是否相等
- `is` 比较是否是同一个对象

工程里的通用规则：

- 比较 `None` 用 `is None`
- 比较值用 `==`

### 2. 可哈希对象是什么

一个对象能否做字典键或集合成员，核心看它是否可哈希。

通常：

- 不可变对象更适合可哈希
- 可变对象一般不适合，因为值变了后哈希语义会出问题

所以这些通常可以做键：

- `str`
- `int`
- `tuple`（前提是内部元素也可哈希）

而这些通常不能做键：

- `list`
- `dict`
- `set`

### 3. 为什么这件事重要

因为字典和集合底层依赖哈希表。  
你至少要能解释：

- 为什么 `list` 不能直接做字典键
- 为什么 `tuple` 有时可以，有时不可以

## 六、浅拷贝、深拷贝与共享内部对象

### 1. 浅拷贝复制什么

浅拷贝只复制最外层容器，不递归复制内部对象。

```python
import copy

data = {"names": ["alice", "bob"]}
cloned = copy.copy(data)
```

这时：

- `data` 和 `cloned` 是不同字典
- 但它们内部的 `names` 列表仍然可能是同一份对象

### 2. 深拷贝复制什么

深拷贝会递归复制内部对象。

```python
deep_cloned = copy.deepcopy(data)
```

### 3. 什么时候会踩坑

最常见的情况：

- 配置模板复制
- 多层嵌套字典或列表
- 给多个对象分配“默认结构”

### 4. 一个更重要的工程建议

不要把“出了问题就 `deepcopy`”当万能解法。  
更好的思路通常是：

- 先搞清共享关系是否合理
- 尽量减少无意识共享的可变对象
- 对只读数据优先使用不可变结构或重新构造

## 七、最常见的五类陷阱

### 1. 可变默认参数

```python
def add_tag(tag, bucket=[]):
    bucket.append(tag)
    return bucket
```

为什么错：

- 默认值在函数定义时只创建一次
- 多次调用会共享同一个列表

正确写法：

```python
def add_tag(tag, bucket=None):
    if bucket is None:
        bucket = []
    bucket.append(tag)
    return bucket
```

### 2. 列表乘法创建二维数组

```python
matrix = [[0] * 3] * 3
matrix[0][0] = 1
```

结果会连带改掉多行，因为内层列表被重复引用。

正确写法：

```python
matrix = [[0] * 3 for _ in range(3)]
```

### 3. 闭包的延迟绑定

```python
funcs = []
for i in range(3):
    funcs.append(lambda: i)
```

最后得到的通常都是同一个最终值。  
原因是闭包捕获的是名字，不是当时的值。

修复方式：

```python
funcs = []
for i in range(3):
    funcs.append(lambda i=i: i)
```

### 4. 一边遍历一边修改容器

```python
items = [1, 2, 3, 4]
for item in items:
    if item % 2 == 0:
        items.remove(item)
```

这类写法容易漏元素或行为混乱。

更稳的方式：

- 构造新列表
- 倒序处理
- 明确使用索引控制

### 5. 原地修改方法返回 `None`

很多容器原地方法返回 `None`，比如：

- `list.sort()`
- `list.append()`
- `dict.update()`

```python
items = [3, 1, 2]
result = items.sort()
print(result)  # None
```

要区分：

- 原地修改方法
- 返回新对象的方法或函数，比如 `sorted(items)`

## 八、可迭代对象、迭代器与生成器

### 1. 先分清三个角色

| 概念 | 说明 |
| --- | --- |
| Iterable | 可以被 `for` 遍历 |
| Iterator | 可以不断产出下一个值 |
| Generator | 通过 `yield` 构造出的特殊迭代器 |

### 2. 为什么这件事属于数据模型

因为 `for` 并不是“会魔法地遍历一切”，它依赖对象遵守迭代协议。

你至少要知道：

- 并不是所有对象都能被遍历
- 一个对象能不能 `for`，和它是否实现迭代协议有关
- 生成器是实现流式处理的重要工具

### 3. 什么时候优先用生成器

- 数据量大，不想一次性加载
- 处理链天然是一条一条消费
- 你只需要顺序访问，不需要随机索引

## 九、对象协议与常见 dunder 方法

Python 很多语法糖都建立在“对象协议”上。

你至少要对这些方法有概念：

| 方法 | 含义 |
| --- | --- |
| `__init__` | 初始化对象 |
| `__repr__` | 便于调试的表示 |
| `__str__` | 面向用户的字符串表示 |
| `__len__` | 支持 `len(obj)` |
| `__iter__` | 支持迭代 |
| `__next__` | 迭代器取下一个值 |
| `__getitem__` | 支持下标访问 |
| `__contains__` | 支持 `in` |
| `__enter__` / `__exit__` | 支持 `with` |

你不一定需要一开始自己大量实现这些方法，但必须知道：

- Python 的很多“像内置语法一样自然”的行为，本质上都是协议
- 自定义对象也可以通过这些协议接入语言能力

## 十、`dataclass`、不可变建模与更稳的接口

### 1. `dataclass` 的定位

对主要承载数据的对象，`dataclass` 很适合。

```python
from dataclasses import dataclass

@dataclass
class User:
    name: str
    age: int
```

它能减少样板代码，让模型更清晰。

### 2. 为什么要关注不可变建模

如果某个对象天然表示：

- 配置
- 快照
- 事件
- 查询条件
- 只读领域对象

那么尽量让它更不可变，通常会更稳。

收益包括：

- 状态不会被无意改掉
- 更容易推理
- 更容易做缓存或哈希
- 更适合并发和测试

## 十一、理解这层后，工程代码会有哪些变化

当你真正理解数据模型之后，代码风格通常会变得更稳：

- 不再随手在多个地方共享同一个可变容器
- 写函数时会明确区分“修改输入”还是“返回新值”
- 对默认参数和闭包更警惕
- 知道什么时候该返回不可变结构
- 更容易看懂别人写的装饰器、生成器和上下文管理器

## 十二、面试里最容易被继续追问的点

### 1. Python 变量是怎么工作的

标准表达：

- 变量名本质上是对象引用的绑定标签
- 赋值通常不会复制对象

### 2. 函数为什么会改掉外部列表

标准表达：

- 形参和实参共享同一个列表对象
- 如果函数内部做的是原地修改，外部会感知到

### 3. 为什么可变默认参数会出问题

标准表达：

- 默认参数在函数定义时求值，只创建一次
- 后续调用会重复复用它

### 4. `is` 和 `==` 有什么区别

标准表达：

- `is` 比较对象身份
- `==` 比较值是否相等

## 十三、一个最实用的记忆框架

如果你想快速判断一段 Python 代码是否容易出问题，可以连续问自己五个问题：

1. 这里是新建对象，还是共享旧对象
2. 这里是原地修改，还是重新绑定
3. 这个对象是否可变
4. 这里比较的是身份，还是值
5. 这个语法糖背后依赖什么对象协议

很多“玄学 bug”问到最后，基本都能落到这五个问题上。
