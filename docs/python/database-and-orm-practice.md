# Python 数据库、事务与 ORM 实践

很多人写 Python 后端，真正开始吃亏的地方不是语法，而是数据库边界没立住。

最常见的问题通常是：

- 路由、service、repository 都在偷偷提交事务
- ORM 模型同时承担请求模型、领域模型和持久化模型
- 查询能跑，但一上量就出现 N+1、慢 SQL、连接池耗尽
- 表结构和代码一起演化时，没有迁移策略，只能手工改库

这篇文档的目标，是把 Python 项目里数据库、事务和 ORM 的最小工程主线讲清楚。

## 一、先建立一个判断：数据库问题通常不是 ORM 能自动解决的

ORM 能帮你做的主要是：

- 把表和对象做映射
- 减少一部分样板 SQL
- 统一 session、事务和查询表达方式

但 ORM 不能替你自动解决：

- 表设计是否合理
- 索引是否合理
- 事务边界是否清楚
- 一致性是否真的成立
- 查询是否高效

所以先记住一句话：

- ORM 是访问数据库的工具层，不是数据库设计能力的替代品

## 二、数据库基本盘：即使使用 ORM，也必须懂这几件事

### 1. 连接和连接池

你至少要清楚：

- 应用不是每次请求都新建物理连接
- 通常会通过连接池复用连接
- 连接池过小会排队，过大也可能把数据库压垮

如果服务一上来就“连接耗尽”，通常不是框架问题，而是：

- session 没有及时释放
- 长事务太多
- 下游慢查询拖住连接

### 2. 事务

事务解决的是一组操作要么一起成功，要么一起失败。

你至少要知道：

- 什么地方开始事务
- 什么地方提交事务
- 出错时在哪里回滚
- 哪些跨表操作必须放在同一个事务里

### 3. 索引

索引不是“建得越多越好”，而是为了让高频查询更稳定。

要特别关注：

- 查询条件字段
- 排序字段
- 唯一约束字段
- 联合索引的顺序是否匹配查询模式

### 4. 迁移

只要项目会长期维护，就不要靠手工改表。

更稳的做法是：

- 用迁移工具维护表结构演进
- 让开发、测试、生产环境都走同一套变更路径
- 把表变更纳入代码评审和发布流程

## 三、一个更稳的最小分层

在 Python Web 项目里，更稳的数据库分层通常是：

- 路由层：接 HTTP 请求、做输入输出转换
- Service 层：编排业务流程、定义事务边界
- Repository / DAO 层：封装查询与持久化细节
- Model 层：ORM 模型或持久化模型

比较重要的一点是：

- repository 负责“怎么查、怎么写”
- service 负责“什么时候提交、什么时候回滚”

如果 repository 层自己偷偷 `commit`，后面一旦涉及多表一致性，流程会很难收住。

## 四、事务边界应该尽量收敛在 service 层

### 1. 为什么不要多层乱提交

如果你在：

- 路由层提交一次
- repository 方法里再提交一次
- 某个工具函数里又顺手提交一次

那么后面一出问题，你很难判断：

- 哪一步已经落库
- 哪一步还能回滚
- 补偿逻辑应该从哪里接

### 2. 一个更稳的习惯

通常建议：

- session 在请求边界或任务边界创建
- service 层负责事务完成与失败处理
- repository 只负责操作 session，不负责决定最终提交

### 3. 长事务要谨慎

事务不是越大越安全。

长事务会带来：

- 连接占用更久
- 锁持有更久
- 冲突概率更高
- 重试成本更高

所以更稳的原则是：

- 把真正需要一致性的写操作放进事务
- 不要把无关的外部调用、长计算、网络请求塞进同一个事务

## 五、SQLAlchemy / ORM 的核心心智模型

### 1. ORM 对象不等于领域对象

ORM 模型更偏向持久化映射，主要关心：

- 表名和字段
- 主键和外键
- 关系映射
- 持久化状态

但业务领域对象更关心：

- 业务规则
- 语义约束
- 状态变化

如果把两者混成一个对象，项目容易出现：

- 请求模型污染数据库层
- 领域规则直接绑死在 ORM 生命周期上
- 改一个字段牵连很多层

### 2. session 是工作单元，不是全局垃圾桶

`Session` 通常代表一次工作单元。

要避免这些习惯：

- 全局共享一个 session
- 把 session 挂成单例长期复用
- 在异步或多线程场景里混用同一个 session

更稳的做法是：

- 一次请求一个 session
- 一次后台任务一个 session
- 用完及时关闭

### 3. ORM 查询不是“写起来像对象，就没有 SQL 成本”

无论你用 ORM 还是手写 SQL，最终都会落到数据库执行计划上。

所以你仍然要关心：

- 查询到底生成了什么 SQL
- 有没有全表扫描
- 有没有走到合适索引
- 一次请求里查了多少次数据库

## 六、最容易踩坑的几个 ORM 问题

### 1. N+1 查询

典型现象是：

- 先查一批主对象
- 再对每个对象单独查关联数据

小数据量时看不出来，一上量就会慢得很明显。

更稳的方式通常是：

- 提前做关联加载
- 明确哪些场景用 eager load
- 只加载真正需要的字段

### 2. 关系加载太重

有时不是 N+1，而是一次把太多关联对象一起拖出来。

结果就是：

- 查询结果巨大
- 序列化成本高
- 内存占用大

要始终记住：

- ORM 关系很方便，但方便不代表应该默认全展开

### 3. 把所有逻辑写进 model method

这样做短期很顺手，长期通常会变成：

- 模型职责失控
- 难以测试
- 查询和业务耦合过深

更稳的方式是：

- 持久化行为留在 repository
- 业务编排放在 service
- 复杂规则不要全塞进 ORM 模型

## 七、表设计和查询设计的几个高价值原则

### 1. 唯一约束优先于“代码里自己保证不重复”

凡是业务上必须唯一的东西，都应该优先考虑数据库层约束。

例如：

- 用户名
- 外部单号
- 幂等键

只靠应用层 `select` 后再 `insert`，并发下很容易失效。

### 2. 分页不要默认用深分页

如果列表会越来越大，要尽早考虑：

- 是否真的需要大偏移量分页
- 是否应该改成基于主键或时间的游标分页

深分页常见问题是：

- 越翻越慢
- 结果不稳定
- 数据变动时容易漏或重

### 3. 软删除不是零成本方案

软删除看似安全，但会带来：

- 所有查询都要记得过滤
- 唯一约束变复杂
- 表越来越大

所以要先判断：

- 业务是否真的需要恢复
- 是否可以归档而不是长期软删除

### 4. 时间字段和状态字段要设计清楚

很常见也很重要的字段包括：

- `created_at`
- `updated_at`
- 状态字段
- 幂等键
- 版本号或乐观锁字段

这些字段能明显提升排障、审计和并发控制能力。

## 八、迁移与表结构演进

### 1. 为什么迁移工具是必需项

因为表结构会持续变化：

- 新增字段
- 拆表
- 补索引
- 修正约束

如果不把这些变化纳入版本控制，团队很快会陷入：

- 每个人本地库结构不一致
- 测试环境和生产环境不一致
- 回滚成本极高

### 2. 迁移设计的几个原则

- 一次迁移只做一类清晰变更
- 先考虑向前兼容，再考虑清理旧结构
- 生产环境大表变更要评估锁表和耗时风险

### 3. 不要忽视数据迁移

很多变更不只是改 schema，还包括：

- 数据回填
- 状态修正
- 历史数据兼容

这类动作要和结构迁移分开设计，避免一锅炖。

## 九、同步 ORM 和异步 ORM 怎么选

### 1. 不要为了“看起来先进”强行全异步

如果你的服务：

- 并发量一般
- 团队对异步不熟
- 数据库交互也不复杂

同步方案完全可能更稳。

### 2. 什么时候异步更值得

- 你的整个调用链都是异步生态
- 服务主要是高并发 IO
- 团队已经能稳定处理 `asyncio` 的边界

### 3. 一个很关键的原则

不要在异步项目里混入一堆阻塞数据库访问，也不要在同步项目里为了局部接口强行拆出复杂异步链路。

## 十、数据库错误要转成业务可理解的错误

数据库层常见错误包括：

- 唯一约束冲突
- 外键约束冲突
- 连接超时
- 死锁或事务冲突

这些错误不应该原样直接冒给用户侧接口。

更稳的做法是：

- 在 repository 或 service 层识别错误类型
- 转成业务层可处理的异常
- 在边界层映射成稳定的 HTTP 响应或任务失败语义

## 十一、测试应该怎么覆盖数据库层

### 1. 不要只测“ORM 语法能不能跑”

更有价值的是测试：

- 关键查询是否满足业务过滤条件
- 事务回滚是否正确
- 唯一约束和幂等逻辑是否成立
- 异常路径是否被正确转换

### 2. 单元测试和集成测试都需要

- 单元测试：更关注 service 编排和 repository 接口行为
- 集成测试：更关注真实数据库交互、事务和约束

### 3. 一个更实用的原则

数据库相关逻辑最终最好有一层真实数据库验证，不要永远停留在 mock。

## 十二、性能排查时最该先看的地方

如果你发现数据库慢，不要第一反应就是“换框架”。

先排查：

- 哪条 SQL 最慢
- 查询次数是否异常
- 是否有 N+1
- 是否走索引
- 是否一次查太多列或太多行
- 是否有长事务和连接泄漏

很多问题最后都不是 ORM 本身，而是：

- 查询设计不合理
- 分页设计不合理
- 索引没跟上
- 事务边界失控

## 十三、常见坏味道

- ORM 模型同时充当请求模型、响应模型和领域模型
- repository 方法内部直接 `commit`
- 所有列表查询都默认 `select *`
- 没有迁移体系，全靠手工改表
- 用了 ORM 之后就不看 SQL 和执行计划
- 把慢查询问题误判成“Python 太慢”

## 十四、面试或 code review 最值得说清的点

- 为什么事务边界通常应该由 service 层统一控制
- 为什么 ORM 不能替代索引、约束和表结构设计
- 为什么 session 生命周期必须短而明确
- 为什么 N+1 和关系加载是 ORM 项目里的高频性能问题
- 为什么数据库相关逻辑最终要用真实数据库做一层验证
