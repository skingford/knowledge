# Python 异常、文件处理与上下文管理

Python 工程代码里，很多稳定性问题不在“逻辑算错了”，而在：

- 错误没有被正确表达
- 资源没有被正确释放
- 文件和网络操作边界模糊
- 失败路径没有设计清楚

这篇文档重点讲这条主线：  
怎么把“异常处理”和“资源管理”写成可维护的工程代码。

## 一、先建立一个基本原则

异常处理的目标不是“让程序别报错”，而是：

- 正确表达失败
- 保留足够上下文
- 保证资源收尾
- 让调用方知道该如何继续

如果只是“为了不让堆栈打印出来”去吞异常，后面通常会更难排查。

## 二、异常不是坏事，而是失败路径的一部分

### 1. 什么情况适合抛异常

- 输入不合法
- 依赖调用失败
- 文件不存在
- 权限不足
- 状态不满足当前操作前提

### 2. 什么情况不该到处拿异常做普通分支

例如：

- 明明可以先判断，却强行用异常代替正常流程控制
- 把频繁、可预期的普通业务分支全写成异常

更稳的经验是：

- 真正的异常情况才用异常
- 普通业务判断保持在显式分支里

## 三、`try / except / else / finally` 的分工

### 1. `try`

只包裹真正可能出错的那一小段，不要把太大范围一股脑包进去。

### 2. `except`

用来处理明确预期的异常类型。

### 3. `else`

当 `try` 成功时执行，适合放“成功后续逻辑”。

### 4. `finally`

无论成功失败都会执行，适合做清理。

```python
try:
    data = load_data()
except FileNotFoundError:
    ...
else:
    process(data)
finally:
    cleanup()
```

## 四、捕获异常时最重要的两个原则

### 1. 只捕获你真正知道怎么处理的异常

不建议随手写：

```python
except Exception:
    pass
```

这种写法最大的问题不是“写得丑”，而是：

- 把真实错误吞掉
- 让系统进入错误但无感知的状态

### 2. 异常信息要带上下文

仅仅看到一个 `ValueError` 往往不够。  
更有价值的是：

- 哪个文件
- 哪条记录
- 哪个请求
- 哪个阶段

## 五、异常链：为什么 `raise ... from ...` 很重要

```python
try:
    raw = int(value)
except ValueError as exc:
    raise RuntimeError(f"invalid user age: {value}") from exc
```

这样做的价值：

- 保留原始错误原因
- 同时提供更贴近业务语义的错误说明

工程里很重要，因为调用栈需要同时服务：

- 排障的人
- 业务代码调用方

## 六、自定义异常：什么时候值得做

### 1. 什么时候需要自定义异常

- 业务失败语义需要明确分类
- 调用方需要按类型区分处理
- 你想把底层异常抽象成更清晰的领域错误

### 2. 什么时候没必要过度设计

如果项目很小、错误类型简单，不需要一开始就设计很多复杂异常层次。

更稳的原则是：

- 从少量、清晰、稳定的异常类型开始
- 真有分支需求再细化

## 七、文件处理：最常见的资源管理场景

### 1. 文件操作最重要的不是 API，而是生命周期

你至少要明确：

- 谁打开文件
- 谁关闭文件
- 异常时是否也能关闭

### 2. 为什么推荐 `pathlib`

相比字符串拼路径，`Path` 更清晰、更工程化。

```python
from pathlib import Path

path = Path("data") / "users.txt"
```

### 3. 编码问题必须明确

文本文件处理时，要尽量显式指定编码，尤其在跨环境、跨平台项目中更重要。

```python
with path.open("r", encoding="utf-8") as f:
    content = f.read()
```

## 八、`with`：资源管理的核心语法

### 1. 为什么 `with` 这么重要

因为它把资源生命周期写得非常清楚：

- 进入时申请资源
- 离开时释放资源
- 即使异常发生也保证收尾

### 2. `with` 不只是文件语法

很多场景都适合：

- 文件
- 锁
- 数据库连接
- 临时目录
- 网络会话

### 3. 背后的协议是什么

上下文管理器依赖两个方法：

- `__enter__`
- `__exit__`

这意味着：

- 只要对象实现了这个协议，就可以配合 `with`

## 九、自己写上下文管理器时要注意什么

### 1. 适合自定义上下文管理器的场景

- 需要统一申请和释放资源
- 需要封装成更清晰的使用边界
- 需要减少重复的 `try/finally`

### 2. 设计时的重点

- `__enter__` 返回什么
- `__exit__` 是否吞异常
- 资源释放是否幂等

### 3. 一个很重要的原则

不要轻易在 `__exit__` 里吞掉异常，除非你非常明确那就是想要的行为。

## 十、异常和日志要怎么配合

### 1. 不要同一层又记日志又重复抛很多次

否则常见后果是：

- 同一个错误被打印很多遍
- 日志噪音巨大

### 2. 更稳的方式

- 在最靠近失败现场的地方补充足够上下文
- 在真正负责边界处理的地方决定是否记录日志、返回错误响应或中止任务

## 十一、批量处理时的异常策略

批量文件处理、导入导出、数据清洗里经常会遇到一个问题：

- 遇到一条错误，是全停，还是跳过继续

更稳的做法是明确策略，而不是临时拍脑袋：

- 全部失败立即停止
- 记录坏数据后继续
- 超过一定错误阈值后停止

如果没有策略，后面很难解释为什么结果不完整。

## 十二、什么时候该用返回值，什么时候该用异常

### 更适合返回值的场景

- 正常业务分支
- 查询结果为空
- 条件未命中

### 更适合异常的场景

- 违反前提条件
- 依赖失败
- 文件损坏
- 状态不一致

关键不是绝对规则，而是：

- 让调用方容易理解
- 让边界稳定

## 十三、常见坏味道

- `except Exception: pass`
- 一个很大的 `try` 把半个函数都包进去
- 出错只打印一句没上下文的话
- 打开文件不用 `with`
- 失败路径没有资源释放
- 错误语义模糊，调用方只能猜

## 十四、一个更稳的失败路径设计思路

写文件、网络、数据库相关逻辑时，可以依次问自己：

1. 哪一步可能失败
2. 失败后谁负责处理
3. 失败后哪些资源必须释放
4. 调用方需要知道什么上下文
5. 这是普通业务结果，还是异常状态

## 十五、面试或 code review 最值得说清的点

- 为什么不应该乱吞异常
- 为什么异常链对排障很有价值
- 为什么 `with` 是资源管理的主线能力
- 为什么文件处理要显式编码和清晰生命周期
- 为什么异常策略应该和边界层责任一起设计
