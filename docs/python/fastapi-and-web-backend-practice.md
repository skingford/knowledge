# Python FastAPI 与 Web 后端实践

如果你学 Python 是为了写后端服务，那么只停留在语法、标准库和并发层面还不够。  
真正进入 Web 后端后，你会很快碰到这些问题：

- 路由、业务逻辑、数据库代码怎么分层
- Pydantic 模型和领域模型怎么区分
- 异步接口是不是应该全都 `async`
- 数据库事务、认证、错误响应和日志应该放在哪一层

这篇文档的目标，就是把 FastAPI 常见工程实践收成一条清晰主线。

## 一、为什么很多人会选 FastAPI

FastAPI 常见优势包括：

- 写法简洁，和 Python 类型标注结合自然
- 请求参数、响应模型、校验能力比较完整
- 对异步 IO 支持友好
- 文档体验好，原型和中小型服务启动很快

但也要先有一个判断：

- FastAPI 不会自动替你解决分层、边界、事务和一致性问题
- 它是 Web 框架，不是架构方案本身

## 二、先建立一个后端项目的最小分层

一个更稳的 FastAPI 项目，至少建议分清这些层：

- 路由层：HTTP 参数接收、响应返回、状态码、认证挂钩
- Service 层：业务规则和流程编排
- Repository / DAO 层：数据库读写
- Schema 层：请求和响应模型
- Domain / Model 层：领域对象或内部模型

一个常见目录可以长这样：

```text
app/
├── main.py
├── api/
│   └── users.py
├── schemas/
│   └── user.py
├── services/
│   └── user_service.py
├── repositories/
│   └── user_repo.py
├── models/
│   └── user.py
├── db/
│   ├── session.py
│   └── base.py
└── core/
    ├── config.py
    ├── logging.py
    └── security.py
```

## 三、请求模型、响应模型和内部模型不要混成一个对象

### 1. 为什么要分开

很多初学者会把同一个模型同时拿来做：

- 请求体解析
- 数据库存储
- API 响应
- 业务内部状态

短期看起来省事，后面通常会越来越混乱。

更稳的边界是：

- 请求模型：只关心输入校验
- 响应模型：只关心输出结构
- 数据库模型：只关心持久化
- 领域模型：只关心业务语义

### 2. Pydantic 模型适合做什么

非常适合：

- 请求体验证
- 响应结构约束
- 配置对象

但不要默认让 Pydantic 模型承担所有层的职责。

## 四、路由层应该薄

一个健康的路由处理函数，通常只做这些事：

- 接参数
- 调用 service
- 把结果转换成响应
- 处理清晰的 HTTP 语义

不要在路由层里堆这些逻辑：

- 大量业务判断
- 多表事务编排
- 外部服务调用细节
- 大量数据转换

路由层越重，后面越难测试，也越难复用。

## 五、依赖注入：FastAPI 的强项，但别滥用

FastAPI 的依赖注入常见用途：

- 数据库 session
- 当前用户
- 配置对象
- 权限检查
- 公共请求上下文

它很适合拿来做“边界能力注入”，但不适合用成：

- 隐式魔法过多
- 到处嵌套依赖看不出执行链

更稳的原则：

- 注入边界能力
- 保持依赖图尽量简单
- 复杂业务流程仍然放到 service 层

## 六、同步接口和异步接口怎么选

### 1. 不是所有接口都必须 `async`

如果你的数据库客户端、HTTP 客户端、SDK 都是同步的，那么一味把路由写成 `async` 并不会自动带来收益。

### 2. 更稳的判断方式

- 下游依赖是异步生态，且请求量高：优先异步链路
- 下游依赖主要是同步库，服务规模不大：同步也可以

### 3. 一个关键边界

如果你写了 `async` 路由，但里面全是阻塞调用，那事件循环优势基本发挥不出来。

## 七、数据库与事务边界

### 1. session 生命周期必须清楚

你至少要明确：

- 谁创建 session
- 谁提交或回滚
- 谁关闭连接

### 2. 事务不要在多层乱提交

比较稳的做法通常是：

- repository 负责数据访问
- service 负责业务流程和事务边界

如果每个 repository 方法里都偷偷提交事务，后面跨表一致性会很难控制。

### 3. 一个常见误区

不要把 ORM 当成“业务逻辑容器”。  
ORM 模型主要是持久化对象，不要把所有业务规则全塞到 ORM 层。

## 八、错误处理与响应设计

### 1. 后端错误至少分三层看

- 用户输入错误
- 业务规则冲突
- 系统内部错误

### 2. 不同类型错误的处理方式应该不同

- 输入错误：明确告诉调用方哪里不合法
- 业务错误：返回稳定的业务语义
- 系统错误：记录日志，避免泄露内部细节

### 3. 一个很重要的原则

不要把 Python 原始异常细节直接原样暴露给外部调用方。

## 九、认证、鉴权与请求上下文

### 1. 认证和鉴权不是一回事

- 认证：你是谁
- 鉴权：你能做什么

### 2. 更稳的设计方式

- 在边界层拿到当前用户
- 在 service 层或权限策略层做业务授权判断

不要把授权逻辑散在每个路由里到处 `if role == ...`。

## 十、日志、追踪与可观测性

FastAPI 服务至少应具备：

- 请求级日志
- 错误日志
- 关键业务埋点
- 慢请求识别

如果服务一出问题只能看零散 `print`，排查成本会很高。

## 十一、后台任务、定时任务和异步处理

### 1. 不要把所有异步需求都塞进 Web 请求里

适合从接口里拆出去的场景：

- 发邮件
- 发消息
- 大批量导出
- 重计算任务

### 2. 更稳的思路

- 请求链路做快速响应
- 重任务异步化
- 明确重试、幂等和失败补偿

## 十二、测试应该怎么覆盖

一个更健康的 FastAPI 项目，至少要有这几类测试：

- service 层单元测试
- 路由层接口测试
- 鉴权和错误路径测试
- 数据访问集成测试

不要只测“200 成功”，更要测：

- 参数错误
- 权限不足
- 依赖失败
- 业务冲突

## 十三、部署前最值得检查的几件事

- 配置是否通过环境变量或配置层统一管理
- 数据库连接池和超时是否合理
- 日志是否结构化
- CORS、认证、错误响应是否清楚
- 健康检查和 readiness 是否完善

## 十四、常见坏味道

- 路由层塞满业务逻辑
- 一个模型同时当请求、响应、ORM、领域对象
- 异步路由里混大量阻塞调用
- repository 层私自提交事务
- 错误处理全靠裸异常
- 授权逻辑散在每个接口里

## 十五、面试或 code review 最值得说清的点

- 为什么 FastAPI 项目一定要分清路由层和 service 层
- 为什么请求模型和数据库模型不建议混用
- 为什么不是所有接口都要强行写异步
- 为什么事务边界通常应放在 service 层
- 为什么 Web 接口必须单独设计错误响应和鉴权边界
