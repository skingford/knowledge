# Python 函数、模块与装饰器

很多人学 Python 会在“会写脚本”之后卡住，问题通常不在语法，而在于：

- 函数接口越来越乱
- 模块拆分没有边界
- 导入关系开始混乱
- 装饰器能用但讲不清本质

这篇文档专门解决这些问题，把 Python 从“会写几个函数”推进到“能组织成可维护模块”。

## 一、为什么这一层很关键

Python 的工程代码很大一部分都建立在这四个能力上：

- 函数接口是否清晰
- 作用域和闭包是否理解透
- 模块和包是否组织合理
- 装饰器是否用得克制且可维护

如果这一层没打牢，项目很容易出现：

- 参数越来越多
- 工具函数到处乱放
- 循环导入
- 包装逻辑越来越看不懂

## 二、函数的定位：先表达边界，再表达实现

### 1. 一个函数至少应该回答三件事

- 输入是什么
- 输出是什么
- 它是否会修改外部状态

写函数时最重要的不是“能不能跑”，而是调用者能不能看懂。

### 2. 函数尽量只做一件事

不是说每个函数都必须很短，而是要尽量保持职责单一。

坏味道通常包括：

- 同时做参数解析、业务判断、日志、数据库写入、格式转换
- 一段函数既能处理单条，也能处理批量，还能做重试

更稳的方式通常是拆成：

- 参数准备
- 核心逻辑
- 边界调用

## 三、参数设计：Python 函数最容易越写越乱的地方

### 1. 位置参数和关键字参数

原则上：

- 语义明显、顺序自然的参数可以用位置参数
- 含义容易混淆的参数更适合关键字参数

```python
def connect(host: str, port: int, timeout: float = 3.0) -> None:
    ...
```

调用时如果参数含义不够直观，显式关键字通常更清楚：

```python
connect("db.local", 5432, timeout=1.5)
```

### 2. 默认参数怎么设计更稳

默认参数最常见的坑不是“有没有默认值”，而是：

- 默认值是不是可变对象
- 默认值是否掩盖了调用方必须显式决策的事情

更稳的经验：

- 简单标量可以直接给默认值
- 列表、字典、集合这类默认值要谨慎
- 如果语义是“未提供”，优先用 `None`

### 3. `*args` 和 `**kwargs` 不要滥用

它们很强大，但也很容易把函数边界写模糊。

适合使用的场景：

- 写包装器
- 转发参数到下层库
- 需要兼容多个可选扩展参数

不适合的场景：

- 普通业务函数为了省事把一切都塞进 `**kwargs`

如果业务函数长期依赖 `**kwargs`，调用成本和维护成本会越来越高。

### 4. 参数过多往往说明建模有问题

如果函数长成这样：

```python
def create_user(name, age, city, phone, email, role, status, source, operator, remark):
    ...
```

更好的选择通常是：

- 引入数据对象
- 用 `dataclass`
- 把相关参数收敛成配置对象或请求对象

## 四、返回值设计：比“返回什么”更重要的是“稳定返回什么”

### 1. 返回值语义要稳定

不建议一个函数在不同路径下随意返回：

- `None`
- `False`
- 空字典
- 字符串错误信息

这种设计会让调用方不断加猜测逻辑。

更稳的思路：

- 成功返回明确结构
- 失败抛异常，或者返回稳定的结果对象

### 2. 多返回值不是问题，关键是是否清晰

Python 支持多返回值，本质上通常是返回元组。

```python
def parse_line(line: str) -> tuple[str, int]:
    ...
```

如果返回值太多，建议：

- 用命名结构
- 用 `dataclass`
- 用 `TypedDict`

## 五、作用域、闭包与名字查找

### 1. LEGB 规则必须真正理解

名字查找顺序：

- Local
- Enclosing
- Global
- Built-in

这件事会直接影响：

- 嵌套函数
- 闭包
- 装饰器
- 模块级变量使用

### 2. 闭包的价值是什么

闭包不是为了炫技，而是为了：

- 保存一部分上下文
- 生成定制化函数
- 写装饰器

```python
def make_multiplier(factor: int):
    def multiply(x: int) -> int:
        return x * factor
    return multiply
```

### 3. 延迟绑定问题必须能解释

```python
funcs = []
for i in range(3):
    funcs.append(lambda: i)
```

最后拿到的值通常都是最终的 `i`。  
这不是 lambda 的问题，而是闭包捕获了名字。

## 六、模块：代码组织的第一层边界

### 1. 什么时候应该拆模块

当一个文件同时承担这些职责时，就该考虑拆了：

- 数据结构定义
- 业务逻辑
- 配置读取
- 外部 API 访问
- CLI 或 HTTP 入口

### 2. 模块拆分的基本原则

- 按职责拆，而不是按“代码多少”拆
- 高内聚，低耦合
- 一个模块尽量能被一句话概括

例如：

- `config.py`：配置加载与校验
- `models.py`：数据模型
- `service.py`：核心业务逻辑
- `client.py`：外部依赖调用

### 3. 不要把工具函数堆成“万能 util.py”

`utils.py` 并不是绝对不能有，但如果越来越大，通常说明：

- 模块边界没设计好
- 很多逻辑没有真正归属

更稳的方式：

- 让每类工具尽量靠近自己的业务领域
- 真的跨领域通用，再抽成独立模块

## 七、包与导入系统

### 1. 为什么导入问题总容易变复杂

因为 Python 的导入不仅和文件结构有关，还和这些因素有关：

- 项目根目录
- 当前工作目录
- 包结构
- 解释器搜索路径

### 2. 更推荐的导入风格

工程里一般更推荐：

- 清晰的绝对导入
- 避免 `from x import *`

这样更容易：

- 搜索引用
- 避免命名污染
- 减少歧义

### 3. 循环导入的本质

循环导入通常不是“语法坏了”，而是模块依赖关系设计有问题。

常见原因：

- 数据模型和业务逻辑相互引用
- 配置、工具、服务层职责混在一起
- 入口代码和库代码耦合过深

解决思路通常是：

- 重新拆分职责
- 延迟导入
- 提取更底层的公共模块

## 八、`__name__ == "__main__"` 的正确定位

它最适合做这些事：

- 命令行入口
- 简单演示
- 轻量调试

不太适合做这些事：

- 把大量业务逻辑全塞在里面
- 让模块既是库又是复杂脚本入口，还混在一起

## 九、装饰器：本质是函数包装，不是魔法

### 1. 装饰器到底是什么

装饰器本质上是：

- 接收一个函数
- 返回一个新函数
- 在不改变调用方式的前提下增强行为

```python
from functools import wraps
import time

def timing(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        try:
            return func(*args, **kwargs)
        finally:
            cost = time.perf_counter() - start
            print(f"{func.__name__} cost={cost:.4f}s")
    return wrapper
```

### 2. 装饰器最适合做什么

特别适合这些横切逻辑：

- 日志
- 计时
- 重试
- 权限检查
- 缓存
- 参数校验前置封装

### 3. 装饰器最不适合做什么

不适合把复杂业务主流程藏进去。  
如果一个装饰器已经开始：

- 修改业务入参
- 偷偷改返回值结构
- 带很多隐式状态
- 嵌套很多层后没人看得懂

那通常说明设计已经过重了。

### 4. `wraps` 为什么重要

不用 `functools.wraps` 的后果通常是：

- 函数名丢失
- 文档字符串丢失
- 调试和追踪困难

## 十、高阶函数与函数式工具

Python 不是纯函数式语言，但高阶函数很常见。

你至少要熟悉：

- `map`
- `filter`
- `sorted(..., key=...)`
- `functools.partial`

更实用的经验是：

- 逻辑简单时可以用
- 可读性下降时宁可写普通循环

## 十一、什么时候该用函数，什么时候该用类

这是 Python 里特别值得问自己的问题。

优先考虑函数的情况：

- 逻辑纯
- 不需要长期保存状态
- 输入输出边界明确

考虑类的情况：

- 需要封装状态
- 需要一组行为围绕同一份数据协同工作
- 需要清晰抽象某种对象

不要因为“面向对象更高级”就强行写类。

## 十二、一个更稳的模块演进路径

很多 Python 项目比较健康的演进方式是：

1. 先从小函数开始
2. 把相关函数组织成模块
3. 把复杂输入输出抽成数据对象
4. 再在真正需要状态封装时引入类

这是比“一开始就设计很多类和继承关系”更符合 Python 风格的路径。

## 十三、常见坏味道清单

- 一个函数参数太多
- 大量依赖 `**kwargs`
- 返回值结构不稳定
- 模块之间互相导入
- 装饰器层数过多
- 把大量业务逻辑藏在装饰器里
- 巨大的 `utils.py`
- 入口文件混着所有业务实现

## 十四、面试或 code review 最值得说清的点

- 为什么参数太多通常说明建模有问题
- 为什么模块拆分按职责比按文件长度更重要
- 为什么装饰器适合横切逻辑，不适合隐藏主业务流程
- 为什么循环导入本质上经常是依赖设计问题
- 为什么 `wraps` 是装饰器的基本礼貌
