# Python NumPy、pandas 与数据处理实践

很多人学 Python 最终会走到数据处理方向。  
这时最常见的问题不是“语法不够”，而是：

- 什么时候该用 `list`，什么时候该用 `ndarray`
- DataFrame 到底应该怎么想
- 缺失值、类型、日期时间怎么处理更稳
- 为什么 Notebook 里能跑，换到正式任务里就越来越乱

这篇文档的目标，是把 NumPy、pandas 和数据处理工程实践串成一条更实用的主线。

## 一、先建立一个判断：不是所有数据都需要 pandas

### 1. 什么时候 pandas 很合适

- 表格数据
- CSV / Excel / Parquet 等批量数据
- 清洗、过滤、聚合、透视
- 分析型任务

### 2. 什么时候不一定需要 pandas

- 数据量很小、逻辑很简单
- 只是少量 JSON 转换
- 主要是业务规则，而不是表格分析

### 3. 什么时候 NumPy 更关键

- 数值计算
- 向量化操作
- 矩阵运算
- 需要高效数组处理

## 二、NumPy 的核心心智模型

### 1. `ndarray` 不是 Python `list`

它更接近：

- 同构数据
- 连续或规则的数组结构
- 支持向量化计算

### 2. 为什么它通常比纯 Python 循环更快

因为很多操作不是在 Python 层一项一项循环，而是底层批量计算。

### 3. 一个很重要的原则

如果你已经在 NumPy 里，就尽量用向量化思路，而不是把它当成 `list` 再回到 Python 循环。

## 三、pandas 的核心心智模型

### 1. DataFrame 可以先把它理解成“带索引和列名的二维表”

但它不只是表格，还包含：

- 列类型
- 索引语义
- 缺失值处理
- 批量运算能力

### 2. Series 和 DataFrame 的关系

- `Series` 可以看成一列或一维带索引数据
- `DataFrame` 是多列组合

### 3. 不要忽视索引

很多数据处理问题最后都绕不过索引语义。

你至少要明确：

- 当前索引是不是业务字段
- 是否需要重置索引
- 合并和筛选时索引会不会带来歧义

## 四、数据处理的典型主线

一个比较常见的数据处理流程是：

1. 读取数据
2. 理解字段和类型
3. 清洗缺失值和脏数据
4. 做过滤、转换、聚合
5. 输出结果

真正的难点通常在第 2 到第 4 步，不是在“怎么读 CSV”。

## 五、类型、缺失值和日期时间是最容易反复踩坑的地方

### 1. 类型问题

最常见的问题包括：

- 数字列被读成字符串
- ID 被错误地当成数值
- 混合类型让运算和比较变得混乱

### 2. 缺失值问题

必须尽早搞清：

- 哪些值是真缺失
- 哪些值只是空字符串或占位符
- 缺失值应该删掉、填充还是单独处理

### 3. 日期时间问题

时间字段一旦处理不好，后面排序、聚合、分组都可能出错。  
要尽量尽早统一：

- 时间解析
- 时区语义
- 统计粒度

## 六、过滤、分组、聚合和合并是最常用的能力

### 1. 过滤

核心是先把条件表达清楚，再关注写法。

### 2. 分组和聚合

这是数据分析和报表里非常高频的主线。

要特别注意：

- 分组字段是否合理
- 聚合口径是否一致
- 缺失值是否影响结果

### 3. 合并

合并前必须想清楚：

- 主键是什么
- 是一对一、一对多还是多对多
- 合并后行数为什么变多或变少

很多结果错，不是 merge 不会写，而是数据关系没想清楚。

## 七、性能和内存边界

### 1. pandas 不是无限大的“内存数据库”

如果数据量很大，要特别关注：

- 是否一次性全读进内存
- 是否有很多中间副本
- 列类型是否过重

### 2. 常见优化方向

- 只读取需要的列
- 分块读取
- 提前指定更合理的类型
- 减少不必要的中间 DataFrame

### 3. 一个很实用的判断

如果你已经开始明显被内存卡住，就要重新思考：

- 能不能分批处理
- 能不能减少列
- 能不能把部分处理下推到数据库或更适合的引擎

## 八、从 Notebook 到工程化脚本

### 1. Notebook 的价值

适合：

- 探索数据
- 快速验证
- 画图和分析

### 2. Notebook 的边界

不适合长期承担：

- 稳定批处理入口
- 可复现生产任务
- 多人协作下的核心逻辑沉淀

### 3. 更稳的演进方式

- 在 Notebook 里探索
- 把稳定逻辑抽成模块和脚本
- 让输入、输出、配置和日志清楚

## 九、数据处理任务也需要工程化

很多数据任务最终也要具备：

- 配置管理
- 日志
- 错误处理
- 可重跑性
- 测试

不要因为“只是分析脚本”就忽略这些边界。

## 十、常见坏味道

- 一切都堆在一个 Notebook
- 不管数据大小，默认一次性读全量
- 类型和缺失值不先处理就开始聚合
- merge 前不确认主键关系
- 大量链式操作之后没人能解释中间状态

## 十一、面试或 code review 最值得说清的点

- 为什么 `ndarray` 和 `list` 的思维方式不同
- 为什么 DataFrame 的类型、缺失值和索引必须优先搞清
- 为什么大部分数据处理问题本质上是口径和关系设计问题
- 为什么 Notebook 适合探索，不适合长期承担生产入口
- 为什么数据处理任务也需要工程化治理
