# Python 面向对象、类设计与 dataclass

Python 支持面向对象，但真正难的地方从来不是“类怎么写”，而是：

- 什么时候该写类
- 什么时候不该写类
- 状态和行为应该怎么组织
- 继承、组合、`dataclass` 分别该怎么选

这篇文档的重点不是机械讲 OOP 术语，而是把 Python 工程里最实用的类设计能力讲清楚。

## 一、先建立一个现实原则

在 Python 里，类不是默认答案。

很多场景其实更适合：

- 函数
- 字典或简单容器
- `dataclass`
- 轻量对象而非复杂继承体系

更稳的思路是：

- 先问是否真的需要状态封装
- 再问行为是否围绕同一对象长期协作
- 最后才决定是否建类

## 二、类与实例的核心区别

### 1. 类是模板，实例是具体对象

```python
class User:
    role = "member"

    def __init__(self, name: str):
        self.name = name
```

这里：

- `User` 是类
- `User("alice")` 是实例

### 2. 类属性和实例属性必须分清

- 类属性属于类本身
- 实例属性属于每个实例

典型风险是把本该属于实例的可变对象放到了类属性上。

```python
class BadUser:
    tags = []
```

这样会导致所有实例共享同一个列表。

## 三、`self`、实例方法、类方法、静态方法

### 1. 实例方法

实例方法操作的是实例状态。

```python
class Counter:
    def __init__(self):
        self.value = 0

    def inc(self):
        self.value += 1
```

### 2. 类方法

类方法更适合：

- 备用构造器
- 需要基于类本身做逻辑

```python
class User:
    def __init__(self, name: str):
        self.name = name

    @classmethod
    def from_dict(cls, data: dict):
        return cls(data["name"])
```

### 3. 静态方法

适合：

- 和类相关，但不依赖实例状态
- 逻辑放在类下面是为了语义归组

但如果一个静态方法和类本身关系很弱，也可以直接写成模块函数。

## 四、属性管理：不是所有字段都应该裸露着改

### 1. `@property` 的价值

它适合这些场景：

- 需要只读视图
- 需要在读取时做派生计算
- 需要在设置时校验

```python
class Temperature:
    def __init__(self, celsius: float):
        self._celsius = celsius

    @property
    def celsius(self) -> float:
        return self._celsius
```

### 2. 不要把 `property` 用成“到处隐藏副作用”

更稳的经验：

- 简单、可预测、轻量的逻辑适合 `property`
- 复杂 IO、昂贵计算、不明显副作用不适合藏进属性访问

## 五、继承、组合、Mixin：怎么选更稳

### 1. 继承适合什么

适合真正稳定的“is-a”关系。

### 2. 组合通常更稳

组合更适合：

- 复用能力
- 组装行为
- 避免深继承链

Python 工程里，大量情况下组合比继承更容易维护。

### 3. Mixin 的适用边界

Mixin 适合提取小而明确的横向能力：

- 日志支持
- 序列化支持
- 时间戳字段封装

但不适合：

- 混入大段业务主逻辑
- 形成难以理解的多继承网络

## 六、`dataclass`：Python 工程里非常值得掌握的能力

### 1. 它解决什么问题

对主要用来表达数据的类，`dataclass` 可以显著减少样板代码。

```python
from dataclasses import dataclass

@dataclass
class User:
    name: str
    age: int
```

### 2. 为什么它很适合工程代码

因为大量对象其实主要是：

- 请求对象
- 配置对象
- DTO
- 结果对象
- 领域快照

这些对象的重点不是复杂方法，而是：

- 字段清晰
- 初始化清晰
- 表达语义清晰

### 3. 什么时候优先用 `dataclass`

- 数据字段多于行为
- 需要可读的初始化
- 需要更清晰的模型边界
- 需要和类型标注自然结合

### 4. 什么时候不要滥用 `dataclass`

如果一个对象：

- 有复杂生命周期
- 有很多行为逻辑
- 需要非常细的状态控制

那它未必一定适合只当成数据类来处理。

## 七、不可变建模值得尽早关注

### 1. 哪类对象特别适合更不可变

- 配置
- 查询条件
- 事件
- 快照
- 返回结果对象

### 2. 为什么不可变更稳

- 降低共享状态风险
- 更容易推理
- 更适合缓存
- 测试更简单

### 3. 一个很实用的判断

如果一个对象在创建后“理论上就不该再变”，那就尽量往更不可变的方向建模。

## 八、面向对象不等于“把所有函数塞进类”

这是 Python 初学者非常常见的问题。

### 什么时候不用类反而更好

- 逻辑是纯函数
- 不需要共享内部状态
- 模块函数已经足够表达

### 什么时候类更有价值

- 需要维护内部状态
- 需要一组行为围绕同一份数据
- 需要清晰抽象某种对象

## 九、特殊方法和对象协议

你不需要一开始实现很多 dunder 方法，但至少要知道它们是对象能力的入口。

常见高频的有：

- `__init__`
- `__repr__`
- `__str__`
- `__eq__`
- `__len__`
- `__iter__`

### 为什么 `__repr__` 值得重视

调试时，好的对象表示能显著提升排查效率。

### 为什么 `__eq__` 要谨慎

你一旦定义对象相等规则，就等于定义了：

- 测试如何比较它
- 容器中如何判断它
- 业务上什么叫“同一个语义对象”

## 十、接口设计：关注能力，而不是出身

Python 很强调鸭子类型。  
工程里更重要的问题往往不是：

- “它是不是某个基类”

而是：

- “它有没有我需要的能力”

这意味着设计类接口时，要更关注：

- 输入输出边界
- 是否易替换
- 是否易 mock
- 是否依赖具体实现过深

## 十一、一个更实用的类设计检查清单

设计一个类前，可以先问：

1. 这个对象真的需要长期状态吗
2. 它的行为是否围绕同一份状态聚合
3. 用函数或 `dataclass` 是否已经足够
4. 是否有不必要的继承
5. 是否暴露了太多可随意修改的内部状态

## 十二、常见坏味道

- 类只是函数容器，没有真正对象状态
- 深继承链
- 多继承关系复杂
- 到处用 `@property` 藏逻辑
- 一个类既负责配置、业务、存储、网络调用
- 类属性里放可变默认值

## 十三、面试或 code review 最值得说清的点

- 为什么 Python 中组合通常比复杂继承更稳
- 为什么 `dataclass` 非常适合 DTO、配置和结果对象
- 为什么类属性放可变对象会有共享状态风险
- 为什么不是所有逻辑都应该塞到类里
- 为什么对象设计的重点是状态边界和职责边界
