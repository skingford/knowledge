# Python pytest 进阶与测试替身

会写 `pytest` 的基础测试，只是第一步。  
真正的 Python 项目一旦变大，很快就会碰到这些问题：

- 测试准备代码越来越重复
- 外部依赖太多，测试跑得慢又不稳定
- mock 越写越难看
- 不知道该写单元测试、集成测试还是接口测试

这篇文档重点讲 `pytest` 在真实工程里的高频用法和设计边界。

## 一、先建立一个目标

测试不是为了追求“看起来很专业”，而是为了：

- 降低修改风险
- 提高重构信心
- 让动态语言项目边界更清晰

如果测试写完后仍然：

- 很难改代码
- 很难定位失败原因
- 很依赖运气和环境

那测试设计本身也该被反思。

## 二、先分清几类测试

### 1. 单元测试

关注局部逻辑是否正确。

适合：

- 纯函数
- 规则判断
- 数据转换

### 2. 集成测试

关注模块之间组合起来是否正确。

适合：

- service + repository
- 接口 + 鉴权 + 校验
- 配置 + 启动流程

### 3. 端到端或接口测试

关注系统对外行为是否符合预期。

适合：

- HTTP API
- CLI 命令
- 任务入口

## 三、fixture 是 pytest 真正高效的关键

### 1. fixture 解决什么问题

它解决的是测试准备阶段的重复：

- 重复构造对象
- 重复初始化配置
- 重复创建临时目录

### 2. fixture 的设计原则

更稳的 fixture 通常具备：

- 名字清晰
- 作用小而明确
- 可组合
- 不带太多隐藏副作用

### 3. scope 要怎么选

常见 scope 包括：

- function
- class
- module
- session

经验上：

- 默认先用 function
- 只有明确需要复用且开销大时，再扩大 scope

scope 过大时，测试之间更容易互相污染。

## 四、`parametrize`：把一组规则测全

很多规则型逻辑，如果只测一个样例，保护力很有限。  
`parametrize` 非常适合：

- 边界值
- 状态组合
- 输入输出映射

```python
import pytest

@pytest.mark.parametrize(
    ("raw", "expected"),
    [("1", 1), ("2", 2), ("10", 10)],
)
def test_parse_int(raw, expected):
    assert int(raw) == expected
```

## 五、`monkeypatch` 和 mock 的边界

### 1. `monkeypatch` 适合什么

适合临时替换：

- 环境变量
- 模块函数
- 全局对象
- 路径或配置

### 2. mock 适合什么

适合验证：

- 某个依赖是否被调用
- 调用参数是否正确
- 某个外部协作是否按预期发生

### 3. 一个关键原则

不要为了 mock 而 mock。  
如果一段测试需要 mock 五六层依赖，通常说明设计本身耦合太深。

## 六、测试替身：fake、stub、spy、mock 怎么理解

### 1. fake

用一个可工作的轻量实现替代真实依赖。

适合：

- 内存版仓储
- 简化版缓存

### 2. stub

只提供固定返回结果。

适合：

- 外部接口结果替换
- 配置读取替换

### 3. spy

主要用来观察调用行为。

### 4. mock

强调交互验证。

更稳的工程经验：

- 能用 fake 的地方，通常比一堆 mock 更自然
- 只有真正要验证协作行为时，再强调 mock

## 七、`pytest` 里最值得优先掌握的几个工具

### 1. `pytest.raises`

适合验证异常路径。

### 2. `tmp_path`

适合测文件系统逻辑，不要手写很多临时路径和清理代码。

### 3. `caplog`

适合验证日志是否记录了关键上下文。

### 4. `monkeypatch`

适合替换环境变量和边界依赖。

## 八、异步测试怎么做

如果项目里有 `asyncio` 或异步 Web 服务，就不能只会同步测试路径。

你至少要关注：

- 异步函数如何运行
- 异步依赖如何替换
- 超时、取消、失败路径怎么测

更重要的是：

- 异步测试也要保持边界清晰
- 不要把一切都混成“跑起来就算行”

## 九、HTTP 接口测试怎么更稳

接口测试至少应覆盖：

- 正常返回
- 参数错误
- 鉴权失败
- 资源不存在
- 业务冲突
- 下游失败时的错误响应

不要只停在“能返回 200”。

## 十、数据和状态污染是测试不稳定的常见根因

常见问题来源：

- fixture 作用域过大
- 共享可变对象
- 测试之间依赖执行顺序
- 没有正确清理数据库或临时资源

测试如果依赖顺序，后面几乎一定会越来越脆。

## 十一、一个更稳的测试设计思路

写测试前先问：

1. 我要保护的是哪条规则
2. 这条规则属于单元、集成还是接口层
3. 哪些依赖应该替换，哪些应该真实接入
4. 如何让测试失败时足够容易定位

## 十二、什么时候该少 mock，多重构

如果测试越来越依赖：

- 深层 patch
- 很多返回值拼装
- 很多内部调用顺序验证

更值得做的通常不是继续堆 mock，而是：

- 抽清职责
- 减少隐藏依赖
- 提升接口可替换性

## 十三、常见坏味道

- 一个 fixture 做太多事
- 测试只验证“被调用了”，不验证结果语义
- 大量依赖内部实现细节
- 测试名称模糊，看不出保护哪条规则
- 同一类准备代码复制很多次

## 十四、面试或 code review 最值得说清的点

- 为什么 fixture 是 pytest 的核心能力
- 为什么 fake 往往比大量 mock 更稳
- 为什么测试要按层次区分单元、集成和接口
- 为什么测试不稳定通常是状态污染问题
- 为什么好的测试能反推代码边界设计
