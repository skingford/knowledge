# Python 测试、项目结构与工程化

很多 Python 项目一开始都很顺手，越写越乱的原因通常不是语言本身，而是工程习惯没有跟上。

最常见的症状是：

- 所有代码都堆在一个目录
- 依赖版本没人说得清
- 新同事拉下来跑不起来
- 不敢改，因为没测试
- 线上问题只能靠 `print`

这篇文档就是把 Python 最基本、最值得尽早建立的工程化习惯收起来。

## 一、先建立一个现实目标

不是每个 Python 项目都要上复杂平台化体系，但只要项目会被反复运行、反复修改、多人协作，就应该至少做到：

- 环境可复现
- 依赖可追踪
- 代码有结构
- 核心逻辑有测试
- 日志和配置有边界

## 二、项目结构：先把目录变清楚

### 1. 一个实用的最小结构

```text
my_project/
├── pyproject.toml
├── README.md
├── src/
│   └── my_project/
│       ├── __init__.py
│       ├── main.py
│       ├── config.py
│       ├── service.py
│       └── models.py
└── tests/
    ├── test_service.py
    └── test_models.py
```

### 2. 为什么建议这样分

- `src/` 放业务代码，避免导入路径混乱
- `tests/` 独立出来，方便测试组织
- 配置、模型、服务逻辑分模块，而不是全塞一个文件

### 3. 不同类型代码尽量分层

至少要分清：

- 入口层：命令行、HTTP、任务触发
- 业务层：规则和核心逻辑
- 基础设施层：数据库、文件、外部 API、消息队列

如果这些混在一起，后面测试和替换依赖会非常痛苦。

## 三、虚拟环境与依赖管理

### 1. 为什么虚拟环境是必须项

虚拟环境解决的是：

- 不同项目依赖冲突
- 全局 Python 环境污染
- 团队环境不一致

这是 Python 工程化的基础，而不是可选优化项。

### 2. 至少要做到什么

- 每个项目有独立环境
- 依赖安装有统一入口
- 团队知道如何创建和更新环境

### 3. 依赖要分层

至少区分：

- 运行依赖
- 开发依赖

开发依赖包括：

- 测试工具
- 格式化工具
- lint 工具
- 类型检查工具

### 4. 版本管理要有策略

不能长期依赖“本地正好能跑”。

至少要回答清楚：

- 新同事怎么装依赖
- CI 用什么版本
- 线上部署依赖版本怎么锁定

## 四、配置管理与环境变量

### 1. 不要把配置硬编码进业务逻辑

例如这些不应散落在代码里：

- 数据库地址
- 密钥
- 外部服务域名
- 超时配置
- 开关项

### 2. 配置建议分层

- 默认配置
- 环境变量覆盖
- 必要时加外部配置文件

### 3. 一个简单但重要的原则

配置应该在入口层收敛，而不是在业务代码里到处读取环境变量。

更稳的方式是：

- 入口层统一加载配置
- 校验后注入业务层

这样测试和复用都会容易很多。

## 五、日志：别把 `print` 当运维体系

### 1. 日志至少要有级别

常见级别：

- `DEBUG`
- `INFO`
- `WARNING`
- `ERROR`

### 2. 日志至少要有上下文

好的日志要能回答：

- 哪个模块出的事
- 哪个请求或任务出的事
- 输入条件是什么
- 错在哪一步

### 3. 工程上的几个基本建议

- 业务正常流程不要刷太多噪音日志
- 错误日志要带异常栈
- 不要打印密钥、Token、手机号、身份证等敏感信息

## 六、测试：先从最有价值的部分开始

### 1. Python 项目为什么更需要测试

因为它很多问题会在运行时才暴露。  
测试的价值尤其体现在：

- 帮你防止重构回归
- 降低动态语言带来的不确定性
- 让多人协作更敢改代码

### 2. `pytest` 为什么值得优先掌握

因为它：

- 写法简洁
- 生态成熟
- fixture 机制非常实用
- 团队普遍接受度高

### 3. 优先测什么

不要一上来追求“全覆盖”。  
优先测试：

- 纯函数
- 业务规则
- 边界条件
- 异常路径
- 很多人会改的核心模块

### 4. fixture 的价值

fixture 解决的是测试准备代码重复问题。

适合场景：

- 构造公共测试数据
- 初始化临时目录
- 提供 mock 依赖

### 5. 单元测试和集成测试都需要

#### 单元测试

关注局部逻辑是否正确。

#### 集成测试

关注模块之间拼起来是否正确。

例如：

- 配置加载 + 业务调用
- 数据库存取流程
- HTTP 接口到 service 层的打通

## 七、如何写出更好测的代码

### 1. 依赖注入优于硬编码依赖

如果函数内部直接创建数据库连接、HTTP 客户端或时间对象，测试会很难做。

更稳的设计是：

- 把依赖作为参数传入
- 或在对象初始化时注入

### 2. 把业务逻辑和 IO 分开

这是 Python 项目里非常高价值的一条经验：

- 业务规则尽量纯
- IO 操作放边界层

这样测试会简单很多。

### 3. 时间、随机数、外部请求都要可替换

这些是测试里最容易引入不稳定性的来源。

## 八、格式化、lint 与静态检查

### 1. 为什么这些工具不是“洁癖”

它们的价值不是“看起来更整齐”，而是：

- 降低 review 噪音
- 提前发现低级错误
- 保证团队一致性

### 2. 至少要有哪几类检查

- 代码格式化
- import 顺序
- 基础 lint
- 类型检查

### 3. 一个更实用的原则

工具规则应该帮助团队减少争论，而不是制造更多争论。  
所以要：

- 统一规则
- 自动执行
- 把风格问题交给工具，而不是 code review

## 九、命令行入口、任务脚本与发布边界

### 1. 入口代码要薄

无论是：

- `main.py`
- CLI 命令
- Web 路由
- 定时任务入口

都不应该塞满业务逻辑。

更稳的分工是：

- 入口负责接参数、组装依赖、处理错误
- 业务逻辑放 service 或 domain 层

### 2. 脚本不是不能工程化

即使只是内部工具脚本，只要会反复用，也应该逐步具备：

- 明确参数
- 日志
- 错误码
- 配置入口
- 基础测试

## 十、CI 的最小闭环

一个健康的 Python 项目，在提交时至少应该能自动做：

1. 安装依赖
2. 格式检查
3. lint
4. 类型检查
5. 测试

如果这些全靠本地自觉，稳定性通常会越来越差。

## 十一、几个高频工程误区

### 1. 业务代码里到处直接读环境变量

问题：

- 不好测试
- 不好统一校验
- 配置来源不透明

### 2. 所有逻辑写在入口文件

问题：

- 重用困难
- 测试困难
- 越改越重

### 3. 依赖版本完全不锁

问题：

- 今天能跑，明天未必能跑
- CI 和本地环境可能不一致

### 4. 测试只靠手点

问题：

- 回归成本高
- 动态语言风险高
- 重构不敢做

## 十二、一个更容易执行的落地顺序

如果你现在项目还比较乱，可以按下面顺序补：

1. 先加虚拟环境和依赖声明
2. 再整理目录结构
3. 再给核心逻辑补 `pytest`
4. 再接格式化、lint、类型检查
5. 最后再完善 CI 和发布流程

不要试图一次补齐所有体系，容易拖慢推进。

## 十三、面试或实际协作里最值得说清的点

- 为什么 Python 项目必须做环境隔离
- 为什么入口层应该薄、业务层应该纯
- 为什么测试优先覆盖核心规则而不是盲目追求覆盖率
- 为什么配置要集中收口而不是散落在代码里
- 为什么动态语言更需要格式、lint 和类型检查做前置约束
