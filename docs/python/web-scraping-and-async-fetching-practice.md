# Python 爬虫、HTTP 抓取与异步采集实践

很多人学 Python 到一定阶段，都会开始写抓取脚本、同步外部数据或定时采集任务。

这类任务最常见的问题不是“不会发请求”，而是：

- 代码第一次能跑，第二次就不稳
- 一并发就超时、封禁或把目标站点打挂
- 抓到的数据没法增量、去重和重跑
- HTML 解析逻辑特别脆，一改页面就全坏

这篇文档的目标，是把 Python 在爬虫、HTTP 抓取和异步采集里的工程主线收起来。

## 一、先建立边界：不是能抓就应该抓

在做抓取或采集前，先判断：

- 是否有官方 API
- 是否允许采集
- 是否有频率限制
- 是否涉及账号、隐私或受保护内容

更稳的原则是：

- 优先使用官方接口
- 遵守站点规则和速率限制
- 不绕过明确的安全边界

工程能力的重点是把任务做稳，不是去对抗站点。

## 二、先把“采集任务”看成一条完整流水线

一个更稳定的采集流程通常包括：

1. 发现目标链接或任务
2. 发起请求
3. 解析结构化数据
4. 清洗和规范化
5. 去重、入库或落盘
6. 记录状态，支持失败重试和增量续跑

真正容易出问题的，通常不是第 2 步，而是：

- 第 5 步是否幂等
- 第 6 步是否可恢复

## 三、HTTP 客户端怎么选

### 1. 同步抓取

适合：

- 任务量不大
- 流程简单
- 团队更熟悉同步模型

### 2. 异步抓取

适合：

- 请求量较大
- 目标主要是 IO 密集
- 你已经能处理超时、取消、限流和连接复用

### 3. 一个更重要的判断

先选“你能稳定维护的方案”，再谈理论并发上限。

对于很多内部采集任务来说：

- 稳定的同步方案

往往比：

- 结构混乱的异步方案

更有价值。

## 四、连接复用、超时和重试是基本盘

### 1. 不要每个请求都新建客户端

无论同步还是异步，都要重视：

- 连接池
- keep-alive
- session / client 复用

### 2. 超时必须显式设置

至少要考虑：

- 连接超时
- 读取超时
- 整体请求超时

如果没有超时，采集任务在网络波动时很容易无限挂住。

### 3. 重试要有边界

更稳的重试策略通常包括：

- 只对可重试错误重试
- 有退避，而不是毫无节制地连打
- 总重试次数有限

不要把所有异常都无脑重试。

## 五、抓网页时，解析策略比“选择器写法”更重要

### 1. 先判断数据来自哪里

常见来源包括：

- 静态 HTML
- 页面内嵌 JSON
- 后端接口返回 JSON
- 动态渲染后的 DOM

有时真正稳定的数据入口不是页面结构，而是页面背后的接口。

### 2. HTML 解析要尽量选择稳的锚点

更稳的做法是优先依赖：

- 语义清楚的属性
- 标题层级
- 固定结构区域

要避免过度依赖：

- 很长的 CSS 层级链
- 容易变化的样式类名
- 纯视觉位置关系

### 3. 解析失败要能发现

不要让解析异常静默吞掉。

更稳的做法是：

- 对关键字段做存在性校验
- 记录解析失败原因
- 为字段缺失建立监控或报警

## 六、同步、异步和并发控制怎么选

### 1. 不要把“异步”理解成“无限并发”

真正重要的是：

- 最大并发数
- 每域名限速
- 每任务超时
- 重试和退避

### 2. 并发过大常见后果

- 目标服务限流
- 本地连接耗尽
- CPU 在解析环节被打满
- 错误日志暴涨但没人知道哪部分真的成功

### 3. 一个更稳的策略

- 先从小并发跑通
- 再观察吞吐、错误率和目标站点稳定性
- 最后再逐步放大

## 七、任务必须可重跑、可去重、可续跑

这是采集任务和“一次性脚本”最重要的分界线之一。

### 1. 去重

要尽早确定：

- 数据的唯一键是什么
- URL 是否足够唯一
- 外部 ID 是否稳定

### 2. 幂等

你要能回答：

- 同一条数据重复抓到会怎样
- 同一批任务重跑会不会重复入库

### 3. 续跑

对长任务尤其重要：

- 当前跑到哪一步
- 哪些页面成功了
- 哪些页面失败了
- 下次从哪里继续

## 八、入库和落盘不要放到最后才考虑

很多抓取任务失败，不是“没抓到”，而是“抓到了但后面不可维护”。

要尽早设计：

- 原始数据是否留存
- 清洗后数据放哪里
- 错误记录是否保留
- 字段版本变化如何兼容

一个更稳的策略通常是：

- 原始输入和规范化结果分开存

这样出问题时更容易回放和修复。

## 九、反爬、异常站点和不稳定目标怎么处理

### 1. 先做正常工程手段

例如：

- 合理请求头
- 合理频率
- 重试退避
- 连接复用

### 2. 不要把绕过限制当成默认目标

如果站点已经明确设置了保护机制，更稳的选择通常是：

- 降低频率
- 改成合作接口
- 调整业务方案

### 3. 异常页面和脏数据要视为常态

采集任务里，以下情况都很常见：

- 404
- 302 跳转
- 空页面
- 半结构化字段
- 局部字段缺失

代码要围绕“这些情况会发生”来设计。

## 十、测试和观测不能省

### 1. 解析逻辑很适合写测试

优先测试：

- 关键 HTML 片段解析
- JSON 字段转换
- 缺字段时的降级行为

### 2. 采集任务至少要有这些指标

- 成功数
- 失败数
- 重试数
- 平均耗时
- 解析失败比例

### 3. 日志要能回答两类问题

- 哪个 URL 或任务失败了
- 失败发生在请求、解析还是入库阶段

## 十一、什么时候该从同步脚本升级成完整任务系统

如果你已经开始遇到这些情况，就该升级了：

- 任务量大到需要队列调度
- 抓取周期稳定重复
- 失败重试逻辑越来越复杂
- 需要多 worker 协同
- 需要监控和告警

这时要逐步引入：

- 任务队列
- 状态表
- 统一配置
- 并发治理

## 十二、常见坏味道

- 每个请求都重新创建客户端
- 没有超时，任务挂住只能手工杀
- 并发直接拉满，不做限速
- 解析逻辑完全依赖脆弱选择器
- 抓完直接覆盖结果，没有原始数据和失败记录
- 任务不能重跑，也不能断点续跑

## 十三、面试或 code review 最值得说清的点

- 为什么采集任务的关键不只是“会发请求”，而是可重跑和可恢复
- 为什么连接复用、超时和重试是 HTTP 采集基本盘
- 为什么异步抓取不等于无限并发
- 为什么 HTML 解析要围绕稳定锚点设计
- 为什么去重、幂等和状态记录决定了任务能不能长期维护
