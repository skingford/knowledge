# React 后台数据权限与列表查询参数设计

很多后台权限系统做到一半都会卡在一个地方：

页面权限已经做了，按钮权限也做了，但列表数据还是不稳。

典型现象是：

- 页面能进，也能看到列表
- 但到底该看到全部数据、组织内数据，还是仅自己数据，没有统一口径
- 筛选器、URL 参数、默认查询条件和权限范围互相覆盖
- 某些页面能导出全部，但列表本身却只该看部分

这说明问题已经不再只是“有没有权限”，而是：**这份权限到底对应什么数据范围，以及这个范围应该怎么落到查询参数里。**

这篇就聚焦这个最容易被低估、但又极其关键的点。

## 适合谁看

- 已经做了 React 中后台权限系统，但列表数据范围还在页面里手动拼
- 你的系统开始出现 `all / org / self` 这类数据权限层级
- 你想让列表页、筛选器、URL 参数和数据权限保持同一条主线

## 学习目标

- 区分页面权限、按钮权限和数据权限的边界
- 理解 `all / org / self` 这类 scope 该怎么影响查询参数
- 建立一套“权限 -> 数据范围 -> 默认查询条件 -> 可编辑筛选器”的稳妥设计

## 先给结论：数据权限要单独建模

先说最重要的一句：

**数据权限不是按钮权限的附属品，也不是页面权限的细节。**

它应该被单独建模。

更稳的拆法通常是：

- 页面权限：能不能进页面
- 按钮权限：能不能执行动作
- 数据权限：能看到哪个范围的数据

如果你把这三件事混在一起，列表页一定会越来越乱。

## 最常见的数据范围层级

后台最常见的第一版，通常就是这三层：

- `all`：可以看全部数据
- `org`：只能看当前组织 / 门店 / 部门数据
- `self`：只能看自己相关的数据

例如订单列表：

- `all`：可看全部订单
- `org`：只能看当前组织下订单
- `self`：只能看自己负责的订单

这个模型不一定覆盖所有业务，但已经足够作为统一入口。

## 为什么列表页最容易写乱

### 1. 权限范围和筛选条件被混成一件事

很多页面一开始会写成：

- 如果是自己数据，就默认塞 `ownerId = currentUser.id`
- 如果是组织数据，就默认塞 `orgId = currentOrgId`
- 如果是全部数据，就不塞条件

问题是这些条件到底是：

- 权限强制条件
- 默认筛选条件
- 还是用户可修改条件

如果不先区分，后面很快就会乱。

### 2. URL 参数和权限条件互相覆盖

比如用户手动在 URL 上带了别的 `orgId`，而页面又在代码里偷偷补了一个当前组织 ID。

最后你根本说不清：

- 到底哪个参数生效
- 用户是否真的越权了
- 页面显示的筛选项和实际请求条件是不是一致

### 3. 列表导出和列表查询走了两套口径

这是后台高频大坑：

- 页面列表按组织范围查
- 导出接口却没带同样范围

结果就是列表看起来没问题，但导出越权。

## 一套更稳的查询参数设计

更推荐的思路不是让页面自己拼权限条件，而是统一分三层参数：

1. **用户筛选条件**：用户自己在筛选器里选的东西
2. **默认条件**：页面初始状态给的条件
3. **强制权限条件**：无论用户怎么改都必须带上的条件

最终请求参数应该是：

**用户筛选条件 + 默认条件 + 强制权限条件**

但三者优先级不同，其中最重要的是：

- 强制权限条件永远不能被用户输入覆盖

## 一个更务实的数据结构思路

例如可以先抽成：

```ts
type DataScope = 'all' | 'org' | 'self'

type QueryConstraint = {
  orgId?: string
  ownerId?: string
}
```

再根据 scope 生成权限约束：

```ts
function buildScopeConstraint(scope: DataScope, context: {
  orgId?: string
  userId: string
}) {
  if (scope === 'all') return {}
  if (scope === 'org') return { orgId: context.orgId }
  return { ownerId: context.userId }
}
```

然后最终请求前再统一合并。

## 推荐的合并顺序

更稳的顺序一般是：

1. 初始化默认查询参数
2. 读取 URL 参数
3. 合并用户当前筛选条件
4. 最后覆盖强制权限条件

也就是说，**权限条件最后落地。**

这样用户无论怎么改筛选器，都不会突破权限范围。

## 筛选器、URL 和权限怎么配合

### 1. 先明确哪些字段是“可编辑筛选项”

例如：

- 订单状态
- 时间范围
- 关键词
- 标签

这些通常可以被用户自由修改。

### 2. 再明确哪些字段是“权限控制字段”

例如：

- `orgId`
- `ownerId`
- `storeId`

这些字段不应该总是暴露给用户自由输入。

### 3. 如果用户有更高权限，再决定是否开放筛选器

例如：

- `all` 权限用户可以切组织查看
- `org` 权限用户不能跨组织切换
- `self` 权限用户甚至不应该看到“负责人”筛选器

也就是说，筛选器本身也要受数据权限约束。

## 一个更稳的页面心智模型

以订单列表页为例，可以这样理解：

1. 用户有没有 `order.read`
2. 如果没有，不能进页面
3. 如果有，再看 `order.read` 对应的数据范围是 `all`、`org` 还是 `self`
4. 页面根据 scope 生成强制查询条件
5. 用户只在允许的维度上继续筛选
6. 列表查询和导出查询都复用同一份 scope 约束

这样页面、导出、筛选器、URL 参数就不会各走各的。

## 一个可直接落地的模板思路

```ts
// src/permissions/data-scope.ts
export type DataScope = 'all' | 'org' | 'self'

export function buildDataScopeConstraint(scope: DataScope, context: {
  userId: string
  orgId?: string
}) {
  switch (scope) {
    case 'all':
      return {}
    case 'org':
      return { orgId: context.orgId }
    case 'self':
      return { ownerId: context.userId }
  }
}
```

```ts
// src/features/order-management/queries/build-order-query.ts
export function buildOrderQuery({
  filters,
  defaultFilters,
  scopeConstraint,
}: {
  filters: Record<string, unknown>
  defaultFilters: Record<string, unknown>
  scopeConstraint: Record<string, unknown>
}) {
  return {
    ...defaultFilters,
    ...filters,
    ...scopeConstraint,
  }
}
```

这里最重要的不是代码多少，而是顺序：`scopeConstraint` 放最后。

## 导出接口要不要复用同一套约束

答案通常是：必须。

如果列表查询和导出查询不是同一套 scope 约束来源，后面几乎一定出事故。

更稳的原则是：

- 列表怎么限制
- 导出就怎么限制
- 批量操作怎么限制
- 详情查询怎么限制

至少在前端这一层，尽量统一走同一套 scope 构造逻辑。

## 最常见的反模式

### 1. 在页面里到处写 `if (scope === 'self')`

短期快，长期会让每个页面都长出自己的权限拼接逻辑。

### 2. 让 URL 参数覆盖权限字段

只要允许用户自由改 `orgId` 且最终直接带进请求，就很容易造成越权风险。

### 3. 把默认筛选条件和强制权限条件混在一个对象里

后面你会越来越难解释哪些条件是用户能改的，哪些是不能改的。

### 4. 列表页有限制，导出接口没限制

这是最常见、也最危险的权限假象之一。

### 5. 所有业务都硬套 `all/org/self`，不留扩展口

这个三层模型很好用，但不一定覆盖所有业务。某些系统还会有：

- `store`
- `department`
- `project`
- `custom`

所以实现上要允许继续扩展，而不是写死成永远只有三种。

## 最后给一个落地顺序

如果你现在正准备整理后台数据权限，建议按这个顺序做：

1. 先把页面权限和数据权限在概念上拆开
2. 给关键列表页补一层统一的 scope 约束构造函数
3. 让列表查询、导出、批量操作复用同一套约束来源
4. 再去收敛筛选器和 URL 参数的权限边界

这套顺序会比“每个页面单独修”稳很多。
