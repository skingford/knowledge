# React 后台多租户与多组织权限扩展设计

单组织、单角色的后台权限模型，很多团队都能做出来。

真正开始变复杂，通常是在业务往前走一步之后：

- 一个用户可能属于多个组织
- 一个组织下面又有多个空间、项目或门店
- 同一个用户在不同租户下角色不同
- 数据权限不再只是“能不能看页面”，而是“能看到哪个范围的数据”

这时候如果还用最原始的单层 permission key 思路硬扛，系统很快就会失控。

这篇的目标不是把权限系统讲成一套抽象教科书，而是从 React 中后台工程角度，回答一个现实问题：**当权限模型进入多租户 / 多组织场景后，前端应该怎么扩展，才不会把整个后台拖进泥潭。**

## 适合谁看

- 你的后台已经不是单一组织结构，而是开始出现租户、组织、空间、门店、项目等层级
- 你已经有基础权限系统，但开始遇到“同一个人切组织后权限变了”的问题
- 你想提前把多租户扩展点设计好，避免后面整体推倒重来

## 学习目标

- 区分租户隔离、组织归属、角色授权、数据范围这几层边界
- 理解为什么多租户场景下不能只靠角色名或简单 permission key
- 建立一套适合 React 后台扩展的上下文模型、切换模型和权限消费方式

## 先给结论：多租户权限要多一层上下文

在单组织后台里，你可能只需要回答：

- 当前用户有没有 `user.read`

但在多租户场景里，你真正要回答的通常是：

- 当前用户在**哪个租户**下
- 当前用户在**哪个组织 / 空间**下
- 当前上下文里有没有 `user.read`
- 这份权限对应的数据范围是什么

所以更准确的说法不是“用户有没有权限”，而是：

**用户在当前上下文里有没有某个权限。**

这个“上下文”通常至少包括：

- 当前租户 `tenant`
- 当前组织 / 空间 `organization` 或 `workspace`
- 当前激活身份或角色集合

## 最容易变复杂的 4 个维度

### 1. 租户隔离

这是最大的一层边界。

典型问题：

- 用户 A 属于租户 X，也属于租户 Y
- 切换租户后，菜单和页面权限都要变化
- 不同租户下的数据完全不能串

这层解决的是“你现在在哪个客户或业务隔离域里”。

### 2. 组织 / 空间归属

很多系统即使在同一个租户里，也还会继续细分：

- 部门
- 门店
- 项目组
- 工作区
- 子公司

这就意味着：就算租户不变，切到不同组织，权限和数据范围也可能不同。

### 3. 角色不是全局固定的

同一个用户完全可能出现：

- 在租户 A 是管理员
- 在租户 B 是普通成员
- 在组织 X 有导出权限
- 在组织 Y 只能查看

所以角色不能再被当成“用户身上固定不变的属性”。

### 4. 数据范围控制

到这一步，权限已经不仅仅是：

- 能不能进页面
- 能不能点按钮

还会变成：

- 能看全部订单，还是只能看自己负责的订单
- 能看所有门店数据，还是只能看当前门店
- 能编辑全部成员，还是只能编辑自己组内成员

这已经是“数据权限”问题，而不只是 UI 权限问题。

## 为什么单层权限模型会失效

### 1. permission key 不再是完整答案

`user.read` 这种 key 仍然有价值，但它只表示“动作类型”。

它通常回答不了：

- 在哪个租户下生效
- 在哪个组织下生效
- 数据范围是全部、组织内，还是仅本人

所以多租户场景里，permission key 仍然保留，但必须配合上下文和 scope 使用。

### 2. `currentUser.role` 基本失去表达力

在多租户系统里，下面这种写法会很快失效：

```tsx
currentUser.role === 'admin'
```

因为真正的状态更像：

- 当前租户是哪个
- 当前组织是哪个
- 当前租户下有哪些角色
- 当前组织下有哪些范围限制

### 3. 菜单和路由不再是“登录后固定一份”

单组织后台常见做法是：登录后拉一次菜单树，然后整天都用那一份。

多租户系统里不行，因为：

- 切租户会变
- 切组织会变
- 切工作空间会变

所以菜单、路由权限、按钮权限都要跟当前上下文绑定。

## 一套更稳的上下文设计

更建议的前端模型是把“权限”拆成两层：

1. **能力层**：permission key，本身只表达动作能力
2. **上下文层**：tenant / organization / workspace / scope，表达能力在哪个边界内生效

### 一个更务实的数据结构思路

例如后端返回给前端的，不要只是简单权限数组，而更像：

```ts
type PermissionGrant = {
  tenantId: string
  organizationId?: string
  permissions: string[]
  dataScopes?: Record<string, 'all' | 'org' | 'self'>
}
```

再进一步，当前前端真正消费的状态可以是：

```ts
type AccessContext = {
  currentTenantId: string
  currentOrganizationId?: string
  grants: PermissionGrant[]
}
```

然后所有权限判断都基于当前上下文做。

### 为什么要显式建 `AccessContext`

因为多租户系统里，“当前在哪”本身就是权限系统的一部分。

如果你没有显式上下文层，后面这些判断就会散得到处都是：

- 当前租户 ID
- 当前组织 ID
- 当前菜单树
- 当前页面权限
- 当前数据筛选条件

## 前端工程里该怎么组织

如果接前一篇目录结构，这里建议继续多加一层上下文能力：

```txt
src/
  app/
    providers/
      auth-provider.tsx
      permission-provider.tsx
      access-context-provider.tsx
  permissions/
    permission-keys.ts
    permission-service.ts
    data-scope.ts
  access/
    current-tenant.ts
    current-organization.ts
    access-context.ts
  navigation/
    menus.ts
    menu-builder.ts
```

一句话理解：

- `permissions/` 负责“能做什么”
- `access/` 负责“当前在哪个边界里做”

### 推荐补一个 `AccessContextProvider`

这个 provider 至少应该统一提供：

- 当前租户
- 当前组织 / 空间
- 切换方法
- 当前上下文对应的授权集合

这样页面和菜单都不需要自己拼上下文。

## 一个更合理的判断方式

单组织时你可能写：

```ts
hasPermission('order.read')
```

多租户时更合理的写法会更像：

```ts
hasPermission({
  permission: 'order.read',
  tenantId: currentTenantId,
  organizationId: currentOrganizationId,
})
```

或者更进一步，把上下文收进 provider，页面只写：

```ts
hasPermission('order.read')
```

但这个函数内部已经默认基于当前 `AccessContext` 判断，而不是全局静态判断。

## 数据范围要单独抽象

很多系统踩坑，是因为只做了按钮和页面权限，却没有单独处理数据范围。

更稳的做法通常是把数据范围单独抽一层，例如：

```ts
getDataScope('order.read') // 'all' | 'org' | 'self'
```

然后页面查询参数基于这个范围自动补过滤条件。

例如：

- `all`：不额外限制
- `org`：自动带上当前组织 ID
- `self`：自动带上当前用户 ID

这样“看得到页面”和“能看到哪些数据”才不会混成一件事。

## 菜单和租户切换怎么配合

一个高频问题是：切换租户之后，菜单要不要整棵重算？

答案通常是：要。

因为菜单不是纯静态导航，而是“当前上下文下的可见入口”。

更稳的流程通常是：

1. 用户切换租户 / 组织
2. 更新 `AccessContext`
3. 重新计算当前授权
4. 重新生成可见菜单
5. 校验当前路由是否仍然可访问
6. 如果不可访问，跳到当前上下文下的默认首页

这一步很关键，不然会出现：

- 菜单变了
- 但页面还停在无权限路由里

## 最常见的反模式

### 1. 只切顶部租户名，不切真实权限上下文

很多界面看起来支持租户切换，但实际上只是 UI 文案变了，权限和数据查询上下文没变。这是最危险的一类假切换。

### 2. 把组织 ID 写死在页面请求里

这会让多组织扩展变得非常难，因为每个页面都得回头改。

### 3. 用角色名表示数据范围

例如把 `org_admin`、`store_admin`、`self_user` 混成角色体系，最后权限语义会越来越乱。更稳的是把“动作权限”和“数据范围”拆开。

### 4. 切换上下文后不重新校验当前路由

这会导致用户停留在旧上下文可访问、但新上下文不可访问的页面里。

### 5. 前端和后端对上下文模型理解不一致

如果前端说“组织”，后端说“空间”，接口又返回“项目”，最后前端权限系统一定会被一层层适配拖垮。

## 最后给一个扩展顺序

如果你现在已经有单组织后台，准备往多租户走，建议按这个顺序扩：

1. 保留原来的 permission key，不要推翻重来
2. 先补当前租户 / 当前组织上下文
3. 再让权限判断函数绑定上下文
4. 然后把菜单和路由跟上下文联动起来
5. 最后单独补数据范围控制

这样改造成本会比“整套重写权限系统”低很多。
