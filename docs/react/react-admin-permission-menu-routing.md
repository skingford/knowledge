# React 后台权限、菜单与路由设计：一套不容易失控的方案

React 管理后台做到后面，最容易乱的往往不是组件本身，而是这三件事开始互相打架：

- 路由能不能进
- 菜单要不要显示
- 按钮和字段能不能操作

很多项目一开始把这三件事混在一起处理：

- 在路由组件里顺手判断菜单
- 在菜单配置里顺手写权限
- 在页面 JSX 里到处写 `role === "admin"`

短期能跑，长期一定会乱。

更稳的做法是：**把路由、菜单、权限看成三层相关但不同的系统，再用统一的权限模型把它们串起来。**

## 适合谁看

- 正在做 React 管理后台、运营平台、ERP、CMS、内部工作台
- 已经遇到“菜单看得到但点不进去”或“能进页面但按钮权限不对”的问题
- 想把权限设计从零散条件判断升级成一套长期可维护的工程结构

## 学习目标

- 区分路由权限、菜单权限、按钮权限、字段权限各自的职责
- 建立一套统一的权限模型，避免权限判断散在页面里
- 学会把 React Router、菜单配置和权限判断挂到同一条主线上

## 先给结论：后台权限设计的分层原则

如果你只记一件事，就记这个：

1. **路由决定能不能到页面**
2. **菜单决定要不要给入口**
3. **页面权限决定能不能看到操作区块**
4. **按钮 / 字段权限决定能不能执行具体动作**

也就是说，权限不是一个 `if`，而是一套层次化约束。

一个更务实的原则是：

- 不要在 JSX 到处直接写角色判断
- 不要让菜单配置自己发明一套权限字段
- 不要让路由守卫和按钮守卫各自维护一套规则
- 尽量让所有权限判断都回到一份统一的 permission key 上

## 最常见的 4 种权限层次

### 1. 路由权限

判断用户能不能进入某个页面，例如：

- 能否进入“用户管理”
- 能否进入“订单详情”
- 能否进入“系统设置”

它解决的是“入口之后的页面边界”。

### 2. 菜单权限

判断某个导航项要不要展示，例如：

- 左侧菜单是否显示“权限管理”
- 某个二级菜单是否对当前角色隐藏

它解决的是“是否给用户展示入口”。

### 3. 按钮权限

判断某个操作能不能触发，例如：

- 是否能新建
- 是否能删除
- 是否能导出
- 是否能审批

它解决的是“能看到页面，但不代表能执行动作”。

### 4. 字段权限

判断某些表单字段或详情字段是否：

- 可见
- 只读
- 可编辑
- 可脱敏显示

这类权限在后台非常高频，但经常被低估。

## 为什么权限逻辑总会失控

### 1. 角色名被直接写死在页面里

例如：

```tsx
{user.role === 'admin' && <DeleteButton />}
```

这样写的问题不是能不能跑，而是：

- 角色名会变化
- 新角色会增加
- 同一个动作不一定只由一个角色拥有
- 页面里到处写死后，后期根本改不动

### 2. 菜单、路由、按钮各有一套判断方式

常见混乱状态是：

- 菜单配置用角色名
- 路由守卫用权限码
- 按钮权限用布尔字段

结果就是三套系统表面都对，组合起来却经常冲突。

### 3. 把前端权限当成真正安全边界

前端权限的主要作用是：

- 控制展示
- 优化交互
- 减少无效操作

真正的安全边界必须在后端。

所以前端权限系统应该被理解为**界面与交互治理层**，不是最终授权系统。

### 4. 页面组件自己决定一切

如果每个页面都自由发挥：

- 自己判断菜单
- 自己判断按钮
- 自己判断字段禁用

那这个系统一定无法长期统一。

## 一套更稳的设计模型

更建议的结构是：

- **权限源**：后端返回的 permission keys
- **前端权限中心**：统一的 `hasPermission` / `hasAnyPermission` / `hasAllPermissions`
- **路由配置**：挂接访问所需权限
- **菜单配置**：挂接展示所需权限
- **页面组件**：只消费统一权限能力，不自己发明规则

### 第一步：先统一 permission key

不要直接以角色名作为判断主轴，而是统一抽象成权限点，例如：

```txt
user.read
user.create
user.update
user.delete
role.read
role.assign
order.read
order.export
system.config.update
```

这样做的好处是：

- 角色和权限解耦
- 一个角色可以映射多个权限点
- 新角色加入时不需要改页面逻辑
- 按钮、菜单、路由都可以复用同一套 key

### 第二步：建立统一权限判断函数

例如只暴露少量核心能力：

```ts
hasPermission('user.create')
hasAnyPermission(['order.read', 'order.export'])
hasAllPermissions(['role.read', 'role.assign'])
```

页面层只依赖这些能力，不直接解析用户角色结构。

### 第三步：让路由和菜单都挂接同一套权限 key

路由配置和菜单配置里都只挂 permission keys，而不是各写一套布尔逻辑。

### 第四步：按钮和字段权限走组件级封装

与其每页手写：

```tsx
{hasPermission('order.export') ? <Button>导出</Button> : null}
```

更稳的是抽一层：

```tsx
<PermissionGate permission="order.export">
  <Button>导出</Button>
</PermissionGate>
```

这样页面会干净很多，也更容易全局统一行为。

## 菜单、路由、权限如何映射

一个务实原则：**菜单是路由的可视化投影，但不是路由本身。**

也就是说：

- 不是所有路由都要出现在菜单里
- 菜单项通常应该指向一个路由
- 菜单显示与否，应该参考访问权限，但不应替代路由校验

### 推荐的数据结构思路

可以把路由和菜单配置拆成两层：

1. **路由记录**：定义路径、组件、所需权限、布局
2. **菜单记录**：定义标题、图标、排序、对应路由、展示权限

例如：

```ts
const routes = [
  {
    path: '/users',
    permission: 'user.read',
    element: <UserPage />,
  },
]

const menus = [
  {
    key: 'users',
    title: '用户管理',
    to: '/users',
    permission: 'user.read',
  },
]
```

### 为什么不要让菜单代替路由守卫

因为隐藏菜单不等于不能访问页面。

如果用户直接输入 URL：

- 菜单系统拦不住
- 只有路由守卫或页面入口权限检查能拦住

所以正确关系应该是：

- 菜单隐藏：减少无权限入口
- 路由守卫：阻止真正进入页面
- 页面内部权限：限制页面内操作能力

## 一套更清晰的工程分工

### 1. `permissions/` 目录

放：

- permission key 常量
- 权限判断函数
- 角色到权限的映射适配
- `PermissionGate` 这类权限组件

### 2. `router/` 目录

放：

- 路由配置
- 守卫逻辑
- 需要登录、需要权限、匿名可访问等边界

### 3. `menus/` 或导航配置

放：

- 菜单树
- 分组
- 图标
- 排序
- 对应路由
- 展示权限

### 4. 页面层

页面层只做消费：

- 不解析角色原始数据
- 不写散落的角色名判断
- 尽量通过统一权限函数或组件做判断

## 一个更实用的页面判断顺序

以“用户管理”页为例，可以按下面顺序理解：

1. 用户登录了吗
2. 有 `user.read` 吗
3. 有的话允许进入页面
4. 页面里是否有 `user.create`
5. 有的话显示“新建用户”按钮
6. 是否有 `user.delete`
7. 有的话显示删除操作，否则隐藏或禁用

这样你会发现：**同一个页面可以有页面级权限，也可以有动作级权限。**

## 字段权限为什么要单独看

很多后台做了一半才发现，字段权限和按钮权限根本不是一个问题。

例如：

- 财务金额只能部分角色看到
- 手机号要脱敏
- 某些字段只能查看不能编辑
- 某些审批状态下字段全部只读

这时不要再用“能不能点按钮”的思路硬套，而是单独抽象：

- `canViewField`
- `canEditField`
- `maskField`

字段权限经常和业务状态一起出现，所以要特别注意别把它们全塞进表单组件里写死。

## 哪些反模式最容易踩

### 1. 在每个组件里直接读取 `currentUser.role`

这会让权限逻辑散到整个项目里，后期无法替换。

### 2. 菜单隐藏就当作没有权限

这会导致直接访问 URL 时出现漏控。

### 3. 一个页面只有“能进 / 不能进”两种状态

后台大量页面其实是“能进，但只能看；能看，但不能改”。

### 4. 把按钮权限和字段权限混为一谈

“能提交”不等于“所有字段都能编辑”。

### 5. 前端不和后端统一 permission key

前后端各叫各的名字，最后一定需要大量映射补丁。

## 最后给一个务实建议

如果你现在就在做 React 后台权限系统，优先做这几件事：

1. 先统一 permission key，不要再让角色名直接进页面层
2. 把路由权限、菜单权限、按钮权限、字段权限明确分层
3. 给项目补一个统一权限中心，而不是每页自己判断
4. 菜单隐藏、路由守卫、页面内按钮控制三层都补齐

这样后面无论是加新角色、改权限模型，还是做多租户 / 多组织扩展，都会稳很多。
