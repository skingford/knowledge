# React 高频问题：渲染、状态与副作用排错清单

React 很多问题看起来像“玄学”，其实都能回到几个固定原则：

- state 是否最小且来源单一
- effect 是否只在同步外部系统
- key 是否稳定
- 组件位置是否变化
- 渲染阶段是否保持纯净

这篇文档把高频问题压缩成一份排错清单，方便在开发和复盘时快速定位。

## 1. 为什么 `setState` 后马上打印，还是旧值？

因为一次事件处理中的读取，看到的是当前这次渲染的快照。调用 state 更新函数后，React 会安排下一次渲染，而不是立刻把当前作用域里的变量改掉。

稳妥写法：

- 不要假设 `setX` 后当前作用域里的 `x` 立刻变化
- 如果新值依赖旧值，用函数式更新

```tsx
setCount((current) => current + 1)
```

## 2. 为什么列表不能随便用 `index` 当 key？

因为 key 用来标识“这是不是同一个元素”。当插入、删除、排序发生时，`index` 会漂移，React 可能把旧 state 复用到错误项上，表现为：

- 输入框内容串行
- 勾选状态错位
- 动画和焦点跳动

优先使用稳定 ID。

## 3. 为什么 effect 一直死循环？

最常见的原因有两个：

1. effect 里更新了它自己依赖的 state
2. 依赖数组里放了每次渲染都会新建的对象或函数

先不要急着“删依赖”，先反查：

- 这段逻辑真的需要 effect 吗
- 这是不是派生 state
- 这个对象或函数能不能移出 effect，或者让依赖更稳定

## 4. 为什么开发环境里 effect 会执行两次？

在开启 Strict Mode 的开发环境中，React 会额外做一次 setup 和 cleanup，帮助你发现不安全的副作用。这是开发期检查，不代表线上一定会双执行。

如果因此报错，通常说明你的 effect：

- 没有正确清理
- 订阅逻辑不幂等
- 依赖了“只允许执行一次”的全局副作用

## 5. 为什么“把 props 复制到 state”经常出问题？

因为你等于维护了两份真相：

- 父组件 props 一份
- 子组件本地 state 一份

后面只要有一边更新不同步，就会出现 UI 不一致。只有在“初始化后允许脱离 props 独立编辑”的场景下，才考虑用 props 初始化本地 state，并明确后续同步策略。

## 6. 如何正确重置一个组件的内部 state？

看你想重置的是哪一层：

- 只重置某个字段：直接更新对应 state
- 重置整个表单：恢复到初始 state
- 重置整棵子树：给子组件换一个新的 `key`

通过 `key` 重置子树，本质是在告诉 React：“这已经不是同一个组件实例了。”

## 7. 为什么条件渲染切换后，state 有时会丢，有时不会？

因为 React 记住 state 的依据，不是 JSX 文本长得像不像，而是组件在树里的位置。

要点：

- 位置不变，state 可能保留
- 位置改变或 key 改变，state 会重置

排查这类问题时，先看组件树位置和 key，而不是先怀疑 Hook。

## 8. 什么时候该用 `ref`，什么时候该用 `state`？

用 `state`：

- 值变化后要更新 UI

用 `ref`：

- 值变化不需要触发 UI 更新
- 要拿 DOM
- 要保存定时器 ID、上一次值、请求中标记等

如果一个值放进 state 后完全不参与渲染，就很可能放错了。

## 9. `Context` 会不会让整棵树都重渲染？

不会“整棵树都重渲染”，但会让读取该 Context 的消费者在 value 变化时重新读取。实际问题通常出在：

- 一个 Context 里塞太多东西
- value 每次都创建新对象
- 高频变化数据和低频配置数据混在一起

改进方向：

- 拆 Context
- 缩小消费者范围
- 不要把整个页面状态一股脑塞进全局 Context

## 10. 受控输入为什么会卡顿？

先看是不是每次输入都触发了太多额外工作，例如：

- 大列表过滤
- 大表单联动
- 整页重渲染
- effect 里又触发其他更新

排查顺序通常是：

1. 把输入 state 尽量局部化
2. 把昂贵计算移出输入链路
3. 必要时用 `useDeferredValue`
4. 再考虑局部 `memo`

## 11. `useMemo`、`useCallback`、`memo` 是不是应该到处加？

不是。它们是性能优化工具，不是代码结构工具。

更稳的顺序是：

1. 先确认真的有性能问题
2. 先优化 state 位置、effect 设计和组件边界
3. 再局部使用缓存

如果项目已经启用 React Compiler，可以进一步减少手写 `memo`、`useMemo`、`useCallback` 的需求；如果没启用，仍然要按传统方式分析热点后再加。

## 12. 为什么点击按钮后总是拿到旧闭包里的值？

这通常不是 React “失效”，而是函数在创建时捕获了当时的变量。常见场景：

- `setInterval`
- 事件监听器
- Promise 回调
- 第三方库回调

排查方向：

- 是否该改成函数式更新
- 是否该在 effect 里重新订阅
- 是否该用 ref 持有最新值

## 13. 为什么“删除依赖数组里的变量”看起来能让问题消失，但后面更难查？

因为 effect 依赖描述的是 effect 真正读取了什么。硬删依赖，通常只是把逻辑改成“碰运气同步”。表面看起来不循环了，实质是留下了过期数据和隐藏 bug。

更稳的处理顺序：

- 先问这段逻辑是否需要 effect
- 再问能否把依赖对应的值改成派生值或局部变量
- 最后再调整 effect 结构

## 14. 为什么请求逻辑放在 effect 里后，取消、竞态和闪烁问题变多了？

因为请求不是“发出去就完”，还涉及：

- 依赖切换时旧请求是否取消
- 返回顺序是否和发起顺序一致
- 组件卸载后是否还会回写 state

如果 effect 负责请求，就要同步把取消、清理和竞态处理一起设计。

## 15. React 页面变慢时，第一步该看什么？

不是先加缓存，而是先确认慢在哪里：

- 渲染慢
- 提交慢
- 列表慢
- 输入卡
- 网络慢
- 包体积大

优先用：

- React DevTools Profiler 看组件重渲染
- 浏览器 Performance 面板看主线程阻塞
- Network 面板看请求和资源加载

## 一份务实的排错顺序

1. 先确认现象属于状态错乱、effect 错误、列表复用错误，还是纯性能问题
2. 看 state 是否重复、来源是否唯一、位置是否合理
3. 看 effect 是否在做本不该做的事
4. 看 key 是否稳定、组件位置是否变化
5. 再用 Profiler 或 Performance 面板量化热点
