# React 性能优化指南：渲染分析、列表性能与稳定性优化

React 性能优化最容易走偏的地方，是把优化理解成“给每个组件都包一层缓存”。真正更稳的顺序通常是：

1. 先量化问题
2. 先优化状态结构和组件边界
3. 再局部用缓存、分片和列表治理

这篇文档重点整理 React 性能问题的排查顺序和常见优化手段。

## 适合谁看

- 页面已经能跑，但开始出现切换卡顿、输入掉帧、长列表慢和频繁重渲染
- 团队里已经开始大量使用 `memo`、`useMemo`、`useCallback`，但收益和代价都说不清
- 需要把性能问题从“感觉慢”变成“有数据、有定位、有改法”

## 学习目标

- 分清 React 性能问题来自哪里
- 掌握 DevTools Profiler 和浏览器面板的基础用法
- 建立从状态局部化到列表治理再到 memo 化的优化顺序
- 能处理输入卡顿、长列表、重计算和包体积等高频问题

## 先判断慢在哪

React 页面慢，常见来源大致有五类：

- 不必要的重渲染
- 大列表渲染
- 渲染内有昂贵计算
- 输入链路阻塞主线程
- 包体积和资源加载过大

### 先用工具量化

优先使用：

- React DevTools Profiler：看哪些组件频繁重渲染、每次耗时多少
- 浏览器 Performance 面板：看主线程是否被长任务阻塞
- Network 面板：看首屏资源、接口和瀑布图

如果没有量化数据，就很容易陷入“这块看起来应该慢”的错觉。

## 最优先的优化：状态局部化

React 官方文档一直强调，性能优化首先不是缓存，而是让 state 不要提得过高。

### 一个典型误区

把输入框、弹层开关、hover、高亮、展开折叠都放在页面最外层，结果任何一点变化都让整页重渲染。

更稳的做法：

- 输入框 state 放在输入附近
- 行级展开状态放在行组件或列表局部
- 只对真正共享的数据做状态提升

### 局部化的收益

- 减少渲染波及范围
- 让组件职责更清晰
- 后续更容易判断是否真的需要 `memo`

## 把不必要的 Effect 清掉

很多性能问题表面是“渲染多”，本质是 effect 触发了额外更新。

常见问题：

- 把派生值放到 effect 里再 `setState`
- effect 依赖变化后又触发链式更新
- 输入变化时 effect 里做过多同步逻辑

优化方向：

- 派生值直接在渲染阶段计算
- 用户动作驱动的逻辑放回事件处理函数
- effect 只保留同步外部系统的职责

## 列表性能：稳定 key、拆分与虚拟化

长列表是 React 性能问题最高发的场景之一。

### 第一步：保证 key 稳定

如果 key 不稳定，React 连“是不是同一项”都判断不准，后面的 memo 和虚拟化都很难奏效。

### 第二步：把每一项做成职责单一的行组件

不要让父组件一边做筛选、一边做排序、一边直接渲染很复杂的列表项。更稳的做法是：

- 父组件负责数据准备
- 子项组件专注渲染和局部交互

### 第三步：必要时做虚拟化

当列表项很多且同时在屏幕内显示的只有一小部分时，优先考虑窗口化 / 虚拟化，而不是先堆更多 memo。

适合虚拟化的信号：

- 数百到数千条以上
- 每项都比较重
- 滚动明显卡顿

## 缓存工具的正确使用边界

### `memo`

适合：

- 子组件渲染成本明显较高
- 父组件经常重渲染
- 子组件在同样 props 下输出稳定

不适合：

- 组件本身很轻
- props 每次都会变
- 只是为了“感觉更专业”

### `useMemo`

适合：

- 有昂贵计算
- 或者需要给 memoized child 提供稳定对象 / 数组引用

```tsx
const visibleTodos = useMemo(
  () => todos.filter((todo) => todo.title.includes(keyword)),
  [todos, keyword],
)
```

### `useCallback`

适合：

- 函数作为 props 传给已 memo 的子组件
- 函数作为依赖，且你明确需要稳定引用

不要为了“防止函数重复创建”而默认加。函数本身通常不贵，错误的缓存反而会增加理解成本。

### React Compiler 的影响

如果项目已经启用 React Compiler，官方文档说明它可以自动做一部分类似 `memo` 的优化，这会降低手写 `memo`、`useMemo`、`useCallback` 的必要性。

但前提是：

- 你的组件本身足够纯
- 项目已经实际接入并启用编译器

没有启用时，仍然需要按传统方式做热点分析和手工优化。

## 让输入和交互更顺滑

### 场景 1：输入时同时触发重筛选

如果用户每输入一个字符，就会触发大列表筛选和重渲染，可以考虑把输入值和“慢结果”分开：

```tsx
const deferredKeyword = useDeferredValue(keyword)
const visibleTodos = useMemo(
  () => todos.filter((todo) => todo.title.includes(deferredKeyword)),
  [todos, deferredKeyword],
)
```

### 场景 2：切换筛选条件时页面大面积更新

如果某个更新不是必须立刻阻塞交互，可以考虑 `startTransition`，把较重的更新标成低优先级。

```tsx
function handleFilterChange(nextFilter: string) {
  setFilterInput(nextFilter)

  startTransition(() => {
    setAppliedFilter(nextFilter)
  })
}
```

注意：`startTransition` 解决的是交互流畅性，不是替你消灭重活。真正的重计算和重渲染，仍然需要拆解。

## 代码分割与 Suspense

不是所有性能问题都发生在渲染阶段，很多问题出在包太大、首屏加载太重。

### 可以优先拆分的内容

- 路由级页面
- 富文本编辑器、图表、地图、可视化等重组件
- 低频弹层和管理后台功能块

### 务实建议

- 先按路由做分割
- 再拆低频大依赖
- 对于异步加载组件，用 `Suspense` 提供清晰的加载占位

## 一份性能排查清单

### 现象是“切换慢”

- 看是不是父组件持有太多无关 state
- 看列表和区块是否拆得过粗
- 看是否有不必要 effect 触发额外更新

### 现象是“输入卡”

- 看输入 state 是否放太高
- 看输入时是否触发了过滤、排序、图表刷新
- 看是否需要 `useDeferredValue`

### 现象是“列表卡”

- 看 key 是否稳定
- 看是否真的需要渲染这么多项
- 看是否要虚拟化
- 再看是否要对子项加 `memo`

### 现象是“首屏慢”

- 看 bundle 体积
- 看是否可以路由级拆包
- 看图片、字体和第三方脚本

## 一个推荐的优化顺序

1. 先用 Profiler 或 Performance 面板确认热点
2. 先缩小 state 影响范围
3. 清掉不必要的 effect 和派生 state
4. 处理列表 key、拆分和虚拟化
5. 对明显热点再加 `memo`、`useMemo`、`useCallback`
6. 最后处理代码分割、资源加载和首屏体验
