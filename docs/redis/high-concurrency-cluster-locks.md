# Redis 高并发、集群部署与分布式锁

这篇文档专门补 Redis 在高并发场景下的生产级知识：什么时候该上集群，热点 Key 和大 Key 怎么治理，分布式锁到底能解决什么问题，Lua 脚本在并发控制里该怎么用。

如果你已经看过 [Redis 实践与缓存穿透/击穿/雪崩](./redis-and-cache-patterns.md)，这篇可以继续往“高并发治理”和“生产部署”方向补齐。

## 一张总表先看清

| 主题 | 典型问题 | 核心关注点 |
| --- | --- | --- |
| 高并发治理 | 热点 Key、回源风暴、慢命令 | 限流、分片、降级、批量化、异步化 |
| 集群部署 | 单机扛不住、故障切换、容量不足 | Sentinel、Cluster、持久化、主从拓扑 |
| 分布式锁 | 重复执行、并发冲突 | 加锁原子性、解锁安全性、锁续期、最终兜底 |
| Lua 脚本 | 多步判断 + 写入需要原子执行 | 原子性、脚本时长、跨槽限制、EVALSHA |

## Redis 在高并发链路里到底承担什么角色

Redis 在高并发系统里通常不是“最终正确性系统”，而是下面四类角色：

1. 热点数据缓存：减少数据库读压力
2. 并发控制器：做锁、去重、限流、滑动窗口
3. 短状态存储：会话、验证码、短期幂等结果
4. 削峰缓冲层：做轻量队列、异步聚合、预扣减

一旦定位清楚这个角色，设计上就会更稳：

- 读性能和低延迟交给 Redis
- 最终一致性和资金正确性交给数据库或状态机
- Redis 故障时必须有降级和兜底思路

## 高并发场景最常见的五个坑

### 1. 热点 Key

某个 Key 的访问量远高于其他 Key，单点命中同一条数据，容易把单线程 Redis 的单核处理能力吃满。

典型场景：

- 热门商品库存
- 首页活动配置
- 爆款排行榜
- 某个商户的超热点配置

常见治理方式：

- 永不过期 + 后台异步刷新
- 本地缓存 + Redis 二级缓存
- 对可聚合的数据做分片，例如把一个计数器拆成多个分片 Key
- 在入口加限流，避免热点失效时瞬时回源

### 2. 大 Key

单个 Key 挂载的数据太大，比如几十万元素的 List、Hash、Set、ZSet。

风险点：

- 删除、遍历、迁移成本高
- 主从复制和集群迁移时延迟明显
- 容易触发慢查询，拖垮主线程

治理建议：

- 单个集合按业务维度拆分
- 避免超大 Hash 和超长 List
- 删除大 Key 优先用异步删除能力，避免阻塞主线程
- 对排行榜和窗口统计控制元素规模，定期裁剪

### 3. 回源风暴

热点 Key 失效时，大量请求同时回源数据库或下游服务。

应对方式：

- 热点数据逻辑过期 + 异步刷新
- `singleflight`、互斥锁或请求合并
- 下游高压时直接降级返回默认值

### 4. 慢命令

Redis 单线程最怕 O(N) 或 O(N log N) 的命令在高峰期长时间占住主线程。

典型危险操作：

- `KEYS *`
- 大范围 `SMEMBERS`
- 超大 `ZRANGE`
- 对大 Key 做全量遍历

生产原则：

- 宁可多做结构设计，也不要在线上高峰扫大集合
- 优先使用分页、游标、分桶和裁剪

### 5. 网络往返太多

Redis 很快，但如果每个请求都要发十几条小命令，RT 还是会被网络开销拖垮。

优化手段：

- 批量读写优先用 Pipeline
- 多步判断 + 更新优先合并成 Lua
- 避免在关键链路里频繁做小颗粒度往返

## 集群部署到底怎么选

很多团队一上来就说“Redis 上集群”，但其实要先分清你到底是要**高可用**还是要**水平扩容**。

| 方案 | 解决什么问题 | 适合场景 | 主要限制 |
| --- | --- | --- | --- |
| 单实例 | 最简单、最低维护成本 | 开发环境、小流量服务 | 无高可用，容量和 QPS 都有限 |
| 主从 + Sentinel | 自动故障转移 | 需要高可用，但数据量和写流量还没到必须分片 | 不能解决单主写入瓶颈 |
| Redis Cluster | 分片 + 高可用 | 数据量大、写入高、需要水平扩展 | 多 Key 操作有跨槽限制 |
| 客户端分片（Ring） | 简化分片 | 某些轻量场景或历史系统 | 运维和迁移体验不如 Cluster |

### 主从 + Sentinel 适合什么场景

如果你的核心诉求是：

- 想要自动故障切换
- 数据量还没大到必须分片
- 大多数请求是读，写集中在一个主节点也能扛住

那么优先考虑主从 + Sentinel。它解决的是高可用，不是水平扩容。

### Redis Cluster 适合什么场景

如果你的问题变成了：

- 单主内存已经接近上限
- 写 QPS 已经接近单实例瓶颈
- 热点数据很多，必须把数据分散到多个主节点

那才该上 Redis Cluster。

Cluster 的关键点要记住：

- 数据按 16384 个 hash slot 分布
- 一个主节点负责一部分 slot
- 每个主节点最好至少配一个从节点
- 多 Key 命令要求 Key 落在同一个 slot

### 生产部署建议

高并发生产环境一般至少这样起步：

- 3 主 3 从，确保主节点故障后还能自动切换
- 开启 AOF，每秒刷盘一次作为常见折中
- RDB 作为周期性备份，不单独依赖 RDB 保证恢复点
- 监控内存、慢查询、复制延迟、网络流量、连接数和主从切换事件

### 什么时候别急着上 Cluster

如果你现在的问题只是：

- 缓存命中率低
- 热点 Key 设计不合理
- 慢命令太多
- 代码层面没有用 Pipeline / Lua

那先别急着上 Cluster。很多 Redis 的“扛不住”其实是使用方式不合理，不是部署模式落后。

## Cluster 下的 Key 设计要点

Redis Cluster 最大的变化不是“多台机器”，而是**多 Key 原子操作不再天然成立**。

比如这些操作都要求相关 Key 落在同一个 slot：

- Lua 同时操作多个 Key
- 事务里同时改多个 Key
- 一次性对多个 Key 做强一致判断

这时候要用 **hash tag**：

```text
order:{123}:lock
order:{123}:status
order:{123}:quota
```

只要大括号里的内容一样，这些 Key 就会落到同一个 slot，才能在 Cluster 下安全做 Lua 或事务。

一个实战原则：

- 需要一起原子操作的 Key，设计时就要放同一 slot
- 不需要一起原子操作的 Key，尽量均匀分散，避免单 slot 过热

## 分布式锁到底能解决什么问题

Redis 分布式锁能解决的是：

- 同一时刻只允许一个 worker 处理某个任务
- 减少重复执行和并发冲突
- 在高并发入口处先挡一波流量

Redis 锁**不能**解决的是：

- 资金类最终正确性
- 跨系统事务一致性
- 锁失效后仍然绝不出错

所以正确表述应该是：

> Redis 锁负责“尽量少冲突”，数据库唯一索引、乐观锁、状态机负责“最终不做错”。

## 分布式锁的标准做法

### 1. 加锁必须原子

标准写法：

```text
SET lock:order:123 unique_value NX PX 30000
```

这里四个点缺一不可：

- `NX`：只有不存在时才加锁
- `PX 30000`：锁一定要有 TTL，避免死锁
- `unique_value`：标识锁归属，防止误删别人的锁
- 整条命令原子执行

### 2. 解锁必须校验归属

不能直接 `DEL lock:key`，否则会出现经典误删：

1. A 拿到锁
2. A 执行太久，锁过期
3. B 拿到同名新锁
4. A 执行结束后把 B 的锁删掉

正确做法是 Lua：

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
  return redis.call("DEL", KEYS[1])
end
return 0
```

### 3. TTL 要覆盖业务执行时长

锁 TTL 太短，业务还没执行完锁就过期；TTL 太长，异常时其他请求要等很久。

经验上至少要覆盖：

- 业务处理的 P99 时延
- 下游 RPC 波动
- 数据库慢 SQL 和重试时间

### 4. 长任务要考虑续期

如果临界区执行时间不可预测，比如：

- 大批量补偿任务
- 聚合计算
- 远程调用链路长

那就要考虑 watchdog/续期机制，而不是只靠固定 TTL。

但要注意：续期失败并不等于业务一定失败，所以最终一致性仍然不能靠锁兜底。

## 分布式锁的几个典型误区

| 误区 | 问题 | 正确做法 |
| --- | --- | --- |
| 只用 `SETNX` 不设过期 | 宕机会造成死锁 | 必须带 TTL |
| 直接 `DEL` 解锁 | 可能删掉别人的锁 | 校验 value 再删除 |
| 锁住了就一定正确 | 锁失效后仍可能并发写错 | 数据库约束兜底 |
| 任何场景都上 Redlock | 复杂度高，收益未必匹配 | 先把单实例锁模式用对 |
| 在 Cluster 下随意做多 Key Lua | 可能跨槽失败 | 用 hash tag 让相关 Key 同槽 |

## Lua 脚本在高并发里的正确用法

Lua 的价值不是“能写脚本”，而是把多条命令合成一次原子执行，减少并发竞争和网络往返。

典型适用场景：

- 校验并释放锁
- 限流计数
- 库存判断后扣减
- 幂等判断 + 状态写入

不适合的场景：

- 长时间运行的复杂脚本
- 大量遍历大集合
- 跨很多 Key 的复杂编排

### 例子 1：安全释放锁

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
  return redis.call("DEL", KEYS[1])
end
return 0
```

### 例子 2：限流时原子计数

```lua
local current = redis.call("GET", KEYS[1])
if not current then
  current = 0
else
  current = tonumber(current)
end

if current + 1 > tonumber(ARGV[1]) then
  return 0
end

redis.call("INCR", KEYS[1])
redis.call("EXPIRE", KEYS[1], ARGV[2])
return 1
```

### 例子 3：库存扣减

```lua
local stock = tonumber(redis.call("GET", KEYS[1]) or "-1")
local need = tonumber(ARGV[1])

if stock < 0 then
  return -1
end

if stock < need then
  return 0
end

redis.call("DECRBY", KEYS[1], need)
return 1
```

对应业务语义：

- `-1`：库存 Key 不存在
- `0`：库存不足
- `1`：扣减成功

### Lua 在线上要注意什么

1. 脚本要短小，避免长时间阻塞 Redis 主线程
2. 热路径优先使用 `EVALSHA`，减少脚本文本重复传输
3. Cluster 下脚本涉及的多个 Key 必须落同一 slot
4. 不要把 Lua 当数据库事务，它不提供自动回滚

## 高并发场景下 Redis 的推荐组合拳

如果你在做秒杀、支付、抢券、风控这类系统，常见组合一般是：

1. 热点数据缓存：本地缓存 + Redis
2. 瞬时并发控制：Redis 锁 / 限流 / 去重
3. 原子判断更新：Lua
4. 最终一致性：数据库唯一索引、乐观锁、状态机
5. 异步削峰：MQ 或 Stream，而不是把所有写操作都压到同步链路里

一句话总结：

> Redis 负责快、挡、削峰；数据库负责准、兜底、最终一致。

## 一个落地清单

上线前至少检查这些项：

- 是否识别了热点 Key 和大 Key
- 是否避免了高峰期慢命令
- 是否对关键路径做了 Pipeline 或 Lua 合并
- 是否明确了单实例、Sentinel、Cluster 的选型原因
- 是否给分布式锁设置了 TTL、唯一 value 和安全解锁
- 是否为 Cluster 下的多 Key 操作设计了 hash tag
- 是否把最终一致性放在数据库或状态机，而不是 Redis 锁
