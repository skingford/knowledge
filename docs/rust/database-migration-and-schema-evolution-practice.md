# Rust 数据库迁移、Schema 演进与兼容发布实践

很多 Rust 服务做到一定阶段，问题就不再是：

- 会不会连数据库
- 会不会写 migration

而是：

- 老版本服务还在线时，这次 schema 变更能不能兼容
- 给大表加列、改约束、补索引会不会把线上流量卡住
- 历史数据怎么回填
- 缓存、消息、对象存储和外部回调要不要一起改
- 一旦发布一半发现问题，能不能安全回滚

这些问题的共同点是：

**数据库迁移不是一段 SQL，而是一整条演进与发布链路。**

这页补的就是 Rust 服务里的数据库迁移、Schema 演进与兼容发布主线。

## 先分清：schema migration、数据回填、代码发布、回滚不是一回事

很多团队第一次做迁移时，最容易把下面几件事揉成一个动作：

- 改表结构
- 写迁移脚本
- 发布新代码
- 回填历史数据
- 删除旧字段

这几件事经常相关，但职责不同。

### schema migration

回答的是：

- 数据库结构要怎么变

例如：

- 加列
- 改索引
- 新建表
- 补约束

### 数据回填 backfill

回答的是：

- 历史数据怎么补齐到新结构

例如：

- 给旧用户补 `tenant_id`
- 给新列填默认值
- 把旧格式数据转成新格式

### 代码发布 release

回答的是：

- 应用代码什么时候开始依赖新结构
- 新旧版本混跑时能不能同时工作

### 回滚 rollback

回答的是：

- 如果新版本发布失败，旧版本还能不能继续跑
- 已经执行的迁移是否还能安全回退

一个务实原则通常是：

- migration 只是链路中的一个步骤
- 不要把“写了迁移文件”误当成“迁移方案已经设计好了”

## 第一原则：先扩张，再收缩，不要一步到位替换

数据库演进里最稳的主线通常是：

- expand
- migrate / backfill
- contract

也就是：

1. 先加新结构，不急着删旧结构
2. 让代码逐步兼容新旧两边
3. 回填和验证完成后，再删除旧结构

这比“一次性替换”稳得多。

例如你要把：

- `nickname`

改成：

- `display_name`

更稳的顺序通常不是：

1. 直接删 `nickname`
2. 立刻改代码全读 `display_name`

而是：

1. 先加 `display_name`，先允许为空
2. 新代码开始双写 `nickname` 和 `display_name`
3. 后台回填历史数据
4. 读路径优先读 `display_name`，缺失时回退 `nickname`
5. 验证完成后再去掉旧字段和兼容逻辑

## 一个高频误区：把 `sqlx::migrate!()` 当成迁移方案本身

`sqlx` 确实提供了：

- migration 文件管理
- `sqlx::migrate!()`
- `cargo sqlx migrate ...`

但这些主要解决的是：

- 迁移怎么执行

并不自动解决：

- 迁移顺序怎么和代码发布配合
- 新旧版本混跑怎么兼容
- 大表回填怎么做
- 索引和约束变更会不会锁表

所以更稳的理解通常是：

- `sqlx` 负责 migration 机制
- 你负责 schema 演进策略

- [SQLx 数据库访问实践](./sqlx-database-practice.md)

## 哪些变更最容易出事故

下面几类变更尤其值得谨慎：

### 1. 给大表直接加 `NOT NULL`

风险通常是：

- 历史数据还没补齐
- 老代码还不会写这个字段
- 数据库执行时间和锁影响不可控

### 2. 直接 rename 或 drop 列

风险通常是：

- 老版本服务还在读旧字段
- 异步 worker、脚本、管理后台没一起升级

### 3. 大规模同步回填

风险通常是：

- 把数据库 I/O、锁和复制延迟一起打满

### 4. 给热点表补索引时没考虑在线方式

风险通常是：

- 线上流量直接被阻塞

### 5. 只改数据库，不改缓存 / 消息 / 审计 / 对象 key

风险通常是：

- 数据库结构更新了
- 但边缘系统还在按旧模型工作

这些问题真正危险的地方是：

- 它们通常不是编译错误
- 而是发布后才暴露

## 一个够用的迁移心智模型：结构迁移和数据迁移分开

很多迁移失败，本质上是把两件事绑死了：

1. 结构变化
2. 历史数据处理

更稳的做法通常是：

### 先做结构迁移

例如：

- 加新列
- 加新表
- 加新索引

这一阶段尽量做到：

- 可快速执行
- 可重复执行
- 对在线流量影响可控

### 再做数据回填

例如：

- 分批扫描旧记录
- 按主键范围回填
- worker 异步推进

这样做的价值通常是：

- 发布和大规模数据处理可以解耦
- 回填失败不等于应用无法启动
- 你能更清楚观察迁移进度和回退点

- [Rust 后台任务与 Worker 实践](./background-jobs-and-worker-practice.md)

## 典型 rollout：新增一个最终必填字段

假设你要给 `users` 表增加：

- `tenant_id`

并且最终希望：

- 非空
- 带索引
- 参与唯一约束

更稳的主线通常是：

### 第一步：先加可空列

先让 schema 接受新字段，但不立刻要求历史数据满足。

### 第二步：新代码开始写入新列

这时要求：

- 新写入路径全部补 `tenant_id`
- 老数据先保留为空

### 第三步：回填历史数据

常见做法是：

- 按主键范围分批更新
- 控制批大小和速率
- 记录进度、失败和重试

### 第四步：加约束和索引

确认历史数据已补齐后，再逐步收紧：

- `NOT NULL`
- 复合唯一约束
- 查询索引

### 第五步：删除兼容逻辑

只有在：

- 所有实例都升级
- 历史数据都已补齐
- 下游系统都已切到新字段

之后，才去删临时兼容代码。

这类问题如果放在多租户语境里，继续看：

- [Rust 多租户与数据隔离实践](./multi-tenant-and-data-isolation-practice.md)

## 双写、双读不是默认好方案，但有时是必要过桥

很多 schema 演进都需要短期兼容：

- 新旧列同时存在
- 新旧表同时存在
- 新旧消息字段同时存在

这时常见做法就是：

- 双写
- 双读回退

它们的价值是：

- 让新旧版本可以平滑过渡

但也要注意：

- 双写会增加复杂度和不一致风险
- 双读回退会延长兼容窗口

所以更稳的原则通常是：

- 只在迁移窗口内使用
- 一开始就明确退出条件
- 迁移完成后尽快删除

不要把临时兼容逻辑变成永久结构。

## 缓存、消息和对象存储也必须跟着演进

数据库 schema 变了，不代表系统就只需要改数据库。

常见还要一起考虑的边界包括：

- Redis key 是否要补新维度
- 缓存 value 结构是否兼容新旧字段
- MQ / event payload 是否要加版本
- 审计记录和导出字段是否要同步变化
- 对象 key、元数据表和下载权限是否一起改

例如：

- 数据库里加了 `tenant_id`

但如果：

- Redis key 还是只有 `order:{id}`
- 对象 key 还是没有 tenant 前缀
- 事件 payload 没带 tenant

那迁移其实并没有完成。

## 索引和约束变更要当成流量治理问题来看

很多迁移不是业务逻辑错，而是数据库操作本身对线上影响太大。

例如：

- 大表建索引
- 重建唯一约束
- 修改列类型
- 回填时大批量更新

这些操作要提前考虑：

- 是否支持在线方式
- 是否需要分阶段执行
- 是否会长时间锁表
- 是否会放大复制延迟
- 是否需要错峰或限速

一个务实原则通常是：

- 结构可兼容
- 执行可观测
- 流量可控

不要只盯着 migration 文件能不能跑通。

## 回填任务更像后台批处理，不像一次普通 SQL

历史数据回填很容易被写成：

- 一条巨大的 `UPDATE`

这在小表上可能没问题，但到真实系统经常会出事。

更稳的做法通常是：

- 分批扫描
- 限速更新
- 可中断、可重试
- 记录进度

一个最小思路通常是：

1. 按主键范围分页读取待回填数据
2. 每批只处理有限行数
3. 每批单独提交
4. 把失败批次记下来

这比：

- 一口气全表更新

更容易控制。

## 回滚首先要看“旧代码还能不能读新 schema”

很多人一提回滚，就先想：

- migration 能不能写 down 脚本

这当然重要，但更务实的问题通常是：

- 新 schema 上，旧代码还能不能正常运行

例如：

- 你新增了列，旧代码一般还能跑
- 你删了列，旧代码很可能立刻炸
- 你把旧字段语义改了，旧代码可能会悄悄读错

所以更稳的回滚策略通常是：

- 先保证 schema 对旧代码尽量向后兼容
- 再考虑是否真的要物理回退数据库结构

一个务实原则通常是：

**应用回滚通常应该比数据库回滚更容易。**

## 混部兼容测试很重要

很多迁移在开发环境看起来都没问题，真正出事是在：

- 新旧版本实例同时在线

这时值得重点验证：

### 1. 新代码配老 schema

如果发布顺序里新代码可能先启动，就必须确认这一点。

### 2. 老代码配新 schema

滚动发布和回滚时，这通常更常见。

### 3. 回填进行中

也就是：

- 部分记录是新格式
- 部分记录还是旧格式

这时读路径和校验逻辑是否仍然稳定。

### 4. 缓存和消息的混合版本

例如：

- 老 worker 发旧事件
- 新 worker 发新事件

下游是否还能消费。

## 观测要覆盖迁移版本、回填进度和兼容错误

迁移类发布比普通发布更需要观测。

至少值得看这些信号：

- 当前应用版本
- 当前 migration 版本
- 回填任务进度
- 每批处理耗时和失败数
- 锁等待和死锁
- schema 不匹配错误
- 新旧字段命中比例

特别适合打到日志 / trace 的字段包括：

- `migration_version`
- `backfill_job`
- `batch_start_id`
- `batch_end_id`
- `rows_processed`
- `compat_mode`

## 一个很务实的发布顺序

很多数据库演进都可以先按这条线想：

1. 写 migration，先只做可兼容的 expand 变更
2. 发布兼容新旧结构的应用代码
3. 启动回填 worker 或批处理任务
4. 验证查询、写入、缓存、消息和观测都正常
5. 再补约束、删旧字段、删兼容逻辑

不要把：

- 改 schema
- 发新代码
- 回填历史数据
- 删除旧结构

压成一次发布动作。

## 常见误区

### 误区 1：迁移就是写一段 SQL

真实问题往往出在兼容发布、回填、缓存和回滚，不在 SQL 语法本身。

### 误区 2：先删旧列，再发布新代码

只要还有旧实例、旧 worker 或旧脚本在线，这通常就是事故入口。

### 误区 3：把结构迁移和历史数据回填绑成一步

这样会把发布窗口、数据库压力和失败恢复难度一起放大。

### 误区 4：双写双读加上以后就不着急删

临时兼容逻辑一旦常驻，后面只会越来越难维护。

### 误区 5：只看数据库，不看缓存、消息和对象存储

很多演进失败，本质上是边缘系统模型没有一起升级。

### 误区 6：down migration 能写就等于一定能安全回滚

很多线上数据一旦向前演进，真正安全的通常是应用回滚，不是强行把数据库物理倒回去。

## 自检

你至少应该能回答：

1. 为什么数据库迁移不能只理解成“写 migration 文件”？
2. 为什么很多 schema 演进都应该先 expand，再回填，最后 contract？
3. 为什么结构迁移和历史数据回填最好拆成两段？
4. 为什么缓存、消息和对象存储经常也要跟着 schema 变更一起升级？
5. 为什么真正务实的回滚策略通常优先保证“旧代码仍能读新 schema”？

这些问题理顺后，Rust 服务里的数据库演进才会从“能改表”进入“能安全发布、能兼容混部、能长期演进”的状态。
