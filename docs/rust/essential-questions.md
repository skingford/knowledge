# Rust 必备问题清单

这页不追求把 Rust 所有知识点列完，而是把最能区分理解深度的问题按主线排好。

建议的回答顺序是：

- 先说它解决什么问题
- 再说 Rust 怎么做
- 最后补一句代价、边界或常见误区

## 1. 入门必答：先把资源模型说对

**1. Rust 为什么没有 GC 也能强调内存安全？**

- 因为它把资源管理尽量前移到编译期，用所有权、借用和生命周期规则约束内存访问
- 核心不是“完全不需要运行时”，而是大部分安全约束在编译时就能验证

**2. move、`Copy`、`Clone` 的区别是什么？**

- move 是所有权转移
- `Copy` 是低成本隐式复制，复制后原值仍可用
- `Clone` 是显式复制，可能成本较高

**3. 为什么同一时刻不能同时有可变借用和不可变借用？**

- 因为 Rust 要阻止读写重叠带来的不一致、悬垂引用和数据竞争
- 这是它安全模型的核心约束之一

**4. 生命周期标注到底是在做什么？**

- 它通常不是“延长对象寿命”，而是在描述多个引用之间的有效期关系
- 本质是帮助编译器推断返回引用不能超过哪个输入引用的寿命

**5. 为什么很多 API 都是“参数借用，返回值拥有”？**

- 参数借用能降低调用成本和耦合
- 返回值拥有所有权更容易跨作用域使用，减少生命周期复杂度

## 2. 类型系统：回答不清这里，Rust 代码很难写稳

**6. `struct` 和 `enum` 分别更适合表达什么？**

- `struct` 适合稳定字段集合
- `enum` 适合互斥状态和分支建模

**7. 为什么 Rust 很强调 `enum + match`？**

- 因为它鼓励把状态显式建模，并通过穷尽匹配减少漏分支
- 这比魔法值、字符串状态码和布尔组合更稳

**8. trait 在 Rust 里除了“像接口”，还承担什么职责？**

- 行为抽象
- 泛型约束
- 默认实现
- 动态分发边界

**9. `impl Trait` 和 `dyn Trait` 有什么区别？**

- `impl Trait` 更偏静态分发
- `dyn Trait` 更偏运行时多态
- 选择核心在于性能、可读性和是否需要异构对象集合

**10. 为什么泛型经常和 trait bound 一起出现？**

- 因为泛型只表示“类型可变”，trait bound 才表示“这个类型必须具备什么能力”

## 3. 日常编码：集合、字符串和迭代器是高频坑点

**11. `String` 和 `&str` 的区别是什么？**

- `String` 拥有堆上字符串数据
- `&str` 是对字符串切片的借用视图

**12. 为什么 Rust 字符串不能按下标直接取第 N 个字符？**

- 因为字符串是 UTF-8，字节偏移和字符边界不等价

**13. `iter()`、`iter_mut()`、`into_iter()` 的区别是什么？**

- `iter()` 产生不可变借用迭代
- `iter_mut()` 产生可变借用迭代
- `into_iter()` 消耗集合并转移所有权

**14. 为什么边拿 `Vec` 元素引用边 `push` 容易出错？**

- 因为 `push` 可能触发底层重新分配，原引用可能失效

**15. `HashMap::entry` 解决了什么问题？**

- 把“存在则修改，不存在则初始化”的常见模式收敛成统一写法

## 4. 工程化：从“会写代码”到“会做项目”

**16. `Option<T>` 和 `Result<T, E>` 的职责边界是什么？**

- `Option` 表达有无
- `Result` 表达成功与失败及失败原因

**17. 什么时候应该 `panic!`，什么时候应该返回错误？**

- `panic!` 更适合不可恢复错误和内部不变量被破坏的场景
- 业务流程和预期内失败一般应返回 `Result`

**18. module、crate、workspace 分别解决什么问题？**

- module 解决单个 crate 内部组织
- crate 是编译和依赖单元
- workspace 管多个 crate 的协作

**19. 为什么 Rust 工程里通常强调控制 `pub` 暴露面？**

- 因为默认私有有助于收敛 API 边界，降低后续重构成本

**20. 为什么 `cargo fmt` 和 `cargo clippy` 应该进入日常循环？**

- 它们能把格式和大量低价值问题前移，减少协作摩擦和 review 噪音

## 5. 智能指针与并发：很多“难点”都在这里汇合

**21. `Box`、`Rc`、`Arc` 分别在解决什么问题？**

- `Box`：堆分配和递归类型等单拥有场景
- `Rc`：单线程共享所有权
- `Arc`：多线程共享所有权

**22. `RefCell` 为什么只适合单线程？**

- 因为它靠运行时借用检查实现内部可变性，不提供线程同步语义

**23. `Send` 和 `Sync` 分别表示什么？**

- `Send` 表示所有权可安全在线程间转移
- `Sync` 表示共享引用可安全在线程间共享

**24. 为什么 `Arc<Mutex<T>>` 不是默认答案？**

- 它虽常见，但意味着共享可变状态和锁竞争成本
- 很多时候消息传递更清晰

**25. async/await 为什么不等于自动并行？**

- async 更偏等待期间的并发组织，不自动保证多核并行执行

## 6. 进阶专题：决定你是“知道名词”还是“能做判断”

**26. `serde` 最值得解决的核心问题是什么？**

- Rust 类型和外部协议/配置格式之间的稳定映射

**27. 为什么 DTO 和领域模型不一定应该合并？**

- 协议结构、业务规则和存储模型的演进节奏往往不同，硬合在一起容易相互污染

**28. 为什么宏不该成为默认抽象手段？**

- 宏虽然强，但会提高可读性、调试和维护门槛
- 能用函数、trait、泛型解决时，通常不应先上宏

**29. `unsafe` 到底把什么责任交给了开发者？**

- 把编译器无法自动验证的安全前提交由开发者手动保证

**30. FFI 最容易出问题的边界是什么？**

- 内存所有权
- 数据布局和 ABI
- 空指针、长度和生命周期约束

## 面试前至少要能说清的结论

- Rust 的核心不是语法，而是资源和共享边界如何在编译期表达
- 类型系统不是为了复杂而复杂，而是为了把状态和能力显式化
- `Result`、`Option`、trait、`enum`、借用规则是日常编码主线
- 并发和 async 仍然受所有权模型约束，不是另一套世界
- `serde`、宏、unsafe、FFI 都是进阶能力，但必须建立在主线稳固之后
