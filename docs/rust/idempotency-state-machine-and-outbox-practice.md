# Rust 幂等、状态机与 Outbox 实践

Rust 服务一旦进入真实业务，很快就会碰到一类比“代码能不能跑”更难的问题：

- 超时后到底要不要重试
- 同一个请求打进来两次会不会重复扣款
- worker 失败重试会不会把副作用放大
- 写库成功了，但消息没发出去怎么办
- 回调乱序到了，状态会不会被回滚

这些问题的共同点是：

**它们都在处理“有副作用的操作怎么只正确推进一次”。**

这页补的就是这条主线。

## 先分清：幂等不是“不报错”，而是“重复执行结果可控”

幂等最容易被误解成：

- 调多次也没事

但更稳的理解应该是：

- 同一业务意图被重复提交时，系统不会把副作用无限放大

这意味着幂等关注的不是“有没有第二次请求”，而是：

- 第二次请求落下来时，系统怎么识别它和第一次是同一意图
- 怎么保证状态不会被重复推进
- 怎么把“重复请求”变成“可预测结果”

## 幂等、去重、重试、状态机不是一回事

这几个词经常一起出现，但职责不同。

### 幂等

回答的是：

- 同一业务操作重复执行时，结果是否可控

### 去重

回答的是：

- 系统怎么识别两次输入属于同一业务操作

### 重试

回答的是：

- 失败后是否重新执行

### 状态机

回答的是：

- 当前状态是否允许推进到下一个状态

一个很务实的对应关系是：

- 去重负责识别“是不是同一件事”
- 状态机负责判断“这一步能不能再做”
- 幂等负责保证“就算再来一次也不会把副作用放大”
- 重试只是触发重复执行的来源之一

## 什么时候必须明确设计幂等

下面这些场景，不做幂等通常迟早出事：

- 创建订单
- 发起扣款
- 支付回调
- 发货 / 发券
- worker 重试执行任务
- 超时后自动重试写请求
- 消费 MQ / 任务表 / Outbox 事件

如果操作没有副作用，幂等压力通常没那么大；但一旦操作会改状态、扣资源、发消息、触发外部系统，就必须认真想。

## 幂等入口通常有 4 层

### 1. 请求级幂等键

适合：

- API 写请求
- 外部回调
- 明确存在“客户端可能重放”的入口

常见做法：

- 客户端传 `Idempotency-Key`
- 服务端把它和业务主体一起落库
- 重复请求直接返回第一次结果或当前状态

如果你想把第三方 callback 的原始 body 验签、重放防护、2xx / 5xx 返回语义和异步解耦单独理顺，继续看：

- [Rust Webhook、回调与签名校验实践](./webhook-and-callback-practice.md)

### 2. 业务唯一键

适合：

- 一个业务天然有唯一业务编号
- 例如 `order_no`、`payment_no`、`biz_id`

这层通常比“随机请求 ID”更稳，因为它更贴近业务真实意图。

### 3. 数据库约束

适合：

- 最终防线
- 防止并发写穿透应用层判断

常见形式：

- unique constraint
- `INSERT ... ON CONFLICT ...`
- 条件更新

### 4. 状态机推进约束

适合：

- 同一资源会经历多阶段推进
- 例如订单、支付、退款、任务状态

这层的核心不是“有没有重复请求”，而是：

- 当前状态还能不能再次推进

## 第一原则：先找“业务意图键”，不要先找技术随机 ID

很多系统第一反应是：

- 给每个请求生成一个 UUID

这不一定错，但经常不够。

更稳的顺序通常是：

1. 先看业务上是否已经有天然唯一键
2. 没有时再补请求级幂等键
3. 最后用数据库约束把最终一致性收住

因为真正该被识别成“同一件事”的，往往是：

- 同一个订单支付
- 同一个用户领券
- 同一个任务执行

而不是“HTTP 请求包的字节是否完全一样”。

## 请求级幂等适合什么，不适合什么

### 适合

- `POST /payments`
- `POST /orders`
- 第三方回调入口
- 外部 API 写请求网关

### 不适合单独依赖

- 没有持久化落地
- 不带业务主体约束
- 多实例部署但只做内存去重

如果幂等键只存在进程内内存里，服务重启或扩容后基本等于没做。

## 一个很务实的数据库幂等思路

最常见做法通常是：

1. 先把业务唯一键或幂等键落到数据库
2. 用 unique constraint 保证最终只有一条成功记录
3. 对冲突结果做显式业务映射

例如创建支付记录时，重点通常不是“先查有没有”，而是：

- **让数据库帮你兜住并发下的唯一性**

因为“先查再写”在并发下很容易被穿透。

## 状态机比“布尔字段 + if”更稳

很多副作用重复执行问题，本质上是状态流转没有被显式建模。

坏味道通常像这样：

- `is_paid`
- `is_shipped`
- `is_refunded`

字段一多，组合关系就开始失控。

更稳的做法通常是：

- 用 `enum` 表达有限状态
- 明确合法迁移
- 更新时带条件

例如：

- `Pending -> Paid`
- `Paid -> Shipped`
- `Paid -> Refunded`

但不能：

- `Refunded -> Paid`
- `Shipped -> Pending`

## 条件更新比“先查状态，再决定要不要更新”更稳

很多并发 Bug 都出在：

1. 先查当前状态
2. 代码里判断可以推进
3. 再执行更新

在并发下，这三步之间完全可能被别的请求插队。

更稳的思路是：

- 把“当前状态必须是 X 才能更新成 Y”直接写进 SQL 条件里

这样受影响行数本身就是结果信号。

可以简单理解成：

- `rows_affected == 1`：本次推进成功
- `rows_affected == 0`：状态已经被别人推进过，或当前状态不允许推进

## `sqlx` 事务里不要夹外部 RPC

这是幂等和一致性设计里很关键的一点。

如果事务里夹了：

- 调支付网关
- 发 MQ
- 调用户服务

你会立刻遇到几个问题：

- 锁持有时间变长
- 超时语义变复杂
- 回滚无法撤销外部副作用

更稳的原则通常是：

- 事务里只做本地状态变更
- 外部副作用通过 Outbox 或事务外协调

## Outbox 解决的不是“发消息方便”，而是“写库和投递之间不能留黑洞”

最典型的问题是：

1. 业务状态写库成功
2. 准备发消息
3. 进程崩了，消息没出去

这时系统就会进入一种很危险的状态：

- 本地状态已经变了
- 下游却永远不知道

Outbox 的核心思路是：

1. 在同一个数据库事务里，同时写业务表和 outbox 表
2. 后台 worker 再去扫描 outbox 并投递
3. 投递成功后把 outbox 标成已发送

这样真正被事务保证的是：

- “业务变化”和“待发送事件”要么都成功，要么都失败

## 一个最小 Outbox 心智模型

可以先把 outbox 表想成：

- 待发送事件队列表

常见字段包括：

- `event_id`
- `aggregate_type`
- `aggregate_id`
- `event_type`
- `payload`
- `status`
- `attempt_count`
- `next_retry_at`
- `created_at`

这张表的重点不是长什么样，而是职责：

- 业务事务只负责写进去
- worker 负责取出来投递
- 重试、退避、死信策略在投递侧处理

## 消费方也必须幂等

Outbox 只能保证：

- 至少一次把事件送出去

它通常**不能保证恰好一次**。

所以消费方仍然要处理：

- 重复事件
- 重放事件
- 乱序事件

常见做法包括：

- 记录已处理事件 ID
- 对业务主键做唯一约束
- 用状态机限制重复推进

如果下游消费者不幂等，Outbox 只能把问题从“消息没发出去”换成“消息重复消费导致副作用重复”。

## 重试必须先分“临时错误”和“永久错误”

这一点在幂等链路里特别关键。

### 更像临时错误

- 网络抖动
- 连接超时
- 502 / 503 / 504
- 短暂锁冲突

### 更像永久错误

- 参数非法
- 状态不合法
- 业务约束不成立
- 数据缺失且不会自动恢复

这两类错误不该共用一套重试策略。

## 超时和重试为什么必须和幂等绑在一起看

因为 timeout 通常只意味着：

- 本地等不下去了

它不意味着：

- 远端一定没成功

所以对写请求而言，timeout 后继续重试之前，必须回答：

1. 这次操作有没有业务唯一键或幂等键？
2. 远端会不会把两次请求都当真？
3. 失败后能不能查状态，而不是盲重试？

如果这些问题说不清，就不要把超时后的自动重试当默认动作。

## Worker 幂等通常要比 HTTP 幂等更严格

因为 worker 场景里，重复执行往往更常见：

- 进程崩溃后重新拉任务
- 投递超时后再次消费
- worker 重平衡
- 重试策略重新执行同一任务

所以更稳的 worker 处理习惯通常是：

- 输入里带业务主键和幂等键
- handler 本身先查状态再决定是否推进
- 推进动作用条件更新或唯一约束兜底

## 返回“重复成功”通常比返回错误更实用

很多幂等接口一个高频设计问题是：

- 第二次请求来了，到底该报错，还是返回第一次结果

更务实的做法通常是：

- 如果业务意图相同且第一次已经成功，就直接返回已存在结果或当前状态

原因很简单：

- 调用方最关心的是“这件事现在到底成没成”
- 而不是“你这次是不是第一次调用”

当然前提是：

- 你能确认两次请求确实代表同一业务意图

## 一组很实用的落地策略

### 创建类操作

优先考虑：

- 业务唯一键
- unique constraint
- 重复创建直接返回已有结果

### 状态推进类操作

优先考虑：

- 显式状态机
- 条件更新
- 非法状态推进直接拒绝

### 写库后发消息

优先考虑：

- 本地事务 + Outbox
- worker 异步投递
- 消费方幂等

### 写请求超时后

优先考虑：

- 查状态
- 依赖幂等键
- 如果新旧处理链路受运行时开关控制，确保重试仍能复用同一策略版本

## 测试要覆盖的不是“成功一次”，而是“重复来三次还不出事”

至少值得覆盖：

### 单元测试

- 状态机合法迁移
- 重复请求的结果映射
- 幂等键冲突后的行为

### 集成测试

- 并发重复提交是否只成功一次
- unique constraint 冲突是否被正确映射
- Outbox 写入是否和业务写库同事务

### 回归测试

- timeout 后重复提交是否产生重复副作用
- worker 重试三次后状态是否仍然正确
- 消费方收到重复事件时是否仍然只推进一次

## 观测必须能回答“重复执行到底发生了没有”

最少建议记录：

- 幂等键命中次数
- 状态推进冲突次数
- Outbox 待发送数量
- Outbox 重试次数
- 消费去重命中次数

很实用的字段包括：

- `biz_id`
- `idempotency_hit`
- `state_from`
- `state_to`
- `outbox_event_type`
- `attempt`

注意：

- `biz_id` 适合日志和 trace，不一定适合直接做 metric label

## 常见误区

### 误区 1：先查有没有，再决定要不要写

并发下一样可能重复写入。

### 误区 2：把幂等完全等同于 request UUID

没有业务语义约束时，经常只能防住“同一个包重放”，防不住“同一业务被重新发起”。

### 误区 3：事务里直接调外部系统

这样会把事务、锁和外部失败语义全部缠在一起。

### 误区 4：Outbox 做了，就以为消费方不用幂等

这通常会在重复投递时把问题放大。

### 误区 5：超时后自动重试一切写请求

这非常容易把一次不确定结果放大成重复副作用。

## 一组很实用的落地原则

1. 先找业务意图键，再决定是否额外引入请求级幂等键。
2. 应用层判断不够，最终还要靠数据库唯一约束或条件更新兜底。
3. 状态推进优先建模成显式状态机，不要靠零散布尔字段拼流程。
4. 事务里只做本地状态变更，跨边界副作用优先走 Outbox。
5. Outbox 解决的是“写库和待发送事件一起提交”，不解决消费方天然幂等。
6. timeout 和重试永远要跟幂等一起设计。
7. 重复请求更常见的正确结果不是报错，而是返回当前已确认状态。
8. 进入 MQ / event-driven 场景后，消费方仍然要按至少一次投递来设计去重、重试和死信。

你至少应该能回答：

1. 幂等、去重、重试、状态机分别在解决什么？
2. 为什么“先查再写”在并发下不够稳？
3. 为什么条件更新通常比“先查状态再更新”更可靠？
4. Outbox 真正保证了什么，又没有保证什么？
5. 为什么消费方在 Outbox 场景下仍然必须幂等？
6. timeout 后为什么不能对有副作用请求盲重试？

把这些问题讲清楚，你才算真正把 Rust 服务里“有副作用的操作怎么安全推进”这条主线理顺。
