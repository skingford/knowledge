# 宏与元编程基础

Rust 的宏能力很强，但也最容易让人刚接触时走偏。

正确的切入方式不是“宏很高级”，而是先分清：

- 哪些问题适合用普通函数、trait、泛型解决
- 哪些问题真的需要在编译期生成代码

## 先分两类宏

### 1. 声明宏

常见形式：

```rust
macro_rules! say_hello {
    () => {
        println!("hello");
    };
}
```

它更像“基于模式匹配的语法展开”。

### 2. 过程宏

常见形式：

- `#[derive(...)]`
- 属性宏
- 类函数宏

它们能直接处理 Rust 语法树，能力更强，也更复杂。

## 日常最常接触的是 `derive`

很多常用能力其实都是 derive 宏在帮你生成实现：

- `Debug`
- `Clone`
- `Serialize`
- `Deserialize`
- `PartialEq`

这也是为什么初学 Rust 时，宏虽然没系统学，但其实早就在用了。

## 宏到底解决什么问题

它最适合处理下面几类场景：

- 样板代码重复且结构稳定
- 需要基于类型定义自动生成实现
- 普通函数和泛型无法表达需要的语法层能力

不适合的场景：

- 只是想少写几行普通逻辑
- 业务规则本身经常变化
- 团队还不具备维护宏的能力

## 为什么 Rust 需要宏

因为有些能力普通函数做不到，例如：

- `println!` 接受可变参数并做格式检查
- `vec![]` 提供类似字面量的初始化体验
- `#[derive(Debug)]` 根据结构体字段自动生成实现

它们都发生在“编译期代码展开”这一层。

## `macro_rules!` 适合什么

声明宏更适合：

- 小型语法糖
- 重复模板展开
- 简单可预测的代码生成

它的优点是：

- 不需要单独操作完整语法树
- 成本相对低

它的限制是：

- 复杂度一上来就很难读
- 调试体验一般
- 维护成本会迅速上升

## 过程宏为什么更复杂

过程宏通常要做这些事：

1. 接收 token stream
2. 解析语法结构
3. 生成新的 token stream

所以它更接近“编译器插件式能力”。

典型场景：

- 自定义 derive
- 属性驱动代码生成
- 框架层 DSL

## 宏不是泛型和 trait 的替代品

这是最重要的一条边界。

如果问题能靠：

- 普通函数
- trait 默认实现
- 泛型约束
- 明确的数据结构

解决，就先别上宏。

因为宏的代价包括：

- 可读性下降
- 错误信息更难理解
- IDE 和调试体验更复杂
- 团队维护门槛更高

## 框架里为什么经常看到宏

因为框架通常需要：

- 降低用户样板代码量
- 把结构声明转成样板实现
- 提供声明式 API 体验

例如：

- 序列化 derive
- Web 路由属性宏
- ORM 模型映射宏

但框架能用宏，不代表业务代码也应该到处造宏。

## 一条务实判断线

可以按这个顺序判断：

1. 普通函数能不能解决？
2. trait + 泛型能不能解决？
3. 构建脚本或代码生成能不能解决？
4. 只有都不合适时，再考虑宏。

## 宏带来的维护风险

### 1. 展开后代码不直观

调用点很短，但真正生成了什么实现，不一定一眼能看出。

### 2. 错误可能离用户代码很远

问题出在宏展开后的结果，报错位置却看起来像在调用点。

### 3. 过度 DSL 化会削弱普通 Rust 可读性

如果一段业务代码只有“框架自定义注解和宏”，而没有明确的 Rust 结构，长期维护通常更痛苦。

## 工程建议

### 优先消费成熟宏，而不是自己写宏

先学会正确使用成熟生态里的宏，再决定是否真的需要自定义。

### 宏要解决“重复结构问题”，不要承载易变业务逻辑

业务逻辑越经常变化，越不适合塞进宏里。

### 团队里如果只有少数人看得懂宏，要慎用

否则短期省样板，长期变维护债务。
