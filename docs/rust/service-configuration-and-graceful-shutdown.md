# 服务配置与优雅关闭

> 本页结合 Tokio 官方 `ctrl_c`、任务取消与 runtime shutdown 主线整理，重点讲 Rust 服务从启动到退出的完整生命周期。

很多服务代码“能跑”，但不算“能稳定上线”。最常见的缺口就出在这条链路上：

- 配置怎么读
- 启动前怎么校验
- 收到退出信号后怎么停
- 后台任务怎么协同结束
- 哪些资源要先停，哪些请求要先摘流量

这页讲的是 Rust 服务生命周期，而不是某个框架的局部 API。

## 先把服务生命周期想成 4 段

1. 读取和构造配置
2. 初始化依赖并启动服务
3. 运行期间处理请求和后台任务
4. 收到关闭信号后做优雅退出

如果这四段没有设计清楚，服务会很容易出现：

- 启动半天才发现配置错
- Ctrl+C 直接把任务打断
- 连接、任务、消费者没有收尾
- 发布时短暂丢请求

## 配置要尽早读、尽早校验

如果你想先把“默认值、文件、环境变量、命令行、校验和来源合并”这条配置主线理顺，直接看：

- [Rust 配置管理实践](./configuration-management-practice.md)

配置通常包括：

- 服务监听地址
- 数据库连接串
- Redis/MQ/外部 API 配置
- 超时、重试和池大小
- tracing / 日志级别

如果你想把 Redis 连接复用、TTL 和缓存失效策略单独理顺，继续看：

- [Rust 缓存与 Redis 实践](./cache-and-redis-practice.md)

比较稳的原则是：

- 启动最早阶段就读取
- 读完立刻校验
- 不合法就直接失败退出

不要让服务先跑起来，再在第一次打到某条路径时才发现配置不对。

## 配置结构体最好显式建模

即使来源是环境变量、文件或命令行，进入程序内部后也最好先收敛成明确结构体。

这样更容易：

- 校验默认值
- 传递依赖
- 做测试
- 区分应用级和模块级配置

## 启动阶段要分层

比较稳的启动顺序通常是：

1. 初始化配置
2. 初始化 tracing
3. 初始化数据库池、缓存客户端、HTTP 客户端等依赖
4. 拼应用状态
5. 构造 Router / server
6. 注册关闭信号
7. 开始监听

这样出错时更容易定位，也更容易保证日志从一开始就完整。

## Tokio 官方主线里的关闭入口

Tokio 官方文档里最常见的服务关闭触发点之一是：

```rust
tokio::signal::ctrl_c().await?;
```

它的意义是：

- 等待进程收到 Ctrl+C
- 把操作系统信号变成 async 等待点

这通常是优雅关闭链路的起点。

## 关闭不是“收到信号就直接退出”

更稳的思路通常是：

1. 收到关闭信号。
2. 停止接受新流量或新任务。
3. 通知后台任务开始收尾。
4. 等待一段合理时间。
5. 超时后再强制终止剩余任务。

这和“直接 kill”最大的差别在于：你给了系统一个把状态收干净的机会。

## `select!` 是协调生命周期的关键工具

在 async 服务里，常见会有多个事件竞争：

- 服务器运行结束
- Ctrl+C 到来
- 某个关键后台任务异常退出

这类场景很适合 `select!`。

核心价值不是语法，而是：

- 把“谁先发生就进入关闭流程”写成显式控制流

## 后台任务不能只管 `spawn` 不管收尾

很多服务会有这些后台任务：

- 定时刷新配置
- 消费队列
- 心跳或健康上报
- 批量刷盘

如果只会 `tokio::spawn`，但没有统一关闭机制，退出时就很容易出现：

- 任务被粗暴取消
- 状态丢失
- 日志没刷完
- 资源没释放

如果你的服务还有 WebSocket / SSE 这类长连接推送，也要把连接任务和广播 channel 纳进同一条关闭主线，继续看：

- [Rust WebSocket 与 SSE 实践](./websocket-and-sse-practice.md)

如果你想把进程内 worker、队列消费者、定时任务、`CancellationToken` 和后台重试边界系统收成一页，继续看：

- [Rust 后台任务与 Worker 实践](./background-jobs-and-worker-practice.md)
- [Rust 定时任务、调度与 Leader Election 实践](./scheduler-and-leader-election-practice.md)

## Tokio 的取消边界要会说清

Tokio 官方文档强调：

- 任务被取消，通常会在下一个 `.await` 点停下来
- `JoinHandle::abort()` 会给任务发取消信号
- runtime 关闭也会取消任务

这意味着：

- 取消不是“立刻中断任意指令”
- 任务能不能及时停，和它是否会 yield 有关
- 长时间阻塞的逻辑不会优雅地响应 async 取消

## 优雅关闭时最该停什么

通常按这个顺序思考比较稳：

### 1. 先停新流量入口

例如：

- HTTP listener 不再接新连接
- 消费者不再拉新消息

### 2. 再让在途任务收尾

例如：

- 当前请求处理完
- 当前消息消费到安全点
- 当前批处理刷完结果

### 3. 最后释放底层资源

例如：

- 数据库池
- 外部客户端
- metrics/tracing exporter

如果你想把 metrics exporter、OTLP provider、flush / shutdown 和标签设计单独理顺，继续看：

- [Rust Metrics 与 OpenTelemetry 实践](./metrics-and-opentelemetry-practice.md)

## timeout 很重要

优雅关闭不是无限等。

比较稳的策略通常是：

- 给收尾设一个最大等待时间
- 超时后进入强制终止

否则一个卡住的任务就可能让整个发布或退出流程无限挂住。

## 运行时关闭和服务层关闭不是一回事

Tokio 官方文档里有 runtime 的 shutdown API，例如：

- `shutdown_timeout`
- `shutdown_background`

但在服务应用里，更关键的通常不是 runtime API 本身，而是你在进入 runtime 关闭之前，是否已经把：

- listener
- in-flight requests
- background tasks
- shared resources

协调好。

## 配置和关闭为什么要放在一页讲

因为它们其实是同一条服务生命周期问题。

如果启动和退出都没有统一边界，服务就很难做到：

- 行为可预测
- 发布可控
- 排障清晰

## 工程建议

### 配置一开始就收敛成结构体

不要在业务里到处直接读环境变量。

### 启动前就做硬校验

端口、连接串、超时、必要开关不合法时直接失败。

这里的“必要开关”更接近启动期静态配置，不等于按租户、按比例动态求值的运行时 feature flag。

如果你想把静态配置和运行时灰度开关的边界单独理顺，继续看：

- [Rust Feature Flag、运行时开关与灰度治理实践](./feature-flag-and-runtime-governance-practice.md)

### 关闭流程要有统一入口

不要让每个模块各自决定什么时候退出。

### 后台任务要有协同关闭机制

不是只存个 `JoinHandle` 就算完事。

### 优雅关闭必须配超时

否则它就可能变成永远关不掉。
