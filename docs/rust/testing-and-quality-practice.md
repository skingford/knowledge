# 测试与质量实践

Rust 的强项之一，是它把很多质量动作做成了默认工作流的一部分。

但这不代表“编译过了就等于没问题”。真实项目里，仍然需要把测试、lint、格式化和回归策略配合起来。

## 质量闭环先分 4 层

### 1. 编译期约束

例如：

- 所有权和借用规则
- 类型系统约束
- trait bound

这能挡掉一大批错误，但挡不掉业务逻辑错误。

### 2. 测试

包括：

- 单元测试
- 集成测试
- 文档测试

### 3. 代码质量工具

包括：

- `cargo fmt`
- `cargo clippy`

### 4. 回归与演进策略

包括：

- 增量补测试
- 修 bug 先补复现
- 重要边界做表驱动或场景测试

## 单元测试更适合什么

单元测试适合：

- 小范围纯逻辑
- 边界条件
- 错误路径
- 数据转换

```rust
fn is_even(n: i32) -> bool {
    n % 2 == 0
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn even_number_returns_true() {
        assert!(is_even(4));
    }
}
```

## 集成测试更适合什么

集成测试更像“从 crate 外部看你这个库或应用”。

适合：

- 对外 API
- 模块协作
- 配置装配
- 边界层行为

如果你发现某些测试必须知道大量内部细节，可能更适合放回单元测试。

## 文档测试的价值

Rust 文档测试有一个很实用的价值：

- 文档示例不是摆设，而是可以被执行验证

这会迫使：

- 示例代码保持可运行
- 文档和实际 API 不至于长期漂移

## 错误路径测试很重要

Rust 项目里，很多逻辑价值恰恰在错误处理边界。

所以不要只测 happy path，还要测：

- 非法输入
- 缺配置
- 超时
- 空结果
- 重复调用
- 状态冲突

## `cargo clippy` 不是“可有可无的建议”

`clippy` 很多提示都非常实用，例如：

- 明显不 idiomatic 的写法
- 冗余 clone
- 潜在性能问题
- 可读性较差的表达

它不等于“绝对真理”，但在团队里通常很值得进入日常循环和 CI。

## `cargo fmt` 的价值不是美观，而是降低协作摩擦

统一格式能减少：

- 无意义 diff
- code review 风格争论
- 大量琐碎手工修正

它应该成为默认动作，而不是“上线前再整理一下”。

## 测试替身和边界隔离

Rust 项目做测试时，一个常见问题是：

- 怎么隔离数据库、HTTP、文件系统、外部服务

常见思路包括：

- trait 抽象依赖边界
- 用小型 fake / stub 替代真实实现
- 把纯逻辑和 I/O 边界拆开

这里的关键不是“为了 mock 而抽象”，而是让核心逻辑不必依赖真实外部环境。

如果你想把 fake、stub、mock、本地 HTTP server、Tokio async 测试和边界隔离系统理顺，继续看：

- [Rust 测试替身与依赖隔离实践](./test-doubles-and-dependency-isolation-practice.md)

## 修 bug 的更稳流程

一个很实用的质量策略是：

1. 先写一个最小复现测试。
2. 确认当前测试失败。
3. 再修实现。
4. 确认测试转绿。

这样 bug 修复不会只停留在“这次好像好了”，而是进入长期回归保护。

## 表驱动和场景测试很适合边界较多的逻辑

例如：

- 参数校验
- 状态流转
- 协议解析

这类逻辑常常适合把输入、预期输出、错误场景列成表。

如果你正在补 DTO 校验、领域不变量和错误映射，也可以继续看：

## async 和并发代码测试要特别关注什么

重点通常不是“跑起来了”，而是：

- 超时
- 取消
- 顺序问题
- 锁竞争
- 共享状态一致性

如果你正在测试 timeout、限流、并发限制和 overload 拒绝边界，也可以继续看：

- [Rust 限流、超时与负载保护实践](./rate-limit-timeout-and-overload-protection-practice.md)

很多并发 bug 不是类型错误，而是时序问题，所以更需要针对边界设计测试。

## 工程建议

### 先测纯逻辑，再测边界层

不要一开始就只写“大而全”的端到端测试。

### 修 bug 必补回归测试

否则同类问题很容易再次出现。

### fmt 和 clippy 放进日常循环

不要等到最后统一清理。

### 重要公共 API 尽量有文档示例

这样文档和测试能形成闭环。
