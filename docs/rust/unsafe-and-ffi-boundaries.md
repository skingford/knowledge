# Unsafe 与 FFI 边界

Rust 的安全承诺建立在一组严格规则上，但它也明确保留了一个逃生口：`unsafe`。

这并不意味着 Rust 安全可以被随意关闭，而是表示：

- 编译器不再替你验证某些前提
- 你要自己承担这些前提成立的责任

## `unsafe` 到底在说什么

很多人会把 `unsafe` 理解成“危险代码”。这不够准确。

更准确的理解是：

这里有一段代码需要依赖某些编译器无法自动证明的安全前提，因此这部分责任转交给开发者。

## 常见 `unsafe` 能做什么

- 解引用原始指针
- 调用 `unsafe fn`
- 访问或修改可变静态变量
- 实现 `unsafe trait`
- 访问 `union` 字段

这些操作本身不是自动错误，但都要求你保证额外不变量。

## 最核心原则：把 `unsafe` 缩到最小

工程上更稳的做法是：

1. 把不安全操作封装在很小的一层里。
2. 对外暴露尽可能安全的 API。
3. 在封装层明确写出安全前提。

目标不是“项目里完全没有 unsafe”，而是“unsafe 的作用域清晰、可审计、可验证”。

## 一个常见误区

很多人第一次卡 borrow checker，会想：

- 能不能直接上指针
- 能不能 `unsafe` 绕过去

大多数时候都不应该。

因为这通常说明：

- 数据流还没设计好
- 生命周期边界不清楚
- 应该先改结构，而不是先改安全级别

## 什么时候 `unsafe` 才合理

更常见的合理场景包括：

- FFI 边界
- 高性能底层数据结构
- 封装编译器无法推断但人工可证明的安全不变量
- 与操作系统或底层运行时直接交互

也就是说，它更像“底层基础设施工具”，而不是日常业务代码默认选项。

## FFI 在解决什么

FFI 是 Foreign Function Interface，用来让 Rust 和其他语言或原生库交互。

最常见的是和 C ABI 交互：

- 调已有 C 库
- 被其他语言调用
- 接系统级接口

## FFI 的主要风险点

### 1. 内存所有权不一致

最危险的问题通常不是“能不能调通”，而是：

- 谁分配内存
- 谁释放内存
- 用哪边的分配器释放

如果这件事没说清，最容易出严重问题。

### 2. 数据布局和 ABI 约定

跨语言边界时，要关心：

- 字段顺序
- 对齐
- 调用约定
- 枚举和布尔值的表示方式

这也是为什么 FFI 常会配合 `repr(C)`。

### 3. 空指针、悬垂指针、长度不匹配

跨边界后，Rust 类型系统的很多保障会弱化，必须自己补前置校验。

## 安全封装思路

一个更稳的模式通常是：

1. 最外层暴露安全 Rust API。
2. 内部单独一小层处理 `unsafe` / FFI。
3. 把所有前提条件写清楚。
4. 在边界处做尽量多的参数校验和转换。

也就是：把危险控制在固定边界，而不是让调用方到处直接碰原始指针。

## `unsafe fn` 不代表函数体自动 unsafe

这也是一个容易混淆的点。

`unsafe fn` 表示：

- 调用它需要调用方满足某些前提

而 `unsafe { ... }` 表示：

- 这一段代码执行了编译器无法自动验证的操作

这两者相关，但不是同一层含义。

## 审计 `unsafe` 时重点看什么

可以重点问下面几件事：

1. 这段 `unsafe` 为什么不可避免？
2. 依赖了哪些不变量？
3. 这些不变量由谁保证？
4. 是否已经把作用域缩到最小？
5. 对外是否暴露成了安全 API？

## 什么时候不该碰 `unsafe`

- 只是为了绕过借用报错
- 只是为了少写几次 `clone`
- 只是因为当前不想重构数据结构
- 团队没人能稳定审计这段底层逻辑

这些场景里，`unsafe` 基本都是坏信号。

## 工程建议

### 把 unsafe 留在底层封装

业务层尽量只看到安全 API。

### 每段 unsafe 都应该能说明安全前提

如果你解释不清“为什么这段是安全的”，通常就还不该提交。

### FFI 边界先明确内存责任

谁创建、谁销毁、谁拥有，必须先说清。

### 用测试和文档补编译器无法证明的部分

编译器不能证明，不代表可以不证明；只是证明方式要换成文档、断言、测试和审计。

1. `unsafe` 把什么责任从编译器转交给了开发者？
2. 为什么“借用报错就上 unsafe”通常是错误方向？
3. FFI 最危险的几个边界是什么？
4. 为什么 `unsafe` 应该缩在很小的封装层里？
5. `unsafe fn` 和 `unsafe {}` 分别在表达什么？
