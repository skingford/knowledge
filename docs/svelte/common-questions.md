# Svelte 高频问题：响应式、组件通信与模板陷阱排错清单

Svelte 的很多问题看起来像“编译器太聪明”或“语法太轻”，实际往往都能回到几个固定点：

- 值是不是放错了容器
- 派生值是不是误用了 `$effect`
- 组件通信是不是走成了隐式双向绑定
- 列表是不是没有稳定 key

这页把高频问题压成一份排错清单。

## 1. `$state`、`$derived`、`$effect` 到底怎么选？

务实口径：

- 需要响应式状态，用 `$state`
- 需要根据状态算值，用 `$derived`
- 需要和外部系统同步，用 `$effect`

只要你写的是“根据 A 推出 B”，优先怀疑它是不是该回到 `$derived`。

## 2. 为什么不要用 `$effect` 维护派生值？

因为这会把本来稳定的数据关系改造成“变化后再手动同步”的流程，后面更难追踪，也更容易出现循环和重复执行。

更稳的写法是：

```svelte
<script>
  let count = $state(0)
  let doubled = $derived(count * 2)
</script>
```

## 3. 为什么我在 `$effect` 里改 state，结果一不小心就循环了？

因为 effect 会追踪它读到的依赖；如果你在 effect 里又回写了这些依赖，很容易形成自触发链路。

排查顺序：

1. 先看这是不是派生值
2. 再看是否能移回事件处理函数
3. 最后才保留成真正的外部同步 effect

## 4. 什么时候该用 `$state.raw`？

当你拿到的是大对象或大数组，而且常见模式是整包替换而不是深层字段变更时，可以考虑 `$state.raw`，避免深代理带来的额外成本。

如果你需要频繁深层修改它，通常还是普通 `$state` 更自然。

## 5. 为什么 Svelte 5 更推荐回调 props，而不是 dispatcher？

因为回调 props 更直接，组件之间的依赖关系也更显式。父组件传什么动作给子组件，一眼就能看清楚。

适合：

- `submit`
- `select`
- `remove`
- `toggle`

## 6. `bind:` 很方便，为什么还是不能到处用？

因为双向绑定会让数据来源变得模糊。局部输入控件很好用，但页面级复杂状态如果到处 `bind:`，后面会越来越难判断是谁在改值。

更稳的口径：

- 局部输入：可以 `bind:`
- 跨组件业务协作：优先 props + 回调

## 7. 为什么列表最好写 keyed each？

因为一旦列表会插入、删除、排序，Svelte 需要靠 key 判断前后两次渲染的对应关系。没有稳定 key，局部状态、输入框和动画都更容易错位。

## 8. 为什么低频大组件不该一上来就同步打进首屏？

因为这类组件更适合延后加载。图表、编辑器、弹层、管理面板等低频模块，通常更适合用动态导入或路由级拆包。

## 9. 为什么 DOM 更新后拿不到最新结果？

Svelte 的 DOM 更新不是“你改 state 的那一行就立刻完成”。如果你要在更新后读取 DOM，通常需要等到 `tick()` 之后再操作。

## 10. context 和 store 什么时候用？

- context：一棵组件子树里深层共享状态
- store：需要独立模块化、跨组件树共享，或者要和外部订阅源协作

如果只是几个组件之间的普通协作，先别急着上 store。

## 11. snippet 什么时候有价值？

当你希望父组件把一段可渲染结构交给子组件，而不想再堆一串布尔 props 或模板分支时，snippet + `{@render}` 通常更清晰。

## 12. 页面慢的时候先看什么？

先区分：

- 是不是列表更新太多
- 是不是 effect 太多，把数据流写成了同步链
- 是不是低频重组件被提前加载了
- 是不是加载链路有瀑布请求

不要一上来就堆工具或改构建参数。

## 一份务实的排错顺序

1. 先确认是响应式问题、组件通信问题，还是纯性能问题
2. 看值是不是放错：`$state`、`$derived`、`$effect`
3. 看通信方式是不是选错：回调 props、`bind:`、context、store
4. 再看列表 key、加载链路和热点组件
