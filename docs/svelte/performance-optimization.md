# Svelte 性能优化指南：更新粒度、懒加载与列表性能

Svelte 性能优化最容易走偏的地方，是因为它默认已经很快，于是很多人直接跳过“先测量、再定位”这一步。更稳的顺序通常是：

1. 先确认是更新慢还是加载慢
2. 先处理列表、effect 和热点组件
3. 再考虑懒加载、代码分割和加载链路

## 适合谁看

- 页面已经能跑，但开始出现交互卡顿、列表更新慢、首屏重
- 团队里开始引入懒加载、拆包和复杂状态组织，但收益不稳定
- 想把 Svelte 性能问题变成“可测量、可定位、可回收”的优化工作

## 学习目标

- 分清 Svelte 的更新优化和加载优化分别看哪一层
- 掌握 `$derived`、`$state.raw`、keyed each 和列表拆分的主线
- 理解动态导入、代码分割和 SvelteKit 加载策略的适用边界

## 先分清更新慢还是加载慢

Svelte 性能问题通常分两类：

- 更新慢：交互后局部更新太重、列表太大、effect 链太长
- 加载慢：首屏包太大、低频组件被提前打进首屏、请求链路瀑布

问题没分清，就很容易把工具用错。

## 更新优化先看数据流和 effect

Svelte 的更新足够细，但如果你把很多本该写成 `$derived` 的逻辑改写成 `$effect` 链，更新成本和调试成本都会上升。

先看这些信号：

- effect 里大量回写 state
- 同一份数据被多处“监听后再同步”
- 模板中的派生值没有回收到 `$derived`

更稳的思路是：

- 状态放 `$state`
- 派生值放 `$derived`
- effect 只保留外部同步

## 列表性能与 keyed each

### 1. 优先 keyed each

这是列表优化的前提：

```svelte
{#each items as item (item.id)}
  <Row {item} />
{/each}
```

稳定 key 能让 Svelte 正确地插入、移动和删除节点，而不是把列表更新做成“整段重算”。

### 2. 列表项职责保持单一

更稳的做法是：

- 父组件负责筛选、排序和数据准备
- 列表项只负责展示和局部交互

不要让一个列表项同时承担重计算、复杂副作用和大段模板条件。

### 3. 特大列表优先考虑虚拟化

当一次要展示几百上千项时，真正瓶颈通常不是框架本身，而是 DOM 数量。这个阶段优先考虑虚拟列表，而不是先微调 runes。

## 大对象和大列表的状态组织

### `$state.raw`

如果接口返回的数据很大，而且常见更新方式是整包替换，可以考虑 `$state.raw`，减少深代理的成本。

### `$derived` 保持派生值稳定

过滤、排序、聚合和展示文案尽量回收到 `$derived`，这样依赖关系更清晰，也更容易定位更新热点。

## 懒加载、代码分割与 SvelteKit

### 动态导入低频组件

编辑器、图表、低频弹层、后台配置面板适合动态导入：

```svelte
{#await import('./HeavyPanel.svelte') then { default: HeavyPanel }}
  <HeavyPanel />
{/await}
```

### SvelteKit 默认支持拆包

如果使用 SvelteKit，优先让路由和低频模块保持自然拆分，不要急着把所有代码并回一个大包。

### 避免加载瀑布

如果一页里要拿多个资源，优先并发获取，避免串行等待把首屏拉长。

## 一份务实的优化顺序

1. 先确认是更新慢还是加载慢
2. 先看 `$effect` 是否误用了数据流同步
3. 处理 keyed each、列表项拆分和大列表 DOM 数量
4. 处理大对象状态组织，必要时考虑 `$state.raw`
5. 再做动态导入、路由拆包和加载链路治理
