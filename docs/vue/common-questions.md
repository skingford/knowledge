# Vue 高频问题：响应式、组件通信与模板陷阱排错清单

Vue 的很多问题看起来像“响应式失效”，实际往往能回到几个固定点：

- 值是不是放错了容器
- 派生值是不是误用了 `watch`
- 对象 / props / 列表是不是被错误解构或复用
- 组件切换是不是选错了 `v-if` / `v-show`

这页把高频问题压成一份排错清单。

## 1. `ref` 和 `reactive` 到底怎么选？

务实口径：

- 单值、简单状态，优先 `ref`
- 一组相关字段，考虑 `reactive`

不要为了“少写 `.value`”盲目上 `reactive`。团队协作里，`ref` 往往更明确。

## 2. 为什么模板里不用 `.value`，脚本里却要？

因为模板会自动解包 `ref`，脚本不会。模板里省略 `.value`` 是 Vue 的模板语法便利，不代表脚本里也一样。

## 3. 为什么我解构之后就不更新了？

最常见场景是直接解构 `reactive` 对象：

```ts
const state = reactive({ count: 0 })
const { count } = state
```

这样拿出来的 `count` 就不再是响应式来源本身了。普通 `reactive()` 对象直接解构要谨慎；如果确实需要拆开用，通常要改成 `ref` 组织，或显式处理引用关系。

## 4. `computed` 和 `watch` 怎么区分？

- `computed`：表达“值”
- `watch`：表达“动作”

如果你写的是“根据 A 推出 B”，优先 `computed`。如果写的是“当 A 变化时去请求 / 同步 / 清理”，才是 `watch`。

## 5. `watch` 和 `watchEffect` 怎么选？

- 已知监听源，用 `watch`
- 依赖自动收集更自然，用 `watchEffect`

如果你只是想“省事”，结果把副作用依赖写得越来越隐蔽，后面通常更难排查。

## 6. 为什么 watch 越写越多，页面越来越乱？

这通常不是缺少某个 API，而是状态设计已经开始重复：

- 本可 `computed` 的值被手动同步
- 本该事件里完成的逻辑被拆成监听链
- 表单状态、展示状态、请求状态耦在一起

先回头看状态结构，不要一味再加 watcher。

## 7. `v-if` 和 `v-show` 什么时候容易选错？

- `v-if` 会创建 / 销毁节点和组件
- `v-show` 只是切换显示状态

如果组件切换频繁，只是单纯隐藏，通常 `v-show` 更合适；如果你需要真正卸载和释放资源，用 `v-if`。

## 8. 为什么列表必须要稳定 `key`？

因为 Vue 需要靠 `key` 判断前后两次渲染中是不是同一个节点 / 组件。`index` 在插入、删除、排序后会漂移，局部状态、输入框、动画都可能错位。

## 9. 为什么子组件里不能直接改 props？

因为 props 仍然是单向下行数据。子组件改 props，会让数据流来源不清，也更容易和父组件状态冲突。

更稳的做法：

- 需要本地编辑时，复制成局部状态并明确同步策略
- 需要通知父组件更新时，用 `emit`

## 10. 为什么模板里一复杂就开始难维护？

典型信号：

- 模板里写很多链式表达式
- 条件判断嵌很多层
- 列表项里直接塞复杂计算

更稳的方式：

- 派生值移到 `computed`
- 行为移到方法
- 重复结构拆子组件

## 11. 为什么 DOM 更新后拿不到最新结果？

Vue 的 DOM 更新是异步批处理的。如果你需要在更新后再读取 DOM，通常要等到 `nextTick()`。

## 12. `provide` / `inject` 是不是可以代替所有 props？

不是。它适合深层共享，但不适合把一切状态都“藏起来”。一旦过度使用，依赖关系会变得隐式，调试成本会上升。

## 13. 为什么 `computed` 比 method 更适合模板里的派生值？

因为 `computed` 更明确表达“这是一个依赖响应式源的派生结果”，并且能避免模板中反复计算同样逻辑。method 更适合动作，不适合大量承担派生状态。

## 14. 页面慢的时候先看什么？

先区分：

- 是不是列表更新太多
- 是不是 props 不稳定导致子组件都在更新
- 是不是模板里有重计算
- 是不是加载太慢，需要异步组件 / 懒加载

不要先上来就到处缓存。

## 一份务实的排错顺序

1. 先确认是响应式问题、组件通信问题，还是纯性能问题
2. 看状态是不是放错容器：`ref`、`reactive`、`computed`、`watch`
3. 看 props、列表 key、`v-if` / `v-show` 是否选错
4. 再量化更新热点和加载热点
