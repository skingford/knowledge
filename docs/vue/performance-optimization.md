# Vue 性能优化指南：更新机制、异步组件与列表性能

Vue 性能优化最容易走偏的地方，是把所有问题都理解成“响应式太神秘”。实际更稳的顺序通常是：

1. 先确认是更新慢还是加载慢
2. 先优化 props、组件边界和列表更新
3. 再考虑缓存、异步组件和页面级缓存

## 适合谁看

- 页面已经能跑，但开始出现切换卡顿、列表更新慢、页面首屏重
- 团队里开始大量引入缓存、懒加载和组件抽象，但收益不清楚
- 想把 Vue 性能问题变成“可测量、可定位、可回收”的优化工作

## 学习目标

- 分清 Vue 的更新优化和加载优化分别看哪一层
- 掌握 props 稳定性、`computed` 稳定性和列表更新优化的主线
- 理解异步组件、路由懒加载、`KeepAlive`、`v-once`、`v-memo` 的适用边界

## 先分清更新慢还是加载慢

Vue 性能问题通常分两类：

- 更新慢：交互后局部或整页更新太重
- 加载慢：首屏包太大、组件加载太晚、路由切换成本高

如果问题都还没分清，就很容易把工具用错。

## 更新优化先看 props 稳定性

Vue 官方性能建议里，一个很核心的点是保持传给子组件的 props 尽量稳定。

典型反模式：

- 父组件每次更新都把“大对象”整包传下去
- 子组件实际只关心一个布尔状态，却拿到了整个上下文

更稳的方式：

- 在父组件先算好子组件真正需要的值
- 让子组件只接最稳定、最小的 props

```vue
<ListItem
  v-for="item in list"
  :key="item.id"
  :id="item.id"
  :active="item.id === activeId"
/>
```

这样变化时，只有真正 `active` 状态变化的项更容易触发必要更新。

## 让 computed 和模板保持稳定

Vue 3.4+ 对 `computed` 的稳定性做了进一步优化：如果 `computed` 的结果值没有变化，依赖它的 effect 不会无意义重跑。

工程上更稳的思路仍然是：

- 派生值优先放 `computed`
- 模板里不要塞重计算
- 不要让 method 替代一切派生状态

## 列表性能与组件边界

### 1. 稳定 `key`

这是前提。没有稳定 `key`，后续更新优化很难成立。

### 2. 列表项保持职责单一

不要让父组件一边筛选、一边排序、一边渲染复杂列表项。更稳的做法是：

- 父组件负责准备数据
- 子组件负责展示和局部交互

### 3. 大列表优先考虑虚拟化

当一次需要展示很多项时，真正的瓶颈通常不是单个指令，而是 DOM 数量本身。这个阶段优先考虑虚拟列表，而不是先到处加缓存。

## 异步组件、代码分割与 KeepAlive

### 异步组件

重组件、低频组件、弹层、图表、编辑器适合用异步组件拆开加载。

### 路由级懒加载

如果页面是多路由应用，优先从路由级拆包开始。

### `KeepAlive`

适合：

- tab 切换
- 多视图切换
- 希望保留组件内部状态的页面片段

同时要注意：

- 可以用 `include` 控制缓存范围
- 可以用 `max` 限制缓存数量，避免无限堆积

```vue
<KeepAlive :max="10">
  <component :is="activeComponent" />
</KeepAlive>
```

## `v-once` 和 `v-memo`

### `v-once`

适合完全不会再变化的静态片段。

### `v-memo`

适合你明确知道“只有某些依赖变了，才值得重新渲染”的场景。

这类指令属于局部优化工具，不应替代正常的组件边界和 props 设计。

## 一份务实的优化顺序

1. 先确认是更新慢还是加载慢
2. 先看 props 是否稳定、组件边界是否过粗
3. 把模板里的派生计算收回 `computed`
4. 处理列表 key、列表项拆分和虚拟化
5. 再考虑异步组件、路由懒加载、`KeepAlive`
6. 最后在明确热点处使用 `v-once`、`v-memo`
