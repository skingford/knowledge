# Vite 与 Vue 工程化指南：脚手架、环境变量与构建边界

Vue 项目里把 Vite 用好，关键不在于背配置项，而在于先搞清楚三件事：

1. Vite 负责什么，不负责什么
2. 开发期、构建期、类型检查分别在哪一层完成
3. 环境变量、静态资源和拆包边界应该放在哪里

如果这三件事混在一起，后面就很容易把构建工具当成“万能中间层”，项目越大越难维护。

## 适合谁看

- 已经在用 Vue + Vite，但对模式、环境变量和构建边界还没有统一口径
- 会跑 `vite dev` / `vite build`，但不知道类型检查、资源处理和拆包到底是谁在做
- 需要把 Vue 项目从“能启动”提升到“能解释工程链路、能定位构建问题”

## 学习目标

- 建立 Vite 在 Vue 项目里的统一位置：开发服务器、构建器和插件宿主
- 分清 `vite dev`、`vite build`、类型检查、ESLint、单测等职责边界
- 掌握环境变量、静态资源、依赖预构建和生产构建的稳妥用法

## 先明确 Vite 的职责

Vite 在 Vue 项目里主要承担三件事：

- 本地开发服务器
- 基于原生 ESM 的模块加载与 HMR
- 生产构建入口

它不负责：

- TypeScript 全量类型检查
- 业务代码分层设计
- 自动替你决定状态管理和请求层怎么写

官方文档明确把类型检查和静态分析从 transform 流程里拆开，这是 Vite 快的前提之一。

## 开发期和构建期各做什么

### 开发期

Vite 会按需编译模块，处理裸模块导入，并对依赖做预构建。Vue 项目里你通常感知到的是：

- 冷启动快
- HMR 快
- 改一个 `.vue` 文件不需要整包重编

### 构建期

生产环境直接跑：

```bash
npm run build
```

本质上就是 `vite build`。默认以项目根下的 `index.html` 为入口构建生产产物。

### 类型检查不要和 Vite 混成一件事

更稳的口径是：

- 开发编译交给 Vite
- Vue + TS 类型检查交给 `vue-tsc --noEmit`
- ESLint、测试各跑各的流水线

如果把这些都塞回 Vite transform 管线，通常只会拖慢反馈速度。

## 模式与环境变量

Vue 项目里最容易写乱的是环境变量。

### 稳定规则

- 客户端只读 `import.meta.env`
- 暴露到客户端的变量使用 `VITE_` 前缀
- 不把服务端密钥直接塞进前端构建变量

```ts
const apiBaseUrl = import.meta.env.VITE_API_BASE_URL
const isProd = import.meta.env.PROD
```

### 常见误区

- 把后端密钥放进 `VITE_` 变量
- 用很多分散的布尔变量控制环境
- 把业务开关和部署环境混成同一层

更稳的做法是：

- 环境变量只放部署相关配置
- 业务开关走独立配置中心或 Feature Flag

## 静态资源与类型检查

### 静态资源

Vite 会处理 HTML 和模块中的静态资源引用。Vue 项目里最常见的是：

- 模板里的图片、视频等资源
- CSS `url()`
- 组件中导入的 SVG、图片、字体

大原则很简单：

- 静态字符串资源路径让构建器处理
- 动态路径和复杂逻辑自己显式导入

### 类型检查

Vue + TS 项目里，建议把类型检查脚本单独设清楚：

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vue-tsc --noEmit && vite build"
  }
}
```

这样团队对“为什么本地能跑、CI 却失败”会更容易达成共识。

## 生产构建与拆包边界

### 1. 先做天然拆分

先按路由、重组件和低频模块做自然拆分，不要一上来就手写一堆 `manualChunks`。

### 2. 静态分析和构建配置分开

如果项目需要：

- 额外类型检查
- ESLint 浏览器提示
- bundle 分析

优先通过独立命令或插件补充，而不是把所有规则写进一个巨大 `vite.config.ts`。

### 3. 依赖预构建是性能手段，不是问题掩盖层

`optimizeDeps` 适合解决：

- 某些依赖在 dev 环境解析慢
- CommonJS 包开发体验差

但如果项目大量依赖解析异常，先排查依赖结构和导入方式，不要只靠“强行 include 一切”。

## 一套务实的工程建议

1. 先把 `dev`、`build`、`typecheck` 责任拆清楚
2. 环境变量只放部署配置，不混业务密钥和业务开关
3. 先靠路由和重组件做天然拆分，再考虑构建优化
4. 不把 Vite 配置文件写成项目总控中心
